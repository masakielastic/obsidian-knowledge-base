# James M Snell 氏の新しい Stream API 提案をどう読むか
## Web Streams の代替案ではなく、責務分解の再提案として読む

James M Snell 氏の `new-streams` は、現時点では実験的な提案として読むのがよい。  
これは「すぐに標準 API が置き換わる話」ではなく、JavaScript の Stream API をどう再設計すべきかを考えるための設計実験である。

この提案の重要点は、単に「今の Web Streams は遅い」「もっと軽い API がほしい」という不満ではない。  
むしろ、既存の Streams API が 1 つの抽象に多くの責務を詰め込みすぎているのではないか、という問題提起にある。

自分なりに整理すると、従来の Web Streams は少なくとも次の責務をまとめて扱っている。

- データ供給
- 読み取りと書き込み
- バッファ管理
- backpressure
- 分岐
- transform
- cancel / abort / error propagation
- reader / writer の所有権管理

James 氏の提案は、これらをいったんまとめて握るのではなく、もっと露骨に分けて考えようという方向に見える。

## 1. 「Streams are just iterables」という発想

James 氏の案で最も象徴的なのは、ストリームを特別な巨大オブジェクトではなく、反復可能なデータ列として捉え直す点である。  
中心にあるのは `AsyncIterable` / `Iterable` であり、専用の複雑な reader / writer 儀式を減らすことが意図されている。

これはかなり大きな発想転換だと思う。  
従来の Web Streams は「ストリーム専用の世界」を持っているが、James 案は JavaScript がもともと持っている反復プロトコルへ寄せる。

つまり、

- ストリーム固有の抽象を増やす
- 既存の言語機能に近づける

の後者を選ぼうとしている。

この発想は、Promise と Future の整理にも少し似ている。  
新しい概念を増やすより、すでにある言語の中心概念に寄せることで責務を減らす、という方向である。

## 2. bytes only という割り切り

James 氏の案は、対象を基本的に bytes に限定しようとしている。  
ここで見えてくるのは、「何でも流せる抽象」にするか、「I/O の本流に特化した抽象」にするかという設計判断である。

自分の理解では、James 案は前者を捨てている。  
つまり、

- 任意の値の流通は iterable 側で扱う
- Stream API はバイト列 I/O に集中する

という責務分離である。

これはかなり筋が良い。  
なぜなら、HTTP、ファイル、ソケット、圧縮、暗号化のような現実の I/O は、最終的にはほぼ bytes の問題だからである。  
逆に言えば、オブジェクトストリームまで 1 つの抽象に押し込むと、汎用性は上がるが責務が膨らみやすい。

## 3. backpressure を内部事情ではなく、利用者の方針として露出する

既存の高水準 API では、backpressure はしばしば「内部でうまくやってくれるもの」と見なされる。  
しかし実際には、

- 遅い読者をどう扱うか
- バッファをどこまで許すか
- 溢れたときに待つのか、落とすのか、失敗させるのか

は、アプリケーションやランタイムの運用ポリシーである。

James 案はそこを API の表面に戻そうとしている。  
これは、サーバーサイドやランタイム設計の文脈ではかなり自然である。  
逆にブラウザ寄りの「使いやすい共通 API」としては、少し生々しすぎるかもしれない。

ここで重要なのは、backpressure を単なる内部実装の都合ではなく、**流量制御の方針** として扱っていることである。

## 4. tee の便利さより、分岐の意味を分ける

分岐も単なる便利機能ではなく、

- 複製するのか
- 共有するのか
- 遅い読者をどう扱うのか

を分けて考えるべきだ、という発想が見える。

ここでも責務分解がある。  
「分岐」という 1 語でまとめると簡単そうに見えるが、実際にはそこに

- メモリ消費
- 同期ずれ
- バッファの蓄積
- 読者ごとの差異

が入り込む。

James 案は、そこを API の都合で丸めず、意味の違いを露出しようとしているように見える。  
この点は、ランタイム実装や高負荷サーバーの観点ではかなり重要である。

## 5. transform を eager ではなく pull-through に近づける

「変換」と言うと単純そうだが、実際には

- 上流が書いたらすぐ変換するのか
- 下流が必要になった時点で変換するのか

で挙動が変わる。

前者は push 寄り、後者は pull 寄りである。  
James 案は後者を強めようとしている。

これは SSR、プロキシ、圧縮、段階的変換のような処理ではかなり理にかなっている。  
必要になるまで仕事を進めすぎないことで、不要な先行処理や隠れたバッファリングを減らせるからである。

この論点は、単なる Stream API の話というより、**変換の実行責務をどこに置くか** の問題として読むべきだと思う。

## 6. sync / async 分離を API に出すか、実装に隠すか

ここは設計思想の差がかなりはっきりしている。

- 性能上の差異を API に露出してでも、明示的に扱いたい
- 性能最適化は実装側の責務であり、利用者には高水準 API を見せたい

この対立は、単なる好みではなく、どこまでを抽象の責務とみなすかの違いである。

自分はこの点を、Promise / Future や Fiber / Event Loop の整理と同じ文脈で見たい。  
つまり、利用者モデルを単純化するためにランタイムが責務を負うのか、それとも性能や制御性のために API 側へ責務を露出するのか、という問題である。

この論点は、JavaScript の Stream API に限らず、PHP や Rust の async 設計を考えるときにもかなり重要になる。

## 7. 反論側から見て重要な点

この提案に対する反論で重要なのは、少なくとも次の 2 点である。

1つ目は、**標準 API の性能問題と実装品質の問題を混同してはいけない** という点である。  
ナイーブな実装が遅いからといって、API 自体が悪いとは限らない。  
仕様が定める observable behavior と、実装がどこまで最適化されるかは分けて考える必要がある。

2つ目は、**error handling の議論が薄いと危険** という点である。  
Stream の設計では、happy path よりも

- cancel
- abort
- underlying source / sink の failure
- 接続された処理への伝播

のほうが難しいことが多い。

とくに I/O 系は、壊れたときにどう止まるかが設計の核心になる。  
したがって、新しい抽象を評価するときは、使い勝手や性能だけでなく、失敗時の責務分解も確認する必要がある。

## 8. この提案から何を学ぶか

この提案の価値は、「James 案を採用すべきか」にすぐ飛びつくことではないと思う。  
むしろ次の問いを学ぶ材料として重要である。

**Stream API は何を抽象化すべきで、何を隠しすぎてはいけないのか。**

自分なりのまとめはこうである。

- Web Streams は、高水準で統一的な API を提供する
- James 案は、その統一のコストを見える化する
- 議論の本質は、API の見た目の好みではなく責務分解にある

その意味で、James M Snell 氏の提案は「次世代標準候補」というより、**Stream を値・流れ・分岐・圧力・変換・所有権・実行経路に分けて再考するための学習材料** として価値が高い。

## 自分向けメモ

この話は、JavaScript だけの話ではない。  
PHP の Stream API、Fiber、Event Loop、HTTP クライアント設計、さらには HTTP/2 や QUIC のようなプロトコル実装ともつながる。

今後の自分の関心としては、次を整理したい。

- Web Streams と James 案の責務分解の違い
- Node.js streams との違い
- PHP の Stream / Socket / Fiber / Event Loop の責務分解との対応
- Abort / cancel / error propagation をどう比較するか
- bytes only という割り切りを、HTTP やプロトコル実装の観点からどう評価するか

この論点は、単なる API 比較ではなく、**ランタイム設計と抽象化の境界をどう引くか** の問題として読むと面白い。

## 一言でまとめる

James M Snell 氏の新しい Stream API 提案は、  
「Web Streams の置き換え候補」というより、  
**Stream API の責務をどこまで統合し、どこから分離すべきかを再点検するための設計メモ**  
として読むと理解しやすい。
