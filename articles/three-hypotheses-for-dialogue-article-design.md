# 3つの仮説思考は、なぜクラス設計の対話形式記事づくりに便利なのか

仮説思考には、**論点仮説**、**アプローチ仮説**、**答え仮説**の3種類があると言われる。  
多くの入門書では「仮説思考」と言うと答え仮説、つまり「結論はおそらくこうだろう」という予想を指すことが多い。だが、実際に複雑なテーマを解説したり、対話形式の記事を書いたりしようとすると、それだけでは足りない。

特に、**クラス設計**、**オブジェクトの責務分解**、**Stream / Socket / Fiber / Event Loop の違い**のような、読者が混乱しやすいテーマでは、この3つを区別して考えることが非常に役に立つ。

## 対話形式の記事は「答え」だけでは成立しない

通常の解説記事では、書き手が頭の中で整理した結論をそのまま並べても、ある程度は記事として成立する。  
しかし対話形式では、登場人物が疑問を持ち、言い間違えたり、途中で誤解したりしながら理解を進めていく。そのため、単に最終結論を知っているだけでは不十分になる。

たとえばクラス設計の話で、

- Stream はデータオブジェクトなのか
- Socket は自律的に振る舞う主体なのか
- Fiber は制御の主体なのか、それとも単なる実行コンテキストなのか
- Event Loop は誰が何を判断しているのか

といった問いを扱う場合、読者がどこでつまずくのかを見越して、対話の流れそのものを設計しなければならない。

ここで必要になるのが、3つの仮説である。

## 1. 論点仮説は「何を争点にするか」を決める

対話形式の記事づくりでまず重要なのは、**どこを論点として立てるか**である。  
これが論点仮説だ。

クラス設計の解説でよくある失敗は、テーマが広すぎることだと思う。  
「オブジェクト指向とは何か」から始めてしまうと、話が広がりすぎて、読者が何を持ち帰ればよいのか分からなくなる。

そこで論点仮説として、たとえば次のように置く。

- 読者は「オブジェクト＝自己判断する主体」と思い込みがちではないか
- そのため Stream や Socket をうまく分類できず、設計が曖昧になるのではないか
- ならば今回の論点は「オブジェクトの分類方法」に絞るべきではないか

このように論点仮説を立てると、記事の焦点が定まる。  
つまり対話形式の記事において論点仮説は、**登場人物が何について悩むのか**を決める役割を持つ。

ここを外すと、対話がどれだけ自然でも、結局は論点がぼやけた記事になる。

## 2. アプローチ仮説は「どう理解させるか」を決める

次に必要なのがアプローチ仮説である。  
これは「この論点は、どう料理すれば読者に伝わるか」という仮説だ。

クラス設計の話は抽象的になりやすい。  
そのため、定義を先に並べるだけでは読者が置いていかれやすい。  
ここでアプローチ仮説として、

- まず誤解しやすい見方を出す
- 次にその見方の限界を示す
- そのあとで分類の枠組みを示す
- 最後に具体例で再確認する

という流れが有効ではないか、と考えることができる。

あるいは、

- 初学者に素朴な疑問を言わせる
- 解説者が整理の軸を示す
- 実装者が現実の設計判断に引き戻す

という三者構成が有効ではないか、という仮説も立てられる。

これがアプローチ仮説である。  
つまりアプローチ仮説は、**対話の進め方そのものの設計図**になる。

対話形式の記事は、単に会話調で書けばよいわけではない。  
どの順番で誤解を出し、どの段階で整理し、どこで具体例を出すかを設計する必要がある。  
その設計を支えるのがアプローチ仮説だと言える。

## 3. 答え仮説は「今回どこに着地するか」を決める

最後に答え仮説である。  
これは「最終的にどんな理解にたどり着くか」の見取り図になる。

たとえばクラス設計の記事なら、答え仮説として、

- オブジェクトは一律に「主体」と見なすべきではない
- 少なくとも「主体」「値」「ハンドル」のように分類して考えたほうが理解しやすい
- Stream / Socket はデータではなく、外部資源への操作能力を持つハンドルとして捉えると整理しやすい
- Fiber はI/Oの担当ではなく、実行の中断と再開の文脈で理解すべきだ

という着地を想定できる。

この答え仮説があることで、対話の途中に出てくる話題を取捨選択しやすくなる。  
面白そうな脇道があっても、今回の着地点に寄与しないなら削る判断ができる。

対話形式の記事では会話が広がりやすいが、答え仮説があると、**どこまで寄り道してよいか**の判断がしやすくなる。

## 3つの仮説が揃うと、対話は「探索」ではなく「案内」になる

対話形式の記事が難しいのは、書き手が自由にしゃべらせようとすると、会話がただの雑談になりやすいからだと思う。  
逆に結論を急ぎすぎると、対話形式にした意味がなくなる。

ここで3つの仮説を使うと、記事づくりを次のように整理できる。

- **論点仮説**：今回、どこを問題として切り出すか
- **アプローチ仮説**：その問題をどういう順番と形式で理解させるか
- **答え仮説**：最終的にどこへ着地させるか

この3つがあると、対話形式の記事は「思いつきの会話」ではなく、**読者を迷わせずに考えさせるための設計物**になる。

言い換えると、3つの仮説思考は、対話形式の記事における**探索空間を制御するための道具**である。  
何を話すか分からないまま会話を始めるのではなく、

- 何を探すかを決める
- どう探すかを決める
- どこにたどり着きたいかを決める

という順番で設計できるからである。

## クラス設計の記事との相性が特によい理由

この考え方がクラス設計の記事で特に便利なのは、クラス設計の議論そのものが、**正解を丸暗記する話ではなく、論点の切り方と整理の仕方が重要な領域**だからだ。

同じテーマでも、

- 責務で切るのか
- 所有関係で切るのか
- ライフサイクルで切るのか
- 判断主体か操作ハンドルかで切るのか

によって、説明の仕方は大きく変わる。

つまりクラス設計の解説では、いきなり「正しい答え」を言うよりも、

- 何を論点として選ぶか
- どういう見方で整理するか
- その整理からどんな答えが見えるか

を段階的に見せるほうが、本質が伝わりやすい。  
これはまさに3つの仮説思考と相性がよい。

## 記事を書く前の簡易テンプレート

対話形式の記事を書く前に、次の3点を書き出しておくと整理しやすい。

### 論点仮説
今回、読者がいちばん混乱しているのは何か。  
どの誤解を中心論点に置くか。

### アプローチ仮説
その誤解は、どんな順番で崩していくと理解しやすいか。  
誰にどの役割を持たせるか。

### 答え仮説
最終的に読者に持ち帰ってほしい整理は何か。  
今回の記事であえて扱わない範囲はどこか。

この3点が先に見えているだけで、対話形式の記事の質はかなり安定する。

## まとめ

対話形式の記事づくりでは、答えを知っているだけでは足りない。  
読者がどこで迷うか、どういう順番なら理解しやすいか、最後に何を持ち帰らせるかまで設計する必要がある。

そのとき役に立つのが、論点仮説、アプローチ仮説、答え仮説という3つの仮説思考である。

特にクラス設計のような抽象的で誤解が起きやすいテーマでは、この3つを分けて考えることで、

- 対話の焦点が定まり
- 会話の流れを設計しやすくなり
- 記事の着地点もぶれにくくなる

対話形式の記事を書くとは、単に会話文を並べることではない。  
**読者の理解の探索を設計すること**であり、そのための実践的な道具として3つの仮説思考は非常に有用である。
