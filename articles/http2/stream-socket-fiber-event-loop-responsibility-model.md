# Stream / Socket / Fiber / Event Loop の責務分解を踏まえた再整理

## はじめに

PHP の Stream API や `stream_select()`、Fiber を使った非同期 HTTP クライアントの設計を考えていると、コードの書き方以前に、言葉そのものの意味や責務の切り方を整理し直す必要があると感じる。

特に混乱しやすいのは、`Stream` や `Socket` をひとつのオブジェクトとして見たときに、それが読み書きだけでなく、監視や待機、場合によっては再試行やタイムアウト管理まで自分で引き受けるもののように見えてしまうことである。  
しかし、そのような見方をすると、オブジェクトに意思を持たせすぎることになり、設計が God Object に近づいてしまう。

ここでは、Stream / Socket / Fiber / Event Loop を、オブジェクト指向の観点からいったん整理し直す。

---

## 最初の混乱：Stream が何でもできるように見える

PHP の Stream API を学び始めると、次のような直感を持ちやすい。

- Stream は読み書きできる
- `stream_select()` にも同じ stream を渡す
- ソケット通信も stream として扱える
- 非同期処理でも stream が中心に見える

このため、頭の中では次のような像ができやすい。

- Stream という単一のオブジェクトが存在する
- そのオブジェクトが read / write する
- そのオブジェクトが「今読めるか」「今書けるか」も知っている
- 必要なら待機も担当する

この直感は自然ではあるが、責務分解としては危うい。  
なぜなら、読み書きと監視は本来別の役割だからである。

---

## 読み書きと監視は別の責務

I/O まわりの整理でまず重要なのは、`read` / `write` と `readable` / `writable` の判定を分けることだと思う。

- `read` / `write` は操作そのもの
- `readable` / `writable` の判定は、その操作を今進められるかどうかの観測

つまり、ここには少なくとも二つの層がある。

1. **何をするか**
   - 読む
   - 書く

2. **いつできるか**
   - 今読めるか
   - 今書けるか

`stream_select()` は後者に属する。  
`stream_select()` 自体はデータを読んだり書いたりしているわけではなく、どのストリームでどの操作を進められるかを観測しているだけである。

この意味で、I/O は単なる実行の問題ではなく、**調停の問題**でもある。

---

## Stream / Socket は何者なのか

### 主体ではない

Stream や Socket をオブジェクト指向で考えるとき、まず外したいのは「自己判断する主体」としての見方である。

Stream や Socket は、少なくとも HTTP クライアント設計で問題になる文脈では、

- 自分で次の行動を決めるわけではない
- 自分で待機戦略を決めるわけではない
- 自分で再試行やタイムアウトを判断するわけではない

その意味で、主体ではない。

### ただし単なるデータでもない

一方で、Stream や Socket を単なるデータオブジェクトと呼ぶのも少し違う。

- 単なる値ではない
- 外部資源に対する操作を持つ
- `read` / `write` の入口になる

したがって、より正確には次のように言うほうがよい。

> Stream / Socket は、外部資源への操作能力を表す受動的なハンドルである。

ここで重要なのは「受動的」という点である。  
操作は持つが、意思決定は持たない。

---

## オブジェクトの分類を見直す

この整理をすると、オブジェクトを一律に「状態と振る舞いを持つ主体」と考えるのは無理があるとわかる。  
少なくとも、次のような分類を意識したほうが理解しやすい。

### 1. 主体

判断するもの。状態遷移を持ち、条件に応じて振る舞いを変える。

例:

- HTTP クライアント
- Scheduler
- Retry ポリシー
- Connection Manager

### 2. 値

情報を表すもの。比較的受動的で、不変に近い構造を持つ。

例:

- Request
- Response
- Header
- URI
- 設定値

### 3. ハンドル

外部資源へのアクセス口。自分では判断しないが、操作能力を提供する。

例:

- Stream
- Socket
- File Handle
- Database Connection

この分類で見ると、Stream / Socket は「主体」ではなく、「値」でもなく、**ハンドル**に属する。

---

## Fiber は何者なのか

Fiber はさらに別種の概念である。

Fiber は Stream のような資源ハンドルではない。  
Request / Response のような値でもない。  
また、Scheduler のように全体を見て判断する主体でもない。

Fiber は、

> 中断・再開できる処理の流れ

である。

オブジェクト分類の観点では、Fiber は「制御フロー」や「実行単位」として見るのが自然だと思う。  
これを Stream や Socket と同列に置くと混乱する。

重要なのは、Fiber が I/O を監視するわけではないということだ。  
Fiber は「待つ仕組み」ではなく、「途中で止まり、あとで再開できる実行の流れ」である。  
監視の結果を受けて再開される側であって、監視そのものを担う側ではない。

---

## Event Loop という言葉の誤解

`Event Loop` もかなり誤解を招きやすい言葉だと思う。

多くの場合、この言葉を聞くと「while ループを回す仕組み」を連想する。  
しかし、本質は単なるループ構造ではない。

少なくとも I/O を扱う文脈では、Event Loop の中核は次の三つにある。

1. 監視
2. 選択
3. ディスパッチ

つまり、

- 何が進められるかを観測し
- どの処理を進めるかを決め
- 適切な実行単位を再開する

ということが本体である。

この意味で、Event Loop というより、

- Monitor
- Scheduler
- Dispatcher

と分けて考えたほうが見通しがよい場面も多い。

ループはあくまでその実行形態にすぎず、本質ではない。

---

## 責務ごとに並べ直す

ここまでの整理をもとに、非同期 HTTP クライアントで関係しそうな主要な役割を並べると次のようになる。

### Stream / Socket
- 外部資源への操作口
- 読み書きの対象
- 受動的なハンドル

### Monitor
- readable / writable の観測
- `stream_select()` のような readiness 判定
- 「いつできるか」を扱う

### Scheduler / Dispatcher
- どの処理を進めるかを決める
- Fiber や task を再開する
- 調停者として振る舞う

### Fiber
- 中断・再開される処理の流れ
- 同期風に書かれた手続きを途中停止可能にする

### Protocol
- HTTP のメッセージ構造や状態遷移を理解する
- ヘッダ・ボディ・接続再利用などのルールを扱う

### Client Facade
- 利用者向け API
- `get()` や `send()` のような入口
- 内部の複雑さを隠す

この整理をすると、「誰が何を知っていて、何を知らないべきか」がだいぶはっきりする。

---

## God Object 的な Stream 設計の限界

設計上の誘惑として強いのは、Stream にいろいろ詰め込みたくなることだと思う。

たとえば次のようなものをひとつのオブジェクトに持たせたくなる。

- read / write
- 監視
- timeout
- retry
- parser
- protocol state
- 接続再利用
- 再送制御

しかし、これをやると責務が混線する。

- 監視の都合がプロトコル層に漏れる
- プロトコル状態が資源ハンドルに埋め込まれる
- 再試行やタイムアウトの戦略が低レベル資源に貼り付く
- テストしづらくなる
- HTTP/2 のような多重化ではさらに破綻しやすくなる

背景にあるのは、ストリームを擬人化し、「状況を見て自分で進むオブジェクト」と見なしてしまうことだと思う。  
しかし実際には、進むかどうかを決めるのは外側の監視やスケジューラである。

---

## HTTP/2 を見据えると責務分離の価値が増す

HTTP/1.1 の単純なクライアントであれば、多少責務が混ざっていても動いてしまうことがある。  
1 接続 1 リクエストのような単純な構成なら、設計上の粗さが目立ちにくいからである。

しかし HTTP/2 を考えると事情が変わる。

- 1 接続上に複数ストリーム
- フレーム単位の多重化
- 接続レベルとストリームレベルの状態が分かれる
- フロー制御や dispatcher が重要になる

この状況では、「ひとつの Stream オブジェクトが自分で全部判断して進む」という設計は苦しくなる。  
だからこそ、HTTP/2 の詳細に入る前の段階で、

- ハンドル
- 監視
- 調停
- 実行単位
- プロトコル
- 利用者向け API

という責務分解を身につけておく意味が大きい。

---

## 自分なりの再定義

今回の整理を通じて、自分の中では次のように言い換えるとしっくりくる。

- **Stream は主体ではなく、接続口である**
- **Socket も通信そのものではなく、端点へのハンドルである**
- **Fiber は待つ仕組みではなく、止めて再開できる処理の流れである**
- **Event Loop の本質は while ループではなく、監視とディスパッチである**

さらに言えば、

> 非同期 HTTP クライアントの設計は、便利そうな単一オブジェクトを作ることではなく、意思決定をどこに置くかを整理することである。

---

## まとめ

PHP の Stream API を見ていると、Stream という単一のオブジェクトに読み書きも監視も状態管理もまとめたくなる。  
しかし、その見方はオブジェクトを擬人化しすぎており、責務を混線させやすい。

少なくとも次のように整理したほうがよい。

- Stream / Socket は受動的なハンドル
- Monitor は観測を担当する
- Scheduler / Dispatcher は調停を担当する
- Fiber は中断・再開される処理の流れ
- Protocol は HTTP の意味を扱う
- Client Facade は利用者向けの入口

コードを書く前に、この責務の地図を頭の中で言語化しておくことが重要である。  
特に HTTP/2 のような多重化プロトコルを見据えるなら、God Object 的な設計の誘惑を早めに断ち切る必要がある。

オブジェクト指向で重要なのは、「何でもできるオブジェクト」を作ることではない。  
むしろ、**どのオブジェクトが主体で、どのオブジェクトが値で、どのオブジェクトがハンドルで、どこに意思決定を置くのかを見極めること**が重要なのだと思う。
