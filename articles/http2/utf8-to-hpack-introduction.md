# UTF-8 から学ぶ HPACK 入門 ― HTTP/2 ヘッダ圧縮を「状態を持つバイナリ構造」として読む

HTTP/2 を学び始めると、多くの人はまずフレーム共通ヘッダに出会います。Length、Type、Flags、Stream ID という 9 バイトの共通ヘッダを読み、その後ろに payload が続く。ここまでは比較的理解しやすい領域です。共通ヘッダは固定長で、先頭 9 バイトを読めば次に何バイト読めばいいかがわかるからです。

しかし、HEADERS フレームの中身に踏み込んだ瞬間に、難しさの質が変わります。そこで登場するのが HPACK です。HTTP/2 を「固定長ヘッダ + 可変長 payload を持つフレームのプロトコル」として読めるようになった人でも、HPACK では急に見通しを失いやすい。なぜかといえば、HPACK は単なるバイナリ構造ではなく、「状態を持つ圧縮付きの表現」だからです。

この違いを理解するために、UTF-8 から HTTP/2 フレームへと進んだ見方を、さらに HPACK へ延長してみます。UTF-8 は「可変長をどう読むか」を学ぶ題材として優れており、HTTP/2 フレームは「構造化されたバイナリをどう切り分けるか」を学ぶ題材として優れています。そして HPACK は、その次の段階として「状態を伴う圧縮表現をどう復元するか」を学ぶ題材になります。

## UTF-8 と HTTP/2 フレームの延長線上に HPACK がある

UTF-8 は、先頭バイトを見て 1 バイト文字なのか、2 バイト文字なのか、3 バイト文字なのか、4 バイト文字なのかを判断します。単に長さが変わるだけでなく、継続バイトの形や overlong、surrogate などの妥当性も気にしなければなりません。つまり UTF-8 では、「何バイト読むか」と「その並びが正しいか」が密接に結びついています。

HTTP/2 フレームは少し性質が違います。先頭 9 バイトの共通ヘッダを読むと、そのフレームの payload の長さと種類がわかります。UTF-8 が「先行バイトで長さを知る」方式だとすれば、HTTP/2 は「共通ヘッダで長さを知る」方式です。どちらも、先頭の情報が後続の解釈を決めるという点では共通しています。

元の整理では、こうした読み方を iterator / parser / validator の3層で考えるとわかりやすいとされていました。iterator は境界を切り出す役割、parser は意味のある構造に変換する役割、validator は妥当性や制約を確認する役割です。UTF-8 はこの3つがかなり密接で、HTTP/2 は frame の切り出し自体は単純だが type ごとの制約や状態遷移の検証が難しい、という整理です。

HPACK は、この枠組みをさらに押し広げる題材です。なぜなら HPACK では、iterator / parser / validator に加えて、「state」が強く絡むからです。過去に何が送られたかによって、今読んでいるバイト列の意味が変わる。ここが UTF-8 や単純なフレーム構造との決定的な違いです。

## HPACK とは何か

HPACK は、HTTP/2 でヘッダフィールドを効率よく送るための圧縮方式です。HTTP/1.1 では、`Host` や `User-Agent` や `Cookie` のようなヘッダが毎回長いテキストで繰り返し送られがちでした。HTTP/2 では、ヘッダをただバイナリ化するだけでなく、頻出のヘッダ名・値や過去に送ったヘッダを再利用することでサイズを削減します。

そのため HPACK には、ざっくり言えば次の要素があります。

- 頻出ヘッダを番号で参照する仕組み
- 過去に送ったヘッダを動的に蓄える仕組み
- 整数を短く表す可変長エンコーディング
- 文字列を Huffman 符号化する仕組み

つまり HPACK は、単なる「ヘッダのバイナリ表現」ではありません。圧縮、参照、状態管理が組み合わさった表現です。ここを「難しそう」とだけ感じるのではなく、どの種類の難しさが重なっているのか分解して見ることが重要です。

## HPACK の入口は、UTF-8 に似た「可変長整数」

HPACK を初めて見ると難しそうに感じますが、最初の入口は意外と UTF-8 に近いところにあります。それが整数表現です。

HPACK では、1 バイト目の一部のビットを「種類の判定」に使い、残りのビットを「整数の先頭部分」に使います。もしそのビット幅に収まらない値なら、追加バイトを使って続きを表します。この追加バイトは、単純に 1 バイトずつ増えるのではなく、「まだ続くかどうか」を示しながら値を延長していく形になります。

ここで重要なのは、「最初のバイトが二つの役割を持つ」ことです。

- どの種類の表現かを示す
- 整数の最初の部分を運ぶ

UTF-8 でも先頭バイトは似た役割を持っています。何バイト文字かを示しながら、同時にコードポイントの一部も運んでいる。継続バイトが続くかもしれないという点も似ています。もちろん具体的なビット配置は違いますが、「先頭の情報で分岐し、足りなければ continuation 的に後ろを読む」という発想はかなり近い。

この見方をすると、HPACK の整数は突然現れる異世界の仕組みではなく、「可変長整数の一種」だと理解できます。

## HPACK の1バイト目は、HTTP/2 フレームの Type に似ている

HTTP/2 フレームでは、共通ヘッダの Type を見て、これは DATA なのか HEADERS なのか SETTINGS なのかを判断します。Type が分岐点になっているわけです。

HPACK でも同じように、1 バイト目の上位ビットを見て「この表現は何か」を判断します。たとえば、

- indexed header field なのか
- literal header field with incremental indexing なのか
- dynamic table size update なのか

といった分岐です。

この意味で、HPACK の 1 バイト目は HTTP/2 フレームの Type に相当する役割を持っています。もちろん、HTTP/2 フレームの Type は専用の 1 バイト全体を使っており、HPACK はビット単位でより詰め込んでいます。しかし「先頭の情報が、後続の解釈ルールを決める」という本質は共通です。

ここまでを見ると、HPACK の前半はまだ「可変長 + 分岐」の話として追えます。UTF-8 や HTTP/2 の学習経験が、そのまま土台になります。

## 本当の難所は dynamic table である

HPACK が本当に難しくなるのは、dynamic table が絡んでからです。

HTTP/2 フレームだけなら、基本的にはその場で完結して読めます。あるフレームを受け取ったら、共通ヘッダを読み、その payload を読めばよい。もちろん状態遷移はありますが、少なくとも payload の内容そのものは、その場のバイト列だけでかなり読めます。

しかし HPACK では、indexed representation を見ても、それが何を指すかは static table と dynamic table の内容に依存します。static table は決め打ちなのでまだよいとして、dynamic table は通信の流れの中で増減します。つまり、同じ index 番号でも、いつ読んだかによって意味が変わりうるのです。

ここで HPACK の本質が見えてきます。

HPACK は「今このバイト列をどう解釈するか」だけの問題ではなく、「これまで何が送られてきたか」を含めて解釈する問題です。

この性質のために、HPACK の decode は純粋に入力バイト列だけで決まるわけではありません。直前までの状態が必要です。これは UTF-8 や多くの単純なバイナリ形式と違い、かなり強い特徴です。

## static table と dynamic table をどう教えるか

HPACK を教えるときに、いきなり仕様書的に static table 一覧を並べても、あまり腹落ちしません。むしろ、役割の違いから入ったほうがわかりやすいです。

static table は、最初から全員が共有している辞書です。たとえば `:method: GET` や `:path: /` や `:status: 200` のような、HTTP で頻出するヘッダ表現があらかじめ番号づけされています。これは言わば「共通語彙」です。

dynamic table は、通信の途中で新しく登録される辞書です。たとえば、あるホスト名や cookie や独自ヘッダが何度も出てくるなら、最初に一度送ったあと、それを table に積んで以後は番号で参照できます。これは言わば「会話の途中で共有された略語」です。

この比喩を使うと、HPACK の動作はかなり人間的に見えます。最初から知っている語彙は static table、会話の途中で「以後これを略して呼ぼう」と共有した語彙は dynamic table です。そして decode 側は、その略語帳を相手と同期させながら読まなければならない。

## HPACK は「構造」より「履歴」の学習が必要

HTTP/2 フレーム入門では、9 バイトの共通ヘッダと payload の関係を見れば、かなり機械的に encode / decode の往復ができました。payload を作り、Length を埋めて frame にする。逆に 9 バイトを読んで Length を得て、そのぶん payload を読む。この往復は比較的対称です。

HPACK では、この対称性が少し崩れます。decoder の仕事は復元ですが、encoder の仕事は「どの表現を選ぶのが得か」を考えることまで含むからです。

たとえば、同じヘッダ集合を送るとしても、

- static table の index だけで表すのか
- literal として送るのか
- dynamic table に追加するのか
- Huffman を使うのか

といった選択肢があります。復元だけなら「書いてある通り読む」だけで済みますが、生成側は圧縮効率やテーブル運用を考える必要があります。ここで encode と decode は、同じ仕様の両側でありながら、難しさの質が違ってきます。

したがって HPACK は、「構造からバイト列を作る」「バイト列から構造を復元する」という単純な往復変換よりも、「状態を保ちながら最適な表現を選ぶ」問題に近づきます。ここが HTTP/2 フレーム単体より一段難しい理由です。

## Huffman 符号化は、さらに別の種類の難しさを足す

HPACK には Huffman 符号化もあります。ここでまた別の難しさが加わります。

UTF-8 は文字エンコーディングですが、圧縮ではありません。HTTP/2 フレームの共通ヘッダも圧縮ではありません。しかし HPACK の文字列は、必要に応じて Huffman 符号化された形で入ってきます。つまり「文字列をそのまま読む」のではなく、「一度圧縮表現を展開してから文字列として扱う」段階が入ります。

このため HPACK は、少なくとも次の三つの処理を持つことになります。

1. 表現種別を判定する
2. integer / string を読む
3. 必要なら Huffman を展開する
4. table を参照または更新する

この四段階のうち、1 と 2 は構文解析に近く、3 は圧縮アルゴリズムに近く、4 は状態機械に近い。つまり HPACK は、複数の分野の難しさが重なった場所です。

## iterator / parser / validator / state で HPACK を整理する

元記事で使われていた iterator / parser / validator の見方は、HPACK ではさらに有効になります。むしろ HPACK では state を足して 4 層で考えたほうが整理しやすいです。

### iterator
どこまでが 1 つの表現なのかを切り出します。可変長整数や文字列長を読みながら、次の境界を見つけます。

### parser
切り出したバイト列を、indexed header なのか literal header なのかといった意味のある構造に変換します。

### validator
index が有効か、table size update が妥当か、表現として不正な並びになっていないかを確認します。

### state
dynamic table の内容を保持し、更新し、次の decode に引き継ぎます。

この整理を UTF-8、HTTP/2、HPACK に並べると次のようになります。

| 項目 | UTF-8 | HTTP/2 frame | HPACK |
|---|---|---|---|
| iterator | 文字境界を切る | 9バイトヘッダ＋payload境界を切る | 可変長整数や文字列境界を切る |
| parser | コードポイントにする | frame構造にする | header field 表現にする |
| validator | UTF-8妥当性を確認 | type/flags/stream制約を確認 | index/table/update の整合性を確認 |
| state | 基本なし | ストリーム状態あり | dynamic table が本質 |
| compression | なし | なし | Huffman あり |

この表を見ると、HPACK は単に HTTP/2 の一部というより、「状態付き圧縮表現」という独立した難しさを持っていることがわかります。

## 最初の実験題材は「よくある疑似ヘッダ」がよい

HPACK を学ぶとき、最初から複雑な cookie や長いヘッダ値を扱うと見通しが悪くなります。まずは、static table に載っている頻出の疑似ヘッダを題材にするのがよいです。

たとえば、

- `:method: GET`
- `:scheme: https`
- `:path: /`
- `:authority: example.com`

のようなものです。

このうち static table にあるものは index だけで表せるので、「こんな短いバイト列でこんなヘッダになるのか」という感覚を得やすい。ここで HPACK の「番号参照による圧縮」を体感できます。

その次に、

- literal で新しいヘッダを送る
- incremental indexing で dynamic table に積む
- 次回以降は index 参照で短く送る

という流れに進むと、dynamic table の意味がはっきりします。

## decode の練習は「この1バイト目は何を意味するか」から始める

HPACK を読む練習では、最初から全体を見て理解しようとすると難しいです。むしろ、1 バイト目に注目して、

- これは indexed か
- literal か
- size update か
- prefix は何ビットか

を判定する練習から始めるのが有効です。

このやり方は、UTF-8 で「この先頭バイトは 1 バイト文字か、2 バイト文字か、3 バイト文字か」を見る感覚に近い。あるいは HTTP/2 の共通ヘッダで Type を先に見る感覚にも近いです。

つまり HPACK を読む第一歩は、「まず最初の分岐点を見つける」ことです。そこから整数を読み、必要なら文字列を読み、table 操作を行う。全部を一度にやろうとせず、分岐点から段階的に追うのがよいです。

## 実装設計として見ると、HPACK は層を分けないと破綻しやすい

教育だけでなく実装の観点でも、HPACK は層を分けて設計しないと見通しが悪くなります。たとえば次のような責務分離が考えられます。

- prefix integer を読む小さな関数
- string literal を読む関数
- Huffman decode を行う関数
- representation の種類を判定する parser
- dynamic table を管理する state オブジェクト
- 不正な index やサイズ超過を扱う validator

この分離を意識しないと、「1 つの巨大な decode 関数」が生まれやすく、理解もテストも難しくなります。逆に言えば、HPACK は「小さな primitive に分けて理解する」ことの大切さを教えてくれる題材でもあります。

これは UTF-8 を iterator / validator の観点から整理する発想や、HTTP/2 フレームを fixed header と payload に分けて考える発想の延長線上にあります。全部を一塊で見ないことが重要です。

## HPACK を学ぶと「状態を持つプロトコル」を読む感覚が育つ

HPACK の学習価値は、HTTP/2 のヘッダ圧縮を知ることだけではありません。もっと一般的には、「状態を持つプロトコルをどう読むか」という感覚を鍛えるところにあります。

単純なバイナリ形式なら、バイト列をその場で読めば済みます。しかし実際のネットワークプロトコルでは、

- 以前のやり取りを前提にした圧縮
- テーブル同期
- 状態遷移
- エラー時の扱い

といった要素がよく出てきます。HPACK はその入門として非常に良い題材です。フレーム構造だけでは見えなかった「履歴依存の解釈」が前面に出てくるからです。

## まとめ

UTF-8 は、「先頭情報を見て可変長データを読む」感覚を教えてくれます。HTTP/2 フレームは、「固定長ヘッダで構造を切り出す」感覚を教えてくれます。そして HPACK は、その次の段階として「状態を持つ圧縮表現を復元する」感覚を教えてくれます。

HPACK の難しさは一つではありません。可変長整数の難しさ、表現種別の分岐の難しさ、dynamic table の状態管理の難しさ、Huffman 符号化の難しさが重なっています。だからこそ、「なんとなく難しい」で終わらせず、難しさの種類を分解して理解することが大切です。

見方を変えれば、HPACK はとてもよい教材です。UTF-8 と HTTP/2 フレームで学んだことを、さらに一段深いレベルに進めてくれるからです。構造を読むだけではなく、履歴を読む。固定の形式を読むだけではなく、共有された辞書を前提に読む。そこに HPACK の面白さがあります。

HTTP/2 を学んでいて HEADERS フレームの中身で急に霧が濃くなったと感じたなら、それは自然なことです。そこは、単なるフレーム構造の延長ではなく、「状態付き圧縮プロトコル」の入口だからです。UTF-8 から HTTP/2 へ進んだように、HTTP/2 フレームから HPACK へ進むときも、発想を一段切り替える必要があります。その切り替えを意識できると、HPACK はぐっと読みやすくなります。
