# クラスやオブジェクトの責務をどう切るか  
— Promise / Future / Fiber を題材にした整理

非同期処理の設計を考え始めると、すぐに「このクラスは何を表しているのか」「このオブジェクトは何を担当するのか」が曖昧になりやすい。

とくに PHP で Fiber を使った設計や、Promise / Future 的な抽象を考え始めると、次のような混線が起きやすい。

- Fiber があれば Promise は不要ではないか
- Promise と Future は同じではないか
- cancel は Promise に持たせればよいのではないか
- イベントループが全部面倒を見るなら、結果オブジェクトは不要ではないか

こうした混乱は、個々の技術要素の知識不足だけで起きるわけではない。  
むしろ多くの場合は、**クラスやオブジェクトの責務の切り方が曖昧なこと**によって起きる。

このメモでは、非同期設計を題材にしながら、クラスやオブジェクトをどのように分類して考えると理解しやすいかを整理する。

---

# なぜ「オブジェクトは何でもできる主体」と考えると混乱するのか

オブジェクト指向の入門では、しばしばオブジェクトを「自分で判断して動くもの」として説明することがある。  
この説明は一部では役立つが、設計を考える場面ではそのままだと危うい。

実際のプログラムには、少なくとも次のような違いがある。

- 自分で判断して振る舞いを選ぶもの
- 単に状態や結果を保持するもの
- 外部資源への操作窓口として振る舞うもの
- 複数の要素を調停するもの

これらを全部「主体」として一括りにすると、責務が膨らみやすい。  
Promise が結果も表し、実行も開始し、cancel も持ち、イベントループとの橋渡しまで引き受ける、というような設計になりやすい。

その結果、API 利用者にも実装者にもわかりにくいオブジェクトができてしまう。

---

# まずはオブジェクトを4種類に分けて考える

非同期設計に限らず、クラスやオブジェクトは次の4種類に分けて考えると整理しやすい。

## 1. 主体（判断や進行を担うもの）

これは、状況に応じて次の動きを選び、処理を進めるものだ。

例:

- Scheduler
- EventLoop
- RetryPolicy
- StateMachine
- Dispatcher

これらは単なる値の入れ物ではない。  
いつ何を実行するか、どの順序で進めるか、失敗時にどう振る舞うか、といった判断を持つ。

## 2. 値（状態や結果を表すもの）

これは「何かの結果」や「今の状態」を保持するものだ。

例:

- Future の完了結果
- HTTP レスポンス
- URI
- 設定オブジェクト
- エラー情報

値オブジェクトは、普通は自分から処理を進めない。  
それは「何かを表す」ことが責務であり、「誰かを動かす」ことは本来の役割ではない。

## 3. ハンドル（外部資源への操作窓口）

これは、外部の資源や仕組みに対する操作能力を持つものだ。

例:

- Stream
- Socket
- File handle
- DB connection
- Timer registration

ここで重要なのは、ハンドルは「受動的だから値オブジェクトだ」とは言えないことだ。  
Stream や Socket は自分で意思決定する主体ではないが、単なる値でもない。  
それらは **外部資源に触るための操作窓口** である。

## 4. 調停役（複数の責務のあいだをつなぐもの）

これは、主体・値・ハンドルのあいだをつなぐためのオブジェクトだ。

例:

- Deferred
- CancellationTokenSource
- Adapter
- Bridge
- Resolver

たとえば Deferred は、利用者に見せる Future と、内部で結果を確定する処理のあいだをつなぐ。  
これを Future 自身に全部持たせると、読み取り側と書き込み側が混ざる。

---

# Promise / Future / Fiber をこの分類で見る

この4分類で見ると、Promise / Future / Fiber の違いがかなりはっきりする。

## Fiber は主体そのものではなく、実行機構に近い

Fiber は「処理を途中で止めて、あとで再開する」ための仕組みである。  
これは Future のように将来値を表すものではないし、Promise のように結果を確定させるための窓口でもない。

Fiber は、役割としては **実行制御のための低レイヤー機構** であり、分類としては「主体が使う機構」に近い。

Fiber 自体をそのまま公開 API の中心に置くと、利用者は次のことまで意識しなければならなくなる。

- いつ開始するか
- いつ再開するか
- どこで待つか
- 例外はどう伝播するか
- 複数処理はどう合成するか

つまり Fiber は、それ単体で利用者向けの意味づけが完成しているわけではない。

## Future は将来値である

Future は「まだ来ていない結果」を表す。  
これは基本的に値の側に属する。

もちろん `await` されたり、コールバック登録の窓口を持ったりはするが、その中心的責務は「将来得られる結果を表す」ことにある。

Future の利用者は、普通は次のことだけをしたい。

- 完了を待つ
- 失敗を受け取る
- 複数の結果を合成する

この Future に `resolve()` や `reject()` を持たせると、利用者側と実装側の責務が混ざる。

## Promise / Deferred は調停役である

Promise という言葉は文脈によって揺れるが、設計上は「結果を確定させる側」として分離して考えたほうがよい。  
PHP では `Deferred` という名前のほうが役割が見えやすい場合も多い。

Deferred は、Future に対応する「書き込み側のハンドル」であり、外から見れば調停役である。

- 内部では I/O 完了やタイマー発火を受け取る
- その結果を Future に流し込む
- 利用者には Future だけを返す

この構造にすると、Future は読み取り専用に保てる。

---

# JavaScript の Promise をそのまま模倣しないほうがよい理由

JavaScript の Promise は歴史的にはとても合理的だった。  
コールバック地獄を整理し、`then` や `await` による統一的な非同期 API を広めた意味は大きい。

ただし、PHP でクラスやオブジェクトの責務を整理する観点から見ると、そのまま模倣しないほうがよい理由がある。

## 1. Promise が多くの役割を背負いやすい

JavaScript の Promise は、利用者から見ると

- 将来値であり
- コールバック登録先であり
- `await` の対象であり

という複数の顔を持っている。

これは JavaScript の文脈では便利だが、PHP で同じ形をそのまま持ち込むと、  
「このオブジェクトは結果なのか、制御窓口なのか」が曖昧になりやすい。

## 2. eager 実行の感覚を持ち込みやすい

JavaScript の Promise は生成時に処理が始まる eager な性質を持つ。  
しかし PHP の非同期設計では、開始タイミングをもう少し明示的にしたい場面がある。

たとえば:

- 並列数を制御したい
- まだ実行キューに入れたくない
- timeout や cancellation を先に設定したい

このとき、結果オブジェクトと開始トリガーが強く結びついていると扱いづらい。

## 3. cancel を結果オブジェクトに持たせたくなる

JavaScript では Promise 自体は cancel を直接持たず、AbortController のような外部トークンが使われる。  
これは結果オブジェクトと実行制御を分けるための整理でもある。

PHP でも同様に、

- Future は結果
- CancellationToken は中断要求
- Scheduler や I/O 層が実際の停止処理を担う

と分けたほうが責務がきれいになる。

## 4. PHP は実行環境の幅が広い

PHP には、ブラウザ JavaScript のような一枚岩の実行モデルはない。

- 通常のリクエスト実行
- CLI
- ワーカー
- イベントループ常駐型プロセス
- 各種拡張による I/O
- Stream / Socket / curl など複数の経路

この環境で長く使える API を考えるなら、単一の便利オブジェクトに寄せるより、責務分離を優先したほうが保守しやすい。

---

# Stream / Socket は主体ではないが、値オブジェクトでもない

ここは誤解されやすいので、非同期設計から少し離れても整理しておきたい。

Stream や Socket は、自分で考えて次の行動を決める主体ではない。  
だからといって、URI や日付のような純粋な値オブジェクトとも違う。

これらは **ハンドル** として理解するのがよい。

ハンドルの特徴は次の通り。

- 外部資源に紐づいている
- 操作メソッドを持つ
- 状態は外部世界の影響を受ける
- 同じ見た目でも、中で何が起きるかは外部依存である

たとえば `read()` は単なる計算ではない。  
OS やネットワークや相手側の状態に依存する操作であり、そこには「外界への窓口」という責務がある。

この視点を持つと、Stream や Socket を「ただのデータ入れ物」と誤解しにくくなる。

---

# EventLoop や Scheduler は「全部入りの神オブジェクト」ではない

非同期設計では EventLoop や Scheduler が強い存在感を持つので、何でもここに押し込みたくなる。  
しかしこれも責務分離を崩しやすい。

EventLoop の責務は、基本的には次のようなものに留めるべきだ。

- 監視対象の登録
- 準備完了やイベント発火の受け取り
- 対応するコールバックや再開処理の実行

そこにさらに

- Future の結果管理
- タイムアウトポリシーの詳細
- キャンセル理由の解釈
- リトライ戦略
- ログ整形
- HTTP クライアント固有の状態遷移

まで押し込むと、EventLoop は巨大な神オブジェクトになる。

EventLoop は主体に近いが、あくまで「進行役」であって、アプリケーション固有の意味づけまで全部持つべきではない。

---

# クラス設計の目安

非同期設計で責務を見失いにくくするために、クラスを切るときの目安をまとめる。

## Future
責務:
- 将来得られる結果を表す
- 完了待ちや結果取得の窓口になる
- 合成の対象になる

持たせすぎないもの:
- 実際の I/O 開始責任
- cancel の直接実装
- スケジューリング方針

## Deferred / Promise
責務:
- Future を完了させる
- 成功・失敗を一度だけ確定する
- 内部実装と公開 Future の橋渡しをする

持たせすぎないもの:
- 利用者向けの複雑な合成ロジック
- 実行全体のポリシー判断

## Fiber
責務:
- suspend / resume による実行の切り替え

持たせすぎないもの:
- 将来値の意味づけ
- API 利用者向けの完了モデル
- timeout / cancel / all / race の高レベル抽象

## CancellationToken
責務:
- 中断要求を表す
- 実行主体へ中断意思を伝える

持たせすぎないもの:
- 結果そのもの
- 実際の完了通知

## EventLoop / Scheduler
責務:
- 実行の順序や再開タイミングを管理する
- イベントや readiness を監視する

持たせすぎないもの:
- ドメイン固有の意味づけ
- 値オブジェクトの責務
- すべての設定と状態の集中管理

## Stream / Socket
責務:
- 外部資源への操作窓口になる

持たせすぎないもの:
- 上位プロトコル全体の状態管理
- 再試行戦略やポリシー
- 将来値の意味づけ

---

# 実装者向けの整理

実装時には、次の3層で考えるとかなり見通しがよくなる。

## 低レイヤー: 実行機構と外部資源
- Fiber
- EventLoop
- Stream
- Socket
- Timer

ここは「どう動かすか」の層である。

## 中レイヤー: 非同期抽象
- Future
- Deferred
- CancellationToken
- timeout
- all / race

ここは「利用者にどう見せるか」の層である。

## 高レイヤー: 具体的なアプリケーション API
- HTTP クライアント
- DB クライアント
- 非同期ストリーム API
- チャネル
- ワーカー制御

ここは「何をしたいか」の層である。

この3層を混ぜると、たとえば HTTP クライアントが Fiber の再開条件やイベントループ登録の細部まで直接抱え込むようになりやすい。  
逆に分けておけば、低レイヤーの変更が公開 API を壊しにくくなる。

---

# まとめ

クラスやオブジェクトの責務を整理するときは、まず「オブジェクトは何でもできる主体である」という見方から少し距離を取ったほうがよい。

非同期設計では特に、次の区別が重要になる。

- 主体: 進行や判断を担うもの
- 値: 結果や状態を表すもの
- ハンドル: 外部資源への操作窓口
- 調停役: 境界をつなぐもの

この観点で見ると、

- Fiber は実行機構
- Future は将来値
- Deferred / Promise は完了させる側
- CancellationToken は中断要求
- EventLoop は進行役
- Stream / Socket はハンドル

として整理できる。

重要なのは、これらを一つの便利オブジェクトに押し込まないことだ。  
JavaScript の Promise は歴史的には合理的だったが、PHP で設計するときは、その形をそのまま真似るよりも、責務分離を意識したほうが長く使える API になりやすい。

非同期設計で迷ったときは、「このオブジェクトは結果なのか、実行主体なのか、外部資源の窓口なのか、それとも境界をつなぐ役なのか」と問い直すと、かなり整理しやすくなる。
