# php-fpm との比較から ReactPHP の運用を学ぶ

date:2026-04-02
tags: #php
#reactphp
#php-fpm
#operations
#learning-memo

ReactPHP を php-fpm と比較して学ぶとき、最初に押さえたいのは「同じ PHP でも、運用モデルがかなり違う」という点です。ReactPHP はイベントループを中心にした低レベルの非同期基盤で、HTTP サーバー、ソケット、子プロセスなどを部品として提供します。一方の php-fpm は FastCGI のためのプロセスマネージャで、子プロセス数や再起動回数などの運用パラメータをあらかじめ備えています。つまり、php-fpm は運用機能込みの土台であり、ReactPHP は非同期アプリを組み立てるための基盤です。

php-fpm から学べるのは、「Web アプリの安定運用では、コード以外にプロセス管理が重要だ」ということです。たとえば php-fpm には `pm.max_children`、`pm.start_servers`、`pm.max_requests` などがあり、同時処理数の上限や、一定回数の処理後に子プロセスを入れ替える運用が標準で考慮されています。これはメモリリークや状態汚染を前提に、プロセスを計画的にリサイクルする思想です。

ReactPHP を学ぶときは、この php-fpm の「当たり前」をいったん分解して見ると理解しやすいです。ReactPHP ではイベントループが中心で、プログラムの冒頭で loop を作り、タイマーやソケット監視を登録し、最後に loop を回します。つまり発想の出発点が「リクエストごとに PHP を起動する」ではなく、「常駐プロセスの中で複数のイベントをさばく」です。

ここで学習上の大きな転換があります。php-fpm では、1 リクエストごとに世界がかなり初期化されるので、多少雑に状態を持っても被害が限定されます。ReactPHP では常駐プロセスなので、雑な static キャッシュやグローバル状態がそのまま残り続けます。だから ReactPHP の学習は、非同期 API の書き方だけでなく、**長寿命プロセスの責務**を学ぶことでもあります。これは「コードの書き方」より「運用前提の設計」の学習です。

さらに、ReactPHP では重い処理をそのままイベントループに載せてはいけません。ReactPHP は non-blocking I/O の基盤なので、同期的な外部 API 呼び出し、重い DB 処理、CPU 負荷の高い変換処理を混ぜると、1 件の遅延ではなく全体の詰まりになります。必要なら子プロセスに逃がす、という発想が重要です。ReactPHP には child-process コンポーネントがあり、子プロセスの起動、入出力のストリーム化、終了監視、シグナル送信ができます。

この比較から見えてくる学び方はかなり明確です。  
最初から「ReactPHP で本番 Web サーバーを作る」と考えるより、次の順番で学ぶほうがよいです。

まず php-fpm で暗黙に提供されていたものを列挙する。たとえば、プロセス数管理、一定回数での再起動、接続の受け皿、異常終了時の再起動です。次に ReactPHP では、それらが標準では自動で与えられないことを確認する。そのうえで、イベントループ、HTTP、Socket、ChildProcess を小さく試し、「どこまでがアプリの責務で、どこからが外部 supervisor の責務か」を切り分けて学ぶのがよいです。

学習課題としては、次の3段階がわかりやすいです。  
第一段階は、タイマーとソケットを使った小さな常駐プログラムを書くこと。第二段階は、HTTP サーバーを立てて、重い処理を child process に分離すること。第三段階は、systemd など外部のプロセス管理と組み合わせて、「ReactPHP 単体では足りない運用機能」を埋めることです。ここまで進むと、ReactPHP は単なる非同期ライブラリではなく、php-fpm とは別の運用思想を持つ基盤だと理解しやすくなります。

要するに、php-fpm との比較をもとに ReactPHP を学ぶとは、非同期プログラミングを学ぶこと以上に、**PHP における運用責務の所在を学び直すこと**です。php-fpm が隠してくれていたものを、ReactPHP では自分で見に行く必要があります。その意味で ReactPHP は、非同期のための道具であると同時に、運用の構造を理解するための教材でもあります。
