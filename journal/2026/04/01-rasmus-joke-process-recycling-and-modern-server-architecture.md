# Rasmus Lerdorf の「10リクエストごとに Apache を再起動すればいい」のジョークから学ぶ、現代サーバーアーキテクチャ

Rasmus Lerdorf の「メモリリークしていても Apache を 10 リクエストごとに再起動すればいい」という話は、文字通りの運用手順ではなく、初期 Web 開発の現実主義を誇張したジョークだと理解している。要するに、「完璧に壊れないものを作る」だけでなく、「壊れても短命なプロセスとして交換できるように設計する」という発想である。

このジョークが成立した背景には、当時の Web サーバーが短命な子プロセスやワーカーを入れ替えながら全体の健全性を保つ文化を持っていたことがある。CGI や mod_php の時代は、1 リクエストごと、あるいは一定回数ごとに実行単位を捨てやすく、メモリリークや状態汚染の影響を局所化しやすかった。ジョークの本質は「リークを直さなくていい」ではなく、「運用で被害を閉じ込める」実務感覚にある。

この発想は現代でも消えていない。PHP-FPM でも、ワーカーが一定回数のリクエストを処理したら再生成する設定が使われる。つまり「一定回数ごとにワーカーを入れ替える」は、昔話ではなく今でも通用する運用技法である。短命ワーカー型の利点は、アプリ側が多少不完全でも、プロセス寿命の制御で健全性を回復しやすいことにある。

ただし、現代のサーバーアーキテクチャはそこから広がった。従来の nginx や Apache + PHP-FPM のような構成は、接続管理と PHP 実行環境を分離し、HTTP サーバーとアプリ実行基盤の責務を分けやすい。一方で、ReactPHP、AMPHP、Swoole、RoadRunner、FrankenPHP のような長寿命プロセス型の世界では事情が変わる。ここではワーカーが生き続けるため、メモリリークやグローバル状態の汚染が次のリクエストにも持ち越されやすい。

だからこのジョークは、現代では対比として効いてくる。短命ワーカー型では「壊れたら入れ替える」がかなり有効だが、長寿命サーバー型ではそれだけでは足りない。状態管理、リソース解放、メモリ使用量の監視がずっと重要になる。言い換えると、Rasmus のジョークは昔の雑な笑い話ではなく、サーバーの実行モデルの違いを理解する入口になっている。

結論として、このジョークから学べるのは次のことだ。サーバー設計には「壊れないように作る」だけでなく、「壊れても交換できるように作る」という発想がある。そして現代は、短命ワーカー型と長寿命イベントループ型の両方を選べる時代になった。だからサーバーを理解するうえでは、言語やフレームワークの名前より先に、「このシステムはプロセスの再生成で健全性を保つ設計なのか、それとも長寿命のまま整合性を守る設計なのか」を見ることが大事になる。
