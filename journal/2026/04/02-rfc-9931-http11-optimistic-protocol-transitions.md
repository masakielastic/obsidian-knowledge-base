# 【学習メモ】RFC 9931: HTTP/1.1 の楽観的プロトコル切替はなぜ危険なのか

date: 2026-04-02
tags: #RFC9931 #HTTP11 #HTTP #ProtocolDesign #Security #RequestSmuggling #CONNECT #Upgrade #WebSocket #Networking #WebProtocols #ObsidianNotes

RFC 9931 を読んで整理したいポイントは、HTTP/1.1 では `Upgrade` や `CONNECT` の要求を送った直後に、切替先プロトコルのデータを先回りして送る実装が危険になりうる、ということだ。RFC 9931 は 2026年3月公開の Standards Track 文書で、RFC 9112 と RFC 9298 を更新している。主題は、切替成功の応答を受ける前に後続データを送る **optimistic protocol transition** の安全性だ。

要するに、HTTP/1.1 では「要求を送り終えた」ことと「遷移が成立した」ことは別物として扱わなければならない。クライアントは切替が通るつもりで新プロトコルのデータを流しても、サーバーはそれを拒否できる。拒否された場合、その同じバイト列が HTTP/1.1 の次のリクエストとして解釈される余地がある。RFC 9931 は、この解釈のずれが Request Smuggling や Parser Exploit の入口になると整理している。

ここで重要なのは、単なるクライアント対サーバーの二者間だけではなく、第三者の入力が混ざると危険が増すことだ。たとえばブラウザが他オリジン由来のデータを送る場合や、HTTP プロキシクライアントがローカルの別アプリの通信を中継する場合、切替後データの生成者と HTTP クライアント本体の trust boundary がずれる。すると、失敗した `CONNECT` の後ろに紛れ込んだデータが、接続単位の認証文脈を借りた HTTP/1.1 リクエストとして処理される危険が出てくる。

HTTP/1.1 で特に危ない理由は、HTTP/2 や HTTP/3 と違って、逐次的にバイト列を解釈するからだ。HTTP/2/3 は明示的なストリーム境界を持つが、HTTP/1.1 は前の要求がどう終わったかに依存して次を読む。そのため、クライアントは「もう新プロトコルに入った」と思い、サーバーは「まだ HTTP/1.1 のまま」と思う、という interpretation gap が生じやすい。RFC 9931 も、この問題は HTTP/2/3 では構造的に起きにくいと説明している。

既存仕様の扱いも興味深い。WebSocket は RFC 6455 側で、ハンドシェイク応答前にアプリケーションデータを送らない規律があり、さらにクライアント→サーバー方向のフレームにマスキングがあるので、防御的に働く。一方で `connect-udp` は、RFC 9931 により HTTP/1.x 上での楽観送信が禁止方向に修正された。つまり RFC 9931 は、HTTP/1.1 上の遷移では「先に少し流して latency を削る」という発想を抑制する文書だと読める。

実装者目線では教訓はかなり明快だ。HTTP/1.1 の `Upgrade` や `CONNECT` では、成功応答を受ける前に post-transition data を流さないこと。信頼できないデータ源を背後に持つプロキシやトンネル実装では特に厳守すること。RFC 9931 は HTTP/1.1 の CONNECT について、2xx を受けるまで TCP payload を転送しないか、`Connection: close` を付けることを求め、さらに拒否したサーバー側にも接続を閉じる要件を加えている。

学んだことを一文でまとめるなら、RFC 9931 は「HTTP/1.1 では、状態遷移前後の parser を同じバイトストリーム上に楽観的に共存させるな」と警告する文書だった。成功前と成功後を厳密に分離することが、安全性の中心になる。
