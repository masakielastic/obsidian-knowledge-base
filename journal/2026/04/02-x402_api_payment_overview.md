# HTTP 402でAPI課金を埋め込む決済プロトコルは何を変えるのか

date: 2026-04-02
tags: #payment #api #x402 #ai-agent #blockchain

x402 は、長く予約されていた HTTP の `402 Payment Required` を実際の支払いフローに結びつけるためのオープン標準だ。API やデータ、デジタルサービスに対して、クライアントがリクエストし、サーバーが 402 で支払い条件を返し、支払い後に同じリクエストを再試行する、という形で課金を HTTP の往復に埋め込む。2025年5月に公開され、2025年12月には V2 が発表された。

この話を理解する近道は、いまの AI API の使い方から考えることだ。開発者はすでに、カードでクレジットを買って内部残高を減らす方式や、カード登録後の従量課金に慣れている。x402 はその延長線上で、**チャージ元や都度決済をウォレットと HTTP に寄せる**発想だと見るとわかりやすい。公式ホワイトペーパーも、x402 の狙いを API キー、サブスクリプション、手動決済を減らし、stablecoin を使ったリアルタイムな machine-native transactions を可能にすることだと説明している。

従来の API 課金は、事前登録、API キー発行、利用量計測、月次請求や前払いクレジット管理が中心だった。これは人間向け SaaS や法人契約には強いが、単発利用や AI エージェント同士の少額決済には重い。x402 はそこを切り替えようとしている。サーバーは「このエンドポイントは有料です」と 402 を返し、クライアントは支払いを済ませて再試行する。つまり「契約ベースの API 提供」から「リクエストベースの API 販売」への転換だ。

実務では、x402 を毎回のフル決済だけで使う必要はない。V2 では reusable access、wallet-based identity、subscription-like や prepaid workflow が強化され、毎回 402→決済→再試行を繰り返さない構成も想定されている。つまり、`/topup` だけ x402 で課金し、その後はサービス内クレジットを減らす設計や、最初の支払い後は一定期間のアクセス権を再利用する設計も取りやすくなっている。これは AI API のプリペイド文化にかなり近い。

手数料の観点では、一般ユーザー向けには依然としてカードが強い。利用者が慣れており、経費処理や返金、請求管理も既存の枠組みに乗せやすいからだ。一方で、x402 が主に使う stablecoin 決済は、少額・越境・機械間決済で有利になりやすい。Stripe は machine payments と x402 対応を案内しており、Base と Solana 上の USDC を扱う。ホワイトペーパーも low fees や instant settlement を強く打ち出している。

したがって、AI サービス企業にとっての現実解は二者択一ではない。**人間向け UI ではカード、開発者向けではプリペイドクレジット、bot / agent 向けでは x402**という三層構造が自然だ。Cloudflare も 2025年9月に Coinbase と x402 Foundation を立ち上げ、Agents SDK と MCP Servers で x402 を支援すると発表した。x402 は一般消費者向けの万能決済というより、まず API、エージェント、機械間経済のための決済レイヤーとして伸びている。

要するに x402 は、「暗号資産で払う技術」よりも、**支払いを HTTP の標準的な会話に戻す試み**として見ると面白い。API キーでアクセス権を配る世界から、支払い可能性そのものをアクセス制御に使う世界への移行だ。AI エージェントが外部ツールやデータを自律的に買う未来を考えるなら、x402 はかなり筋の良い実験に見える。少なくとも、AI API のプリペイド文化を知っている人には、その延長として理解しやすい。
