# Pre-PEP: Rust for CPython の論点整理 — 段階設計とロードマップ設計の観点から

Pre-PEP: Rust for CPython で現在議論されているのは、CPython に Rust を全面導入するかどうかではない。現時点の提案は、**optional extension modules という限定範囲で Rust を認めるか、その実験をどう評価し、将来どこまで広げうるか**にある。required modules や interpreter core への導入は今回の提案には含まれておらず、未確定なのは、対象モジュールの選定、評価基準、将来の拡大範囲である。したがって、この議論は **現時点の提案範囲、将来のロードマップ、実験対象、評価基準**を分けて読む必要がある。

この整理が必要なのは、「CPython に Rust を導入するらしい」という話だけが独り歩きしやすいからである。実際のスレッドでは、提案のスコープ修正、optional modules の位置づけ、将来の拡張可能性、ビルドや保守体制への影響が複雑に絡んでおり、しかも議論の途中で提案内容そのものも調整されている。そこで本メモは、賛否を述べるためではなく、**何が提案され、どこが未確定で、どの論点を切り分けるべきか**を整理するためのメモとして位置づける。

背景として、PyPI 側では Rust 製 Python モジュール開発の足場がすでに整っている。maturin は PyO3・cffi・uniffi などを使った Rust 製 Python パッケージのビルドと公開を支援しており、複数 OS 向け wheel 構築にも対応している。さらに Linux カーネルでは Rust 用ドキュメントと Quick Start が整備され、2025年12月には Rust in the kernel が no longer experimental と整理された。Windows 側でも Rust による OS 周辺開発の取り組みが進んでいる。つまり、**Rust をシステムソフトウェアや Python 周辺で使うこと自体は、すでに特別な話ではない**。ただし、そのことと、CPython がどの段階でどこまで Rust を導入するかは別問題である。

現行案が optional modules に絞られているのは、そのためである。提案側は、まず小さな範囲でビルドシステム変更と C↔Rust 相互運用を試し、知見を積むことを第1段階の目的にしている。候補例としては `base64` や `json` のような既存 C 拡張の置き換えが挙がっているが、この二つは重みが違う。`base64` は比較的軽い実験対象だが、`json` は利用頻度が高く、名目上 optional でも事実上必須に見えやすい。したがって第1段階で重要なのは、Rust で書けることではなく、**失敗しても引き返せる対象を選べるか**である。

そのうえで論点は三段階に整理できる。  
**第1段階**は、optional modules でどこまで試すか。  
**第2段階**は、その実験をビルド、保守、配布、開発者受容まで含めてどう評価するか。  
**第3段階**は、その結果を受けて required modules や core まで広げる前提があるのか、あるならどこまでか。  
もちろん実装技術、保守者コミュニティ、埋め込み用途など別の切り口もあるが、本メモでは論点の散らばりを整理するために、**段階ごとのスコープとその先のロードマップ**を一つの整理軸として採用する。
