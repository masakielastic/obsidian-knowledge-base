# エンジニア求人情報の書き方 — エンジニア求人情報の書き方について

エンジニアの求人票を読むたびに、技術名だけが長く並んでいる欄に引っかかることがある。PHP、Go、TypeScript、AWS、Docker、Kubernetes と並んでいても、それだけでは会社の実態はほとんど見えない。情報量が多いようで、実は意味が薄い。応募者が知りたいのは、何を使っているかそのものではなく、何のために使っているか、どこで使っているか、どれくらい重要か、今後も使うのか、入社後にどのくらい関わるのかだと思う。そこが書かれていない求人票は、会社にとっても応募者にとってもミスマッチの原因になりやすい。

求人票には、技術の一覧ではなく、技術の位置づけを書くべきだと感じる。同じ PHP や Go でも、ある会社では中核技術で、別の会社では補助技術かもしれない。TypeScript も、フロントエンドの主力なのか、管理画面限定なのか、社内ツール中心なのかで意味が変わる。だから「中核技術」「補助技術・周辺技術」「歴史的資産」「実験・探索領域」のように整理するだけでも、技術一覧は会社の判断の地図になる。さらに「採用理由」「限定利用の理由」「期待役割」「温度感」を一言添えるだけで、解像度はかなり上がる。

ここで大事なのは、役に立つ技術だけを書く必要はないということだ。今は主力ではない技術、創業時に助けられた技術、既存顧客資産を支える技術、チームが愛着を持っている技術も、十分に書く価値がある。そうした記述から見えるのは、単なる技術選定ではなく、その会社がレガシーをどう扱うか、合理性と歴史をどう両立するか、個人の探究や好奇心をどの程度尊重するかという文化そのものだからだ。「主力ではないが知見として大事」「好きだから触っている」も、文脈つきで書かれていれば有益な情報になる。

複数の言語やフレームワークを使う会社ほど、さらに重要になるのがコミュニケーションの説明だと思う。技術名が多い会社を見ると、応募者は「結局どうやって会話を成立させているのか」と不安になる。技術スタックの多さは、整理された分業にも見えるし、属人的な混乱にも見える。その違いを分けるのは、共通言語と責務分解の説明があるかどうかだ。どの単位で責務を分けているのか。仕様変更や障害対応では、どのレイヤーの言葉で議論するのか。業務の言葉と設計の言葉をどうつないでいるのか。そこまで書いてある求人票は、組織としての成熟度が伝わる。

この話は DDD のユビキタス言語に近いが、専門用語を前に出しすぎなくても十分説明できる。重要なのは、業務の言葉、設計の言葉、実装の言葉の三層を混線させないことだと思う。強い組織は、実装言語が違っても、前二者がある程度そろっている。弱い組織は、実装の言葉だけが乱立して、肝心の業務や責務の会話ができなくなる。求人票にこの観点が入ると、「この会社はどんな問題を、どんな単位で、どんな言葉で扱っているのか」が見えてくる。

だから、良い求人票はスペック表ではない。「うちにはこんな技術があります」というカタログではなく、「うちはこういう問題を、こういう考え方で解いています」という自己紹介であるべきだと思う。応募数を増やすために単語を盛る文書ではなく、合う人に届き、合わない人とのミスマッチを減らすための文書として書く。その発想に切り替わると、技術欄の書き方もかなり変わるはずだ。技術スタックを書くなら、せめて用途、位置づけ、温度感、背景、会話の仕組みまで書く。そこまで書いて初めて、求人票は技術組織としての自己説明になる。
