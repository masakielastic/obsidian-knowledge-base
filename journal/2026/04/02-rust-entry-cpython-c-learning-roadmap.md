# Rust を入口に CPython を目指すなら、C 言語はどう学ぶべきか — 段階設計とロードマップ設計で考える学習ルート

date:2026-04-02
tags: #CPython #Rust #CLanguage #PythonInternals #LearningRoadmap #OSSDevelopment

Pre-PEP: Rust for CPython の議論を見ていると、「これからは C より Rust を学べばよいのか」と受け取りたくなります。ですが、少なくとも現時点の論点はそこまで単純ではありません。いま議論されているのは、CPython 全体を Rust で置き換える話ではなく、optional extension modules という限定範囲で Rust を認めるか、その実験をどう評価するか、そして将来どこまで広げうるかです。required modules や interpreter core への導入は今回の提案には含まれていません。

この整理を学習の話に引き寄せると、C 言語の学び方も同じく段階設計で考えるべきだとわかります。つまり、「C を全部学んでから CPython に行く」のではなく、「CPython に必要な C を、段階ごとに学ぶ」という設計です。Rust を入口にすること自体はかなり合理的です。所有権、エラー処理、境界設計、FFI、安全性への感覚は、C や CPython の実装を読むときの助けになります。ですが、CPython に参加するには最終的に C を避けて通れません。必要なのは、一般的な C 教養を網羅することよりも、CPython の都合に合わせて C を学ぶことです。

学習ルートは三段階で考えると整理しやすいです。

第1段階は、Rust を足場にしてシステムプログラミングの視点を得る段階です。ここでは高度な Rust 実装そのものより、メモリ、所有権、ビルド、テスト、FFI、配布といった観点に慣れることが大事です。CPython に関わると、言語仕様だけでなく、ビルドシステム、外部ライブラリ、保守体制、配布方法まで視野に入ります。Rust はその入口として優れています。Python パッケージ側では maturin や PyO3 のような足場もあり、「Rust で Python 周辺を触る」こと自体はすでに現実的です。

第2段階は、CPython に必要な C を限定的に学ぶ段階です。ここで重要なのは、C を教科書的に網羅することではありません。優先順位が高いのは、ポインタ、構造体、配列と文字列、ヘッダファイル、マクロ、関数ポインタ、メモリ管理、デバッガです。さらに CPython 文脈では、参照カウント、エラー時の戻り値、オブジェクト表現、GIL、拡張モジュール初期化といった独特の作法があります。Rust 経験者にとっての本当の課題は、「C は危険」という一般論を知ることではなく、「CPython ではどこが事故点で、どこに保守コストが出るのか」を理解することです。

第3段階は、CPython の開発文化に接続する段階です。ここまで来ると、問題は C の文法だけではありません。ローカルでビルドできること、テストを回せること、失敗時のログを読めること、小さな修正を安全に出せること、レビューで説明できることが重要になります。Rust for CPython の議論でも、技術的に書けるかどうかだけでなく、ビルド、保守、配布、コミュニティ受容が評価軸になっています。学習者側も同じで、単に C のコードを書けるだけでは足りません。どの変更が安全で、どの変更が引き返しやすく、どの変更が保守負担を増やすのかを判断する視点が必要です。

この観点から見ると、C 言語学習の目標も組み替えたほうがよいと思います。目標は「C をマスターすること」ではなく、CPython の C コードを読んで追跡できること、小さな修正を書けること、参照カウントやエラー処理の危険地点がわかること、そして変更範囲を小さく切って保守コストを意識できることです。

結局のところ、Rust を入口に CPython を目指す学習ルートとは、Rust から C に乗り換える話ではありません。安全な抽象化の世界から、互換性と保守責任を背負う実装の世界へ、段階的に降りていく学習ルートとして設計するのが自然です。Rust for CPython の議論が問うているのも、まさにその現実なのだと思います。
