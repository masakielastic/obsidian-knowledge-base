# UTF-8 から学ぶ HTTP/2 フレーム入門 ― 固定長・可変長・往復変換で読むバイナリデータ

date: 2026-04-03
tags: #utf8 #http2

バイナリデータというと、16進数が並ぶ難しい世界に見えがちです。しかし実際には、UTF-8 も HTTP/2 も「先頭の情報を見て、どこまでを一つの単位として読むか決める」という共通の発想で動いています。文字列処理とネットワークプロトコルは別分野に見えますが、どちらもストリームから構造を復元する技術だと考えると理解しやすくなります。

まず基本になるのが、固定長と可変長の違いです。固定長データは、たとえば4バイトごとに読むだけなので境界が明確です。一方、可変長データは無駄が少なく柔軟ですが、「次は何バイト読むのか」を知る仕組みが必要です。UTF-8 はその情報を先頭バイトに埋め込みます。ASCII は1バイト、より大きなコードポイントは2〜4バイトで表し、先行バイトのビットパターンを見ると長さがわかります。

UTF-8 はよく1〜4バイトの4分類として説明されますが、実装ではもう少し細かく見ます。E0、ED、F0、F4 のように、先行バイトによって後続バイトの許容範囲が変わるからです。つまり UTF-8 は、単に「何バイトか」を決めるだけでなく、「その並びが正しい UTF-8 か」を同時に確かめながら読む必要があります。ここに文字コード処理の難しさがあります。

HTTP/2 も可変長ですが、長さの持ち方が違います。HTTP/2 フレームは、先頭に9バイトの共通ヘッダを持ち、その中に Length、Type、Flags、Stream ID が入っています。UTF-8 が「先行バイトで長さを知る」方式なのに対し、HTTP/2 は「ヘッダで長さを知る」方式です。標準のフレームタイプは DATA、HEADERS、SETTINGS など10種類あり、Type を見れば次の解釈の分岐先が決まります。この意味では、UTF-8 の先行バイトと HTTP/2 の Type は、どちらも先頭情報による分岐点だと言えます。

ここで大事なのが、往復変換の視点です。構造からバイト列を作るのが encode、バイト列から構造を復元するのが decode です。HTTP/2 フレームは HPACK を除けば、共通ヘッダが固定なので比較的わかりやすく往復できます。payload を作り、その長さを Length に入れて frame を組み立てる。逆に decode 側では、9バイト読んで Length を得て、その分だけ payload を読む。この流れはかなり機械的です。対して UTF-8 は、decode の途中で継続バイトや overlong、surrogate などの妥当性検査が強く絡みます。HTTP/2 よりも「読むこと自体」が難しい面があります。

この違いは、iterator / parser / validator という3層で考えると整理しやすくなります。iterator は境界を見つけて次の単位を切り出す役割、parser はその単位を意味のある構造にする役割、validator はそれが妥当かどうかを確かめる役割です。UTF-8 は、この3つがかなり密接です。一方 HTTP/2 は、frame を切り出す iterator の部分は比較的単純で、難しいのは type ごとの制約や状態遷移を確かめる validator 側です。

この見方を持つと、文字コードの学習は単なる文字列処理にとどまりません。UTF-8 を読む感覚は、そのまま HTTP/2 や他のバイナリプロトコルを読む感覚につながります。大切なのは、コードを丸暗記することではなく、仕様から最小のテストデータを組み立て、逆にバイト列から構造を読み解く練習をすることです。UTF-8 から HTTP/2 へ進む流れは、その訓練としてとても相性がよい題材です。
