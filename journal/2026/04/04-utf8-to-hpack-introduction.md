# UTF-8 から学ぶ HPACK 入門 ― HTTP/2 ヘッダ圧縮を「状態を持つバイナリ構造」として読む

HTTP/2 を学ぶとき、最初の山場はフレーム共通ヘッダです。Length、Type、Flags、Stream ID からなる 9 バイトを読めば、次に何バイト読めばよいかがわかる。ここまでは比較的機械的に理解できます。ところが HEADERS フレームの中身に入ると、急に霧が濃くなります。その中心にあるのが HPACK です。

HPACK を難しく感じるのは自然です。HTTP/2 フレームまでは「構造を読む」問題でしたが、HPACK は「状態を持つ圧縮表現を読む」問題だからです。UTF-8 が可変長データの読み方、HTTP/2 フレームが構造化されたバイナリの読み方を教えてくれるとすれば、HPACK はその次の段階として、履歴に依存する表現の読み方を教えてくれます。

UTF-8 では先頭バイトを見て 1〜4 バイトの長さを判断します。HTTP/2 フレームでは 9 バイトの共通ヘッダを読んで payload の長さと種類を判断します。どちらも「先頭の情報が後続の解釈を決める」という点で共通しています。これを iterator / parser / validator の3層で整理します。iterator は境界を切り出し、parser は意味のある構造に変換し、validator は妥当性を確かめる、という見方です。

HPACK もこの延長で理解できます。ただし HPACK では、ここに state を足す必要があります。なぜなら HPACK は、今見ているバイト列だけでは意味が決まらず、過去に送受信されたヘッダ表現に依存するからです。

HPACK の入口としてわかりやすいのは整数表現です。1 バイト目の一部のビットで表現の種類を示し、残りのビットで整数の先頭部分を持ち、足りなければ追加バイトで続きを表します。これは UTF-8 の先頭バイトと継続バイトの関係に少し似ています。細部は違っても、「最初のバイトで分岐し、必要なら続きを読む」という発想は共通です。

さらに、HPACK の 1 バイト目は HTTP/2 フレームの Type にも似ています。indexed header field なのか、literal header field なのか、dynamic table size update なのかを、先頭ビットで判定するからです。つまり HPACK の第一歩は、UTF-8 の長さ判定や HTTP/2 の Type 判定と同じく、「最初の分岐点を見抜くこと」にあります。

しかし HPACK の本当の難所は dynamic table です。static table は最初から共有されている辞書ですが、dynamic table は通信の途中で追加・更新される辞書です。そのため、同じ index 番号でも、通信の時点によって意味が変わりえます。ここで HPACK は単なるバイナリ構造ではなくなります。decode は入力バイト列だけで完結せず、「これまでに何が送られたか」という履歴が必要になるのです。

この性質のため、HPACK は「構造を読む」よりも「履歴を読む」プロトコルだと言えます。HTTP/2 フレームまでは encode と decode が比較的対称でしたが、HPACK では encoder 側が「どの表現を選ぶと効率がよいか」まで考える必要があります。static table を使うのか、literal にするのか、dynamic table に積むのか、さらに Huffman を使うのか。decoder は復元すればよい一方、encoder は圧縮戦略まで担います。

したがって HPACK は、iterator / parser / validator に加えて state を意識すると理解しやすくなります。iterator は可変長整数や文字列境界を切り出し、parser は header field 表現に変換し、validator は index や update の妥当性を確認し、state は dynamic table を保持します。UTF-8 が「可変長を読む教材」、HTTP/2 フレームが「構造を読む教材」だとすれば、HPACK は「状態付き圧縮を読む教材」です。

HTTP/2 の HEADERS フレームで急に難しく感じるのは、そこで問題の種類が切り替わるからです。UTF-8 から HTTP/2 フレームへ進んだように、HTTP/2 フレームから HPACK へ進むときも発想の段差があります。その段差を「状態を持つ圧縮表現への移行」として意識できると、HPACK はずっと見通しよく読めるようになります。
