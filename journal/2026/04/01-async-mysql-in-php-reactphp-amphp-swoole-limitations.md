# ReactPHP / AMPHP / Swoole で MySQL 非同期処理をどう見るか — 待ち時間を重ねられるが、SQL そのものが速くなるわけではない

ReactPHP / AMPHP / Swoole での MySQL 非同期処理を考えるとき、いちばん大事なのは「DB を魔法のように高速化する技術」ではなく、「MySQL 待ちのあいだに別の仕事を進める技術」だと整理することだ。非同期化の対象は MySQL サーバーそのものではなく、PHP 側で MySQL ソケットを non-blocking に扱うことにある。書けるときに送信し、読めるときに受信し、その待ち時間をイベントループやコルーチンで他の処理に振り向ける。だから価値の本体は、個々の SQL の実行時間短縮ではなく、アプリ全体の I/O 待ちの重ね方にある。

この価値が見えやすいのは、php-fpm よりも ReactPHP / AMPHP / Swoole のような常駐型の実行環境だ。php-fpm では 1 リクエストが 1 ワーカーを占有し、その中で同期的に処理されるので、まずは SQL 改善やインデックス、N+1 解消、キャッシュのほうが優先されやすい。一方で、常駐プロセスでは 1 プロセスが多数の接続や I/O を抱えながら動くので、MySQL 待ちのあいだに別の問い合わせや別クライアントの処理を進められる意味が大きくなる。WebSocket、SSE、チャット、ジョブワーカーのような長寿命・多接続の文脈では特に効いてくる。

ReactPHP はイベントループと Promise 的な流儀が見えやすく、MySQL 非同期化も「I/O を待っているあいだに別の処理を回す」という構図が理解しやすい。AMPHP は非同期処理をより構造的に書きやすく、コードの見通しを保ちながら MySQL 待ちを扱いやすい。Swoole はさらにランタイムやサーバー機能まで含んだ強い基盤で、coroutine や hook によって同期風の書き味を保ちながら内部で待ち時間中の切り替えを行える。この 3 つは見た目の API や書き味はかなり違うが、内部で起きていることとしては、MySQL の I/O 待ちを他の仕事と重ねようとしている点で共通している。

ただし、ここで誤解しやすい限界もある。非同期化しても、遅い SQL は遅いままだし、ロック競合、不適切なインデックス、DB サーバー側の飽和は消えない。非同期化は「待ち方の改善」であって、「DB 内部の計算を速くする技術」ではない。また、1 本の MySQL 接続で何でも自由に多重化できるわけでもない。protocol の順序や状態管理があるので、実務では接続プールが重要になる。つまり、非同期 MySQL は単なる便利 API ではなく、接続の割り当て、packet のやり取り、state machine、read/write readiness をどう扱うかまで含んだ設計課題になる。

なので、ReactPHP / AMPHP / Swoole による MySQL 非同期処理は、普通の CRUD アプリですべてに優先する万能策ではないが、常駐型 PHP の文脈ではかなり重要な基盤技術になる。価値は SQL 単体の高速化ではなく、MySQL・Redis・外部 API など複数 I/O が絡むアプリ全体の待ち時間設計にある。限界を理解したうえで見ると、PHP におけるイベントループ、コルーチン、接続プール、I/O 待ちの考え方を学ぶ入口として、とても筋の良い題材だと感じる。
