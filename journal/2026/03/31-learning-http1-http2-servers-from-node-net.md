# Node.jsでHTTP/1/2の簡易サーバーをどう学ぶか — `node:net` から始めて、HTTPの層を自分の手で切り分ける

date: 2026-03-31
tags: #Nodejs #HTTP1 #HTTP2 #TCP #TLS #WebServer #Backend #Programming #SoftwareEngineering #Obsidian

私は、Node.js で HTTP を学ぶときこそ、最初から `node:http` や `node:http2` に頼りすぎないほうが理解が深まると思っています。完成された標準モジュールは実務では便利ですが、学習では「何を肩代わりしてくれているのか」が見えにくいからです。そこで私は、まず `node:net` を起点に簡易サーバーを作るやり方を勧めたいです。

最初の入口は TCP です。`net.createServer()` で接続を受け、`socket.on('data')` で届いたバイト列を読み、`socket.write()` や `socket.end()` で返す。この段階で大事なのは、TCP にはメッセージ境界がないことです。`data` イベント1回が、HTTPリクエスト1回に対応するとは限りません。ここを理解すると、「HTTPの前に、まずストリームとしての TCP を理解する必要がある」と腹落ちします。

HTTP/1 の学習は、その TCP 上を流れるテキストをどう切り出すかです。私はまず、`Buffer` にデータを溜めて `\r\n\r\n` を探し、リクエスト行とヘッダーを分離するところから始めるのがよいと思います。さらに `Content-Length` を見てボディ長を判断し、レスポンス行とヘッダーを書いて返す。学習初期なら、keep-alive や chunked transfer encoding、厳密な検証は省略して構いません。重要なのは、HTTP/1 が「テキストを切るプロトコル」だと体で理解することです。

HTTPS を学ぶなら、次は `node:tls` です。TLS は HTTP の代わりではなく、その下に載る暗号化と認証の層です。`tls.createServer()` を使えば、暗号化されたソケットの上で、HTTP/1 に近い処理をそのまま試せます。学習用なら自己署名証明書で十分です。ここで TCP 層、TLS 層、HTTP 層を分けて考えられるようになります。

HTTP/2 になると、学び方はかなり変わります。HTTP/1 の延長で文字列を読むのではなく、9バイトのフレームヘッダーを読むところから始めるべきです。length、type、flags、stream identifier を取り出し、SETTINGS、PING、HEADERS、DATA を観察する。それだけでも十分に学習になります。最初から HPACK の動的テーブルや Huffman 符号化まで抱え込む必要はありません。HTTP/2 は「テキストを切る」のではなく、「バイナリフレームを切る」学習だと考えると整理しやすいです。

私なら学習順序はこうします。TCP エコーサーバー、行区切りや長さ付きの簡易独自プロトコル、最小 HTTP/1 サーバー、TLS 化、その後に HTTP/2 のフレーム切り出しです。この順番なら、`node:net`、`node:tls`、`Buffer`、そして Stream の感覚が自然につながります。Node.js で簡易サーバーを自作する意味は、本番用の再実装ではなく、「どのモジュールがどの層を担当しているか」を自分の手で観察できることにあります。gRPC や Connect、さらに QUIC を学ぶ前段としても、この順番はかなり有効だと思います。

**この記事の要点まとめ**  
`node:http` や `node:http2` は便利ですが、学習の入口としては `node:net` が最重要です。HTTP/1 はテキストを切る学習、HTTP/2 はバイナリフレームを切る学習です。簡易サーバーは完成品を作るためではなく、層構造と責務分担を理解するために作るのがよい、と私は考えています。
