nghttp2 拡張モジュールの準公式化について検討しました。

nghttp2 拡張モジュールの準公式化を考える出発点は、単なる「HTTP/2 を PHP で使いたい」ではありません。 むしろ、この種の拡張を個人で維持し続ける負担が大きすぎる こと、そして コミュニティ全体への影響が大きすぎる ことです。

通信基盤に関わる拡張は、API の安定性、依存ライブラリ追従、保守、利用者の期待、周辺ツールとの整合まで背負います。 それを個人開発のまま支えるのは限界がある。 一方で、完全に放置すると、PHP コミュニティとして HTTP/2 の基礎部品をどう扱うかが曖昧なままになります。

論点は、単なる採用可否ではなく、どの API を、どのスコープで、どの責任分担のもとに公共部品化するか です。 高水準 API まで抱えるのか、protocol primitive に徹するのか。 さらにこれは、コミュニティ開発の体制づくりの話でもあります。

そして将来的には、ビルトインサーバーの TLS 対応や、条件付きの HTTP/2 対応の議論にも接続します。 nghttp2 拡張の準公式化は、PHP の通信基盤を個人技から公共財へどう移すかを問うテーマだと思っています。

## PHP の責任分担の先行事例

PHP では昔から、重い領域を全部 userland に任せてきたわけではありません。

たとえば mod\_php は Apache の中に PHP を組み込む形で、Web サーバー連携の複雑さをかなり吸収してきました。php-fpm も FastCGI Process Manager として、プロセス管理やプール管理を PHP 側で担う仕組みです。

一方で ext/curl は libcurl を通じて多様なプロトコル接続を提供し、ext/openssl は TLS や証明書処理の基盤を提供します。つまり PHP は、低レイヤーを丸ごと標準化する場合もあれば、外部ライブラリを橋渡しする拡張として責任を分担する場合もある、ということです。

nghttp2 拡張の準公式化も、この延長線上で考えられます。 HTTP/2 を全部 core に入れるのではなく、どこまでを公共部品として支え、どこからを利用者や周辺ライブラリに委ねるのか。 その責任分担を整理する議論として見るのが大事だと思います。

## HTTP/2 の標準化と責任分担は、言語ごとにかなり違う

Node.js は http2 を標準モジュールとして持ち、Core API では HTTP/2 機能に即した低水準 API を、Compatibility API では既存の HTTP/1 系 API に近い形を用意しています。つまり「標準で載せるが、API の層を分けて責任分担する」モデルです。

Python は標準ライブラリに http.client、http.server、ssl はありますが、http.server は本番向けではなく、TLS は ssl が担当します。HTTP/2 までを標準で強く抱え込まず、その先は利用者や周辺ライブラリ側の責任に寄せるモデルと言えます。

Rust はさらに分離が明確で、h2 crate は HTTP/2 仕様の実装に徹し、TCP 接続管理や TLS は非目標です。そこを hyper など上位ライブラリが受け持つ構造になっています。つまり「protocol primitive を細かく分けて組み合わせる」モデルです。

PHP で nghttp2 拡張の準公式化を考えるなら、この3類型のどこに寄せるのかが重要だと思います。

## コミュニティ方針と API 論の議論が一体化している PHP RFC 作成の難しさ

nghttp2 拡張の RFC が難しいのは、単なる機能提案ではなく、コミュニティの方針 と API の設計 が同時に問われるからです。

PHP の RFC は、変更コストが大きいものほど慎重な合意形成が求められ、採択には強い賛成が必要です。 そのため「HTTP/2 の基盤は重要だから、何らかの形で支えるべきだ」という総論に賛成でも、いざ各論になると、

- どこまでを core / 準公式 / 外部拡張に置くのか
    
- API は高水準にするのか、protocol primitive に徹するのか
    
- 保守責任や依存ライブラリ追従を誰が負うのか
    

といった論点で反対が出やすいです。

つまり難しさは、HTTP/2 支援の必要性そのものより、どういう責任分担モデルなら PHP コミュニティとして持続可能か を示さないといけない点にあります。 方針だけ語ると API が曖昧になり、API だけ語るとコミュニティ上の正当性が弱くなる。 この二つを切り離さず、しかも「総論賛成、各論反対」に崩れない形で束ねる必要がある。そこが最大の難所だと思います。
