# Python の asyncioはどう進化したのか――ループ中心から構造化並行性へ

Python の `asyncio` を追っていて面白いのは、API が増えたこと以上に、**並行処理をどう頭の中で捉えるか**が変わってきたことだ。整理すると、`asyncio` の歴史は **ループ中心 → コルーチン中心 → スコープ中心** への移行として見ると理解しやすい。

初期の `asyncio` はイベントループが主語だった。`Future` や `Task` は「ループに登録する仕事」で、感覚としては受付に処理を置いて、終わったら呼ばれる世界に近い。これはコールバック時代の延長としては自然だが、自由度が高いぶん、タスクの寿命管理が散らばりやすい。気づかないうちに生き残るタスク、例外の取りこぼし、キャンセル処理の不整合が起きやすい、いわゆる非構造化並行の弱さが見えてくる。 

次の段階で `async` / `await` が入ると、主語はループからコルーチンへ移る。`await` は「ここで一度中断して待つ」という読みやすい記法で、書き味はかなり逐次処理に近づいた。ここで `asyncio` は一気に身近になったと思う。ただし、見た目が逐次風になっても、`create_task()` のように外へ投げた処理は視界から消えやすい。つまり、読みやすさは上がっても、責任の置き場までは自動では整理されない。

そこで重要になるのが、Python 3.11 以降の `TaskGroup` と `ExceptionGroup` だ。ここで `asyncio` は「何を待つか」だけでなく、**誰が誰に責任を持つか**をスコープで表現できるようになった。タスクをグループとして扱い、例外は上に伝わり、キャンセルは下へ伝わる。寿命がコードの構造と結びつくので、野良タスクやゾンビタスクが設計上起きにくくなる。これは単なる便利機能ではなく、並行処理の責任設計を標準語にした転換点だと感じる。

この流れを理解すると、Trio がなぜ強い思想を持って見えるのかもわかる。Trio は後方互換を背負わず、安全性重視で最初から構造化並行を前提に設計できた。一方の `asyncio` は、既存利用者や歴史的経緯を抱えたまま、現実と折り合いをつけて進化してきた。だから `asyncio` は「街を歩きながら設計図を書き直した」系で、Trio は「最初から設計図を引いた」系だと言える。

この話は OS プロセスの親子関係とも少し似ているが、同じではない。OS の親子は主にリソース管理の木で、親が死んでも子が残ることがある。構造化並行は責任管理の木で、子の失敗を親が引き受け、親の終了に子が巻き込まれる。つまり `asyncio` の進化は、「イベントループの道具箱」から「並行処理の壊れ方まで設計する枠組み」への進化だった、と捉えると腹落ちしやすい。

今 `asyncio` を学ぶなら、関数名を丸暗記するよりも、「今どのメンタルモデルで書いているか」を意識したい。ループに仕事を置く感覚なのか、`await` で逐次風に読む感覚なのか、それとも `TaskGroup` でチームとして寿命を管理する感覚なのか。その違いが見えると、コードの読み方も、事故の防ぎ方もかなり変わる。
