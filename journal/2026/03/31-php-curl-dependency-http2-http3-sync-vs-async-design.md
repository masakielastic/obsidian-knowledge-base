# PHP の HTTP/2/3 は curl だけでよいのか
――同期利用者の利便性と非同期利用者の足場をどう両立するか

PHP で HTTP 通信を扱う現場では、curl 依存はきわめて合理的である。TLS、ALPN、HTTP/2/3 の交渉を自前で抱えずに済み、多くの利用者にとっては「安全かつ便利に通信できること」が最優先だからだ。同期的な Web アプリケーション開発を中心に見るなら、curl は今後も重要な基盤であり続けるだろう。

しかし、PHP の将来を考えるうえでは、それだけでは整理しきれない問題がある。ここで重視したいのは、**同期的な利用者の利便性を損なわずに、非同期的な利用者や基盤実装者をどのように支援するか**という判断軸である。curl の価値を認めたうえで、実装経験や設計知識が PHP 側に共有されにくい点まで含めて考える必要がある。

### 1\. 同期的な利用者にとって curl 依存は合理的である

まず確認すべきなのは、curl 依存は弱さではなく、かなり強い実務判断だという点である。多くの PHP 利用者は、HTTP/2 のフレーム制御や QUIC の状態遷移を触りたいのではなく、外部 API に接続し、レスポンスを受け取り、業務処理を進めたい。その意味で、完成度の高い高水準 API に乗ることは合理的であり、同期的な request/response 開発の体験を守るうえでも有効である。

したがって、議論の出発点は「curl をやめるべきだ」ではない。むしろ、**同期利用者のための便利で安定した入口は維持すべきだ**という点を先に確認したほうがよい。

### 2\. 問題は、非同期利用者や基盤実装者が育つ足場が弱いことにある

一方で、curl ベースの利用体験は主に高水準 API に閉じている。利用者が触れるのは request/response モデルであり、HTTP/2/3 の本質である多重化、flow control、HPACK/QPACK、接続状態管理といった層ではない。そのため、PHP 側に蓄積されるのは「使い方」の知識であって、「なぜこの API 制約なのか」「どう設計すべきか」という実装知ではない。curl は利用経験を広げるが、実装経験までは広げにくいのである。

この構造は、同期的な利用者には大きな問題として見えにくい。しかし、非同期処理、独自ランタイム、gRPC/Connect、長寿命接続、プロトコル制御に関心を持つ利用者にとっては、共有された足場の不足として現れる。つまり課題は、curl が便利すぎることではなく、**curl が強いがゆえに、その外側の層が育ちにくいこと**にある。

### 3\. API を三層に分けると論点が見えやすい

この問題を整理するには、API を少なくとも三層に分けて考える必要がある。  
第一は、URL を渡してレスポンスを得る高水準 API。  
第二は、接続、stream、headers、data、trailers、多重化、flow control を扱うコネクションレベル API。  
第三は、frame parse/serialize、HPACK/QPACK、state machine のような低レベル primitive である。

curl は第一層に非常に強い。ここを充実させることは、同期的な利用者の利便性を守るうえで正しい。一方、非同期的な利用者を支援するには、第二層や第三層の整備が必要になる。ここを混同して「curl があるから十分」と言ってしまうと、同期利用者の満足と非同期利用者の不足を同じ言葉で扱ってしまうことになる。

### 4\. 純 PHP だけに委ねるのも現実的ではない

では非同期利用者のための足場を、純 PHP のユーザーランド実装に委ねればよいのか。ここでも慎重な整理が必要である。HTTP/2/3 は、バイト列処理、圧縮、状態遷移、フロー制御が絡む複合問題であり、HTTP/3 / QUIC では transport 層の複雑さまで加わる。教育用や実験用の最小実装には価値があるが、実用的な基盤をすべて純 PHP に背負わせるのは重い。まさに「ライブラリとして書くには重すぎるが、アプリとして書くには低レベルすぎる」領域である。

ここで重要なのは、非同期利用者を支援すると言っても、「全部を PHP スクリプトで書けるようにする」ことが目標ではない、という点である。必要なのは、無理のない責任分担である。

### 5\. 既存実装はあるが、共有可能な形では存在しにくい

Swoole、FrankenPHP、RoadRunner、curl には、それぞれ強力な実装や基盤がある。だが共通する構造的課題もある。**実装は存在していても、エコシステム全体で再利用しやすい形で存在しているとは限らない**ということだ。経験や知見が閉じた実装の中に留まると、PHP 全体の標準的な足場にはなりにくい。

この観点から見ると、同期利用者は既存の高水準基盤で十分に恩恵を受けている一方、非同期利用者やフレームワーク作者は、閉じた実装の外に出たとたん共通部品を見つけにくい。ここに支援の非対称性がある。

### 6\. 判断軸として必要なのは「同期は守る、非同期には足場を出す」

したがって、中心的な判断軸はかなり明確である。  
**同期的な利用者には、curl を中心とした高水準 API の利便性を維持する。**  
そのうえで、**非同期的な利用者には、完成品 API ではなく、作るための足場を提供する。**  
この二段構えが現実的である。

その候補として、nghttp2 / ngtcp2 / nghttp3 のようなデファクト実装を、Sans-I/O な protocol primitive として扱う発想が出てくる。高水準の「便利 API」を標準に抱え込むのではなく、フレームワークや基盤作者が使える低レベル部品に責務を限定する。これは同期利用者の体験を壊さず、非同期利用者を見捨てない設計として筋がよい。

### 7\. これは curl の話であると同時に、PHP の将来像の話でもある

この問題設定は、単なる HTTP/2/3 の技術論にとどまらない。PHP は今後、同期的な request/response を主戦場とするのか、embed PHP のように他ランタイムとの統合を広げるのか、あるいは True Async のように非同期 I/O をより広く扱える言語を目指すのか。そこには少なくとも複数の将来像がある。

ここで大事なのは、「同期か非同期か」という単純な対立にしないことである。同期中心の強みを維持する選択にも合理性はある。ただし、それだけでは HTTP/2/3 や QUIC のような変化に対する主体性を失いやすい。重要なのは、**どこまでを PHP 自身の責務とし、どこから先を外部基盤に委ねるのか**を設計判断として整理することだ。

### 8\. フロンティアがないとコミュニティは停滞しやすい

さらに、この論点はコミュニティの継続性にもつながっている。オープンソースのコミュニティは、既存機能を安定提供するだけでは長期的に維持しにくい。共有された未解決問題があり、そこに新しい実装者や学習者が参加できる状態、つまりフロンティアが必要になる。高水準 API の世界は利用を広げるが、実装の入口や挑戦の場を作りにくい。

Swoole、FrankenPHP、RoadRunner のような実装には高度な問題解決がある。しかし、それが reusable な形で広く共有されなければ、コミュニティ全体のフロンティアにはなりにくい。HTTP/2/3 や QUIC に PHP から触れにくい現状は、単なる機能不足ではなく、**最前線の問題に触れる入口が失われつつある状態**として見ることもできる。

高水準 API が充実しても、上級者や基盤実装者が挑戦できる領域がなければ、コミュニティは「使う人」だけが増えて「作る人」が減りやすい。フロンティアがない言語は、安定するのではなく停滞する。この観点でも、protocol primitive の整備は単なる機能追加ではなく、実装者を育てる入口の設計と結びついている。

### まとめ

調査結果を整理すると、結論は単純な賛否ではない。curl 依存は、同期的な利用者の利便性を守るうえで今後も合理的である。一方で、それだけでは非同期利用者や基盤実装者に必要な足場は育ちにくい。純 PHP 実装に全面的に委ねるのは重く、閉じた実装だけに頼っても共有は進まない。

したがって、中心的な判断軸はこうなる。  
**同期利用者には完成度の高い高水準 API を維持する。**  
**非同期利用者には再利用可能な protocol primitive を整備して支援する。**

PHP における curl 依存の課題とは、便利か不便かではなく、**同期中心の強みを守りながら、非同期の挑戦者にどんな公共部品を渡せるか**という設計問題なのである。  
そしてこれは、curl を使うべきかどうかだけの話でもない。PHP がこれからも「HTTP の上で便利に動く言語」でいるのか、それとも「HTTP や非同期 I/O の変化に対して自前の問題設定を持てる言語」であり続けるのか、という自己定義の問題でもある。
