# 文字列関数でも、すぐ失敗しないほうがよいことがある
— `str_iter` で考える、アプリと低レイヤーのエラーハンドリングの違い

date: 2026-03-31
tags: #PHP #PHPRFC #StringProcessing #ErrorHandling #LowLevelDesign #APIDesign #UTF8 #SoftwareArchitecture #NetworkProgramming #DesignPrinciples

以前取り組んだ PHP RFC の `str_iter` では、UTF-8 文字列をコードポイント単位で順にたどる関数を考えた。このとき重要だったのは、不正なバイト列を見つけたら即座にエラーにするべきか、という論点だった。

近年は静的型付けの言語や Result 型の普及もあって、異常を見つけた瞬間に失敗を返すのが正しい、という感覚が広まりやすい。もちろんそれが適切な場面は多い。アプリケーションレベルでは、早めに止めて利用者に明確な失敗を返すほうが安全なことも多いからだ。

ただし、その感覚をそのまま低レイヤーに持ち込めるとは限らない。

ここで大事になるのが、**観測とポリシーにもとづく対応は原則として分離する**という設計である。低レイヤーの部品は、まず何が入力されてきたのかを観測する責務を持つ。その入力を拒否するのか、置換するのか、記録だけして先に進むのかは、本来は上位のポリシーで決めるべきことだ。

`str_iter` で不正なバイト列をチェックしない方針になったのも、この考え方による。関数の役割は、まずバイト列を走査することにある。検証や拒否まで一体化すると、用途ごとの判断を低レイヤーに押し込めることになり、部品としての再利用性が落ちる。

文字列関数というと、`trim()` や `substr()` のようなアプリ寄りの API を連想しやすい。しかし `str_iter` のような低レイヤーの走査関数は、実際にはネットワーク関数に近い面がある。ネットワークでは、壊れたバイト列や途中までしか届いていないデータは珍しくない。そのたびに基盤部品が即座に全部を失敗させていたら、復旧、記録、部分的処理、再試行といった上位の判断が難しくなる。だから低レイヤーでは、まず観測を進め、失敗の扱いは後段に委ねる設計がよく現れる。

この発想は文字列処理でも同じである。入力が常に整っているとは限らない以上、「壊れているかもしれないが、それでもどこまで読めるか」を扱える部品には意味がある。

もちろん別の設計も理屈の上ではありうる。複数の文字列関数を一つのオブジェクトのメソッド群に見立てて、不正なバイト列を「汚染モード」のような状態として内部に保持しながら動かす方法である。しかしその方式は、状態管理、API 設計、相互運用、最適化の負担が大きい。コストが高すぎるため、大規模な採用事例はほとんどない。

要するに、即座にエラーを返すことは常に正義ではない。とくに低レイヤーでは、まず現実の入力を観測できることのほうが重要な場合がある。`str_iter` の設計判断は、文字列関数を単なる便利 API としてではなく、未検証入力を扱う基盤部品として見ると理解しやすい。世界をすぐにきれいにするのではなく、まず壊れた現実を観測する。それが低レイヤーの役割である。
