# QUIC のユーザーランド API はなぜこんなに設計がむずかしいのか

TCP → HTTP/2 → QUIC と進むにつれて、API の前提そのものが崩れていく。この流れを意識すると、QUIC の API 設計が単なるラッパー作りでは済まない理由が見えてくる。

TCP の時代は比較的わかりやすかった。OS が接続管理や順序制御、再送などをかなり引き受けてくれるので、アプリ側からは「接続する」「読む」「書く」「閉じる」という byte stream の世界に見えやすい。だから言語が違っても、だいたい似た API に落ち着きやすかった。PHP の stream API みたいな発想が成立しやすいのもこの土台があるからだ。

でも HTTP/2 になると事情が変わる。1本の接続の中に複数 stream があり、やり取りは frame ベースで進み、flow control や state machine も無視できない。ここまで来ると、もう「ソケットを読む・書く」だけでは全体像を表現しにくい。Connection を前に出すのか、Stream を主役にするのか、Event 駆動で見せるのかだけでも設計が割れやすい。

QUIC はさらに難しい。TCP の上の話ではなく、UDP の上にユーザー空間でトランスポートを実装するので、TCP では OS の奥に隠れていたものが前に出てくる。再送、順序制御、接続管理、暗号ハンドシェイク、タイマー処理まで API に影響する。つまり QUIC は「ソケットっぽい何か」ではなく、datagram を渡し、送信パケットを取り出し、タイマーを進め、接続状態を観測する仕組みまで含めて考えないと形になりにくい。

しかも QUIC ライブラリは流儀がかなり違う。sans-I/O 型なら柔軟だが利用者に責務が多い。ランタイム内蔵型は使いやすいが既存の event loop やフレームワークと衝突しやすい。HTTP/3 一体型は高水準で便利だが、QUIC そのものの共通 API というより完成品に近づく。だから「基本 API はこれ」と言い切りにくい。

さらに厄介なのは、API 設計が見た目の問題だけではないことだ。どこまで抽象化するか、利用者にどれだけ状態管理を見せるかによって、実装コストや性能特性も変わってくる。しかも比較条件には TLS、UDP、イベントループ、単一 stream か多重 stream か、ロス環境かどうかまで絡む。きれいな API を決めてから性能を見る、という順番では済まない。

その結果、既存フレームワークにも統合しづらい。多くの HTTP ライブラリは TCP 的前提で育ってきたので、QUIC/HTTP/3 を後付けしにくい。共通 API がないから統合しにくく、統合されないから利用者が増えず、利用者が増えないから共通理解も育たない。ここがいちばん大きな鶏卵問題だと感じる。

結局、QUIC の API 設計は単なるラッパー作りではない。API 抽象、性能評価、フレームワーク統合、そしてエコシステムの合意形成が全部つながっている。QUIC が TCP ソケットの延長として自然に見えない理由は、まさにそこにある。
