# HTTP サーバーは全部同じじゃないですか？にどう答えるか
— PHP と Node.js から考える、標準機能・政治コスト・自作できる基盤の意味

date: 2026-03-31
tags: #HTTP #WebServer #PHP #NodeJS #Backend #SystemDesign #Architecture #Async #EventLoop #TLS #OpenSSL #OSS #SoftwareDesign #EngineeringManagement #RFC

「HTTP サーバーなんて、結局は全部同じでは？」  
この疑問は自然です。どれもリクエストを受けてレスポンスを返すので、外から見ると似ています。

でも実際には、HTTP サーバーは全然同じではありません。  
違うのは、性能だけではなく、**どこまでを標準で面倒を見るのか**、**どこから先を利用者や専門家に委ねるのか**という責任分担です。

Node.js の `createServer()` は、アプリ側が自分でサーバーを組み立てていく文化と相性がよい API です。  
一方で PHP は、長いあいだ Apache、nginx、php-fpm のような外部の Web サーバーや実行基盤と組み合わせて使う文化を育ててきました。  
つまり、同じ「HTTP サーバー」でも、言語コミュニティごとに前提が違います。

ここで重要なのは、**標準機能として全部を専門家にお任せする方式には、強い安心感がある反面、政治コストが非常に高い**ということです。

たとえば PHP のビルトインサーバーを TLS 対応させる議論を考えるとわかりやすいです。  
TLS 自体は「今どき必要な機能」に見えます。けれど、標準機能に入れるとなると話は急に重くなります。

OpenSSL 依存をどう扱うのか。  
証明書や秘密鍵の設定をどこまで面倒見るのか。  
開発用だけでよいのか、本番利用を誘発しないか。  
既存の運用文化やドキュメントをどう変えるのか。  
脆弱性対応や保守責任は誰が負うのか。

こうした論点は、単なる実装課題ではありません。  
**コミュニティ全体で責任を背負う合意形成の問題**です。  
そのため、「便利だから入れよう」では済まず、政治コストがものすごく高くなります。

だからこそ、「標準に全部載せる」だけが正解ではありません。  
むしろ重要なのは、**利用者が必要に応じて自作サーバーや独自基盤を組み立てられる土台を整備すること**です。

これは「全部を各自で頑張れ」という意味ではありません。  
低レベルのソケット、TLS、イベントループ、HTTP パーサ、ストリーム API のような基盤を整え、必要な人がその上に自分の用途に合ったサーバーを作れるようにする、ということです。

この考え方の利点は大きいです。  
標準機能に全責任を集中させなくて済むので、コミュニティ全体の合意形成コストを下げられます。  
同時に、Node.js 的にサーバーを組み立てたい人にも道を開けます。  
標準機能は最小限に保ちつつ、拡張や実験の余地を残せるわけです。

要するに、HTTP サーバーは全部同じではありません。  
違いは「HTTP をしゃべれるかどうか」ではなく、**誰が何を責任持って引き受ける設計になっているか**にあります。

そして PHP で本当に大事なのは、何でも標準に押し込むことではなく、  
**政治コストの高い標準化と、利用者が自作できる基盤整備をどう分担するか**を考えることだと思います。
