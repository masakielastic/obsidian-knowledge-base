# HTTP クライアントの設計 — Stream / Socket / Fiber / Event Loop の責務をどう切り分けるか

HTTP クライアントを考え始めると、つい「Stream オブジェクトが読み書きも監視も状態管理も全部やる」と想像しがちです。ですが、その見方は危ういです。今回の整理で重要なのは、コードを書く前に責務の地図を作ることでした。とくに Stream / Socket / Fiber / Event Loop を別物として理解することが中心に置かれています。

まず Stream です。名前からは「データの流れそのもの」を想像しやすいのですが、設計上は「流れを受け渡す受動的な接続口・操作対象」と見たほうが安定します。自分で何かを判断して動く主体ではなく、外側から read / write される対象です。Socket も同様で、「通信そのもの」ではなく接続の端点やハンドルに近いものとして捉えるのがよい、という整理でした。つまり Stream や Socket は、ネットワークを理解して自律的に振る舞う存在ではありません。外側の仕組みがそれを使っているだけです。

ここで大事なのが、**読み書きと監視は別責務**だという点です。read / write は実際の I/O 操作ですが、readable / writable の判定は観測です。`stream_select` の役割は「読むこと」でも「書くこと」でもなく、「今どの操作が進められるかを観測すること」です。この区別が曖昧になると、Stream オブジェクトに監視、タイムアウト、再試行、状態遷移まで押し込みたくなり、God Object になりやすいです。I/O は単なる実行の問題ではなく、誰をいつ進めるかという調停の問題でもある、という見方が必要です。

Event Loop も誤解されやすい言葉です。単なる `while` ループのことではありません。本質は**監視・選択・ディスパッチ**にあります。何が読み込み可能か、何が書き込み可能かを観測し、その結果をもとに次に進める処理を選ぶ。この機能を見たほうが理解しやすいです。だから、場合によっては Event Loop という名前よりも、Monitor や Scheduler / Dispatcher と分けて考えたほうが責務が見えやすくなります。

Fiber もまた別種です。Fiber は I/O を監視する仕組みではありません。**中断・再開できる処理の流れ**です。「Fiber が待つ」という言い方は雑で、実際には監視側が I/O の準備完了を観測し、その結果を受けて Fiber を再開します。つまり待っているのは Fiber そのものというより、Scheduler 側が「今は進めない」と判断して保留している状態です。Fiber はデータでもハンドルでもなく、制御フローを表す実行単位として理解したほうがよいです。

この整理をオブジェクト指向で言い換えると、少なくとも次のように分けて考えると見通しがよくなります。  
Stream / Socket は **ハンドル**。  
Request / Response / Header は **値**。  
Monitor や Scheduler、Client は **主体**。  
Fiber はそのどれでもなく、**実行単位・制御フロー**です。  
この分類を曖昧にすると、「オブジェクトは全部自己判断する主体だ」という誤解が入り込み、責務の境界が崩れます。

非同期 HTTP クライアントの最小構成を言葉で置くなら、こんな地図になります。

- Stream / Socket: 読み書きの対象
    
- Monitor: readable / writable の観測
    
- Scheduler / Dispatcher: どの処理を進めるか決める調停者
    
- Fiber / Task: 中断・再開される処理の流れ
    
- Protocol: HTTP メッセージや状態遷移の理解
    
- Client Facade: 利用者向け API
    

重要なのは、「誰が何を知っていて、何を知らないべきか」です。Stream は HTTP を知らなくてよいし、Fiber はソケット監視を知らなくてよい。Protocol は I/O 多重化そのものを知らなくてもよい。この分離が、HTTP/1.1 の段階でも保守性を上げますし、HTTP/2 のような多重化になるとさらに重要になります。HTTP/1.1 では雑に一体化しても動くことがありますが、HTTP/2 では接続共有やフレーム単位の進行が入るため、God Object 的な Stream 設計はかなり苦しくなります。

結論として、HTTP クライアント設計で先に疑うべきなのはコードではなく言葉です。  
Stream は主体ではなく受動的な接続口。  
Socket も通信そのものではなくハンドル。  
監視と読み書きは分離すべき。  
Event Loop の本質はループではなく監視・選択・ディスパッチ。  
Fiber は I/O 監視機構ではなく、中断・再開できる処理の流れ。  
このメンタルモデルを先に作ると、便利そうな単一オブジェクトに全部背負わせる誘惑から距離を取れます。非同期 HTTP クライアント設計の核心は、賢い Stream を作ることではなく、**意思をどこに置くかを明確にすること**です。
