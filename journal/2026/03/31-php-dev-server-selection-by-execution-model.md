# PHP開発環境のサーバーは「本番の実行モデル」に合わせて選ぶ — ReactPHP・Swoole・php-fpm の使い分けメモ

結論から書くと、PHP 開発環境のサーバー選びは「手軽さ」より「本番の実行モデルにどこまで合わせるか」で考えるのがよい。
本番が php-fpm 型なら、開発も同期モデル中心で大きな問題は出にくい。
本番が ReactPHP や Swoole のような長寿命・非同期モデルなら、開発でも近い実行モデルを使ったほうが安全だ。
理由は、設定をそろえるためではなく、バグの出方をそろえるためである。

まず php-fpm + nginx のような従来型の構成は、リクエストごとに処理が区切られる同期モデルとして理解しやすい。1 回のリクエストが終わるたびに状態が切れやすく、前の処理の影響が次に残りにくい。この性質のおかげで、PHP の多くのアプリは「毎回まっさらな気持ちで始まる」前提で設計しやすかった。開発環境としても扱いやすく、一般的な Web アプリでは今でも自然な選択肢になる。

次に ReactPHP は、イベントループの上で動く長寿命プロセス型の選択肢として見るとわかりやすい。開発用の軽いサーバーとして便利だが、本質は「サーバーが生き続ける」ことにある。すると、静的変数、キャッシュ、接続、シングルトン、後始末漏れの影響が次の処理に残りうる。さらに、重い同期処理を 1 つ混ぜるだけで、ほかの処理まで待たせることがある。つまり ReactPHP を使うと、同期モデルでは見えにくい問題が早く見えるようになる。これは本番も非同期モデルなら大きな利点だが、本番が php-fpm なら開発補助以上の意味を持たせすぎないほうがよい。

Swoole も方向性は近い。長寿命で、状態共有やライフサイクル管理の影響を受けやすく、アプリ側にも同期モデルとは違う注意が必要になる。本番で Swoole を使うなら、開発時点からその癖を踏める環境にしておいたほうが、状態汚染やブロッキングのような問題を後回しにせずに済む。見た目の構成一致より、壊れ方の一致を優先するべきだ、という考え方はここで効いてくる。

一方で、「開発だけ ReactPHP、本番は php-fpm」という構成が常に悪いわけではない。ローカルで素早く立ち上げる、ちょっとした確認をする、といった軽い開発ループでは十分役に立つ。ただし、イベントループ前提の書き方や、常駐メモリ前提の状態管理に依存し始めると危ない。判断基準はシンプルで、「そのサーバーを外しても同じアプリとして成立するか」で見るのがよい。外しても成立するなら開発補助、外すと壊れるなら実行モデル依存である。

実務では、開発環境を二層で考えると整理しやすい。ひとつは起動が速く、確認が楽な軽い開発ループ。もうひとつは、本番に近い実行モデルで状態共有や後始末漏れまで確認する統合確認である。サーバー選択は道具選びではなく、どの段階でどの種類のバグを踏みたいかを決める設計判断だと考えると、判断しやすくなる。
