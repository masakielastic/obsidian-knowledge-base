# Planner / Generator / Evaluator は、AIによる対話形式の記事設計にも使える

対話形式の記事を書こうとすると、意外と難しい。役割を分けたはずなのに雑談に流れたり、説明役が全部しゃべって単調になったり、「結局この会話は何を整理していたのか」がぼやけやすい。そこで役立つのが Planner / Generator / Evaluator という見方だ。これは単なるマルチエージェント流行語ではなく、「何を決める役か」「何を作る役か」「何を疑う役か」を分けるための設計モデルとして読むと強い。今回のテーマも、その考え方をAIによる対話形式の文章作成へ応用することにある。

まず雑に言えば、Planner は「今回どこまで話すか」を決める役、Generator は「その枠の中で説明を作る」役、Evaluator は「その説明で本当に通じるのか」を疑う役だ。ここで大事なのは、Planner は偉い人、Generator は作業者、Evaluator は否定屋、という理解で止まらないことだ。Planner は最初に全部詳細設計する人ではなく、論点と非目標を切り分ける人だし、Generator は無限に話を広げる人ではなく、枠内で具体化する人だし、Evaluator も単なるダメ出し係ではなく、読者がつまずく地点を先に見つける役である。

この3つを分ける理由は単純だ。1人で全部やろうとすると、論点整理と本文生成と自己評価が頭の中で混線しやすい。AIに記事を書かせるときも同じで、いきなり「わかりやすく書いて」と投げると、論点設定が甘いまま、それっぽい説明だけが増殖しやすい。対話形式では特にそれが起きやすい。疑問役が雑に問い、説明役がなんとなく答え、最後にふんわり納得して終わる。これでは読み物にはなっても、技術記事としての骨格が弱い。

AIによる対話記事に応用するときのコツは、キャラ属性ではなく、各発話の機能で考えることだ。たとえば一つの節では、最初に Planner が「今回は何を扱い、何を扱わないか」を示す。次に Evaluator が「つまりこういうことか」と早とちりや雑な理解を出す。そこに Generator が比喩や対比や具体例を使って説明を進める。最後に Planner が境界条件を確認して、その節の要点を締める。この流れがあるだけで、対話はかなり安定する。今回の設計方針でも、整理役・説明役・懐疑役を主担当として置きつつ、必要に応じて補助的な重なりを許すのが自然だとされている。

ここで重要なのは、1人1役に機械的に固定しないことだ。懐疑役が読者の誤解を代弁するのは有効だが、単なるボケ役にすると品質が落ちる。説明役が全部を抱え込むと独演会になる。整理役が話を止めるだけでも前に進まない。むしろ「主担当はあるが、節ごとに少し重なる」くらいが強い。技術記事は、問いを立てる、誤解を表に出す、説明する、境界線を引く、という往復でできているからだ。対話形式はキャラ遊びではなく、その往復を見える化する器として使うと安定する。

このモデルは記事だけの話でもない。RFC なら Planner は論点整理と非目標の明示、Generator は仕様本文や例示、Evaluator はFAQや反論想定になる。AIへの指示設計なら Planner はタスク定義、Generator は下書き生成、Evaluator はレビュー観点になる。コードレビューでも同じで、設計意図の確認、変更案の提示、破綻しそうな点の検査を分けるだけで議論がかなり整う。つまり Planner / Generator / Evaluator は、キャラ設定ではなく、思考の交通整理そのものなのだ。

対話形式の記事が雑談化しやすいのは、発話の機能が未設計だからだ。AIで対話形式を使うなら、「誰がしゃべるか」より先に、「この発話は論点整理なのか、説明なのか、疑義提示なのか」を決めたほうがいい。そうすると、対話は遊びではなく、読者の理解を前に進める構造になる。Planner / Generator / Evaluator は、そのための実用的な設計道具だと思う。

**この記事の要点**

- Planner / Generator / Evaluator は、キャラ属性ではなく思考機能の分担モデル
    
- Planner は論点と非目標を決め、Generator は枠内で具体化し、Evaluator は誤解や弱点を可視化する
    
- 対話記事では「登場人物の個性」より「各発話の役割」で設計したほうが強い
    
- AIによる対話記事でも、問い・誤解・説明・境界条件の流れを作ると雑談化しにくい
    
- 1人1役に固定しすぎず、主担当を持ちつつ少し重ねるほうが自然
    
- この考え方は技術記事、RFC、AIプロンプト、レビュー設計にも応用できる
