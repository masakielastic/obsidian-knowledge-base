# イベントループは何を回しているのか――イベントキュー、readiness、completion モデル

イベントループを一言でいうと、「今やるべき仕事があるかを見て、あれば順に処理し、なければ待つ仕組み」だと整理できる。そこで回っている材料がイベントキューで、タイマー満了、ソケット読込可能、書込可能、処理完了通知などがキューに入り、ループはそれを取り出してコールバックや継続処理を実行する。

readiness モデルは epoll の世界観で、「準備ができたら呼んで」だ。epoll は監視対象を見張り、「今なら読める」「今なら書ける」を知らせる。ここで重要なのは、epoll 自身が I/O を代わりに実行するわけではないことだ。通知を受けたあと、実際に read/write を進めるのはアプリ側である。つまり readiness は、「処理可能になった」という合図のモデルと考えるとわかりやすい。

completion モデルは io_uring の世界観で、「これをやっといて、終わったら教えて」だ。こちらは「今ならできる」を知らせるだけではなく、やってほしい I/O を先に渡し、終わった結果をあとで受け取る。readiness が「実行可能通知」寄りなのに対して、completion は「仕事の発注と結果受取」寄りのモデルである。

雑に言えば、epoll は「レジが空いたら呼んで」、io_uring は「この伝票を渡すので、終わったら番号で呼んで」に近い。前者は自分でその場の処理を進める前提で、後者はカーネルに仕事を投げて完了を受け取る前提だ。この差が、API 設計やアプリ側の責務の違いにつながる。

libuv が面白いのは、この二つの橋渡しをしている点にある。ネットワーク I/O では readiness 系の監視を使いながら、ファイル I/O では長くスレッドプールも活用してきた。利用者から見ると同じ event loop API でも、裏側では「監視して起こす」処理と「別の場所でやらせて完了を返す」処理を吸収している。libuv は、モデルの違いを隠しながら統一的な非同期実行の見た目を提供する実装だと捉えると理解しやすい。

では、なぜ io_uring は最近になって実装されたのか。背景には、従来の Linux AIO が使いにくく、適用範囲も狭かったことがある。ネットワークでは epoll が強力だった一方で、ファイル I/O 側の非同期は弱かった。そこで io_uring は、共有リングを使ってシステムコール往復やデータ受け渡しのコストを減らし、まとめて投入・完了回収しやすい形を作った。さらに Spectre/Meltdown 以後は、ユーザー空間とカーネル空間の往復コストがより意識されるようになり、この方向の設計の価値が高まった。

要するに、イベントキューとイベントループは「仕事の交通整理」の話であり、readiness と completion は「何を通知するか」の違いである。epoll は「準備できたよ」、io_uring は「頼まれた仕事が終わったよ」。libuv はその差を現実的に吸収する層として見ると、かなり整理しやすい。
