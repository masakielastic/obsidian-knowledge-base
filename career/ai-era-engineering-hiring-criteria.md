# 概要

AIコーディングエージェントの普及によって、エンジニア採用で重視すべき点は変わりつつある。  
従来のように「手を速く動かしてコードを書けるか」だけを見るのでは足りず、AIの出力を評価し、設計の妥当性を判断し、何を作るべきかを定義できるかが重要になっている。

特に採用側は、候補者を単なる実装者としてではなく、AIを含んだ開発システム全体の健全性を支えられる人材かどうかで見極める必要がある。

# なぜ採用基準が変わるのか

AIによって速くなったのは、既存パターンに沿った実装、CRUD、テスト追加、ドキュメント生成のような「やり方が比較的明確な仕事」である。  
一方で、複雑な仕様変更、複数サービスにまたがる修正、アーキテクチャ設計、ビジネス制約を踏まえた意思決定は、依然として人間に強く依存する。

この結果、採用で問うべき能力は次のように移る。

- 実装速度
- コード量
- パターン暗記

から、

- AI生成物を見極めるレビュー力
- 設計判断の妥当性を説明する力
- 何を作るべきかを定義する力
- チームのルールや責任分界を理解する力

へと重心が移る。

# 採用側が見るべき判断基準

## 1. AI生成コードを批判的に読めるか

これからの採用では、候補者が「自分で全部書けるか」だけでなく、AIが出したコードに対して何を指摘できるかが重要になる。

見るべき点:

- コードの見た目の綺麗さにだまされないか
- バグや設計上の違和感を説明できるか
- 境界条件や例外系を見落とさないか
- 変更の影響範囲を想像できるか

面接では、ライブコーディングだけでなく、AIが生成したコードをレビューさせる形式が実務に近い評価になりやすい。

## 2. 設計の妥当性を判断できるか

AIはもっともらしい実装案を出せるが、その案が目の前の組織やプロダクトに本当に合っているかは別問題である。  
そのため採用側は、候補者が設計を一般論で語るだけでなく、制約条件の中で判断できるかを見る必要がある。

見るべき点:

- どの責務をどこに置くかを説明できるか
- データ整合性や依存関係をどう扱うか考えられるか
- 将来の変更や運用コストまで見越せるか
- スケールや保守性の観点を持てるか

シニア採用では特に、この能力が中核になる。

## 3. 「何を作るべきか」を定義できるか

AIは作る速度を上げるが、作るべきものを決めてくれるわけではない。  
したがって採用では、候補者が仕様や要求をそのまま受け取るだけでなく、目的や優先順位を整理できるかが重要になる。

見るべき点:

- 要求をそのまま実装課題として受け取らず、目的に言い換えられるか
- ビジネスロジックの解釈を確認できるか
- 作るべきものと後回しにするものを分けられるか
- 合意形成の前提を言語化できるか

実装者というより、論点整理と意思決定支援ができる人材かどうかを見る必要がある。

## 4. セキュリティと品質担保の感覚があるか

AI生成コードは整って見えやすいため、危険な実装が紛れ込んでも見逃されやすい。  
そのため、採用では「安全に作る感覚」を持っているかを以前より強く確認したい。

見るべき点:

- 入力値検証や権限境界への感度があるか
- セキュリティ上の危険を具体的に指摘できるか
- レビュー、Lint、テストなど仕組みで守る発想があるか
- 品質を個人技でなく運用で担保しようとするか

「安全なコードを書けます」と言えるだけでは弱い。  
どのように事故を防ぐかを具体的に説明できるかが重要である。

## 5. チームで運用ルールを回せるか

AI導入の成否は、ツール選定そのものよりも、何をAIに任せ、何を人間が握るかという運用ルールの設計に左右される。  
したがって採用でも、個人として優秀かだけでなく、ルール運用に参加できるかを見るべきである。

見るべき点:

- チームの規約や既存パターンを尊重できるか
- 暗黙知を言語化することに抵抗がないか
- AIに渡す前提知識やルールを整理できるか
- 生成物の責任の所在を曖昧にしないか

AI時代では、優秀な個人よりも、組織の学習速度を高める人材の価値が上がる。

# ジュニア採用で見るべきこと

ジュニアはAIによって見た目の整ったコードを書きやすくなる。  
しかしそれは設計判断の成熟を意味しない。

そのためジュニア採用では、現時点の完成度だけでなく、次の点を重視したい。

- フィードバックを吸収できるか
- 既存ルールに沿って学べるか
- わからないことを切り分けて相談できるか
- AIの出力をそのまま出さず、自分で確かめようとするか

AI時代のジュニア採用は、即戦力性よりも学習のしかたを見る比重が増す。

# シニア採用で見るべきこと

シニア人材には、AIでは代替しにくい判断の責任を持てることが求められる。

重視したい点:

- アーキテクチャの意思決定
- 技術選定と優先順位づけ
- 技術負債を返済する判断
- チーム間調整や仕様の擦り合わせ
- AI利用ルールとレビュー基準の整備

シニア採用では、「自分で速く書ける人」よりも、「組織として正しく作れる状態を設計できる人」を高く評価すべきである。

# 面接設計への示唆

AI時代の採用面接では、従来型のアルゴリズム問題や単純なライブコーディングだけでは評価が偏る可能性がある。  
実務に近づけるなら、次のような設計が考えられる。

- AI生成コードをレビューしてもらう
- 仕様の曖昧さをどう解釈するかを議論する
- 設計上の懸念点を説明してもらう
- セキュリティ上の危険を洗い出してもらう
- ルールが未整備なチームで何を整えるかを聞く

これにより、実装力だけでなく、判断力、レビュー力、合意形成力を確認しやすくなる。

# まとめ

AI時代のエンジニア採用では、コードを書く速さの価値は相対的に下がる。  
その代わりに重要になるのは、AIが出したものを見極める力、設計や優先順位を判断する力、組織の制約を踏まえて何を作るべきかを定義する力である。

採用側が見るべきなのは、単独で手を動かせる人かどうかではない。  
AIを含む開発体制の中で、品質、責任、設計、合意形成を支えられる人かどうかである。

# 自分用メモ

- AI導入で消えたのは「単純作業の負荷」であって、判断の仕事ではない
- 採用は実装試験からレビュー試験へ寄せていく余地がある
- ジュニアには完成度より学習能力を見る
- シニアには設計責任と運用設計能力を見る
- 個人技よりも組織能力を強く支える人材を評価する
