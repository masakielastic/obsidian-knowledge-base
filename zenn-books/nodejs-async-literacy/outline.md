# Node.js 教養編 アウトライン案

## 本書の役割

本書は、Node.js を入口として、非同期処理を日常の言葉で説明できるようにするための教養編である。  
目的は、Node.js の API を覚えることではなく、Promise や `async/await` の下にある event loop、libuv、OS の待機機構までを見渡せるようにすることにある。

## 全体の流れ

本書は次の流れで進む。

1. Node.js の日常語から出発する
2. 非同期処理を「待ち方の設計」として捉え直す
3. event loop を待機と再開の中心として整理する
4. `select` を古典として理解する
5. `kqueue` と `epoll` を Node.js 前史として位置づける
6. libuv が OS 差分をどう吸収しているかを見る
7. Event Loop と Worker Pool の違いを整理する
8. Promise / `async/await` を地図に戻す
9. 補講として `io_uring` を扱い、別方向の進化を示す
10. 最後に、Node.js を非同期処理の共通語として位置づけ直す

## 第5章 kqueue と epoll の時代

### 章の目的
`select` のあとに、OS ごとにより効率的なイベント通知方式が発展したことを理解する。

### この章で答える問い
- なぜ `select` だけでは足りなくなったのか
- `kqueue` と `epoll` は何を改善したのか
- Node.js にとってこれらはどういう意味を持つのか

### 主な内容
- `select` の限界を Node.js ユーザー向けの粒度で説明
- Linux では `epoll`
- BSD/macOS 系では `kqueue`
- OS ごとに高効率な待機方式が育ったこと
- Node.js はこれらを直接触らないが、その恩恵の上にあること
- 「Node.js 前史」としての位置づけ

### この章の着地点
読者が、Node.js の下にある世界は Linux 固有の `epoll` だけではなく、OS ごとに異なる待機機構の系譜を持っていると理解できること。

- イベントループの基盤としての libuv
- I/O、タイマー、周辺機能を支える runtime 側の責務
- 「Node.js は JavaScript だけで動いているわけではない」という再整理

### この章の着地点
読者が、libuv を「Node.js を支える翻訳層・基盤層」として説明できること。

### この章の着地点

## 第8章 Promise と async/await を地図に戻す

### 章の目的

### 主な内容

- Promise と event loop の関係
- Promise と OS の readiness の距離感
- 日常語としての Promise と、土台にある待機モデルの違い

### この章の着地点
読者が、Promise や `await` を過大評価も過小評価もせず、runtime と I/O 待機の上に乗る表現として説明できること。

## 第9章 補講: io_uring は何が違うのか

### 章の目的
`epoll` と同じ系統の話としてではなく、別方向の I/O モデルとして `io_uring` を簡潔に紹介する。

### この章で答える問い
- `io_uring` は `epoll` の延長なのか
- readiness と completion は何が違うのか
- Node.js ユーザーは `io_uring` をどの程度知ればよいのか

### 主な内容
- `epoll` の世界では「今なら進められる」が中心
- `io_uring` の世界では「投げておき、終わったら受け取る」という色合いが強い
- readiness と completion の対比
- Node.js にとっては主役というより比較対象であること
- 深掘りしすぎず、「別の方向の進化」として示す

### この章の着地点
読者が、非同期 I/O の世界には readiness 中心の発想だけでなく、completion 寄りの発想もあると知ること。

## 用語ガイドの下書き

本書でぶれやすい語は、最初から方針を置いておく。

### 非同期処理
高速化の魔法ではなく、待機で全体を止めないための設計。

### event loop
順番管理だけではなく、待機・通知・再開の中心。

### Promise
未来の結果や継続を扱う抽象。I/O 多重化そのものではない。

### non-blocking
処理が即完了することではなく、待機で全体を止めないための前提。

### readiness
今なら少し進められる、という通知。

### completion
処理が完了した、という通知。

### libuv
OS ごとの違いと runtime の基盤責務を吸収する層。
