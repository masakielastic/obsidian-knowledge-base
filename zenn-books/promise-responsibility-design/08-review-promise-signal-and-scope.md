title: "第8章　復習 ― Promise・Signal・責任スコープをつなぎ直す"
---

**初学者**  
ここまで読んでくると、Promise 周辺で混ざりやすいものをかなり分けて考えられるようになってきました。  
でも正直に言うと、最後にもう一度、全体を一枚の地図として見直したいです。  
章ごとにはわかってきたんですが、実務では全部が同じ場所に出てくるので、また混ざりそうです。

**解説者**  
それがこの最終章の役割です。  
この本は Promise の API を順番に覚える本ではありませんでした。  
Promise、AbortSignal、timeout、retry、cleanup、責任スコープを、別々の棚に置き直す本でした。  
最後にもう一度、それぞれの位置をつなぎ直しましょう。

**実装者**  
実務では、一つの非同期処理の中に全部出てきます。  
`await` して、必要なら `signal` を渡して、timeout を入れて、失敗したら retry を考え、最後に cleanup する。  
だからこそ、近くに書かれるものを責務でも近いと思い込まないことが重要です。

**初学者**  
この章は、知識を増やすというより、棚を固定するための復習章なんですね。

**解説者**  
その通りです。  
持ち帰るのは、便利な書き方の一覧ではなく、責務の見取り図です。

## Promise は結果待機と失敗伝播の道具である

**初学者**  
まず、いちばん中心にある Promise から確認したいです。  
結局、Promise は何だったんでしょうか。

**解説者**  
Promise は、結果待機と失敗伝播の道具です。  
まだ手元にない結果を扱えるようにし、成功と失敗を同じ流れに載せます。  
`await` が自然に書けるのも、その基盤に Promise があるからです。

**実装者**  
ここで大事なのは、Promise を弱い道具だと思わないことです。  
Promise はかなり強い。  
ただし強いのは、自分の守備範囲に集中しているからです。  
将来値、成功、失敗、待機。  
この領域では非常に整理された道具です。

**初学者**  
つまり Promise は、非同期処理全体の王様ではなく、結果の流れを扱う中心なんですね。

**解説者**  
その言い方で大丈夫です。  
Promise は中心ではありますが、全部ではありません。

## AbortSignal は中止伝播の道具である

**初学者**  
では AbortSignal は、Promise の隣にある別の中心なんでしょうか。

**解説者**  
別の中心というより、別の棚にある重要な道具です。  
AbortSignal は結果を表しません。  
表しているのは、「もう不要だ」「続けなくてよい」という中止の意思です。  
そして、その意思を下位処理へ伝える経路になります。

**実装者**  
実務では `await` の近くに出てくるので Promise と混ざりやすいですが、役割は違います。  
Promise は何が返るかを待つ。  
AbortSignal はもう待つ必要があるのか、もう続けるべきなのかという判断を運ぶ。  
この違いは最後まで崩さないほうがいいです。

**初学者**  
Promise が結果の窓口、AbortSignal が中止の窓口。  
やはりこの二本立てで覚えるのがよさそうですね。

**解説者**  
はい。  
少なくともこの小冊子の範囲では、その棚分けが一番安定します。

## timeout と retry は制御ポリシーである

**初学者**  
timeout と retry は、何度も「別棚」と言われてきました。  
最終的にはどう覚えるのがいいでしょうか。

**解説者**  
どちらも制御ポリシーとして覚えるのがよいです。  
timeout は「どこまで待つか」「どこで諦めるか」という方針です。  
retry は「失敗にどう対応するか」「何回までやり直すか」という方針です。  
どちらも結果そのものや中止そのものではなく、外側から振る舞いに条件を与えるものです。

**実装者**  
ここを Promise の本体や AbortSignal の代用品だと思うと崩れます。  
timeout は「もう待たない」と決めるだけで、対象が本当に止まるとは限らない。  
retry は「もう一度やる」と決めるだけで、その仕事がまだ所有されているとは限らない。  
つまり、制御ポリシーは責任線と接続してはじめて安全に動きます。

**初学者**  
なるほど。  
timeout も retry も便利ですが、どちらも単独で完結した正しさを持つわけではないんですね。

**解説者**  
その通りです。  
方針は方針であって、責任線や後始末を代行してくれるわけではありません。

## cleanup は後始末である

**初学者**  
cleanup は最後まで別責務として出てきました。  
でも実務では、どうしても Promise のチェーンの最後や `finally` のような場所に書かれるので、同じ棚に見えてしまいます。

**解説者**  
書かれる場所と、責務の種類は別です。  
cleanup は後始末です。  
何を解除するか、何を閉じるか、何を元に戻すか。  
これは外部資源や所有関係に依存します。  
Promise がそのタイミングに関わることはあっても、cleanup の意味そのものを持っているわけではありません。

**実装者**  
イベントリスナーの解除、タイマーの停止、接続のクローズ、途中状態の破棄。  
こうしたものはすべて cleanup です。  
中止判断が届いたからといって自動では終わりません。  
だから「signal を渡したから大丈夫」「timeout があるから安心」という発想が危ないわけです。

**初学者**  
後始末は、最後にあることが多いだけで、Promise の一部ではないんですね。

**解説者**  
そうです。  
最後に書かれる責務であって、結果待機の責務ではありません。

## `all` / `allSettled` / `race` は待機集約の棚にある

**初学者**  
`Promise.all`、`Promise.allSettled`、`Promise.race` は最後にどう整理すればいいですか。  
使い分けは理解できても、棚の言葉で言い直したいです。

**解説者**  
この三つは、待機集約や観測の棚にあります。  
`all` は複数の結果待機をまとめる。  
`allSettled` は成功と失敗を保持したまま全部を観測する。  
`race` は最初に決着したものを観測する。  
つまり、どれも結果待機の見え方を調整している道具です。

**実装者**  
だから、この三つを責任スコープそのものと思わないほうがいいです。  
同じ配列に入れたから同じ所有者とは限らない。  
最初に決着したものを採用したから、他が自動で止まるとも限らない。  
待機集約と責任設計は、別々に見たほうが安全です。

**初学者**  
待機のまとめ方と、責任のまとめ方は違う。  
ここも最後まで大事な区別ですね。

**解説者**  
その通りです。  
Promise 系の便利 API は、待機の形をそろえてくれます。  
でも責任境界まで自動で作ってくれるわけではありません。

## 責任スコープは別の設計問題である

**初学者**  
ここで何度も出てきた責任スコープですが、最後にもう少しだけはっきり言葉にしたいです。  
責任スコープって、結局何なんでしょうか。

**解説者**  
一言で言えば、「誰がこの仕事の面倒を見るのか」という境界です。  
誰が起動し、誰が不要と判断し、どこまで中止を伝え、どこまで後始末を引き受けるのか。  
この線引きが責任スコープです。

**実装者**  
実務で設計が壊れるときは、たいていここが曖昧です。  
Promise は返している。  
signal も渡している。  
timeout も retry もある。  
でも「その仕事は誰のものか」が見えない。  
すると、所有者を失った retry、止まったつもりの通信、片づかないイベント購読が生まれます。

**初学者**  
つまり責任スコープは、Promise の API ではなく、設計の地図そのものなんですね。

**解説者**  
そうです。  
Promise や AbortSignal は、その地図の上で使う道具です。  
地図そのものの代わりにはなりません。

## 実務で最後に見るべき問い

**初学者**  
全体の棚は見えてきました。  
でも実際のレビューや実装で、最後に何を自分へ問いかければいいでしょうか。

**解説者**  
この本の内容を実務に持ち帰るなら、問いの形にしておくのがよいです。  
たとえばこうです。  
これは結果待機の問題か。  
これは失敗伝播の問題か。  
これは中止伝播の問題か。  
これは制御ポリシーか。  
これは cleanup か。  
そして、これは誰の責任範囲の中にあるのか。

**実装者**  
この問いを持っているだけで、かなり見え方が変わります。  
便利 API を追加する前に、「いま足りないのは本当に API か、それとも責任線の説明か」を考えられるようになるからです。  
Promise 周辺では、それがとても重要です。

**初学者**  
結局、この本は Promise の再学習というより、責務の再配置だったんですね。

**解説者**  
まさにその通りです。  
Promise を学び直すとは、便利な書き方を覚え直すことではありません。  
Promise を中心にしつつも、Promise だけに全部を背負わせない地図を作り直すことです。

### 整理

- Promise は結果待機と失敗伝播の道具である  
- AbortSignal は中止の意思と伝播経路の道具である  
- timeout は「どこで諦めるか」を決める制御ポリシーである  
- retry は「失敗にどう対応するか」を決める失敗処理ポリシーである  
- cleanup は後始末であり、結果待機や中止伝播とは別責務である  
- `Promise.all`、`Promise.allSettled`、`Promise.race` は待機集約や観測の道具である  
- 責任スコープは、「誰がこの仕事の面倒を見るのか」という境界を定める別の設計問題である  
- 近い場所に出てくるものを同じ棚に入れないことが、Promise 周辺の設計を安定させる  

### この章の核心

Promise の再学習とは、便利な書き方を覚え直すことではありません。  
Promise を結果待機と失敗伝播の中心として正しく置き直し、AbortSignal、中止伝播、timeout、retry、cleanup、責任スコープを別の棚に戻すことです。  
この地図を持つことで、Promise 周辺の「動くのに壊れる設計」を責務混線として読めるようになります。

## まとめ

- Promise は結果待機と失敗伝播の中心である  
- AbortSignal は中止伝播の道具であり、Promise の代用品ではない  
- timeout と retry は制御ポリシーである  
- cleanup は後始末であり、別責務として扱う必要がある  
- `all`、`allSettled`、`race` は待機集約や観測の棚にある  
- 責任スコープは、誰が仕事の面倒を見るのかという設計上の境界である  
- Promise 周辺の破綻は、便利 API 不足ではなく責務混線として起こりやすい  
- 持ち帰るべきものは、API 一覧ではなく責務の見取り図である  

この小冊子はここで終わりです。  
ただし、Promise の学び直しはここで止まりません。  
Rust や Python の async に進むときも、あるいは JavaScript でより複雑な非同期設計に向き合うときも、まず見るべきなのは「どの道具を使うか」だけではなく、「どの責務をどこに置くか」です。  
その問いを持ち帰れたなら、本書の目的は達成できています。
