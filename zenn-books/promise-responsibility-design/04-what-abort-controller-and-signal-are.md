---
title: "第4章　AbortController / AbortSignal は何者か"
---

**初学者**  
前の章で、Promise はキャンセルを単独では完結しないと整理しました。  
そこで出てくるのが AbortController / AbortSignal ですよね。  
でも正直、ここはいつも少し曖昧です。  
`fetch` に `signal` を渡すもの、くらいの理解で止まっています。

**解説者**  
その止まり方はかなり自然です。  
AbortController / AbortSignal は、Promise よりも役割が見えにくい道具だからです。  
ですが、この章で大事なのは機能一覧ではありません。  
まず棚を決めます。  
AbortSignal は結果の道具ではなく、中止の意思とその伝播経路に関わる道具です。

**実装者**  
実務でも、ここを Promise と同じ棚に入れてしまうと混乱します。  
`await` しているものの近くに `signal` が出てくるので、全部ひとまとめに「非同期の便利機能」に見えてしまう。  
でも実際には、Promise は結果待機、AbortSignal は中止伝播です。  
近い場所に出てくるだけで、責務は違います。

**初学者**  
つまりこの章は、「AbortSignal は Promise の代用品ではない」とはっきりさせる章なんですね。

**解説者**  
その通りです。  
AbortSignal を魔法のキャンセル API として見るのではなく、中止の責務を扱う別の道具として見直していきましょう。

## AbortSignal は結果ではない

**初学者**  
まず一番基本的なところですが、AbortSignal は何を表しているんでしょうか。  
Promise みたいに、成功か失敗かを持っているわけではないですよね。

**解説者**  
はい。  
AbortSignal は結果を表すものではありません。  
Promise のように「あとで何が返るか」を待つ窓口でもない。  
AbortSignal が表しているのは、「この処理はもう不要になった」「止めたい」という中止の意思です。

**実装者**  
ここをはっきり言うと、AbortSignal は値の箱ではなく、状態の共有窓口に近いです。  
ただし共有するのは結果ではなく、中止の状態です。  
「もう止めてよい」「止めるべきだ」という合図がここに乗る。

**初学者**  
Promise が「結果をどう受け取るか」の窓口だとすると、AbortSignal は「もう続けなくていい」という判断を伝える窓口なんですね。

**解説者**  
その理解で大丈夫です。  
だから AbortSignal を Promise と同じものとして扱うと、最初から話がずれます。  
片方は結果待機、もう片方は中止伝播です。

## AbortController は中止の起点を作る

**初学者**  
では AbortController は何をしているんですか。  
Signal と Controller が二つあるのも少しややこしいです。

**解説者**  
ここも責務で分けると見やすくなります。  
AbortController は、中止を発行する側の起点です。  
AbortSignal は、その中止状態を受け取る側に渡すための窓口です。

**実装者**  
つまり Controller は「止めると決める側」、Signal は「その決定を受け渡す側」と考えるとわかりやすいです。  
実際のコードでも、親側や呼び出し側が Controller を持ち、下位処理には Signal だけを渡すことが多い。  
この分離は大事です。

**初学者**  
なるほど。  
「止める権限」と「止める意思を受け取る窓口」を分けているんですね。

**解説者**  
そうです。  
この構造があるから、「誰が止める判断を持っているか」と「誰がその判断を受けて動くか」を分けて設計できます。  
ここは Promise の世界とはかなり違います。

## `fetch` に `signal` を渡す意味

**初学者**  
`fetch` に `signal` を渡す例はよく見ます。  
あれは結局、何を表しているんでしょうか。

**解説者**  
あれは、「この処理は中止の責任線の中にあります」と明示する行為です。  
単にオプションを足しているのではなく、この通信は外側の中止判断に従う、という設計上の接続をしているんです。

**実装者**  
実務ではここがとても重要です。  
たとえば画面遷移、タブの切り替え、検索条件の変更などで、前の通信が不要になることがあります。  
そのとき `signal` を渡していると、「親がもういらないと判断したら、この下位処理もそれに従う」という線が引けます。

**初学者**  
`signal` を渡すというのは、「止められるかもしれないから受け取っておいて」ではなく、「あなたはこの中止判断の配下にいます」という宣言に近いんですね。

**解説者**  
その表現はかなり本質に近いです。  
AbortSignal は便利オプションではなく、中止の責任関係をつなぐための道具です。

## AbortSignal は Promise の代用品ではない

**初学者**  
でも、実際のコードでは Promise と Signal がかなり近い場所にあります。  
`await fetch(..., { signal })` みたいに並ぶので、つい同じ棚に置いてしまいます。

**解説者**  
そこが一番の混線ポイントです。  
`await` しているからといって、`signal` まで結果待機の道具だと思ってはいけません。  
Promise は「結果がどうなるか」を扱う。  
AbortSignal は「もう続けるべきでない」という判断を運ぶ。  
両者は協力はしますが、同じ責務ではありません。

**実装者**  
極端に言うと、Promise がなくても中止判断の伝播という設計課題は存在しますし、AbortSignal がなくても Promise は結果待機と失敗伝播の道具として成立します。  
たまたま実務の非同期処理では、この二つが近くに現れるだけです。

**初学者**  
同じコード片に出てくるから同じ役目だと思ってしまうけれど、本当は「結果」と「中止」で話が違うわけですね。

**解説者**  
その通りです。  
この違いを崩すと、「Promise でキャンセルできるはずだ」とか「AbortSignal が結果も表してくれるはずだ」といった誤解が起きます。

## `Promise.withResolvers()` との違い

**初学者**  
前の章で `Promise.withResolvers()` が出てきました。  
あれも「外から何かを伝える窓口」を作る感じがあって、少し似て見えます。

**解説者**  
似て見えるのは自然ですが、責務は違います。  
`Promise.withResolvers()` が整理しているのは、結果待機の窓口と、その Promise を解決・失敗させる手段です。  
つまり Promise の世界の中で、結果の入口を見えやすくしている。  
一方の AbortSignal は、中止の意思を伝えるための窓口です。

**実装者**  
言い換えるなら、`withResolvers()` は「結果をどう流し込むか」の整理であって、「もう不要だから止める」という話ではありません。  
逆に AbortSignal は、中止状態を伝えるけれど、成功値や失敗値の受け渡しはしません。  
見た目が少し似ていても、流しているものが違います。

**初学者**  
なるほど。  
`withResolvers()` は結果の窓口、AbortSignal は中止の窓口。  
同じ「窓口」でも中身が違うんですね。

**解説者**  
その整理が大切です。  
窓口という言葉だけで同列に置かず、何を受け渡しているかまで見ないといけません。

## `Promise.try()` との違い

**初学者**  
では `Promise.try()` との違いはどうでしょう。  
これも Promise の入口をそろえる話でしたよね。

**解説者**  
はい。  
`Promise.try()` が整理しているのは、同期的に始まる処理も Promise の結果待機と失敗伝播の流れに載せることです。  
つまり、これも結果の棚の話です。  
AbortSignal のように「止める判断をどう渡すか」は扱っていません。

**実装者**  
ここも混ぜないほうがいいです。  
`Promise.try()` は、「失敗を Promise の流れに揃えたい」という入口整理です。  
AbortSignal は、「不要になったことを下位へ伝えたい」という責任線の整理です。  
一方は結果と失敗、もう一方は中止です。

**初学者**  
Promise 系の API はみんな結果の棚にいる。  
AbortSignal は別の棚にいる。  
そう覚えるとかなりすっきりします。

**解説者**  
その覚え方でよいです。  
むしろこの章では、それくらい明確に棚分けしたほうが混線を防げます。

## 中止は失敗と同じではない

**初学者**  
少し気になるのですが、中止されたときって、結果としては失敗っぽく見えますよね。  
それでも中止と失敗は別に考えたほうがいいんでしょうか。

**解説者**  
はい。  
観測上は失敗の形に見えることがあっても、意味としては別に考えたほうが整理しやすいです。  
失敗は「うまくいかなかった結果」です。  
一方で中止は、「もう続ける必要がない、または続けるべきでないという判断」です。  
原因も責任線も違います。

**実装者**  
ここを混ぜると、retry の判断も cleanup の設計も崩れます。  
本当に失敗したのか、途中でやめたのかで、次に取るべき行動は変わるからです。  
だから AbortSignal は、単なる失敗生成装置として見るべきではありません。

**初学者**  
なるほど。  
見え方としては失敗の形に落ちることがあっても、設計上は別の意味を持っているわけですね。

**解説者**  
その通りです。  
Promise が運ぶ失敗結果と、AbortSignal が運ぶ中止の意思は、関係はあっても同一ではありません。

## AbortSignal を渡しただけでは設計は終わらない

**初学者**  
ここまで聞くと、`signal` を渡せばかなり整理できるように思えてきます。  
でも、それでキャンセル設計は完成なんでしょうか。

**解説者**  
完成ではありません。  
AbortSignal は中止の意思を伝える道具ですが、それを受け取った下位処理がどう振る舞うかまでは自動で決めてくれません。  
実際にどこで止まるのか、何を cleanup するのか、どこまで責任を持つのかは別途設計が必要です。

**実装者**  
実務ではここが落とし穴です。  
`signal` を渡したことで安心してしまう。  
でも下位の関数が signal を受け取っていない、受け取っても見ていない、cleanup をしない、といったことが普通に起きます。  
すると「中止できるつもり」の設計だけが残ります。

**初学者**  
つまり AbortSignal は、中止伝播の線を引く道具ではあるけれど、その線の先で何をするかは別に考えないといけないんですね。

**解説者**  
その理解で大丈夫です。  
AbortSignal はとても重要ですが、万能停止装置ではありません。  
中止の意思と責任線を表すための道具です。

### 整理

- AbortSignal は結果を表す道具ではなく、中止の意思を表す道具である  
- AbortController は中止を発行する起点であり、AbortSignal はその状態を受け渡す窓口である  
- `fetch` に `signal` を渡すことは、中止の責任線を下位処理へ接続することである  
- AbortSignal は Promise の代用品ではなく、結果待機とは別の棚にある  
- `Promise.withResolvers()` は結果の窓口の整理、AbortSignal は中止の窓口の整理である  
- `Promise.try()` も結果待機と失敗伝播の入口整理であり、中止伝播の道具ではない  
- 中止は失敗と関係することはあるが、設計上は別の意味を持つ  
- AbortSignal を渡しただけでは設計は終わらず、cleanup や下位処理の対応が別途必要である  

### この章の核心

AbortSignal は Promise の代用品ではありません。  
Promise が結果待機と失敗伝播の道具であるのに対し、AbortSignal は中止の意思とその伝播経路を扱う別の道具です。  
`signal` を渡すことは、便利オプションを付けることではなく、中止の責任線を下位処理へ接続することだと理解する必要があります。

## まとめ

- AbortSignal は結果ではなく、中止の意思を表す  
- AbortController は中止の起点、AbortSignal は中止の受け渡し窓口である  
- Promise は結果待機、AbortSignal は中止伝播であり、責務が違う  
- `Promise.withResolvers()` や `Promise.try()` は結果の棚にある  
- 中止は失敗と同じではなく、意味と責任線が異なる  
- `signal` を渡しただけでは設計は完成せず、下位処理や cleanup の設計が必要になる  

次章では、ここで見た AbortSignal を踏まえて、キャンセルを単なる「止める命令」ではなく、責任の伝播として捉え直します。  
誰が止めるのか、どこまで伝えるのか、なぜ cleanup が必要になるのか。  
キャンセル設計を責任線の問題として整理していきます。
