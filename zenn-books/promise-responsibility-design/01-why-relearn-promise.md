---
title: "第1章　なぜ Promise を学び直すのか"
---

**初学者**  
Promise って、そんなに学び直す必要がありますか。  
`async` / `await` で普通に書けていますし、困ったら `try...catch` を付ければだいたい済みますよね。

**解説者**  
そこが、この章の出発点です。  
書けることと、説明できることは違います。  
Promise は毎日使えるけれど、何を担当していて、何を担当していないのかは意外と曖昧になりやすいんです。

**実装者**  
実務でもそこはよく崩れます。  
たとえば「キャンセル対応しました」と言いながら、画面では止まったように見えるだけで、通信自体は裏で続いていたりします。  
あるいは timeout を入れて安心したつもりでも、cleanup が抜けていて資源が残ったりします。

**初学者**  
でもそれって Promise が弱いというより、周辺の設計の話ですよね。

**解説者**  
その通りです。  
この本も Promise を弱い道具として描くつもりはありません。  
むしろ Promise は、将来値を扱い、成功と失敗を同じ流れに載せ、`await` と接続し、結果待機の形をそろえる強力な道具です。  
だからこそ、どこまでが Promise の責務で、どこから先が別の責務なのかを見直す価値があります。

## Promise は「非同期そのもの」ではない

**初学者**  
でも、Promise って非同期処理そのものじゃないんですか。

**解説者**  
そこが最初の誤解です。  
Promise は「非同期の世界にいる何でも屋」ではありません。  
まずはもっと狭く見たほうがいい。  
Promise は、結果をあとで受け取るための窓口であり、その結果が成功か失敗かを同じ流れで扱うための道具です。

**実装者**  
たとえば API 呼び出しで Promise が返ってくると、つい「通信している主体」みたいに見えますよね。  
でも Promise 自体が通信しているわけではありません。  
Promise は、通信の結果を待ち受ける箱に近い。

**初学者**  
じゃあ Promise は、処理そのものというより、結果の受け取り方をそろえる道具なんですね。

**解説者**  
そうです。  
この本では、まず Promise を「結果待機と失敗伝播」の道具として捉え直します。  
そのうえで、キャンセル、timeout、retry、cleanup は別の棚に置いて整理していきます。

## それでも Promise は強力である

**初学者**  
でも、そんなふうに切り分けると、Promise の仕事って思ったより小さく見えます。

**解説者**  
小さいのではなく、焦点が合うんです。  
Promise の仕事は狭いけれど、その狭さが強さでもあります。  
結果待機と失敗伝播に責務を集中させたから、`then` でも `catch` でも `await` でも、一貫した扱いができる。

**実装者**  
逆に言うと、Promise に他の責務まで背負わせ始めると、設計が濁ります。  
キャンセルも timeout も retry も cleanup も、全部 Promise の話に見えてくる。  
でもその状態は便利そうに見えて、あとで整理不能になります。

**初学者**  
Promise が弱いのではなく、Promise に何でも押し込むほうが危ないわけですね。

**解説者**  
その理解で大丈夫です。  
この本は Promise を否定するための本ではありません。  
Promise の価値を認めたうえで、どこから先が別の棚なのかを見えるようにする本です。

## なぜ実務では混線しやすいのか

**初学者**  
でも、実際のコードでは全部近くに出てきますよね。  
`await` して、必要なら timeout して、`signal` を渡して、失敗したら retry して……。  
なら、まとめて考えたくなるのも自然では。

**解説者**  
自然です。  
だからこそ混線しやすい。  
同じ場所に出てくるものと、同じ責務であることは違います。

**実装者**  
実際の現場ではこうです。  
結果を待っているのに、いつの間にか中止判断まで同じ関数に埋まり、さらに retry の方針まで入り、最後に cleanup も雑に追加される。  
動くコードにはなるけれど、「誰が待つのか」「誰が止めるのか」「誰が失敗を拾うのか」「どこまでが責任範囲か」が説明しにくくなる。

**初学者**  
たしかに、説明しにくいコードはあります。  
直せと言われても、どこをどう切ればいいのか迷います。

**解説者**  
その迷いを減らすために、この本では最初から棚を分けます。  

- Promise は結果待機と失敗伝播  
- AbortSignal は中止伝播  
- timeout と retry は制御ポリシー  
- cleanup は後始末  
- 責任スコープはさらに別の設計問題  

まずはこの地図を持つことが大事です。

## この本で学び直すもの

**初学者**  
つまり、この本は Promise の書き方を覚え直す本ではないんですね。

**解説者**  
その通りです。  
Promise を API としてではなく、責務として見直す。  
それがこの章の役割です。

**実装者**  
先の章では、Promise が何を引き受けるのか、何を引き受けないのか、AbortSignal は何者か、キャンセルはなぜ責任の伝播なのか、`Promise.all` や timeout や retry をどの棚に置くべきかを順番に見ていきます。

**初学者**  
学び直すというより、散らばっていた理解を地図に戻す感じですね。

**解説者**  
その言い方が近いです。  
Promise 周辺は、知識が足りないから混乱するとは限りません。  
むしろ、使えているからこそ混ざる。  
この本は、その混ざった理解をほどいて、説明できる形に戻していきます。

### 整理

- Promise は非同期処理そのものではなく、結果待機と失敗伝播の中心にある道具である  
- Promise は強力だが、キャンセル、timeout、retry、cleanup まで同じ責務ではない  
- 実務で混線しやすいのは、近い場所に出てくるものを同じ棚に入れてしまうからである  
- この本は API の網羅ではなく、責務の切り分けを学び直すための小冊子である  

### この章の核心

Promise を学び直す理由は、使えていないからではありません。  
使えているのに、何を担当し、何を担当していないのかが曖昧になりやすいからです。  
本書では Promise を結果待機と失敗伝播の道具として捉え直し、そこから中止伝播、制御ポリシー、後始末、責任スコープを切り分けていきます。

## まとめ

- Promise は毎日使えていても、責務は混線しやすい  
- Promise は結果待機と失敗伝播に強い  
- Promise を弱い道具としてではなく、責務のはっきりした強い道具として見ることが重要である  
- AbortSignal、timeout、retry、cleanup、責任スコープは別の棚として扱う必要がある  
- Promise の再学習とは、便利な書き方を覚え直すことではなく、責務を置き直すことである  

次章では、Promise が実際に何を引き受けているのかをもう少し正面から整理します。  
`await`、成功、失敗、`Promise.try()`、`Promise.withResolvers()` まで含めて、まずは Promise 自身の守備範囲をはっきりさせます。
