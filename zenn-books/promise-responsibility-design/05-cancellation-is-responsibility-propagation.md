---
title: "第5章　キャンセルは責任の伝播である"
---

**初学者**  
前の章で、AbortSignal は中止の意思と伝播経路の道具だと整理できました。  
でも、まだ少し抽象的です。  
キャンセルって結局、「止める」ということではないんですか。

**解説者**  
もちろん「止める」という面はあります。  
ただ、その言い方だけだと重要なことが抜けます。  
キャンセルは単なる停止命令ではなく、「誰がもう不要だと判断したのか」「その判断をどこまで下位へ伝えるのか」という責任の問題でもあります。

**実装者**  
実務で壊れやすいのもそこです。  
ボタンを押した、画面を閉じた、タイムアウトした。  
その瞬間に「止めたつもり」にはなります。  
でも実際には、どの処理が誰の配下にあり、どこまで中止を伝えるのかが設計されていないと、通信やタイマーやイベント待機が残ります。

**初学者**  
つまりキャンセルは、気分として「もういらない」と思うだけでは終わらないんですね。

**解説者**  
その通りです。  
この章では、キャンセルを「停止の API」ではなく、「責任の伝播」として見直します。  
そうすると、なぜ signal を下位関数へ渡す必要があるのか、なぜ cleanup が別に必要なのかも見えてきます。

## キャンセルは「もう不要」という判断である

**初学者**  
まず最初に確認したいのですが、キャンセルって失敗とは違うんですよね。

**解説者**  
はい。  
キャンセルの中心にあるのは、「失敗した」という事実ではなく、「もうこの処理を続ける必要がない」という判断です。  
これは技術的な異常というより、所有者や呼び出し側の都合に近いこともあります。

**実装者**  
たとえば検索ボックスで文字を打ち直したとき、前の検索リクエストは「失敗した」わけではありません。  
ただ、もうその結果はいらない。  
画面遷移でも同じです。  
遷移前の画面のために始めた通信は、新しい画面に移った時点で不要になります。

**初学者**  
なるほど。  
キャンセルはエラー処理というより、所有者が「この仕事はもう終わっていい」と判断することなんですね。

**解説者**  
そうです。  
だからキャンセルを考えるときは、まず「誰がこの処理の所有者なのか」を見ないといけません。  
所有者が不要と判断したとき、その判断が下位へ届くようになっているか。  
そこが本質です。

## 親が不要と判断したら、子へどう伝えるか

**初学者**  
「所有者」という言葉が出ましたが、これは親子関係のように考えるといいんでしょうか。

**解説者**  
はい。  
キャンセルを責任の伝播として理解するとき、親子関係で考えるとかなり整理しやすくなります。  
上位の処理が下位の処理を起動したなら、上位は「この仕事はもういらない」と判断する立場になりえます。  
そのとき、その判断を子へ渡す経路が必要です。

**実装者**  
たとえば画面コンポーネントが API 呼び出しを開始したなら、その通信は画面の責任線の内側にあります。  
画面が閉じたら、配下の通信も不要になる。  
でも、その線がコード上に引かれていないと、親だけが「終わったつもり」になって、子の処理は生き続けます。

**初学者**  
つまりキャンセルは、「今すぐ止まれ」という号令というより、「この親の配下にある処理だから、親の判断に従う」という関係性なんですね。

**解説者**  
その理解が大切です。  
AbortSignal は、その関係をコードに落とし込むための経路です。  
単なる便利オプションではなく、責任線をつなぐためのものとして見る必要があります。

## signal を下位関数へ渡す意味

**初学者**  
では、signal を下位関数へ渡すことには、どんな意味があるんでしょうか。  
単に API がそうなっているから渡す、ではだめなんですよね。

**解説者**  
はい。  
signal を渡すというのは、「この下位処理は、上位の中止判断の配下にあります」と明示することです。  
つまり、中止の責任線をその関数の中まで伸ばしているわけです。

**実装者**  
ここは実務でかなり差が出ます。  
上位で controller を作っていても、途中の関数が signal を受け取らない、あるいは下位へ渡さないと、その地点で責任線が切れます。  
すると、親はキャンセルしたつもりでも、途中から先は別世界になってしまいます。

**初学者**  
signal を受け取らない関数って、「自分は親の中止判断とは無関係です」と言っているのに近いんですね。

**解説者**  
極端に言えばそうです。  
もちろん、すべての関数が signal を受け取るべきだという話ではありません。  
ですが、親の責任線の内側にある処理なら、その線をどこでつなぎ、どこで切るのかを意識しないといけません。

## 「止めたつもり」と「実際に止まる」は違う

**初学者**  
ここまで聞いてくると、キャンセルの難しさは「止める API を呼ぶかどうか」よりも、「その影響が本当に届くかどうか」にありそうです。

**解説者**  
その通りです。  
キャンセル設計で危ないのは、「止めたつもり」で満足してしまうことです。  
上位が不要と判断しても、その判断が下位に伝わらなければ、実際の処理は続きます。

**実装者**  
たとえば UI ではすでに古い検索結果を捨てたつもりなのに、通信は裏で継続していて、あとから遅れて返ってくることがあります。  
あるいは timeout で諦めたつもりでも、接続やイベント待ちが残っていることもあります。  
このとき問題なのは「キャンセル API がなかったこと」ではなく、「責任線が途中で切れていたこと」です。

**初学者**  
見た目として処理を無視するだけでは、本当に止まったことにはならないわけですね。

**解説者**  
そうです。  
キャンセルを観測の問題だけで済ませると危険です。  
重要なのは、責任を持つ側の判断が、対象の処理に届き、その処理がそれに応じて振る舞うことです。

## timeout と手動キャンセルは競合しうる

**初学者**  
キャンセルには、ユーザーが止める場合もあれば、timeout のように時間切れで止める場合もありますよね。  
この二つは同じように考えていいのでしょうか。

**解説者**  
起点は違いますが、どちらも「もう続けない」という判断を発生させる点では似ています。  
ただし、設計上は区別したほうがよいです。  
手動キャンセルはユーザーや上位の意思決定です。  
timeout は制御ポリシーです。  
理由は違いますが、どちらも下位へ中止を伝える責任線に接続される必要があります。

**実装者**  
現場ではここが競合します。  
ユーザーが画面を閉じたのと、timeout が発火したのがほぼ同時、ということは普通にあります。  
そのとき「誰がキャンセルしたのか」だけでなく、「キャンセル後に何を片づけるのか」「二重に片づけても大丈夫か」まで考えないといけません。

**初学者**  
たしかに、止める理由が複数あると、処理の終わり方も複雑になりますね。

**解説者**  
だからこそ、timeout を Promise の一部として雑に押し込むのではなく、制御ポリシーとして別に置きつつ、中止伝播とどう接続するかを考える必要があります。

## cleanup はキャンセルとは別責務である

**初学者**  
前の章でも cleanup は別責務だと聞きました。  
でもキャンセルの話をしていると、どうしても一緒に見えてきます。

**解説者**  
近い場所に出てくるので混同しやすいですが、やはり別です。  
キャンセルは「もう不要だ」という判断と、その伝播です。  
cleanup は、その結果として何を片づけるかという後始末です。  
中止が伝わっただけで後始末が自動で終わるわけではありません。

**実装者**  
たとえばイベントリスナーの解除、タイマーの停止、途中状態の破棄、接続のクローズなどは cleanup です。  
中止判断が届いても、これらをどこで誰がやるかは別に決めないといけません。  
ここを省くと、「キャンセルはしたのに何かが残る」という状態になります。

**初学者**  
つまりキャンセルは責任線、cleanup は片づけの内容なんですね。  
つながってはいるけれど、同じ仕事ではない。

**解説者**  
その理解で大丈夫です。  
この区別を持っておくと、「signal を渡したから大丈夫」という思い込みを避けられます。

## 下位関数が signal を受け取らないと何が起きるか

**初学者**  
もう少し具体的に聞きたいです。  
下位関数が signal を受け取らないと、どんな壊れ方をするんでしょうか。

**解説者**  
典型的には、親の責任線が途中で切れます。  
上位は「この仕事はもう不要」と判断しているのに、その判断が奥まで届かない。  
すると、途中から先の処理だけが独立して生き続けます。

**実装者**  
たとえば `loadUser(signal)` の中でさらに `fetchProfile()` や `fetchPosts()` を呼んでいるのに、それらへ signal を渡していないとします。  
すると `loadUser` の呼び出し元は中止したつもりでも、実際には内側の通信が継続することがあります。  
この壊れ方は厄介で、見た目ではわかりにくいのに、帯域やメモリや状態整合性に影響します。

**初学者**  
責任線が切れると、動いてはいるけれど所有者不明の仕事が残るわけですね。

**解説者**  
まさにそこです。  
キャンセル設計は、孤立した仕事を作らないための設計でもあります。

## キャンセル設計は所有関係の設計である

**初学者**  
ここまでの話をまとめると、キャンセルってかなり設計論ですね。  
単なる API の使い方ではない気がしてきました。

**解説者**  
その通りです。  
キャンセル設計は、「誰が誰を起動し、誰が不要と判断し、どこまでその判断を伝えるか」という所有関係の設計です。  
AbortSignal は、その所有関係をコード上に落とすための重要な道具ですが、設計そのものを代行してくれるわけではありません。

**実装者**  
実務でも、キャンセルがうまくいっているコードは、たいてい ownership が見えます。  
親がいて、子がいて、責任線があり、cleanup の場所も決まっている。  
逆に壊れやすいコードは、「どの処理が誰のものか」が曖昧です。  
その状態で signal だけ追加しても、根本は直りません。

**初学者**  
キャンセルは停止機能というより、責任の線をきちんと引くための設計問題なんですね。

**解説者**  
そうです。  
この章で持ち帰りたいのはそこです。  
キャンセルを「止める命令」とだけ見ると浅くなります。  
「誰の判断が、どの仕事まで届くべきか」として見ると、設計の輪郭が見えてきます。

### 整理

- キャンセルは「失敗」ではなく、「もう不要である」という判断である  
- キャンセルを考えるには、まず処理の所有者と責任線を見る必要がある  
- 親が不要と判断したら、その判断を子へ伝える経路が必要である  
- signal を下位関数へ渡すことは、中止の責任線をその先まで伸ばすことである  
- 「止めたつもり」と「実際に止まる」は違い、責任線が切れると下位処理は残る  
- timeout と手動キャンセルは理由は違っても、中止判断として競合しうる  
- cleanup はキャンセルと近いが別責務であり、後始末を別に設計する必要がある  
- キャンセル設計は、所有関係と伝播経路の設計である  

### この章の核心

キャンセルは、単なる停止命令ではありません。  
上位の処理が「もう不要だ」と判断したとき、その判断を下位の処理へどこまで伝えるかという、責任の伝播です。  
AbortSignal はその責任線をつなぐ道具ですが、cleanup や所有関係の設計まで自動で解決するわけではありません。

## まとめ

- キャンセルは「失敗」ではなく「不要になった」という判断である  
- キャンセル設計では、誰が処理の所有者かを明確にする必要がある  
- 親の中止判断を子へ伝える経路として signal が使われる  
- signal を下位へ渡さないと、責任線が途中で切れる  
- 「止めたつもり」ではなく、実際にどこまで止まるかを設計しなければならない  
- timeout と手動キャンセルは別起点だが、中止伝播の上では接続されうる  
- cleanup はキャンセルとは別責務であり、別に設計する必要がある  
- キャンセル設計は、所有関係と責任線の設計である  

次章では、ここまで整理してきた責務マップを使って、`Promise.all`、`Promise.allSettled`、`Promise.race`、timeout、retry をまとめて見直します。  
よく使う道具ほど、同じ棚に入って見えやすいものです。  
待機集約、制御ポリシー、中止伝播の違いをもう一度はっきりさせていきます。
