---
title: "第7章　動くのに壊れる Promise 設計"
---

**初学者**  
ここまでで責務の地図はかなり整理できました。  
でも実務で困るのは、むしろ「ちゃんと動いているように見えるコード」なんです。  
エラーも出ていないし、画面上も一応動く。  
それなのに、あとから妙な不具合が出たり、設計が説明しにくかったりします。

**解説者**  
その感覚はとても重要です。  
Promise 周辺で本当に厄介なのは、「まったく動かないコード」よりも、「動いているので見逃されるコード」です。  
この章では、そうした壊れ方をケーススタディとして読みます。  
ポイントは、便利 API が足りないから壊れるのではなく、責務が混線しているから壊れると見ることです。

**実装者**  
実務では、signal も渡している、timeout もある、retry も入っている、`Promise.all` でもまとめている。  
一見するとかなり頑張っているコードに見えます。  
でも実際には、「誰が待つのか」「誰が止めるのか」「誰が片づけるのか」が曖昧で、結果として不安定になることがあります。

**初学者**  
つまりこの章は、「どの API を使うか」よりも、「その API をどの責務として置いているか」を点検する章なんですね。

**解説者**  
その通りです。  
動くことと、責任が正しく置かれていることは別です。  
この違いを見抜けるようになると、Promise 周辺の設計はかなり安定します。

## ケース1　signal を受け取らない下位関数

**初学者**  
まず一つ目は、やはり signal 周りでしょうか。

**解説者**  
はい。  
とてもよくあるのが、上位では AbortController を持っていて、途中までは `signal` を渡しているのに、下位関数のどこかでその線が切れているケースです。

**実装者**  
たとえば画面側が `loadPage(signal)` を呼び、その中で `loadUser()` と `loadPosts()` を呼んでいるとします。  
でも `loadUser()` や `loadPosts()` が signal を受け取らず、さらにその先の `fetch` にも渡していない。  
このとき上位はキャンセルしたつもりでも、内側の通信は普通に続きます。

**初学者**  
でも画面側では、もう結果を使わないようにしていれば問題ない気もします。

**解説者**  
それは「観測をやめた」だけであって、「仕事を止めた」わけではありません。  
帯域、メモリ、接続、イベント待機、遅延した状態更新の危険は残ります。  
責任線が途中で切れているからです。

**実装者**  
このケースの問題は、signal API を知らないことではありません。  
「親の中止判断をどこまで下位へ伝えるか」という責務の線を、途中で見失っていることです。

**初学者**  
つまり壊れているのは、キャンセル機能ではなく、所有関係の設計なんですね。

## ケース2　timeout だけあって cleanup がない

**初学者**  
timeout があると安心してしまうケースもありそうです。

**解説者**  
あります。  
しかも非常に多いです。  
一定時間で「もう待たない」と決めたことで、何か大きな問題を解決した気になってしまう。  
でも timeout は制御ポリシーであって、cleanup ではありません。

**実装者**  
たとえば timeout で待機を打ち切っても、タイマーの解除、イベントリスナーの解除、途中状態の破棄、接続のクローズなどが別途必要なことがあります。  
これらが設計されていないと、「ユーザーには終わったように見えるが、内部には何かが残る」という状態になります。

**初学者**  
確かに timeout は「諦める」だけで、「片づける」とは言っていないですね。

**解説者**  
その通りです。  
このケースで壊れているのは、timeout が弱いからではありません。  
制御ポリシーと後始末を同じ棚に入れてしまっていることです。

**実装者**  
特に Promise 周辺では、`race` と timeout の組み合わせで「終わった感」が強く出るので危険です。  
見えなくなったことと、片づいたことは違います。

## ケース3　retry がキャンセルを無視する

**初学者**  
retry 周りも危なそうですね。  
失敗したら自動で再試行してくれるのは便利ですが。

**解説者**  
便利です。  
ただし、便利だからこそ責任線を踏み越えやすい。  
典型例は、上位がもう不要だと判断しているのに、retry ロジックだけが勝手に仕事を続けてしまうケースです。

**実装者**  
たとえば画面はすでに閉じられた。  
でも内部の retry ループは「まだ規定回数に達していないから」と言って再試行を続ける。  
すると、その仕事はもう誰のためのものなのかわからなくなります。  
所有者を失ったまま走る処理になります。

**初学者**  
retry って、失敗を受けてやり直すだけの仕組みに見えますけど、本当は「まだこの仕事が必要か」も確認しないといけないんですね。

**解説者**  
その通りです。  
retry は失敗処理ポリシーですが、責任スコープの外で暴走してはいけません。  
中止伝播や所有者の判断と接続されてはじめて、安全な retry になります。

**実装者**  
このケースで壊れているのは、retry という機能ではありません。  
失敗処理ポリシーが、中止伝播や所有関係から切り離されていることです。

## ケース4　`Promise.all` でまとめたつもりが責任スコープになっていない

**初学者**  
`Promise.all` を使うと、かなり整理できた気がしてしまいます。  
でもこれも危ないことがあるんですよね。

**解説者**  
はい。  
`Promise.all` は待機集約としてはとても強いですが、それをそのまま責任スコープだと思ってしまうのが典型的な混線です。

**実装者**  
たとえば三つの通信を `Promise.all` に入れたとします。  
でも、その三つが本当に同じ親の配下にあるとは限りません。  
あるものは画面の寿命に従うべきで、あるものはキャッシュ更新として継続してよいかもしれない。  
あるものは部分失敗を許容したいかもしれない。  
それなのに、同じ配列に入れたことで「同じ責任の仕事だ」と思い込むと、設計が雑になります。

**初学者**  
待ち方をまとめたことと、責任の境界を決めたことは別なんですね。

**解説者**  
そうです。  
`Promise.all` は結果待機を集約しているだけです。  
所有関係、中止伝播、cleanup の境界は、別に設計しなければなりません。

## ケース5　UI 側はキャンセルしたつもりだが通信は続いている

**初学者**  
これまでの話をまとめたようなケースですが、UI ではキャンセル済みなのに通信が続いている、というのはやはり典型ですか。

**解説者**  
典型です。  
しかも見つけにくい。  
画面はすでに新しい状態になっているので、古い通信の存在が見えにくいからです。

**実装者**  
検索ボックスで文字を打ち直したとき、前の検索結果を無視するようにしておけば、ぱっと見では問題ないように見えます。  
でも裏では前の通信が継続し、帯域を使い、サーバーに負荷をかけ、タイミングによっては遅れて状態更新を引き起こすこともあります。

**初学者**  
「表示しない」と「止めた」は違うわけですね。

**解説者**  
その通りです。  
このケースの問題は、結果の観測を止めたことを、仕事の中止と混同していることです。  
Promise の観測と AbortSignal による中止伝播を、同じ棚に入れてしまっている典型例です。

## ケース6　`Promise.withResolvers()` で窓口を作ったが責任設計は空のまま

**初学者**  
`Promise.withResolvers()` も便利そうですが、これも壊れ方がありそうです。

**解説者**  
あります。  
`withResolvers()` は結果待機の窓口を明示しやすくする道具ですが、それだけで責任設計が整うわけではありません。

**実装者**  
たとえばイベントを Promise 化したいから `withResolvers()` で解決・失敗の窓口を外に出した。  
それ自体は整理として有効です。  
でも、その Promise がどの所有者に属するのか、不要になったらどう破棄するのか、イベント購読をいつ解除するのかが決まっていないと、窓口だけきれいで責任設計は空のままです。

**初学者**  
確かに、Promise の入口や出口が見やすくなっても、その仕事が誰のものかは別問題ですね。

**解説者**  
その通りです。  
`withResolvers()` は結果の棚の整理です。  
中止伝播や cleanup や所有関係の設計までは肩代わりしません。

## ケース7　`Promise.try()` で失敗を Promise 化したが cleanup は別問題

**初学者**  
`Promise.try()` も似ていますか。  
失敗を Promise の流れに乗せられるなら、かなり整理された感じがしますが。

**解説者**  
整理はされます。  
ただし、あくまで結果待機と失敗伝播の入口がそろうだけです。  
cleanup や中止伝播まで同時に解決するわけではありません。

**実装者**  
たとえば前処理で同期的に設定を読み、途中でイベントを登録し、その後で Promise の流れに乗せる処理があるとします。  
`Promise.try()` によって失敗はきれいに Promise 化できます。  
でも、途中で確保した資源や登録したフックをいつ外すかは、やはり別に考えないといけません。

**初学者**  
失敗を一つの流れにそろえることと、仕事の後始末を設計することは別なんですね。

**解説者**  
そうです。  
`Promise.try()` を使っても、責務の地図そのものは消えません。  
むしろ入口がそろったぶん、他の責務を意識的に分けないと見落としやすくなります。

## 壊れているのは「動き」ではなく「責務の置き方」である

**初学者**  
こうして並べてみると、どのケースも「API の使い方が完全に間違っている」わけではないですね。  
むしろ、ちゃんと動いているからこそ危ない。

**解説者**  
その通りです。  
この章で見た壊れ方の多くは、構文エラーではありません。  
Promise も signal も timeout も retry も、個々には使われています。  
それでも壊れるのは、責務の棚が崩れているからです。

**実装者**  
結果待機、失敗伝播、中止伝播、制御ポリシー、cleanup、責任スコープ。  
これらが同じ関数の近くに出てくると、人はつい一つの塊として扱いたくなります。  
でも実際には、そこを分けて読めないと「動いているのに不安定」という状態から抜け出せません。

**初学者**  
つまり、Promise 周辺の設計レビューで見るべきなのは、「便利 API をちゃんと使っているか」よりも、「責務がどこに置かれているか」なんですね。

**解説者**  
まさにそこです。  
API の有無より、責任の配置。  
この視点を持てると、壊れたコードを責務混線として読めるようになります。

### 整理

- signal を受け取らない下位関数は、中止の責任線を途中で切ってしまう  
- timeout があっても cleanup がなければ、後始末が残る  
- retry がキャンセルを無視すると、所有者を失った仕事が走り続ける  
- `Promise.all` は待機集約であって、責任スコープそのものではない  
- UI が結果を無視しても、下位の通信が止まるとは限らない  
- `Promise.withResolvers()` は結果の窓口を整理するが、責任設計までは作らない  
- `Promise.try()` は失敗の入口を揃えるが、cleanup や中止伝播は別問題である  
- 「動くのに壊れる」コードの多くは、API 不足ではなく責務混線として読める  

### この章の核心

Promise 周辺で厄介なのは、動かないコードよりも、動いているので見逃されるコードです。  
signal、timeout、retry、`Promise.all`、`withResolvers()`、`Promise.try()` は、それぞれ一定の役割を果たします。  
それでも設計が壊れるのは、結果待機、中止伝播、制御ポリシー、cleanup、責任スコープが同じ棚に押し込まれているからです。

## まとめ

- Promise 周辺では、「動く」ことと「責任が正しい」ことは別である  
- signal を渡さない地点では、中止の責任線が切れる  
- timeout は後始末を代行しない  
- retry は所有者の中止判断を無視してはいけない  
- `Promise.all` は待機集約であり、責任スコープではない  
- UI が結果を捨てても、下位処理が止まるとは限らない  
- `withResolvers()` や `Promise.try()` も、責任設計そのものを解決するわけではない  
- Promise 設計の破綻は、便利 API 不足ではなく責務混線として読める
