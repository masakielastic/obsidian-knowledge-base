# Promise 小冊子 執筆ガイドライン

## このガイドラインの目的

本ガイドラインは、JavaScript ユーザー向けの小冊子
**「Promise 再学習 ― 結果待機・失敗・キャンセルを整理する」**
を、一貫した方針で執筆するための基準をまとめたものです。

この小冊子は、Promise の API を網羅する本ではありません。  
また、AbortController / AbortSignal の使い方だけを集めた実用メモでもありません。

本書の目的は、Promise 周辺で混線しやすい責務を切り分け、

- Promise は何を引き受けるのか
- Promise は何を引き受けないのか
- AbortSignal はどの責務に関わるのか
- timeout / retry / cleanup はどこに置くべきか
- なぜ「動くのに壊れる設計」が起きるのか

を、対話形式で学び直せるようにすることです。

---

## 本書の基本方針

### 1. Promise を API ではなく責務で書く

本書は、Promise の構文やメソッド一覧を覚えることを目的としません。  
Promise を **結果待機と失敗伝播の道具** として位置づけ、その責務を明確にすることを優先します。

そのため、各章では常に次の問いに戻ります。

- Promise は何をしているのか
- Promise は何をしていないのか
- その周辺で別の責務を担っているものは何か

### 2. Promise を過小評価しない

本書は Promise の限界を扱いますが、Promise を弱い道具として描いてはいけません。

Promise は、

- 将来値を扱う
- 成功と失敗を同じ流れに載せる
- `await` と接続する
- 結果待機の形をそろえる

という点で非常に強力です。

したがって、本書では必ず

1. Promise の価値を認める  
2. そのうえで限界を示す  
3. 別責務との境界を引く

という順番で説明します。

### 3. AbortSignal を魔法のキャンセル API として書かない

AbortController / AbortSignal は、本書において重要な題材です。  
ただし、それを「止めるボタン」「便利な中断 API」とだけ書いてはいけません。

本書では AbortSignal を、

- 中止の意思
- 中止の伝播経路
- 下位処理への責任の受け渡し
- cleanup を考える契機

として扱います。

つまり、AbortSignal は **結果** の道具ではなく、**中止の責務** に関わる道具として描きます。

### 4. Promise と AbortSignal を同じ棚に入れない

本書で特に重要なのは、Promise と AbortSignal を別責務として描くことです。

- Promise は結果待機と失敗伝播
- AbortSignal は中止の意思と伝播

この違いを崩してはいけません。

「Promise でキャンセルする」
「AbortSignal で結果を表す」
のような書き方は避け、必ずどちらの責務かを明示します。

### 5. timeout / retry / cleanup を別責務として扱う

本書では、次のものを同じ棚に入れてはいけません。

- Promise
- AbortSignal
- timeout
- retry
- cleanup

それぞれの基本的な位置づけは次の通りです。

- Promise = 結果待機 / 失敗伝播
- AbortSignal = 中止伝播
- timeout = 制御ポリシー
- retry = 再試行ポリシー
- cleanup = 後始末

これらが近い場所に現れるからといって、同じ責務として書かないでください。

### 6. 本書は「動くコード」より「責任が説明できる設計」を重視する

Promise 周辺では、動いているコードがそのまま正しい設計に見えやすいです。  
しかし本書は、便利な実装例を集める本ではありません。

本書が重視するのは、次の問いに答えられることです。

- 誰が待つのか
- 誰が失敗を拾うのか
- 誰が止めるのか
- どこまでが一つの責任範囲か

コード例を出す場合も、「どう動くか」だけでなく「責任がどこに置かれているか」を説明してください。

## 想定読者

本書の想定読者は、次のような JavaScript ユーザーです。

- `Promise` と `async` / `await` は日常的に使っている
- `Promise.all` もある程度使っている
- `fetch` と AbortSignal を見たことがある、または少し使ったことがある
- しかし、責務分解やキャンセル設計はまだ曖昧
- Rust や Python の async に進む前に、JavaScript の Promise 周辺を整理したい

したがって、読者は完全な初学者ではありません。  
一方で、責務の言語化にはまだ慣れていない前提で書きます。

## 章構成の基本フォーマット

各章は原則として次の形で構成します。

1. 導入  
2. 対話本文  
3. `### 整理`  
4. `### この章の核心`  
5. `## まとめ`  
6. 次章への接続

### 導入

導入では、その章で扱う誤解や違和感を初学者に言わせます。  
解説者は、その章の中心命題を早めに示してください。

### 対話本文

対話本文では、新しい概念を一気に増やしすぎないようにします。  
1つの節で扱う中心論点は1つに絞ってください。

### `### 整理`

本書では `### 整理` を積極的に使ってよいです。  
理由は、Promise 周辺は責務が近接して見えやすく、視覚整理が理解に直結するからです。

ただし、整理は長くしすぎず、3〜5項目程度に抑えます。

### `### この章の核心`

各章の終盤には、その章の中心命題を2〜4文程度で圧縮して示します。  
ここでは必ず、その章で扱った責務の違いを言い切ってください。

### `## まとめ`

まとめでは、章の要点を箇条書きで再提示します。  
新情報は入れません。

## 用語の扱い

### 1. 用語は最初に棚を決める

新しい用語が出たら、まず次のどれに属するかを明示します。

- 結果待機
- 失敗伝播
- 中止伝播
- 制御ポリシー
- 後始末
- 責任スコープ

たとえば、

- `Promise.try()` は結果待機 / 失敗伝播の入口
- `Promise.withResolvers()` は結果待機の窓口を明示する道具
- AbortSignal は中止伝播
- timeout は制御ポリシー
- cleanup は後始末

のように位置づけます。

### 2. 同じ場所に出るものを同じ責務として書かない

Promise 周辺では、同じコード断片の中に

- `await`
- `Promise.all`
- `AbortSignal`
- timeout
- retry
- cleanup

が同時に出てくることがあります。  
しかし、それらは別責務です。  
同じ現場に出ることと、同じ種類であることを混同しないようにしてください。

### 3. 流行語・表面的な比喩で済ませない

「キャンセル対応」「非同期の便利機能」「裏で頑張ってくれる」などの表現だけで済ませず、

- 誰が待つか
- 誰が止めるか
- 誰が失敗を拾うか
- 誰が cleanup するか

を言えるように書いてください。

## コード例の扱い

### 基本方針

コード例は使ってよいですが、主役にしません。  
コード例は、責務の違いを見せるための補助です。

### コード例を出すときの順番

必ず次の順で書いてください。

1. 何が混線しているのかを言う  
2. コード例を出す  
3. どの責務がどこに置かれているかを説明する

### コード例で避けること

- 便利そうな書き方だけを並べる
- cleanup を省略したままよい例として扱う
- signal を渡しただけで設計が終わったように見せる
- `Promise.all` を責任スコープそのものとして扱う

## 本書全体の核心

この小冊子で学び直したいのは、Promise の便利な書き方そのものではありません。  
学び直したいのは、Promise 周辺で混線しやすい責務です。

- Promise は結果待機と失敗伝播
- AbortSignal は中止伝播
- timeout / retry は制御ポリシー
- cleanup は後始末
- 責任スコープは別の設計問題

この切り分けを保ちながら書くことが、本書全体の品質を決めます。
