---
title: "第6章　Promise.all・race・timeout・retry を責務で見直す"
---

**初学者**  
ここまでで、Promise は結果待機と失敗伝播、AbortSignal は中止伝播、timeout と retry は制御ポリシー、cleanup は後始末だと整理してきました。  
でも実際のコードでは、これらが全部かなり近い場所に出てきますよね。  
だからまだ、ときどき同じ棚に見えてしまいます。

**解説者**  
その感覚は自然です。  
この章では、まさにその「近くに出てくるもの」を責務で見直します。  
対象は `Promise.all`、`Promise.allSettled`、`Promise.race`、timeout、retry です。  
どれも非同期コードではよく登場しますが、やっている仕事は同じではありません。

**実装者**  
現場で崩れやすいのもこのへんです。  
`Promise.all` でまとめたから一つの仕事になった気がする。  
`race` を使ったから timeout もキャンセルも片づいた気がする。  
retry を入れたから堅牢になった気がする。  
でも実際には、待機集約、観測、制御ポリシー、中止伝播は別々に見ないと設計が濁ります。

**初学者**  
つまりこの章は、便利な道具を否定するのではなく、どの棚の道具なのかを言い直す章なんですね。

**解説者**  
その通りです。  
似た場所に出てくるから同じ責務に見える。  
その錯覚をほどくのがこの章の役割です。

## `Promise.all` は待機集約である

**初学者**  
まず `Promise.all` ですが、これはかなりよく使います。  
複数の Promise をまとめる道具、という理解でいいですよね。

**解説者**  
はい。  
ただし「まとめる」と言うと広すぎるので、もう少し正確に言いましょう。  
`Promise.all` は、複数の Promise の待機を集約する道具です。  
全部の成功を待ち、どれか一つが失敗したら失敗として観測する。  
つまり中心にあるのは、結果待機の集約です。

**実装者**  
ここで重要なのは、`Promise.all` が仕事の所有関係や中止伝播を自動で作るわけではないことです。  
あくまで「複数の結果をどう待つか」を整える道具です。  
責任線まで定義してくれるわけではありません。

**初学者**  
`Promise.all` を使うと、なんとなく「一つの仕事の束」になった感じがします。  
でも実際には、待ち方をまとめているだけなんですね。

**解説者**  
その理解が大切です。  
`Promise.all` は強力ですが、強いのは結果待機の集約においてです。  
責任スコープそのものではありません。

## `Promise.allSettled` は観測の粒度を変える

**初学者**  
では `Promise.allSettled` はどうでしょう。  
こちらは成功と失敗を両方見られるので、より管理しやすい感じがします。

**解説者**  
`Promise.allSettled` も基本は待機集約です。  
違うのは、失敗があっても全体を失敗で打ち切らず、各 Promise の結果を成功・失敗つきで保持して観測できることです。  
つまり待ち方と見え方が変わっているのであって、責務の棚そのものが変わるわけではありません。

**実装者**  
ここを誤解すると、「`allSettled` を使えば全体管理ができた」と思い込みやすいです。  
でも実際には、観測の情報量が増えただけです。  
誰が誰の面倒を見るのか、失敗したものをどう後始末するのか、不要になったらどこまで止めるのかは、別に考えないといけません。

**初学者**  
なるほど。  
`all` は全成功を待つ集約、`allSettled` は成功と失敗を保持したまま全部を待つ集約。  
どちらもまずは「待機集約の道具」なんですね。

**解説者**  
その通りです。  
観測のしかたは違いますが、棚としてはどちらも結果待機の側にあります。

## `Promise.race` は「最初に観測されるもの」を選ぶ

**初学者**  
`Promise.race` はさらにややこしいです。  
これもよく timeout と一緒に出てきますよね。

**解説者**  
まず `Promise.race` 自体の責務を切り分けましょう。  
`Promise.race` は、複数の Promise のうち、最初に決着したものを観測する道具です。  
ここでも中心にあるのは結果待機です。  
「誰が一番先に終わるか」を観測するための待機集約だと言えます。

**実装者**  
注意したいのは、`race` は「先に見えた結果を採用する」ことと、「他を止める」ことを同時にはやらない、という点です。  
ここが実務ではかなり重要です。  
観測上は勝者が決まっても、敗者側の処理が自動で止まるとは限りません。

**初学者**  
ああ、`race` って「どちらを採用するか」を決めるだけで、「負けたほうを消す」わけではないんですね。

**解説者**  
そうです。  
この違いを見落とすと、`race` を使っただけで timeout やキャンセルまで終わった気になってしまいます。

## timeout は制御ポリシーである

**初学者**  
では timeout は、ここでどう位置づければいいですか。  
`race` と組み合わせて書かれることが多いので、同じ仲間に見えてしまいます。

**解説者**  
timeout は待機集約の道具ではありません。  
timeout は「どれだけ待つか」「どこで諦めるか」という制御ポリシーです。  
つまり結果を運ぶ道具ではなく、待機に対して外側から制約をかける判断です。

**実装者**  
たとえば「3秒経ったら諦める」という方針は、Promise の本体からは出てきません。  
それはアプリケーション側や呼び出し側が決めることです。  
通信の性質、UI の都合、ユーザー体験、システム負荷などによって、許容時間は変わるからです。

**初学者**  
`Promise.race` で timeout Promise を混ぜる例をよく見ますが、あれは「待機集約の道具」と「制御ポリシー」を組み合わせているだけなんですね。

**解説者**  
その理解で大丈夫です。  
`race` は最初に決着したものを観測する。  
timeout は、一定時間で打ち切る方針を作る。  
近くに書けても、棚は別です。

## timeout は中止伝播そのものではない

**初学者**  
timeout したら、処理は止まったと考えていいんでしょうか。

**解説者**  
そこが危ないところです。  
timeout は「もう待たない」という判断を作ります。  
けれど、それだけで下位処理が本当に止まるとは限りません。  
中止伝播は別に接続しないといけません。

**実装者**  
実務では「timeout したから安心」と思ってしまいがちです。  
でも実際には、観測だけ打ち切って、通信やイベント待機は継続していることがあります。  
この状態は、見えなくなっただけで仕事が残っているので厄介です。

**初学者**  
つまり timeout は「待つのをやめる方針」であって、「対象を止める仕組み」ではないんですね。

**解説者**  
その通りです。  
timeout を中止伝播と混同すると、設計が曖昧になります。  
timeout は制御ポリシー、AbortSignal は中止伝播。  
ここは分けて考える必要があります。

## retry は失敗処理ポリシーである

**初学者**  
retry も、非同期処理の近くに必ず出てきますよね。  
これはどの棚に置けばいいですか。

**解説者**  
retry は失敗処理ポリシーです。  
Promise が失敗を運んできたあと、その失敗に対して「再試行するか」「何回まで試すか」「どういう失敗ならやり直すか」を決める外側の方針です。  
つまり Promise の失敗伝播の上に乗る制御です。

**実装者**  
ここで大事なのは、retry が成功や失敗を作る仕組みそのものではないことです。  
失敗を受けてどう振る舞うかを決めているだけです。  
だから retry は、呼び出し側の責任やドメインの都合に依存しやすい。  
通信ならやり直せても、ユーザー操作や決済処理では慎重に扱う必要があります。

**初学者**  
たしかに retry は、「失敗したらもう一度」という単純な話ではないですね。  
どの失敗なら許されるかを判断しないといけない。

**解説者**  
そうです。  
だから retry を Promise の本体だと思ってしまうと、責務が重くなりすぎます。  
retry は失敗処理のポリシーであって、結果待機や失敗伝播そのものではありません。

## retry はキャンセルを無視できない

**初学者**  
retry は失敗処理ポリシーだとして、キャンセルとはどう関係するんでしょうか。

**解説者**  
ここはかなり重要です。  
retry は失敗に対する方針ですが、だからといって中止判断を無視してよいわけではありません。  
上位が「もう不要だ」と判断したあとまで retry が続くなら、責任線に逆らうことになります。

**実装者**  
実務でありがちなのは、失敗したので自動的に再試行し続けるけれど、画面はもう閉じられている、あるいは呼び出し元はもうその結果を必要としていない、という状態です。  
これは retry のロジックが、所有者のキャンセル判断を見ていない状態です。  
動いてはいますが、責任設計としては壊れています。

**初学者**  
なるほど。  
retry は「失敗したら機械的にやり直す」だけでは足りなくて、「この仕事はまだ所有されているか」も見ないといけないんですね。

**解説者**  
その通りです。  
retry は制御ポリシーですが、責任線の外側で暴走してはいけません。  
中止伝播と接続してはじめて、安全な retry になります。

## よくある混線 1: `race` で timeout したから全部終わったと思う

**初学者**  
ここまでの話を聞くと、よくある壊れ方も見えてきそうです。  
たとえば `Promise.race` で timeout を入れたら、それで全部終わったと思い込むのは危ないんですね。

**解説者**  
その通りです。  
それは典型的な混線です。  
`race` は最初に決着したものを観測する。  
timeout は一定時間で諦める方針を作る。  
この二つを組み合わせることはできますが、それだけで敗者側の処理の中止や cleanup は保証されません。

**実装者**  
実際には、timeout 側が勝ったあとも、元の通信が継続していることがあります。  
観測上は「もう終わった」と見えているので、気づきにくい。  
でも資源消費や状態更新の遅延などの形で後から問題になります。

**初学者**  
見えなくなったことと、消えたことは違う。  
かなり大事な区別ですね。

## よくある混線 2: `Promise.all` でまとめたから責任も一つになったと思う

**初学者**  
もう一つ気になるのは、`Promise.all` で複数の仕事をまとめたときです。  
あれで責任スコープまで一つになったと考えるのは危ないんでしょうか。

**解説者**  
危ないです。  
`Promise.all` が統一してくれるのは、待機の形です。  
責任の所有関係や中止伝播の経路、cleanup の境界までは自動ではそろいません。

**実装者**  
たとえば三つの通信を `all` で待っていても、そのうちどれが親の中止判断に従うのか、どれが部分失敗に耐えるのか、どこで片づけるのかは別です。  
同じ配列に入れたからといって、同じ責任線にいるとは限りません。

**初学者**  
`all` は「同じ箱に入れた」のであって、「同じ親のもとに整理した」とは限らないんですね。

**解説者**  
その理解が大切です。  
待機集約と責任スコープは別の設計問題です。

## 似た場所に出るものを同じ棚に入れない

**初学者**  
だいぶ整理できてきました。  
でも結局、この章の一番大事な教訓は何でしょうか。

**解説者**  
一言で言えば、「似た場所に出るものを同じ棚に入れない」です。  
`Promise.all`、`allSettled`、`race` は待機集約や観測の道具です。  
timeout と retry は制御ポリシーです。  
AbortSignal は中止伝播です。  
cleanup は後始末です。  
近い場所に出るからこそ、責務の地図が必要になります。

**実装者**  
実務では便利な書き方ほど、責務を隠します。  
一行で書けるほど、全部同じものに見えてしまう。  
だからこそ、「これは何をしているのか」を API 名ではなく責務で言い直せるかどうかが重要です。

**初学者**  
便利な道具を覚えるだけではなく、その道具がどの棚に属しているかを言えるようにする。  
それがこの章のポイントなんですね。

**解説者**  
その通りです。  
棚が見えれば、コードが長くなっても設計は見失いにくくなります。

### 整理

- `Promise.all` は複数の Promise の待機を集約する道具である  
- `Promise.allSettled` は成功と失敗を保持したまま全部を観測する待機集約である  
- `Promise.race` は最初に決着したものを観測する待機集約である  
- timeout は「どこで諦めるか」を決める制御ポリシーであり、中止伝播そのものではない  
- retry は「失敗にどう対応するか」を決める失敗処理ポリシーである  
- retry は中止判断を無視してよいわけではなく、責任線と接続される必要がある  
- 待機集約、制御ポリシー、中止伝播、cleanup は近くに出てきても別責務である  

### この章の核心

`Promise.all`、`Promise.allSettled`、`Promise.race` は、どれも結果待機や観測の道具です。  
一方で timeout と retry は、外側から待機や失敗に方針を与える制御ポリシーです。  
似た場所に出てくるからといって同じ棚に入れてしまうと、待機集約、中止伝播、責任スコープ、cleanup の違いが見えなくなります。

## まとめ

- `Promise.all` は待機集約である  
- `Promise.allSettled` も成功と失敗を保持した待機集約である  
- `Promise.race` は最初に決着したものを観測する待機集約である  
- timeout は制御ポリシーであり、中止伝播そのものではない  
- retry は失敗処理ポリシーであり、Promise の本体責務ではない  
- retry はキャンセルや所有者の判断を無視してはいけない  
- 待機集約、制御ポリシー、中止伝播、cleanup は別責務である  
- 似た場所に出るものを同じ棚に入れないことが重要である  

次章では、ここまでの責務マップを使って、実際に「動くのに壊れる」Promise 設計を読み解きます。  
signal を渡していない、timeout だけあって cleanup がない、retry がキャンセルを無視する。  
そうした実務の破綻を、便利 API 不足ではなく責務混線として読めるようにしていきます。
