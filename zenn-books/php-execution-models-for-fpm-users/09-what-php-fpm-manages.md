title: "php-fpm は何を管理しているのか"
---

前の章では、FastCGI が CGI の「外に渡す」という構図を残しながら、毎回起動するつらさを減らし、Web サーバーとアプリケーション実行側を分けたまま実用的に運用できるようにしたことを見ました。

その流れの先にあるのが **php-fpm** です。  
今ではかなり当たり前の構成として扱われますが、ただ「FastCGI で動く PHP」くらいの理解で済ませると、本書の核心が見えにくくなります。

php-fpm の重要さは、単に PHP を FastCGI の向こう側で動かすことではありません。  
**分離された PHP 実行側を、常駐プロセスとしてどう待機させ、どう管理し、どう運用可能な単位にするか** を引き受けているところにあります。

この章では、php-fpm が何を実行しているのかではなく、**何を管理しているのか** に焦点を当てます。

## php-fpm は「PHP そのもの」ではなく「PHP 実行側の管理者」である

**新人**  
php-fpm って、普段は「PHP を動かしているもの」くらいの理解でした。  
でもここまでの流れを読むと、それだけでは足りなさそうですね。

**ベテラン**  
足りません。  
もちろん PHP を実行する側にいるのはその通りです。  
でも本書の観点では、php-fpm を単なる実行器として見るより、**PHP 実行側を管理する仕組み** として見るほうが重要です。

**新人**  
管理する仕組み。  
つまり、FastCGI の向こう側にいる PHP のまとまりを面倒見ているわけですね。

**ベテラン**  
その理解でよいです。  
Web サーバーは HTTP の受け口として振る舞い、FastCGI を通じて PHP 側へ仕事を渡す。  
そのとき、PHP 側で待機している実行プロセスをどう持つか、どう増減させるか、どう動かし続けるか。  
そこを引き受けているのが php-fpm です。

**運用の人**  
現場感覚でも、php-fpm は「PHP が動く場所」というだけではなく、「PHP 実行プロセスを運用しやすいまとまりにしているもの」と見たほうがしっくりきますね。

**新人**  
なるほど。  
FastCGI が作った「外に分ける構図」の先で、その分けた PHP 側をどう扱うかが php-fpm なんですね。

## 「待機している PHP 実行側」をどう扱うか

**新人**  
前の章では、「あらかじめ待機している実行側に処理を渡す」という話がありました。  
php-fpm は、その待機の部分を管理しているんですか。

**ベテラン**  
その通りです。  
CGI のように毎回新しく起動するのではなく、あらかじめ動いている PHP 側に仕事を渡したい。  
でも、その「動いている PHP 側」を放っておくだけでは実務になりません。

**新人**  
たしかに。  
何個待たせるのか、どう動かすのか、止まったらどうするのか、いろいろありそうです。

**ベテラン**  
まさにそこが php-fpm の担当範囲です。  
本書では細かな設定値には入りませんが、少なくとも次の感覚は持っておきたいです。

- PHP 実行側を常駐させる
- 複数の実行プロセスを持てるようにする
- それらをまとまりとして調整できるようにする
- Web サーバーから渡された仕事を受けられるようにする

**運用の人**  
現場では、この「待たせ方」をどう設計するかがかなり重要です。  
少なすぎれば詰まりやすい。  
多すぎれば無駄が出る。  
しかもそれは Web サーバー側の設定とは別の単位で見たい。  
だから php-fpm のような管理層が効いてきます。

**新人**  
なるほど。  
FastCGI が「外に分ける」なら、php-fpm は「分けた先をどう待たせるか」を担当しているんですね。

**ベテラン**  
はい。  
そして、そのことが Web サーバーと PHP 実行側の責務分離を、概念だけでなく運用可能な構成にしています。

## プール管理とは何をしているのか

**新人**  
php-fpm の話になると、よく「プール」という言葉が出てきます。  
これはどう理解すればいいですか。

**ベテラン**  
本書の範囲では、**待機している PHP 実行プロセスを、あるまとまりとして管理する考え方** と捉えるのがよいです。

**新人**  
まとまりとして管理する。

**ベテラン**  
はい。  
ただ 1 個の実行プロセスを動かすのではなく、複数の実行プロセスをどう持つか、どの単位で扱うか、というまとまりを作るわけです。  
そのまとまりがあることで、PHP 実行側を運用しやすい対象として見られるようになります。

**新人**  
つまり、PHP を 1 回ずつ起動する世界ではなく、待機中の実行プロセス群を管理する世界なんですね。

**ベテラン**  
その通りです。  
ここで重要なのは、プールという言葉を設定項目として暗記することではありません。  
本質は、**PHP 実行側を、調整可能で管理可能な単位にしている** ところにあります。

**運用の人**  
現場でも、これがかなり大きいです。  
Web サーバーの受け口と、PHP 実行側のまとまりを別単位で見られるからこそ、構成や監視や調整の視点が整理しやすくなります。

**新人**  
なるほど。  
FastCGI が境界を作り、php-fpm がその境界の向こう側を管理しやすい形に整えているんですね。

## Web サーバーとの役割分担はどう見ればよいか

**新人**  
では、Nginx や Apache と php-fpm の関係は、どう整理すればいいですか。  
普段は一緒に使っているので、まだ少し混ざります。

**ベテラン**  
ここも本書の重要ポイントです。  
まず、Web サーバーは **HTTP の受け口** です。  
接続を受け、リクエストを読み、必要なら静的な応答を返し、PHP に渡すべきものを渡します。

一方で php-fpm は、**PHP 実行側の待機と管理** を担います。  
受け取った仕事を、待機している PHP 実行プロセスに割り振って処理させる側だと見ればよいです。

**新人**  
なるほど。  
Web サーバーは受け口。  
php-fpm は処理役を待たせて管理する側。  
かなり分かれてきました。

**ベテラン**  
そうです。  
ここで大事なのは、php-fpm を「もう一つの Web サーバー」のように見ないことです。  
また逆に、Web サーバーが PHP 実行側の管理までやっていると考えないことです。

**運用の人**  
現場でも、この分け方ができるとトラブルシュートしやすいですね。  
HTTP の入口で詰まっているのか。  
FastCGI の受け渡しで困っているのか。  
PHP 実行側の待機数や処理側で詰まっているのか。  
見る場所を分けやすくなります。

**新人**  
確かに、普段「サーバーが重い」と一言で言ってしまいがちですけど、本当は層ごとに意味が違いますね。

## php-fpm があると何が運用しやすいのか

**新人**  
php-fpm の役割は見えてきました。  
でも、現場で「運用しやすい」とは具体的に何が違うんでしょう。

**ベテラン**  
一言でいえば、**PHP 実行側を独立した管理対象として扱えること** です。

**新人**  
独立した管理対象。

**ベテラン**  
はい。  
mod_php のように Web サーバーの中に PHP がいると、HTTP の受け口と PHP の実行がかなり近い場所にあります。  
php-fpm では、PHP 実行側が別の単位として見えるので、

- PHP 実行側の数を考える
- PHP 実行側の状態を監視する
- PHP 実行側だけを再読込や再起動の対象として考える
- Web サーバー側の調整と PHP 側の調整を分ける

といった発想を取りやすくなります。

**運用の人**  
ここは本当に大きいです。  
「Web サーバー」と「PHP の実行プロセス」を別々に見られると、運用上の問いがかなり整理されます。  
何を増やすのか。  
どこを監視するのか。  
どこを触るのか。  
その単位が明確になりますから。

**新人**  
つまり php-fpm は、ただ速くするための部品ではなく、PHP 実行側を人間が扱いやすい形にしているわけですね。

**ベテラン**  
その理解が大事です。  
本書では、そこを「責務分離の運用可能化」として見てほしいんです。

### 整理

php-fpm は、FastCGI の向こう側にある PHP 実行側を、**常駐プロセスとして待機させ、管理する仕組み** です。  
単に PHP コードを実行するだけでなく、**PHP 実行側を運用可能な単位として整えている** ところに意味があります。

この章で押さえたい役割は次の通りです。

- PHP 実行側を常駐させる
- 複数の実行プロセスを持てるようにする
- それらをまとまりとして管理する
- Web サーバーから渡された仕事を処理できるようにする
- PHP 実行側を独立した管理対象として扱えるようにする

また、Web サーバーとの役割分担は次のように整理できます。

- Web サーバー  
  HTTP の受け口として接続やリクエストを扱う

- php-fpm  
  FastCGI の向こう側で PHP 実行プロセスを待機・管理し、処理を担当する

ここで重要なのは、php-fpm の価値が単なる高速化ではないことです。  
php-fpm は、**分離された PHP 実行側を、人間が運用しやすい形で管理する仕組み** なのです。

### この章の核心

php-fpm は、FastCGI の先にある PHP 実行側を常駐プロセスとして待機させ、まとまりとして管理する仕組みです。  
その本質は、PHP を実行することそのものより、分離された PHP 実行側を運用可能な単位として成立させていることにあります。

## まとめ

- php-fpm は、FastCGI の向こう側にある PHP 実行側を管理する仕組みである
- 重要なのは、単に PHP を動かすことではなく、待機中の PHP 実行プロセスを常駐・管理することにある
- プール管理は、PHP 実行側をまとまりとして調整・運用しやすくする考え方である
- Web サーバーは HTTP の受け口、php-fpm は PHP 実行側の待機と管理という分担で考えると整理しやすい
- php-fpm の価値は、責務分離を概念だけでなく運用可能な構成にしている点にある

次章では、プロセス・スレッド・接続待機の基礎を整理します。  
ここまでに出てきた Web サーバー側と PHP 実行側の役割分担を、より土台の概念から見直すことで、なぜこの分離が自然だったのかをさらにはっきりさせていきます。
