---
title: "はじめに — PHP の知識と PHP の実行形態の知識は別である"
---

PHP を学ぶとき、多くの人は文法、フレームワーク、ライブラリ、データベース接続といった話から入ります。  
それ自体は自然なことです。実務でも、最初に触れるのはアプリケーションのコードだからです。

ですが、PHP を長く使っていると、別の種類の疑問が出てきます。  
なぜ Nginx の後ろで php-fpm を動かすのか。  
mod_php とは何が違うのか。  
CGI は何が古くて、FastCGI は何を変えたのか。  
Apache MPM という言葉は見かけるのに、それが PHP の話なのか Web サーバーの話なのか、うまく説明できない。

この本は、そうした疑問に答えるための本です。  
昔の方式を懐かしむためでも、用語を暗記するためでもありません。  
今ふつうに使っている php-fpm 周辺の構成を、歴史と責務分解の結果として理解するための本です。

## いま動いている PHP は、いつも同じ形ではない

**新人**  
PHP って、PHP は PHP ですよね。  
同じコードを動かしているんだから、そんなに実行形態の違いを気にしなくてもいい気がします。

**ベテラン**  
その感覚自体は自然です。  
実際、アプリケーションを書く段階では、同じ PHP のコードを書いているように見えますからね。

でも、ここで一度分けて考えたほうがいいです。  
PHP の文法やライブラリの知識と、PHP がどんな形で実行されているかの知識は、同じではありません。

**新人**  
同じではない、ですか。

**ベテラン**  
はい。  
たとえば CLI で `php script.php` と実行する PHP と、Web サーバー経由でリクエストを受けて動く PHP では、見えているものが違います。  
さらに、Web サーバー経由で動く PHP でも、Apache の中に組み込まれて動く場合と、外部のプロセスとして待機して動く場合では、構成も責務分担も変わります。

**運用の人**  
現場感覚でいうと、そこを知らなくても日常運用はできます。  
でも、トラブル対応や構成変更になると急に必要になりますね。

たとえば、

「これって Apache 側の話ですか」  
「php-fpm のプール設定の話ですか」  
「そもそも PHP をどこで実行しているんですか」

みたいな切り分けが必要になります。

**新人**  
ああ、たしかに。  
エラーが出たときに、Nginx の設定を見るべきなのか、php-fpm の設定を見るべきなのか、PHP コードを見るべきなのかで迷うことがあります。

**ベテラン**  
そうです。  
この本でやりたいのは、まさにその迷いを減らすことです。  
PHP の実行形態を知るというのは、単に昔の名前を覚えることではありません。  
誰が何を担当しているのかを見分ける地図を持つことです。

## なぜ今さら CGI や mod_php の話を学ぶのか

**新人**  
でも、今は php-fpm が普通ですよね。  
だったら php-fpm だけ学べばよくないですか。  
CGI とか mod_php とか、昔の話を知らなくても困らない気がします。

**ベテラン**  
それももっともな疑問です。  
ただ、php-fpm がなぜ自然な構成として広まったのかは、前の方式と比べないと見えにくいんです。

**新人**  
前の方式と比べると見える、ですか。

**ベテラン**  
はい。  
たとえば CGI は、仕組みとしてはかなり分かりやすいです。  
でも、その分かりやすさには大きなコストがありました。  
mod_php は非常に便利でしたが、その便利さはある前提の上で成り立っていました。  
その前提が変わると、別の構成のほうが自然になります。

つまり、  
CGI が何に困っていたのか。  
mod_php がなぜ強かったのか。  
mod_php がなぜ主流でなくなったのか。  
FastCGI が何を分離したのか。  
そこを順に見ると、php-fpm の位置づけが分かるんです。

**運用の人**  
現場でも同じです。  
今の構成だけ見ていると、「そういうものだから」で終わりがちです。  
でも歴史を少し知ると、「なぜ Web サーバーと PHP を分けるのか」が説明しやすくなります。

**新人**  
なるほど。  
昔の方式を知ることが目的じゃなくて、今の方式がどういう整理の結果なのかを見るためなんですね。

**ベテラン**  
その理解でいいです。  
この本は、昔の方式の雑学集ではありません。  
現在の php-fpm 周辺を理解するために、必要なだけ過去へ戻る本です。

## この本で区別したいこと

**新人**  
さっきから「分けて考える」と何度も言っていますけど、特にどこを分けるんですか。

**ベテラン**  
まずは大きく三つあります。

一つ目は、実行方式の違いです。  
CGI、mod_php、FastCGI、php-fpm は似た話題として語られますが、同じものではありません。

二つ目は、Web サーバー側の話と PHP 側の話です。  
Apache MPM のような話は Web サーバー側の並行処理の話であって、PHP の実行形態そのものとは別です。

三つ目は、接続を待つことと、PHP のコードを実行することの違いです。  
この二つが混ざると、構成を理解しにくくなります。

**新人**  
ちょっとずつ見えてきました。  
今まで私の頭の中では、「Apache」「Nginx」「php-fpm」「worker」「プロセス」みたいな言葉が、全部なんとなく同じ箱に入っていました。

**運用の人**  
そこは多くの人が一度は通るところですね。  
現場だと、設定ファイルの場所も別ですし、監視の見方も別ですし、ログの出方も別です。  
でも概念の地図がないと、全部が「サーバーまわりの難しい話」に見えてしまいます。

**ベテラン**  
この本では、その箱をいったん分け直します。  
どの言葉がどの層の話なのか。  
誰が接続を受け、誰が PHP を実行し、誰がその実行プロセスを管理しているのか。  
そこを区別しながら進めます。

## この本の読み方

**新人**  
読むときのコツはありますか。

**ベテラン**  
あります。  
まず、この本を「最強の構成を選ぶ本」だと思わないことです。  
構成の優劣を単純に決める話ではありません。  
どの方式が、どの時代の、どの条件で自然だったのかを理解する本です。

次に、用語を急いで暗記しようとしないことです。  
大事なのは名前を覚えることではなく、責務の違いをつかむことです。

最後に、歴史順の説明を昔話として読まないことです。  
この本では、歴史は設計比較の材料です。  
「なぜ次の方式が必要になったのか」を見るために、前の方式を学びます。

**運用の人**  
実務の感覚でいうと、  
「いま使っている構成を他人に説明できるようになる」  
という読み方が合っています。

**新人**  
たしかに、それなら目的がはっきりしていますね。  
知識を増やすというより、説明できるようになるために読む感じです。

**ベテラン**  
そうです。  
この本のゴールは、php-fpm を中心とした現在の構成を、歴史と責務分解の地図として説明できるようになることです。

### 整理

この章では、次の点を確認しました。

- PHP の文法やライブラリの知識と、PHP の実行形態の知識は別である
- php-fpm を理解するには、CGI や mod_php との比較が役に立つ
- 本書の目的は昔の雑学ではなく、現在の構成を歴史と責務分解の地図として理解することにある
- 特に、Web サーバー側の話と PHP 実行側の話を混同しないことが重要である
- まずは SAPI を入口として、PHP の実行形態の見取り図をつくる必要がある

### この章の核心

PHP を理解するとは、コードの書き方だけを知ることではありません。  
どのような形で PHP が実行され、誰が何を担当しているのかを区別できてはじめて、現在の構成を説明できるようになります。

## まとめ

- この本は、php-fpm 周辺を昔の雑学ではなく、現在の実行形態を理解するための地図として説明する
- CGI、mod_php、FastCGI、php-fpm の比較は、今の構成を理解するために必要である
- Apache MPM のような Web サーバー側の話と、PHP の実行形態の話は分けて考える必要がある
- 用語の暗記よりも、責務の違いをつかむことが重要である
- 次章では SAPI を入口にして、PHP の実行形態の見取り図を整理する

次章では、PHP がどのような形で実行されるのかを、SAPI という地図から見ていきます。  
ここで全体像をつかんでおくと、その後の CGI、mod_php、FastCGI、php-fpm の比較が読みやすくなります。
