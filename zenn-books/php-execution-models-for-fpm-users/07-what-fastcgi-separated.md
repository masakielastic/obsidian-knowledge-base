---
title: "FastCGI は何を分離したのか"
---

前の章では、Apache MPM が Web サーバー側の並行処理モデルであり、PHP の実行形態そのものではないことを整理しました。  
この区別が見えてくると、mod_php のように Apache の中で PHP を動かす構成が、なぜ便利でもあり、同時に分離しにくくもあったのかがわかりやすくなります。

ここで次に見るのが **FastCGI** です。  
FastCGI は、単に CGI を少し速くした仕組み、と雑に理解されることがあります。  
ですが、本書ではそうは扱いません。  
FastCGI の本質は、**毎回起動するつらさを減らしたこと** だけでなく、**Web サーバーとアプリケーション実行側の責務を分けやすくしたこと** にあります。

この章では、その「分離」が何を意味していたのかを整理します。

## CGI の何を引き継ぎ、何を変えたのか

**新人**  
FastCGI という名前を見ると、まず「CGI を高速化したもの」という印象があります。  
それでだいたい合っていますか。

**ベテラン**  
半分は合っています。  
たしかに CGI の問題を改善しようとした流れです。  
ただ、「速くなった」という言い方だけだと、本質を見落としやすいんです。

**新人**  
本質は別のところにもあるんですね。

**ベテラン**  
そうです。  
まず CGI の何が良かったかを思い出してください。  
Web サーバーがリクエストを受けて、外部プログラムに処理を渡す。  
この構図そのものは、とてもわかりやすかったですね。

**新人**  
はい。  
Web サーバーとアプリケーションの境界が見えやすい、という話でした。

**ベテラン**  
その通りです。  
FastCGI は、この **外に渡す** という発想を引き継ぎます。  
ただし、CGI のように **毎回新しく起動する** のではなく、あらかじめ動いている実行側に処理を渡す方向へ進みます。

**新人**  
なるほど。  
外にある、という構図は残す。  
でも、毎回ゼロから起動するのはやめる。  
そういう改善なんですね。

**運用の人**  
現場感覚でも、それはかなり自然です。  
CGI の良さは責務の境界が見えやすいことでした。  
その良さは残したまま、繰り返しに弱いところだけ何とかしたい。  
FastCGI は、その方向の整理として見るとわかりやすいですね。

## 「外に置く」ことの意味

**新人**  
でも、ここで改めて聞きたいです。  
どうして「外に置く」ことがそんなに重要なんでしょう。  
mod_php のように中で動かせるなら、そのほうが短くて便利にも見えます。

**ベテラン**  
そこが分離の核心です。  
Web サーバーの役割と、アプリケーションを実行する役割は、本来は別のものとして考えやすいんです。

**新人**  
Web サーバーは HTTP を受ける。  
アプリケーション実行側は PHP を動かす。  
そういう分け方ですね。

**ベテラン**  
はい。  
CGI では、その境界はかなり見えやすかった。  
ただし毎回起動するのがつらかった。  
mod_php では毎回起動するつらさは減りましたが、今度は Web サーバーと PHP 実行側が近すぎました。

FastCGI は、その中間ではありません。  
むしろ、**境界を保ったまま、実用性を上げる方向** だと見るほうが正確です。

**新人**  
境界を保ったまま実用性を上げる。  
それはかなり本書のテーマに合っていますね。

**運用の人**  
現場でも、その差は大きいです。  
Web サーバーは受け口として動く。  
PHP の実行側は別で待機していて処理を受ける。  
この形だと、どちらが何を担当しているかを分けて見やすいです。

**新人**  
なるほど。  
FastCGI は、「PHP を外に出す」こと自体が重要なんですね。  
単に高速化のためだけじゃなくて。

**ベテラン**  
その通りです。

## 毎回起動しない、とはどういうことか

**新人**  
では、CGI との違いをもう少しはっきりさせたいです。  
「毎回起動しない」というのは、実際にはどういう見え方になるんですか。

**ベテラン**  
CGI では、リクエストが来るたびに、そのための実行を新しく始めていました。  
FastCGI では、すでに動いている実行側が待っていて、そこへ仕事を渡すイメージになります。

**新人**  
つまり、処理役が先に待機しているんですね。

**ベテラン**  
そうです。  
だから、毎回ゼロから起動するコストを減らしやすくなります。  
繰り返しの多い Web では、これは大きいです。

**運用の人**  
現場感覚で言うと、  
「受けたらそのたびに新しい作業者を呼ぶ」  
のではなく、  
「あらかじめ作業者を待機させておいて、仕事を回す」  
感じですね。

**新人**  
その比喩はかなりわかりやすいです。  
しかも作業者が Web サーバーの中ではなく、外にいるわけですね。

**ベテラン**  
そうです。  
ここが重要です。  
FastCGI は、**外に置いたまま常駐させる** 方向です。  
CGI のように外に渡す構図を保ちながら、毎回起動するつらさを減らしています。

## 分離されたのは、速度だけではない

**新人**  
ここまで聞くと、FastCGI はかなり筋がいい改善に見えます。  
では「何を分離したのか」と言われたら、どう答えればいいですか。

**ベテラン**  
まず一番大きいのは、**Web サーバーの責務とアプリケーション実行側の責務** です。

**新人**  
Web サーバーは HTTP を受ける。  
FastCGI の向こう側は PHP を実行する。  
そういう分け方ですね。

**ベテラン**  
はい。  
mod_php では、この二つがかなり近い場所で一体化していました。  
FastCGI では、構成としてそこを切り離しやすくなります。

もう一つ重要なのは、**接続の受け口とアプリケーション処理の単位** を分けやすくなることです。  
Web サーバーは多数の接続を受けることに集中し、PHP 実行側はアプリケーションの処理に集中しやすくなります。

**新人**  
なるほど。  
Web サーバーが全部抱えるのではなく、受け口と処理役を別々に考えやすくなるんですね。

**運用の人**  
現場ではその差が大きいです。  
HTTP の入口としての調整と、PHP の実行数や待機数の調整を、別の単位として考えやすくなりますから。

**新人**  
それは確かに、単なる高速化では言い切れないですね。  
構成の考え方そのものが変わっています。

**ベテラン**  
その通りです。  
FastCGI の重要さは、実行を外に出したことそのものより、**外に出したまま実用的に運用できる形を作ったこと** にあります。

## Web サーバーを Web サーバーとして扱いやすくなる

**新人**  
FastCGI の分離が効いてくると、Web サーバー側の見え方も変わりますか。

**ベテラン**  
大きく変わります。  
Web サーバーを、より純粋に **HTTP の受け口** として考えやすくなります。

**新人**  
mod_php だと、受け口でもあり、PHP の実行場所でもあったんですよね。

**ベテラン**  
そうです。  
だから Apache の並行処理モデルの話と PHP の実行の話が近づきやすかった。  
FastCGI では、Web サーバーは受けて渡す側、PHP 実行側は受け取って処理する側、と構図を分けやすくなります。

**運用の人**  
その結果、Web サーバーの選択や構成も、PHP 実行側とは別に考えやすくなります。  
HTTP の受け口として何を使うか。  
PHP 実行側をどう待機させるか。  
この二つを別々の視点で見られるのは大きいです。

**新人**  
ここでやっと、「Web サーバーと PHP を分ける」という言葉の重みがわかってきました。  
単に別プロセスにする、というだけじゃないんですね。

**ベテラン**  
そうです。  
責務の境界を、構成として見える形にする、という意味があります。

## FastCGI は完成形そのものではない

**新人**  
ここまで聞くと、FastCGI がかなり理想的に見えます。  
では、これで全部解決したんでしょうか。

**ベテラン**  
そこは一歩止まって考えたいところです。  
FastCGI は非常に重要な整理ですが、それ自体がそのまま「完成形」というわけではありません。

**新人**  
どういう意味ですか。

**ベテラン**  
FastCGI は、Web サーバーとアプリケーション実行側を分けるための重要な仕組みです。  
ただし、その先で **PHP 実行側をどう管理するか** という問題は、まだ別に残ります。

**新人**  
ああ、確かに。  
外に置くことと、その外にいる PHP をどう運用するかは別問題ですね。

**運用の人**  
現場ではそこが重要です。  
待機している PHP 実行側を、何個持つのか。  
どう管理するのか。  
どう再起動するのか。  
どう監視するのか。  
そういう実務的な問いは、FastCGI のその先にあります。

**新人**  
なるほど。  
FastCGI は「分離の構図」を作る。  
でも、その分離された PHP 側をどう管理するかは、さらに次の段階なんですね。

**ベテラン**  
その理解で正しいです。  
だから次章では、php-fpm がその管理をどう引き受けるのかを見ていきます。

## FastCGI をどう位置づければよいか

**新人**  
この章の内容を短く言うと、FastCGI はどんな存在なんでしょう。

**ベテラン**  
こう言うのがよいでしょう。  
FastCGI は、CGI の「外に渡す」という発想を保ちながら、毎回起動するつらさを減らし、Web サーバーとアプリケーション実行側の責務分離を実用的な形にした仕組みです。

**新人**  
かなり整理されました。  
CGI の延長線上にあるけれど、単なる高速化ではない。  
責務分離を現実的にした仕組みなんですね。

**ベテラン**  
そうです。  
この理解があると、php-fpm も「なんとなく今の標準」ではなく、FastCGI の上で PHP 実行側の管理を担うものとして読めるようになります。

### 整理

FastCGI は、CGI の「Web サーバーが外部の実行側に処理を渡す」という発想を引き継ぎながら、**毎回新しく起動するモデル** をやめ、**あらかじめ待機している実行側に処理を渡す** 方向へ進めた仕組みです。

この変化によって、次の分離が見えやすくなります。

- Web サーバーの責務と、アプリケーション実行側の責務
- HTTP の受け口と、PHP の実行処理
- 接続をさばく側と、アプリケーション処理を担う側

重要なのは、FastCGI の価値が単なる高速化ではないことです。  
FastCGI は、**外に分けるという構図を保ったまま、実用的に運用できる形を作った** ところに意味があります。

一方で、FastCGI は「分離の構図」を与えるものであって、PHP 実行側の管理そのものをすべて解決するわけではありません。  
その管理の問題は、次に見る php-fpm へとつながります。

### この章の核心

FastCGI は、CGI のわかりやすい責務分離を保ちながら、毎回起動するつらさを減らし、Web サーバーと PHP 実行側を現実的に分けて運用できる形にしました。  
その本質は、単なる高速化ではなく、責務分離を実用的な構成として成立させたことにあります。

## まとめ

- FastCGI は、CGI の「外に渡す」という発想を引き継いでいる
- CGI との大きな違いは、毎回新しく起動せず、待機している実行側に処理を渡す点にある
- FastCGI の価値は、単なる高速化ではなく、Web サーバーとアプリケーション実行側の責務分離を実用化したことにある
- これにより、HTTP の受け口と PHP の実行処理を別々の単位として考えやすくなる
- ただし、FastCGI は PHP 実行側の管理そのものを完結させるわけではなく、その先に php-fpm の役割がある

次章では、php-fpm を取り上げます。  
FastCGI の上で、PHP 実行側をどのように待機させ、管理し、運用しやすい単位として扱うのかを整理していきます。
