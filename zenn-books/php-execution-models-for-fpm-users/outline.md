# PHP-FPMユーザー向け基礎編 アウトライン改訂案

本アウトラインは、  
php-fpm 周辺の知識を **「昔の雑学」ではなく「現在の実行形態を理解するための責務と歴史の地図」** として理解させる、という執筆ガイドラインの方針に沿って再構成したものです。

また、企画案で示されていた  
CGI / mod_php / FastCGI / php-fpm の比較、Apache MPM と PHP 実行形態の区別、そして「なぜ php-fpm は自然な選択だったのか」という到達点を保ちつつ、章の粒度を揃えやすい形に圧縮しています。

## この改訂案の基本方針

- 1章につき1つの中心命題に寄せる
- 「地図」「歴史比較」「責務分解」「総回収」の流れを明確にする
- Apache MPM と PHP 実行形態の混同を防ぐ
- async ランタイムの詳細には踏み込みすぎない
- 第2冊への橋渡しは行うが、本冊の主題は php-fpm の理解に固定する

## 推奨フォルダ名

`php-execution-models-for-fpm-users`

## 推奨章構成

### 01-introduction.md

# はじめに — PHP の知識と PHP の実行形態の知識は別である

**中心命題**  
PHP を理解するには、文法やフレームワークだけでなく、どのような形で実行されるのかという見取り図が必要である。

**この章で扱うこと**

- なぜ今さら CGI や mod_php の話を学ぶのか
- 「普段の運用」と「実行モデルの理解」は別物であること
- 本書が歴史の雑学ではなく、現在の構成を理解するための地図であること

**章の役割**

- 本書の目的を明確にする
- 読者の問題意識を立てる
- 次章の SAPI の地図につなぐ

### 02-sapi-and-execution-models.md

# PHP はどう実行されるのか — SAPI と実行形態の見取り図

**中心命題**  
SAPI は、PHP がどの環境でどのように実行されるかを整理するための入口である。

**この章で扱うこと**

- `apache2handler`
- `cgi-fcgi`
- `fpm-fcgi`
- `cli`
- `embed`

**章の役割**

- 本書全体の基礎地図をつくる
- 「Web サーバーの中で動く」「外で動く」という違いを示す
- 後続の CGI / mod_php / php-fpm 比較の前提を揃える

### 03-why-cgi-was-simple-and-painful.md

# CGI はなぜわかりやすく、なぜつらかったのか

**中心命題**  
CGI は単純で理解しやすいが、その単純さはリクエストごとの起動コストという大きな代償を伴っていた。

**この章で扱うこと**

- CGI の基本動作
- 1リクエスト1プロセスというモデル
- 分かりやすさと限界
- FastCGI が必要になった背景

**章の役割**

- 最初期の素朴な実行形態を説明する
- なぜ改善が必要になったのかを読者に納得させる
- 次章以降の比較の土台をつくる

### 04-why-mod-php-was-so-convenient.md

# mod_php はなぜ便利だったのか

**中心命題**  
mod_php は、当時の Apache と PHP の組み合わせとして非常に自然で、導入も理解もしやすい方式だった。

**この章で扱うこと**

- Apache の中で PHP を動かす意味
- 当時の実用性
- 導入の容易さ
- なぜ広く普及したのか

**章の役割**

- mod_php を単なる旧方式として扱わない
- 強かった理由を先にきちんと説明する
- 次章の「限界」の説明に説得力を持たせる

### 05-why-mod-php-hit-its-limits.md

# mod_php はなぜ限界を迎えたのか

**中心命題**  
mod_php の限界は、責務分離の必要性と、Web サーバー側の並行処理モデルとの緊張関係を露出させた。

**この章で扱うこと**

- 責務分離の必要
- 運用上の都合
- MPM との関係
- サーバー構成の変化
- 主流交代の理由

**章の役割**

- 「mod_php が悪かった」のではなく「整理しきれなくなった」と説明する
- php-fpm への移行が自然だった理由を準備する
- Apache MPM の章へつなぐ

### 06-apache-mpm-and-the-web-server-side.md

# Apache MPM とは何か — prefork・worker・event はどの層の話か

**中心命題**  
Apache MPM は Web サーバー側の並行処理モデルの話であり、PHP の実行形態そのものではない。

**この章で扱うこと**

- prefork
- worker
- event
- プロセスとスレッドの基本的な見え方
- Apache 側の責務

**章の役割**

- Apache MPM と PHP 実行形態の混同を防ぐ
- Web サーバー側と PHP 側の層を分ける
- mod_php / php-fpm 比較の精度を上げる

### 07-what-fastcgi-separated.md

# FastCGI は何を分離したのか

**中心命題**  
FastCGI の本質は、CGI の改善だけでなく、Web サーバーとアプリケーション実行側の責務を分離した点にある。

**この章で扱うこと**

- CGI との違い
- 常駐プロセス化の意味
- 通信相手として PHP を外に出す考え方
- Web サーバーとの役割分担

**章の役割**

- 歴史の変化を責務分解として理解させる
- php-fpm の位置づけの前提をつくる
- 「PHP を外に分ける」とは何かを言語化する

### 08-what-php-fpm-manages.md

# php-fpm は何を管理しているのか

**中心命題**  
php-fpm は、PHP を FastCGI として動かすための実行側管理者であり、プロセス管理とプール管理を担っている。

**この章で扱うこと**

- php-fpm の基本的な役割
- プール管理
- ワーカープロセス
- 常駐プロセスとしての見え方
- Web サーバーとの責務分担

**章の役割**

- php-fpm ユーザーの現場感覚と接続する
- 設定の細部ではなく、何を担当しているのかを整理する
- 本書の中心対象を明確に回収する

**中心命題**  
プロセス、スレッド、接続待機の区別を理解すると、なぜ Web サーバー側と PHP 側で責務分担が必要だったのかが見えやすくなる。

**この章で扱うこと**

- プロセスとスレッドの違い
- 接続待機とアプリケーション処理の違い

### 10-review-why-php-fpm-made-sense.md

# まとめ — なぜ php-fpm は自然な選択だったのか

**中心命題**  
php-fpm は偶然広まったのではなく、CGI・mod_php・Apache MPM・FastCGI の流れを経て、責務分離の観点から自然に選ばれた構成だった。

**この章で扱うこと**

**章の役割**

## このアウトラインの狙い

この改訂案の狙いは、  
CGI → mod_php → FastCGI → php-fpm という流れを、単なる歴史順ではなく、**責務分解と実行モデルの整理の流れ** として読めるようにすることです。執筆ガイドラインでも、その観点が本書の最重要方針として示されています。

そのため、本書は async 本の前日譚ではありますが、主題はあくまで **php-fpm ユーザーが現在の構成を説明できるようになること** に置いています。企画案の狙いとも整合しています。
