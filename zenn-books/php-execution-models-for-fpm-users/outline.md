# PHP-FPMユーザー向け基礎編 アウトライン改訂案

本アウトラインは、  
php-fpm 周辺の知識を **「昔の雑学」ではなく「現在の実行形態を理解するための責務と歴史の地図」** として理解させる、という執筆ガイドラインの方針に沿って再構成したものです。

また、企画案で示されていた  
CGI / mod_php / FastCGI / php-fpm の比較、Apache MPM と PHP 実行形態の区別、そして「なぜ php-fpm は自然な選択だったのか」という到達点を保ちつつ、章の粒度を揃えやすい形に圧縮しています。

今回はこれに加えて、**Nginx がなぜ php-fpm と相性のよい構成として定着したのか** を説明するための章を追加しています。  
ただし、本書は Nginx の内部実装やイベントループの詳細を解説する本ではありません。  
Nginx はあくまで **Web サーバー側の責務分離を理解するための比較対象** として扱います。

## この改訂案の基本方針

- 1章につき1つの中心命題に寄せる
- 「地図」「歴史比較」「責務分解」「総回収」の流れを明確にする
- Apache MPM と PHP 実行形態の混同を防ぐ
- Nginx は Web サーバー側の設計思想として位置づける
- async ランタイムやイベントループ実装の詳細には踏み込みすぎない
- 第2冊への橋渡しは行うが、本冊の主題は php-fpm の理解に固定する

### 12-license.md

# ライセンス

## 章構成の見直しポイント

今回の改訂では、Nginx を独立章として追加しました。  
理由は、Nginx が単なる Web サーバー製品ではなく、**Web サーバー側の責務をどう見るか** という本書の中心論点と深く関わっているためです。

一方で、本冊では Nginx の内部実装やイベントループ詳細には踏み込みません。  
本書の主題はあくまで **php-fpm ユーザーが現在の構成を説明できるようになること** にあります。

そのため、Nginx は次のように扱います。

- PHP を実行するものではない
- HTTP の受け口としての Web サーバー側の設計として位置づける
- FastCGI / php-fpm との責務分担の相性として説明する

## このアウトラインの狙い

この改訂案の狙いは、  
CGI → mod_php → FastCGI → php-fpm という流れを、単なる歴史順ではなく、**責務分解と実行モデルの整理の流れ** として読めるようにすることです。

さらに、Nginx を加えることで、  
**なぜ Web サーバーを HTTP の受け口として整理し、PHP 実行側を外に分ける構成が広く受け入れられたのか** を、読者の現場感覚に近い形で説明しやすくしています。

そのため、本書は async 本の前日譚ではありますが、主題はあくまで **php-fpm ユーザーが現在の構成を説明できるようになること** に置いています。
