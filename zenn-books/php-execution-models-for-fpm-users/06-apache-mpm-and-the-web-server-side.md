title: "Apache MPM とは何か — prefork・worker・event はどの層の話か"
---

前の章では、mod_php の便利さが、そのまま Apache と PHP の結びつきの強さでもあった、という話をしました。  
Web サーバーの中で PHP を動かせることは自然で便利でしたが、その近さは、責務分離や運用の観点ではだんだん重く見えてきます。

ここで一度、話をはっきり分けておきます。  
この章の主役は **Apache MPM** です。  
ただし、これは PHP の実行形態そのものの話ではありません。  
**Web サーバー側が、接続やリクエストをどうさばくか** という、別の層の話です。

mod_php を理解しようとすると、この二つの話が混ざりやすくなります。  
だからこそ、ここで Apache 側の話だけをいったん切り出して整理しておく必要があります。

Apache の設定っぽい感じはするんですが、PHP の話でもあるんですか。

**ベテラン**  
そこが、まさにこの章で分けたいところです。  
まず結論から言うと、**MPM は Apache 側の話** です。  
PHP の実行形態そのものを指す言葉ではありません。

**新人**  
Apache 側の話。  
つまり、Web サーバーの都合の話なんですね。

**ベテラン**  
そうです。  
もう少し丁寧に言うと、Apache が複数の接続やリクエストをどう扱うか、という並行処理の話です。  
誰が待つのか。  
何を並べて処理するのか。  
どの単位で仕事を分けるのか。  
そういう設計に関係します。

**運用の人**  
現場感覚でも、MPM は Web サーバーのふるまいを見るときの話ですね。  
Apache が接続をどう受けて、どう分担して処理するか、という見え方に関わります。

**新人**  
でも、mod_php の話では MPM と PHP の相性みたいな話が出てきましたよね。  
だったら、やっぱり PHP の話でもあるんじゃないですか。

**ベテラン**  
そこが混ざりやすいところです。  
**関係はあるけれど、同じ層の話ではない**、というのが正確です。  
まず MPM 自体は Apache 側の話。  
その上で、Apache の中で PHP を動かす mod_php では、Apache 側の並行処理モデルが PHP の動かし方と近くなりやすい。  
だから影響し合って見えるんです。

**新人**  
では、MPM は具体的には何を決めているんですか。

**ベテラン**  
ざっくり言えば、**Apache が仕事をどう分担して処理するか** を決めています。  
HTTP リクエストが来たときに、それをどんな単位で受け持つのか。  
プロセスで分けるのか。

そうした基本方針です。

**新人**  
つまり、「PHP をどう解釈するか」ではなくて、「Apache がどう働くか」の話なんですね。

**ベテラン**  
その通りです。  
ここを最初に固定しておくと、後の話がかなり楽になります。

**運用の人**  
現場で見るなら、Apache のプロセスやスレッドの増え方、接続の扱い方、負荷時のふるまいに関係する話、と考えると近いですね。

**新人**  
なるほど。  
PHP ファイルをどう実行するかより前に、Apache という Web サーバーがどうリクエストを抱えるかを決めているわけですね。

**ベテラン**  
はい。  
だから MPM は、PHP 実行形態と並べて比較する言葉ではありません。  
むしろ、PHP 実行形態が Apache の中に入っているときに、その影響が見えやすくなる、と考えるほうが自然です。

## prefork・worker・event は何が違うのか

**新人**  
MPM の代表例として、prefork、worker、event という名前が出てきますよね。  
ここもまだ、名前だけ知っている状態です。

**ベテラン**  
この章では、細かな設定値ではなく、考え方の違いだけ押さえましょう。

### prefork

**ベテラン**  
prefork は、かなり素朴なモデルです。  
大づかみに言えば、**プロセス単位で仕事を受け持つ** 方向です。

**新人**  
プロセス単位。  
つまり、仕事を分ける単位が比較的大きいんですね。

**ベテラン**  
そうです。  
まずはその理解で十分です。  
並行処理のモデルとしては比較的わかりやすく、直感的です。

**運用の人**  
現場感覚でも、「まずはプロセスで分けて考える」と思えば追いやすいですね。  
ただし、その素朴さには当然コストもあります。

### worker

**ベテラン**  
worker は、prefork よりも **スレッドを使って仕事をさばく** 方向に寄っています。

**新人**  
プロセスだけではなく、スレッドも使っていくわけですね。

**ベテラン**  
はい。  
ここで重要なのは、Apache 側が「もっと効率よく多数の仕事を扱いたい」と考えたとき、仕事の持ち方の単位が変わってくることです。

**新人**  
なるほど。  
Web サーバーとしては、接続やリクエストをより効率よく扱いたい。  
そのためにモデルが変わるわけですね。

### event

**ベテラン**  
event は、さらに **接続待機の扱い** を意識したモデルです。

**新人**  
接続待機、ですか。

**ベテラン**  
はい。  
Web サーバーの仕事は、単に PHP を実行することではありません。  
接続を受け、待ち、読み書きし、必要ならその先の処理へ渡す。  
この中には、「ただ待っている時間」もあります。  
event は、その待ち方の扱いをより整理しようとする方向だと見てください。

**運用の人**  
現場でも、接続をどう待つかは効いてきます。  
特に多数の接続があると、ただ待っていること自体の扱い方が重要になりますからね。

**新人**  
つまり prefork、worker、event は、全部 Apache 側で「どう仕事をさばくか」の違いなんですね。

**ベテラン**  
その理解で大丈夫です。  
この章では、詳細よりも **プロセス中心か、スレッドも使うか、接続待機をどう意識するか** という輪郭を押さえることが大切です。

**ベテラン**

すると、Apache の並行処理モデルの話と、PHP をどう安全に動かすかの話が近い場所に並ぶことになります。

**新人**

**ベテラン**  
そうです。  
ここで大事なのは、  
**MPM 自体が PHP の機能ではない**  
ということと、  
**mod_php のように Apache の中で PHP を動かすと、その影響を受けやすい**

**運用の人**

Apache の並行処理モデルを調整しているのか、PHP 実行側の構成を見直しているのか。  
そこが混ざると、問題の切り分けもしづらくなります。

**新人**  
なるほど。  
MPM は Apache の話。  
でも mod_php は Apache の中に PHP がいるから、Apache の都合が PHP に近く見える。  
だから分けて理解しないといけないんですね。

**ベテラン**

## 接続待機の話と PHP 実行の話は別である

**新人**  
さっき event のところで「接続待機」という話が出ました。

**ベテラン**  
別です。  
もちろん、最終的にはひとつの Web リクエストの流れの中でつながります。  
でも、概念としては分けたほうがよいです。

接続を受ける。  
必要なら待つ。  
データを読み書きする。  
そして、その先で PHP のアプリケーション処理を行う。  
これらは連続していますが、同じ責務ではありません。

**新人**  
たしかに、「接続を待つこと」と「PHP のコードを実行すること」は、同じに見えて違いますね。

**運用の人**  
現場では、その違いがかなり効きます。  
ボトルネックが接続待機にあるのか、PHP 側の処理にあるのかで、見る場所が変わりますから。

**ベテラン**  
この本で大事にしたいのは、そうした切り分けです。  
Apache MPM は、接続やリクエストをどうさばくかという Web サーバー側の話。  
PHP の実行形態は、その先でアプリケーション処理をどう担当させるかという話。  
ここを混同しないことが、本書全体の理解の土台になります。

### 整理

Apache MPM は、Apache が接続やリクエストをどう並行に扱うかを決める、**Web サーバー側の並行処理モデル** です。  
PHP の実行形態そのものを指す言葉ではありません。

この章で押さえたい代表例は次の三つです。

- `prefork`  
  プロセス中心で仕事を受け持つ、比較的素朴でわかりやすい方向です。

- `worker`  
  スレッドも使いながら、より効率よく仕事をさばく方向です。

- `event`  
  接続待機の扱いも意識しながら、多数の接続をより整理して扱う方向です。

重要なのは、これらがすべて **Apache 側の都合** だということです。  
ただし mod_php のように Apache の中で PHP を動かす構成では、Apache 側の並行処理モデルと PHP の実行の話が近く見えやすくなります。

したがって、次の区別を崩さないことが重要です。

- Apache MPM は Web サーバー側の話
- PHP の実行形態は PHP 側の話
- 接続待機の問題と、PHP のアプリケーション処理の問題は別である

### この章の核心

Apache MPM は、Web サーバーが接続やリクエストをどうさばくかという、Apache 側の並行処理モデルです。  
mod_php ではその話が PHP の実行形態に近く見えやすくなりますが、だからこそ両者を別の層として分けて理解する必要があります。

## まとめ

- Apache MPM は、Apache 側の並行処理モデルであり、PHP の実行形態そのものではない
- prefork、worker、event は、Apache が仕事をどう分担して扱うかの違いである
- mod_php では Apache の中で PHP を動かすため、MPM の話と PHP 実行の話が近く見えやすい
- 接続待機の問題と PHP のコード実行の問題は分けて考える必要がある
- この切り分けがあると、後に Nginx や FastCGI、php-fpm を読むときにも層の違いを見失いにくくなる

次章では、Nginx はなぜ php-fpm と相性がよかったのかを見ていきます。  
ここで Web サーバー側の責務をさらに見やすくすることで、HTTP の受け口と PHP 実行側を分ける構成が、なぜ自然に広がっていったのかを整理していきます。
