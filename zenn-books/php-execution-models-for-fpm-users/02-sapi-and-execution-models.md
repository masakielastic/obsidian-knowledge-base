---
title: "PHP はどう実行されるのか — SAPI と実行形態の見取り図"
---

前の章では、PHP の知識と PHP の実行形態の知識は別だ、という話をしました。  
同じ PHP のコードを書いていても、どこで、どのように動いているかによって、構成の見え方も責務分担も変わります。

その話を具体的にするために、この章では **SAPI** という見取り図を入口にします。  
細かな内部実装を覚えるためではありません。  
PHP がどんな場所で、どんな形で実行されるのかをざっくり分類し、あとで CGI や mod_php や php-fpm を比べやすくするためです。

## SAPI とは何か

**新人**  
SAPI って、正直あまり意識したことがありません。  
PHP の専門用語という感じがして、ちょっと身構えてしまいます。

**ベテラン**  
その反応は自然です。  
でも、この章では難しいものとして捉えなくて大丈夫です。  
まずは、**PHP が外の世界とどうつながるかを表す入口** くらいに考えてください。

**新人**  
外の世界とどうつながるか、ですか。

**ベテラン**  
はい。  
PHP はいつも同じ形で単独で動いているわけではありません。  
コマンドラインから実行されることもあれば、Web サーバーの中で動くこともありますし、Web サーバーとは別プロセスとして待機していることもあります。

その「どんな環境で、どういう接続のされ方をしているか」を整理するときの見取り図が SAPI です。

**運用の人**  
現場でいえば、  
「この PHP はどこで動いているのか」  
「誰から呼ばれているのか」  
「どこまでが Web サーバーの責務で、どこからが PHP 側の責務なのか」  
を考える入口ですね。

**新人**  
なるほど。  
PHP の文法の話じゃなくて、PHP の置かれ方の話なんですね。

**ベテラン**  
そうです。  
この本では、SAPI を細かく分類し尽くすことが目的ではありません。  
まずは、あとで必要になる代表的な実行形態を見分けられるようになることが大事です。

## 同じ PHP でも、動いている場所が違う

**新人**  
でも、PHP は PHP ですよね。  
そんなに置かれ方が違うんですか。

**ベテラン**  
違います。  
たとえば一番わかりやすいのは CLI です。  
端末で `php script.php` と実行するあれですね。

**新人**  
それはイメージできます。  
自分でコマンドを打って、その場で動いて終わる形ですね。

**ベテラン**  
はい。  
これは Web サーバーとは関係ありません。  
リクエストを受けてページを返すのではなく、コマンドとして PHP を実行している形です。

一方で、Web の PHP は、ブラウザからの HTTP リクエストを受けて動きます。  
ただし、そのときも PHP がどこで実行されるかは一通りではありません。

**新人**  
そこがこの本の本題に近いところですね。

**ベテラン**  
その通りです。  
ざっくり言えば、次のような違いがあります。

- Web サーバーの中に PHP が組み込まれて動く形
- Web サーバーの外で PHP が別プロセスとして動く形
- そもそも Web と関係なくコマンドとして動く形

この違いがあると、設定の置き場所も、責務分担も、障害の切り分け方も変わります。

**運用の人**  
監視やログの見方も変わりますね。  
「Apache の中で PHP が動いている」のか、  
「Nginx から php-fpm に投げている」のかで、見に行く場所がかなり変わります。

**新人**  
たしかに。  
今までは全部まとめて「PHP のサーバー設定」みたいに見ていましたけど、本当は層が違うんですね。

## まず押さえたい代表的な実行形態

**ベテラン**  
この本で特に意識したいのは、次の代表的な実行形態です。

- `cli`
- `apache2handler`
- `cgi-fcgi`
- `fpm-fcgi`
- `embed`

**新人**  
いきなり名前が増えました。

**ベテラン**  
ここでは名前を暗記するより、配置の違いをつかむことが大事です。  
ひとつずつざっくり見ていきましょう。

### cli

**ベテラン**  
`cli` はコマンドラインです。  
端末から PHP を直接実行する形です。

**新人**  
これは一番わかりやすいですね。  
バッチ処理とか、簡単なスクリプトとか、Composer のコマンドとかもこの仲間ですか。

**ベテラン**  
その理解で大丈夫です。  
Web サーバーが前にいないので、HTTP のリクエストを受ける PHP とは前提が違います。

**運用の人**  
現場だと cron で定期実行しているスクリプトや、デプロイ時の補助コマンドなどもこの系統ですね。  
同じ PHP でも、Web リクエストの PHP とは切り分けて考えたほうがよい場面が多いです。

### apache2handler

**ベテラン**  
`apache2handler` は、Apache の中で PHP を動かす形です。  
いわゆる mod_php の世界ですね。

**新人**  
ああ、ここで mod_php が出てくるんですね。  
つまり Apache が PHP を中に抱えている感じですか。

**ベテラン**  
はい。  
Web サーバーの中で PHP をそのまま実行する形だと考えると、まずは十分です。  
この方式の自然さや強み、そして後で限界が見えてくることは、後の章で詳しく扱います。

**新人**  
今の段階では、「Web サーバーの中で動く PHP」という位置づけで見ればいいんですね。

**ベテラン**  
その通りです。

### cgi-fcgi

**ベテラン**  
`cgi-fcgi` は、CGI や FastCGI 系の入口です。  
ここでは、**Web サーバーの外にいる PHP と通信して動かす形** がある、という理解を作ることが大事です。

**新人**  
外にいる PHP と通信する、ですか。

**ベテラン**  
はい。  
Apache の中に PHP を組み込むのではなく、外側にいる PHP 実行系に処理を渡すわけです。  
このとき、CGI と FastCGI では性質がかなり違いますが、まずは  
「Web サーバーの外に出す流れがある」  
と押さえてください。

**新人**  
ここではまだ CGI と FastCGI の違いを細かく見なくていいんですね。

**ベテラン**  
そうです。  
この章の目的は地図をつくることです。  
細かな比較は後でやります。

### fpm-fcgi

**ベテラン**  
`fpm-fcgi` は、php-fpm の形です。  
今の現場では、これが一番身近な人が多いでしょう。

**新人**  
Nginx の後ろで php-fpm を動かす、あの構成ですね。

**ベテラン**  
はい。  
Web サーバーの外に PHP 実行側を分け、その PHP 側でプロセス管理まで行う形です。  
ここも後で詳しく扱いますが、今は  
「FastCGI 系の中でも、php-fpm は PHP 実行側の管理を強く意識した形」  
と捉えておけば十分です。

**運用の人**  
現場感覚では、これが「PHP の実行プロセスを別口で管理している」構成ですね。  
だから Web サーバーの設定と php-fpm の設定が別になり、ログや監視も分かれやすくなります。

### embed

**ベテラン**  
最後に `embed` です。  
これは PHP を他のプログラムの中に埋め込んで使う形です。

**新人**  
そんなこともできるんですか。

**ベテラン**  
できます。  
ただ、この本では主役ではありません。  
重要なのは、PHP という言語処理系が、必ずしも「Web サーバーの後ろの PHP」だけではないということを知ることです。

**新人**  
CLI もあれば、Apache の中もあれば、外部プロセスとして待つ形もあれば、埋め込む形もある。  
たしかに「PHP はいつも同じ形で動いているわけではない」という話が実感できます。

## この章で特に分けたいこと

**新人**  
だんだん地図っぽくなってきました。  
でも、まだ頭の中では少し混ざっています。

**ベテラン**  
それで大丈夫です。  
ここで特に分けたいのは、次の三つです。

一つ目は、**Web か Web でないか** です。  
CLI は Web ではありません。  
Apache や php-fpm の話は Web の文脈です。

二つ目は、**Web サーバーの中で動くか、外で動くか** です。  
`apache2handler` は中で動きます。  
`fpm-fcgi` は外で動きます。

三つ目は、**PHP を実行するだけなのか、PHP 実行側を管理するのか** です。  
php-fpm は、単に PHP を動かすだけでなく、その実行プロセスの管理まで引き受けます。

**新人**  
ああ、ここを分けるとかなり見やすいです。

- CLI はそもそも Web じゃない
- mod_php は Apache の中
- php-fpm は Web サーバーの外
- php-fpm は外にあるだけじゃなくて管理もしている

という感じですね。

**ベテラン**  
そうです。  
この段階では、その粒度で十分です。

**運用の人**  
実務でも、その切り分けがあるだけでかなり楽です。  
たとえば「この問題は Apache 側か、php-fpm 側か」という問いは、Web サーバーの責務と PHP 実行側の責務を分けて見ないと立てられません。

## なぜこの地図が必要なのか

**新人**  
でも、最初から CGI の話に入ってもよかった気がします。  
どうしてわざわざ SAPI の地図を先に作るんですか。

**ベテラン**  
理由は単純です。  
地図なしで歴史に入ると、名前だけが増えて混乱しやすいからです。

CGI、mod_php、FastCGI、php-fpm を順番に見ていくとき、  
それぞれが

- Web なのか Web でないのか
- Web サーバーの中なのか外なのか
- PHP の実行なのか、その管理まで含むのか

この三つのどこに位置するのかが分かっていると、比較しやすくなります。

**新人**  
たしかに、今の時点でも「何を比べる話なのか」が少し整理されてきました。

**ベテラン**  
この本の中心は、単なる用語集ではありません。  
実行形態の違いを通じて、責務の違いを見ることです。  
だから最初に地図が要るんです。

「その違いは、置かれている場所と責務分担の違いとして見える」  
というところまでを押さえます。

**運用の人**  
現場でも、最初に全部を理解しようとすると詰まりやすいです。  
まずは配置をつかむ。  
その上で、次の章からひとつずつ比較していくほうが頭に入りやすいですね。

**新人**  
了解です。  
この章は地図。  
次の章から個別比較に入る。  
そう思って読めばいいわけですね。

その通りです。

### 整理

この章で見た代表的な実行形態は、次のように整理できます。

- `cli`  
  Web サーバーを介さず、コマンドラインから PHP を実行する形です。

- `apache2handler`  
  Apache の中で PHP を動かす形です。  
  いわゆる mod_php の世界です。

- `cgi-fcgi`  
  Web サーバーの外にいる PHP 実行側と通信する流れの入口です。  
  CGI や FastCGI の比較は後の章で詳しく扱います。

- `fpm-fcgi`  
  php-fpm の形です。  
  Web サーバーの外で PHP を動かし、その実行側の管理も担います。

- `embed`  
  PHP を他のプログラムに埋め込んで使う形です。

この章で特に重要なのは、次の区別です。

- Web か、Web でないか
- Web サーバーの中で動くか、外で動くか
- PHP を実行するだけか、実行側の管理も担うか

### この章の核心

SAPI は、PHP がどこでどのように動いているかを見分けるための地図です。  
この地図があると、CGI、mod_php、FastCGI、php-fpm の違いを、単なる名前の違いではなく、配置と責務分担の違いとして読めるようになります。

## まとめ

- SAPI は、PHP がどの環境でどのように実行されるかを整理する入口である
- `cli` は Web ではなく、コマンドラインから PHP を実行する形である
- `apache2handler` は Apache の中で PHP を動かす形であり、mod_php の世界に対応する
- `fpm-fcgi` は Web サーバーの外で PHP を動かし、その実行側の管理も担う形である
- この章の目的は分類を暗記することではなく、後の章で実行形態を比較するための地図をつくることである

次章では、まず CGI を取り上げます。  
もっとも素朴で分かりやすい実行形態から出発することで、なぜ後の方式が必要になったのかを見やすくしていきます。
