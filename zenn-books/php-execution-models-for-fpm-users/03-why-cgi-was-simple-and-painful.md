---
title: "CGI はなぜわかりやすく、なぜつらかったのか"
---

前の章では、SAPI を入口にして、PHP がいくつかの形で実行されることを見ました。  
その地図を踏まえて、この章からは個別の実行形態を順番に見ていきます。

最初に扱うのは CGI です。  
CGI は、今では昔の方式として語られることが多いのですが、最初に学ぶ対象としてはとても重要です。  
なぜなら、仕組みが単純であるぶん、「Web サーバーがリクエストを受け、外部プログラムを起動して結果を返す」という構図が見えやすいからです。

ただし、その単純さはそのまま限界にもつながります。  
この章では、CGI のわかりやすさと、そのわかりやすさがなぜ運用上のつらさになったのかを整理します。

## CGI は何をしているのか

**新人**  
CGI って、名前は何度も見たことがあります。  
でも正直、「古いやつ」という印象しかありません。

**ベテラン**  
最初はそれで構いません。  
ただ、この章では「古いから終わったもの」として見るのではなく、まずはどういう発想だったのかを押さえましょう。

CGI は、とても素朴な仕組みです。  
Web サーバーが HTTP リクエストを受け取り、その内容を外部プログラムに渡して、そのプログラムが返した結果をレスポンスとして返します。

**新人**  
外部プログラムに渡す、ですか。  
つまり、Web サーバーの中に PHP がいるわけではないんですね。

**ベテラン**  
そうです。  
ここでは PHP に限りません。  
Perl でも C でも、外部プログラムとして動けばよいわけです。  
Web サーバー自身が全部を処理するのではなく、「このリクエストは外のプログラムに任せよう」という形ですね。

**運用の人**  
現場感覚でいうと、Web サーバーとアプリケーションがかなりはっきり分かれています。  
Web サーバーは受け口で、実際の処理は外に投げる。  
その意味では、責務分担の構図は見えやすいです。

**新人**  
たしかに、構図としてはわかりやすいです。  
Web サーバーが受けて、外のプログラムが処理する。  
それだけならすごく自然です。

**ベテラン**  
はい。  
だから CGI は、概念の入門としてはむしろ優れています。  
Web サーバーが何でも抱え込むのではなく、外部プログラムに仕事を渡せる、という考え方がはっきり見えるからです。

## なぜ CGI はわかりやすいのか

**新人**  
「わかりやすい」と言われる理由は、今の説明でなんとなくわかります。  
でも、もう少し言葉にできますか。

**ベテラン**  
できます。  
CGI がわかりやすいのは、大きくいって三つの点です。

まず、リクエストごとの流れが素直です。  
リクエストが来た。  
そのためのプログラムを起動する。  
出力を返す。  
流れが直列で見えます。

次に、Web サーバーとアプリケーションの境界がはっきりしています。  
Web サーバーは受け付ける。  
アプリケーションは処理する。  
この役割分担がかなり見えやすいです。

そして最後に、1回のリクエストが1回の実行に対応しているので、頭の中で追いやすいんです。

**新人**  
ああ、たしかに。  
「1回アクセスされたら、1回起動して、終わったら終わり」という感じなら、初心者でもイメージしやすいです。

**ベテラン**  
そうです。  
常駐プロセスやワーカープールのような話をまだ知らなくても理解できます。  
だから CGI は、実行モデルの原型としてかなり教育的です。

**運用の人**  
運用の観点でも、最初の理解としては素直ですね。  
「毎回まっさらな実行が始まる」と思えばよいので、長生きするプロセス特有の悩みは少ないです。

**新人**  
それだけ聞くと、むしろ安全そうにも見えますね。

**ベテラン**  
その感覚も半分は当たっています。  
ただし、その素直さは、別の形で大きなコストを生みます。

## 1リクエストごとに起動する、という意味

**新人**  
ここが本題ですね。  
CGI は何がつらかったんですか。

**ベテラン**  
一番大きいのは、**1リクエストごとに外部プログラムを起動する** ことです。

**新人**  
1回アクセスが来るたびに、毎回起動するんですか。

**ベテラン**  
そうです。  
たとえば 100 回リクエストが来たら、100 回そのプログラムを起動することになります。  
処理が終われば、その実行も終わります。

**新人**  
毎回新しく始めて、毎回終わる。  
さっきの「わかりやすさ」の裏返しですね。

**ベテラン**  
その通りです。  
わかりやすいということは、言い換えれば「毎回ゼロからやり直す」ということでもあります。

**運用の人**  
現場で考えると、これが積み重なると重いです。  
リクエストが来るたびにプロセスを起こして、環境を整えて、処理して、終わったら片付ける。  
これを高頻度で繰り返すわけです。

**新人**  
それは確かに大変そうです。  
PHP のコードを実行する前に、起動の手間そのものが毎回かかるんですね。

**ベテラン**  
そうです。  
ここで重要なのは、「PHP の処理が重い」のとは別に、「処理を始めるためのコスト」が毎回発生することです。

**新人**  
つまり、アプリケーションの中身が軽くても、入口で毎回コストを払うんですね。

**ベテラン**  
その理解で合っています。  
CGI のつらさは、アプリケーションロジック以前のところにあります。

## 単純さの代償は、起動コストの反復にある

**新人**  
でも、昔はそれでも使われていたんですよね。  
ということは、最初はそこまで困らなかったんですか。

**ベテラン**  
そうです。  
リクエスト数が少ない、処理規模が小さい、Web 自体の使われ方が今ほど重くない。  
そういう時代には、単純さのほうが価値を持ちやすかったわけです。

**新人**  
なるほど。  
最初からダメだったわけではなくて、状況に合っていたんですね。

**ベテラン**  
はい。  
ただ、アクセスが増え、アプリケーションが大きくなり、Web サーバーにかかる負荷が高くなると、毎回起動して毎回終わるモデルはだんだん厳しくなります。

**運用の人**  
現場では、単発なら気にならなくても、繰り返しが増えると効いてきます。  
1回あたりの起動コストがそこまで大きく見えなくても、それが多数のリクエストで積み上がると無視できません。

**新人**  
しかも Web は、同じ種類の処理を何度も受けるわけですもんね。

**ベテラン**  
その通りです。  
だから CGI の問題は、単発の正しさではなく、繰り返しに弱いことにあります。

## CGI は悪い設計だったのか

**新人**  
ここまで聞くと、CGI は非効率で古いやり方、という印象になります。  
でも、そこまで単純に言っていいんでしょうか。

**ベテラン**  
そこは大事なところです。  
CGI は「悪い設計だった」わけではありません。  
むしろ、当時の条件で考えれば、とても自然で理解しやすい方式でした。

**新人**  
自然だった、ですか。

**ベテラン**  
はい。  
Web サーバーがリクエストを受け、必要に応じて外部プログラムを実行する。  
この構図は素朴ですが、責務が見えやすい。  
実装の出発点としても理解の出発点としても筋がいいんです。

問題は、その素朴さが大量のリクエストに対しては高くつくことでした。

**運用の人**  
現場でも、仕組みのわかりやすさと、大規模運用への向き不向きは別です。  
CGI は、前者では優れていて、後者でだんだん厳しくなったと見るほうが公平ですね。

**新人**  
つまり、設計の思想自体が間違っていたというより、時代と負荷に対してつらくなっていったんですね。

**ベテラン**  
その通りです。  
この本では、そういう整理を大事にします。  
昔の方式を見下すためではなく、何が自然で、どこで苦しくなったのかを見るために学ぶんです。

## では、何が欲しくなったのか

**新人**  
CGI がつらいなら、次に欲しくなるのは何ですか。

**ベテラン**  
一言でいえば、**毎回起動しなくてよい仕組み** です。

**新人**  
ああ、それはわかりやすいです。  
毎回ゼロから始めるのがつらいなら、あらかじめ待機していてくれればよさそうです。

**ベテラン**  
まさにその発想です。  
リクエストのたびに新しくプロセスを起こすのではなく、あらかじめ動いている実行側に処理を渡したい。  
そうすると、毎回の起動コストを減らせます。

**運用の人**  
現場でも、その方向が自然です。  
「受けたら毎回新しく立ち上げる」より、「あらかじめ処理役を待たせておく」ほうが、繰り返しには強いですからね。

**新人**  
なるほど。  
CGI の問題は、外に投げるという発想そのものではなく、毎回新しく起動することにあった。  
だから次は、外に置いたまま常駐させる方向に進むわけですね。

**ベテラン**  
その理解で大丈夫です。  
次の章以降で見る mod_php は、別の方向からこの問題に応えた方式ですし、FastCGI は CGI の延長線上で改善しようとした方式です。  
どちらも、CGI のつらさを背景にして見えてきます。

## CGI を先に学ぶ意味

**新人**  
だんだん、最初に CGI から入る理由がわかってきました。  
これが原型なんですね。

**ベテラン**  
そうです。  
CGI を先に理解すると、後の方式が何を改善しようとしたのかが見えやすくなります。

- mod_php は、外に投げるのではなく Web サーバーの中で PHP を動かす
- FastCGI は、外に置く発想を残しつつ毎回起動をやめる
- php-fpm は、その実行側の管理まで担う

こうした違いも、CGI の素朴な構図を知っていると整理しやすいです。

**新人**  
最初の形が見えると、後の方式が「何を足したのか」「何を変えたのか」で見られるわけですね。

**ベテラン**  
その通りです。  
だから CGI は、今の主流ではなくても、本書の出発点としてはとても重要です。

### 整理

CGI は、Web サーバーがリクエストを受け、その内容を外部プログラムに渡し、その結果をレスポンスとして返す仕組みです。

この方式がわかりやすいのは、次の理由によります。

- Web サーバーとアプリケーションの境界が見えやすい
- 1リクエストごとの流れが直列で追いやすい
- 常駐プロセスや複雑な管理を前提にしなくても理解できる

一方で、この単純さはそのまま限界にもつながります。

- 1リクエストごとに外部プログラムを起動する
- 処理開始前の起動コストが毎回発生する
- リクエストが増えるほど、その反復コストが効いてくる

つまり CGI の問題は、外部プログラムに処理を任せるという発想そのものではなく、**毎回新しく起動するモデル** にあります。

### この章の核心

CGI は、Web サーバーとアプリケーションの境界をわかりやすく見せてくれる、素朴で教育的な実行形態です。  
ただし、その単純さは 1 リクエストごとの起動コストを意味し、繰り返しの多い Web では次第に大きな負担になっていきました。

## まとめ

- CGI は、Web サーバーが外部プログラムを起動して処理を任せる素朴な仕組みである
- CGI がわかりやすいのは、Web サーバーとアプリケーションの境界が見えやすいからである
- CGI の中心的な負担は、1リクエストごとに外部プログラムを起動することにある
- CGI は悪い設計だったのではなく、単純さの代償が大規模な繰り返し処理で重くなったと見るべきである
- 後の方式は、CGI のわかりやすさを踏まえつつ、この起動コストの問題にどう向き合うかという流れで理解できる

次章では、CGI とは別の方向から実用性を高めた方式として、mod_php を見ていきます。  
Web サーバーの中で PHP を動かすという発想が、なぜ当時とても便利だったのかを整理します。
