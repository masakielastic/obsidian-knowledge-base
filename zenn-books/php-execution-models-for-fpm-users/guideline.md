# PHP-FPMユーザー向け基礎編 執筆ガイドライン

## 1. 書籍の目的

この書籍の目的は、PHP-FPM を使っている読者が、現在の構成を  
**「いつもの運用」ではなく「歴史と責務分解の結果としての実行モデル」**  
として理解できるようにすることです。

読者が最終的にできるようになるべきことは次の通りです。

- CGI / mod_php / FastCGI / php-fpm の違いを説明できる
- Apache MPM と PHP 側の実行形態を混同しない
- mod_php が衰退した理由を説明できる
- php-fpm が主流になった理由を説明できる
- ReactPHP や FrankenPHP の話を学ぶ前の土台を持てる

## 2. 想定読者

- 普段は php-fpm を使っている PHP ユーザー
- Nginx や Apache の背後で PHP が動いていることは知っているが、歴史や構造には自信がない人
- CGI や mod_php の違いを断片的にしか知らない人
- ReactPHP や FrankenPHP に興味はあるが、まず現行の構成を理解したい人
- API の暗記ではなく、実行形態の見取り図を求めている人

## 3. 書籍の基本方針

### 重視すること

- 歴史的経緯の整理
- 実行形態の区別
- 責務の分解
- php-fpm が自然だった理由の説明
- 読者の現場感覚との接続
- 「昔の方式」と「今の方式」の比較可能な地図

### 深入りしないこと

- event loop 実装の詳細
- ReactPHP / Swoole / FrankenPHP の詳説
- OS 内部実装の深掘り
- ベンチマーク中心の優劣比較
- 各製品の設定値チューニング集
- ライブラリや拡張の API カタログ化

## 4. この本で特に区別すべきもの

本書では、特に次の区別を崩さないことを重視します。

- CGI / FastCGI / php-fpm
- mod_php / php-fpm
- Apache MPM / PHP 実行形態
- prefork / worker / event
- プロセス / スレッド
- Web サーバーの責務 / PHP 実行側の責務
- 接続待機の問題 / PHP アプリケーション処理の問題
- 歴史的な事情 / 設計上の必然

## 5. 対話形式の役割分担

登場人物は3役で固定します。

### 新人

役割：

- 疑問・混乱・誤解を出す
- 読者の「そういえばよく知らない」を代弁する
- 早い結論や雑な理解を口にする
- 「今の構成で困っていないのに、なぜ昔の話を学ぶのか」という疑問を出す

注意：

- 賢くしすぎない
- 用語を自力で整理しすぎない
- 正しい結論を先回りして言わない
- 説明役にならない
- ベテランや運用の人の役割を奪わない

### ベテラン

役割：

- 論点を切り分ける
- 歴史を設計比較として整理する
- 用語を定義し、層の違いを言語化する
- 章の中心命題に会話を戻す
- 「なぜその方式が選ばれたのか」を説明する

注意：

- 長い講義調の独演にしない
- 何でも一気に説明しない
- 歴史話を年表のように並べない
- 比喩だけで終わらせず、正式な用語に戻す
- 運用の細部まで自分だけで説明しない

### 運用の人

役割：

- 実際の運用や構成の見え方を補足する
- Web サーバー、php-fpm、プロセス管理の現実感を与える
- 設定や常駐プロセスの感覚を具体化する
- ベテランの整理を現場の視点に接続する
- 「現場ではどこが問題になりやすいか」を示す

注意：

- 抽象整理を主導しない
- ベテランの役割を奪わない
- 特定製品の運用ノウハウだけに寄りすぎない
- スコープを超えて Swoole や ReactPHP の詳説に入らない
- 章の主題を「設定ハウツー」に変えない

## 6. 本書で扱う3つの仮説

### 論点仮説

- この章では何を問題として切り出すか
- どの層の話をしているか
- 何と何を比較する章なのか

### アプローチ仮説

- 歴史順で説明するべきか
- 責務順で説明するべきか
- どの比較軸なら混同が減るか

### 答え仮説

- 現時点ではどう整理するのが妥当か
- この章の読者が持ち帰るべき結論は何か

### 運用ルール

- 新人は答え仮説を早めに口にしてよい
- ベテランは論点仮説を調整する
- 運用の人はアプローチ仮説を現場側から支える
- 誤解や試行錯誤は教育的価値があるので必要に応じて残してよい
- ただし、誤解を放置したまま章を終えない

## 7. 文章スタイル

### 基本トーン

- 技術的に正確
- 初学者にやさしい
- 歴史の説明が設計理解につながる
- 簡潔だが冷たすぎない
- 対話として自然
- 音声読み上げでも不自然になりにくい

### 文体ルール

- です・ます調を基本とする
- 一文を長くしすぎない
- 一文一義を意識する
- 定義・整理・結論は明確に言い切る
- 比喩は補助的に使い、最後は正式な用語に戻す
- 「まず」「次に」「ここで分けたいのは」などの段階整理を積極的に使う
- 歴史順と責務順を適切に往復する
- 用語の初出では、その章に必要な範囲で説明する

### 避けるべき文体

- 曖昧な断定の連続
- 専門用語の説明なしの連打
- 歴史を昔話として消費する書き方
- 製品名の列挙だけで意味を説明しない書き方
- 「便利」「速い」「裏でやってくれる」といった責務を曖昧にする表現
- php-fpm と Apache MPM を同列に比較する雑な整理

## 8. 音声向け情報と視覚向け情報の分離

### 対話本文に置くもの

- 疑問
- 誤解
- 因果関係
- 歴史の流れ
- 概念説明
- 責務の切り分け
- 「なぜそうなったのか」の説明

### 対話の外に置くもの

- 箇条書き
- 用語整理
- 比較表の代わりになる短い一覧
- 責務マップ
- 実行形態の比較
- 章末まとめ
- 「この章の核心」

### 運用ルール

- 対話本文では原則として箇条書きを使わない
- 2〜3項目程度の短い確認は、音声で自然なら例外的に許容してよい
- それ以上の比較や一覧は対話の外へ出す
- 一覧や整理は原則として `### 整理` に置く
- 長い整理は短文連続か、視覚ブロックに分離する

## 9. 見出し構造の基準

Zenn の書籍では、基本的な見出しレベルは **2まで** とします。

### 基本ルール

- 各ファイルのタイトルは先頭の `#` を使ってよい
- 章タイトル相当の大見出しは `##`
- 章内セクションは `###`
- 原則として `####` 以降は使わない
- 「整理」は `### 整理`
- 「この章の核心」は `### この章の核心`
- 「まとめ」は `## まとめ`

### 禁止事項

- 章によって見出しレベルを変える
- `####` 以降を常用する
- 「まとめ」の見出しレベルを章ごとに変える
- 見出しだけ細かく増やして本文が薄くなる構成にする

## 10. 章フォーマットの基準

各章は原則として次の流れで構成します。

### 1. 導入

- 新人が疑問や混乱を出す
- ベテランがこの章で扱う範囲を示す
- 必要に応じて、誤った答え仮説を先に出してよい

### 2. 本文

- 対話で概念を分解する
- 必要に応じて `###` セクションを置く
- 運用の人が現場の見え方や運用上の補足を入れる
- 章の中心命題から逸れたら戻す

### 3. 整理ブロック

- 見出し名は原則として `### 整理`
- 本文で扱った内容を、用語・責務・流れ・比較軸として視覚的に整理する
- 新情報は追加しない
- 比較や責務マップを簡潔に圧縮する

### 4. この章の核心

- 見出し名は `### この章の核心`
- 1〜3文程度の短い本文で書く
- 箇条書きは使わない
- その章の中心命題を圧縮して言い切る
- 新情報は追加しない

### 5. まとめ

- 見出し名は `## まとめ`
- 箇条書き3〜5項目で書く
- 用語の区別・責務・流れを再確認する
- 「この章の核心」を箇条書きに言い換えるだけにしない

### 6. 次章への接続

- 1〜2文で次章へつなぐ
- 次章で何の比較や層を扱うかを軽く示す

## 11. 章タイトルと中心命題

- 各章は原則として1つの中心命題に絞る
- 章タイトルを見ただけで、その章が何を整理する章か分かるようにする
- 1章の中で、主役となる概念と補助的に触れる概念の比重を明確にする
- 本文の比重は章タイトルに対応していること
- 話題が広がった場合も、その章の中心命題に必ず戻る

## 12. 校正時の最重要品質基準

### A. 概念の区別がぶれていないか

特に以下は必ず明確であること。

- CGI / FastCGI / php-fpm
- mod_php / php-fpm
- Apache MPM / PHP 実行形態
- prefork / worker / event
- プロセス / スレッド
- Web サーバーの責務 / PHP 実行側の責務
- 接続待機の問題 / アプリケーション処理の問題

### B. 実行モデルへの接続があるか

- 誰が何を担当しているかが説明されているか
- Web サーバー、FastCGI、php-fpm の役割分担が明確か
- Apache MPM がどの層の話か説明されているか
- 歴史の説明が単なる昔話で終わっていないか

### C. 読者の混乱ポイントに答えているか

- 「CGI の何が問題だったのか」
- 「mod_php はなぜ強かったのか」
- 「mod_php はなぜ主流でなくなったのか」
- 「Apache MPM と php-fpm はどう違うのか」
- 「なぜ php-fpm は自然な選択だったのか」

こうした疑問に、本文がきちんと答えていること。

### D. 圧縮が効いているか

- 長い説明のあとに整理があるか
- まとめが機能しているか
- 読了後に3〜5点で復習できるか

### E. 章ごとの粒度が揃っているか

- 見出しの粒度が揃っているか
- まとめの長さがバラバラすぎないか
- 章タイトルと本文の比重が合っているか

## 13. 具体的なレビュー観点

### 用語

- 初出時に説明されているか
- 同じ意味の言葉が複数混在していないか
- 「常駐」「worker」「プロセス」「サーバー」が曖昧に使われていないか
- 「速い」「便利」だけで説明していないか

### 対話

- 新人が説明役になっていないか
- ベテランが独演しすぎていないか
- 運用の人が補足役に収まっているか
- 誤解と修正の流れがあるか
- 3役の役割分担が章ごとに崩れていないか

### 構成

- 1セクション1テーマになっているか
- 歴史順と責務順が混線していないか
- 「この章の核心」と「まとめ」が重複しすぎていないか
- 章の中で論点仮説がぶれていないか

### 可読性

- 一文が長すぎないか
- 読点が多すぎて息継ぎしにくくないか
- 長文の台詞が箇条書きの代わりになって崩れていないか
- 比較や整理が本文に埋もれていないか

### 読み上げ適性

- 対話部分が声に出して自然か
- 箇条書きは視覚ブロックに分離されているか
- 「まず」「次に」「ここで分けたいのは」が機能しているか

## 14. まとめの基準

各章の最後の「まとめ」は、次の基準に従ってください。

- 箇条書き3〜5項目
- 新しい情報を追加しない
- 章タイトルと対応する内容に絞る
- 読者が短時間で復習できるようにする
- 用語の区別と責務の違いが再確認できる内容にする

## 15. リライト方針

校正や改善を行う際は、必要に応じて以下を行ってください。

- 冗長な部分の圧縮
- 曖昧な主語や対象の明確化
- 長い台詞の分割
- 箇条書きと対話の役割分離
- 見出し粒度の統一
- まとめの統一フォーマット化
- 実行形態や責務分解への接続の補強
- 誤解を修正する流れの明確化
- 歴史的経緯を設計比較として読めるように再配置する

ただし、以下は避けてください。

- 詳細すぎる専門知識の追加
- 章のスコープを勝手に拡張すること
- 対話形式を崩して説明文だけにしてしまうこと
- 書籍全体のトーンを急に変えること
- 製品比較や宗教論に流れすぎること

## 16. 出力時の期待

レビュー・校正・リライトを行うときは、必要に応じて次の観点で整理してください。

- 構成上の問題
- 文章上の問題
- 用語運用上の問題
- 対話役割上の問題
- 仮説運用上の問題
- 修正案
- 必要なら統一版のリライト

## 17. この基礎編で特に守るべき一文

この本は、php-fpm 周辺の知識を  
**「昔の技術の雑学」ではなく「現在の実行形態を理解するための責務と歴史の地図」**  
として理解させる本である。

レビュー・改善・リライトの判断は、常にこの方針に従って行ってください。
