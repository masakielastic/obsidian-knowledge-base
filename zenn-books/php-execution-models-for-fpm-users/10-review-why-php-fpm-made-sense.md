---

ここまで、本書では CGI、mod_php、Apache MPM、FastCGI、php-fpm、そしてプロセス・スレッド・接続待機の基礎を順番に見てきました。

それぞれを個別に見ると、別々の技術史の話にも見えます。  
ですが、本書でやりたかったのは、単なる昔話ではありません。  
**現在の php-fpm 周辺の構成が、どんな責務分解と歴史的な流れの上に成立しているのか** を見取り図として理解することでした。

この最終章では、その流れをまとめ直します。  
焦点は一つです。  
**なぜ php-fpm は自然な選択だったのか。**

## 最初にあったのは、わかりやすい外部実行だった

**新人**  
ここまで読んできて、だいぶつながってきました。  
でも最後に、最初から順番にもう一度整理したいです。  
結局、どこからどう自然につながっていったんでしょう。

**ベテラン**  
まず出発点として見たのが CGI でした。  
CGI はとても素朴です。  
Web サーバーがリクエストを受け、外部プログラムに処理を渡し、その結果を返す。  
この構図は、Web サーバーとアプリケーションの境界が見えやすいという意味で、とても教育的でした。

**新人**  
はい。  
「外に渡す」という構図そのものは、かなりわかりやすかったです。

**ベテラン**  
その通りです。  
問題は、そのわかりやすさが **1リクエストごとに毎回起動する** という形と結びついていたことでした。  
だから CGI の弱点は、責務分離そのものではなく、繰り返しに弱いことにありました。

**運用の人**  
現場感覚でもそうですね。  
外に渡す発想は悪くない。  
でも、そのたびに新しく起動するのは、繰り返しの多い Web ではだんだん重くなる。  
ここが最初の苦しさでした。

## 次に強かったのは、Web サーバーの中で完結する便利さだった

**新人**  
そこで mod_php が出てきたんですよね。

**ベテラン**  
はい。  
mod_php は、Apache の中で PHP をそのまま動かす方式でした。  
CGI のように毎回外部プログラムを起動しない。  
構成も短い。  
導入もしやすい。  
Apache が中心だった時代には、これはかなり自然で便利でした。

**新人**  
今振り返ると、mod_php の強みはよくわかります。  
単に「古いやり方」ではなくて、「当時の自然な実用解」だったんですね。

**ベテラン**  
その理解が大事です。  
この本では mod_php を雑に切り捨てないようにしてきました。  
強かった理由を認めないと、なぜ長く使われたかも、なぜ後で限界が見えたかも説明できないからです。

**運用の人**  
現場でも、構成が短いこと、把握しやすいこと、すぐ動かせることは強いです。  
mod_php はまさにそこに強みがありました。

## 便利さの前提が変わると、一体化は重くなる

**新人**  
でも、その便利さが後には限界になるわけですよね。

**ベテラン**  
そうです。  
mod_php の便利さは、Apache と PHP が近いことに支えられていました。  
しかし、その近さは、Web サーバー側の都合と PHP 実行側の都合を分けにくくもします。

**新人**  
つまり、最初は「近いから便利」だったのが、後では「近いから分けにくい」に変わるんですね。

**ベテラン**  
その通りです。  
HTTP の受け口としての Web サーバー。  
アプリケーションを実行する PHP 側。  
この二つを別々に設計し、別々に管理したいという要求が強くなると、一体化は次第に重く見えてきます。

**運用の人**  
監視、再起動、調整、負荷の見方。  
そうした運用の問いも、「ひとつの塊」より「別々の単位」で見たい場面が増えますからね。

## Apache MPM は Web サーバー側の話だった

**新人**  
ここで Apache MPM の章が効いてきますね。  
あれは、Web サーバー側の並行処理モデルの話でした。

**ベテラン**  
そうです。  
prefork、worker、event というのは、Apache が接続やリクエストをどうさばくかの話でした。  
PHP の実行形態そのものではありません。

**新人**  
でも mod_php では、その二つがかなり近く見えてしまう。

**ベテラン**  
そこが重要です。  
MPM 自体は Apache 側の話ですが、Apache の中で PHP を動かすと、Web サーバーの並行処理モデルの話と、PHP をどう安全に動かすかの話が近くなりやすい。  
だからこそ、層を分けて理解する必要がありました。

**運用の人**  
この切り分けがないと、Apache 側の問題なのか、PHP 側の構成の問題なのかが曖昧になります。  
本書で MPM を挟んだのは、その混同をほどくためでした。

## FastCGI は「外に分ける」を実用化した

**新人**  
そして FastCGI ですね。  
ここでかなり流れがつながった感じがしました。

**ベテラン**  
FastCGI は非常に重要です。  
CGI の「外に渡す」という構図を残しながら、毎回起動するつらさを減らしたからです。

**新人**  
つまり、境界は保つ。  
でも毎回ゼロから始めるのはやめる。  
この二つを両立したんですね。

**ベテラン**  
その通りです。  
FastCGI を「CGI の高速版」とだけ言ってしまうと足りません。  
本質は、**Web サーバーとアプリケーション実行側の責務分離を、現実に運用できる形にしたこと** にあります。

**運用の人**  
現場感覚でも、これでかなり見通しがよくなります。  
Web サーバーは HTTP の受け口。  
その向こうに PHP 実行側が待っている。  
この構図が、概念としてだけでなく構成として見えるようになるわけです。

## php-fpm は、分離された PHP 実行側を管理可能にした

**新人**  
そして、その先が php-fpm なんですね。

**ベテラン**  
はい。  
FastCGI は分離の構図を与えました。  
しかし、その分離された PHP 実行側を、どう待機させ、どう増減させ、どう管理するかは、さらに別の問題です。  
php-fpm は、その問題を引き受けています。

**新人**  
この本では、「PHP を動かすもの」より「PHP 実行側の管理者」として見るのが大事でしたね。

**ベテラン**  
その通りです。  
php-fpm の本質は、FastCGI の向こう側にある PHP 実行プロセスを、常駐・待機・管理可能な単位として成立させることにあります。

**運用の人**  
ここが今の現場にとって大きいんです。  
Web サーバーと PHP 実行側を別々の対象として見られる。  
HTTP の入口は入口として、PHP 側の待機数や状態は別で見る。  
その整理が、日常運用にもトラブルシュートにも効いてきます。

## なぜ php-fpm は「自然」だったのか

**新人**  
ここまで来ると、php-fpm が自然だった理由を自分でも少し言えそうです。  
でも、最後にちゃんと言葉にしたいです。

**ベテラン**  
よいですね。  
では順番に整理しましょう。

CGI は、外に渡す構図はよかったが、毎回起動するのがつらかった。  
mod_php は、Web サーバーの中で完結する便利さがあったが、Web サーバー側と PHP 側が近すぎた。  
Apache MPM を見ると、そもそも Web サーバー側の並行処理の話と PHP 実行側の話は別レイヤーだとわかる。  
FastCGI は、外に分ける構図を保ちながら毎回起動の問題を減らした。  
php-fpm は、その分離された PHP 実行側を常駐・管理可能な形にした。

この流れをまとめると、php-fpm は単なる流行ではありません。  
**待機と実行、Web サーバー側と PHP 側、受け口と処理役という責務の違いに沿って、自然に整理された構成** だと言えます。

**新人**  
つまり、「たまたま今の標準になった」のではなく、「それまでの問題を順に整理していくと、かなり自然にここへ来る」ということなんですね。

**ベテラン**  
まさにそれです。  
だから本書のタイトルにあるように、「php-fpm を前提として使っている人が、その前提を学び直す」ことに意味があるんです。

## この本で持ち帰ってほしいこと

**新人**  
最後に、この本を読み終えて読者が持ち帰るべきことは何でしょう。

**ベテラン**  
大きく三つあります。

一つ目は、PHP の知識と PHP の実行形態の知識は別だということです。  
コードが書けても、どこでどう動いているかが見えないと、構成の説明はしにくい。

二つ目は、昔の方式を雑学として見るのではなく、現在の構成を理解するための比較軸として見ることです。  
CGI、mod_php、FastCGI は、今の php-fpm を理解するための前史です。

三つ目は、責務分離の地図を持つことです。  
Web サーバー側の話なのか。  
PHP 実行側の話なのか。  
接続待機の話なのか。  
アプリケーション処理の話なのか。  
そこを見分けられるようになることが大切です。

**運用の人**  
現場で言えば、「今の構成を説明できるようになること」が大きいですね。  
Nginx や Apache の後ろで php-fpm を動かしているとき、それがどういう責務分担の上に成り立っているかを言葉にできる。  
それだけでも構成の見え方がかなり変わります。

**新人**  
なるほど。  
この本は、新しい技術を覚える本というより、「今ふつうに使っている構成を、ちゃんと説明できるようになる本」だったんですね。

**ベテラン**  
その理解で締めてよいと思います。

### 整理

本書で見てきた流れは、次のように整理できます。

- **CGI**  
  Web サーバーが外部プログラムに処理を渡す、わかりやすい構図を持っていた  
  ただし、1リクエストごとに毎回起動するため、繰り返しに弱かった

- **mod_php**  
  Apache の中で PHP を動かすことで、構成の短さ、導入のしやすさ、実用性を実現した  
  ただし、Web サーバー側と PHP 実行側が近すぎて、後には分離しにくさが目立つようになった

- **Apache MPM**  
  Web サーバー側の並行処理モデルであり、PHP の実行形態そのものではなかった  
  それでも mod_php では両者が近づいて見えやすかった

- **FastCGI**  
  CGI の「外に渡す」構図を保ちながら、毎回起動するつらさを減らした  
  Web サーバーとアプリケーション実行側の責務分離を実用的にした

- **php-fpm**  
  分離された PHP 実行側を、常駐・待機・管理可能な単位として成立させた  
  Web サーバーと PHP 側を別々の管理対象として扱いやすくした

したがって、php-fpm は単なる流行ではなく、**Web サーバー側と PHP 実行側、待機と実行、受け口と処理役という責務の違いを整理した結果として自然に見えてくる構成** です。

### この章の核心

php-fpm は、CGI のわかりやすい境界、mod_php の実用性、FastCGI の責務分離を受け継ぎながら、分離された PHP 実行側を常駐・管理可能にした構成です。  
だからこそ、php-fpm は偶然広まったのではなく、歴史と責務分解の流れから見て自然な選択だったと言えます。

## まとめ

- CGI は外に渡す構図を見せてくれたが、毎回起動するつらさがあった
- mod_php は Apache の中で PHP を動かす便利さを持っていたが、後には一体化が重くなった
- Apache MPM は Web サーバー側の並行処理モデルであり、PHP の実行形態とは別レイヤーの話だった
- FastCGI は、外に分ける構図を保ちながら毎回起動の問題を減らし、責務分離を実用化した
- php-fpm は、その分離された PHP 実行側を常駐・管理可能な形にし、現在の構成として自然な整理を与えた

これで、本書の主な内容はひととおり終わりです。  
次の書籍で ReactPHP や FrankenPHP、あるいはより広い非同期ランタイムの話を読むときも、この本で作った地図が比較軸として役立つはずです。  
今ふつうに使っている php-fpm 周辺の構成を説明できるようになることが、その先の学びの土台になります。
