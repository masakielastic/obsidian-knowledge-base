---
title: "FastCGI は何を分離したのか"
---

前の章では、Nginx が PHP を実行するものではなく、HTTP の受け口としての Web サーバーであること、そして php-fpm と役割分担がきれいに揃いやすいことを見ました。

その流れを踏まえると、次に重要になるのが **FastCGI** です。  
なぜなら、Nginx と php-fpm の組み合わせを単なる定番構成としてではなく、責務分離の結果として理解するには、その途中にある FastCGI の意味を押さえる必要があるからです。

FastCGI は、単に CGI を少し速くした仕組み、とまとめられがちです。  
ですが本書では、そういう理解では足りないと考えます。  
FastCGI の重要さは、**毎回起動するつらさを減らしたこと** と同時に、**Web サーバーとアプリケーション実行側を分けたまま実用的に運用できるようにしたこと** にあります。

この章では、その「分離」が何を意味していたのかを整理します。

## CGI の何が残り、何が変わったのか

**新人**  
FastCGI という名前を見ると、やっぱり最初に思うのは「CGI を速くしたもの」です。  
それだと不十分なんですよね。

**ベテラン**  
はい。  
間違いではありませんが、それだけだと本質が抜けます。  
まず CGI を思い出しましょう。  
CGI は、Web サーバーがリクエストを受けて、外部プログラムに処理を渡す構図でした。

**新人**  
はい。  
Web サーバーとアプリケーションの境界が見えやすい、という話でした。

**ベテラン**  
その通りです。  
つまり CGI の良さは、**外に渡す** という構図にありました。  
FastCGI は、その構図自体は捨てません。  
捨てたのは、**1リクエストごとに毎回新しく起動する** という部分です。

**新人**  
なるほど。  
外に渡すという考え方は残す。  
でも毎回ゼロから起動するのはやめる。  
そこが変化なんですね。

**運用の人**  
現場感覚でも、その整理がいちばんしっくりきます。  
CGI の構図は悪くない。  
でも、そのたびに作業者を新しく呼ぶのはつらい。  
だから、作業者は外に待たせておいて、そこへ仕事を流したい。  
FastCGI はその方向ですね。

## 毎回起動しない、とはどういうことか

**新人**  
ここをもう少し具体的に聞きたいです。  
「毎回起動しない」というのは、どう見ればいいですか。

**ベテラン**  
CGI では、リクエストが来るたびに、そのリクエストのためだけに外部プログラムを起動していました。  
処理が終われば、その実行も終わります。

一方で FastCGI では、**あらかじめ動いている実行側** に対して仕事を渡します。  
つまり、処理役はすでに外で待っているわけです。

**新人**  
たしかに、それなら毎回ゼロから始める必要がありませんね。

**ベテラン**  
そうです。  
この違いは大きいです。  
Web は同じような処理を何度も繰り返します。  
そのたびに実行環境を一から起こすのではなく、待機中の実行側に仕事を回すほうが自然になります。

**運用の人**  
現場では、これでやっと「繰り返しに向いた構成」になります。  
単発なら CGI の素朴さでもよい。  
でも繰り返しが増えると、毎回起動するモデルは苦しくなる。  
FastCGI はそこを改善しています。

**新人**  
つまり FastCGI は、CGI の構図を壊さずに、Web らしい繰り返しに耐えやすくしたわけですね。

**ベテラン**  
その理解でよいです。

## 分離されたのは、速度だけではない

**新人**  
ここまで聞くと、やっぱり「速くするための工夫」に見えるんですが、それだけではないんですよね。

**ベテラン**  
そうです。  
FastCGI の価値を速度だけで見ると、本書の中心が抜けます。  
本当に大きいのは、**Web サーバーの責務とアプリケーション実行側の責務を分けたまま、現実的に運用できるようにしたこと** です。

**新人**  
「分けたまま」が大事なんですね。

**ベテラン**  
はい。  
mod_php では、Web サーバーの中に PHP が入り込んでいました。  
それは便利でしたが、Web サーバー側の都合と PHP 実行側の都合が近づきやすい構成でもありました。

FastCGI では、PHP を Web サーバーの外に置いたまま実用化できます。  
つまり、  
**HTTP の受け口は Web サーバー**、  
**アプリケーション実行は外の実行側**  
という分担を保ちやすくなるわけです。

**運用の人**  
現場でも、これがかなり重要です。  
Web サーバーは入口として振る舞う。  
アプリケーション実行側は別で待機していて、そこに仕事を流す。  
この構図だと、どちらの責務なのかを切り分けやすいんです。

**新人**  
なるほど。  
FastCGI は「速さの改善」より、「責務分離の実用化」として見たほうがよさそうです。

**ベテラン**  
まさにそれです。

## Web サーバーは何を手放し、何を残したのか

**新人**  
FastCGI にすると、Web サーバーの役割はどう見えるようになりますか。

**ベテラン**  
まず、Web サーバーが **PHP を自分の中で直接実行する必要がなくなる** という点が大きいです。  
つまり、PHP アプリケーション処理そのものを外へ渡しやすくなります。

その一方で、Web サーバーの責務そのものが消えるわけではありません。  
HTTP の受け口であること、リクエストを読み、必要に応じて振り分けること、そうした仕事は Web サーバー側に残ります。

**新人**  
つまり、Web サーバーは仕事をやめるのではなく、仕事を絞るんですね。

**ベテラン**  
その言い方がよいです。  
Web サーバーは Web サーバーとしての仕事に寄り、アプリケーション実行は外に任せる。  
その整理が見えやすくなります。

**運用の人**  
この見え方が、後の Nginx + php-fpm の自然さにもつながりますね。  
Nginx は HTTP の入口に集中する。  
PHP 実行側は別で待つ。  
FastCGI はその間の橋渡しの仕組みとして理解しやすいです。

## FastCGI は Nginx と php-fpm の間をどうつないでいるのか

**新人**  
ここで Nginx と php-fpm の話につながるんですね。

**ベテラン**  
そうです。  
前の章で見たように、Nginx は PHP を実行するものではありません。  
HTTP の受け口として働き、PHP で処理すべきものを外に渡します。  
その「外に渡す」構図を実現するうえで、FastCGI は非常に重要です。

**新人**  
つまり、Nginx が受けた仕事を、FastCGI を通じて php-fpm 側へ回すわけですね。

**ベテラン**  
そうです。  
もちろん本書では通信プロトコルの詳細には入りません。  
ここで大事なのは、  
**Web サーバー側と PHP 実行側のあいだに、きちんとした受け渡しの境界がある**  
ということです。

**運用の人**  
現場でも、Nginx と php-fpm を別の部品として見られるのは大きいです。  
そのあいだに受け渡しの仕組みがあるからこそ、HTTP の受け口と PHP 実行側を別々に管理しやすくなります。

**新人**  
なるほど。  
FastCGI は単に速い仕組みではなく、Nginx と php-fpm のような役割分担を成立させる境界でもあるんですね。

**ベテラン**  
その理解がこの章ではいちばん重要です。

## では、FastCGI があれば十分なのか

**新人**  
ここまで聞くと、FastCGI でかなり整理できているように見えます。  
では、これで PHP 側の問題は全部解決したんでしょうか。

**ベテラン**  
そこは一歩止まって考えたいところです。  
FastCGI は、**分離の構図** を与えてくれます。  
でも、その分離された PHP 実行側を **どう待機させ、どう増減させ、どう管理するか** までは、それだけでは決まりません。

**新人**  
ああ、たしかに。  
外に分けることと、その外側をどう運用するかは別問題ですね。

**運用の人**  
現場ではそこがいちばん実務的です。  
何個待たせるのか。  
どう管理するのか。  
どう監視するのか。  
FastCGI だけでは、その運用の単位までは与えてくれません。

**新人**  
つまり FastCGI は、境界を作る。  
でも、その境界の向こう側の PHP 実行側をまとめて運用可能にするには、さらに仕組みが必要なんですね。

**ベテラン**  
その通りです。  
そして、それを担うのが次章で扱う php-fpm です。

## FastCGI をどう位置づければよいか

**新人**  
この章の内容を短く言うと、FastCGI はどんな存在なんでしょう。

**ベテラン**  
こう言うのがよいでしょう。  
FastCGI は、CGI の「外に渡す」というわかりやすい構図を保ちながら、毎回起動するつらさを減らし、Web サーバーとアプリケーション実行側を分けたまま実用的に運用できるようにした仕組みです。

**新人**  
かなり整理されました。  
単なる高速化ではなく、責務分離を実務の構成にした存在なんですね。

**ベテラン**  
その理解で十分です。  
この視点があると、次の php-fpm も「FastCGI で動く PHP」ではなく、「分離された PHP 実行側の管理者」として見やすくなります。

### 整理

FastCGI は、CGI の「Web サーバーが外部の実行側に処理を渡す」という構図を引き継ぎながら、**1リクエストごとに毎回新しく起動するモデル** をやめ、**あらかじめ待機している実行側に処理を渡す** 方向へ進めた仕組みです。

この変化によって、次の分離が見えやすくなります。

- Web サーバーの責務とアプリケーション実行側の責務
- HTTP の受け口と PHP の実行処理
- リクエストを受けて振り分ける側と、アプリケーション処理を担う側

重要なのは、FastCGI の価値が単なる速度改善ではないことです。  
FastCGI は、**外に分けるという構図を保ったまま、それを現実の運用で使いやすい形にした** ところに意味があります。

また、Nginx と php-fpm の組み合わせを考えるときも、FastCGI はそのあいだにある受け渡しの境界として理解できます。  
Nginx は HTTP の受け口として振る舞い、PHP 実行側は外で待機する。  
FastCGI は、その責務分担を成立させる仕組みです。

ただし、FastCGI はあくまで **分離の構図** を与えるものであり、PHP 実行側そのものの管理を完結させるものではありません。  
その管理の問題は、次章の php-fpm へとつながります。

### この章の核心

FastCGI は、CGI のわかりやすい境界を保ちながら、毎回起動するつらさを減らし、Web サーバーとアプリケーション実行側を分けたまま実用的に運用できるようにしました。  
その本質は単なる高速化ではなく、責務分離を現実の構成として成立させたことにあります。

## まとめ

- FastCGI は、CGI の「外に渡す」という発想を引き継いでいる
- CGI との大きな違いは、毎回新しく起動せず、待機している実行側に処理を渡す点にある
- FastCGI の価値は、単なる高速化ではなく、Web サーバーとアプリケーション実行側の責務分離を実用化したことにある
- Nginx と php-fpm の組み合わせも、FastCGI があることで受け口と PHP 実行側の境界が見えやすくなる
- ただし FastCGI は PHP 実行側の管理そのものを完結させるわけではなく、その先に php-fpm の役割がある

次章では、php-fpm が何を管理しているのかを見ていきます。  
FastCGI が作った境界の向こう側で、待機中の PHP 実行プロセスをどのようにまとめ、管理し、運用可能な単位にしているのかを整理します。
