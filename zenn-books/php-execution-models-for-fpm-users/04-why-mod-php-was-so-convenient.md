---
title: "mod_php はなぜ便利だったのか"
---

前の章では、CGI がとても素朴でわかりやすい実行形態である一方で、1リクエストごとに外部プログラムを起動するため、繰り返しに弱いという話をしました。

その流れを受けて、この章では **mod_php** を見ます。  
mod_php は、今では昔の方式として語られがちです。  
ですが、それを単に「古い方式」として片づけると、なぜ長く広く使われたのかが見えなくなります。

mod_php は、当時の Web と PHP の組み合わせとしてかなり自然で、便利で、実用的でした。  
この章では、その便利さがどこから来ていたのかを整理します。

## mod_php は何を変えたのか

**新人**  
CGI の話を聞いたあとだと、mod_php は「毎回外部プログラムを起動しない方式」なんだろうな、という感じはします。

**ベテラン**  
その理解はよい入口です。  
CGI では、Web サーバーがリクエストを受け、そのたびに外部プログラムを起動していました。  
一方で mod_php は、**Apache の中に PHP を組み込んで動かす** 方式です。

**新人**  
つまり、外に投げないんですね。

**ベテラン**  
そうです。  
PHP を外部プログラムとして毎回呼び出すのではなく、最初から Web サーバー側に PHP の実行機能を持たせるわけです。  
だから、CGI のような「毎回起動する」という感覚とはかなり違います。

**運用の人**  
現場感覚で言えば、Web サーバーと PHP がかなり近いです。  
別プロセスの PHP 実行系に受け渡すのではなく、Apache 側でそのまま PHP を実行する。  
構成としてはかなり素直に見えます。

**新人**  
なるほど。  
CGI が「受けて、外に渡す」なら、mod_php は「受けて、そのまま中で処理する」んですね。

**ベテラン**  
その対比でまずは十分です。  
この違いが、便利さの出発点になります。

## なぜそれが便利だったのか

**新人**  
でも、「中で動く」ことがそんなに便利なんですか。

**ベテラン**  
便利です。  
理由は一つではありませんが、まず大きいのは、**外部プログラムを毎回起動しなくてよい**ことです。

**新人**  
CGI のつらさが、そのまま改善される感じですね。

**ベテラン**  
はい。  
毎回外へ渡して、毎回起動して、毎回終わる。  
その流れを避けられるだけでも、かなり実用的になります。

さらに、Web サーバーの中で完結するぶん、構成が見えやすい。  
「Apache が PHP を処理する」という理解で進められるので、最初の導入としても扱いやすいです。

**新人**  
たしかに、別の口で php-fpm を立てて、Web サーバーと通信させて、みたいな構成より、だいぶ単純そうです。

**運用の人**  
そうなんです。  
構成要素が少ないというのは、それだけで強いです。  
当時の感覚でいえば、Apache を入れて、PHP モジュールを有効にして、そのまま PHP のページを動かせる。  
これはかなり手触りがよかったはずです。

**新人**  
いまの感覚だと、Web サーバーと PHP 実行系を分けるのが普通に見えますけど、当時は一体になっているほうが自然だったわけですね。

**ベテラン**  
そうです。  
今の標準感覚から過去を見てしまうと、「なぜそんなに密結合だったのか」と感じやすいのですが、当時はむしろそれが実用的だったわけです。

## 「Apache の中で PHP が動く」の自然さ

**新人**  
でも、どうしてそんなに自然だったんでしょう。  
今の私の感覚だと、Web サーバーと PHP を分けたほうが整理されていてきれいに見えます。

**ベテラン**  
そこは時代背景も大きいです。  
まず、Apache 自体が非常に強い存在でした。  
その Apache の中で PHP がそのまま動くなら、Web アプリケーションを作るための流れがかなり短くなります。

**新人**  
Apache が中心にあって、その拡張として PHP を使う感じですか。

**ベテラン**  
そうです。  
PHP は、Web のための言語として広まっていった側面が強いです。  
その文脈では、「Web サーバーの中でそのまま PHP が動く」というのは、とても自然な配置だったんです。

**運用の人**  
実務でもわかりやすいですよ。  
Apache がリクエストを受ける。  
PHP ファイルが来たら、その場で PHP を実行する。  
結果を返す。  
流れがかなり短いです。

**新人**  
たしかに、CGI よりも「Web の中で完結している感じ」がありますね。

**ベテラン**  
はい。  
CGI は Web サーバーと外部プログラムの連携として理解する必要がありますが、mod_php は「Apache の機能として PHP を使う」と考えやすい。  
そのわかりやすさも大きかったんです。

## 導入しやすさは大きな価値だった

**新人**  
便利さというと性能の話を想像していましたけど、導入のしやすさも大きかったんですね。

**ベテラン**  
大きかったです。  
技術は、理屈の美しさだけで広まるわけではありません。  
使い始めやすいこと、構成がわかりやすいこと、動かしやすいことは、とても重要です。

**新人**  
PHP のファイルを置いて、Apache でそのまま動かせる、みたいな感覚は強そうです。

**運用の人**  
現場感覚でも、その強みはかなり大きいです。  
別プロセスの管理が要らない。  
Web サーバーと PHP の通信経路を別に考えなくてよい。  
最初の構成として把握しやすい。  
これは教育コストや運用コストの面でも有利です。

**新人**  
「構成が少ない」というのは、単に楽というだけじゃなくて、理解しやすいということでもあるんですね。

**ベテラン**  
その通りです。  
しかも、当時は今ほど「責務分離を厳密にしよう」という発想が前面に出ていたわけでもありません。  
まずは Web アプリケーションを動かせることが重要でした。  
その意味で mod_php はかなり噛み合っていたんです。

## CGI と比べたときの強み

**新人**  
ここまで聞くと、CGI より便利なのはかなり納得できます。  
比べると、どう整理できますか。

**ベテラン**  
大きな違いはこうです。

CGI は、Web サーバーが外部プログラムを起動して処理を任せます。  
mod_php は、Web サーバーの中で PHP を実行します。

その結果として、

- 毎回外部プログラムを起動する負担を避けやすい
- Web の処理の流れが短くなる
- 構成要素が少なく見えやすい
- 導入や理解のハードルが下がる

という強みが出ます。

**新人**  
つまり、CGI の「外に渡す」構図より、mod_php の「中でそのまま処理する」構図のほうが、当時の用途には合っていたんですね。

**ベテラン**  
そうです。  
しかも重要なのは、これは単なる性能比較ではないということです。  
理解しやすさ、導入しやすさ、構成の短さも含めて便利だったんです。

**運用の人**  
実務では、その総合力が効きます。  
理論的に完全分離されていることより、すぐ動くこと、把握しやすいこと、運用しやすいことが勝つ場面は多いですからね。

## mod_php は「昔の雑な方式」ではない

**新人**  
ここまで聞くと、mod_php に対する見方が少し変わりますね。  
正直、私は「昔の雑なやり方」という印象を持っていました。

**ベテラン**  
その印象を持つ人は多いです。  
でも、この本ではそこを雑に処理しません。  
mod_php は、当時の Web と PHP の関係を考えると、かなり筋の通った方式でした。

**新人**  
筋が通っていた、ですか。

**ベテラン**  
はい。  
Web サーバーで HTTP を受け、そのまま PHP を実行して、すぐにレスポンスを返す。  
この流れは、Web アプリケーションの普及期に非常に相性がよかったんです。

問題は、あとで見るように、この便利さがそのまま将来の制約にもつながっていくことです。  
でも、それは最初から雑だったという意味ではありません。

**運用の人**  
現場でも、後から見て「古い」と言うのは簡単です。  
でも、その時代に何が求められていて、どこが強みだったのかを見ないと、公平な比較にはなりません。

**新人**  
なるほど。  
この章では、まず便利だった理由をしっかり認めることが大事なんですね。

**ベテラン**  
その通りです。  
強みをきちんと理解しておかないと、なぜ長く使われたのかも、なぜ後で限界が見えてきたのかも、どちらも説明しにくくなります。

## 便利さは、どんな前提の上に成り立っていたのか

**新人**  
ここまでで、mod_php が便利だった理由はかなり見えました。  
でも、次の章では限界の話になるんですよね。  
ということは、この便利さには前提があったんですか。

**ベテラン**  
その通りです。  
mod_php の便利さは、Apache の中で PHP を動かすことが自然で、実用的で、運用上も扱いやすい、という前提の上に成り立っています。

ただ、その前提がずっと同じとは限りません。  
Web サーバー側の並行処理の考え方や、責務分離の要求や、構成の作り方が変わってくると、「中で一体化していること」が逆に整理しにくさを生む場面が出てきます。

**新人**  
便利さの条件が変わると、見え方も変わるわけですね。

**ベテラン**  
そうです。  
だから次の章では、「mod_php が悪かった」のではなく、「便利だったものが、どこで限界を迎えたのか」を見ていきます。

### 整理

mod_php は、Apache の中で PHP を動かす方式です。  
CGI のように外部プログラムを毎回起動するのではなく、Web サーバーの中でそのまま PHP を実行します。

この方式が便利だった理由は、主に次の点にあります。

- 外部プログラムを毎回起動する負担を避けやすい
- Web サーバーの中で処理が完結し、流れが短い
- 構成要素が少なく、把握しやすい
- 導入しやすく、Web アプリケーション用途と相性がよい
- Apache が中心だった時代には、配置としてかなり自然だった

つまり mod_php の強みは、単なる速度の話だけではありません。  
**構成の短さ、導入のしやすさ、理解しやすさ、実用性の高さ** が組み合わさっていたことが大きいのです。

### この章の核心

mod_php は、Apache の中で PHP をそのまま動かせることで、CGI より短く、わかりやすく、導入しやすい構成を実現しました。  
だからこそ長く広く使われたのであり、後から見て単純に「昔の雑な方式」と片づけるべきではありません。

## まとめ

- mod_php は、Apache の中で PHP を動かす方式である
- CGI のように毎回外部プログラムを起動しないため、実用上の負担を減らしやすかった
- mod_php の便利さは、構成の短さ、導入のしやすさ、理解しやすさにもあった
- Apache が中心だった時代には、Web サーバーの中で PHP を動かす構成はかなり自然だった
- mod_php は単なる旧式ではなく、当時の条件に強く噛み合った実用的な方式だった

次章では、mod_php の強みを踏まえたうえで、なぜそれでも限界が見えてきたのかを整理します。  
便利だった構成が、どのような条件の変化によって整理しにくくなっていったのかを見ていきます。
