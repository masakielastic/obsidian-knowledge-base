---
title: "mod_php はなぜ限界を迎えたのか"
---

前の章では、mod_php がなぜ便利だったのかを見ました。  
Apache の中で PHP をそのまま動かせることは、CGI より短く、導入しやすく、理解しやすい構成につながっていました。

ただし、便利だったことと、ずっと最適だったことは同じではありません。  
ある時代には自然だった構成も、前提が変われば別の見え方をします。

この章では、mod_php を「悪い方式」として断罪するのではなく、**なぜ便利だった構成が次第に限界を見せるようになったのか** を整理します。  
焦点になるのは、責務分離、運用、そして Web サーバー側の並行処理モデルとの関係です。

## 限界とは、何が苦しくなったということか

**新人**  
前の章で mod_php の便利さはかなり納得できました。  
だからこそ逆に気になります。  
そんなに便利だったのに、なぜ主流ではなくなっていったんですか。

**ベテラン**  
まず大事なのは、ここでいう「限界」は、ある日突然使えなくなったという意味ではない、ということです。  
動くものが急に壊れたわけではありません。  
むしろ、便利さの前提がだんだん揺らいでいった、と考えるほうが正確です。

**新人**  
便利さの前提、ですか。

**ベテラン**  
はい。  
mod_php は、Apache の中で PHP を動かす方式でした。  
これは構成が短く、理解もしやすい。  
ただその反面、**Web サーバーの都合と PHP 実行側の都合がかなり近い場所で結びつく** ことになります。

**運用の人**  
現場感覚で言うと、最初はそれが楽なんです。  
でも、運用の規模が上がったり、Web サーバー側と PHP 側を別々に調整したくなったりすると、近すぎることがかえって扱いづらくなります。

**新人**  
なるほど。  
「近いから便利」だったのが、「近いから切り分けにくい」に変わってくるんですね。

**ベテラン**  
その通りです。  
この章では、その変化を順番に見ていきます。

## Web サーバーと PHP の責務を分けたくなる

**新人**  
責務分離、という言葉が出てきました。  
ここでは何を分けたくなるんですか。

**ベテラン**  
大きくいえば、**HTTP を受ける側の責務** と **PHP を実行する側の責務** です。

**新人**  
mod_php だと、その二つが Apache の中でかなり近いんですよね。

**ベテラン**  
そうです。  
もちろん概念としては別です。  
Apache は HTTP を受け、PHP はアプリケーションの処理をします。  
でも実際の構成としては、同じ場に乗っています。

この状態では、Web サーバーとしての都合と、PHP 実行系としての都合を、別々に扱いにくい場面が出てきます。

**新人**  
たとえば、どんな場面ですか。

**ベテラン**  
たとえば、Web サーバー側の並行処理の方針と、PHP の実行のさせ方を切り離して考えたい場面です。  
あるいは、PHP だけを別の単位で増減させたい、監視したい、再起動したい、といった場面ですね。

**運用の人**  
実務でも、  
「HTTP の受け口としての Web サーバー」  
と  
「PHP の実行プロセス」  
を別々に見たいことがあります。

ログ、監視、再起動、負荷の見方、調整の仕方。  
そういうものが全部ひとつの塊に見えると、だんだん扱いにくくなるんです。

**新人**  
便利な一体化が、後になると管理単位の粗さになるわけですね。

**ベテラン**  
はい。  
一体化は、初期には簡潔さですが、規模が上がると分離のしにくさにもなります。

## 「Apache の中にいる」ことの重さ

**新人**  
でも、Apache の中にいること自体は、前の章ではかなり自然に見えました。  
それがどうして重さになるんでしょう。

**ベテラン**  
ポイントは、PHP が Apache の拡張として振る舞うことです。  
つまり PHP の実行が、Apache という Web サーバーのあり方に強く結びつきます。

**新人**  
独立した実行側ではなく、Apache の一部として見えるわけですね。

**ベテラン**  
その通りです。  
すると、PHP をどう動かすかという問題が、Apache の都合から完全には自由になりません。  
逆に、Web サーバー側の設計選択も、PHP をどう抱えているかの影響を受けやすくなります。

**運用の人**  
現場では、これがじわじわ効いてきます。  
Web サーバーだけを軽く入れ替えたい。  
PHP 実行側だけを別で調整したい。  
そういう発想を取りにくくなるんです。

**新人**  
たしかに、「Apache の中で PHP を動かす」が前提だと、Web サーバーと PHP を別の部品として考えにくいですね。

**ベテラン**  
そうです。  
mod_php の便利さは、Apache と PHP が密接だからこそ出ていました。  
でも、その密接さは構成の自由度を下げる面もあります。

## MPM との関係が見えてくる

**新人**  
ここで Apache MPM が関係してくるんですか。

**ベテラン**  
はい。  
ただし、この章では深入りせず、まず輪郭だけ押さえます。  
Apache MPM は、Web サーバー側の並行処理モデルの話です。  
PHP の実行形態そのものとは別レイヤーですが、mod_php では両者がかなり近く見えてきます。

**新人**  
Web サーバー側の並行処理の話が、PHP の動かし方に影響してくるんですね。

**ベテラン**  
そうです。  
Web サーバー側がどう接続をさばくかという問題と、PHP をどう安全に動かすかという問題が、切り離しにくくなります。  
ここで、「Web サーバーの都合」と「PHP 実行側の都合」を別々に設計したい、という欲求が強くなります。

**運用の人**  
このあたりから、  
「Apache の並行処理モデルの話なのか」  
「PHP の実行形態の話なのか」  
を混同すると話が見えにくくなりますね。

**新人**  
なるほど。  
mod_php の限界というのは、PHP 単体の問題ではなくて、Apache 側との結びつきの強さが効いてくるんですね。

**ベテラン**  
その理解でよいです。  
後の章で MPM を個別に扱うのは、まさにその混同をほどくためです。

## 運用上の都合も変わっていく

**新人**  
責務分離や MPM の話はわかってきました。  
運用上の都合というのは、もう少し具体的にいうと何ですか。

**ベテラン**  
一言でいえば、**調整したい単位が変わってくる** ということです。

**新人**  
調整したい単位。

**ベテラン**  
はい。  
小さな構成なら、「Apache に PHP が乗っている」で十分なことも多いです。  
でも、規模が大きくなると、Web サーバー側と PHP 側で別々に見たいものが増えます。

- どこで負荷を受けているのか
- どのプロセスを増減させたいのか
- どこを再起動したいのか
- どこを監視したいのか

こうした単位が、ひとつの塊だと見えにくくなります。

**運用の人**  
たとえば、PHP 側の実行数だけを調整したいとか、PHP 側だけを再読込したいとか、そういう発想ですね。  
Web サーバーと PHP 実行系が別であれば、その操作もしやすくなります。

**新人**  
つまり、機能として動くかどうかだけじゃなくて、運用するときの見え方や触り方が問題になるんですね。

**ベテラン**  
その通りです。  
技術の主流が変わるときは、性能だけでなく、運用しやすさや責務分離のしやすさが強く効きます。

## Web サーバーの選び方も変わっていく

**新人**  
mod_php が Apache と強く結びついているなら、Web サーバーの選び方にも影響しますか。

**ベテラン**  
大きく影響します。  
mod_php の世界では、Apache の中で PHP を動かすことが前提になりやすい。  
すると、「Web サーバーは Apache であること」がかなり強い前提になります。

**新人**  
ああ、PHP の実行形態が Web サーバーの選択と結びついてしまうんですね。

**ベテラン**  
そうです。  
でも、Web サーバーの役割と PHP 実行側の役割を分けられれば、その前提を弱められます。  
Web サーバーは Web サーバーとして選ぶ。  
PHP 実行側は PHP 実行側として管理する。  
この構図のほうが、後の時代には自然になっていきます。

**運用の人**  
現場でも、その分離は大きいです。  
HTTP の受け口として何を使うかと、PHP の実行をどう管理するかを別々に考えられるのは運用上かなり助かります。

**新人**  
なるほど。  
mod_php の限界は、Apache の中で完結しすぎていたことでもあるんですね。

**ベテラン**  
そう言えます。  
便利さの裏側で、選択の単位が一体化しすぎていたわけです。

## 主流交代は「敗北」ではなく整理の結果

**新人**  
ここまで聞くと、mod_php が主流ではなくなったのは、単に新しいものに負けたというより、構成の整理が進んだ結果に見えます。

**ベテラン**  
その見方が大事です。  
この本では、技術の流行を勝ち負けだけで語らないようにしたいんです。

mod_php は、便利でした。  
だから長く使われました。  
ただし、その便利さは Apache と PHP が強く結びつくことに支えられていました。  
その結びつきが、責務分離や運用の観点ではだんだん重くなった。  
だから、Web サーバーと PHP 実行側を分ける構成が自然になっていったわけです。

**新人**  
「昔は雑だったから捨てられた」ではなくて、「当時は合理的だったけれど、後の条件では別の整理のほうが自然になった」ということですね。

**ベテラン**  
まさにそれです。  
この整理ができると、次に出てくる FastCGI や php-fpm も、単なる新技術ではなく、責務分離を進めた結果として読めるようになります。

## では、何が欲しくなったのか

**新人**  
CGI の章では、「毎回起動しなくてよい仕組みが欲しくなった」という話がありました。  
mod_php の次では、何が欲しくなるんですか。

**ベテラン**  
一言でいえば、**PHP を Web サーバーの外に分けつつ、毎回起動はしない仕組み** です。

**新人**  
ああ、CGI の素朴さと mod_php の実用性の、別の取り方みたいですね。

**ベテラン**  
そうです。  
外に分けたい。  
でも毎回起動は避けたい。  
さらに、PHP 実行側を独立した単位として管理したい。  
この欲求が、FastCGI やその先の php-fpm の方向につながっていきます。

**運用の人**  
現場でも、その方向は自然です。  
Web サーバーは Web サーバーとして動かし、PHP の実行側は別で待たせて管理する。  
そのほうが役割分担も見えやすいですからね。

**新人**  
なるほど。  
mod_php の限界は、便利だった構成が、責務分離や運用の観点では近すぎる構成になっていた、ということなんですね。

**ベテラン**  
その理解で大丈夫です。

### 整理

mod_php は Apache の中で PHP を動かすことで、構成の短さと導入のしやすさを実現しました。  
ただし、その便利さは Web サーバーと PHP 実行側が強く結びつくことの上に成り立っていました。

その結果、次のような点が次第に苦しくなっていきます。

- HTTP を受ける側と PHP を実行する側を別々に扱いにくい
- Web サーバー側の設計と PHP 側の実行形態が強く結びつく
- Apache MPM のような Web サーバー側の並行処理モデルとの関係が重くなる
- 監視、再起動、調整の単位を分けにくい
- Web サーバーの選択と PHP 実行の構成が一体化しやすい

つまり mod_php の限界とは、性能だけの問題ではありません。  
**便利だった一体化が、後には責務分離しにくい構成として見えてきたこと** が重要です。

### この章の核心

mod_php は便利でしたが、その便利さは Apache と PHP が近すぎる構成に支えられていました。  
そのため、Web サーバー側と PHP 実行側を別々に設計し、運用したいという要求が強まるにつれて、次第に限界が見えやすくなっていきました。

## まとめ

- mod_php の限界は、動かなくなったことではなく、便利さの前提が揺らいだことにある
- Apache の中で PHP を動かす構成は、Web サーバー側と PHP 実行側を分けにくい
- 責務分離、監視、再起動、調整の単位を分けたい要求が強くなると、一体化が重く見えてくる
- Apache MPM のような Web サーバー側の並行処理モデルとの関係も、mod_php では切り離しにくい
- 主流交代は mod_php の敗北ではなく、責務分離を進めた構成のほうが自然になった結果と見るべきである

次章では、Apache MPM を取り上げます。  
ここで Web サーバー側の並行処理モデルを切り出して見ることで、PHP の実行形態との混同をほどき、mod_php の限界がどの層の問題だったのかを整理していきます。
