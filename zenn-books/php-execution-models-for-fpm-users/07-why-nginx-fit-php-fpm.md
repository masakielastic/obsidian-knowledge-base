前の章では、Apache MPM が Web サーバー側の並行処理モデルの話であり、PHP の実行形態そのものではないことを整理しました。  
この区別が見えてくると、次に気になるのが Nginx です。

実際の現場では、  
「PHP は php-fpm で動かす」  
「Web サーバーは Nginx を使う」  
という組み合わせをかなり自然なものとして目にします。

ただ、この組み合わせを単に「よく使われているから」と理解すると、少しもったいないです。  
本書の観点から見ると、Nginx と php-fpm の相性は、流行や雰囲気だけで説明するよりも、**責務分離の整理** として理解したほうが見通しがよくなります。

この章では、Nginx を単なる製品紹介としてではなく、**Web サーバー側の責務をどう見るか** という観点から位置づけます。

## まず、Nginx は PHP を実行するものではない

**新人**  
最初に確認したいんですが、Nginx って PHP を動かしているわけではないんですよね。

**ベテラン**  
その通りです。  
ここは最初に強く分けておきたいところです。  
Nginx は **Web サーバー** です。  
PHP を解釈して実行する役目そのものは持っていません。

**新人**  
でも現場だと、「Nginx で PHP サイトを動かしている」という言い方をよく聞きます。

**ベテラン**  
その言い方自体は実務では通じます。  
ただ、本書のように実行モデルを整理する観点では、もう少し分解して言ったほうがよいです。

Nginx は HTTP の受け口としてリクエストを受ける。  
そして、PHP で処理すべきものは php-fpm のような PHP 実行側へ渡す。  
つまり、  
**Nginx が PHP を動かすのではなく、Nginx は PHP 実行側へ仕事を渡す**  
と考えたほうが正確です。

**運用の人**  
現場でも、この区別ができるとかなり楽です。  
Nginx の設定を見直す話なのか、php-fpm 側の設定や状態を見る話なのかが分けやすくなりますからね。

**新人**  
なるほど。  
最初にそこを分けないと、「Nginx で PHP を動かす」という言い方に引っぱられて、責務が混ざってしまうんですね。

では、Nginx は何を担当していると考えればいいですか。

**ベテラン**

HTTP の受け口、ですか。

**ベテラン**

接続を受ける。  
リクエストを読む。  
必要なら静的ファイルを返す。  
そして、アプリケーション側へ渡すべきものを渡す。  
まずはこの役割に注目してください。

**新人**  
つまり、PHP を自分で抱え込むよりも、Web サーバーとしての責務をはっきり持っている感じなんですね。

**ベテラン**  
そうです。  
もちろん Web サーバーにはいろいろな機能がありますが、本書の主題に引きつけるなら、  
**Nginx は Web サーバー側の仕事を引き受けることに重心がある**  
と整理するとわかりやすいです。

**運用の人**  
現場感覚でも、その理解はかなりしっくりきます。  
Nginx 側では HTTP の入口や配信のふるまいを見る。  
PHP の実行や待機の話は php-fpm 側で見る。  
この分け方が自然です。

**新人**  
ここまで聞くと、Apache + mod_php との違いが気になります。

**ベテラン**  
その通りです。  
Apache + mod_php では、Web サーバーの中で PHP をそのまま実行できる便利さがありました。

**新人**  
でも Nginx + php-fpm では、そこが分かれるわけですね。

**ベテラン**  
はい。  
この違いを雑に「新しい vs 古い」で見るのではなく、**どこに重心を置いているかの違い** として見るのが大事です。

Apache + mod_php は、  
**中で完結する便利さ**  
に強みがありました。

一方で Nginx + php-fpm は、  
**Web サーバー側と PHP 実行側を分けて整理しやすいこと**  
に強みがあります。

**新人**

**ベテラン**

## Nginx と php-fpm の相性は、責務分担の一致として見える

**新人**  
では、Nginx と php-fpm の「相性がよい」というのは、具体的にはどういう意味ですか。

**ベテラン**  
一言でいえば、**役割分担がきれいに揃う** ということです。

**新人**  
役割分担が揃う。

**ベテラン**  
はい。  
Nginx は Web サーバーとして、HTTP の受け口や振り分けを担う。  
php-fpm は PHP 実行側として、待機中の PHP 実行プロセスを管理し、渡された処理を担当する。  
この分担がかなり明快です。

**新人**  
ああ、両方とも「自分の仕事」がはっきりしているんですね。

**ベテラン**  
その通りです。  
しかもこの分担は、本書でここまで見てきた歴史の流れとも一致します。

- CGI では外に渡す構図は見えていた
- mod_php では中で完結する便利さがあった
- FastCGI では外に分ける構図が実用的になった
- php-fpm では分けた先の PHP 実行側を管理しやすくなった

Nginx は、この「外に分ける構図」と非常に噛み合いやすい Web サーバー側の姿として見えるわけです。

**運用の人**  
現場でも、その整理は実感しやすいです。  
HTTP の入口で考えることと、PHP の実行側で考えることが別の単位になっている。

### 整理

この章で整理したいのは、Nginx と php-fpm の関係を、製品の人気ではなく責務分担として見ることです。

まず、Nginx は **PHP を実行するものではなく、Web サーバーである** という点が重要です。  
Nginx は HTTP の受け口としてリクエストを受け、必要なものを PHP 実行側へ渡します。

一方で php-fpm は、FastCGI の向こう側で **PHP 実行プロセスを待機・管理し、処理を担当する側** です。

このため、両者の関係は次のように整理できます。

- Nginx  
  HTTP の受け口として振る舞う  
  Web サーバー側の責務に集中する

- php-fpm  
  PHP 実行側を常駐・管理する  
  渡された処理を PHP 側で担当する

この組み合わせが自然に見えるのは、  
**Web サーバー側と PHP 実行側を分ける構成と、Nginx の役割の重心がよく一致しているから** です。

したがって、Nginx + php-fpm は単に「よくある構成」なのではなく、  
**責務分離の観点から見て整理しやすい構成** として理解するのがよいです。

### この章の核心

Nginx は PHP を実行するものではなく、HTTP の受け口としての Web サーバーです。  
だからこそ、PHP 実行側を外に分けて管理する php-fpm と役割分担がきれいに揃い、Nginx + php-fpm は責務分離の観点から自然な構成として広く受け入れられました。

## まとめ

- Nginx は PHP 実行系ではなく、Web サーバーである
- Nginx は HTTP の受け口としての責務に集中し、PHP の実行は php-fpm 側に渡す
- Apache + mod_php は中で完結する便利さ、Nginx + php-fpm は分けて整理できる便利さに強みがある
- Nginx と php-fpm の相性がよいのは、Web サーバー側と PHP 実行側の役割分担が明快だからである
- Nginx + php-fpm は、流行だけでなく責務分離の筋のよさから理解すると見通しがよくなる
