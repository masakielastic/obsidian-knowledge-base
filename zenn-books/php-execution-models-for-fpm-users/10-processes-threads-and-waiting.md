title: "プロセス・スレッド・接続待機の基礎"
---

ここまでの章で、CGI、mod_php、Apache MPM、Nginx、FastCGI、php-fpm を順番に見てきました。  
その流れの中で、Web サーバー側と PHP 実行側を分ける、という話も何度も出てきました。

ただ、その説明を本当に腹落ちさせるには、もう少し土台になる言葉を整理しておく必要があります。  
それが、**プロセス**、**スレッド**、そして **接続待機** です。

この章の目的は、OS の内部実装を詳しく学ぶことではありません。  
また、非同期ランタイムや event loop の詳細に踏み込むことでもありません。  
ここでやりたいのは、これまで出てきた

- Web サーバー側の並行処理
- PHP 実行側の管理
- 接続を受けること
- アプリケーションを実行すること

を、より基礎的な視点から見直し、なぜ責務分離が自然だったのかを確認することです。

**ベテラン**

でも、その言葉だけでは、何が足りないのか、どこで待っているのか、どの単位で仕事を持っているのかが見えません。

**運用の人**  
現場でも、  
「接続が多い」のか  
「PHP の処理が詰まっている」のか  
「プロセス数の見積もりが合っていない」のか  
を分けないと、対処の方向が変わってきます。

**新人**

つまり、この章は細かい用語の暗記ではなく、問題の場所を見分けるための基礎なんですね。

ここでは、まず「仕事を持つ単位」と「待つこと」と「処理すること」を分けて見ていきましょう。

## プロセスとは何として見ればよいか

**新人**  
では、まずプロセスからお願いします。  
いちばんよく聞く言葉ですが、いちばん曖昧かもしれません。

**ベテラン**  
この本では、まず **独立した実行の単位** と考えるのがよいです。  
ひとつのプログラムが動いているまとまり、と言ってもよいでしょう。

**新人**  
独立した実行の単位。  
たとえば、php-fpm のワーカープロセスとか、Apache のプロセスとか、そういうものですか。

**ベテラン**  
その理解で大丈夫です。  
重要なのは、「何かを実行しているまとまり」であり、しかもある程度独立して扱える単位だということです。

**運用の人**  
現場感覚でも、プロセスは監視や制御の単位として見えやすいですね。  
増えている、減っている、再起動した、落ちた、といった話をしやすいです。

**新人**  
なるほど。  
だから php-fpm の話でも「PHP 実行側を独立した管理対象として扱える」が重要だったんですね。

**ベテラン**  
そうです。  
プロセスとして分かれていると、少なくとも「どの実行のまとまりをどう扱うか」を考えやすくなります。

## スレッドとは何として見ればよいか

**新人**  
では、スレッドはどう違うんでしょう。  
名前は知っているんですが、プロセスとの違いをうまく説明できません。

**ベテラン**  
この本では、まず **ひとつの実行のまとまりの中で仕事を進める流れの単位** くらいに押さえるのがよいです。

**新人**  
プロセスが大きな箱で、スレッドはその中で動く流れ、という感じですか。

**ベテラン**  
はい、そのイメージで十分です。  
大事なのは、プロセスとスレッドは同じ言葉の言い換えではない、ということです。  
どの単位で仕事を持つか、どの単位で並べるか、という設計に関わります。

**運用の人**  
現場では、そこを厳密に OS 教科書的に説明する必要はありません。  
でも少なくとも、  
「プロセスで分ける世界」  
と  
「プロセスの中でスレッドも使う世界」  
では、仕事の持ち方が違う、という感覚は大事です。

**新人**  
前の Apache MPM の章で prefork と worker の違いが出てきたのも、まさにそこなんですね。

**ベテラン**  
その通りです。  
prefork はプロセス中心で仕事を持つ方向でした。  
worker はスレッドも使う方向でした。  
つまり、Web サーバー側がどう並行性を持つかという話の中で、プロセスとスレッドの違いが出てくるわけです。

## プロセスとスレッドは「待つこと」そのものではない

**新人**  
ここで少し見えてきました。  
でも、まだ「接続待機」との関係が気になります。  
プロセスとかスレッドって、待つための仕組みなんですか。

**ベテラン**  
そこは分けたほうがよいです。  
プロセスやスレッドは、**仕事を持ち、進める単位** の話です。  
一方で接続待機は、**外から何かが来るのを待つ状態や、その扱い方** の話です。

**新人**  
ああ、役割が違うんですね。

**ベテラン**  
そうです。  
たとえば Web サーバーは、接続を受け、必要に応じて待ち、読み書きし、それをどこかの処理単位に渡します。  
ここには「待つ」という性質があります。  
でも、それを持っている単位がプロセスなのかスレッドなのかは、別の設計問題です。

**運用の人**  
現場ではここがかなり大事です。  
接続が多いからといって、すぐに「PHP が遅い」とは限りません。  
逆に PHP 実行側が詰まっているのに、接続待機の話だと思ってしまうこともあります。

**新人**  
なるほど。  
「どの単位で仕事を持つか」と「何を待っているのか」は別に考えないといけないんですね。

## 接続待機とは何か

**新人**  
では、接続待機そのものはどう理解すればいいですか。

**ベテラン**  
この本では、まず **外部からの通信や入出力の準備が整うのを待つこと** と捉えれば十分です。

**新人**  
準備が整うのを待つ。

**ベテラン**  
はい。  
Web サーバーは、ただ計算だけしているわけではありません。  
接続を受ける。  
データが届くのを待つ。  
書き込めるタイミングを待つ。  
そうした「待ち」がたくさんあります。

**新人**  
たしかに、HTTP の処理って、計算だけじゃなくて通信そのものがありますね。

**ベテラン**  
その通りです。  
そして重要なのは、**待っている時間と、PHP のコードを実行している時間は同じではない** ということです。

**運用の人**  
現場で言うと、Web サーバー側は「多くの接続をどう待つか」が重要になります。  
一方で PHP 実行側は「渡された仕事をどう処理するか」が重要になります。  
この二つは連続していますが、責務としては分けたほうがよいです。

**新人**  
ここでまた、Web サーバーと PHP 実行側の分離がつながってきますね。

**ベテラン**  
そうです。  
接続待機をうまく扱うことと、PHP アプリケーションをうまく実行することは、どちらも必要ですが、同じ問題ではありません。

## Nginx や Apache の問いと php-fpm の問いは違う

**新人**  
ここまで来ると、Nginx や Apache の話と php-fpm の話が、たしかに違う種類の問いだとわかってきます。

**ベテラン**

Nginx や Apache のような Web サーバー側では、まず接続やリクエストをどう受け、どう待ち、どう振り分けるかが大きな関心になります。

一方で php-fpm 側では、渡された仕事を処理する PHP 実行プロセスをどう持ち、どう管理するかが中心になります。

**新人**  
つまり、  
Web サーバー側は「入口と待機」の問い、  
php-fpm 側は「処理役の管理」の問い、  
という感じですね。

**ベテラン**  
はい。  
もちろん実際には連携しています。  
でも、責務の地図としては分けたほうがずっと見やすいです。

**運用の人**

入口側で詰まっているのか。

それともアプリケーションの中身が重いのか。

## だから Web サーバー側と PHP 側を分けたくなる

**新人**  
ここまで来ると、なぜ分離したくなるのかがかなり見えてきました。  
でも改めて言うと、どう整理できますか。

**ベテラン**  
一言でいえば、**待つことに向いた仕事** と **処理することに向いた仕事** は、同じ単位で考えないほうが整理しやすいからです。

**新人**  
待つことに向いた仕事と、処理することに向いた仕事。

**ベテラン**  
はい。  
Web サーバーは、多数の接続を受け、待ち、必要に応じて振り分ける役割を持ちます。  
PHP 実行側は、渡されたリクエストに対してアプリケーション処理を行う役割を持ちます。

この二つを別の責務として見ると、

- Web サーバー側では接続や待機の扱いを考える
- PHP 側では実行プロセスの数や処理能力を考える

というふうに、問いを分けやすくなります。

**運用の人**  
現場では、その分離がかなり効きます。  
接続をさばく側の設計と、アプリケーション実行側の設計を同じ箱に入れてしまうと、何を増やせばよいのか、どこを監視すべきかが曖昧になりますからね。

**新人**  
なるほど。  
mod_php の限界も、ここで見直せますね。  
待つ側と処理する側が近すぎると、設計上も運用上も整理しにくいわけだ。

**ベテラン**  
まさにそれです。

### 整理

この章で整理したいのは、次の三つです。

- **プロセス**  
  独立した実行の単位として見やすいまとまりです。  
  php-fpm のワーカープロセスや Apache のプロセスのように、管理や監視の対象としても捉えやすい単位です。

- **スレッド**  
  ひとつの実行のまとまりの中で仕事を進める流れの単位です。  
  プロセスと同じ意味ではなく、仕事の持ち方や並べ方に関わります。

- **接続待機**  
  通信や入出力の準備が整うのを待つことです。  
  Web サーバーでは、この「待ち」をどう扱うかが重要になります。

ここで重要なのは、**プロセスやスレッドは仕事を持つ単位の話であり、接続待機は外部とのやり取りをどう待つかという話である** という点です。

したがって、次の区別を崩さないことが大切です。

- プロセス / スレッドの話
- 接続待機の話
- PHP のアプリケーション処理の話

この区別があると、

- Web サーバー側では接続や待機をどう扱うかを考える
- PHP 実行側では処理を担当する実行プロセスをどう管理するかを考える

という分担が自然に見えてきます。

### この章の核心

プロセスとスレッドは仕事を持つ単位の話であり、接続待機は通信や入出力をどう待つかの話です。  
この違いを分けて考えると、Web サーバー側と PHP 実行側を分離する構成が、単なる流行ではなく、待機と実行の責務の違いに沿った自然な整理だったことが見えてきます。

## まとめ

- プロセスは独立した実行の単位として見やすいまとまりである
- スレッドは、ひとつの実行のまとまりの中で仕事を進める流れの単位である
- 接続待機は、通信や入出力の準備が整うのを待つことであり、プロセスやスレッドそのものとは別の話である
- Web サーバー側では接続や待機の扱いが重要であり、PHP 実行側ではアプリケーション処理の実行が重要である
- そのため、Web サーバーと PHP 実行側を分ける構成は、待機と実行の責務の違いから見ても自然である

次章では、ここまでの流れを振り返りながら、なぜ php-fpm が自然な選択だったのかを全体として回収します。  
CGI、mod_php、Apache MPM、Nginx、FastCGI、php-fpm が、それぞれどの問題にどう応えていたのかを、最後にもう一度つなぎ直していきます。
