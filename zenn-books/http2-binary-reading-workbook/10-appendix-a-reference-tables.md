## 付録A　参照表

この付録では、本書の本文で繰り返し使った表をまとめて掲載します。  
ここに置く表は、暗記用の一覧ではありません。  
本書で重視してきたのは、**必要な場面で表に戻ること** です。  
したがって、この付録も「覚えるため」より「引くため」に使ってください。

---

### A-1. UTF-8 先頭バイト表

UTF-8 では、まず先頭バイトを見て、何バイト文字候補かを決めます。  
最初に使う表は次の通りです。

| 先頭バイト範囲 | 意味 |
| --- | --- |
| `00`–`7F` | 1バイト文字候補 |
| `C2`–`DF` | 2バイト文字候補 |
| `E0`–`EF` | 3バイト文字候補 |
| `F0`–`F4` | 4バイト文字候補 |

補足として、`80`–`BF` は continuation byte の範囲です。  
ただし、これは先頭バイトではありません。  
また、`C0`、`C1`、`F5` 以上は本書で扱う正しい UTF-8 の先頭バイト範囲には入りません。

この表で分かるのは **長さ候補** までです。  
それだけで受理できるとは限りません。  
2バイト目以降が continuation byte かどうか、さらに特例条件が必要かどうかを次の表で確認します。

---

### A-2. UTF-8 continuation byte 範囲

UTF-8 で 2バイト目以降に現れる continuation byte の範囲は次です。

| バイト範囲 | 意味 |
| --- | --- |
| `80`–`BF` | continuation byte |

ただし、この範囲に入っていれば常に受理できるわけではありません。  
`E0`、`ED`、`F0`、`F4` では 2バイト目に特例条件があります。  
したがって、本文で扱ったように、**continuation byte の形だけで通してはいけない** 場面があります。

---

### A-3. UTF-8 特例表

本書で扱った特例は次の四つです。

| 先頭バイト | 2バイト目の許容範囲 |
| --- | --- |
| `E0` | `A0`–`BF` |
| `ED` | `80`–`9F` |
| `F0` | `90`–`BF` |
| `F4` | `80`–`8F` |

この表は、本文の第3章で何度も使いました。  
意味としては次のように覚えると見通しがよくなります。

- `E0` は 2バイト目の下限が上がる
    
- `ED` は 2バイト目の上限が下がる
    
- `F0` は 2バイト目の下限が上がる
    
- `F4` は 2バイト目の上限が下がる
    

したがって、3バイト形式や 4バイト形式で continuation byte が続いているように見えても、この表を引かないと受理・不受理を誤ることがあります。

---

### A-4. UTF-8 典型例早見表

本文で繰り返し使った典型例を、短く並べておきます。

| バイト列 | 読み方 |
| --- | --- |
| `41` | 1バイト文字 1個 |
| `C3 A9` | 2バイト文字 1個 |
| `E3 81 82` | 3バイト文字 1個 |
| `C3` | 2バイト文字候補だが途中切れ |
| `E3 81` | 3バイト文字候補だが途中切れ |
| `E0 9F 80` | 不受理 |
| `E0 A0 80` | 受理 |
| `ED 9F BF` | 受理 |
| `ED A0 80` | 不受理 |
| `F0 8F 80 80` | 不受理 |
| `F0 90 80 80` | 受理 |
| `F4 8F BF BF` | 受理 |
| `F4 90 80 80` | 不受理 |

この表の使い方は、「答えを暗記すること」ではありません。  
似た例を並べて、**どこが境界なのか** を見直すために使ってください。

### A-5. HTTP/2 共通ヘッダ表

HTTP/2 の各フレームは、先頭 9バイトに共通ヘッダを持ちます。  
その内訳は次の通りです。

| バイト数 | フィールド |
| --- | --- |
| 3バイト | Length |
| 1バイト | Type |
| 1バイト | Flags |
| 4バイト | R + Stream Identifier |

合計 9バイトです。

本書では、HTTP/2 を読むとき、まず先頭 9バイトを切ることを最優先しました。  
payload 側から先に読まないこと、Type だけで済ませないことが重要でした。  
この表は、「どこに何が置かれているか」を毎回確認するための土台です。

---

### A-6. HTTP/2 Length の見方

HTTP/2 の Length は、**payload の長さ** を表します。  
共通ヘッダ 9バイトを含めた長さではありません。

| 項目 | 意味 |
| --- | --- |
| Length | payload のバイト数 |
| フレーム全体の長さ | `9 + Length` |

この一点は重要です。  
本文で扱ったように、

- Length=0 なら payload なし
    
- Length=1 なら payload 1バイト
    
- Length=2 なら payload 2バイト
    

となります。

Length をフレーム全体の長さだと誤解すると、フレーム境界が崩れます。  
したがって、HTTP/2 を読むときは、常に

1. 先頭 9バイトを切る
    
2. Length を読む
    
3. Length の値だけ payload を取る
    

という順番を守ります。

---

### A-7. HTTP/2 Type 一覧（本書で使う範囲）

本書で本文中に繰り返し出てきた Type は、次の二つです。

| Type 値 | フレーム種別 |
| --- | --- |
| `00` | DATA |
| `04` | SETTINGS |

本書ではまずこの二つに絞って扱いました。  
理由は、共通ヘッダを切る練習に集中するためです。  
Type の種類を増やしすぎると、「何のフレームか」を当てることに意識が寄りすぎて、まず境界を切るという本書の中心がぼけるからです。

したがって、この付録でも、あえて本文で使った範囲に絞っています。

---

### A-8. HTTP/2 典型例早見表

本文で扱った最小例と典型例を並べます。

| バイト列 | 読み方 |
| --- | --- |
| `00 00 00 04 00 00 00 00 00` | payload なしの SETTINGS |
| `00 00 01 00 00 00 00 00 01 FF` | payload 1バイトの DATA |
| `00 00 02 00 00 00 00 00 01 AA BB` | payload 2バイトの DATA |
| `00 00 02 00 00 00 00 00 01 FF` | Length=2 だが payload が 1バイトで途中切れ |
| `00 00 01 00 00 00 00 00 01 AA BB` | Length=1 なので `AA` が payload、`BB` は余剰 |
| `00 00 00 04 00 00 00 00 00 00 00 01 00 00 00 00 00 01 FF` | SETTINGS のあとに DATA が続く 2フレーム連結 |

この表を見るときも、答えだけでなく、**どこで 1フレームが終わるか** に注目してください。  
本書で HTTP/2 に対して一番身につけてほしかったのは、その境界感覚です。

### A-9. HPACK 形式判定の最初の目印

本書で最初に扱ったのは Indexed Header Field です。  
その判定に使った目印は次です。

| 先頭の見方 | 意味 |
| --- | --- |
| 先頭ビットが 1 | Indexed Header Field として読む |
| 先頭ビットが 1 ではない | 本書の初歩範囲では indexed 以外の別形式 |

ここで重要なのは、「値を覚える前に形式を決める」ことです。  
`82` を見てすぐ `:method: GET` と言うのではなく、まず「Indexed Header Field だ」と判定してから index を取り出します。

また、`40` のように先頭ビットが 1 でない値を見たときは、「Indexed Header Field ではない」と分かります。  
その場合、本書の範囲では static table だけでは読めないと判断しました。

### A-10. HPACK static table 抜粋（本書で使う範囲）

本書本文で繰り返し使った static table 項目を抜粋して載せます。

| Index | ヘッダー |
| --- | --- |
| 2 | `:method: GET` |
| 3 | `:method: POST` |
| 4 | `:path: /` |
| 5 | `:path: /index.html` |
| 6 | `:scheme: http` |
| 7 | `:scheme: https` |

本書では、この範囲だけでも十分に「表を引いて読む」練習ができます。  
特に `82`、`84`、`85`、`87` は何度も使いました。

ただし、この表も暗記が主目的ではありません。  
本文で一貫していたように、**index を見たら表に戻る** ことを優先してください。

### A-11. HPACK 典型例早見表

本文で扱った最小例と典型例を並べます。

| バイト列 | 読み方 |
| --- | --- |
| `82` | `:method: GET` |
| `84` | `:path: /` |
| `85` | `:path: /index.html` |
| `87` | `:scheme: https` |
| `82 84 87` | `:method: GET`、`:path: /`、`:scheme: https` |
| `82 85 87` | `:method: GET`、`:path: /index.html`、`:scheme: https` |
| `40` | Indexed Header Field ではないので、この本の初歩範囲では static だけでは読めない |
| `82 40` | `82` までは読めるが、その次は static だけでは読めない |
| `82 84 40` | `82`、`84` までは読めるが、その次は static だけでは読めない |

この表では、読める例と読めない例をあえて混ぜています。  
その理由は、「知っている値だけ拾う」のではなく、**どこまで safe に読めるか** を見極める感覚を保つためです。

### A-12. HTTP/1 と HPACK の対応メモ

本書の第8章で、HTTP/1 の見慣れた表現を足場にして HPACK の疑似ヘッダーを見ました。  
そのときの最小対応をまとめておきます。

| HTTP/1 側の意味単位 | HPACK 側の疑似ヘッダー |
| --- | --- |
| `GET` | `:method: GET` |
| `/` | `:path: /` |
| `/index.html` | `:path: /index.html` |
| `https` | `:scheme: https` |

対応例として、本文では次の二つを扱いました。

| HTTP/1 側 | HPACK 側 |
| --- | --- |
| `GET / HTTP/1.1` | `82 84 87` |
| `GET /index.html HTTP/1.1` | `82 85 87` |

ここでも、HTTP/1 をそのまま答えに使うのではなく、**HPACK 側を先に復元し、その結果を対応づける** という順番が大切でした。

### A-13. 付録全体の使い方

この付録で一番大切なのは、「表を持っていること」より「表に戻ること」です。  
本書を通して、何度も同じことをしてきました。

UTF-8 では、

- 先頭バイト表に戻る
    
- continuation byte の範囲を見る
    
- 必要なら特例表に戻る
    

HTTP/2 では、

- 共通ヘッダ表に戻る
    
- Length の意味に戻る
    
- Type 一覧に戻る
    

HPACK では、

- 先頭ビットの形式判定に戻る
    
- static table に戻る
    

つまり、表は知識の飾りではありません。  
**読む動作の途中に差し込む道具** です。
