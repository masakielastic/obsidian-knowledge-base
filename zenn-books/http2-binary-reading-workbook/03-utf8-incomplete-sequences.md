## 第3章　特例を見落とさない

第2章では、UTF-8 を読むときの基本として、次の流れを確認しました。

1. 先頭バイトを見る
    
2. 何バイト候補かを決める
    
3. 必要な continuation byte がそろっているかを確認する
    
4. そこまでを 1文字として切る
    
5. 残りの先頭へ進む
    

この流れは重要です。  
しかし、UTF-8 はこれだけでは十分ではありません。

なぜなら、UTF-8 には「3バイト形式なら何でもよい」「4バイト形式なら何でもよい」とは言えない箇所があるからです。  
先頭バイトの形が合っていて、後続バイトも continuation byte の形をしていても、それだけでは受理してはいけない場合があります。

この章では、その代表例として、まず `E0` と `ED` の特例を扱います。  
次に `F0` と `F4` の特例を扱います。

ここでの狙いは、細かい仕様の暗記ではありません。  
大切なのは、**「形だけ合っていればよい」という危ない近道をやめること** です。  
第2章で身につけた「先頭を見る」「後続バイトを確認する」に加えて、**必要なら特例表も引く** という一段深い読み方を身につけます。

---

### 3-1. continuation byte だけ見れば十分なのか

まず、第2章までの読み方を思い出します。

たとえば 3バイト文字であれば、

- 先頭バイトが `E0` から `EF`
    
- 2バイト目と 3バイト目が `80` から `BF`
    

であれば、いったん 3バイト形式の形は整っているように見えます。

しかし UTF-8 では、これだけで受理してはいけないケースがあります。  
つまり、**3バイト形式らしく見えること** と **正しい UTF-8 として受理できること** は同じではありません。

この章で扱う特例は、そのことを体で覚えるためのものです。

ここでの読み方は次のようになります。

- 先頭バイトを見る
    
- 3バイト候補または 4バイト候補だと分かる
    
- continuation byte の形を確認する
    
- さらに、先頭バイトが特例対象なら 2バイト目の範囲を確認する
    
- そこで初めて受理、不受理を決める
    

つまり、先頭バイト表だけで終わらず、必要なら特例表まで読むわけです。

---

### 3-2. `E0` の特例

最初に `E0` の例を見ます。  
次の二つを比べてください。

`E0 9F 80`

`E0 A0 80`

どちらも先頭バイトは `E0` です。  
したがって 3バイト文字候補です。  
また、2バイト目も 3バイト目も `80` から `BF` の範囲に見えます。  
見た目だけなら、どちらも 3バイト文字として通してしまいそうです。

しかし、この二つは同じ扱いにはなりません。

`E0` のときは、2バイト目に特別な条件があります。  
`80` から `9F` ではなく、`A0` から `BF` の範囲でなければなりません。

そのため、

- `E0 9F 80` は不受理
    
- `E0 A0 80` は受理
    

となります。

ここで重要なのは、2バイト目 `9F` も continuation byte の形ではあることです。  
つまり、「continuation byte だからよい」という読み方では `E0 9F 80` を誤って通してしまいます。  
このため、`E0` を見たら「3バイト形式だ」で終わらず、**2バイト目の下限が上がる** と意識しなければなりません。

---

### 3-3. `ED` の特例

次に `ED` を見ます。  
これも二つ並べて比較します。

`ED 9F BF`

`ED A0 80`

どちらも先頭バイトは `ED` です。  
したがって 3バイト文字候補です。  
また、2バイト目と 3バイト目は continuation byte の範囲に見えます。  
しかし、ここでも両方を同じように受理してはいけません。

`ED` の場合は、2バイト目に上限があります。  
`80` から `9F` の範囲でなければなりません。  
`A0` から `BF` は受理してはいけません。

したがって、

- `ED 9F BF` は受理
    
- `ED A0 80` は不受理
    

となります。

ここでも、第2章までの「3バイト候補で continuation byte が二つあるから通す」という読み方では不十分です。  
`ED` を見た時点で、2バイト目の許容範囲が狭くなることを確認しなければなりません。

---

### 3-4. 比較して読む

ここで `E0` と `ED` をまとめて見ます。

- `E0` のときは、2バイト目が `A0` から `BF`
    
- `ED` のときは、2バイト目が `80` から `9F`
    

どちらも 3バイト形式ですが、2バイト目に追加条件があります。  
しかも条件の向きが逆です。

`E0` は下限が上がります。  
`ED` は上限が下がります。

ここを曖昧にすると、「3バイト形式で continuation byte が続いているから大丈夫」という雑な一般化に戻ってしまいます。  
本書では、このような場面でこそ表を引きます。  
感覚ではなく、先頭バイトごとの条件に戻って判断します。

### 3-6. 模範解答：`E0` / `ED`

#### 問題1の考え方

先頭バイト `E0` は 3バイト文字候補です。  
2バイト目 `9F` と 3バイト目 `80` は continuation byte の範囲に入っています。  
しかし `E0` のときは、2バイト目が `A0` から `BF` でなければなりません。  
`9F` はその範囲に入りません。  
したがって不受理です。

#### 問題2の考え方

先頭バイト `E0` は 3バイト文字候補です。  
2バイト目 `A0` と 3バイト目 `80` は continuation byte の範囲に入っています。  
さらに `E0` の特例条件である「2バイト目が `A0` から `BF`」も満たしています。  
したがって受理できます。

#### 問題3の考え方

先頭バイト `ED` は 3バイト文字候補です。  
2バイト目 `9F` と 3バイト目 `BF` は continuation byte の範囲に入っています。  
さらに `ED` の特例条件である「2バイト目が `80` から `9F`」も満たしています。  
したがって受理できます。

#### 問題4の考え方

先頭バイト `ED` は 3バイト文字候補です。  
2バイト目 `A0` と 3バイト目 `80` は continuation byte の範囲に入っています。  
しかし `ED` のとき、2バイト目は `80` から `9F` でなければなりません。  
`A0` はその範囲を外れています。  
したがって不受理です。

### 3-7. 3人で読む：`E0` の特例

扱う例は `E0 9F 80` と `E0 A0 80` です。

**初学者**「どちらも `E0` で始まっていて、後ろも `80` 以上 `BF` 以下に見えます。どちらも 3バイト文字として通してよさそうに見えます。」

**解説者**「その見方が、この章で修正したい近道です。まず `E0` なので、たしかに 3バイト候補です。2バイト目と 3バイト目が continuation byte の範囲にあることも確認できます。ですが、`E0` の場合はそれだけでは足りません。」

**実装者**「つまり、形式だけ見れば通りそうでも、`E0` は追加条件があるわけですね。」

**解説者**「そうです。`E0` のときは、2バイト目が `A0` から `BF` でなければなりません。では `E0 9F 80` の 2バイト目は何ですか。」

**初学者**「`9F` です。これは continuation byte ではありますが、`A0` 以上ではありません。」

**解説者**「その通りです。だから `E0 9F 80` は不受理です。では `E0 A0 80` はどうですか。」

**初学者**「2バイト目が `A0` なので、特例条件を満たします。これは受理できます。」

**実装者**「見た目の形は似ているのに、境目が `9F` と `A0` にあるんですね。」

**解説者**「そうです。この章では、そういう境界を表で確認する読み方を身につけます。」

この対話で大切なのは、「continuation byte の形だから通す」という考えを止めることです。  
`E0` を見たら、2バイト目の特例条件まで確認する。  
それがここでの型です。

### 3-9. `F0` の特例

次に 4バイト形式へ進みます。  
まず `F0` を見ます。

比較するのは次の二つです。

`F0 8F 80 80`

`F0 90 80 80`

どちらも先頭バイトは `F0` です。  
したがって 4バイト文字候補です。  
また、2バイト目から 4バイト目までは continuation byte の範囲に見えます。

しかし、ここでも両方を同じように受理してはいけません。

`F0` の場合は、2バイト目に下限があります。  
`90` から `BF` の範囲でなければなりません。

そのため、

- `F0 8F 80 80` は不受理
    
- `F0 90 80 80` は受理
    

となります。

ここでも構図は同じです。  
「4バイト形式で continuation byte が三つ続いているからよい」という考え方では、`F0 8F 80 80` を誤って通してしまいます。  
`F0` を見たら、2バイト目に追加条件があると考えなければなりません。

### 3-10. `F4` の特例

続いて `F4` を見ます。

`F4 8F BF BF`

`F4 90 80 80`

どちらも先頭バイトは `F4` です。  
したがって 4バイト文字候補です。  
また、後続の三つは continuation byte の範囲に見えます。  
しかし、ここでも両方を受理してはいけません。

`F4` の場合は、2バイト目に上限があります。  
`80` から `8F` の範囲でなければなりません。

したがって、

- `F4 8F BF BF` は受理
    
- `F4 90 80 80` は不受理
    

となります。

`F0` と `F4` は、`E0` と `ED` に対応する形で覚えると見通しがよくなります。

- `F0` は 2バイト目の下限が上がる
    
- `F4` は 2バイト目の上限が下がる
    

この章では、細かな理屈を長く追うよりも、まずこの境界を正確に読めるようになることを重視します。

### 3-13. 3人で読む：`F0` と `F4`

**実装者**「ここでも continuation byte かどうかだけでは足りないわけですね。」

**解説者**「そうです。`F0` のとき、2バイト目は `90` から `BF` です。では `F0 8F 80 80` はどうですか。」

**初学者**「`8F` は continuation byte ですが、`90` 未満です。だから不受理です。」

**解説者**「正しいです。では `F0 90 80 80` はどうですか。」

見た目の形式が整っていることと、UTF-8 として正しいことを切り分ける。

### 3-15. 誤読修正問題

ここで、もっともらしいが危ない説明を取り上げます。

> UTF-8 は、先頭バイトで長さを決めて、残りが continuation byte なら実装できる。

この説明は、第2章までならかなり自然に見えます。  
しかし、この章まで読むと不十分だと分かります。

なぜなら、

- `E0 9F 80`
    
- `ED A0 80`
    
- `F0 8F 80 80`
    
- `F4 90 80 80`
    

のような列は、先頭バイトの長さ候補と continuation byte の形だけでは弾けないからです。

つまり、この説明は「基本形の観察」までは合っていますが、「特例条件がある」という事実を落としています。  
本書では、この落ち方を単なる知識不足とは見ません。  
**筋は良いが、一般化が早すぎる** 読み方だと見ます。

この説明は、次のように言い直すべきです。

**UTF-8 では、先頭バイトを見て長さ候補を決め、必要な continuation byte を確認する。  
ただし `E0`、`ED`、`F0`、`F4` では 2バイト目に追加条件があるため、必要なら特例表まで確認しなければならない。**

この言い直しができれば、第2章の基本形を保ったまま、第3章の特例も取り込めています。

### 3-17. この章のまとめ

この章では、UTF-8 を読むとき、先頭バイト表だけでは足りない場合があることを見ました。

第2章で確認した基本は、今も変わりません。

- 先頭バイトを見る
    
- 何バイト候補かを決める
    
- continuation byte の形を確認する
    
- そこまでを 1文字として切る
    

ただし、第3章でそこに一つ追加されました。

- 必要なら特例表を引く
    

特に `E0`、`ED`、`F0`、`F4` では、2バイト目に追加条件があります。  
そのため、先頭バイトの長さ候補と continuation byte の形だけで受理を決めてはいけません。

この章で修正したかったのは、次のような近道です。

- 3バイト形式なら continuation byte が二つあればよい
    
- 4バイト形式なら continuation byte が三つあればよい
    
- 見た目がそれらしければ通してよい
    

本書では、こうした近道を止めて、**必要な場面では表に戻る** 読み方を重視します。

UTF-8 の章は、ここで終わりではありません。  
ただし本書の狙いとしては、この時点でかなり重要なことを学んでいます。  
それは、「先頭を見る」「後続を確認する」「必要なら特例表まで引く」という、段階的な読み方です。

この型が身につくと、後の HTTP/2 や HPACK でも、「見た目だけで決めない」「境界条件を表で確認する」という姿勢をそのまま持ち込めます。

次章からは、題材を HTTP/2 に移します。  
そこで扱う単位は 1文字ではなく 1フレームです。  
しかし、やることは変わりません。  
最初に見る場所を決め、表を引き、1単位を確定させ、次へ進みます。
