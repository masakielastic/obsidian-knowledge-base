# 表を引いて読むバイナリ実習 執筆ガイドライン

## このファイルの役割

このファイルは、書籍  
**『表を引いて読むバイナリ実習 ― UTF-8 / HTTP/2 / HPACK で学ぶ「切る・数える・復元する」』**  
の企画・執筆・校正・リライトを支援するための品質基準と文章スタイルガイドである。

本書は概念書ではなく、**少数の典型データを深く読む実習書** である。  
したがって執筆時には、抽象概念の説明量そのものよりも、読者が次の動作を身につけられるかを優先する。

1. 先頭を見る  
2. 表を引く  
3. 1単位の長さや意味を決める  
4. 数える / 復元する  
5. 次へ進む  

本書で重視するのは、答えの暗記ではなく、**どこを見て、どの表を引き、どこで切るか** という読解動作である。

---

## 1. 書籍の目的

この書籍の目的は、UTF-8、HTTP/2、HPACK を題材として、  
**表を引きながら読む技法** を読者に身につけてもらうことにある。

読者が最終的にできるようになるべきことは次の通りである。

- 生のバイト列を前にしたとき、まず先頭を見るという発想を持てる
- 表を引いて 1単位の長さや意味を決められる
- UTF-8 では 1文字ずつ切りながら文字数を数えられる
- HTTP/2 では 9バイト共通ヘッダからフレーム境界を切れる
- HPACK では static table を引いてヘッダーを復元できる
- 途中切れ、不整合、特例条件を見落とさずに読める
- もっともらしい誤読や危ない近道を言語化して修正できる
- HTTP/1 の既存知識を足場にして HTTP/2 / HPACK の生データを読める

---

## 2. 想定読者

- HTTP/2 や HPACK に興味はあるが、生のダンプや 16 進数列に身構えてしまう人
- UTF-8 を知識としては知っているが、実際にバイト列を切りながら読んだことが少ない人
- 仕様を丸ごと読む前に、どこを見るべきかの型を身につけたい人
- HTTP/1 の既存知識を持ち、それを足場に HTTP/2 / HPACK を理解したい人
- 典型例を少数深掘りする教材を好む人
- 実装やデバッグの前段階として、読む力そのものを鍛えたい人

補足として、本書の主対象はフル実装を直ちに目指す読者ではない。  
むしろ、**仕様の全面読解や実装に入る前に、観察と切り分けの型を整えたい読者** を主対象とする。

## 3. 書籍の基本方針

### 重視すること

- 生データから始めること
- 先頭を見ること
- 表を引くこと
- 1単位を決めること
- 数える / 復元すること
- 次へ進むこと
- HTTP/1 の既存知識を足場にすること
- もっともらしい誤読を教材化すること
- アプローチ仮説を見せること
- 少数の典型例を深く扱うこと

### 深入りしすぎないこと

- RFC 全体の網羅
- exploit 的な話
- 特定言語でのフル実装チュートリアル
- HPACK dynamic table や Huffman の完全解説
- HTTP/2 の詳細な状態遷移の網羅
- RST_STREAM / GOAWAY / Sans-IO の本格実装論
- HTTP/3 / QPACK の詳細
- 大量の問題をこなすドリル型演習

本書は、すべてを教える本ではない。  
**読者が生データを前にして立ち止まらず、読むための最初の一歩を切れるようにする本** である。

## 4. 本書の中核メンタルモデル

本書では、UTF-8 / HTTP/2 / HPACK を次の共通フレームで扱う。

1. 先頭を見る  
2. 表を引く  
3. 1単位の長さや意味を決める  
4. 数える / 復元する  
5. 次へ進む  

この共通フレームを崩さないことが、本書全体の最重要原則である。

### UTF-8 での対応

- 先頭バイトを見る
- 文字長候補を決める
- 必要なら特例表を引く
- 1文字を切る
- 文字数を数える

### HTTP/2 での対応

- 先頭から 9バイト共通ヘッダを切る
- Length / Type / Flags / Stream ID を読む
- フレーム境界を決める
- 次のフレームへ進む

### HPACK での対応

- 先頭ビットを見る
- 表現形式を決める
- index や長さを読む
- static table を引いて復元する
- 次の表現へ進む

## 5. 本書における対話形式の位置づけ

本書では、**模範解答と誤読修正問題で 3人対話形式を採用する**。  
全面対話形式にはしない。  
問題提示、参照表、整理ブロック、典型例一覧は本文側で淡々と示してよい。

### 3人対話を使う理由

- 正答だけでなく、もっともらしい誤読を見せやすい
- 誤答そのものではなく、誤ったアプローチ仮説を修正しやすい
- 初学者の自然なつまずきと、実装者の危ない近道を分けて見せられる
- 解説者を「答えを言う人」ではなく「読み方の司会」にできる

### 3人対話を使う場所

- 模範問題
- 誤読修正問題
- 特例や不整合を扱う問題
- 一般化しすぎる危険を示したい問題

### 3人対話を使いすぎない場所

- 単純反復の練習問題
- 典型例一覧
- 参照表
- 短い確認問題

---

## 6. 3人対話の役割分担

### 初学者

役割：

- 自然な第一印象を言う
- 素朴な誤読や不安を出す
- 「どこを見ればよいか分からない」状態を言語化する
- 解説後に理解を言い換える

注意：

- すでに整理済みの説明を先回りして言わない
- 上級者の視点を先取りしない
- 仕様用語を勝手に再定義しない
- 毎回同じ誤り方をしない

### 解説者

役割：

- 何を先に見て、何をまだ決めてはいけないかを示す
- どの表を引くべきかを明示する
- 切り方、判断根拠、進み方を整理する
- その問題での論点仮説を整える
- 最後に手順を圧縮してまとめる

注意：

- 本文で扱う詳細を対話の中で全部先に言わない
- 長い講義にしない
- 仕様の網羅に脱線しない
- 実装論を主導しない

### 実装者

役割：

- 筋は良いが危ない近道を持ち込む
- 実装やデバッグで起こりがちな省略を指摘する
- 「たまたま動くが危ない」読み方を表面化する
- 現実のダンプ、ログ、テストデータに接続する

注意：

- 抽象理論を主導しない
- 解説者の役を奪わない
- 特定言語の枝葉に広げすぎない
- 実装自慢や最適化の話に逸れない

## 7. 仮説思考の三層をどう扱うか

本書では、各問題を次の 3層で見られるようにする。

### 論点仮説

この問題で最初に何を問うべきか。

例：
- UTF-8: 何文字あるかより、どこで 1文字が終わるか
- HTTP/2: 何のフレームかより、どこで 1フレーム切れるか
- HPACK: 答えを当てるより、どの表現形式か

### アプローチ仮説

どう見るべきか。  
どの表を引くべきか。  
どこで切るべきか。

本書ではここを最重視する。  
通常の解説書が答え仮説に寄りがちなところを、本書ではアプローチ仮説の修正に重点を置く。

### 答え仮説

最終的な結論。

例：
- 正しい文字数
- フレーム種別
- 復元されるヘッダー
- 正常 / 不正 / 不整合の判定

### 運用ルール

- 問題文や模範解答では、必要に応じて 3層のどこがずれていたかを明示してよい
- 誤読修正問題では、論点仮説 / アプローチ仮説 / 答え仮説の順に整理するとよい
- ただし、毎回この用語を露骨に出す必要はない。読解動作に埋め込むことを優先する

## 8. ユニット設計の標準フォーマット

各ユニットは原則として次の流れで構成する。

### 1. ユニット名

例：
- UTF-8: 先頭バイトを見て文字数を数える
- HTTP/2: 9バイト共通ヘッダを切る
- HPACK: static table を引いて疑似ヘッダーを復元する

### 2. このユニットの論点

その問題で最初に立てるべき論点仮説を 1文で示す。

### 3. 使う表

例：
- UTF-8 先頭バイト表
- UTF-8 特例表
- HTTP/2 共通ヘッダ表
- HTTP/2 Type 一覧
- HPACK Static Table

### 4. 導入データ

説明から始めず、生データを先に置く。

### 5. 最初の問い

観察動作に寄せる。

例：
- どこで 1文字が終わるか
- どこで 1フレーム切れるか
- static table だけで読めるか

### 6. 模範問題

- 生データ
- 作業指示
- 使用してよい表
- 禁止する近道
- 3人対話の模範解答

### 7. 練習問題

- 2〜3問程度
- 型は散らしすぎない
- 同型反復 / 1か所だけ違う比較 / 途中切れ・不整合 を中心にする

### 8. 誤読修正問題

- 誤った解答例を先に置く
- どこが危ないかを問う
- 誤答ではなく、誤った見方を修正する

### 9. 典型例一覧

- 1行1例でよい
- 暗記用ではなく、見慣れ用に置く

### 10. ユニット末尾のまとめ

- 最初に何を見たか
- どの表を引いたか
- どこで 1単位を切ったか
- 何を根拠に正常 / 不正 / 未確定を分けたか

## 9. 問題設計の基準

### 問題数

本書は大量演習型にしない。  
1ユニットあたりの目安は次の通り。

- 模範問題 1〜2問
- 練習問題 2〜3問
- 誤読修正問題 1問
- 典型例一覧 1ページ相当

### 問題のよい作り方

- 比較で読ませる
- 1か所だけ違うデータを並べる
- 正常と不正を対にする
- 途中切れを混ぜる
- 似ているが危険な値を並べる
- HTTP/1 の既存知識を足場にする

### 問題の悪い作り方

- 似ていない例を大量に並べる
- 仕様知識の暗記テストにする
- いきなり重い応用例を出す
- 動作より用語当てに寄る
- 正常系だけで終わる
- すぐに上級書の守備範囲へ踏み込む

## 10. 題材別の重点

### UTF-8

UTF-8 は文字コードの章としてではなく、**可変長データを表を引いて読む教材** として扱う。

特に重視すること：

- 先頭バイトで文字長候補を決める
- continuation byte の確認
- 途中切れの判定
- E0 / ED / F0 / F4 の特例
- 「形だけ合っていればよい」という誤読の修正

避けること：

- Unicode 全般の大きな講義
- 正規化や書記素クラスタの本格議論
- 文字コード史への脱線

### HTTP/2

HTTP/2 は仕様項目の列挙ではなく、**構造化バイナリを 9バイト共通ヘッダから切る教材** として扱う。

特に重視すること：

- 共通ヘッダを切る
- Length / Type / Flags / Stream ID の読み分け
- payload との照合
- 2フレーム連結
- Length 不整合

避けること：

- フレーム種別の網羅
- 状態遷移の全解説
- 実装 API 設計論への深入り

### HPACK

HPACK は圧縮方式の一般論としてではなく、**先頭ビットと static table を使って復元する教材** として扱う。

特に重視すること：

- Indexed Header Field の最小例
- static table の利用
- static だけで読めるものと読めないものの区別
- 「1バイト 1ヘッダー」という誤一般化の修正
- HTTP/1 との対応づけ

避けること：

- dynamic table の完全解説
- Huffman の詳細実装
- 圧縮率や性能の本格議論

---

## 11. HTTP/1 を足場にする運用ルール

HTTP/2 / HPACK では、必要に応じて HTTP/1 の見慣れたメッセージを先に置く。

例：

```http
GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Accept: text/html
```

そのうえで、

- HTTP/2 の論理ヘッダー
- HEADERS フレーム
- HPACK 生データ

へ落としていく。

### この方針の狙い

- 読者にとって既知の意味単位を足場にできる
- raw bytes 側が別世界に見えることを防ぐ
- 差分を比較しながら学べる
- static table の役割を変換表として理解しやすくなる

### 注意

- HTTP/1 と HTTP/2 を「同じ文字列が見える世界」と誤解させない
- 既知知識への依存が強すぎて、生データの観察が疎かにならないようにする
- 逆算解答に寄りすぎない

## 12. 文章スタイル

### 基本トーン

- 技術的に正確
- 実習書として明快
- 判断の筋道が見える
- 読者の観察動作を邪魔しない
- 説明より先にデータを置く
- 具体例から読み方へ一般化する

### 文体ルール

- 本文はです・ます調を基本とする
- 一文一義を意識する
- 結論と根拠を近くに置く
- 何をまだ決めてはいけないかを必要に応じて明示する
- 例は必要最小限に絞る
- 抽象語を使ったら、すぐ観察対象に戻す
- 読者の動作を動詞で書く

### 本書で特に重視すること

- 生データから始める
- 見る場所を具体的に言う
- 引く表を具体的に言う
- 切り方を途中経過として見せる
- 最終結論だけでなく、根拠を答えさせる
- 一般化しすぎる危険をその場で止める

### 避けるべき文体

- 抽象概念の説明だけで終わる文章
- 16進数の見た目の感想で終わる文章
- 用語の定義を並べるだけの文章
- 答えだけを言って過程を飛ばす文章
- 「雰囲気で分かる」ことを許してしまう文章
- 仕様の網羅に流れる文章
- 章ごとに密度やトーンが大きく変わる文章

## 13. 音声向け情報と視覚向け情報の分離

本書でも、**声に出して読める説明** と **目で追うべき情報** を分ける。

### 対話や本文に置くもの

- 問い
- 誤解
- どこを見るか
- なぜその表を引くか
- どこで切るか
- なぜその判断になるか
- 何をまだ決めてはいけないか

### 整理ブロックに置くもの

- バイト列
- 切り分け結果
- 比較表
- 対応表
- 典型例一覧
- 章末チェックリスト

### 運用ルール

- 3人対話の中で長い箇条書きを多用しない
- 長い比較は表や整理ブロックに逃がす
- バイト列はインデントや分割で読みやすく示す
- 視覚的に理解したほうがよい情報は、対話の中に埋め込まない

## 15. 校正時の最重要品質基準

### A. 読者の動作が見えるか

- どこを見ればよいかが書かれているか
- どの表を引くかが明示されているか
- どこで 1単位を切るかが示されているか
- 次へどう進むかが書かれているか

### B. 問題がアプローチ仮説を鍛えているか

- 答えだけで終わっていないか
- 読み方の手順が見えるか
- 誤った近道が教材化されているか
- 正常系だけに偏っていないか

### C. 比較が効いているか

- 1か所だけ違う例があるか
- 正常と不正が対になっているか
- 読者が差分から学べるか

### D. 章の問いに答えているか

- 導入の問いに本文が答えているか
- 話題が途中で広がりすぎていないか
- 上級編の話題が侵食しすぎていないか

### E. 分量が適切か

- 模範問題が長すぎないか
- 練習問題が単なる水増しになっていないか
- 典型例一覧が実用的か
- 1ユニットの認知負荷が高すぎないか

## 16. 具体的なレビュー観点

### 問題文

- 作業指示が明確か
- 禁止する近道が必要な場面で明示されているか
- 参照してよい表が明記されているか

### 模範解答

- 先頭を見るところから始まっているか
- 途中経過が見えるか
- 表を引いた痕跡があるか

- 初学者が賢くなりすぎていないか

- 実装者が脱線していないか
- 3役の違いが出ているか

### 練習問題

- 模範問題と地続きか

### 誤読修正問題

### 典型例一覧

- 暗記表ではなく、見慣れ用の一覧になっているか

## 17. リライト方針

校正や改善では、必要に応じて次を行う。

- 抽象概念の説明を削り、データと動作へ戻す
- 1ユニット 1論点に絞る
- 長い模範解答を途中経過つきで分割する
- 誤読修正問題を追加して、危ない近道を可視化する
- 比較しやすいデータ対を作り直す
- 典型例一覧を追加して見慣れ用の足場を作る
- HTTP/1 を足場にする前後対応を補う
- 章末のまとめを「読み方の手順」に寄せる

ただし、次は避ける。

- 上級書向けの実装論を大量追加すること
- 問題数を増やして密度を落とすこと
- 3人対話を長い講義へ変えること
- 用語解説を足しすぎて実習の主線を壊すこと
- 題材ごとのトーンをばらつかせること

## 18. 出力時の期待

## 19. 本書で特に守るべき一文

この本は、バイト列を  
**「意味不明な 16進数列」として眺める対象ではなく、「先頭を見て、表を引いて、1単位ずつ切りながら読める構造」として理解させる本**  
である。

校正・改善・リライトの判断は、常にこの方針に従って行う。

## 20. 本書で繰り返し守るべき補助原則

- UTF-8 は可変長データ読解の教材として扱う
- HTTP/2 は構造化バイナリ読解の教材として扱う
- HPACK は static table を引く変換表型の教材として扱う
- 個別知識を増やすより、読む観点を揃えることを優先する
- 読者が「どこから手をつければよいか分からない」状態を減らす
- 正答だけでなく、危ない近道も教材化する
- 問題数を増やすより、典型例を深く扱う
- HTTP/1 の既存知識を足場として有効活用する
- 付録的な補足ではなく、独立した実習体験として設計する

---

## 21. シリーズ内での位置づけを崩さないための注意

本書は、HTTP/2 入門書の補助資料ではなく、**raw bytes を読む技法に焦点を当てた独立小冊子** である。  
したがって、次のバランスを崩さない。

- 概念書より実習書として振る舞う
- ただし実装書や上級書の守備範囲へ踏み込みすぎない
- 既存の HTTP/2 本とは競合せず、役割分担を明確にする
- 本書は「分かる本」ではなく「読めるようになる本」であることを保つ

## 23. 最後に

本書は、抽象概念の説明に寄りすぎた本ではなく、  
**具体的なデータをどう見て、どう考えるかを主役にする実習書** である。

したがって執筆・校正・改善のたびに、次を問い直すこと。

- 読者は最初に何を見るのか
- どの表を引くのか
- どこで 1単位を切るのか
- 何を根拠にその判断をするのか
- どんなもっともらしい誤読をしうるのか
- その誤読をどう修正するのか

この問いに答えられない説明は、本書では優先度が低い。  
逆に、この問いに答えられる説明こそが、本書の中心である。
