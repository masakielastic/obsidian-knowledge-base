
## 第1章　三つの題材に共通する読み方

本書では、UTF-8、HTTP/2、HPACK という三つの題材を扱います。  
この三つは、ふつうは別々の知識として学ばれます。

UTF-8 は文字コードです。  
HTTP/2 はバイナリフレームを使う通信プロトコルです。  
HPACK は HTTP/2 で使われるヘッダー圧縮方式です。

見た目も用途も違います。  
そのため、最初は「共通する読み方がある」と言われても、少し不思議に感じるかもしれません。

しかし本書では、この三つを別々の専門知識としてではなく、**生データを前にしたときにどう読むか** という共通の課題として扱います。  
どの題材でも、最初に必要なのは「全部を一気に理解すること」ではありません。  
まず必要なのは、**どこを見て、どの表を引き、どこで一区切りにするか** を決めることです。

本章では、その共通部分だけを先に取り出して整理します。  
個別の詳細に入る前に、まず本書全体で反復する基本動作をそろえておきます。

本書で繰り返す動作は、次の五つです。

1. 先頭を見る
    
2. 表を引く
    
3. 1単位の長さや意味を決める
    
4. 数える、または復元する
    
5. 次へ進む
    
この章では、この五つを順番に見ていきます。

---

### 1-1. まず先頭を見る

生のバイト列を前にすると、つい「全体をなんとなく眺める」か、「知っている部分だけ拾おうとする」かのどちらかになりがちです。  
しかし、このやり方では読みにくい題材ほど手が止まります。

本書では、まずここを変えます。  
**最初にやることは、先頭を見ること** です。

これは「先頭だけ見れば十分」という意味ではありません。  
そうではなく、**最初の分岐点は先頭にあることが多い** という意味です。  
どこから読み始めるべきか迷ったら、まず先頭に戻る。  
これが本書の基本姿勢です。

UTF-8 では、先頭バイトを見ることで、その文字が何バイト候補なのかを決めます。  
たとえば `41` なら 1バイト文字です。  
`C3` で始まっていれば 2バイト文字候補です。  
`E3` で始まっていれば 3バイト文字候補です。  
つまり、最初の 1バイトを見ることで、次に何を確認すべきかが決まります。

HTTP/2 では、先頭から 9バイトを共通ヘッダとして読みます。  
ここで Length、Type、Flags、Stream Identifier を取り出し、そのあとに payload をどう読むかを決めます。  
つまり HTTP/2 でも、最初にやることは payload の中身を当てることではなく、**先頭の構造を切ること** です。

HPACK でも同じです。  
先頭ビットを見ることで、その表現が Indexed Header Field なのか、Literal Header Field なのか、といった大まかな形式を決めます。  
ここでも、最初にすべきことは値を推測することではなく、**何として読むべきかを決めること** です。

三つとも、題材は違います。  
それでも最初の一歩は似ています。

- UTF-8 では、先頭バイトを見る
    
- HTTP/2 では、先頭 9バイトを見る
    
- HPACK では、先頭ビットを見る
    

この時点では、まだ全部は分かっていません。  
ですが、少なくとも **何を調べに行くべきか** は見えてきます。  
本書では、この「最初の向きの決め方」をまず体に入れます。

---

### 1-2. 表を引く

先頭を見たあとに必要なのは、勘ではなく表です。  
本書では、**表を引くことをためらわない** ことを重視します。

生データを見ていると、慣れている人ほど「だいたい分かるから進める」と考えがちです。  
しかし、それは典型例を何度も見た人にしか通じません。  
しかも、そのやり方は似た例に引きずられやすく、危ない近道にもなります。  
本書では、見た感じや記憶に頼るのではなく、必要な場面では必ず表を引くことを基本にします。

UTF-8 で引く表は、主に二種類です。  
一つは、先頭バイトから文字長候補を決める表です。  
もう一つは、E0 / ED / F0 / F4 のような特例を確認する表です。  
先頭バイト表だけで進めると、形は合っているが受理してはいけないバイト列を見落とします。  
そのため、「何バイトか分かったから終わり」にはなりません。  
必要なら特例表まで引きます。

HTTP/2 で引く表は、共通ヘッダの見方とフレーム種別の表です。  
Length が何を表し、Type の値がどのフレームを意味するのかを、表を見ながら確定します。  
知っている値だけ拾うのではなく、まずヘッダを切り、表で意味を取ります。  
これを省くと、「payload が短いから DATA だろう」「空だから特別扱いだろう」といった危ない読み方に流れやすくなります。

HPACK で引く表は、何より static table です。  
たとえば `82` や `84` のような短い例でも、値を丸暗記するのではなく、必ず index を static table に引いて確認します。  
本書では、HPACK を「なんとなく見覚えのある値を当てるもの」としては扱いません。  
**変換表を使って復元するもの** として扱います。

ここで大切なのは、表は補助資料ではないということです。  
本書では、表を引く行為そのものが読解の本体です。  
表を見なくても答えられるかどうかより、**なぜその判断になるのかを表に戻って言えるか** を優先します。

---

### 1-3. 1単位を決める

先頭を見て、表を引いたあとにやることは、**どこまでを 1単位として読むかを決めること** です。  
これが本書の中心です。

生データは、切らなければ読めません。  
しかも、どこで切るかは題材ごとに違います。  
そのため本書では、「何を 1単位とみなすか」を毎回明示的に扱います。

UTF-8 では、1単位は 1文字です。  
ただし、ここでいう 1文字とは、見た目の文字数を雰囲気で数えることではありません。  
先頭バイトから長さ候補を決め、必要な継続バイトがそろっているかを確認し、必要なら特例条件も見たうえで、そこまでを 1文字として切ります。  
つまり UTF-8 では、「1文字ずつ切ること」がそのまま読みの中心になります。

HTTP/2 では、1単位は 1フレームです。  
先頭から 9バイト共通ヘッダを切り、Length の値だけ payload を取って、そこまでを 1フレームとして確定します。  
ここで大事なのは、Type の値だけ見て切らないことです。  
フレーム境界は雰囲気では決まりません。  
共通ヘッダと Length を使って、はじめて 1フレームが決まります。

HPACK では、1単位は 1ヘッダー表現です。  
Indexed Header Field なら index を取り出して復元します。  
Literal Header Field なら、さらに長さや名前や値を追う必要があります。  
いずれにしても、バイト列全体を一気に「HTTP リクエストっぽいもの」として見るのではなく、**1表現ずつ切りながら進む** ことが必要です。

この「1単位を決める」という動作は、三つの題材を横断する本書の核です。

- UTF-8 は 1文字
    
- HTTP/2 は 1フレーム
    
- HPACK は 1ヘッダー表現
    

個別の名前は違っても、やっていることは似ています。  
先頭を見て、表を引いて、そこまでを 1単位と決める。  
本書では、この感覚を何度も反復します。

---

### 1-4. 数える / 復元する

1単位が決まったら、その次にやることは **数える** か **復元する** かです。  
ここで初めて、読者がほしい最終結果に近づきます。

UTF-8 では、1文字ずつ切れたら文字数を数えられます。  
ただし本書では、「文字数を出すこと」だけを目的にしません。  
どこで 1文字が終わったのか、どこで途中切れなのか、どこで不正なのかを区別できることを重視します。  
ただ答えの数字が合っていても、切り方が曖昧なら本書の目的には達していません。

HTTP/2 では、1フレームずつ切れたら、フレーム種別や payload の長さを確定できます。  
連結された複数フレームがあっても、先頭から順に境界を確定しながら読んでいけば、全体の構造を把握できます。  
ここでも重要なのは、「何フレームあるか」を当てることではなく、**どの根拠でそこまでを 1フレームとしたか** を言えることです。

HPACK では、1表現ずつ切れたら、対応するヘッダーを復元できます。  
たとえば indexed header なら、index を static table に引いて `:method: GET` や `:path: /` のような意味に戻せます。  
ここでは「復元」が主役です。  
ただし復元も、雰囲気でやるのではなく、**どの形式として読み、どの表を引いてそう判断したか** が必要です。

本書では、数えることも復元することも、最後のごほうびのような位置づけです。  
その前にある「先頭を見る」「表を引く」「1単位を決める」ができていなければ、最終結果だけ合っていても危うい。  
逆に、途中の手順がしっかりしていれば、まだ完全に慣れていなくても、読み方は崩れません。

---

### 1-5. 次へ進む

1単位を読めたら終わりではありません。  
本書で最後に重視するのは、**次へ進むこと** です。

ここでいう「次へ進む」は、単に残りを見るという意味ではありません。  
いま読んだ 1単位を確定させ、その続きの位置から同じ手順をもう一度始めるという意味です。  
この反復ができるようになると、生データが一気に読めるようになります。

UTF-8 では、1文字を切ったあと、残りの先頭に移動してまた同じことをします。  
先頭バイトを見て、必要なら表を引き、1文字を確定し、次へ進みます。  
この反復によって、文字列全体を数えられるようになります。

HTTP/2 でも、1フレームを切ったら、その次の位置からまた 9バイト共通ヘッダを読みます。  
先頭フレームが確定していないのに先へ進むと、以降の読みは全部ずれます。  
そのため、**今どこまで確定したか** を常に意識しながら進む必要があります。

HPACK でも同じです。  
1ヘッダー表現を復元したら、残りの先頭からまた形式を判定します。  
前の問題に似ているからといって、まとめて処理してはいけません。  
毎回、先頭ビットから入り直すことが大切です。

この「次へ進む」を軽く見ると、典型例では読めても、少し長くなった途端に崩れます。  
本書で扱う題材はどれも、**先頭から 1単位ずつ確定しながら進む** ことで初めて読めるようになるものです。  
したがって、本書の練習問題でも、単発の最小例だけでなく、連結例や混在例を入れていきます。  
それは、読者に「一回だけ読める」状態ではなく、「次へ進める」状態になってほしいからです。

---

### 1-6. 三つの題材を並べて見る

ここまでの内容を、三つの題材で並べてみます。

#### UTF-8

- 最初に見るものは先頭バイトです
    
- 引くのは先頭バイト表や特例表です
    
- 決める 1単位は 1文字です
    
- その結果として文字数を数えられます
    
- 1文字読めたら次のバイト位置へ進みます
    

#### HTTP/2

- 最初に見るものは先頭から 9バイトの共通ヘッダです
    
- 引くのは共通ヘッダの見方や Type 一覧です
    
- 決める 1単位は 1フレームです
    
- その結果として種別や境界を確定できます
    
- 1フレーム読めたら次のフレーム位置へ進みます
    

#### HPACK

- 最初に見るものは先頭ビットです
    
- 引くのは static table や表現形式の表です
    
- 決める 1単位は 1ヘッダー表現です
    
- その結果としてヘッダーを復元できます
    
- 1表現読めたら次の表現位置へ進みます
    

こうして並べると、本書が三つの話を無理にまとめているわけではないことが見えてきます。  
共通しているのは、知識の種類ではなく、**読むときの身体の使い方** です。  
本書はそこをそろえるための本です。

### 1-7. 混合ミニドリル

ここで、ごく短い例を三つだけ見てみます。

`41`

`00 00 00 04 00 00 00 00 00`

`82`

この段階では、まだ詳細な答えをすべて出す必要はありません。  
ここでやりたいのは、「最初にどこを見るか」と「どの表を引くか」を言えるようになることです。

最初の `41` は、UTF-8 の例として見ます。  
ここでまずやることは、先頭バイトを見て文字長候補を決めることです。  
ASCII の 1バイト文字だと分かれば、そこまでを 1文字として切れます。

次の `00 00 00 04 00 00 00 00 00` は、HTTP/2 の最小例として見ます。  
ここでは、最初の 9バイトを共通ヘッダとして切るべきだと分かるかが大切です。  
payload の意味を先に推測するのではなく、まず構造を取ります。

最後の `82` は、HPACK の最小例として見ます。  
ここでは、先頭ビットを見て表現形式を決め、static table を引きに行くという流れに入れるかが大切です。

この三つを前にして、「何の知識が必要か」を全部答えられなくてもかまいません。  
本章の段階で目指すのは、**どの型で読み始めるか** が分かることです。

### 1-8. この章で覚えておきたいこと

本章でいちばん大切なのは、三つの題材に共通しているのは用語ではなく、読み方だという点です。

本書で繰り返すのは、次の五つでした。

- 先頭を見る
    
- 表を引く
    
- 1単位の長さや意味を決める
    
- 数える / 復元する
    
- 次へ進む
    

この五つは、UTF-8 では 1文字を切る動作として現れます。  
HTTP/2 では 1フレームを切る動作として現れます。  
HPACK では 1ヘッダー表現を切る動作として現れます。

次章からは、まず UTF-8 でこの型をしっかり練習します。  
UTF-8 は単純に見えて、先頭バイト、継続バイト、途中切れ、特例条件という要素があり、**表を引いて 1単位ずつ切る** 練習にとても向いています。  
ここで「先頭を見る」「切る」「不足や不正を見分ける」という感覚を固めてから、HTTP/2 や HPACK に進みます。

本章を読んだあとで、まだ個別の仕様が十分に頭に入っていなくても問題ありません。  
その代わり、次の問いを持って次章へ進んでください。

このデータでは、最初にどこを見るのか。  
どの表を引くのか。  
何を 1単位として切るのか。  
それが決まったあと、何を数えるのか、何を復元するのか。  
そして、どこから次へ進むのか。

この問いを持ったまま進めば、本書全体の流れは見失いません
