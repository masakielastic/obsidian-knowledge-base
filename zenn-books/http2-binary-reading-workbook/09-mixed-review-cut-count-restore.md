## 第9章　何を1単位として切るかを言葉にする

ここまで本書では、UTF-8、HTTP/2、HPACK という三つの題材を扱ってきました。  
UTF-8 では 1文字を切りました。  
HTTP/2 では 1フレームを切りました。  
HPACK では 1ヘッダー表現を切りました。

題材は違います。  
使う表も違います。  
見ているバイト列の長さや形も違います。  
それでも、本書がずっと繰り返してきた動作は同じでした。

- 最初にどこを見るかを決める
    
- 必要な表を引く
    
- 何を 1単位として切るかを決める
    
- その単位を数える、または復元する
    
- 次へ進む
    

この最終章では、それをもう一度まとめて扱います。  
狙いは、新しい仕様知識を増やすことではありません。  
ここでやりたいのは、**題材が変わっても、自分が何を 1単位として切っているのかを言葉にできるようになること** です。

本書を読み終えたあと、読者に持っていてほしいのは、たくさんの断片的知識ではありません。  
そうではなく、16進ダンプを見たときに、いきなり止まらず、

- まずどこを見るのか
    
- 何をまだ決めてはいけないのか
    
- どの表を引けばよいのか
    
- どこまでを 1単位とするのか
    

を口に出せる状態です。  
この章は、そのための復習章です。

---

### 9-1. これまで切ってきた 1単位を並べる

まず、本書で扱ってきた 1単位を並べます。

UTF-8 では、1単位は **1文字** でした。  
先頭バイトを見て長さ候補を決め、必要な continuation byte を確認し、必要なら特例表も見て、そこまでを 1文字として切りました。

HTTP/2 では、1単位は **1フレーム** でした。  
先頭 9バイトの共通ヘッダを切り、Length を読み、その Length に従って payload の範囲を決め、そこまでを 1フレームとして確定しました。

HPACK では、1単位は **1ヘッダー表現** でした。  
先頭ビットを見て形式を決め、Indexed Header Field なら index を static table に引いて復元し、そこまでを 1表現として切りました。

この三つは名前が違います。  
けれど、本書の観点では似ています。  
どれも「先頭にある情報を使って、ここまでが 1単位だと決める」という作業だからです。

この章では、その共通性を改めて見える形にします。

---

### 9-2. 混合ミニドリル 1

まず、三つの短い例を並べます。

`E3 81 82`

`00 00 01 00 00 00 00 00 01 FF`

`82 84 87`

この三つを見たとき、本書の読み方に従うなら、最初に考えるべきことは「何が書いてあるか」ではありません。  
まず考えるべきことは、**これは何を 1単位として切る題材なのか** です。

最初の `E3 81 82` では、UTF-8 の 1文字を切ります。  
先頭バイト `E3` を見て 3バイト候補と分かり、後続 2バイトを確認して、3バイトで 1文字です。

二つ目の `00 00 01 00 00 00 00 00 01 FF` では、HTTP/2 の 1フレームを切ります。  
先頭 9バイトを共通ヘッダとして切り、Length=1 と読み、その後ろの 1バイト `FF` までを 1フレームとします。

三つ目の `82 84 87` では、HPACK の 1ヘッダー表現ずつを切ります。  
`82`、`84`、`87` を順に読んで、それぞれを 1表現として復元します。

ここで大切なのは、「同じ 16進数列でも、題材によって切る単位が違う」と意識することです。  
逆に言えば、題材が違っても「先頭を見て 1単位を決める」という動作は同じです。

### 9-3. 模範問題：何を1単位として切るか

次の各バイト列について、最初に何を見て、何を 1単位として切るかを答えてください。

#### 問題1

`E3 81 82`

#### 問題2

`00 00 01 00 00 00 00 00 01 FF`

#### 問題3

`82 84 87`

答えるときは、次の四つを意識してください。

- 最初に見る場所
    
- 引く表
    
- 1単位
    
- 最終的に何が得られるか
    

---

### 9-4. 模範解答：何を1単位として切るか

#### 問題1の考え方

`E3 81 82` では、最初に見る場所は先頭バイト `E3` です。  
引くのは UTF-8 の先頭バイト表です。  
`E3` は 3バイト文字候補なので、後続の `81` と `82` が continuation byte であることを確認します。  
したがって 1単位は **1文字** です。  
最終的には、3バイトで 1文字が得られます。

#### 問題2の考え方

`00 00 01 00 00 00 00 00 01 FF` では、最初に見る場所は先頭 9バイトです。  
引くのは HTTP/2 共通ヘッダの見方と Type 一覧です。  
Length=1、Type=DATA と分かるので、共通ヘッダ 9バイトと payload 1バイトを合わせて 1単位は **1フレーム** です。  
最終的には、payload 長 1 の DATA フレームが得られます。

#### 問題3の考え方

`82 84 87` では、最初に見る場所は各表現の先頭ビットです。  
引くのは HPACK の static table です。  
各バイトは Indexed Header Field として読めるので、1単位は **1ヘッダー表現** です。  
最終的には、3つのヘッダー表現が復元されます。

この三つを比べると、「何を見るか」「引く表」「切る単位」が違う一方で、やっていることの型は揃っていると分かります。

### 9-5. 3人で読む：混合ミニドリル

扱う例は次の三つです。

`E3 81 82`

`00 00 01 00 00 00 00 00 01 FF`

`82 84 87`

**初学者**「三つともバイト列ですが、何を 1単位として切るかがそれぞれ違います。」

**解説者**「その通りです。では最初の `E3 81 82` は何を切りますか。」

**初学者**「UTF-8 の 1文字です。先頭バイト `E3` を見て、3バイト文字候補として切ります。」

**解説者**「二つ目はどうですか。」

**初学者**「HTTP/2 なので、先頭 9バイトの共通ヘッダを切って 1フレームを読みます。Length=1 なので payload は 1バイトです。」

**実装者**「三つ目の `82 84 87` は、まとめて 1つではなく、1表現ずつ読むんですね。」

**初学者**「はい。各バイトの先頭ビットを見て Indexed Header Field と分かるので、1ヘッダー表現ずつ static table に引きます。」

**解説者**「ここで重要なのは、いきなり中身を当てることではなく、先に『何を 1単位として切る題材か』を決めることです。」

**実装者**「つまり、本書全体でやっていたのは、仕様知識を増やすこと以上に、切る単位を決める習慣を身につけることだったわけですね。」

**解説者**「そうです。この章では、それを言葉にできるようにします。」

この対話で見てほしいのは、三つの題材がばらばらに存在しているのではなく、**同じ読解フレームで扱えている** ことです。

### 9-6. 誤読の型を比べる

ここまで本書では、各題材でいろいろな誤読を見てきました。  
この章では、それらを横に並べてみます。

UTF-8 では、こんな誤りがありました。

- 先頭バイトだけ見て確定した気になる
    
- continuation byte の形だけ見て通してしまう
    
- 特例条件を落とす
    
- 途中切れなのに 1文字として数える
    

HTTP/2 では、こんな誤りがありました。

- Type だけ見て済ませる
    
- payload 側から先に意味づけする
    
- Length を見ずにフレーム境界を雰囲気で決める
    
- 手元にある残り全部を payload とみなす
    

HPACK では、こんな誤りがありました。

- 先頭ビットの形式判定を飛ばす
    
- 値を丸暗記して static table に戻らない
    
- 途中で別形式が現れても無理に static で押し切ろうとする
    
- 見覚えのある値だけ拾って後ろまで読めた気になる
    

一見すると別々のミスに見えます。  
しかし、もっと抽象化すると似ています。

- 先頭の判定を早く済ませすぎる
    
- 表を引かずに記憶や雰囲気で進む
    
- 1単位の境界をあいまいにする
    
- たまたま通った近道を一般化する
    

つまり、本書で修正してきたのは個別の知識不足だけではありません。  
**誤った進み方の型** を修正してきたのです。

---

### 9-7. 論点仮説、アプローチ仮説、答え仮説で見る

本書の執筆方針では、各問題を三つの層で見る考え方を重視してきました。

- 論点仮説  
    この問題で最初に何を問うべきか
    
- アプローチ仮説  
    どう見て、どの表を引いて、どこで切るか
    
- 答え仮説  
    最終的な結論は何か
    

この章では、それを三つの題材に軽く当てはめてみます。

UTF-8 で `E3 81` を見たとき、答え仮説だけに飛ぶと「1文字だろう」と言いたくなるかもしれません。  
しかし論点仮説としては「何文字あるか」より先に「どこで 1文字が終わるか」を問うべきでした。  
そしてアプローチ仮説としては「先頭バイトを見て長さ候補を決め、残りが足りるかを見る」必要がありました。

HTTP/2 で `00 00 02 00 00 00 00 00 01 FF` を見たときも同じです。  
答え仮説だけに飛ぶと「DATA フレームだ」と言えます。  
しかし論点仮説としては「何のフレームか」だけでなく「どこまでが 1フレームか」を問うべきです。  
アプローチ仮説としては「共通ヘッダを切り、Length と実データ長を照合する」必要がありました。

HPACK で `82 40` を見たときも同じです。  
答え仮説だけに飛ぶと「GET のあとに何かある」と言いたくなるかもしれません。  
しかし論点仮説としては「どこまで static table だけで読めるか」を問うべきでした。  
アプローチ仮説としては「先頭ビットを見て形式を判定し、別形式が出たところで止まる」必要がありました。

このように、本書で何度も扱ってきたのは、「答えを当てる」よりも **何を問うかと、どう進むかを整えること** でした。

### 9-8. 模範問題：誤読の型を見分ける

次の各説明が、どこでずれているかを考えてください。  
論点仮説、アプローチ仮説、答え仮説のどこが主に問題か、を答えてください。

#### 問題1

「`E0 9F 80` は 3バイト形式で continuation byte もあるので受理できる。」

#### 問題2

「`00 00 02 00 00 00 00 00 01 FF` は Type が DATA なので読めたことにしてよい。」

#### 問題3

「`82 40 87` は `82` と `87` が見覚えのある indexed header なので、たぶんその二つを読めばよい。」

---

### 9-9. 模範解答：誤読の型を見分ける

#### 問題1の考え方

この説明は、主に **アプローチ仮説** がずれています。  
先頭バイトを見て 3バイト候補と決めるところまではよいですが、`E0` の特例条件を確認していません。  
つまり、どの表を引くべきかという進み方が不足しています。

#### 問題2の考え方

この説明も、主に **アプローチ仮説** がずれています。  
Type が DATA と分かることと、フレーム全体が正常に成立していることは別です。  
共通ヘッダを切り、Length と実データ長を照合する手順が抜けています。

#### 問題3の考え方

この説明は、**アプローチ仮説** と **論点仮説** の両方がずれています。  
「見覚えのある値を拾えばよい」と考えており、まず各表現の形式を判定するという進み方が抜けています。  
また、問うべき論点は「どこまで安全に読めるか」なのに、「知っている値を回収すること」が目的にすり替わっています。

ここで見てほしいのは、多くの誤りが単なる答えの間違いではなく、**見方や進み方のずれ** から生まれていることです。

### 9-10. 3人で読む：誤読の比較

**初学者**「`E0` の特例表です。continuation byte の確認だけで終わっています。」

**実装者**「HTTP/2 の例だと、Type が分かったところで安心して、Length と実データ長の照合を省いていますね。」

### 9-12. この本全体で身につけたかったこと

この章の最後に、本書全体で身につけたかったことを整理します。

本書が育てたかったのは、「UTF-8 に詳しい」「HTTP/2 の型をいくつか知っている」「HPACK の static table をいくつか覚えている」といった断片的な状態だけではありません。  
もちろん、そうした知識もあります。  
しかし、それ以上に重要なのは、**生データを前にしたときの最初の入り方** です。

具体的には、次のような反応ができることです。

- まず先頭を見る
    
- 何を 1単位として切る題材かを考える
    
- 必要な表を探す
    
- まだ決めてはいけないことを保留する
    
- 確定できるところまでを丁寧に切る
    
- 不足や不整合や別形式が出たら、そこで止まる
    
- たまたま成功した近道を一般化しない
    

この反応ができるようになると、16進ダンプを見たときに完全停止しにくくなります。  
すぐに全部分からなくても、「まず何をするか」は分かるようになります。  
本書が目指していたのは、その状態です。

### 9-14. この章のまとめ

この章では、UTF-8、HTTP/2、HPACK を並べて、何を 1単位として切るのかを言葉にしました。  
また、題材ごとの誤読を比べながら、危ない近道の型が似ていることも確認しました。

この章で一番大切なのは、次の一文です。

**読むとは、まず何を 1単位として切るかを決めることです。**

UTF-8 では 1文字です。  
HTTP/2 では 1フレームです。  
HPACK では 1ヘッダー表現です。

そして、その 1単位を決めるために、

- 先頭を見る
    
- 表を引く
    
- 境界条件を確認する
    
- 確定できるところまでを切る
    
- 次へ進む
    

という流れを繰り返します。

本書全体を通して、本当に身につけてほしかったのはこの読み方です。  
仕様の断片をたくさん覚えることよりも、**どこを見て、どの表を引き、どこで 1単位を切るかを説明できること**。  
それができれば、この本の役割は果たせています。

これで本文は終わりです。  
ただし、ここで終わりというより、ここから先に自分で raw bytes を見るための足場ができた、というほうが近いでしょう。  
この本で扱った最小例と典型例を何度か見返しながら、まずは「完全停止せずに最初の一歩を切れる」ことを確かなものにしてください。
