## 第2章　先頭バイトを見て文字数を数える

第1章では、本書全体に共通する読み方として、次の五つを確認しました。

1. 先頭を見る
    
2. 表を引く
    
3. 1単位の長さや意味を決める
    
4. 数える、または復元する
    
5. 次へ進む
    

この章では、その型をまず UTF-8 で練習します。

UTF-8 は文字コードとして知られています。  
しかし本書では、文字コードの歴史や Unicode 全般の概説からは始めません。  
ここで中心に置くのは、**バイト列を前にしたとき、どこで 1文字が終わるかをどう決めるか** です。

UTF-8 は、この練習に向いています。  
なぜなら、先頭バイトを見ることで、その文字が何バイト候補なのかが分かるからです。  
つまり UTF-8 は、**先頭を見て、表を引いて、1単位を決める** という本書の基本動作を、もっとも小さな単位で練習できる題材です。

この章では、まず ASCII、2バイト文字、3バイト文字を切るところから始めます。  
そのあとで、途中切れを見抜く問題に進みます。  
ここではまだ特例条件には深入りしません。  
まずは、**長さ候補を決め、必要なバイトが本当にそろっているかを見る** という基本動作を固めます。

---

### 2-1. UTF-8 を読むとき、最初に何を見るか

UTF-8 のバイト列を読むとき、最初にやることは文字を当てることではありません。  
「これはたぶん日本語だろう」とか「見た感じマルチバイト文字だろう」と考える前に、まず先頭バイトを見ます。

UTF-8 では、先頭バイトの形によって、1文字の長さ候補が決まります。  
ここでは、まず次の対応だけ使います。

- `00` から `7F` なら 1バイト文字候補
    
- `C2` から `DF` なら 2バイト文字候補
    
- `E0` から `EF` なら 3バイト文字候補
    
- `F0` から `F4` なら 4バイト文字候補
    

この章で主に使うのは、1バイト、2バイト、3バイトです。

ただし、ここで注意があります。  
先頭バイトを見れば「何バイト候補か」は分かります。  
けれど、それだけでその文字を受理してよいとはまだ言えません。  
後続のバイトが必要な形をしているか、必要な数だけそろっているかを見なければなりません。

つまり、先頭バイトは出発点です。  
結論そのものではありません。

---

### 2-2. 1バイト文字を切る

最初に、もっとも短い例を見ます。

`41`

このバイト列を見たとき、まず先頭バイト `41` を見ます。  
`41` は `00` から `7F` の範囲に入っています。  
したがって、これは 1バイト文字候補です。

1バイト文字候補であれば、その 1バイトだけで 1文字が確定します。  
この場合は、先頭から 1バイト切って終わりです。

ここで大切なのは、「A だと知っているから読めた」のではないことです。  
本書で重視するのは、文字そのものを知っているかどうかではなく、**どこまでを 1文字として切れるか** です。

同じ考え方で、次のような値も同様に読めます。

`24`

`61`

`7A`

どれも `00` から `7F` に入るので、先頭の 1バイトだけで 1文字です。

この段階で読者が身につけるべきことは単純です。  
ASCII かどうかを雰囲気で決めるのではなく、**先頭 1バイトが 1バイト文字候補の範囲にあるかを見る** ことです。

---

### 2-3. 2バイト文字を切る

次に、2バイト文字の最小例を見ます。

`C3 A9`

最初に見るのは先頭バイト `C3` です。  
`C3` は `C2` から `DF` の範囲に入っています。  
したがって、これは 2バイト文字候補です。

ここで終わってはいけません。  
2バイト文字候補であれば、次の 1バイトが continuation byte の形になっている必要があります。  
continuation byte は `80` から `BF` の範囲です。

この例では、2バイト目は `A9` です。  
`A9` は `80` から `BF` の範囲に入っています。  
したがって、`C3 A9` は 2バイトで 1文字として切れます。

このときも、文字そのものを知っている必要はありません。  
本書で大切なのは、次の順番です。

まず `C3` を見て 2バイト候補と決める。  
次に `A9` を見て continuation byte か確認する。  
条件がそろったので、`C3 A9` を 1文字として切る。

この順番が崩れなければ、似た例にも対応できます。

たとえば次の例も同じ型で読めます。

`C2 A2`

`DF BF`

どちらも、先頭バイトが 2バイト候補の範囲にあり、2バイト目が continuation byte の範囲にあります。  
そのため、2バイトで 1文字として切れます。

ここでありがちな誤りは、「先頭が `C3` っぽいから 2バイト文字だろう」とだけ考えて進んでしまうことです。  
本書では、この“っぽい”を許しません。  
必ず 2バイト目まで確認します。

---

### 2-4. 3バイト文字を切る

続いて、3バイト文字の最小例を見ます。

`E3 81 82`

最初に見るのは先頭バイト `E3` です。  
`E3` は `E0` から `EF` の範囲に入っています。  
したがって、これは 3バイト文字候補です。

3バイト文字候補であれば、後ろに 2つの continuation byte が必要です。  
次の 2バイトが `80` から `BF` の範囲に入っているかを見ます。

この例では、2バイト目は `81`、3バイト目は `82` です。  
どちらも continuation byte の範囲に入っています。  
したがって、`E3 81 82` は 3バイトで 1文字として切れます。

ここでもやっていることは変わりません。

- 先頭を見る
    
- 何バイト候補か決める
    
- 必要な数だけ continuation byte があるか確認する
    
- そこまでを 1文字として切る
    

つまり 1バイト文字でも 2バイト文字でも 3バイト文字でも、読解の型は同じです。  
違うのは、必要な後続バイトの数だけです。

同じ型で読める例として、次のようなものがあります。

`E2 82 AC`

これも先頭 `E2` が 3バイト候補で、後ろの `82` と `AC` が continuation byte なので、3バイトで 1文字です。

---

### 2-5. 混在する文字列を切る

ここまでは、1文字だけの例を見てきました。  
しかし実際には、ASCII と multibyte 文字が混在することがよくあります。  
ここで大切なのは、1文字ごとに切って、そのたびに次の先頭へ進むことです。

次の例を見ます。

`41 C3 A9`

先頭の `41` は 1バイト文字候補です。  
したがって、まず `41` を 1文字として切ります。

そこで終わりではありません。  
残りは `C3 A9` です。  
この残りの先頭 `C3` をあらためて見ます。  
`C3` は 2バイト文字候補です。  
次の `A9` は continuation byte です。  
したがって、`C3 A9` で 1文字です。

このバイト列全体は、1バイト文字 1個と 2バイト文字 1個からなる、合計 2文字として読めます。

ここで大事なのは、最初から「3バイト列だ」とひとかたまりに見ないことです。  
UTF-8 は、左から順に 1文字ずつ切って進みます。  
途中でまとめ読みをしない。  
毎回、残りの先頭を新しく見る。  
この手順が重要です。

同じ考え方で、次のような例も読めます。

`41 E3 81 82`

最初の `41` は 1バイト文字です。  
残りの `E3 81 82` は 3バイト文字です。  
したがって全体では 2文字です。

さらに、

`61 E2 82 AC 62`

なら、`61` で 1文字、`E2 82 AC` で 1文字、`62` で 1文字です。  
合計 3文字です。

本書で重要なのは、このような混在例でも、やることが変わらないと分かることです。  
先頭を見る。  
切る。  
次へ進む。  
また先頭を見る。  
この反復を崩さないことが、読み方の土台になります。

### 2-6. 模範問題

ここで、短い模範問題をまとめて見ます。

#### 問題1

`41`

このバイト列は、何文字として切れますか。

#### 問題2

`C3 A9`

このバイト列は、何文字として切れますか。

#### 問題3

`E3 81 82`

このバイト列は、何文字として切れますか。

---

### 2-7. 模範解答

#### 問題1の考え方

`41` の先頭バイトは `41` です。  
これは `00` から `7F` の範囲にあるので、1バイト文字候補です。  
したがって、この 1バイトだけで 1文字です。

#### 問題2の考え方

`C3 A9` の先頭バイトは `C3` です。  
これは `C2` から `DF` の範囲にあるので、2バイト文字候補です。  
2バイト目 `A9` は `80` から `BF` の範囲にあり、continuation byte の条件を満たします。  
したがって、2バイトで 1文字です。

#### 問題3の考え方

`E3 81 82` の先頭バイトは `E3` です。  
これは `E0` から `EF` の範囲にあるので、3バイト文字候補です。  
2バイト目 `81` と 3バイト目 `82` はどちらも continuation byte の範囲にあります。  
したがって、3バイトで 1文字です。

---

### 2-8. 3人で読む

ここでは、同じ問題を対話形式で見てみます。  
扱うのは `41 C3 A9` です。

**初学者**「`41` は 1文字っぽいです。`C3 A9` は見たことがあるので、たぶん 2文字で合っていますよね。」

**解説者**「結論としては 2文字です。ただし、見たことがあるから進むのではなく、順番に確認しましょう。まず先頭の `41` を見ます。これは 1バイト文字候補なので、ここで 1文字切れます。」

**実装者**「このくらい短いなら見た感じでも読めますが、手順を崩すと長い列で危ないですね。」

**解説者**「そうです。では残りの先頭は何ですか。」

**初学者**「残りは `C3 A9` なので、先頭は `C3` です。」

**解説者**「`C3` は何バイト候補ですか。」

**初学者**「2バイト候補です。」

**解説者**「では次の `A9` を見ます。これは continuation byte の範囲に入っています。したがって `C3 A9` で 1文字です。全体では 2文字です。」

**実装者**「つまり、最初から全体を 3バイト列として見るのではなく、`41` を切ったあとで、残りを新しく読むわけですね。」

**解説者**「その通りです。UTF-8 は左から 1文字ずつ切って進みます。」

この対話で大切なのは、初学者が間違えたかどうかではありません。  
大事なのは、「見覚えがあるから進む」という危ない近道を止めて、**先頭を見る → 切る → 残りの先頭を見る** という手順に戻していることです。

---

### 2-9. 途中切れを見抜く

ここまでの例は、必要なバイトがすべてそろっていました。  
しかし UTF-8 では、先頭バイトを見て長さ候補が分かっても、実際のデータが足りないことがあります。  
これを見抜けなければ、「本当は未完成なのに 1文字として数えてしまう」という誤りが起こります。

次の例を見ます。

`C3`

先頭バイト `C3` は 2バイト文字候補です。  
しかし、このバイト列には 2バイト目がありません。  
つまり、必要な continuation byte が足りません。  
したがって、これは 1文字として確定できません。  
2バイト文字候補ではありますが、**途中切れ** です。

同じように、次の例も見ます。

`E3 81`

先頭バイト `E3` は 3バイト文字候補です。  
しかし必要なのは continuation byte が 2つです。  
ここでは `81` しかなく、3バイト目が足りません。  
したがって、これも途中切れです。

ここで重要なのは、「何バイト候補か分かった」と「1文字として受理できる」は別だということです。  
本書では、この区別を厳密に保ちます。

---

### 2-10. 途中切れの混在例

途中切れは、単独で現れるとは限りません。  
途中まで正常に読めて、最後だけ足りないこともあります。

次の例を見ます。

`41 E3 81`

最初の `41` は 1バイト文字候補なので、ここで 1文字切れます。  
残りは `E3 81` です。  
先頭 `E3` は 3バイト文字候補です。  
しかし continuation byte は 2つ必要なのに、ここでは `81` しかありません。  
したがって、残りは途中切れです。

この場合、全体を「2文字」と数えてはいけません。  
正しくは、先頭の `41` までは 1文字として確定できるが、残りは途中切れであり、次の 1文字は確定できない、となります。

この区別は大切です。  
途中まで正常に読めたからといって、最後まで正常だとは限りません。  
本書では、**切れたところまで確定し、その先は未確定として止まる** という判断を重視します。

### 2-12. 模範解答：途中切れ

#### 問題1の考え方

`C3` は 2バイト文字候補です。  
しかし 2バイト目がありません。  
したがって、1文字として確定できません。途中切れです。

#### 問題2の考え方

`E3` は 3バイト文字候補です。  
しかし continuation byte が 2つ必要なのに、`81` しかありません。  
したがって、1文字として確定できません。途中切れです。

#### 問題3の考え方

先頭 `41` は 1バイト文字なので、ここまでは 1文字として確定できます。  
残りの `E3 81` は 3バイト文字候補ですが、必要な 3バイト目がありません。  
したがって、先頭の 1文字までは正常に読めますが、その先は途中切れです。

### 2-13. 3人で読む：途中切れ

扱う例は `41 E3 81` です。

**初学者**「`41` は 1文字で、そのあと `E3 81` があるので、全体では 2文字でしょうか。」

**解説者**「その言い方だと危ないです。まず先頭の `41` は確かに 1文字です。では残りの先頭 `E3` は何バイト候補ですか。」

**初学者**「3バイト候補です。」

**解説者**「では 3バイトそろっていますか。」

**初学者**「`E3 81` までしかないので、1バイト足りません。」

**実装者**「ここで『3バイト文字っぽいから 1文字』と数えるとまずいですね。」

**解説者**「そうです。`E3` は 3バイト候補ですが、まだ 1文字として確定できません。だから全体を 2文字とは数えません。正しくは、先頭の `41` までは 1文字として確定でき、その先は途中切れです。」

**初学者**「なるほど。長さ候補が分かっただけでは、まだ数えてはいけないんですね。」

**解説者**「その通りです。」

この対話のポイントは、**候補と確定を分けること** です。  
先頭バイトを見て候補は立つ。  
しかし必要な後続バイトがそろうまでは、まだ確定ではない。  
この区別を曖昧にしないことが、途中切れを見抜く鍵になります。

### 2-15. 誤読修正問題

ここで、もっともらしいが危ない説明を一つ取り上げます。

> UTF-8 は先頭バイトを見れば文字数が分かる。  
> だから先頭バイトだけ見て、順に数えていけばよい。

この説明は、半分だけ正しく、半分は危険です。

正しいのは、先頭バイトを見ることで文字長候補が分かる、という点です。  
危険なのは、それだけで文字として確定した気になってしまう点です。

実際には、2バイト文字なら後続 1バイト、3バイト文字なら後続 2バイトが必要です。  
それらが continuation byte の形をしているか、必要な数だけそろっているかを見なければなりません。  
さらに次章では、特定の先頭バイトでは特例条件も必要になることを見ます。

したがって、この説明は次のように言い直すべきです。

**UTF-8 では、先頭バイトを見ることで文字長候補を決められる。  
ただし、文字として確定するには、必要な continuation byte が正しくそろっているかを確認しなければならない。**

本書では、この「候補」と「確定」を分ける言い方を大事にします。

### 2-17. この章のまとめ

この章では、UTF-8 を使って、本書全体の基本動作を練習しました。

最初にやることは、先頭バイトを見ることです。  
そこで 1バイト文字候補、2バイト文字候補、3バイト文字候補を決めます。  
そのあとで、必要な continuation byte が正しくそろっているかを確認します。  
条件がそろって初めて、そこまでを 1文字として切れます。  
1文字切れたら、残りの先頭へ進んで同じことを繰り返します。

この章で特に重要だったのは、次の二点です。

一つ目は、**1文字を左から順に切る** ことです。  
全体をまとめて眺めるのではなく、先頭から 1文字ずつ確定させます。

二つ目は、**候補と確定を分ける** ことです。  
先頭バイトを見て長さ候補は分かります。  
しかし、必要な continuation byte が足りなければ、まだ 1文字として確定できません。  
途中切れを見抜くには、この区別が必要です。

次章では、ここにもう一段だけ厳しさを加えます。  
UTF-8 では、continuation byte の形が合っているだけでは十分でない場合があります。  
E0 / ED / F0 / F4 などの特例条件を見落とすと、もっともらしいが危ない読み方になります。  
次章では、その特例を扱います。
