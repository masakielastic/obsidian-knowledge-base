
## 第7章　static で読めるものと読めないものを分ける

第6章では、HPACK の最初の読み方として Indexed Header Field を扱いました。  
そこでは、先頭ビットを見て形式を決め、index を取り出し、static table を引いてヘッダーを復元しました。

この章では、その読み方を一歩進めます。  
狙いは、**static table だけで読めるもの** と **static table だけでは読めないもの** を区別できるようになることです。

ここで大切なのは、いきなり HPACK の全部を説明しようとしないことです。  
本書では、読めるところまでを確定し、読めないところでは無理に先へ進まない、という姿勢を重視します。  
これは UTF-8 で途中切れを見抜いたときと似ています。  
HTTP/2 で Length 不整合を見て止まったときとも似ています。  
HPACK でも、**ここまでは static table で読める、ここから先は別の読み方が必要** と言えることが大切です。

この章では、まず static のみで完結する短い例を確認します。  
次に、途中で literal が現れる最小例を見て、「ここから先は static table だけでは読めない」と判断する練習をします。

---

### 7-1. 「読める」とは何を意味するのか

第6章では、`82`、`84`、`87` のような短い例を static table だけで読めました。  
こうした例では、

- 先頭ビットを見る
    
- Indexed Header Field と決める
    
- index を取り出す
    
- static table を引く
    

という流れで最後まで復元できました。

この章でまず整理したいのは、「static で読める」とは何を意味するのか、ということです。

本書で「static で読める」と言うとき、それは **そのヘッダー表現の復元に static table 以外の追加情報が要らない** という意味です。  
逆に「static だけでは読めない」とは、先頭を見た結果、Indexed Header Field ではない、あるいは index だけでは値が決まらない形式だと分かる、ということです。

ここで大事なのは、無理に最後まで読んだふりをしないことです。  
読める範囲を正確に区切る。  
読めない理由を言葉にする。  
この章は、その練習です。

---

### 7-2. static だけで読める例を見抜く

まず、static だけで読める短い例を見ます。

`82 84 87`

この例では、先頭の `82` を見ると先頭ビットが 1 なので Indexed Header Field です。  
index 2 を static table に引くと `:method: GET` です。

次の `84` も同じように Indexed Header Field です。  
index 4 を static table に引くと `:path: /` です。

最後の `87` も Indexed Header Field です。  
index 7 を static table に引くと `:scheme: https` です。

このように、各表現がどれも index と static table だけで復元できます。  
したがって、この列は **static table だけで最後まで読める例** です。

ここで見てほしいのは、結果そのものよりも、「各表現がどの段階で最後まで読めると分かるか」です。  
HPACK では、最初に先頭ビットを見て形式を決めます。  
その時点で Indexed Header Field と分かれば、この章の範囲では static table に進めます。  
つまり、**形式判定の段階で、読めるかどうかの見通しが立つ** わけです。

---

### 7-3. 並び順が変わっても読み方は同じ

次に、同じく static だけで読めるが、順番の違う例を見ます。

`87 82 84`

この例でも、やることは変わりません。  
先頭 `87` は Indexed Header Field なので static table の 7番を引きます。  
`:scheme: https` です。

次の `82` は static table の 2番で `:method: GET` です。  
最後の `84` は static table の 4番で `:path: /` です。

つまり順番が変わっても、読解動作は同じです。  
大切なのは、「よく出る並びだからこうだろう」と決め打ちしないことです。  
毎回、先頭から 1表現ずつ切って、static table に戻ります。

ここでの学習目標は、値を覚えることではなく、**どんな順番でも 1表現ずつ復元できること** です。

#### 問題1

`82 84 87`

#### 問題2

`87 82 84`

#### 問題3

`83 86`

読むときは、次の順番を守ります。

- 先頭ビットを見る
    
- Indexed Header Field かどうかを決める
    
- index を取り出す
    
- static table を引く
    
- 1表現を確定する
    
- 残りがあれば次の先頭へ進む
    

---

### 7-5. 模範解答：static だけで読める例

#### 問題1の考え方

`82 84 87` は、どのバイトも先頭ビットが 1 なので Indexed Header Field として読めます。  
したがって static table だけで最後まで読めます。

- `82` → index 2 → `:method: GET`
    
- `84` → index 4 → `:path: /`
    
- `87` → index 7 → `:scheme: https`
    

#### 問題2の考え方

`87 82 84` も、どのバイトも Indexed Header Field として読めます。  
したがって static table だけで最後まで読めます。

- `87` → index 7 → `:scheme: https`
    
- `82` → index 2 → `:method: GET`
    
- `84` → index 4 → `:path: /`
    

#### 問題3の考え方

`83 86` も、どちらも Indexed Header Field として読めます。  
したがって static table だけで最後まで読めます。

- `83` → index 3 → `:method: POST`
    
- `86` → index 6 → `:scheme: http`
    

この三つの例に共通しているのは、**各表現がすべて Indexed Header Field であり、index を static table に引くだけで終わる** ことです。

### 7-6. 3人で読む：static だけで読める例

扱う例は次です。

`82 84 87`

**初学者**「これは第6章で見た型なので、全部 static table で読めそうです。」

**解説者**「その見通しはよいです。ただし、最初から全部まとめてそう言うのではなく、1表現ずつ確認しましょう。先頭の `82` はどうですか。」

**初学者**「先頭ビットが 1 なので Indexed Header Field です。index 2 を static table に引くと `:method: GET` です。」

**解説者**「次の `84` はどうですか。」

**初学者**「同じく Indexed Header Field で、index 4、`:path: /` です。」

**実装者**「この段階で『全部読める列だ』と言えるのは、各表現が全部 Indexed Header Field だと確かめたからですね。」

**解説者**「そうです。最後の `87` も static table で読めます。だからこの列は static だけで最後まで読めます。」

**初学者**「つまり、見慣れた値の並びだからではなく、各表現の形式を確認した結果として『全部読める』と判断しているわけですね。」

**解説者**「その通りです。」

この対話で大切なのは、「たまたま知っている並び」ではなく、**形式判定の積み重ねで読めると判断している** ことです。

### 7-7. static だけでは読めない例は何が違うのか

次に、static だけでは読めない最小例を見ます。

`40`

この 1バイトを見たとき、先頭ビットは 1 ではありません。  
したがって、これは第6章で扱った Indexed Header Field ではありません。

ここで重要なのは、すぐに「読めないから終わり」と投げることではありません。  
まず言うべきなのは、**これは Indexed Header Field ではないので、index を static table に引くだけでは復元できない** ということです。

この章の範囲では、literal の詳細までは扱いません。  
したがって、この時点でできる正しい判断は次のようになります。

- 先頭を見た
    
- Indexed Header Field ではないと分かった
    
- したがって static table だけでは最後まで読めない
    
- ここから先は別の形式の読み方が必要
    

この判断ができれば十分です。

ここで避けたいのは、見覚えのある疑似ヘッダーに無理に当てはめることです。  
`40` を見て「たぶん Host だろう」などと推測して進むのは、本書のやり方ではありません。  
分かることと分からないことを分けます。

### 7-8. static のあとに literal が来る最小例

次に、static で読める部分と、そうでない部分が混ざった例を見ます。

`82 40`

先頭の `82` は、第6章と同じように Indexed Header Field です。  
index 2 を static table に引くと `:method: GET` です。  
ここまでは static table だけで読めます。

しかし次の `40` は、先頭ビットが 1 ではありません。  
したがって、この表現は Indexed Header Field ではありません。  
この章の範囲では、ここから先は static table だけでは読めないと判断します。

したがって、この列は次のように読めます。

- 先頭の 1表現 `82` までは static table で読める
    
- その次の `40` から先は、別の形式なので static table だけでは読めない
    

ここで大切なのは、「全体として読める / 読めない」を一気に言うのではなく、**どこまで読めて、どこから読めないか** を切り分けることです。  
これは本書全体で重視している動作です。

---

### 7-9. `82 84 40` をどう読むか

さらに一歩進めて、次の例を見ます。

`82 84 40`

先頭の `82` は Indexed Header Field なので static table の 2番、`:`method: GET`です。 次の`84 `も Indexed Header Field なので static table の 4番、`:path: /\` です。

しかし、その次の `40` は Indexed Header Field ではありません。  
したがって、そこで static table だけによる読解は止まります。

この列について本書の立場で言えるのは、次のことです。

- 1表現目は `:method: GET`
    
- 2表現目は `:path: /`
    
- 3表現目は literal 系の別形式なので、この章の範囲では static table だけでは読めない
    

ここで間違えやすいのは、「最初の二つが読めたから、三つ目も何とか static table で押し切ろう」とすることです。  
しかしそれでは形式判定を飛ばしてしまいます。  
本書では、分かる範囲と分からない範囲をきちんと区切ります。

### 7-11. 模範解答：static で読める範囲を答える

#### 問題1の考え方

`40` は先頭ビットが 1 ではないので Indexed Header Field ではありません。  
したがって、この章の範囲では static table だけでは読めません。  
読める範囲は 0表現です。

#### 問題2の考え方

`82` は Indexed Header Field なので static table の 2番、`:method: GET` です。  
次の `40` は Indexed Header Field ではありません。  
したがって、先頭の 1表現までは static table で読めますが、その次からは読めません。

#### 問題3の考え方

`82` は `:method: GET`、`84` は `:path: /` として static table で読めます。  
次の `40` は別形式です。  
したがって、先頭 2表現までは読めますが、その次からは static table だけでは読めません。

#### 問題4の考え方

`82` は `:method: GET` として読めます。  
次の `40` は別形式です。  
その時点で 1表現目までは読めたと言えますが、`40` の長さや構造をこの章の範囲では確定できません。  
したがって、その後ろの `87` まで自動的に切り分けて読むことはできません。  
この章の範囲で正しく言えるのは、**先頭の `82` までは読めるが、`40` から先は static table だけでは読めない** ということです。

ここで特に重要なのは問題4です。  
`87` 自体は見覚えのある Indexed Header Field ですが、その前の `40` をどう切るか分からない以上、そこへ安全に到達したとは言えません。  
見えている値だけ拾ってはいけません。

### 7-14. 誤読修正問題

ここで、もっともらしいが危ない説明を取り上げます。

> HPACK は static table を見ればだいたい読める。  
> 知っている値が出てきたら、そのまま順に復元していけばよい。

この説明の危ないところは、**形式判定を飛ばしている** ことです。

第6章までの短い例では、たしかに `82` や `84` のような indexed header ばかりを見てきました。  
そのため、「HPACK は static table の番号を読むものだ」と感じやすくなります。  
しかし実際には、HPACK には indexed 以外の形式もあります。  
そのため、先頭ビットを見て形式を判定せずに static table に飛びつくと、読めない表現まで無理に読もうとしてしまいます。

また、「途中で見覚えのある `87` があるから、そこからまた読めるはずだ」と考えるのも危険です。  
その手前の表現の長さや構造が分からなければ、そこへ安全に到達したとは言えません。

したがって、この説明は次のように言い直すべきです。

**HPACK では、まず各表現の先頭を見て形式を判定する。  
Indexed Header Field であれば static table によって復元できる。  
しかし別形式が現れたら、static table だけで最後まで読めるとは限らないので、どこまで読めたかを区切って判断しなければならない。**

この言い直しができれば、HPACK を「知っている値探し」ではなく、**形式を見て切る読解問題** として扱えています。

この一覧で見てほしいのは、答えの暗記ではなく、**読める範囲をどう切るか** です。

### 7-16. この章のまとめ

この章では、HPACK で static table だけで読めるものと、そうでないものを区別しました。

第6章で学んだ基本は変わりません。

- 先頭ビットを見る
    
- Indexed Header Field かどうかを決める
    
- index を取り出す
    
- static table を引く
    
- 1表現を復元する
    

ただし、この章ではそこに一つ追加されました。

- Indexed Header Field でなければ、static table だけでは読めないと判断する
    

この追加によって、HPACK に対する見方が少し変わります。  
HPACK は「番号を見て表を引くだけ」のものではありません。  
まず形式判定があり、その結果として static で読める場合もあれば、そうでない場合もあります。

この章で特に重要だったのは、**どこまで読めたかを答える** ことです。  
全部読めるか全部読めないか、の二択ではありません。  
先頭の 1表現だけは読める。  
2表現までは読める。  
その先はこの章の範囲では読めない。  
そうした区切り方が大切です。

これは、本書全体で一貫している読み方でもあります。

- UTF-8 では、途中切れなら確定できた文字までを答える
    
- HTTP/2 では、Length 不整合なら確定できたフレームまでを区別する
    
- HPACK では、形式判定の結果 static で読める表現までを区切る
    

次章では、HTTP/1 の見慣れたヘッダー表現を足場にしながら、HTTP/2 と HPACK の対応を見ます。  
そこで狙うのは、「生データの側」と「人間に読めるヘッダーの側」が別世界ではないと分かることです。  
ただし、そのときもやることは変わりません。  
先頭を見る。  
表を引く。  
1単位ずつ切る。  
そして、対応を確かめながら次へ進みます。
