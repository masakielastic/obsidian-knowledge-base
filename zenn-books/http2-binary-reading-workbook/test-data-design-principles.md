# 表を引いて読むバイナリ実習 テストデータ設計原則

## このファイルの役割

このファイルは、書籍  
**『表を引いて読むバイナリ実習 ― UTF-8 / HTTP/2 / HPACK で学ぶ「切る・数える・復元する」』**  
で使うテストデータの設計原則をまとめたものである。

本書では、テストデータは単なる例ではない。  
**どこを見て、どの表を引き、どこで切るかを学ばせる教材そのもの** である。  
したがって、データ設計では「正しい例を並べること」よりも、次を優先する。

- 読者の視線をどこに向けるか
- 何を比較させるか
- どの誤読を起こさせるか
- どう修正させるか
- どの動作を反復させるか

---

## 1. 本書におけるテストデータの位置づけ

本書で扱うのは、抽象概念の説明用サンプルではなく、**読解動作を訓練するための最小データ** である。  
したがって、テストデータには次の役割がある。

- 先頭を見るきっかけを作る
- 表を引く必要性を作る
- 1単位の切り方を明示する
- 正常 / 不正 / 途中切れ / 不整合を比較させる
- もっともらしい誤読を誘発し、その危険を示す
- 類似例の差分から学ばせる
- HTTP/1 から raw bytes への対応を見えるようにする

テストデータは「答えを確認するための入力」ではなく、**読むための視線誘導装置** と考える。

---

## 2. 基本原則

### 原則1: まず最小例を作る

各題材では、まず「最小の正常例」を作る。  
最小例がないまま複合例に進むと、読者は何が本質で何がノイズかを見分けにくい。

例:

- UTF-8: `41`, `C3 A9`, `E3 81 82`
- HTTP/2: 空の SETTINGS, 1バイト DATA
- HPACK: `82`, `84`, `87`

### 原則2: 1問1論点を守る

1つの問題に複数の主論点を詰め込まない。  
特に模範問題では、読者が最初に立てるべき論点仮説を 1つに絞る。

悪い例:
- UTF-8 の特例、途中切れ、複数文字列の混在を 1問で全部やらせる
- HTTP/2 の連結、Flags、Length 不整合を同時に扱う
- HPACK の static / literal / Huffman を一度に出す

良い例:
- まず先頭バイトから文字長候補を決める
- まず 9バイト共通ヘッダを切る
- まず static table を引く

### 原則3: 比較で学ばせる

本書では、単発の例より **似ている二つを並べる設計** を優先する。  
比較によって、読者は「ここが違うから判断が変わる」を学びやすい。

例:

- `E0 9F 80` と `E0 A0 80`
- 空 SETTINGS と 1バイト DATA
- `82 84 87` と `82 85 87`

### 原則4: 正常系だけで終わらせない

正常系だけでは、「どこで失敗するか」が見えない。  
各ユニットには、必要に応じて次のいずれかを入れる。

- 途中切れ
- 特例違反
- Length 不整合
- static だけでは読めないケース
- 一般化しすぎると誤るケース

### 原則5: 典型例は少数精鋭にする

問題数を増やしすぎない。  
本書では、量よりも **比較・反復・誤読修正** の設計が重要である。

## 3. テストデータ設計の観点

### 観点A: 先頭を見させられるか

本書の中核動作は「先頭を見る」である。  
したがって、テストデータは先頭情報が意味を持つように設計する。

例:
- UTF-8 では先頭バイトで文字長候補が変わる
- HTTP/2 では先頭 9バイトを共通ヘッダとして読む
- HPACK では先頭ビットで表現形式が変わる

### 観点B: 表を引かないと危険か

「見た感じ」で解けるデータだけだと、本書の狙いが弱くなる。  
データは、**表を引かないと危ない** ものを含むべきである。

例:
- E0 / ED / F0 / F4 の特例
- HTTP/2 Type 一覧を引かないと型が確定しないケース
- HPACK static table を実際に引かなければ答えられないケース

### 観点C: 1単位を切る必要があるか

データは、切り方を問うものであるべきである。  
ただ眺めるだけではなく、どこで 1文字 / 1フレーム / 1表現を区切るかが問われる必要がある。

### 観点D: 誤読が起きる余地があるか

よいデータは、簡単すぎて一目で終わるものではない。  
ただし、難しすぎてもいけない。  
**もっともらしい誤読が起きる余地** があるくらいがよい。

### 観点E: 類似例との比較がしやすいか

単独ではなく、ペアやセットで設計できるとよい。  
読者は比較によって差分を見る。

## 4. 問題タイプごとの設計原則

### A. 模範問題

模範問題は、読解手順そのものを見せる問題である。

#### 目的
- 最初に何を見るかを示す
- どの表を引くかを示す
- どこで切るかを示す
- 誤った近道を止める
- 次へどう進むかを示す

#### 設計原則
- 最小例または典型例を使う
- 主論点は 1つに絞る
- 途中経過を見せやすいデータにする
- 3人対話で誤読を出しやすいものを選ぶ

### B. 練習問題

練習問題は、模範問題と同じ型を読者に自力で反復させる問題である。

#### 目的
- 読み方の反復
- 比較による差分学習
- 正常 / 不正の見分け
- 途中切れや不整合の確認

#### 設計原則
- 模範問題と同型にする
- 1か所だけ違う問題を混ぜる
- 模範問題の一般化範囲を確かめる
- 新要素を増やしすぎない

### C. 誤読修正問題

誤読修正問題は、本書の個性を担う問題である。

#### 目的
- もっともらしい誤読を可視化する
- 論点仮説 / アプローチ仮説 / 答え仮説のずれを示す
- 「筋は良いが危ない近道」を修正する

#### 設計原則
- 誤答は本当にもっともらしいものにする
- バカげた間違いは避ける
- どこで見方がずれたかを説明できるようにする
- 正答ではなく、正しい手順へ戻すことを主眼にする

### D. 典型例一覧

典型例一覧は、暗記表ではなく見慣れ用の足場である。

#### 目的
- 典型パターンを一望する
- 次に似たデータを見たときの手がかりにする
- 差分の感覚を身体化する

#### 設計原則
- 1行1例で短くする
- 類似例を近くに置く
- 正常 / 不正 / 不整合を混在させる場合は整理して見せる
- 本文の代わりにしない

## 5. UTF-8 のテストデータ設計原則

### 重点
- 先頭バイトで文字長候補を決める
- continuation byte を確認する
- 途中切れを見抜く
- 特例条件を落とさない

### 最初に置くべき最小例
- `41`
- `C3 A9`
- `E3 81 82`

### 比較に向く例
- `41 E3 81` と `41 E3 81 82`
- `E0 9F 80` と `E0 A0 80`
- `ED 9F BF` と `ED A0 80`
- `F0 8F 80 80` と `F0 90 80 80`
- `F4 8F BF BF` と `F4 90 80 80`

### 誘発したい誤読
- continuation byte であることだけ見て通す
- 先頭バイト表だけで十分だと考える
- 途中切れでも文字数を数えてしまう
- 特例表を見ずに長さだけで判断する

### 避けるべき設計
- 文字数が多すぎる長文サンプル
- Unicode 一般論が必要になる題材
- 正規化や書記素クラスタの議論を呼び込むデータ
- 学習初期に複雑な 4バイト列を大量投入すること

---

## 6. HTTP/2 のテストデータ設計原則

### 重点
- 9バイト共通ヘッダを切る
- Length / Type / Flags / Stream ID を読む
- payload と照合する
- フレーム境界を誤らない
- Length 不整合を見抜く

### 最初に置くべき最小例
- 空の SETTINGS
- 1バイト DATA

### 比較に向く例
- 空 SETTINGS と短い DATA
- 単一フレームと 2フレーム連結
- Length 一致と Length 不整合
- payload 途中切れあり / なし

### 誘発したい誤読
- payload 側から先に読もうとする
- Type だけ見れば十分だと思う
- Length を見ずに雰囲気でフレーム境界を決める
- 手元にある残り全部を payload だと思う

### 避けるべき設計
- 初期段階から複雑な HEADERS フレームを出す
- Flags やパディングや優先度を最初から大量に混ぜる
- 状態遷移を前提にしないと理解できない例を早く出しすぎる
- 実運用キャプチャをそのまま長く貼るだけの設計

---

## 7. HPACK のテストデータ設計原則

### 重点
- 先頭ビットで表現形式を決める
- static table を引く
- 1ヘッダー表現ずつ復元する
- static だけで読めるものと読めないものを分ける

### 最初に置くべき最小例
- `82`
- `84`
- `87`
- `82 84 87`

### 比較に向く例
- `82 84 87` と `82 85 87`
- static のみの例と literal を含む例
- 1つだけ index が違う例
- 同じ疑似ヘッダー群だが path だけ違う例

### 誘発したい誤読
- 値を丸暗記して読む
- 1バイト 1ヘッダーだと一般化する
- static table から全部読めると思う
- 既知の HTTP/1 メッセージから逆算してしまう

### 避けるべき設計
- dynamic table 前提の複雑な例を早く出しすぎる
- Huffman を含む複雑例を導入に混ぜる
- literal が長すぎて、何を学ぶ問題か分からなくなる例
- static table を引かなくても勘で答えられる例ばかりにすること

---

## 8. HTTP/1 を足場にするデータ設計

特に HTTP/2 / HPACK では、HTTP/1 の見慣れたメッセージを足場にする。  
テストデータは、HTTP/1 から HTTP/2 / HPACK への対応が見えるように設計するとよい。

### 典型的な出発点
```http
GET / HTTP/1.1
Host: www.example.com
```

あるいは

```http
GET /index.html HTTP/1.1
Host: www.example.com
```

### 設計の狙い
- 読者が意味の単位を先に把握できる
- HPACK static table の index が変換表として見える
- path などの差分が raw bytes 側でどう表れるか比較しやすい

### 注意
- HTTP/1 を先に置いても、最終的には raw bytes を読ませる
- HTTP/1 からの逆算で答えが出るだけの問題にはしない
- 既知知識が補助線であり、主役は生データであることを保つ

---

## 9. 「よいテストデータ」の条件

本書にとってよいテストデータは、次の条件を満たす。

- 最初に見るべき場所がある
- 引くべき表がある
- 1単位を切る必要がある
- 比較可能な類似例がある
- もっともらしい誤読が起きる
- 誤読を修正する価値がある
- 模範解答で途中経過を見せやすい
- 練習問題へ自然に展開できる
- 典型例一覧へ圧縮しやすい

逆に、次のようなデータは本書に向かない。

- 何を学ぶ問題かが複数ありすぎる
- 長すぎて焦点がぼける
- 正解が一目で分かりすぎる
- 誤読の余地がない
- 既知知識だけで解けてしまう
- 上級知識が前提になる
- 本書の 5動作につながらない

---

## 10. 実際の設計手順

テストデータを作るときは、次の順で考えるとよい。

### 手順1: その問題の主論点を 1文で書く
例:
- 先頭バイトから文字長候補を決める
- 9バイト共通ヘッダを切る
- static table を引いて復元する

### 手順2: 最小の正常例を作る
まずは成功例を最小サイズで作る。

### 手順3: 1か所だけ違う比較例を作る
どの差分で判断が変わるかを見せる。

### 手順4: 途中切れ / 不整合 / 特例違反を作る
正常系だけで終わらせない。

### 手順5: もっともらしい誤読を言葉にする
そのデータでどんな危ない近道が起きるかを書く。

### 手順6: 模範問題 / 練習問題 / 誤読修正問題 / 一覧に振り分ける
同じデータでも役割が違えば使い方が違う。

---

## 11. 校正時の確認項目

テストデータのレビューでは、次を確認する。

- そのデータは 1ユニット 1論点を守っているか
- 最初に何を見るかが明確か
- 表を引く必要が本当にあるか
- 切り方を学べるか
- 比較対象が適切か
- 誤読が本当にもっともらしいか
- 不自然な難しさになっていないか
- 典型例として再利用しやすいか
- 上級話題を呼び込みすぎていないか

---

## 12. 最後に

本書では、テストデータは飾りではない。  
**本書の内容そのもの** である。

したがって、テストデータを作るときは、まず次を問うこと。

- 読者はこのデータのどこを見るのか
- どの表を引くのか
- どこで 1単位を切るのか
- どんな誤読をしうるのか
- その誤読をどう修正するのか

この問いに答えられないデータは、本書では優先度が低い。  
逆に、この問いに答えられるデータこそが、本書の実習を支える。
