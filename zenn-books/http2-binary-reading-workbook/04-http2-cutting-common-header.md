## 第4章　9バイト共通ヘッダを切る

第3章まで、UTF-8 を題材にして「先頭を見て、表を引いて、1単位を切る」という読み方を練習してきました。  
UTF-8 では、1単位は 1文字でした。  
先頭バイトを見て長さ候補を決め、必要なら特例条件まで確認して、そこまでを 1文字として切りました。

この章からは題材を HTTP/2 に移します。  
HTTP/2 で切る 1単位は、1文字ではなく **1フレーム** です。

HTTP/2 の生データを見ると、最初は UTF-8 よりも固そうに見えるかもしれません。  
しかし、本書の観点ではやることは変わりません。  
最初に先頭を見る。  
そこで何を 1単位として切るべきかを決める。  
必要な表を引いて意味を取る。  
そして次へ進む。

HTTP/2 で最初に身につけるべきことは、細かなフレーム種別の暗記ではありません。  
まず重要なのは、**HTTP/2 のフレームは先頭から 9バイトの共通ヘッダを持つ** という読み方です。  
この 9バイトをきちんと切れれば、payload の長さと種別を読み分ける土台ができます。  
逆にここを曖昧にすると、その先の読みは全部ずれます。

この章では、まず空の SETTINGS フレームを読みます。  
次に 1バイトの DATA フレームを読みます。  
どちらも最小例ですが、この二つで共通ヘッダの切り方をしっかり固めます。

---

### 4-1. HTTP/2 を読むとき、最初に何を見るか

UTF-8 では先頭バイトを見ました。  
HTTP/2 では、まず **先頭 9バイト** を見ます。

HTTP/2 の各フレームは、共通して次の形を持っています。

- Length: 3バイト
    
- Type: 1バイト
    
- Flags: 1バイト
    
- R + Stream Identifier: 4バイト
    

合計 9バイトです。

ここでの最重要ポイントは、HTTP/2 のフレームをいきなり payload から読まないことです。  
何のフレームかを「見た感じ」で当てにいかないことです。  
まず先頭から 9バイトを切り、その 9バイトを共通ヘッダとして読む。  
この順番を崩しません。

本書では、この 9バイトを読むとき、最低限次の問いを持ちます。

- Length はいくつか
    
- Type は何か
    
- Flags は何か
    
- Stream Identifier はいくつか
    

この段階では、まだ payload の中身を詳しく読む必要はありません。  
先にやるべきことは、**1フレームがどこまで続くかを決めること** です。  
HTTP/2 では、そのために Length が必要です。  
したがって、共通ヘッダを飛ばして payload を眺める読み方は、本書では採りません。

---

### 4-2. まず 9バイトを切る

最初の最小例を見ます。

`00 00 00 04 00 00 00 00 00`

このバイト列は 9バイトあります。  
HTTP/2 では、まずこの 9バイト全体を共通ヘッダとして切ります。

順に見ていきます。

- 最初の 3バイト `00 00 00` が Length
    
- 次の 1バイト `04` が Type
    
- 次の 1バイト `00` が Flags
    
- 最後の 4バイト `00 00 00 00` が R + Stream Identifier
    

この時点で分かることは多くありません。  
しかし大切なのは、**何がどこに置かれているか** がはっきりすることです。

まず Length は `00 00 00` なので 0 です。  
つまり、このフレームには payload がありません。  
次に Type は `04` です。  
Type 一覧を引くと、これは SETTINGS です。  
Flags は `00` です。  
最後の Stream Identifier は 0 です。

ここで重要なのは、「空だから特殊ケース」ではなく、「Length が 0 だから payload がない」と読むことです。  
つまり、payload がないという事実も、共通ヘッダを読んだ結果として分かるのであって、見た目で決めるのではありません。

---

### 4-3. 空の SETTINGS フレームを読む

今の例を、フレームとしてまとめてみます。

`00 00 00 04 00 00 00 00 00`

まず先頭 9バイトを共通ヘッダとして切ります。  
Length は 0 です。  
Type は `04` なので SETTINGS です。  
Flags は 0 です。  
Stream Identifier は 0 です。  
Length が 0 なので payload はありません。

したがって、これは **空の SETTINGS フレーム 1個** として読めます。

この例は短いですが、HTTP/2 を読むときの重要な感覚が入っています。

- フレーム全体を一目で当てない
    
- まず 9バイト共通ヘッダを切る
    
- Length を見る
    
- Type を表で確認する
    
- payload があるかないかを Length で判断する
    

この順番が重要です。

ここでよくある危ない近道は、「Type が `04` だから SETTINGS」とだけ見て終わってしまうことです。  
もちろん Type は大事です。  
しかし、それだけではフレーム全体は確定しません。  
Length がいくつか、Stream Identifier がいくつか、payload があるのかないのかまで含めて初めて 1フレームが読めます。

---

### 4-4. 模範問題：空の SETTINGS

次のバイト列を読んでください。

`00 00 00 04 00 00 00 00 00`

このバイト列は、どのような HTTP/2 フレームとして読めますか。

読むときは、次の順番を守ります。

- 先頭から 9バイトを切る
    
- Length を読む
    
- Type を表で確認する
    
- Flags を読む
    
- Stream Identifier を読む
    
- payload の有無を Length で判断する
    

---

### 4-5. 模範解答：空の SETTINGS

先頭から 9バイトを共通ヘッダとして切ります。

`00 00 00 | 04 | 00 | 00 00 00 00`

最初の 3バイト `00 00 00` は Length です。  
値は 0 です。  
したがって payload は 0バイトです。

次の 1バイト `04` は Type です。  
Type 一覧で確認すると SETTINGS です。

次の 1バイト `00` は Flags です。

最後の 4バイト `00 00 00 00` は R + Stream Identifier です。  
Stream Identifier は 0 です。

したがって、このバイト列は **payload を持たない SETTINGS フレーム 1個** です。

ここでの要点は、「空のフレームに見える」ではなく、Length=0 だから payload がない、と言えることです。

---

### 4-6. 3人で読む：空の SETTINGS

扱う例は次です。

`00 00 00 04 00 00 00 00 00`

**初学者**「9バイトしかないので、全部で 1つの短いフレームだとは思います。ただ、どこから意味を取ればいいか少し迷います。」

**解説者**「HTTP/2 では、まず先頭 9バイトを共通ヘッダとして切ります。この例はちょうど 9バイトなので、まず全部を共通ヘッダと見ます。」

**実装者**「payload を見にいく前に、まず構造を切るわけですね。」

**解説者**「そうです。では最初の 3バイトは何ですか。」

**初学者**「Length です。`00 00 00` なので 0 です。」

**解説者**「次の `04` は何ですか。」

**初学者**「Type です。表を見ると SETTINGS です。」

**解説者**「その通りです。Flags は `00`、最後の 4バイトは Stream Identifier です。Length は 0 なので payload はありません。」

**実装者**「つまり、『空だから特殊』ではなく、『共通ヘッダを読んだ結果、Length=0 なので payload がない』と読むわけですね。」

**解説者**「そうです。HTTP/2 はまず共通ヘッダから入ります。」

この対話で大切なのは、payload の意味を推測する前に、まず 9バイトを切って構造を確定していることです。  
HTTP/2 では、この順番が最優先です。

---

### 4-7. 1バイト DATA フレームを読む

次に、payload を 1バイトだけ持つ最小例を見ます。

`00 00 01 00 00 00 00 00 01 FF`

この例は 10バイトあります。  
HTTP/2 では、まず先頭 9バイトを共通ヘッダとして切ります。

`00 00 01 | 00 | 00 | 00 00 00 01 | FF`

順に見ます。

- Length は `00 00 01`
    
- Type は `00`
    
- Flags は `00`
    
- Stream Identifier は `00 00 00 01`
    

そして、Length が 1 なので、共通ヘッダのあとに payload が 1バイト続くはずです。  
実際、残りは `FF` の 1バイトです。  
これが payload です。

Type `00` は DATA です。  
したがって、このバイト列は **1バイトの payload を持つ DATA フレーム** として読めます。

ここで重要なのは、payload `FF` を先に見て意味を考えないことです。  
まず Length=1 だから payload は 1バイトだけ取る。  
そのあとで Type が DATA だと分かる。  
この順番です。

---

### 4-8. Length は何を表しているのか

ここで一度、Length の役割をはっきりさせておきます。

HTTP/2 の Length は、**payload の長さ** を表します。  
9バイト共通ヘッダを含んだ長さではありません。  
したがって、

- フレーム全体の長さ = 9バイト共通ヘッダ + Length バイトの payload
    

となります。

この点を曖昧にすると、1フレームがどこまでかを誤ります。  
たとえば先ほどの DATA の例では、Length=1 なので、共通ヘッダのあとに 1バイトだけ payload を取ります。  
もし Length をフレーム全体の長さだと思ってしまうと、読み方が完全に崩れます。

HTTP/2 では、まず共通ヘッダを切る。  
その共通ヘッダの中の Length を使って payload の範囲を決める。  
この二段階でフレーム境界を確定します。

---

### 4-9. 模範問題：1バイト DATA

次のバイト列を読んでください。

`00 00 01 00 00 00 00 00 01 FF`

このバイト列は、どのような HTTP/2 フレームとして読めますか。

読むときは、次の順番を守ります。

- 先頭 9バイトを共通ヘッダとして切る
    
- Length を読む
    
- Type を表で確認する
    
- Stream Identifier を読む
    
- Length の値だけ payload を取る
    

---

### 4-10. 模範解答：1バイト DATA

先頭 9バイトを共通ヘッダとして切ります。

`00 00 01 | 00 | 00 | 00 00 00 01`

最初の 3バイト `00 00 01` は Length です。  
値は 1 です。  
したがって payload は 1バイトです。

次の 1バイト `00` は Type です。  
Type 一覧で確認すると DATA です。

次の 1バイト `00` は Flags です。

最後の 4バイト `00 00 00 01` は Stream Identifier です。  
Stream Identifier は 1 です。

Length=1 なので、その後ろの 1バイト `FF` を payload として取ります。

したがって、このバイト列は **payload が 1バイトの DATA フレーム 1個** です。

ここでの要点は、`FF` が残っているから payload だと考えるのではなく、Length=1 と読んだから payload を 1バイト取る、という順番です。

---

### 4-11. 3人で読む：1バイト DATA

扱う例は次です。

`00 00 01 00 00 00 00 00 01 FF`

**初学者**「今度は 9バイトより 1バイト多いので、最後の `FF` が何かありそうです。」

**解説者**「その感覚はよいです。ただし、最後の `FF` を先に意味づけないようにしましょう。まず先頭 9バイトを共通ヘッダとして切ります。」

**実装者**「つまり、残りが payload かどうかも、Length を読んでから決めるわけですね。」

**解説者**「その通りです。では Length はいくつですか。」

**初学者**「`00 00 01` なので 1 です。」

**解説者**「では payload は何バイトですか。」

**初学者**「1バイトです。」

**解説者**「次に Type は `00` です。これは何ですか。」

**初学者**「DATA です。」

**解説者**「そうです。したがって、このフレームは payload を 1バイト持つ DATA フレームです。最後の `FF` は、その 1バイトの payload です。」

**実装者**「ここでも payload 側から読まずに、Length で境界を決めてから payload を取るのが大事なんですね。」

**解説者**「まさにそこが大事です。」

この対話のポイントは、残りのバイトを先に眺めて判断しないことです。  
HTTP/2 では、共通ヘッダを読んだ結果として payload の範囲が決まります。  
その順番を逆にしません。

---

### 4-12. Type だけ見れば十分なのか

ここで、HTTP/2 の読み方で起きやすい近道を一つ止めておきます。

たとえば Type が `04` なら SETTINGS、Type が `00` なら DATA です。  
これは正しい知識です。  
しかし、それだけでフレーム全体を読んだことにはなりません。

Type だけを見ても、次のことはまだ分かりません。

- payload が何バイトあるか
    
- Stream Identifier がいくつか
    
- フレームがどこで終わるか
    

つまり Type は重要ですが、**Type だけでは 1フレームを切れません**。  
HTTP/2 でまず必要なのは、共通ヘッダ全体を読むことです。

この章では、ここを体に入れるために、わざと最小例しか扱っていません。  
空の SETTINGS も、1バイト DATA も、payload の意味を深追いしなくてよいので、「まず共通ヘッダを切る」という型だけに集中できます。

#### 練習2

`00 00 02 00 00 00 00 00 01 AA BB`

#### 練習3

`00 00 00 00 00 00 00 00 01`

`00 00 01 04 00 00 00 00 00 FF`

読むときは、次の順番を守ります。

- 先頭 9バイトを共通ヘッダとして切る
    
- Length を読む
    
- Type を表で確認する
    
- Flags を読む
    
- Stream Identifier を読む
    
- Length の値だけ payload を取る
    

この段階では、Flags の詳細な意味に深入りする必要はありません。  
まずは共通ヘッダと payload の境界を正確に取れることを重視してください。

---

### 4-14. 誤読修正問題

ここで、もっともらしいが危ない説明を一つ取り上げます。

> HTTP/2 のフレームは、Type を見ればだいたい分かる。  
> だから先頭の型だけ見て、あとは中身を読めばよい。

この説明は、半分は正しいですが、半分は危険です。

正しいのは、Type がフレーム種別を示していることです。  
危険なのは、Type だけでフレーム境界まで決まった気になってしまうことです。

HTTP/2 のフレームを読むには、まず共通ヘッダ全体を切らなければなりません。  
Length を見て payload の長さを決め、Stream Identifier を読み、そこまでを 1フレームとして確定する必要があります。

したがって、この説明は次のように言い直すべきです。

**HTTP/2 のフレームは、まず先頭 9バイトの共通ヘッダを切って読む。  
Type はその一部としてフレーム種別を示すが、フレーム境界を決めるには Length も含めて共通ヘッダ全体を読まなければならない。**

この言い直しができれば、HTTP/2 を「型当て」ではなく「構造を切る読み方」として捉えられています。

---

### 4-15. 典型例一覧

この章で扱った典型例を並べておきます。

- `00 00 00 04 00 00 00 00 00`  
    payload を持たない SETTINGS フレーム
    
- `00 00 01 00 00 00 00 00 01 FF`  
    payload が 1バイトの DATA フレーム
    
- `00 00 02 00 00 00 00 00 01 AA BB`  
    payload が 2バイトの DATA フレーム
    
- `00 00 00 00 00 00 00 00 01`  
    payload を持たない DATA フレーム
    
- `00 00 01 04 00 00 00 00 00 FF`  
    payload が 1バイトの SETTINGS フレーム
    

この一覧で見てほしいのは、Type の違いだけではありません。  
Length が 0 なら payload はない。  
Length が 1 なら payload は 1バイト。  
Length が 2 なら payload は 2バイト。  
このように、**共通ヘッダを読んだ結果として payload の範囲が決まる** ことが見えていれば十分です。

---

### 4-16. この章のまとめ

この章では、HTTP/2 の読み方の最初の型を学びました。

UTF-8 では先頭バイトを見て 1文字を切りました。  
HTTP/2 では、先頭 9バイトを共通ヘッダとして切って 1フレームを読みます。  
ここが題材の違いです。  
しかし、読み方の骨格は変わりません。

- まず先頭を見る
    
- 表を引く
    
- 1単位を決める
    
- 中身を読む
    
- 次へ進む
    

HTTP/2 で特に重要なのは、**共通ヘッダを飛ばさないこと** です。  
Type だけを見て済ませない。  
payload 側から先に意味づけしない。  
まず先頭 9バイトを切る。  
Length を読む。  
その Length に従って payload の範囲を決める。  
この順番で読みます。

この章で扱ったのは、空の SETTINGS フレームと 1バイト DATA フレームでした。  
どちらも最小例ですが、ここで学んだことはこのあとずっと使います。  
HTTP/2 のフレームは、見た目で切るのではありません。  
**共通ヘッダを読み、Length に従って境界を決める** ことで初めて読めます。

次章では、この型を少し進めて、複数フレームが連結された場合や、Length と実データ長が合わない場合を扱います。  
そこでも基本は同じです。  
まず 1フレームを確定し、その次の位置へ進みます。  
この「切ってから進む」感覚を、HTTP/2 でもしっかり固めていきます。
