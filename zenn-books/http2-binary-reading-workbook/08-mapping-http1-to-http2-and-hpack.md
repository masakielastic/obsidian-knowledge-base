## 第8章　HTTP/1 から raw bytes へ落とす

第7章では、HPACK の表現を読むとき、static table だけで読めるものと、そうでないものを区別しました。  
そこで重視したのは、見覚えのある値を拾うことではなく、**先頭を見て形式を判定し、どこまで読めるかを区切る** ことでした。

この章では、その読み方をもう一歩進めます。  
ここでの狙いは、HTTP/1 の見慣れた表現を足場にしながら、HTTP/2 と HPACK の raw bytes が何を表しているのかを結びつけることです。

ただし、本書は HTTP/1 をそのまま HTTP/2 に置き換える本ではありません。  
また、HTTP/1 の文字列がそのまま HTTP/2 の生データに見える、と考える本でもありません。  
ここでやりたいのは、**意味の対応を足場にして、raw bytes 側を読む視線を育てること** です。

たとえば、次のような HTTP/1 の要求は見慣れているはずです。


```text
GET / HTTP/1.1  
Host: www.example.com
```

この表現をそのまま HPACK のバイト列として読むことはできません。  
しかし、「GET」「/」「https」といった意味単位が、HTTP/2 や HPACK ではどのような 1単位として表れるのかを対応づけることはできます。

この章では、まず `GET /` に相当する最小の疑似ヘッダー群を見ます。  
次に `/` と `/index.html` の違いが、HPACK の index の違いとしてどう現れるかを比較します。  
そこで見てほしいのは、HTTP/1 の知識がそのまま答えになることではなく、**既知の意味を補助線にしながら raw bytes 側を切って読めるようになること** です。

---

### 8-1. HTTP/1 を足場にするとはどういうことか

この章で「HTTP/1 を足場にする」と言うとき、それは HTTP/1 の文字列をそのまま探しにいくことではありません。  
むしろ逆です。  
HTTP/2 や HPACK の raw bytes は、HTTP/1 と同じ見た目をしていません。  
そのため、「文字列が見えないから別世界だ」と感じやすくなります。

本書では、その感じ方を少し和らげたいと考えています。  
つまり、

- HTTP/1 の側には見慣れた意味単位がある
    
- HTTP/2 / HPACK の側には、それを別の形で表した構造がある
    
- その対応を知ると、raw bytes 側が完全な別世界には見えなくなる
    

という状態を目指します。

ここで大事なのは、HTTP/1 の知識を答えの代わりにしないことです。  
あくまで足場です。  
最終的には、raw bytes 側で

- 何を 1単位として切るのか
    
- どの表を引くのか
    
- どこまで復元できるのか
    

を判断できなければなりません。

したがって、この章でも読み方の主役は raw bytes 側にあります。  
HTTP/1 は、その意味を見失わないための補助線です。

---

### 8-2. HTTP/1 の GET は HTTP/2 でどう見えるか

まず、もっとも短い例から始めます。

```text
GET / HTTP/1.1  
Host: www.example.com
```

HTTP/1 では、最初の行にメソッド、パス、バージョンが文字列で並びます。  
しかし HTTP/2 では、こうした情報は疑似ヘッダーとして表されます。  
たとえば、ここで注目したいのは次の三つです。

- `:method: GET`
    
- `:path: /`
    
- `:scheme: https`
    

この章では、Host の詳細までは扱いません。  
まずは、最小の疑似ヘッダー群として、この三つを足場にします。

ここで重要なのは、HTTP/1 の `GET / HTTP/1.1` が、そのまま 1本の文字列として HTTP/2 に残っているわけではないことです。  
HTTP/2 / HPACK では、これらが別々のヘッダー表現として現れます。  
つまり、HTTP/1 側ではひとまとまりに見えていたものが、raw bytes 側では **複数の 1単位** に分かれているわけです。

この見方ができるようになると、HTTP/1 の知識は「文字列を探す知識」ではなく、「何がいくつの意味単位に分かれるかを知る知識」に変わります。

---

### 8-3. `82 84 87` を HTTP/1 の足場で読む

ここで、前章までに扱った HPACK の例をもう一度見ます。

`82 84 87`

第6章では、これを次のように読みました。

- `82` → `:method: GET`
    
- `84` → `:path: /`
    
- `87` → `:scheme: https`
    

この列を、HTTP/1 側の見慣れた要求と対応づけてみます。

```text
GET / HTTP/1.1
```

HTTP/1 のこの行に近い意味を、HPACK 側では次のように持っています。

- `GET` に対応するのが `:method: GET`
    
- `/` に対応するのが `:path: /`
    
- 通信方式の補助情報として `:scheme: https`
    

もちろん、これは完全な 1対1 対応ではありません。  
HTTP/1 のリクエストラインと HTTP/2 の疑似ヘッダーは、構造が同じではないからです。  
それでも、**見慣れた意味が raw bytes 側でどう分かれているか** をつかむには十分です。

ここで見てほしいのは、`82 84 87` を「意味不明な 16進数列」ではなく、**三つの疑似ヘッダー表現の並び** として見られるようになることです。  
そのために、HTTP/1 の `GET /` という既知の形を補助線にしています。

---

### 8-4. 模範問題：GET / を対応で読む

次の二つを見比べてください。

```text
GET / HTTP/1.1
```

`82 84 87`

このとき、HPACK 側の各表現が、HTTP/1 側のどの意味単位に対応しているかを考えてください。

答えるときは、まず HPACK 側を 1表現ずつ切ってから対応づけてください。

---

### 8-5. 模範解答：GET / を対応で読む

HPACK 側の `82 84 87` は、先頭から順に 3つの Indexed Header Field として読めます。

- `82` → static table 2番 → `:method: GET`
    
- `84` → static table 4番 → `:path: /`
    
- `87` → static table 7番 → `:scheme: https`
    

これを HTTP/1 側に対応づけると、次のように見られます。

- `GET` に対応するのが `:method: GET`
    
- `/` に対応するのが `:path: /`
    
- `https` に対応するのが `:scheme: https`
    

ここで重要なのは、HTTP/1 の文字列をそのままバイト列の中から探しているのではないことです。  
HPACK 側では、まず各表現を形式判定と static table によって復元しています。  
そのうえで、それが HTTP/1 の見慣れた意味単位とどうつながるかを見ています。

したがって、この問題の主役はあくまで HPACK 側の読解です。  
HTTP/1 側は、その意味を見失わないための足場です。

### 8-6. 3人で読む：GET / の対応

扱う例は次の二つです。

```text
GET / HTTP/1.1
```

`82 84 87`

**初学者**「HTTP/1 だと `GET /` は見慣れていますが、HPACK 側の `82 84 87` は最初は別物に見えます。」

**解説者**「その感覚は自然です。だからこそ、まず HPACK 側を 1表現ずつ読んでいきます。先頭の `82` はどうですか。」

**初学者**「Indexed Header Field です。index 2 を static table に引くと `:method: GET` です。」

**解説者**「次の `84` はどうですか。」

**初学者**「index 4 なので `:path: /` です。」

**実装者**「つまり、HTTP/1 では `GET /` とひと続きに見えていたものが、HPACK 側では二つのヘッダー表現に分かれているんですね。」

**解説者**「その通りです。最後の `87` は `:scheme: https` です。これで、HTTP/1 側で見慣れた意味単位が、raw bytes 側でどう構造化されているかが見えてきます。」

**初学者**「HTTP/1 の文字列を探すのではなく、HPACK 側を復元した結果として『ああ、これが GET と / に対応しているのか』と分かるわけですね。」

**解説者**「そうです。この順番が大事です。」

この対話で見てほしいのは、HTTP/1 を補助線にしつつも、**HPACK 側の読解手順を飛ばしていない** ことです。

### 8-7. `/` と `/index.html` の違いを対応で見る

次に、path だけが変わる例を見ます。

```text
GET / HTTP/1.1
```

```text
GET /index.html HTTP/1.1
```

これに対応する最小の HPACK 例として、次の二つを比べます。

`82 84 87`

`82 85 87`

この二つは、第6章でも比較しました。  
違うのは中央だけです。

- `84` → `:path: /`
    
- `85` → `:path: /index.html`
    

ここで重要なのは、HTTP/1 側の差分と HPACK 側の差分が対応していることです。  
HTTP/1 では path だけが変わっています。  
HPACK 側でも、method と scheme は同じで、path に対応する index だけが 4 から 5 に変わっています。

この比較によって、HPACK を単なる番号列ではなく、**意味の差分を持った表現の並び** として見られるようになります。  
そしてその差分は、HTTP/1 側で見慣れた path の差分とつながっています。

---

### 8-8. 模範問題：path が変わるとどこが変わるか

次の二組を見比べてください。

```text
GET / HTTP/1.1
```

`82 84 87`

```text
GET /index.html HTTP/1.1
```

`82 85 87`

このとき、

- HTTP/1 側ではどこが違うか
    
- HPACK 側ではどこが違うか
    
- その差分はどの意味単位に対応しているか
    

を答えてください。

---

### 8-9. 模範解答：path が変わるとどこが変わるか

HTTP/1 側では、method はどちらも `GET` です。  
違うのは path だけです。

- 一つ目は `/`
    
- 二つ目は `/index.html`
    

HPACK 側では、先頭の `82` と最後の `87` は共通です。  
違うのは中央だけです。

- `84` → `:path: /`
    
- `85` → `:path: /index.html`
    

したがって、この二つの差分は **path の違い** に対応しています。

ここで見てほしいのは、HPACK 側でも「どこが同じでどこが違うか」が、static table を引くことで明確になることです。  
HTTP/1 側の見慣れた差分が、HPACK 側では index 4 と 5 の差として現れています。

### 8-11. HTTP/1 を足場にするときの注意

ここまで見ると、HTTP/1 を足場にするのは便利です。  
しかし、ここには注意もあります。

最も大きな注意は、**HTTP/1 から逆算して答えだけを当てにいかないこと** です。  
たとえば `GET /index.html` を知っているからといって、HPACK 側を見ずに「たぶん path はこうだろう」と決めてしまうと、本書の目的から外れます。

本書が育てたいのは、raw bytes 側を読めることです。  
したがって、HTTP/1 はあくまで補助線です。  
使う順番は次の通りです。

- まず raw bytes 側を 1単位ずつ読む
    
- static table などを使って復元する
    
- その結果を HTTP/1 側の見慣れた意味と対応づける
    

この順番を守る限り、HTTP/1 の知識は有効な足場になります。  
逆に順番を逆にすると、「知っている文字列を当てるだけ」になってしまいます。

### 8-13. 誤読修正問題

ここで、もっともらしいが危ない説明を取り上げます。

> HTTP/1 を知っていれば、HTTP/2 や HPACK の内容もだいたい逆算できる。  
> だから生データを細かく読まなくてもよい。

この説明は、この章の入口としては魅力的に見えるかもしれません。  
しかし、本書の立場では危険です。

危険な理由は、**raw bytes 側の読解を飛ばしてしまう** からです。  
たしかに HTTP/1 の知識は役に立ちます。  
`GET` や `/index.html` や `Host` のような意味単位を知っていることは、HPACK を読んだあとで「何が表れているか」を理解する助けになります。

しかし、それは raw bytes 側を読んだあとで効く補助線です。  
最初から逆算だけで進むと、

- 形式判定を飛ばす
    
- static table を引く手順を飛ばす
    
- どこまで読めてどこから読めないかを曖昧にする
    

といった問題が起こります。

したがって、この説明は次のように言い直すべきです。

**HTTP/1 の知識は、HTTP/2 や HPACK の意味を理解する足場として役に立つ。  
しかし、答えを逆算するために使うのではなく、raw bytes 側を先に読んだうえで、その復元結果を対応づけるために使うべきである。**

この言い直しができれば、HTTP/1 を補助線として使いつつ、読解の主役を raw bytes 側に保てています。


この一覧で見てほしいのは、HTTP/1 の行を丸ごと対応づけることではありません。  
重要なのは、**見慣れた意味が raw bytes 側では複数の 1表現に分かれている** ことです。

### 8-15. この章のまとめ

この章では、HTTP/1 の見慣れた表現を足場にしながら、HPACK の raw bytes がどのような意味単位を表しているかを見ました。

ここでの中心は、HTTP/1 を答えの代わりにすることではありません。  
あくまで raw bytes 側を先に読み、その復元結果を HTTP/1 側の意味と対応づけることでした。

この章で一貫して守った読み方は次の通りです。

- まず HPACK 側を 1表現ずつ切る
    
- 先頭ビットを見て形式を判定する
    
- Indexed Header Field なら static table を引く
    
- 復元した結果を HTTP/1 側の見慣れた意味単位と対応づける
    
- 差分があれば、どの意味単位の差分かを確認する
    

この読み方によって、`82 84 87` は単なる番号列ではなく、`GET /` に対応する疑似ヘッダー群として見えてきます。  
また `82 84 87` と `82 85 87` の違いは、path の違いとして理解できます。

この章で大切だったのは、**対応づけは復元のあとに行う** ということです。  
HTTP/1 は補助線であり、主役は raw bytes 側の読解です。  
その順番を守ることで、HTTP/1 の既知知識を使いながらも、生データを読む力そのものを育てられます。

次章では、ここまで扱ってきた UTF-8、HTTP/2、HPACK を混ぜて、何を 1単位として切るのかを言葉にする練習に進みます。  
そこで狙うのは、題材が変わっても「最初にどこを見るか」「どの表を引くか」「何を 1単位とするか」を説明できるようになることです。
