## 第0章　本書全体の構成を先に見る

本書は、UTF-8、HTTP/2、HPACK という三つの題材を並べて扱います。  
一見すると、文字コード、通信プロトコル、ヘッダー圧縮という別々の話に見えるかもしれません。  
しかし本書では、それらを別分野の知識としてではなく、**「先頭を見て、表を引いて、1単位ずつ切りながら読む」ための教材** としてまとめて扱います。

この章の役割は、個別の仕様へ入る前に、**本書がどのような順番で、何を読者に身につけてもらおうとしているのか** を示すことにあります。  
最初に全体の地図を見ておくことで、このあと各章で行う作業が、単発の練習ではなく、同じ技法を少しずつ別の題材へ広げていく流れであることが見えやすくなります。

### 0-1. 本書は何を教える本か

本書は、UTF-8 や HTTP/2 や HPACK の仕様を網羅的に説明する本ではありません。  
また、特定言語でフル実装を作るための実装書でもありません。  
本書が中心に置くのは、**生のバイト列を前にしたときに、どこを見て、どの表を引き、どこで切ればよいかを判断する読解動作** です。

そのため本書では、知識を増やすことそのものよりも、次の五つの動作を繰り返し訓練します。

1. 先頭を見る
    
2. 表を引く
    
3. 1単位の長さや意味を決める
    
4. 数える、または復元する
    
5. 次へ進む

この五つは、題材が変わっても崩れません。  
UTF-8 では 1文字を切ります。  
HTTP/2 では 1フレームを切ります。  
HPACK では 1ヘッダー表現を切ります。  
読む対象は違っても、読者が行う基本動作は揃っています。  
本書の全体構成は、この共通性が自然に見えるように設計されています。

### 0-2. なぜ三つの題材を並べるのか

UTF-8 は文字コードです。  
HTTP/2 はバイナリフレームを持つ通信プロトコルです。  
HPACK は HTTP/2 で使われるヘッダー圧縮方式です。  
通常なら別々の章立てや別冊に分かれていても不思議ではありません。

それでも本書でこの三つを並べるのは、どれも **「先頭の情報を見て、表を引き、1単位を決める」** という同じ型で読めるからです。  
UTF-8 では先頭バイトを見て文字長候補を決めます。  
HTTP/2 では 9バイト共通ヘッダを見てフレーム境界を決めます。  
HPACK では先頭ビットを見て表現形式を決め、static table を引いて復元します。

つまり本書は、三つの仕様を横断的に暗記させたいのではありません。  
**ある題材で身につけた「読み方」を、別の題材にも持ち込めるようにすること** を狙っています。  
そのため、全体構成でも「題材ごとの個別知識」より「読解動作の反復」が中心になります。

### 0-3. 本書全体の流れ

本書は、大きく五つの段階で進みます。  
最初に、三つの題材に共通する読み方をまとめて確認します。  
そのうえで、UTF-8、HTTP/2、HPACK の順に、それぞれの最小例と典型例を深く読みます。  
そのあと、HTTP/1 を足場にしながら HTTP/2 と HPACK の対応を見ます。  
最後に、混合復習で「題材が変わっても同じ読み方が使える」ことを確認します。

この順番には理由があります。

まず導入部では、読者に「どこを見ればよいか分からない」という状態から抜けてもらいます。  
いきなり詳細な仕様へ入るのではなく、三つの題材に共通する骨格だけを先に示します。  
これによって、後続の章で題材が変わっても、毎回まったく新しい勉強をしている感じになりにくくなります。

次に UTF-8 を置くのは、もっとも小さな単位である「1文字」を扱うためです。  
先頭バイト、継続バイト、途中切れ、特例条件という流れは、**表を引いて読む** という本書の基本姿勢を最も短い形で練習するのに向いています。

その後に HTTP/2 を置くのは、UTF-8 より大きな構造を読む練習へ進むためです。  
ここでは 9バイト共通ヘッダを切り、Length と実データを照合し、フレーム境界を誤らずに進むことが中心になります。  
UTF-8 で学んだ「先頭を見る」「残り長さと照合する」という感覚が、そのまま別の形で現れます。

続く HPACK では、static table を引いて復元する作業が中心になります。  
ここでは、単に長さを決めるだけでなく、**変換表を使って意味を取り戻す** という作業が前面に出ます。  
ただしここでも、最初に見るのは先頭ビットであり、1表現ずつ切りながら次へ進む点は変わりません。

そして終盤では、HTTP/1 の見慣れたメッセージを足場にして、HTTP/2 の論理ヘッダーや HPACK の indexed header がどう対応しているかを見ます。  
これは、raw bytes の世界を別世界に見せないための章です。  
読者がすでに持っている知識を補助線として使いながら、生データの読解へ戻っていきます。

### 0-4. 各ユニットで何をするのか

本書の各ユニットは、だいたい同じ型でできています。  
最初に生データが置かれます。  
そのあとで、「どこを見るべきか」「どの表を使うべきか」という問いが出ます。  
つづいて模範問題、練習問題、誤読修正問題、典型例一覧が並びます。

この形をそろえているのは、毎回違う読解法を教えたいからではありません。  
むしろ逆で、**毎回同じ基本動作を繰り返したいから** です。  
問題のたびに、「まず何を見るのか」「何をまだ決めてはいけないのか」を確認します。  
そのうえで、正常例だけでなく、途中切れ、不整合、もっともらしい誤読も扱います。  
こうして、答えだけではなく、危ない近道を避ける感覚まで含めて身につけていきます。

### 0-5. なぜ少数の典型例を深く読むのか

本書は大量演習型の問題集ではありません。  
問題数を増やすことより、少数の典型例を比較しながら深く読むことを優先しています。  
これは、読者に求める負荷が高いからです。  
本書で必要なのは、単なる正答ではなく、**何を問うべきか、どう見るべきか、最終的に何が答えか** を同時に扱うことです。

また、本書では誤読修正をかなり重視しています。  
たとえば UTF-8 なら、継続バイトの形だけ見て受理してしまう。  
HTTP/2 なら、Type だけ見てフレームを読んだ気になる。  
HPACK なら、たまたま 1バイトで読めた例を一般化してしまう。  
こうした誤りは、単なる知識不足というより、**読み方の省略** によって起こります。  
そのため、本書では問題数を増やすより、もっともらしい誤読を丁寧に修正するほうを優先します。

### 0-6. 3人対話が担う役割

模範問題や誤読修正問題では、初学者、解説者、実装者の三人対話を使います。  
これは読み物としての演出ではなく、読解動作を見せるための構造です。

初学者は、「どこを見ればいいのか分からない」という自然な戸惑いを表に出します。  
解説者は、すぐに答えを言うのではなく、「いま何を見るべきか」「まだ何を決めてはいけないか」を整理します。  
実装者は、経験者がやりがちな危ない近道を持ち込みます。  
この三役があることで、単なる正答提示ではなく、**誤読の種類と修正の仕方** を立体的に見せられます。

### 0-7. 本書を読み進めるときの見方

本書を読むときは、最初からすべてを理解しようとしなくてかまいません。  
むしろ重要なのは、各章で毎回つぎの問いを持つことです。

このデータでは、最初にどこを見るのか。  
どの表を引くのか。  
何を 1単位として切るのか。  
どこで正常、不正、未確定を分けるのか。  
どんな近道が危ないのか。

この問いを繰り返していくと、三つの題材は別々の難しい話ではなく、同じ技法の応用だと見えてきます。  
本書の全体構成は、その感覚を自然に得られるように作られています。

### 0-8. この本を読み終えたときに目指す状態

本書を読み終えたとき、読者に持っていてほしいのは、「全部知っている」という感覚ではありません。  
そうではなく、16進ダンプを見たときに完全停止せず、まず先頭を見て、必要な表を思い出し、1単位ずつ切りながら進めるという反応です。  
不整合や途中切れを見落としにくくなり、雰囲気で読まず、判断根拠を言葉にできるようになることが、本書の到達点です。

このあと各章では、題材ごとの細部に入っていきます。  
ただし、やることは毎回同じです。  
先頭を見ます。  
表を引きます。  
1単位を決めます。  
数えるか、復元します。  
そして次へ進みます。  
本書全体は、その反復のために組まれています。
