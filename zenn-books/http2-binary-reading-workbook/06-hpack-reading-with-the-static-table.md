## 第6章　static table を引いて読む

第5章まで、HTTP/2 のフレームをどう切るかを見てきました。  
HTTP/2 では、先頭 9バイトの共通ヘッダを読み、Length に従って 1フレームずつ確定しました。

この章からは、題材を HPACK に移します。  
HPACK は HTTP/2 で使われるヘッダー圧縮方式です。  
ただし本書では、圧縮理論の一般論からは入りません。  
ここで中心に置くのは、**先頭を見て表現形式を決め、表を引いて 1ヘッダー表現ずつ復元する読み方** です。

UTF-8 では 1文字を切りました。  
HTTP/2 では 1フレームを切りました。  
HPACK では、**1ヘッダー表現** を切ります。

この章でまず扱うのは、もっとも単純な Indexed Header Field です。  
ここでは static table を引けば復元できる最小例だけを使います。  
つまりこの章の狙いは、HPACK 全体を説明することではなく、**「先頭ビットを見る」「index を取り出す」「static table を引く」** という基本動作を固めることにあります。

---

### 6-1. HPACK を読むとき、最初に何を見るか

UTF-8 では先頭バイトを見ました。  
HTTP/2 では先頭 9バイトを見ました。  
HPACK では、まず **先頭ビット** を見ます。

ここでいう「先頭ビット」とは、その 1バイトの最上位側がどうなっているか、ということです。  
HPACK では、先頭のビットパターンによって、その表現が何形式なのかが分かれます。

この章で扱う Indexed Header Field は、先頭ビットが 1 です。  
したがって、最初の 1バイトを見て「先頭ビットが 1 かどうか」を確認します。  
ここで Indexed Header Field だと分かれば、その後は **index を static table に引く** という流れに入れます。

重要なのは、最初から「これは `:method: GET` っぽい」などと意味を当てにいかないことです。  
HPACK でも、まず必要なのは **何として読むべきか** の判定です。  
つまり、いきなり値を当てるのではなく、**形式を決める** ことが先です。

---

### 6-2. static table とは何か

この章では、HPACK の static table を使います。  
static table は、よく使われるヘッダー名やヘッダーの組み合わせに番号が振られた表です。  
Indexed Header Field では、その番号を使ってヘッダーを表します。

本書では、ここで大事なのは表の内容を暗記することではありません。  
重要なのは、**index を見たら表に戻る** ことです。

たとえば `82` のような短い値を見たとき、慣れると「これは `:method: GET` だ」と覚えてしまうかもしれません。  
しかし本書では、その覚え方を主役にしません。  
必ず「これは Indexed Header Field か」「index はいくつか」「その index を static table に引くと何か」という順番で読みます。

つまり static table は、補助資料ではなく、ここでは読解の本体です。

---

### 6-3. 最小例 `82` を読む

最初の最小例を見ます。

`82`

この 1バイトを見たとき、まず先頭ビットを見ます。  
`82` は 2進数で見ると先頭ビットが 1 です。  
したがって、これは Indexed Header Field として読み始めます。

次に、この値に含まれている index を取り出します。  
この例では index は 2 です。  
そこで static table の 2番を引きます。  
すると `:method: GET` が得られます。

したがって、`82` は **Indexed Header Field 1個** であり、その内容は `:method: GET` です。

ここで重要なのは、「1バイトだから簡単」と思って終わらないことです。  
この 1バイトの中でやっていることは、次の通りです。

- 先頭ビットを見る
    
- Indexed Header Field だと決める
    
- index を取り出す
    
- static table を引く
    
- ヘッダーを復元する
    

つまり、この 1バイトの例の中に、HPACK の読み方の最小形が入っています。

---

### 6-4. `84` と `87` を読む

次に、似た最小例を二つ見ます。

`84`

`87`

どちらも先頭ビットは 1 です。  
したがって、どちらも Indexed Header Field として読みます。

`84` は index 4 です。  
static table の 4番を引くと `:path: /` が得られます。

`87` は index 7 です。  
static table の 7番を引くと `:scheme: https` が得られます。

ここで見てほしいのは、やっていることが `82` のときと変わらないことです。  
違うのは index の値だけです。  
そのため、この章では「何が復元されるか」を覚えるよりも、**どこが変わり、どこが変わらないか** を見ることを優先します。

- 先頭ビットを見る
    
- Indexed Header Field だと決める
    
- index を取り出す
    
- static table を引く
    

この流れは `82` でも `84` でも `87` でも同じです。

### 6-6. 模範解答：1個の indexed header

#### 問題1の考え方

`82` は先頭ビットが 1 なので Indexed Header Field です。  
index は 2 です。  
static table の 2番を引くと `:method: GET` です。  
したがって `82` は `:method: GET` を表します。

#### 問題2の考え方

`84` は先頭ビットが 1 なので Indexed Header Field です。  
index は 4 です。  
static table の 4番を引くと `:path: /` です。  
したがって `84` は `:path: /` を表します。

#### 問題3の考え方

`87` は先頭ビットが 1 なので Indexed Header Field です。  
index は 7 です。  
static table の 7番を引くと `:scheme: https` です。  
したがって `87` は `:scheme: https` を表します。

この三つは短い例ですが、どれも「形式判定 → index → static table」という順番で読めています。

### 6-7. 3人で読む：`82`

扱う例は次です。

`82`

**初学者**「1バイトしかないので、短い値をそのまま覚える問題に見えます。」

**解説者**「短いですが、覚える問題として読むのではなく、手順の練習として読みます。まず何を見ますか。」

**初学者**「先頭ビットです。1 なので Indexed Header Field です。」

**解説者**「その通りです。次は何をしますか。」

**初学者**「index を取り出します。この例では 2 です。」

**実装者**「ここで『82 だから GET』と丸暗記してしまうと、次の例に広がりませんね。」

**解説者**「そうです。本書では必ず static table に戻ります。では 2番は何ですか。」

**初学者**「`:method: GET` です。」

**解説者**「したがって、この 1バイトは `:method: GET` を表しています。」

**実装者**「短い例でも、形式判定と table lookup が主役なんですね。」

**解説者**「そうです。HPACK はまずそこから読みます。」

この対話で大切なのは、`82` を知識問題としてではなく、**HPACK の最小読解問題** として扱っていることです。

### 6-8. 複数の indexed header を並べて読む

次に、1個ではなく複数の Indexed Header Field が並んだ例を見ます。

`82 84 87`

この例を前にしたときも、やることは変わりません。  
全体を一気に「HTTP リクエストっぽいもの」として見ないことが大事です。  
まず先頭の `82` を 1ヘッダー表現として読みます。  
そのあと、残りの先頭 `84` をまた 1ヘッダー表現として読みます。  
最後に `87` を読みます。

順にたどると、

- `82` → `:method: GET`
    
- `84` → `:path: /`
    
- `87` → `:scheme: https`
    

となります。

ここで重要なのは、HPACK でも **1表現ずつ切って進む** ことです。  
UTF-8 で 1文字ずつ切ったのと同じです。  
HTTP/2 で 1フレームずつ切ったのと同じです。  
HPACK では 1ヘッダー表現ずつ切ります。

この章では、まだ literal を含む複雑な例には入りません。  
そのため、この段階では 1バイトごとに 1表現が切れています。  
ただし、ここで「HPACK は 1バイト 1ヘッダーだ」と一般化してはいけません。  
今はたまたま、最小の indexed header だけを扱っているからそう見えるだけです。  
この点は後で改めて確認します。

### 6-9. 似た例を比較する

次の二つを並べて見ます。

`82 84 87`

`82 85 87`

どちらも 3つの Indexed Header Field が並んでいるように見えます。  
最初の `82` と最後の `87` は同じです。  
違うのは中央だけです。

最初の列では中央が `84` です。  
これは static table の 4番で `:path: /` です。

二つ目の列では中央が `85` です。  
これは static table の 5番で `:path: /index.html` です。

つまり、この二つの違いは **path だけ** です。  
ここで見てほしいのは、値の見た目そのものよりも、**index が 4 から 5 に変わると static table 上でどこが変わるか** です。

この比較は、HPACK を暗号のようなものとして見るのではなく、**表を引いて差分を読むもの** として見る練習になります。

### 6-11. 模範解答：複数の indexed header

#### 問題1の考え方

`82 84 87` は、先頭から順に 3つの Indexed Header Field として読めます。

- `82` → index 2 → `:method: GET`
    
- `84` → index 4 → `:path: /`
    
- `87` → index 7 → `:scheme: https`
    

したがって、3個のヘッダー表現からなります。

#### 問題2の考え方

`82 85 87` も、先頭から順に 3つの Indexed Header Field として読めます。

- `82` → index 2 → `:method: GET`
    
- `85` → index 5 → `:path: /index.html`
    
- `87` → index 7 → `:scheme: https`
    

したがって、これも 3個のヘッダー表現からなります。

#### 二つの違い

違いは中央だけです。

- `84` は `:path: /`
    
- `85` は `:path: /index.html`
    

したがって、この二つの列は method と scheme は同じで、path だけが違います。

### 6-14. 誤読修正問題

ここで、もっともらしいが危ない説明を取り上げます。

> HPACK は、よく使う値が短い番号に置き換わっているだけなので、値を覚えれば読める。

この説明は、最小例だけを見ると正しそうに見えます。  
たしかに `82`、`84`、`87` のような短い例では、「覚えているかどうか」で答えられてしまうことがあります。

しかし、本書ではこの読み方を主役にしません。  
理由は二つあります。

一つ目は、**形式判定が抜ける** からです。  
HPACK では、まず先頭ビットを見て何形式かを決める必要があります。  
いきなり値を当てにいくと、この手順が消えます。

二つ目は、**表に戻る習慣が消える** からです。  
本書の狙いは、知っている値を思い出すことではなく、知らない値でも表を引いて読めるようになることです。

したがって、この説明は次のように言い直すべきです。

**HPACK では、まず先頭ビットを見て表現形式を決める。  
Indexed Header Field であれば index を取り出し、static table を引いてヘッダーを復元する。  
値を覚えることより、表に戻る読み方のほうが重要である。**

この言い直しができれば、HPACK を暗記問題ではなく、**変換表型の読解問題** として扱えています。

### 6-16. この章のまとめ

この章では、HPACK の最初の読み方として、Indexed Header Field と static table を扱いました。

HPACK を読むとき、最初にやることは値を推測することではありません。  
まず先頭ビットを見て、何形式として読むかを決めます。  
この章で扱った例では、先頭ビットが 1 なので Indexed Header Field として読めました。  
そのあとで index を取り出し、static table を引いてヘッダーを復元しました。

この章で一貫して守った読み方は次の通りです。

- 先頭ビットを見る
    
- Indexed Header Field だと決める
    
- index を取り出す
    
- static table を引く
    
- 1ヘッダー表現を復元する
    
- 残りがあれば次へ進む
    

これは、UTF-8 や HTTP/2 と同じ骨格です。  
違うのは、切る 1単位の名前だけです。

- UTF-8 では 1文字
    
- HTTP/2 では 1フレーム
    
- HPACK では 1ヘッダー表現
    

この章では、最小の indexed header だけを使ったので、1バイトごとに 1表現が切れました。  
ただし、HPACK 全体がいつもそうだと一般化してはいけません。  
次章では、static table だけで読めるものと、そうではないものを見分けるところへ進みます。  
そこでも大事なのは、見た目や記憶ではなく、**先頭を見て形式を決め、表を引いて、どこまでを 1表現とするかを判断すること** です。
