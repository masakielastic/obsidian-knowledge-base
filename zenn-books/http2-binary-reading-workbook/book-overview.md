# 表を引いて読むバイナリ実習

## 本書の概要

本書は、UTF-8 / HTTP/2 / HPACK を題材に、**生のバイト列を表を引きながら読むための実習書** である。

目的は、仕様の全面読解やフル実装の前に、読者が次の動作を自然に行えるようになることにある。

1. 先頭を見る  
2. 表を引く  
3. 1単位の長さや意味を決める  
4. 数える / 復元する  
5. 次へ進む  

本書は、抽象概念の説明を中心にした概念書ではない。  
また、問題数を大量にこなす演習集でもない。  
少数の典型データを深く読みながら、**どこを見て、どの表を引き、どこで切るか** を身体化することを狙う。

---

## 本書が扱う三つの題材

### UTF-8

UTF-8 は、文字コードの一般論としてではなく、**可変長データを読む最初の教材** として扱う。

読者は、先頭バイトから文字長候補を決め、必要に応じて特例表を引き、1文字ずつ切りながら文字数を数える。  
途中切れや特例条件も含めて読むことで、「形だけ見て通す」危うい読み方を修正する。

### HTTP/2

HTTP/2 は、仕様項目の列挙ではなく、**構造化バイナリを 9 バイト共通ヘッダから切る教材** として扱う。

読者は、Length / Type / Flags / Stream ID を共通ヘッダから読み取り、フレーム境界を決め、payload と照合しながら進む。  
空の SETTINGS、短い DATA、2 フレーム連結、Length 不整合のような典型例を通じて、フレームを読む動作を身につける。

### HPACK

HPACK は、圧縮方式の全体解説としてではなく、**先頭ビットと static table を使ってヘッダーを復元する教材** として扱う。

読者は、先頭ビットから表現形式を決め、index を取り出し、static table を引いて 1 ヘッダーずつ復元する。  
また、static だけで読めるものと、literal が必要なものを区別しながら、「たまたま読めた近道」を一般化しない態度を学ぶ。

---

## なぜこの三つを並べるのか

UTF-8 / HTTP/2 / HPACK は別分野の題材に見える。  
しかし、本書では次の共通した読み方で扱える。

- UTF-8 は **1文字** を切る
- HTTP/2 は **1フレーム** を切る
- HPACK は **1ヘッダー表現** を切る

いずれも、最初に先頭を見て、表を引いて、1単位を確定し、そこから次へ進む。  
この共通性を意識することで、読者は「分野ごとに別々の特殊技能」としてではなく、**バイナリ読解の共通動作** として学べるようになる。

## 本書の特徴

### 1. 生データから始める

本書では、説明より先に生データを置く。  
読者はまずバイト列を見て、何を手がかりにすべきかを考える。  
そのうえで、必要な表を引き、切り方と判断根拠を確認していく。

### 2. 少数の典型例を深く扱う

大量の例題を並べるのではなく、少数の典型例を丁寧に読む。  
これにより、正答の暗記ではなく、**アプローチ仮説の修正** に時間を使える。

### 3. 誤読を教材にする

本書では、誤答そのものよりも、**もっともらしい誤読** を重視する。

たとえば UTF-8 では、

- continuation byte だけ見て受理する
- E0 / ED / F0 / F4 の特例を落とす

といった「筋は良いが危ない読み方」を扱う。  
HTTP/2 や HPACK でも同様に、実装やデバッグで起こりやすい近道を教材化する。

### 4. HTTP/1 を足場にする

特に HTTP/2 / HPACK では、HTTP/1 の見慣れたメッセージを出発点にする。

たとえば、

```http
GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Accept: text/html
```

のような見慣れた形から、

- HTTP/2 の論理ヘッダー
- HEADERS フレーム
- HPACK の生データ

へ落としていく。  
この対応を繰り返すことで、「HTTP/1 的には同じ意味のメッセージが、HTTP/2 の raw bytes ではどう見えるか」を身体化する。

## 本書の学び方

本書の各ユニットは、原則として次の流れで構成される。

1. 生データを置く  
2. 最初の問いを置く  
3. 参照する表を明示する  
4. 模範問題を 3 人対話で読む  
5. 練習問題を解く  
6. 誤読修正問題で危ない近道を見直す  
7. 典型例一覧で見慣れる  
8. まとめで読み方を圧縮する  

ここで重視するのは、答えを知ることではなく、**どこから読み始めるかを迷わなくなること** である。

---

## 3 人対話の役割

模範問題と誤読修正問題では、次の 3 役による対話形式を採用する。

### 初学者
- 自然な疑問や第一印象を出す
- 素朴だがもっともらしい誤読を言語化する

### 解説者
- 何を先に見て、何をまだ決めてはいけないかを整理する
- どの表を引くべきか、どこで切るべきかを示す

### 実装者
- 筋は良いが危ない近道を持ち込む
- 実装やデバッグでやりがちな省略を可視化する

この 3 人対話により、正解だけでなく、**誤ったアプローチから正しいアプローチへの修正過程** を見せる。

---

## 本書が扱う範囲と扱わない範囲

### 本書が扱うこと
- UTF-8 の先頭バイト、途中切れ、特例条件
- HTTP/2 の共通ヘッダ、フレーム境界、Length 不整合
- HPACK の static table を使った最小限の読解
- HTTP/1 から HTTP/2 / HPACK への対応づけ
- 読み方の手順、誤読の修正、典型例への見慣れ

### 本書が深追いしないこと
- RFC 全体の網羅
- HPACK dynamic table や Huffman の完全解説
- HTTP/2 の詳細な状態遷移
- RST_STREAM / GOAWAY / Sans-IO の本格実装論
- HTTP/3 / QPACK の詳細
- 特定言語によるフル実装チュートリアル

本書は、上級書や実装書の代わりではない。  
それらへ進む前に、**生データを読むための基礎体力を作る本** である。

---

## 本書の到達点

本書を読み終えた読者は、次のような状態になることを目指す。

- UTF-8 のバイト列を前にして、どこで 1文字を切るか説明できる
- HTTP/2 の短いダンプを前にして、どこで 1フレーム切るか説明できる
- HPACK の短い例を前にして、static table を引いて復元できる
- 途中切れ、不整合、特例条件を見落とさずに読める
- 「なんとなく読めた」を「どの表を引いたか」に言い換えられる
- 危ない近道や一般化しすぎる癖を自覚して修正できる
- HTTP/2 や HPACK の本格的な仕様読解・実装学習へ進む足場を持てる

---

## 本書の位置づけ

本書は、概念書でも、実装書でも、仕様書の解説書でもない。  
位置づけとしては、**raw bytes を読む技法に特化した独立小冊子** である。

既存の HTTP/2 入門書が「分かる本」だとすれば、本書は「読めるようになる本」である。  
また、HTTP/2 上級書や実装論へ進む前の橋渡しとしても機能する。

---

## この本で特に大事にする一文

この本は、バイト列を  
**「意味不明な 16 進数列」として眺める対象ではなく、「先頭を見て、表を引いて、1単位ずつ切りながら読める構造」として理解させる本**  
である。

この方針を崩さないことが、本書全体の中心である。
