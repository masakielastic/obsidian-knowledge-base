# 本書の設計方針

- 本書は概念書ではなく、**少数の典型データを深く読む実習書** とする
    
- 各ユニットでは必ず
    
    - 生データ
        
    - 問題
        
    - 3人対話の模範解答
        
    - 練習問題
        
    - 誤読修正問題
        
    - 典型例一覧  
        を置く
        
- 読者に反復させる動作は一貫して次の5つ
    
    1. 先頭を見る
        
    2. 表を引く
        
    3. 1単位の長さや意味を決める
        
    4. 数える / 復元する
        
    5. 次へ進む

# 第1部 UTF-8

## 第2章 先頭バイトを見て文字数を数える

### ユニット 2-1 ASCII / 2バイト / 3バイトを切る

#### 目標

- 先頭バイトから 1文字の長さ候補を決める
    
- continuation byte を確認して 1文字ずつ進む
    

#### 模範問題

- `41`
    
- `C3 A9`
    
- `E3 81 82`
    

#### 読者の動作

- 先頭を見る
    
- 何バイト候補か決める
    
- 後続バイトの形を確認する
    
- 1文字切る
    

#### 3人対話で扱う誤アプローチ

- 「見た目で 2バイト文字っぽいから進む」
    
- 「ASCII でないなら multibyte とだけ考える」
    
- 「後続が 80-BF なら十分と考える」
    

#### 練習問題

- `24`
    
- `C2 A2`
    
- `E2 82 AC`
    
- `41 C3 A9`
    

#### 誤読修正問題

- 「UTF-8 は先頭バイトだけ見れば文字数を決められる」という説明の修正
    

#### 典型例一覧

- 1バイト
    
- 2バイト
    
- 3バイト
    
- ASCII + multibyte 混在
    

---

### ユニット 2-2 途中切れを見抜く

#### 目標

- 「候補としては正しい長さ」でも、データが足りない場合を区別する
    

#### 模範問題

- `41 E3 81`
    
- `C3`
    
- `E3 81`
    

#### 読者の動作

- 文字長候補を決める
    
- 実際の残りバイト数と照合する
    
- 正常 / 途中切れを区別する
    

#### 3人対話で扱う誤アプローチ

- 「先頭だけで 3バイト文字だから 1文字と数える」
    
- 「途中まで continuation byte があるので受理する」
    

#### 練習問題

- `41 C3`
    
- `E2 82`
    
- `61 E3 81 82 62`
    

#### 誤読修正問題

- 「文字長が分かった時点で受理できる」という説明の修正
    

#### 典型例一覧

- 正常
    
- 途中切れ
    
- 混在ケース
    

---

## 第3章 特例を見落とさない

### ユニット 3-1 E0 / ED の特例

#### 目標

- continuation byte 判定だけでは不十分であることを理解する
    
- E0 / ED の 2バイト目の範囲条件を確認する
    

#### 模範問題

- `E0 9F 80`
    
- `E0 A0 80`
    
- `ED 9F BF`
    
- `ED A0 80`
    

#### 読者の動作

- 先頭 `E0` / `ED` を見る
    
- 特例表を引く
    
- 2バイト目の範囲を確認する
    
- 受理 / 不受理を決める
    

#### 3人対話で扱う誤アプローチ

- 「3バイト形式で continuation だからOK」
    
- 「ビット演算で形だけ合っているからよい」
    

#### 練習問題

- `E0 A1 80`
    
- `ED 80 80`
    
- `ED 9F 80`
    

#### 誤読修正問題

- 「UTF-8 は continuation byte の形だけ見れば十分」という説明の修正
    

---

### ユニット 3-2 F0 / F4 の特例

#### 目標

- 4バイト形式でも特例表が必要であることを理解する
    

#### 模範問題

- `F0 8F 80 80`
    
- `F0 90 80 80`
    
- `F4 8F BF BF`
    
- `F4 90 80 80`
    

#### 読者の動作

- 先頭 `F0` / `F4` を見る
    
- 特例表を引く
    
- 2バイト目を確認する
    
- 受理 / 不受理を決める
    

#### 3人対話で扱う誤アプローチ

- 「4バイト形式は全部同じ条件で読める」
    
- 「長さが合っているから通す」
    

#### 練習問題

- `F0 91 80 80`
    
- `F4 8E 80 80`
    
- `F4 90 BF BF`
    

#### 誤読修正問題

- 「先頭の長さ表だけで UTF-8 を実装できる」という説明の修正

# 第2部 HTTP/2

## 第4章 9バイト共通ヘッダを切る

### ユニット 4-1 空の SETTINGS を読む

#### 目標

- HTTP/2 フレームをまず 9バイト共通ヘッダとして切る癖をつける
    

#### 模範問題

- `00 00 00 04 00 00 00 00 00`
    

#### 読者の動作

- 9バイトを切る
    
- Length / Type / Flags / Stream ID を読む
    
- フレーム種別を表で確認する
    

#### 3人対話で扱う誤アプローチ

- 「payload がないから特殊ケースと考える」
    
- 「Type だけ見て済ませる」
    

#### 練習問題

- 空の SETTINGS の類題
    
- Type が異なる短い例
    

#### 誤読修正問題

- 「フレームの意味は Type だけ見れば分かる」という説明の修正
    

---

### ユニット 4-2 1バイト DATA を読む

#### 目標

- 共通ヘッダと payload を分けて読む
    
- Length の意味を実データと照合する
    

#### 模範問題

- `00 00 01 00 00 00 00 00 01 FF`
    

#### 読者の動作

- 9バイト共通ヘッダを切る
    
- Length=1 を読む
    
- payload を1バイトだけ取る
    
- フレーム全体を確定する
    

#### 3人対話で扱う誤アプローチ

- 「長さ 1 だから、1バイトだけ見てよい」
    
- 「共通ヘッダを見ずに payload 側から読む」
    

#### 練習問題

- payload 2バイトの DATA
    
- stream id が異なる例
    

#### 誤読修正問題

- 「Length は payload のみ見ればよい」という説明の修正
    

---

## 第5章 フレーム境界を誤らない

### ユニット 5-1 2フレーム連結を切る

#### 目標

- 先頭フレームを確定してから次へ進む手順を身につける
    

#### 模範問題

- 空 SETTINGS + 1バイト DATA の連結例
    

#### 読者の動作

- 先頭 9バイトを読む
    
- Length に従って1フレームを確定する
    
- 次の位置に進む
    
- 2フレーム目も同様に読む
    

#### 3人対話で扱う誤アプローチ

- 「全体を見て何となく2つに分ける」
    
- 「Type が見えたところで切る」
    

#### 練習問題

- DATA + SETTINGS
    
- 連結順を入れ替えた例
    

#### 誤読修正問題

- 「Type の切れ目でフレーム境界を決められる」という説明の修正
    

---

### ユニット 5-2 Length 不整合を見抜く

#### 目標

- Length と実データ長の不整合を読める
    
- 途中切れ / 余剰データを区別する
    

#### 模範問題

- Length=2 なのに payload 1バイトしかない例
    
- 逆に余剰バイトがある例
    

#### 読者の動作

- 共通ヘッダを読む
    
- Length を確定する
    
- 実データ長と照合する
    
- 正常 / 不整合を答える
    

#### 3人対話で扱う誤アプローチ

- 「先頭の型が分かったから十分」
    
- 「手元のバイト列が全部 payload だと考える」
    

#### 練習問題

- 1バイト不足
    
- 2バイト余る
    
- 連結と不整合が混ざる例
    

#### 誤読修正問題

- 「HTTP/2 はヘッダを読めれば payload は後回しでよい」という説明の修正

# 第3部 HPACK

## 第6章 static table を引いて読む

### ユニット 6-1 1個の indexed header を読む

#### 目標

- 先頭ビットを見て Indexed Header Field と判断する
    
- index を static table に引く
    

#### 模範問題

- `82`
    
- `84`
    
- `87`
    

#### 読者の動作

- 先頭ビットを見る
    
- 表現形式を決める
    
- index を取り出す
    
- static table を引く
    

#### 3人対話で扱う誤アプローチ

- 「値を丸暗記する」
    
- 「1バイトだから1ヘッダーとだけ考える」
    

#### 練習問題

- `83`
    
- `86`
    
- `88`
    

#### 誤読修正問題

- 「HPACK は値表を覚えれば読める」という説明の修正
    

---

### ユニット 6-2 複数の indexed header を並べて読む

#### 目標

- 1ヘッダーずつ復元しながら進む
    
- 近い index の差を比較しながら表を引く
    

#### 模範問題

- `82 84 87`
    
- `82 85 87`
    

#### 読者の動作

- 各バイトについて表現形式を決める
    
- index を引く
    
- 復元結果を並べる
    
- 前問との差分を言語化する
    

#### 3人対話で扱う誤アプローチ

- 「前問に似ているから雰囲気で path を決める」
    
- 「今回たまたま1バイト1ヘッダーなので一般化する」
    

#### 練習問題

- `83 86`
    
- `82 86 87`
    
- 類似データの比較
    

#### 誤読修正問題

- 「HPACK は基本的に1バイト1ヘッダー」という説明の修正
    

---

## 第7章 static で読めるものと読めないものを分ける

### ユニット 7-1 static だけで読める例を見抜く

#### 目標

- static table だけで完結するケースを認識する
    

#### 模範問題

- static index のみで構成された短い例
    

#### 読者の動作

- 各表現が indexed かどうか見る
    
- literal なしで終わるか判定する
    

#### 3人対話で扱う誤アプローチ

- 「全部 literal と考えて文字列を探しに行く」
    
- 「HPACK はいつも複雑だと思い込む」
    

#### 練習問題

- static のみの別例
    
- static が続く別順序の例
    

---

### ユニット 7-2 static だけでは読めない例を見抜く

#### 目標

- literal が必要なケースを、深入りしすぎず識別する
    

#### 模範問題

- static の後に literal が来る最小例
    
- 既知の index だけでは復元できない例
    

#### 読者の動作

- 先頭ビットを見る
    
- indexed でないことを確認する
    
- 「ここから先は static table だけでは読めない」と判断する
    

#### 3人対話で扱う誤アプローチ

- 「見慣れた疑似ヘッダーに引きずられて全部 static とみなす」
    
- 「知らない値でも static table を無理に引き続ける」
    

#### 練習問題

- indexed / non-indexed の混在例
    
- static で読める範囲だけ答える問題
    

#### 誤読修正問題

- 「HPACK は全部 static table から復元できる」という説明の修正

# このアウトラインの核

この構成で一貫しているのは、章題ではなく **読者の動作** です。

- UTF-8 では 1文字を切る
    
- HTTP/2 では 1フレームを切る
    
- HPACK では 1ヘッダー表現を切る
    

そして毎回、

- どこを見るか
    
- どの表を引くか
    
- どこで切るか
    
- どこで誤りやすいか
    

を揃えています。
