## 第2章 Connection・Stream・Frame

**初学者**「HTTP/2 を勉強し始めると、Connection と Stream と Frame が何度も出てきますよね。  
でも、正直ぜんぶ“通信の単位”に見えてしまいます。何が違うんでしょうか」

**解説者**「そこは、HTTP/2 の最初の山場ですね。  
逆に言うと、この3つの区別がつくようになると、HTTP/2 はかなり見通しがよくなります。

まず結論から言うと、次のように考えると整理しやすいです。

- **Connection** は、クライアントとサーバーのあいだの“1本の接続”
    
- **Stream** は、その接続の中を流れる“個別のやりとり”
    
- **Frame** は、そのやりとりを構成する“最小の部品”
    

この3段階でできています」

**実装者**「ログを見ると、この違いがかなりはっきり出ます。  
たとえば HTTP/2 対応のクライアントやサーバーのログでは、接続全体に対する話と、特定の stream\_id に対する話が混ざって出てきます。そこを分けて読めるようになると、だいぶ楽になります」

**初学者**「“接続”の中に“やりとり”が複数あって、そのやりとりが“部品”でできている、という感じですか？」

**解説者**「はい。その理解でかなり近いです。  
まずは **Connection** から見ましょう。

HTTP/1.1 でも TCP 接続はありました。HTTP/2 でも、クライアントとサーバーはまず1本の接続を張ります。これが Connection です。  
大事なのは、HTTP/2 ではこの**1本の接続の上で、複数のリクエストとレスポンスを同時に扱える**ことです」

**初学者**「HTTP/1.1 だと、リクエストごとに順番待ちしやすかったけど、HTTP/2 では1本の接続をうまく共有できる、という話ですよね」

**解説者**「そうです。そこで必要になるのが **Stream** です。

HTTP/2 では、1つ1つのリクエスト/レスポンスの組を、接続の中の“独立した通路”のように扱います。これが Stream です。  
たとえば、HTML を取るリクエストが1つ、CSS を取るリクエストが1つ、JavaScript を取るリクエストが1つあれば、それぞれ別の Stream として進みます」

**初学者**「なるほど。じゃあ Stream は、HTTP/1.1 でいう“1回のリクエストとレスポンス”に近いんですね」

**解説者**「入門段階では、その理解で大丈夫です。  
厳密には Stream は“HTTP/2 接続の中に作られる双方向の通信の流れ”ですが、まずは**1つの HTTP メッセージ交換の単位**と思っておくと整理しやすいです」

**実装者**「ログだと、ここに stream\_id が付きます。  
たとえば stream 1, stream 3, stream 5 みたいに番号が出てきたら、“今どのやりとりの話をしているのか”を示していることが多いです」

**初学者**「じゃあ Frame は何ですか？ Stream とどう違うんでしょう」

**解説者**「Frame は、HTTP/2 で実際に送られる**細かい部品**です。  
HTTP/2 では、リクエストやレスポンスを、ひとかたまりの巨大なテキストとして送るのではなく、用途ごとの小さな部品に分けて送ります。その部品が Frame です。

たとえば、ヘッダーを運ぶ **HEADERS フレーム**、本文を運ぶ **DATA フレーム** があります」

**初学者**「つまり、1つの Stream の中に HEADERS フレームや DATA フレームが並ぶんですか？」

**解説者**「その通りです。  
たとえば、ある1本の Stream でリクエストを送るとき、最初に HEADERS フレームでメソッドやパスなどを送り、そのあと必要なら DATA フレームで本文を送ります。  
レスポンス側でも、まず HEADERS フレームでステータスやレスポンスヘッダーが来て、そのあと DATA フレームで本文が来ます」

**初学者**「だんだん見えてきました。  
Connection は大きい箱、Stream はその中の1件ごとの通信、Frame はさらにその中の部品、という感じですね」

**解説者**「はい。その階層感が大切です。  
混乱しやすいのは、3つとも“通信に関係する単位”なので、同じレベルのものに見えてしまうことです。でも実際にはレベルが違います。

- Connection は全体
    
- Stream は個別の流れ
    
- Frame は部品
    

この順番で大きさが違うんです」

**実装者**「しかも、HTTP/2 らしさが出るのは、**複数の Stream の Frame が交互に流れる**ところです。  
たとえば stream 1 の HEADERS が来たあと、すぐに stream 3 の HEADERS が来て、そのあと stream 1 の DATA、さらに stream 5 の HEADERS……というふうに、接続全体では入り混じって流れます」

**初学者**「それが“多重化”ですか？」

**解説者**「そうです。HTTP/2 の重要な特徴の1つです。  
HTTP/1.1 では、1本の接続の上で複数のやりとりをきれいに重ねるのが難しかった。HTTP/2 では Stream という単位を導入し、その中身を Frame に分けることで、複数の通信を1本の接続の上で整理して扱えるようにしました」

**初学者**「でも、フレームが入り混じると、受け取る側は混乱しませんか？」

**解説者**「そこで Frame に Stream 番号が付いていることが効いてきます。  
受け取った側は、“このフレームは stream 3 のものだな”と見分けて、同じ Stream の部品を組み立てていけばよいわけです」

**実装者**「ログを見ると、その感覚がわかります。  
『HEADERS frame <length=... stream\_id=3>』のような表示があれば、“これは stream 3 に属する HEADERS フレームだな”と読めます。  
つまりログでは、**Frame の種類**と**どの Stream に属しているか**を追うことが基本になります」

**初学者**「なるほど。じゃあ Connection には番号はないんですか？」

**解説者**「通常、ログでは接続ごとにまとまりとして扱われることが多いですね。  
大事なのは、**Stream の番号は接続の中で意味を持つ**ということです。stream 3 という番号は、その接続の中の stream 3 です。Connection が変われば、また別の話になります」

**初学者**「少し整理させてください。  
HTTP/2 では、まずクライアントとサーバーが Connection を作る。  
その中で、リクエストごとに Stream が作られる。  
そして各 Stream の中身は HEADERS や DATA などの Frame に分かれて送られる。  
さらに、複数の Stream の Frame は交互に流れてよい。  
こんな理解でいいですか？」

**解説者**「とてもいい整理です。第2章の中心は、まさにそこです」

**実装者**「実際の観察でも、その順番で見ると混乱しにくいです。  
まず“いま1本の接続の中の話をしている”と意識する。  
次に“これは何番の Stream の話か”を見る。  
最後に“どの種類の Frame が流れているか”を見る。  
この3段階で読むと、ログが急に読みやすくなります」

**初学者**「HTTP/2 を難しく感じていた理由が少しわかりました。  
全部を同じ“通信”として見ていたから、頭の中でごちゃごちゃになっていたんですね」

**解説者**「そうです。HTTP/2 では新しい用語が増えたというより、**通信を階層に分けて整理している**と考えるほうが自然です。

最後に一言でまとめましょう。

- **Connection**：クライアントとサーバーのあいだの1本の接続
    
- **Stream**：その接続の中で進む1件ごとのやりとり
    
- **Frame**：そのやりとりを構成する最小の部品
    

この3つを混同しないことが、HTTP/2 理解の土台です」

**初学者**「次からログを見るときは、まず“これは接続全体の話か、個別の Stream の話か、1個の Frame の話か”を意識してみます」

**実装者**「それだけでかなり変わります。  
HTTP/2 は“魔法の高速化”ではなく、接続の中を部品化して整理した仕組みなんだ、と見えてきますから」

**解説者**「次の章では、その Frame の中でも特によく出てくるもの、たとえば HEADERS や DATA などを順に見ていきましょう。そこまで行くと、通信ログの景色がさらに具体的になります」
