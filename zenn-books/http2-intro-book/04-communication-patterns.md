## 第4章 通信パターン（簡略記号）

**初学者**「第3章でフレームの種類はわかってきました。でも、実際の通信ってどういう順番で流れるんですか？ログを見るとバラバラに見えてしまって……」

**解説者**「いいところに気づきましたね。  
HTTP/2 は“部品（Frame）”でできていますが、実際の通信ではそれらが一定の“並び方”をします。これをこの章では“通信パターン”として整理します」

**実装者**「ログが読みにくい原因の多くは、“順番の感覚がない”ことです。  
フレームの種類はわかっていても、“次に何が来るか”の予測ができないと、ただの羅列に見えてしまいます」

---

### シンプルな記号で考える

**解説者**「まず、複雑さを減らすために簡略記号を導入します。

- H = HEADERS
    
- D = DATA
    

まずはこの2つだけで考えましょう」

**初学者**「かなりシンプルになりますね」

**解説者**「入門ではこれで十分です。  
HTTP メッセージの本体は、この2つでほぼ説明できます」

---

### 基本パターン（GETリクエスト）

**解説者**「まず最も基本的なパターンです。

GETリクエスト（本文なし）の場合：

```
リクエスト: H  
レスポンス: H → D
```

これが最初に覚えるべき形です」

**初学者**「リクエストは HEADERS だけで、レスポンスは HEADERS のあとに DATA が来る、ですね」

**実装者**「実際のログでも、この形は頻出です。  
“HEADERS（リクエスト） → HEADERS（レスポンス） → DATA（本文）”という流れが見えたら、だいたいこれです」

---

### POSTなど（リクエストに本文あり）

**解説者**「次に、リクエストにも本文がある場合です。

```
リクエスト: H → D  
レスポンス: H → D
```

たとえば POST や PUT の場合ですね」

**初学者**「リクエスト側にも DATA が増えるんですね」

**解説者**「はい。  
このとき重要なのは、“HEADERS と DATA は1回ずつとは限らない”ということです」

---

### DATA は分割される

**解説者**「HTTP/2 では、DATA は複数に分かれることがあります。

```
レスポンス: H → D → D → D
```

このように、DATA が連続することがあります」

**初学者**「ああ、1回で全部来るとは限らないんですね」

**実装者**「大きなレスポンスだと普通に起きます。  
ログを見ると DATA が何回も続くので、“まだ続きがあるな”と判断できます」

---

### ストリーム単位で見る

**解説者**「ここで第2章の話が効いてきます。  
このパターンは“1つの Stream の中の話”です」

**初学者**「あ、Connection 全体じゃないんですね」

**解説者**「そうです。  
1つの Stream だけを切り出すと、今見たようなシンプルな並びになります」

---

### 実際は複数ストリームが混ざる

**実装者**「ただし現実のログでは、こうは見えません」

**初学者**「えっ？」

**実装者**「実際にはこうなります：

```
stream1: H  
stream3: H  
stream1: H  
stream3: D  
stream1: D
```

つまり、複数の Stream のフレームが交互に流れます」

**初学者**「これは一気に難しくなりますね……」

---

### “ストリームごとに分解する”のがコツ

**解説者**「ここが重要なポイントです。  
ログを読むときは、“Connection 全体”で見るのではなく、“Stream ごとに分解して考える”必要があります」

**初学者**「つまり、stream1 だけを抜き出して並べ直す、ということですか？」

**解説者**「そうです。  
たとえばさっきの例なら：

```
stream1: H → H → D  
stream3: H → D
```

このように整理できます」

**実装者**「これができると、一気に読みやすくなります。  
ログは混ざって見えるけど、実際の通信はちゃんと整理されている、という感覚がつかめます」

---

### よくあるパターンまとめ

**解説者**「ここまでをまとめると、入門段階で覚えるべきパターンは次の通りです」

#### ■ GET（本文なし）

```
H / H → D
```

#### ■ POSTなど（本文あり）

```
H → D / H → D
```

#### ■ DATA分割

```
H → D → D → D
```

---

### 制御フレームは“ノイズではない”

**初学者**「でも実際のログだと、SETTINGS とか PING とかも出てきますよね」

**解説者**「いい観点です。  
それらは“通信パターンの本体”ではありませんが、“周辺の制御”として混ざります」

**実装者**「ログはこんな感じになります：

```
SETTINGS  
stream1: H  
WINDOW\_UPDATE  
stream1: H  
stream1: D  
PING
```

こういう混ざり方です」

**初学者**「これは邪魔に見えてしまいますね」

---

### 本体と制御を分けて考える

**解説者**「そこで考え方を分けます。

- HEADERS / DATA → 本体（HTTPメッセージ）
    
- その他 → 制御
    

まずは本体だけを追うようにすると、理解が安定します」

**初学者**「なるほど、“見るべきものを絞る”ということですね」

---

### この章の核心

**解説者**「この章で一番大事なのは、次の3点です。

1. HTTP/2 の通信はパターンで理解できる
    
2. 基本は H と D の並びで説明できる
    
3. ログは“Streamごとに分解して読む”」
    

**実装者**「これができると、ログが“意味のある流れ”に見えてきます」

---

### まとめ

**解説者**「最後に一言でまとめます。

HTTP/2 の通信は複雑に見えますが、実際には：

- 各 Stream はシンプルなパターン（H と D）で構成される
    
- それが複数同時に流れることで複雑に見える
    

という構造です」

**初学者**「“複雑なのは見た目だけで、中身はシンプル”という感じですね」

**実装者**「まさにそれです。  
HTTP/2 は難しいというより、“整理して見る必要があるプロトコル”です」

---

**解説者**「次の章では、いよいよ実際のログ（curl / nghttpd）を使って、ここで学んだ読み方を実践していきます」
