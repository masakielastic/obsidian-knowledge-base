## 第3章　よく出るフレーム

**初学者**「第2章で Connection と Stream と Frame の違いはだいぶわかってきました。  
でも、Frame も種類がいろいろありますよね。全部覚えないといけないんでしょうか」

**解説者**「入門の段階では、まず“よく出るもの”を押さえれば十分です。  
HTTP/2 にはいくつもフレームがありますが、最初に重要なのは次のようなものです。

- **HEADERS**
    
- **DATA**
    
- **SETTINGS**
    
- **WINDOW\_UPDATE**
    
- **PING**
    
- **RST\_STREAM**
    
- **GOAWAY**
    

この章では、とくに最初の2つ、HEADERS と DATA を中心に見ていきます。ここがいちばん基本だからです」

**実装者**「実際に curl や nghttpd のログを見ても、まず目につくのは HEADERS と DATA です。  
その次に、接続の開始時に SETTINGS、終了や異常時に GOAWAY や RST\_STREAM が出てくる、という感じですね」

**初学者**「やっぱり HEADERS と DATA が主役なんですね」

**解説者**「そうです。  
HTTP 通信をかなり乱暴に言うと、“メタ情報”と“本文”があります。  
HTTP/2 では、その役割を分けて運ぶ代表的なフレームが HEADERS と DATA です」

---

### HEADERS フレーム

**初学者**「HEADERS は名前の通り、ヘッダーを送るフレームですよね」

**解説者**「はい。  
リクエストなら、たとえばメソッド、パス、スキーム、ホストのような情報。  
レスポンスなら、ステータスコードや各種レスポンスヘッダー。  
そういった“この通信が何者か”を示す情報を運ぶのが HEADERS フレームです」

**初学者**「HTTP/1.1 だと、`GET /index.html HTTP/1.1` みたいな開始行と、`Host:` や `User-Agent:` がテキストで並んでいましたよね」

**解説者**「HTTP/2 では、そうした情報がテキストの行としてそのまま流れるのではなく、HEADERS フレームという部品に入って送られます。  
見た目は変わりますが、役割としては“リクエストやレスポンスの条件を伝える部分”です」

**実装者**「ログでは、ある stream に対してまず HEADERS が出ることが多いです。  
つまり“この stream で何を要求しているのか”あるいは“この stream にどう応答するのか”を最初に示すわけですね」

**初学者**「じゃあ、1つの stream は HEADERS から始まることが多いんですか？」

**解説者**「はい、そう考えてよいです。  
少なくとも、HTTP メッセージとして意味のあるやりとりを始めるには、まず HEADERS が必要です。  
何の要求かも何の応答かもわからないまま DATA だけ送られてきても困りますからね」

---

### DATA フレーム

**初学者**「DATA は本文ですね」

**解説者**「その通りです。  
リクエスト本文やレスポンス本文の実データを運ぶのが DATA フレームです。  
たとえば POST の本文、あるいは HTML や JSON の中身などがここに入ります」

**初学者**「じゃあ、画像でも HTML でも JSON でも、とにかく本文なら DATA なんですね」

**解説者**「はい。  
重要なのは、HTTP/2 では本文が“1つの大きな塊”とは限らないことです。  
本文は複数の DATA フレームに分かれて流れることがあります」

**初学者**「分割されるんですね」

**実装者**「ログでもよくあります。  
レスポンス本文が少し大きければ、DATA フレームが何個か続いて出ることがあります。  
だから DATA を1回見たら本文全部、とは限りません」

**解説者**「ここで大事なのは、“1つの HTTP メッセージ = 1フレーム”ではない、ということです。  
たとえばレスポンスは、

- HEADERS でステータスやレスポンスヘッダーを送る
    
- DATA を1個以上使って本文を送る
    

という形になります」

**初学者**「なるほど。HTTP/2 は細かい部品に分けて送る、という話がここでも出てくるんですね」

---

### HEADERS と DATA の関係

**解説者**「入門者がまず覚えるべき通信の基本パターンは、とてもシンプルです。

- 最初に **HEADERS**
    
- 必要ならその後に **DATA**
    

これが基本です」

**初学者**「リクエストもレスポンスも、その考え方で見ていいんですか？」

**解説者**「はい。  
もちろん細かい例外や追加の事情はありますが、最初はそれで十分です。  
たとえば GET リクエストなら本文がないことが多いので、リクエスト側は HEADERS だけで終わることもあります。  
一方でレスポンス側は、HEADERS のあとに DATA が続くことがよくあります」

**実装者**「ログを読むときも、まずはこの並びを探すとわかりやすいです。  
“この stream では HEADERS が来た、そのあと DATA が来た。じゃあレスポンス本文が流れているんだな”と見られるようになります」

---

### SETTINGS フレーム

**初学者**「HEADERS と DATA 以外も少し気になります」

**解説者**「では、接続全体に関わる代表例を見ましょう。  
まず **SETTINGS** です。これは個別のリクエストやレスポンスではなく、接続に関する設定をやりとりするフレームです」

**初学者**「stream ではなく connection の話なんですね」

**解説者**「はい。  
HTTP/2 では通信を始めるときに、“この接続ではこういう設定でやりましょう”という情報を交換します。  
そのために SETTINGS フレームが使われます」

**実装者**「ログのかなり早い段階で出てくることが多いですね。  
だから SETTINGS を見たら、“個別のページ取得というより、接続の準備の話をしているな”と考えるとよいです」

---

### WINDOW\_UPDATE フレーム

**初学者**「名前だけ見ると難しそうです」

**解説者**「これはフロー制御に関係するフレームです。  
詳しい仕組みは第6章で触れますが、ここでは“どれだけ送ってよいかの調整に関わるもの”くらいで十分です」

**初学者**「入門では深入りしなくていいんですね」

**解説者**「はい。  
大切なのは、“本文そのものを運ぶ DATA”とは別に、“通信量の調整に使うフレーム”もある、と知ることです」

---

### PING フレーム

**解説者**「**PING** は、相手との接続確認や往復時間の確認に使われるフレームです」

**初学者**「これは HTTP の本文とは関係ないですね」

**解説者**「そうです。  
HTTP メッセージそのものではなく、接続がちゃんと生きているかを確かめるための補助的なフレームです」

---

### RST\_STREAM フレーム

**解説者**「**RST\_STREAM** は、ある stream を途中で打ち切るためのフレームです」

**初学者**「接続全体を閉じるわけではないんですか？」

**解説者**「そこが重要です。  
HTTP/2 では、接続全体を維持したまま、特定の stream だけを終了させることができます。  
つまり、“この1件のやりとりだけやめる”という制御ができるわけです」

**実装者**「ログでこれが出たら、“この stream は正常完了ではなく途中終了したんだな”と読めます」

---

### GOAWAY フレーム

**解説者**「**GOAWAY** は、接続全体を閉じる方向に進むときのフレームです。  
これは stream 単位ではなく、connection 単位の終了に近い話です」

**初学者**「RST\_STREAM は個別、GOAWAY は全体、という区別ですね」

**解説者**「その通りです。  
この違いは第2章の Connection と Stream の区別がそのまま生きるところです」

---

### まずは「何を運ぶフレームか」で見る

**初学者**「フレームの種類が増えると、また混乱しそうです」

**解説者**「そういうときは、役割で大まかに分けると整理しやすいです。

- **HTTP メッセージの内容を運ぶ**
    
    - HEADERS
        
    - DATA
        
- **接続や制御のために使う**
    
    - SETTINGS
        
    - WINDOW\_UPDATE
        
    - PING
        
    - RST\_STREAM
        
    - GOAWAY
        

最初はこの二分法で十分です」

**実装者**「ログを読むときにも役立ちます。  
HEADERS や DATA が見えたら“実際のリクエスト/レスポンスの話”。  
SETTINGS や PING が見えたら“接続の制御や状態確認の話”。  
そう分けるだけで追いやすくなります」

---

### この章のまとめ

**解説者**「最後にまとめましょう。

HTTP/2 のフレームはたくさんありますが、入門の第一歩としては、まず次を押さえることが重要です。

- **HEADERS**：ヘッダーを運ぶ
    
- **DATA**：本文を運ぶ
    
- **SETTINGS**：接続設定をやりとりする
    
- **WINDOW\_UPDATE**：送受信量の調整に関わる
    
- **PING**：接続確認に使う
    
- **RST\_STREAM**：特定の stream を打ち切る
    
- **GOAWAY**：接続全体の終了に向かう
    

そして、とくに重要なのは **HEADERS と DATA** です。  
HTTP メッセージそのものを観察したいとき、まず見るべきはこの2つだからです」

**初学者**「最初から全部を同じ重さで覚えなくていいのは助かります。  
まずは HEADERS と DATA を中心に見て、ほかは“制御系のフレームもある”くらいで押さえておけばよさそうですね」

**実装者**「それで十分です。  
実際、ログの読み始めでは“この frame は本文なのか、設定なのか、終了処理なのか”を区別できるだけでかなり前に進めます」

解説者「次の章では、そのフレームたちが実際にどう並ぶのかを、通信パターンとして見ていきましょう。  
フレームの名前を知るだけでなく、“どの順番で出やすいのか”がわかると、ログがさらに読みやすくなります」
