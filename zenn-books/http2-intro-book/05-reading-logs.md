## 第5章 ログの読み方

**初学者**「ここまでで Connection・Stream・Frame や、HEADERS と DATA の流れは少しわかってきました。  
でも実際のログを見ると、急に難しく感じます。文字が多いし、何から見ればいいのかわかりません」

**解説者**「それは自然な反応です。  
HTTP/2 のログは、慣れないうちは“全部を同時に読もうとしてしまう”ので難しく見えます。  
でも実際には、見る順番を決めればかなり整理できます」

**実装者**「ログを読むときのコツは、“全部読む”ことではなく“まずどこを見るか決める”ことです。  
HTTP/2 のログは観察の対象を絞ると、一気に読みやすくなります」

---

### まずは3段階で見る

**解説者**「この本では、ログを次の3段階で読むことをおすすめします。

1. **接続全体の話か**
    
2. **どの Stream の話か**
    
3. **どの Frame が流れたか**
    

この順番です」

**初学者**「第2章でやった Connection・Stream・Frame の区別が、そのまま使えるんですね」

**解説者**「そうです。  
ログの読み方は、新しい知識というより、これまで学んだ区別を観察に使う作業です」

---

### 最初に見るべきもの

**実装者**「最初から細部を全部読む必要はありません。  
まず見るべきなのは、次の3つです。

- **stream\_id**
    
- **frame の種類**
    
- **流れの順番**
    

これだけでもかなり読めます」

**初学者**「たとえば `stream_id=1` みたいなものですね」

**実装者**「そうです。  
それが出てきたら、“これは stream 1 の話だな”とわかります。  
さらに `HEADERS` や `DATA` が見えれば、“その stream の中で何が起きているか”が見えてきます」

---

### `curl -v --http2` のログは何を見るか

**解説者**「まずはクライアント側、つまり `curl` のログを考えましょう。  
`curl -v --http2` のようにして HTTP/2 通信を見ると、接続の開始、リクエスト送信、レスポンス受信の流れが出てきます」

**初学者**「`curl` のログは人間向けに見やすくなっている印象があります」

**実装者**「そうですね。  
ただし、“HTTP/2 の内部が全部見える”というより、“クライアントがどう通信したかを追いやすい”ログです。  
入門ではとても使いやすいです」

**解説者**「`curl` のログでは、まず“HTTP/2 が使われたか”を確認します。  
そのうえで、どのリクエストを送り、どういうレスポンスを受けたかを見ていきます。  
最初の観察としては、“HTTP/1.1 の延長で見える部分”と“HTTP/2 特有の構造”を分けて考えるとよいです」

---

### `nghttpd` のログは何を見るか

**初学者**「サーバー側の `nghttpd` はどう違うんでしょう」

**実装者**「`nghttpd` のログは、HTTP/2 のフレームや stream の動きがより直接見えやすいです。  
なので、“HTTP/2 の構造を観察する教材”として向いています」

**解説者**「この本の目的に合っていますね。  
HTTP/2 を“仕様書の文章”ではなく“観察できる構造”として学ぶには、`nghttpd` のようなログはとても有効です」

**初学者**「じゃあ `curl` はクライアント視点、`nghttpd` は構造観察向け、という感じですか？」

**実装者**「だいたいそう考えてよいです。  
もちろん厳密には用途がもっとありますが、入門としてはその整理で十分です」

---

### ログは「1行ずつ」ではなく「まとまり」で見る

**解説者**「ここで重要なのは、ログを1行ずつ孤立して読むのではなく、“まとまり”で見ることです」

**初学者**「まとまり、ですか？」

**解説者**「はい。  
たとえば、ある stream について

- HEADERS が来た
    
- そのあと DATA が来た
    
- さらに DATA が来た
    

という流れが見えたら、それを1つの通信のまとまりとして見るわけです」

**実装者**「逆に、1行だけ見て“これは何だろう”と考えると迷いやすいです。  
HTTP/2 のログは、**順番を含めて意味を持つ**ことが多いので、“前後関係”が大切です」

---

### まずは HEADERS と DATA だけ追えばよい

**初学者**「でもログには SETTINGS や WINDOW\_UPDATE も出てきますよね。全部追うべきですか？」

**解説者**「入門では、まず **HEADERS と DATA だけ追えば十分** です」

**初学者**「かなり絞っていいんですね」

**解説者**「はい。  
第3章と第4章で見たように、HTTP メッセージ本体を理解するには、まず HEADERS と DATA が中心です。  
SETTINGS や WINDOW\_UPDATE は大事ですが、最初から全部同じ重さで追うと混乱しやすいです」

**実装者**「実際の観察でも、

- どの stream に
    
- どんな HEADERS が来て
    
- DATA が何回流れたか
    

をまず見るだけで、かなり状況がつかめます」

---

### Streamごとに読む

**解説者**「ログを読むときの最大のコツは、やはり **Stream ごとに読む** ことです」

**初学者**「Connection 全体でそのまま追うのではなく、ですね」

**解説者**「そうです。  
HTTP/2 では複数の stream が同時に進むので、Connection 全体の表示はどうしても入り混じります。  
でも各 stream の中身は、意外と単純です」

**実装者**「たとえば stream 1 と stream 3 が交互に出てきても、

- stream 1 だけ抜き出す
    
- stream 3 だけ抜き出す
    

と考えれば、それぞれは H → D のような基本形に近づきます」

**初学者**「ログを読むというより、“分解して再構成する”感じですね」

**解説者**「まさにそうです。  
HTTP/2 のログ読解は、分解の作業でもあります」

---

### 接続レベルのログとStreamレベルのログを分ける

**解説者**「もう1つ大切なのは、接続全体の話と、個別 stream の話を分けることです」

**初学者**「SETTINGS や PING は接続寄りの話でしたね」

**解説者**「そうです。  
たとえば SETTINGS を見ているのに、“これはどのリクエストのレスポンスだろう”と考えると混乱します。  
それは個別の HTTP メッセージではなく、接続全体の設定の話だからです」

**実装者**「逆に HEADERS や DATA は、個別の stream に属していることが多い。  
だから“これは接続の話か、stream の話か”を先に分けるだけでも、かなり読みやすくなります」

---

### 読み方の手順を固定する

**解説者**「ここまでをふまえると、ログを読む手順は次のように固定できます。

まず、

1. **HTTP/2 接続が成立しているか**
    
2. **どの stream が出てきているか**
    
3. **HEADERS / DATA の並びはどうなっているか**
    
4. **その途中に制御フレームが挟まっているか**
    

この順で見るのがおすすめです」

**初学者**「順番があるだけで、かなり安心感があります」

**実装者**「慣れてくると、ログの見え方が変わります。  
最初は文字の塊だったものが、“接続の開始”“stream 1 の開始”“レスポンス本文の受信”のように、意味のあるイベントに見えてきます」

---

### `curl` と `nghttpd` をどう使い分けるか

**初学者**「学習するときは、`curl` と `nghttpd` のどちらを中心に見るといいですか？」

**解説者**「最初は `curl` で通信を動かし、`nghttpd` で構造を観察する、という組み合わせがわかりやすいです」

**実装者**「クライアントとしてリクエストを送る体験は `curl` がやりやすいですし、  
サーバー側で stream や frame を観察するなら `nghttpd` が便利です。  
入門では、“自分で通信を起こして、その結果をログで観察する”のが大切です」

**初学者**「受け身で読むより、試しながら読むほうが理解しやすそうです」

**解説者**「その通りです。  
HTTP/2 は見慣れない用語が多いですが、実際に通信を起こしてログを見ると、“何がどこで起きているか”がだんだん結びついてきます」

---

### この章のまとめ

**解説者**「第5章の要点をまとめます。

- ログは **Connection → Stream → Frame** の順で見る
    
- 最初は **stream\_id / frame の種類 / 順番** だけ見ればよい
    
- まず追うべきは **HEADERS と DATA**
    
- ログは Connection 全体ではなく **Stream ごとに分解して読む**
    
- SETTINGS や PING などは、まず“接続側の制御”として整理する
    

これが基本です」

**初学者**「“全部読む”のではなく、“どこから読むか決める”のが大事なんですね」

**実装者**「はい。  
HTTP/2 のログは難しそうに見えますが、観察の軸を固定すればかなり読めます」

**解説者**「次の章では、ここで少し脇に置いていた **フロー制御** と **HPACK** を、“深入りしすぎない範囲で”整理します。  
HTTP/2 を完全に実装する話ではなく、ログや構造を理解するために最低限どこを知っておくべきか、という観点で見ていきましょう」
