# コア・メンタルモデル

## このファイルの役割

本書では、UTF-8、HTTP/2 フレーム、HPACK という一見ばらばらに見える題材を扱う。  
しかし本書が本当に伝えたいのは、個別仕様の断片ではない。

本書の中心にあるのは、**バイト列をどういう観点で読むか**という認知の型である。

このファイルでは、本書全体を貫くメンタルモデルを先に整理する。  
各章の説明や比較は、ここで示す見方に何度も戻りながら進める。

---

## 本書の中心命題

本書の中心命題は次の一文に要約できる。

> バイト列は、16進数の壁としてではなく、境界・分岐・妥当性・状態の問題として読む。

この一文は、本書全体の読み方を決める。  
難しいのは数値そのものではなく、**どこで切るか、どう解釈が分かれるか、何を正しいとみなすか、過去の文脈が現在にどう影響するか**が見えていないことである。

---

## 4つの基本観点

## 1. 境界

最初に考えるべきことは、**どこまでが一単位なのか**である。

バイト列を読むとき、いきなり意味を考え始めると迷いやすい。  
先に必要なのは、「どこで切るのか」という問いである。

### 典型例
- UTF-8 では、1文字が何バイトなのか
- HTTP/2 フレームでは、共通ヘッダがどこまでで、ペイロードがどこからか
- HPACK では、先頭ビット列の形式に応じてどこまで読めば1表現なのか

### この観点で見る意味
境界を誤ると、その後ろにある解釈はすべて崩れる。  
したがって、バイナリ読解では意味づけより先に境界確認が来る。

### 本書での合言葉
> まず意味を見る前に、どこまで読むかを決める。

---

## 2. 分岐

境界が見えた次に重要なのは、**何が次の解釈を決めるのか**である。

バイナリ構造では、先頭バイトや先頭ビットが、その後の読み方を分岐させることが多い。  
つまり「同じ場所にある次の数バイト」でも、先頭の条件次第で意味が変わる。

### 典型例
- UTF-8 の先頭バイトが 1バイト文字か複数バイト文字かを決める
- HTTP/2 フレームの Type がペイロードの意味を決める
- HPACK の先頭ビットが indexed / literal / table update を切り替える

### この観点で見る意味
バイナリは、単純な並びではなく、**分岐つきの読み筋**を持っている。  
読者は数値を丸暗記するのではなく、「どの情報が分岐点なのか」を探す必要がある。

### 本書での合言葉
> 先頭情報は、ただの値ではなく、読み方のスイッチである。

---

## 3. 妥当性

読めたことと、正しいことは同じではない。  
ここで必要になるのが**妥当性**という観点である。

バイト列は、形式上は切り出せても、その表現が仕様上正しいとは限らない。  
また、構造を分解できても、値の範囲や組み合わせが不正な場合もある。

### 典型例
- UTF-8 では、継続バイトの形が不正かもしれない
- HTTP/2 フレームでは、長さやフラグの組み合わせが不正かもしれない
- HPACK では、参照先のテーブル状態が不正かもしれない

### この観点で見る意味
「読める」と「正しい」を分けて考えることで、iterator と validator、parser と validator の役割差が見えやすくなる。

### 本書での合言葉
> 分解できることは、正当化できることを意味しない。

---

## 4. 状態

本書の後半で最も重要になるのが、**状態**である。

状態とは、現在の入力だけでは意味が決まらず、過去に何を読んだかが現在の解釈に影響することを指す。

### 典型例
- HPACK の dynamic table
- 同じ indexed representation でも、現在の table 内容次第で意味が変わる
- 状態更新の順番によって解釈結果が変わる

### この観点で見る意味
状態が入ると、単なる decode では済まなくなる。  
入力だけでなく、履歴、更新規則、現在の文脈も含めて考える必要が出てくる。

### 本書での合言葉
> 状態を持つ構造は、今ある入力だけでは読めない。

---

## 4つの観点の関係

この4つは独立ではなく、段階的につながっている。

1. まず境界を見つける  
2. 次に分岐点を見つける  
3. その上で妥当性を確認する  
4. 必要なら状態を考慮する  

この順序が大切である。  
いきなり状態や意味へ飛びつくと混乱しやすい。  
本書は、**境界 → 分岐 → 妥当性 → 状態**という順番で整理しながら進む。

---

## 本書の3段階モデル

本書では、題材を次の3段階に置いている。

## 1. 可変長

最初の段階では、データの長さが固定ではないことに慣れる。  
ここでの中心課題は、どこまで読めば一単位なのかを見つけることにある。

### 対応する教材
- UTF-8

### 主な問い
- 1文字は何バイトか
- 先頭バイトは何を示しているか
- 途中で壊れたらどう扱うか

---

## 2. 構造

次の段階では、バイト列を複数のフィールドへ分解し、それぞれの役割を考える。  
ここでの中心課題は、単なる連続データを、意味のある構造へ変換することである。

### 対応する教材
- HTTP/2 フレーム

### 主な問い
- 共通ヘッダはどこまでか
- 各フィールドは何を決めるか
- ペイロードの意味は何によって切り替わるか

---

## 3. 状態

最後の段階では、入力だけで意味が決まらない表現に進む。  
ここでの中心課題は、現在の入力と過去の履歴を結びつけることである。

### 対応する教材
- HPACK

### 主な問い
- この表現は現在の table 状態に依存するか
- 更新はいつ行われるか
- 妥当性は入力単体で判断できるか

---

## UTF-8 / HTTP/2 / HPACK を並べる理由

この本で UTF-8、HTTP/2 フレーム、HPACK を並べるのは、単に知名度の高い題材だからではない。  
それぞれが違う種類の難しさを持っており、しかも連続した学習の階段になっているからである。

## UTF-8 が教えること
UTF-8 は、先頭バイトが後続の長さと形を決める。  
したがって、可変長と妥当性判定の最初の教材になる。

## HTTP/2 フレームが教えること
HTTP/2 フレームは、共通ヘッダとペイロードという構造を持つ。  
したがって、構造化バイナリと encode / decode の練習に向いている。

## HPACK が教えること
HPACK は、分岐に加えて dynamic table という状態を持つ。  
したがって、状態付き構造の入口として適している。

この三つを並べることで、本書は  
**可変長 → 構造 → 状態**  
という流れを自然に作ることができる。

---

## iterator / parser / validator / state の整理軸

本書の独自性の一つは、役割を混ぜずに読むための整理軸を前面に出すことである。

## iterator

iterator は、入力を順に進めるための見方である。  
何かを一つずつ取り出して前へ進むが、必ずしも高いレベルの意味づけまでは行わない。

### 典型的な問い
- 次の1単位はどこまでか
- どこで次へ進むか

UTF-8 の文字走査は、この感覚に近い。

---

## parser

parser は、入力を構造へ変換する見方である。  
単なる並びを、フィールドや意味のある単位へ分解する。

### 典型的な問い
- この並びはどのフィールドから成るか
- このヘッダはどう解釈されるか

HTTP/2 フレームの分解は、この感覚に近い。

---

## validator

validator は、入力や構造が仕様上妥当かどうかを判断する見方である。  
読むことと、正しさを確認することは別の責務である。

### 典型的な問い
- この継続バイトは正しい形か
- この長さやフラグの組み合わせは許されるか

UTF-8 の妥当性判定や、HTTP/2 の仕様チェックはここに属する。

---

## state

state は、現在の解釈に過去の履歴が関わることを表す。  
これは単独の関数ではなく、構造全体の性質として現れることが多い。

### 典型的な問い
- この参照先は今どの内容を指すか
- 直前までの更新をどう反映するか

HPACK の dynamic table が代表例である。

---

## 役割を混ぜないことの意味

実装でも理解でも、iterator / parser / validator / state を混ぜると混乱しやすい。  
どこで切るのか、どこで意味づけるのか、どこでエラーにするのか、どこで履歴を更新するのかが曖昧になるからである。

本書では、各題材を読むたびに、次の問いを繰り返す。

- これは iterator の仕事か
- これは parser の仕事か
- これは validator の仕事か
- これは state 管理の仕事か

この問いを持つだけで、複雑な仕様でも見通しが立ちやすくなる。

---

## encode / decode の往復

本書では、読むことだけでなく、**encode / decode の往復**を重視する。

decode だけを見ていると、出来上がった構造を受け取る感覚に偏りやすい。  
一方で encode に目を向けると、どの情報をどの順で並べる必要があるかが見えやすくなる。

### この往復で見えること
- どのフィールドが境界を決めるか
- 何が先頭情報として必要か
- どの値が後続の読み方を分岐させるか
- 最小限のテストデータをどう作るか

### 本書での位置づけ
本書はフル実装本ではないが、encode / decode の両側を見ることで、構造理解を強くする。

---

## 最小テストデータという考え方

本書では、巨大な実装や実運用ログだけに頼らず、**最小限のテストデータを自分で作る**という姿勢を重視する。

これは単なる学習テクニックではない。  
バイナリ構造の理解において、どこが本質かを切り出すための方法である。

### 最小テストデータが有効な理由
- 1ケース1論点にできる
- 境界がどこで決まるかを確認しやすい
- 正常系と異常系を比べやすい
- encode / decode の往復が小さくなる
- state の影響範囲を見やすい

### 本書での役割
UTF-8 の壊れた入力例、HTTP/2 フレームの最小例、HPACK の簡単な状態遷移例などを通じて、理解を手元に引き寄せる。

---

## 本書で繰り返し立てる問い

本書では、各章で次の問いを繰り返す。  
これらは、そのままバイナリ読解のチェックリストでもある。

### 境界に関する問い
- どこからどこまでが1単位か
- 長さはどこで決まるか
- 次に進む位置はどう決まるか

### 分岐に関する問い
- どのビットやフィールドが解釈を切り替えるか
- 分岐の入口は先頭にあるか、中間にあるか
- 同じ長さでも意味が変わるか

### 妥当性に関する問い
- これは形式上読めるだけか、それとも正しいか
- 許されない値や組み合わせは何か
- エラーはどこで検出されるべきか

### 状態に関する問い
- 現在の入力だけで意味が決まるか
- 過去の更新が必要か
- state 管理と decode は分けて考えられるか

## 本書の読後に残したい感覚

本書を読み終えたとき、読者に残したいのは「UTF-8 を知った」「HTTP/2 フレームを覚えた」「HPACK の名前を知った」という断片的な知識だけではない。

むしろ残したいのは、次のような感覚である。

- まず境界を見る
- 次に分岐点を見る
- その上で妥当性を見る
- 必要なら状態を見る
- 役割を混ぜずに整理する
- 小さな入力を自作して確かめる

この感覚が残れば、読者は別のプロトコルやフォーマットに出会っても、同じ見方を持ち運べる。

## 要約

本書のコア・メンタルモデルは、次のように要約できる。

### 1. バイト列はまず境界から読む
意味より先に、どこまでが一単位かを考える。

### 2. 先頭情報は分岐のスイッチである
先頭バイトや先頭ビットが、後続の解釈を決める。

### 3. 読めることと正しいことは別である
妥当性は独立した観点として扱う。

### 4. 状態が入ると入力だけでは意味が決まらない
履歴と更新規則が解釈に参加する。

### 5. 役割を混ぜない
iterator / parser / validator / state を区別する。

### 6. 小さなテストデータで理解を手元化する
最小の正常系と異常系を作ることで、構造が見えやすくなる。
