# バイナリハック入門 執筆ガイド

## このファイルの役割

このファイルは、書籍  
**『バイナリハック入門 ― UTF-8・HTTP/2・HPACK で学ぶ構造と状態』**  
の執筆・校正・リライトを支援するための品質基準と文章スタイルガイドである。

本書は、バイト列を 16 進数の壁としてではなく、**境界・分岐・妥当性・状態** の観点から読むための本である。  
したがって執筆時には、個別仕様の説明量よりも、読者がどの観点で読めるようになるかを優先する。

---

## 1. 書籍の目的

この書籍の目的は、UTF-8、HTTP/2 フレーム、HPACK を題材として、  
**可変長・構造・状態** を段階的に学びながら、バイト列を読むための認知の型を身につけてもらうことにある。

読者が最終的にできるようになるべきことは次の通りである。

- バイト列を読むとき、まず境界を探すという発想を持てる
- 先頭バイトや先頭ビットによる分岐を説明できる
- encode / decode の往復で構造を確認できる
- iterator / parser / validator の違いを整理できる
- state を持つ表現がなぜ難しくなるのかを言語化できる
- 最小限のテストデータを自作して理解する姿勢を持てる
- HTTP/2 上級編や HTTP/3 / QPACK に進むための土台を作れる

---

## 2. 想定読者

- バイト列や 16 進数に苦手意識があるが、避けずに学びたい人
- UTF-8 や HTTP/2 を題材に parser 的な考え方を学びたい人
- 最小限のテストデータを自作しながら理解したい人
- HPACK や状態付き表現の入口を求めている人
- HTTP/2 上級編へ進む前に、前提となる見方を整えたい人

補足として、本書の読者は RFC の全面読解やフル実装をすぐに目指す人に限らない。  
むしろ、仕様を読む前に「どこを見ると構造が見えるのか」を身につけたい読者を主対象とする。

---

## 3. 書籍の基本方針

### 重視すること

- 固定長と可変長の違い
- 境界の見つけ方
- 先頭情報による分岐
- encode / decode の往復
- iterator / parser / validator / state の整理
- 最小テストデータの設計
- UTF-8 と HTTP/2 を同じ認知フレームで読むこと
- HPACK を状態付き構造として理解すること

### 深入りしすぎないこと

- exploit 的な話
- RFC の網羅
- HPACK 実装の完全解説
- HTTP/2 上級編に属する詳細な状態遷移
- RST_STREAM / GOAWAY / Sans-IO の本格実装論
- QPACK や QUIC の詳細
- 特定言語でのフル実装チュートリアル

本書は、実装のすべてを教える本ではない。  
**読む観点を揃え、次に学ぶ対象を切り分けられるようにする本**である。

## 4. 導入対話と本文の役割

この書籍では、全面対話形式は採用しない。  
各章の冒頭に短い導入対話を置き、その後の本文は、書籍としての判断主体が見える専門家向け文章で構成する。  
ただし、総整理に近い章や比較中心の章では、やや対話を厚めにしてよい。

この方針は、添付の中級書ガイドラインにある「短い導入対話＋本文中心＋整理ブロック」という構成思想を継承しつつ、本書の目的に合わせて再設計したものである。:contentReference[oaicite:1]{index=1}

### 導入対話の役割

- その章で起きやすい誤解や混乱を先に表面化する
- 章の問いを明確にする
- 読者が何を見落としているのかを見える形にする
- 本文を読む前のメンタルモデルを準備する

### 本文の役割

- 本書としての判断の筋道を示す
- 境界・分岐・妥当性・状態のどれを扱っているかを明確にする
- 個別知識を、再利用可能な読み方として圧縮する
- その章で何が見えるようになるのかを示す

### 文体上の基本姿勢

- 「私はこう考える」よりも、「本書ではここをこう切る」「ここではこの粒度で整理する」を優先する
- 随筆的な私語りではなく、説明上の方針や境界が見える文体を採る
- 毎文で主語を立てる必要はないが、重要な切り分けでは判断主体が見えるようにする

### 整理ブロックの役割

- 用語の対応関係を整理する
- 構造比較や責務分担を視覚的に示す
- 本文の圧縮度を補う
- 次章へ持ち越すメンタルモデルを明示する

---

## 5. 導入対話の役割分担

導入対話の登場人物は原則として 3 役で固定する。

### 初学者

役割：

- 疑問・誤解・混乱を出す
- 16 進数や仕様を前にして感じやすい不安を代弁する
- 「何を見ればよいかわからない」という状態を言語化する

注意：

- すでに整理された説明を自力で言わない
- 上級者の視点を先取りしない
- 用語を勝手に再定義しない

### 解説者

役割：

- その章で必要な抽象を整理する
- 境界・分岐・妥当性・状態のどこが主題かを示す
- 導入対話の中で章の地図を作る

注意：

- 本文で扱う詳細を先に全部説明しない
- 実装細部に入りすぎない
- 対話だけで章を完結させない

### 実装者

役割：

- ログ、ダンプ、テストデータ、実務での見え方を補足する
- 抽象的な話を観察可能な現実へ接続する
- 「利用者からはどう見えるか」を支える

注意：

- 抽象理論を主導しない
- 解説者の役を奪わない
- 特定言語や実装論の枝葉に広げすぎない

この三役の分担も、添付ファイルの「初学者・解説者・実装者」の分担設計を踏襲している。:contentReference[oaicite:2]{index=2}

## 6. 文章スタイル

### 基本トーン

- 技術的に正確
- 入門書より圧縮度は高いが、不親切にはしない
- 判断の筋道が見える
- 読者を置き去りにしない
- 本文は見通しがよく、導入対話は問いが明確
- 個別仕様の説明で終わらず、読む力に接続する

### 文体ルール

- 本文はです・ます調を基本とする
- 一文一義を意識する
- 一文を長くしすぎない
- 定義・区別・結論は言い切る
- 「なぜそうなるか」を省略しない
- 例は必要最小限に絞る
- 比喩は導入や補助説明で使い、本文では用語へ戻す

### 本書で特に重視すること

- 章冒頭で「何が見えるようになる章か」を明示する
- その章が主に扱う観点を明示する
  - 境界
  - 分岐
  - 妥当性
  - 状態
- 具体例で終わらず、読み方へ一般化する
- 「ここから先は上級編の話題」と必要に応じて明示してよい
- encode / decode の往復や最小テストデータの意味を示す

### 避けるべき文体

- 16 進数の見た目の説明だけで終わる文章
- 抽象語だけが増えて観察対象との接続が消える文章
- 上級編の話題を中途半端に混ぜて焦点がぼける文章
- 「〜と言えるだろう」が連続して結論が弱くなる文章
- 用語や例を並べるだけで、どの観点で読めばよいかが見えない文章

## 7. 音声向け情報と視覚向け情報の分離

本書でも、  
**声に出して自然に読める情報** と  
**目で見たほうが整理しやすい情報** を分ける。  
この整理方針は添付ファイルのスタイルをそのまま継承する。:contentReference[oaicite:3]{index=3}

### 導入対話に置くもの

- 誤解
- 疑問
- 章の問い
- メンタルモデルの下準備
- 見た目の難しさと構造上の難しさのズレ

### 本文に置くもの

- 境界の見つけ方
- 分岐の読み方
- 妥当性判定の位置づけ
- 状態付き構造の難しさ
- iterator / parser / validator / state の整理
- encode / decode の往復の意味

### 本文の外に置くもの

- 箇条書き
- 比較表
- バイト列例
- フィールド対応表
- 壊れた入力の比較
- 章末まとめ
- この章で残すべきメンタルモデル
- 上級編へ送る話題の明示

### 運用ルール

- 導入対話で箇条書きを使わない
- 本文でも長い列挙は避ける
- 一覧や比較は整理ブロックへ出す
- バイト列例は必要最小限にする
- 章末は復習しやすい形式で固定する

## 8. 見出し構造の基準

見出しレベルは全章で統一する。

### 基本ルール

- 章タイトル：`##`
- 章内セクション：`###`
- 「この章で残すべきメンタルモデル」：`###`
- 「まとめ」：`##`

### 禁止事項

- 章ごとに見出しレベルを変える
- 本文中の章タイトルに h1 を使う
- 章末整理の見出し名を毎回変える
- 1 章だけ極端に細かく分割する

この統一ルールも、添付ファイルの見出し設計方針を継承している。:contentReference[oaicite:4]{index=4}

## 9. 章フォーマットの基準

各章は原則として次の流れで構成する。

### 1. 導入対話

- 初学者がその章の誤解や疑問を出す
- 解説者がその章の問いを示す
- 実装者が必要なら現実の見え方を一言補足する

### 2. 本文

- 判断主体が見える専門家向け文章で書く
- 必要に応じて `###` セクションを置く
- その章で扱う観点を明確にする
- 個別例から読み方へ一般化する

### 3. 整理ブロック

- 用語対応
- 境界と分岐の比較
- iterator / parser / validator の区別
- encode / decode の対応
- 上級編へ送る話題の明示

### 4. この章で残すべきメンタルモデル

- 章の中核を短く圧縮する
- 新情報は追加しない
- 次章へ持っていくべき抽象を明示する

### 5. まとめ

- 3〜5 項目程度の箇条書き
- 章タイトルに対応する要点に絞る
- 新しい情報は追加しない

### 6. 次章への接続

- 1〜2 文で次章に渡す
- 抽象の継続性が見えるようにする

## 10. 校正時の最重要品質基準

校正では、特に次の観点を優先して確認する。

### A. 読む観点が明確か

特に以下が明確であること。

- どこが境界か
- 何が分岐点か
- 何を妥当性として扱うか
- どこで状態が入るか
- どこまでが iterator の役割か
- どこからが parser / validator の役割か

### B. 抽象と具体の接続があるか

- UTF-8 / HTTP/2 フレーム / HPACK の具体例と抽象が接続されているか
- バイト列例が単なる例示で終わらず、見るべき点に接続されているか
- ログやダンプやテストデータの見方につながっているか

### C. 章の問いに答えているか

- 導入対話の疑問に本文が答えているか
- 話題が途中で広がりすぎていないか
- 上級編の話題が侵食しすぎていないか

### D. 圧縮が効いているか

- 本文が長くなったあとに整理があるか
- 章末で復習可能な形になっているか
- 抽象語が増えるだけで理解が進まない状態になっていないか

### E. 章ごとの粒度が揃っているか

- 見出し粒度が揃っているか
- 章の重さに応じた分量になっているか
- 導入対話が長くなりすぎていないか
- 本文と整理ブロックの役割が混ざっていないか

この「最重要品質基準」の立て方も、添付ファイルのレビュー観点を本書向けに移植したものである。

---

## 11. 具体的なレビュー観点

### 用語

- 初出時に必要な説明があるか
- 同じ概念に複数の言い換えを乱用していないか
- iterator / parser / validator / state が混同されていないか
- 文字コードの用語とプロトコルの用語が不用意に混ざっていないか

### 導入対話

- 章の問いを作れているか
- 初学者が賢くなりすぎていないか
- 解説者が本文の内容を先に全部言っていないか
- 実装者が補足役に収まっているか

### 本文

- 本書としての判断の筋道が見えるか
- 境界・分岐・妥当性・状態のどれを扱っているかが分かるか
- 個別例を一般化できているか
- 「なぜそう切るか」が書かれているか

### 構成

- 1 セクション 1 テーマになっているか
- 話題の飛躍がないか
- 整理ブロックが本文の代わりになっていないか
- 「メンタルモデル」と「まとめ」が重複しすぎていないか

### 可読性

- 一文が長すぎないか
- 抽象語が連続しすぎないか
- 例が多すぎて主線が見えなくなっていないか
- 章の中心命題が途中で薄れていないか

---

## 12. まとめの基準

各章の最後の「まとめ」は、次の基準に従う。

- 箇条書き 3〜5 項目
- 新しい情報を追加しない
- 章タイトルと対応する内容に絞る
- 読む観点の区別が再確認できるようにする
- 次章に必要な前提が残るようにする

悪い例：

- 抽象的すぎて復習に使えない
- 本文にない上級話題を足す
- 1 項目が長すぎる
- 章ごとにスタイルが変わる

良い例：

- 境界の見つけ方
- 分岐点の見方
- 妥当性の位置づけ
- 状態が入ると何が難しくなるか
- 次章へ持ち越す抽象

---

## 13. リライト方針

校正や改善を行う際は、単なる誤字修正ではなく、必要に応じて以下を行う。

- 冗長な説明の圧縮
- 抽象語の具体化
- 具体例と抽象の対応関係の明確化
- 長い本文の分割
- 導入対話と本文の役割分離
- 整理ブロックの追加や再配置
- 上級話題の切り出し
- 章末まとめの統一フォーマット化

ただし、以下は避ける。

- 上級編向けの詳細知識を勝手に大量追加すること
- 章のスコープを勝手に拡張すること
- 導入対話を長い講義に変えてしまうこと
- 全面対話形式に戻してしまうこと
- 書籍全体の密度やトーンを章ごとに変えてしまうこと

---

## 14. 出力時の期待

レビュー・校正・リライトを行うときは、必要に応じて次の観点で整理する。

- 構成上の問題
- 文章上の問題
- 用語運用上の問題
- 導入対話の役割上の問題
- 本文の抽象化不足または抽象化過多
- 修正案
- 必要なら統一版のリライト

## 15. 本書で特に守るべき一文

この本は、バイト列を  
**「16進数の羅列として眺める対象」ではなく「境界・分岐・妥当性・状態で読める構造」として理解させる本**  
である。

校正・改善・リライトの判断は、常にこの方針に従って行う。

## 16. 本書で繰り返し守るべき補助原則

- UTF-8 は文字コードの章としてではなく、可変長データの教材として扱う
- HTTP/2 フレームは仕様項目の列挙ではなく、構造化バイナリの教材として扱う
- HPACK は圧縮方式の説明で終わらせず、状態付き構造として扱う
- 個別知識を増やすより、読む観点を揃えることを優先する
- 読者が「どこから手をつければよいかわからない」状態を減らす

## 17. シリーズ内での位置づけを崩さないための注意

本書は、HTTP/2 入門書の続編であり、HTTP/2 上級書の前提でもある。  
したがって、次のバランスを崩さない。

- 入門書よりは一段抽象化する
- ただし上級書の実装論へ踏み込みすぎない
- 中級書の API 設計論とも役割を混同しない
- 本書は「観察の次に来る、読む力の本」であることを保つ
