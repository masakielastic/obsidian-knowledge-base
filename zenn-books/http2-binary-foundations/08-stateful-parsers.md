## 第8章　state を持つ parser をどう考えるか

### 導入対話

**初学者**  
「つまり、HPACK の話を続けるけれど、本当に見たいのは HPACK 固有の細かい規則ではなくて、状態付き構造一般の見方なんですね」

**解説者**

**初学者**

**解説者**  
「はい。そこが見えるようになると、状態付き構造を『全部いっぺんに難しいもの』としてではなく、いくつかの責務に分けて考えられるようになります」

### この章で扱うこと

前章では、HPACK を**状態を持つバイナリ構造**として読んだ。  
そこでは、現在の入力だけでは意味が決まらず、dynamic table のような前提状態が必要になること、さらに現在の入力が未来の解釈条件まで変えることを見た。

この章では、その具体例を素材にしながら、  
**state を持つ parser をどう考えるか**  
を整理する。

ここで重要なのは、実装の正解を決めることではない。  
また、HPACK の全仕様を設計論として再現することでもない。  
本章の目的は、状態付き構造に出会ったときに、難しさを分解して見られるようにすることである。

そのために、本章では次の三つを中心に扱う。

- 入力と履歴を分けて考える
    
- 更新規則を独立したものとして捉える
    
- parser と state 管理の境界を意識する
    

第7章では、「状態があると今ここにある入力だけでは足りない」という事実を見た。  
第8章では、その事実をもう少し設計的に言い換える。  
つまり、状態付き構造を読むには、**入力・履歴・更新規則**という少なくとも三つの層を区別しておく必要がある、ということである。

---

### state がない parser と何が違うのか

比較のために、まず state がない parser を思い出したい。

UTF-8 を読む場合、先頭バイトを見て、その文字がどこまで続くかを判断する。  
HTTP/2 フレームを読む場合、共通ヘッダを取り出し、Length や Type を見てフレームを分解する。  
これらにも境界や分岐や妥当性の問題はある。  
しかし主な意味決定は、その場の入力から始められる。

もちろん、途中まで読んだ位置や、現在どこを指しているかという意味での局所的な文脈はある。  
しかしそれは主として「今どこまで進んだか」という処理上の文脈であって、  
**入力の意味そのものが過去の履歴によって変わる**  
という種類の状態ではない。

state を持つ parser では、この点が変わる。  
入力を読むだけでは不十分であり、その入力を解釈するための前提として、すでに維持されている履歴が必要になる。  
しかも、その入力を読んだ結果として履歴そのものが更新されることもある。

つまり、state がない parser では  
「入力を読んで構造を出す」  
という一本の流れが中心だった。  
state を持つ parser ではそこに、  
「どの履歴の上で読むか」  
「読んだ結果として履歴をどう変えるか」  
が加わる。

この差が、難しさの質を変えている。

---

### 現在の入力だけでは決まらない意味

state を持つ parser の核心は、  
**現在の入力だけでは意味が決まらない**  
という点にある。

このことを、単なる感想ではなく、読解上の構造として捉える必要がある。  
現在の入力だけでは意味が決まらないとは、少なくとも次のようなことである。

ある値が何を参照しているかを知るには、あらかじめ存在している table や履歴を参照しなければならない。  
ある入力が妥当かどうかを知るには、過去にどんな更新が行われたかを知らなければならない。  
ある操作の結果を理解するには、その入力が今後の状態をどう変えるかも考えなければならない。

ここで重要なのは、「入力だけでは決まらない」という事実を曖昧にしないことである。  
それは単に「背景知識が必要」という意味ではない。  
もっと具体的に、**入力の意味が、保持されている状態と結びついている**ということである。

この視点がないと、状態付き構造は「規則が多いもの」「覚えることが多いもの」に見えてしまう。  
しかし本質はそこではない。  
大事なのは、意味決定の材料が一箇所にまとまっていないこと、そしてその材料の一部が時間的に蓄積されることにある。

---

### 入力・履歴・更新規則を分けて考える

本章の中心となる整理はここにある。  
state を持つ parser を考えるときは、少なくとも次の三つを分けたほうがよい。

**入力**  
いま目の前にあるバイト列、あるいはその場で処理しているデータである。

**履歴**  
それ以前の入力処理の結果として保持されている状態である。  
HPACK なら、典型例は dynamic table である。

**更新規則**  
現在の入力を処理したときに、履歴をどう変えるかを定める規則である。

この三つを一度分けて考えるだけで、状態付き構造の見え方はかなり変わる。  
難しさが「全部まとめて難しいもの」ではなく、  
「今は入力を見ているのか」  
「履歴参照が必要なのか」  
「更新規則が働いているのか」  
という形で分解されるからである。

前章で見た HPACK をこの枠組みで見ると、かなり整理しやすい。  
indexed header field の読解では、現在の入力だけでなく履歴としての table が必要になる。  
literal header field の一部では、現在の入力が future state を変えるため、更新規則が関わる。  
つまり HPACK の複雑さは、入力・履歴・更新規則が密接に絡むところから生じている。

本章では、この整理を一般化して使う。

---

### dynamic table は「履歴」である

HPACK を使って具体的に考えるなら、dynamic table はもっともわかりやすい履歴の例である。

dynamic table は、その場の入力そのものではない。  
それ以前の処理結果として蓄積され、現在の入力を読むための前提になっている。  
したがって、本書の整理では dynamic table はまず**履歴**として捉えるのがよい。

この見方の利点は大きい。  
dynamic table を「圧縮用の仕組み」としてだけ見ると、どこで現在の読解に効いているのかが見えにくい。  
しかし履歴として見ると、その役割は明確になる。  
今の入力が何を意味するかを決めるために参照される、ということがはっきりするからである。

さらに、dynamic table はただの保存場所ではない。  
入力処理に応じて変化し、その変化が将来の解釈条件になる。  
この意味で dynamic table は、単なるデータ構造ではなく、**時間方向に伸びた意味決定の土台**である。

状態付き parser を考えるときには、この「履歴としての state」という見方が重要になる。  
state は抽象的な言葉だが、履歴と言い換えると、  
何が現在の入力の意味に効いているのかが少し具体的に見えてくる。

---

### 更新規則を独立した層として見る

状態付き構造を難しくしているものの一つは、履歴が勝手に変わるわけではないという点である。  
どの入力が履歴を変えるのか、どう変えるのかには規則がある。  
これが**更新規則**である。

ここで更新規則を独立した層として見ることが大切になる。  
入力を読んだ結果、状態が変わる。  
この一文だけでは、何がどのように変わるのかが曖昧だからである。

たとえば、ある literal 表現が table に追加されるのか、追加されないのか。  
ある操作が table size に影響するのか、単に一時的な表現で終わるのか。  
こうしたことは、現在の入力から履歴への写像として理解したほうがよい。

つまり、状態付き parser を考えるときには、  
「入力をどう読むか」  
だけでなく、  
「その読解結果を使って履歴をどう更新するか」  
も別の論点として立ち上がる。

この区別をしないと、parser は何でも屋になりやすい。  
バイト列を切り、構造を作り、参照を解決し、状態を更新し、妥当性も判定し、エラー処理もする。  
そうなると、どこが複雑なのかさえ見えにくくなる。  
更新規則を別の層として意識することは、この混雑を避ける第一歩である。

---

### parser と state 管理の境界

ここで自然に出てくる問いがある。  
**parser と state 管理はどこで分けて考えるべきか**  
という問いである。

本書はここで、厳密な設計パターンの正解を出すつもりはない。  
ただし、考えるための軸は示せる。

parser の中心的な役割は、入力から構造を露出させることである。  
state 管理の中心的な役割は、履歴を保持し、更新規則に従ってそれを変化させることである。  
この二つは実装上近接することが多いが、責務としては別物として見たほうがよい。

たとえば、ある入力が「これは table の何番を参照する」という構造を持っていることを読み取るのは parser 的な仕事である。  
その番号を実際に履歴へ照らして解決すること、あるいは現在の入力を履歴へ追加して未来の参照条件を変えることは、state 管理の側面が強い。

もちろん現実のコードでは、これらが完全に分離していないこともある。  
しかし頭の中で境界を引けることが重要である。  
今の複雑さは「入力の切り方」の問題なのか。  
それとも「履歴の持ち方・更新の仕方」の問題なのか。  
この区別があるだけで、状態付き構造の設計はかなり見通しがよくなる。

---

### validator はどこまで責任を持つのか

state を持つ parser を考えるとき、validator の位置づけも少し変わる。  
前の章まででも validator は重要だったが、状態付き構造では何を妥当性として見るかが広がるからである。

たとえば、単純な境界や形式の妥当性だけでなく、  
「その参照は現在の履歴の下で有効か」  
「その更新は規則に反していないか」  
といった問いが加わる。  
つまり validator は、入力単体だけではなく、**入力と履歴の関係**まで含めて妥当性を見る必要が出てくる。

ここでもやはり、責務を切り分けて考えることが大事である。  
parser は入力から構造を出す。  
state 管理は履歴を保持し更新する。  
validator は、その参照や更新や関係が許されるかを判定する。  
この三つを区別せずにまとめると、状態付き構造の不正ケースはとたんに扱いづらくなる。

特に壊れた入力を考えるとき、この整理は重要になる。  
第6章で扱ったように、最小の不正ケースを作ろうとすると、  
何が構造の問題で、何が状態関係の問題で、何が更新規則違反なのかを分けて考えなければならないからである。

---

### 状態付き構造を読むときのチェックポイント

ここまでの整理を、読み方の実践に引き寄せてまとめると、state を持つ parser を考えるときは、少なくとも次の点を確認するとよい。

まず、**現在の入力だけで意味が決まる部分**と、**履歴参照が必要な部分**を分ける。  
これを区別しないと、状態が本当に必要な場所が見えない。

次に、**どの入力が履歴を更新するのか**を明確にする。  
すべての入力が履歴を変えるわけではない。  
更新規則が働く箇所を見極める必要がある。

さらに、**更新前の状態で読むのか、更新後の状態で読むのか**という時間順序を意識する。  
状態付き構造では、順序の取り違えが意味の取り違えにつながる。

そして最後に、**妥当性違反がどこで発生しうるか**を分ける。  
入力形式そのものの問題なのか、履歴参照の問題なのか、更新規則の問題なのかを切り分ける。

このようなチェックポイントを持つと、状態付き構造は少し読みやすくなる。  
複雑さを消せるわけではないが、少なくとも何が複雑なのかを局所化できるようになる。

---

### HPACK は状態付き parser の教材である

ここで、本章における HPACK の位置づけを改めて確認しておきたい。

前章では、HPACK は**状態を持つバイナリ構造**の教材だった。  
本章ではさらに一歩進めて、HPACK は**state を持つ parser を考える教材**でもある。

なぜ教材として優れているのか。  
それは、入力・履歴・更新規則の三つが比較的はっきり現れるからである。  
indexed header field では履歴参照が見える。  
literal header field の一部では更新規則が見える。  
dynamic table では履歴そのものが見える。  
そのため、状態付き構造で何が問題になるのかを比較的具体的に追いやすい。

もちろん HPACK は簡単な題材ではない。  
しかし、その難しさは本書の後半に進むためにちょうどよい。  
バイト列読解から実装読解へ橋を架けるうえで、  
「今の入力だけを読めば終わりではない」  
という感覚を身につけるには、適切な題材だからである。

---

### 状態付き parser を設計問題として見すぎないために

ここで一つ注意しておきたい。  
state を持つ parser の話は、そのまま API 設計やクラス分割や Sans-I/O の話へ進みやすい。  
実際、そこへつながっていくのは確かである。

ただし本章では、まだそこへ深く入りすぎない。  
この本の主眼は、設計パターンの紹介ではなく、バイト列と構造をどう読むかにあるからである。  
したがって本章では、設計の正解を与えるよりも前に、  
**何を分けて見ればよいか**  
を整えることを優先する。

入力・履歴・更新規則。  
parser・state 管理・validator。  
このあたりの区別が見えていれば、後で実装設計の議論に入ったときにも、何が論点なのかを把握しやすくなる。  
逆にここが曖昧なまま設計論へ進むと、全部が同時に複雑に見えてしまう。

本章は、その混雑を減らすための章である。

### 次章への橋――バイナリハックから HTTP/2 実装へ

ここまでで、本書は「読む力」の後半に入ってきた。  
UTF-8 で可変長を読み、HTTP/2 フレームで構造化バイナリを読み、HPACK で状態付き構造を読み、さらに本章で state を持つ parser の見方を整理した。

次章では、この蓄積を、**HTTP/2 実装やライブラリ読解への橋**として回収する。  
そこで重要になるのは、いま整理した責務の違いである。  
iterator / parser / validator の違い。  
入力・履歴・更新規則の違い。  
parser と state 管理の境界。  
これらが見えていると、HTTP/2 実装の中で何がどの責務を担っているかを読みやすくなる。

つまり第8章は、後半の理論整理の到達点であると同時に、第9章へ渡すための準備でもある。  
ここで状態付き構造の難しさを役割へ分解して見られるようになることが、そのまま実装読解の足場になる。

### 整理ブロック

#### state を持つ parser を考える三つの層

| 層 | 何を表すか |
| --- | --- |
| 入力 | いま処理しているバイト列や表現 |
| 履歴 | 過去の処理結果として保持されている状態 |
| 更新規則 | 現在の入力によって履歴をどう変えるかを定める規則 |

#### parser / state 管理 / validator の見え方

| 役割 | 中心となる問い |
| --- | --- |
| parser | この入力はどんな構造を持つか |
| state 管理 | この構造をどの履歴の上で解釈し、どう更新するか |
| validator | その参照や更新や関係は妥当か |

#### 状態付き構造を読むときのチェックポイント

- 現在の入力だけで決まる部分はどこか
    
- 履歴参照が必要な部分はどこか
    
- どの入力が履歴を更新するのか
    
- 更新の前後で意味がどう変わるのか
    
- 妥当性違反が入力・参照・更新のどこで起きるのか
    

#### この章で押さえるべきこと

- state がない parser と state がある parser では難しさの質が違う
    
- 状態付き構造では現在の入力だけでは意味が決まらない
    
- 入力・履歴・更新規則を分けて考えると整理しやすい
    
- parser と state 管理は責務としては分けて見たほうがよい
    
- validator は入力単体だけでなく、入力と履歴の関係も扱う
    

---

### この章で残すべきメンタルモデル

state を持つ parser の難しさは、  
**入力を読むことと、履歴を参照・更新することが結びつく**ところにある。

---

## まとめ

- state を持つ parser では、現在の入力だけでは意味が決まらず、履歴が必要になる。
    
- 状態付き構造を考えるときは、入力・履歴・更新規則を分けて考えると整理しやすい。
    
- HPACK の dynamic table は、現在の読解条件であると同時に未来の読解条件にもなる履歴である。
    
- parser は入力から構造を出し、state 管理は履歴を保持・更新し、validator はその関係の妥当性を判定する。
    
- 状態付き構造では、形式の妥当性だけでなく、参照や更新の妥当性も重要な論点になる。

**バイナリハックから HTTP/2 実装へ** という橋渡しを行う。  
ログやダンプを読む力が、実装のどの責務理解につながるのかを整理しながら、上級編で扱う話題への入口を作っていく。
