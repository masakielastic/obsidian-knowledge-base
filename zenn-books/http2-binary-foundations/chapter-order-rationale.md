# 章順設計メモ

## このファイルの役割

このファイルは、本書『バイナリハック入門 ― UTF-8・HTTP/2・HPACK で学ぶ構造と状態』の**章順の意図**を明確にするためのメモである。

本書の章配列は、話題を並べただけの目次ではない。  
読者がどこでつまずきやすいか、どの概念を先に持っていると次の章が読みやすくなるか、どの題材がどの役割を担うかを踏まえて設計している。

本書は、バイト列を「16進数の壁」としてではなく、**境界・分岐・妥当性・状態**で読む本である。  
そのため章順も、仕様の網羅順ではなく、**認知負荷の低い順・理解が積み上がる順**で組み立てる必要がある。

---

## 本書の章順の基本原理

本書の章順は、次の三つの原理に基づいている。

### 1. 難しさの正体を先に言語化する

読者はしばしば、バイト列を見た時点で「難しい」と感じる。  
しかし、その難しさを分解しないまま UTF-8 や HTTP/2 フレームの説明に入ると、個別知識の列挙として受け取られやすい。

そこで本書では最初に、

- どこで切るのか
- 何が次の解釈を決めるのか
- 何をもって正しいとするのか
- どこで過去の状態が効いてくるのか

という読み方の軸を導入する。  
これにより、後続の章が「別々の話題」ではなく、「同じ見方の反復」に見えるようになる。

### 2. 可変長 → 構造 → 状態 の順で進む

本書の中心となる階段は次の三段階である。

1. 可変長
2. 構造
3. 状態

この順序には理由がある。  
状態付き構造は最も難しいが、その前に「どこまで読むか」「どのフィールドに分解するか」が見えていないと、状態の話はほぼ確実に空中戦になる。

そのため本書では、

- UTF-8 で可変長を学ぶ
- HTTP/2 フレームで構造を学ぶ
- HPACK で状態を学ぶ

という順に進める。

### 3. 題材の知識ではなく、読む力を積む

本書の目的は、UTF-8 や HPACK を断片的に知ることではない。  
読者が他のバイナリ形式でも応用できる読み方を獲得することにある。

したがって章順も、

- 知識の網羅
- RFC の順番
- 実装依存の都合

ではなく、

- 読む姿勢が育つ順番
- 役割の違いが見えやすい順番
- 自作テストデータへ自然につながる順番

を優先している。

---

## 章順の全体像

本書の章配列は、次の流れを意図している。

1. 難しさの正体を定義する
2. 固定長と可変長を区別する
3. UTF-8 で可変長を学ぶ
4. HTTP/2 フレームで構造を学ぶ
5. iterator / parser / validator で整理する
6. テストデータ自作で理解を手元化する
7. HPACK で状態を導入する
8. 状態付き parser を整理する
9. HTTP/2 実装への橋を架ける
10. 他の形式へ一般化する

この流れは、単なる説明順ではなく、**理解の抽象度を徐々に上げながら、それを実例で支える構造**になっている。

## 各章の配置理由

## 第1章 バイナリデータはなぜ難しく見えるのか

### この位置に置く理由
最初の章では、具体例に入る前に「難しさの種類」を言語化する。  
これを飛ばすと、読者は UTF-8 も HTTP/2 も「知らない規則が増える話」として受け取りやすい。

ここで本書の中心概念である

- 境界
- 分岐
- 妥当性
- 状態

を提示しておくことで、後続の章がすべて同じ地図の上に載る。

### この章を先頭に置かない場合の問題
いきなり UTF-8 から始めると、文字コード本のように見えやすい。  
いきなり HTTP/2 フレームから始めると、プロトコル仕様の解説本に見えやすい。  
先に「本書は何を学ぶ本なのか」を定義する必要がある。

---

## 第2章 固定長と可変長

### この位置に置く理由
境界の問題を最も基礎的な形で扱えるからである。  
固定長と可変長の違いを理解していないと、UTF-8 の可変長性や HTTP/2 の固定長ヘッダ＋可変長ペイロードという構造が見えにくい。

この章は、後続の各題材に入る前の「地ならし」である。

### 第1章の直後に置く意味
第1章で「境界が重要」と述べた直後に、その最初の具体化として固定長と可変長を置く。  
ここで読者は、境界の取り方には型があると理解できる。

### この章を後ろへ回さない理由
UTF-8 や HTTP/2 フレームの説明の途中で固定長・可変長を都度説明すると、概念が個別論点に埋もれる。  
先に切り出しておいた方が後続で再利用しやすい。

---

## 第3章 UTF-8 は可変長データの教材である

### この位置に置く理由
可変長の最初の教材として、UTF-8 が最も扱いやすいからである。  
読者にとって文字は比較的身近であり、「1文字が何バイトか」という問いは直感とずれやすいため、学習効果が高い。

また UTF-8 では、

- 先頭バイトが長さを決める
- 継続バイトという補助要素がある
- 妥当な並びと壊れた並びを区別できる

という点で、境界・分岐・妥当性を一度に学べる。

### HTTP/2 フレームより先に置く理由
HTTP/2 フレームは構造化バイナリとして優れているが、最初の教材としてはフィールド数が多く、プロトコル知識も混ざりやすい。  
UTF-8 はより小さく、1単位の境界に集中できる。

### この章の役割
この章は「文字コードの章」ではなく、**可変長の読み方の章**である。  
それを明示することで、本全体のテーマから逸れないようにする。

---

## 第4章 HTTP/2 フレームは構造化バイナリの教材である

### この位置に置く理由
UTF-8 で「どこまで読むか」を学んだあとに、HTTP/2 フレームで「どう分解するか」へ進むためである。  
ここで読者は、バイト列を単位ごとに切るだけでなく、フィールド構造として捉える段階へ移る。

HTTP/2 フレームは、

- 9バイト共通ヘッダ
- 長さによって続く可変長ペイロード
- Type や Flags による分岐

という構造を持ち、固定長と可変長が同居する教材として非常に優れている。

### UTF-8 の直後に置く意味
UTF-8 で「可変長の読み方」を身につけた直後に、HTTP/2 フレームで「構造の読み方」へ拡張する。  
この並びにより、両者の共通点と違いが見えやすくなる。

### なぜ HPACK より先か
HPACK は HTTP/2 の文脈に属するが、状態を持つため認知負荷が一段高い。  
その前に、HTTP/2 フレームというより素直な構造を先に押さえておく必要がある。

---

## 第5章 iterator / parser / validator で整理する

### この位置に置く理由
第3章と第4章で具体例を見たあとに、役割の違いを抽象化して整理するためである。  
あまり早くこの章を置くと抽象論になりやすく、逆に遅すぎると後続の HPACK 章で役割混同が起きやすい。

ここで一度、

- UTF-8 のどこが iterator 的か
- HTTP/2 フレームのどこが parser 的か
- 妥当性判定はどこで入るか

を整理しておくことで、後半の state の話が理解しやすくなる。

### この章をここに置く効果
具体例を見たあとに抽象化することで、読者は役割の違いを現物と結びつけて理解できる。  
また、この後に続くテストデータ作成や HPACK の議論でも、この整理軸を繰り返し使える。

---

## 第6章 テストデータを自分で作る

### この位置に置く理由
役割の整理を終えたあと、受け身の理解から能動的な理解へ移るためである。  
ここで最小のテストデータを自作する視点を入れることで、後半の HPACK や状態付き parser の理解が大きく安定する。

### 第5章の直後に置く意味
iterator / parser / validator の区別が見えた状態でテストデータを作ると、

- どこを壊せば境界が崩れるか
- どこを変えれば分岐が変わるか
- どこで妥当性違反が起こるか

を意識しやすい。

### HPACK より前に置く理由
HPACK は状態を含むため、いきなりそこへ入ると入力例の設計も難しくなる。  
まず状態を持たない、あるいは比較的単純な対象でテストデータの作り方に慣れておく方がよい。

---

## 第7章 HPACK は状態を持つバイナリ構造である

### この位置に置く理由
ここが本書の後半への入口であり、「状態」という新しい難しさを導入する章だからである。  
第1章から第6章までで、

- 境界
- 分岐
- 妥当性
- parser 的な分解
- テストデータの考え方

を先に積み上げておくことで、HPACK の難しさを「圧縮だから難しい」と誤解せずに済む。

### この章で導入するもの
- 先頭ビットによる分岐
- integer 表現
- indexed / literal の違い
- dynamic table
- 入力だけでは意味が決まらない構造

### この位置より前に置かない理由
HPACK を早く出しすぎると、読者は状態・分岐・構造を一度に処理しなければならなくなる。  
その負荷を避けるため、ここまで待つ。

---

## 第8章 state を持つ parser をどう考えるか

### この位置に置く理由
第7章で HPACK の具体像を見たあとに、状態付き parser の設計原理を抽象化するためである。  
第7章が「HPACK とは何か」を扱う章なら、第8章は「状態付き構造をどう切り分けて考えるか」を扱う章である。

### 第7章との役割分担
- 第7章：具体的な状態付き構造を観察する
- 第8章：そこから一般化して設計上の論点を取り出す

この順番にすることで、抽象論が地に足のついたものになる。

### ここで扱うこと
- 入力と履歴の分離
- state 更新のタイミング
- validator の責務の位置
- parser と state 管理の境界
- 状態付き decode の難しさ

---

## 第9章 バイナリハックから HTTP/2 実装へ

### この位置に置く理由
本書の学びをシリーズ全体へ接続するためである。  
ここまでの章で読者は、

- 可変長
- 構造
- 妥当性
- 状態
- テストデータ

という観点を獲得している。  
この章では、それらが HTTP/2 のライブラリ設計や実装読解にどうつながるかを示す。

### この章が終盤にある意味
あまり早い段階で実装の話を出すと、読者は設計や API の話題に意識が引っ張られ、本書の中心であるバイト列読解がぼやける。  
終盤に置くことで、「ここまで積んできた見方が、実装ではこう効く」と自然に接続できる。

### 上級書との関係
この章では RST_STREAM / GOAWAY / Sans-IO には深入りしない。  
それらは上級書の役割であり、本章はその前提を整理する橋渡しに留める。

---

## 第10章 他のプロトコルやフォーマットへ広げる

### この位置に置く理由
最後に、本書の学びを HTTP/2 や UTF-8 に閉じないものとして回収するためである。  
本書の価値は、個別仕様の理解だけでなく、別の形式にも持ち運べる読み方を提供することにある。

### 最終章に置く意味
ここまでで読者は三つの具体例を経験している。  
そのうえで初めて、

- 他の文字コード
- 他のバイナリフォーマット
- HTTP/3 / QPACK
- TLV 型の構造
- 状態付き表現一般

へ自然に視野を広げられる。

### この章の役割
終章として、「本書は HTTP/2 の周辺知識本ではなく、バイナリ読解の土台である」ということを明確にする。

---

## なぜこの章順でないといけないのか

本書の章順を入れ替えると、いくつかの問題が起きやすい。

### HPACK を前半に持ってくる場合
状態・分岐・構造が一度に出てしまい、初学者には負荷が高い。  
また、HPACK の難しさを「圧縮だから難しい」と誤解しやすい。

### iterator / parser / validator の整理を前半に持ってくる場合
抽象論が先行し、読者が手触りを持ちにくい。  
具体例を見たあとで整理する方が理解しやすい。

### テストデータ作成を終盤に回す場合
前半の内容が受け身の理解に留まりやすく、後半で状態付き構造に入ったときに手元で確認する習慣が育たない。

### 実装との接続を早く出す場合
ライブラリ設計や API の議論に引っ張られ、本書の中心であるバイト列の読み方がぼやける。

---

## 読者体験として見た章順

この本の読者体験は、次のような変化を想定している。

### 1. 最初は「何が難しいのか」を知る
バイト列が難しい理由は、16進数そのものではなく、読み方の軸が見えないことだと理解する。

### 2. 次に「どこで切るか」を学ぶ
固定長と可変長の違いを通じて、境界を見る習慣を持つ。

### 3. 身近な題材で可変長を体験する
UTF-8 を通じて、先頭バイト・継続バイト・妥当性の話に慣れる。

### 4. 構造化された対象へ進む
HTTP/2 フレームで、共通ヘッダとペイロードを分解して読む感覚を得る。

### 5. 役割分担の言葉を手に入れる
iterator / parser / validator の違いを明確にする。

### 6. 自分の手で確かめる
最小テストデータを作り、理解を能動化する。

### 7. 状態という難所へ入る
HPACK で「入力だけでは意味が決まらない」構造に触れる。

### 8. 状態付き構造の設計を考える
state を持つ parser の見方を整理する。

### 9. 実装読解への橋を渡る
HTTP/2 ライブラリや上級編との接続を理解する。

### 10. 他の題材へ持ち運ぶ
学んだ見方を別のプロトコルやフォーマットへ広げる。

この体験の流れが、本書の章順そのものである。

## シリーズ内で見た章順の意味

本書単体の章順だけでなく、シリーズ全体との関係でもこの順番には意味がある。

### 入門書との接続
入門書で見たフレームや通信パターンを、「観察したもの」から「読めるもの」へ変換するために、本書の前半がある。

### 中級書との接続
中級書でライブラリ設計やクラス責務を読むために、本書の中盤で iterator / parser / validator / state を整理する。

### 上級書との接続
上級書で RST_STREAM / GOAWAY / Sans-IO や最小クライアントを扱う前に、本書後半で状態やテストデータの感覚を育てる。

つまり本書の章順は、シリーズ全体の橋渡し順でもある。

## 要約

本書の章順は、次の意図で設計されている。

### 1. 先に難しさの地図を与える
第1章で、境界・分岐・妥当性・状態という観点を提示する。

### 2. もっとも基礎的な境界問題から始める
第2章で、固定長と可変長を整理する。

### 3. 小さく学んでから大きく学ぶ
第3章の UTF-8 で可変長を学び、第4章の HTTP/2 フレームで構造へ進む。

### 4. 具体例のあとで抽象化する
第5章で iterator / parser / validator を整理する。

### 5. 受け身で終わらせない
第6章でテストデータ自作を導入する。

### 6. 後半で状態という難所に入る
第7章・第8章で HPACK と状態付き parser を扱う。

### 7. 最後に実装と一般化へ接続する
第9章・第10章でシリーズ全体と他分野へ橋を架ける。
