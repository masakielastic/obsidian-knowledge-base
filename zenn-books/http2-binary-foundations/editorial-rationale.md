# 編集方針メモ

## このファイルの役割

このファイルは、本書『バイナリハック入門 ― UTF-8・HTTP/2・HPACK で学ぶ構造と状態』の**編集判断の根拠**をまとめたメモである。

`binary-hacking-outline.md` が「読者に見せる本体の骨格」であるのに対し、このファイルは、

- なぜこの章順にしたのか
- どの章が何の橋渡しを担うのか
- どの抽象概念をどの段階で導入するのか
- 章順を変えると何が起きるのか

を整理するための設計ノートである。

したがって、このファイルの目的はアウトラインの再掲ではない。  
**アウトラインの背後にある編集上の判断を保存すること**が役割である。

---

## 本書の中心命題

本書は、バイナリデータを 16 進数の壁としてではなく、**境界・分岐・妥当性・状態**で読む本である。

この本で育てたいのは、UTF-8 や HTTP/2 や HPACK の個別知識そのものではない。  
それらを題材として、読者が次のような視点を獲得することである。

- どこで切るのか
- 何が次の解釈を決めるのか
- 何をもって正しいとするのか
- どこで過去の状態が現在の意味に効いてくるのか

つまり本書の主題は、仕様の網羅ではなく、**読む力の獲得**にある。

---

## 本書の章順を支える三つの原理

### 1. 先に「何が難しいのか」を言語化する

読者は、バイト列を見た瞬間に「難しい」と感じやすい。  
しかし、その難しさの内訳を分けずに具体例へ入ると、UTF-8 も HTTP/2 も HPACK も、単なる規則の羅列として受け取られやすい。

そこで本書では最初に、難しさの軸を次の四つに分けて示す。

- 境界
- 分岐
- 妥当性
- 状態

この地図を先に与えることで、後続の題材が別々の話ではなく、**同じ見方の反復**として見えるようになる。

### 2. 可変長 → 構造 → 状態 の順で認知負荷を上げる

本書の背骨は、次の三段階である。

1. 可変長
2. 構造
3. 状態

この順番には強い理由がある。  
状態付き構造は最も難しい。  
その前に、まず「どこまで読むか」が見え、次に「どうフィールドに分解するか」が見えていなければ、状態の議論は空中戦になりやすい。

そのため本書では、

- UTF-8 で可変長を学ぶ
- HTTP/2 フレームで構造を学ぶ
- HPACK で状態を学ぶ

という順で進める。

### 3. 抽象化は具体例のあとに置く

iterator / parser / validator のような整理軸は、本書にとって重要である。  
しかし、それをあまり早く出すと、読者には抽象論として映りやすい。

そこで本書では、先に UTF-8 と HTTP/2 フレームという具体例を見せ、そのあとで第5章において役割分担を整理する。  
この順番により、概念が手触りを持った状態で定着しやすくなる。

## 本書の編集上の階段

本書の流れは、次のような階段として設計している。

1. 難しさの正体を定義する
2. 固定長と可変長を区別する
3. UTF-8 で可変長を学ぶ
4. HTTP/2 フレームで構造を学ぶ
5. iterator / parser / validator で整理する
6. テストデータ自作で理解を手元化する
7. HPACK で状態を導入する
8. 状態付き parser を整理する
9. HTTP/2 実装への橋を架ける
10. 他の形式へ一般化する

この並びは、単なる章一覧ではなく、**読者体験の順序**である。

## 各章の配置理由

## 第1章 バイナリデータはなぜ難しく見えるのか

### この位置に置く理由
具体例へ入る前に、「何が難しいのか」を言語化する必要があるからである。  
ここを飛ばすと、以後の章が「新しい規則が増えていく本」に見えやすい。

### この章の編集上の役割
- 本書の射程を定義する
- 文字コード本でもプロトコル本でもないことを示す
- 後続で再利用する観点を先に与える

---

## 第2章 固定長と可変長

### この位置に置く理由
境界という問題を、もっとも基礎的な形で扱えるからである。  
UTF-8 も HTTP/2 フレームも、それぞれ別の形で「どこまで読むか」が問題になるため、その前提をここで整える。

### この章の編集上の役割
- 第1章で示した「境界」を最初に具体化する
- 後続の題材で概念が埋もれないように独立して扱う

---

## 第3章 UTF-8 は可変長データの教材である

### この位置に置く理由
可変長の最初の教材として、UTF-8 が小さく、身近で、学習効果が高いからである。  
先頭バイトが長さを決め、継続バイトが補助要素となり、妥当性も同時に見える。

### この章の編集上の役割
- 可変長の読み方を最小単位で学ばせる
- 文字コードの章ではなく、可変長の章として位置づける
- 妥当性の話題を自然に導入する

---

## 第4章 HTTP/2 フレームは構造化バイナリの教材である

### この位置に置く理由
UTF-8 で「どこまで読むか」を学んだあとに、HTTP/2 フレームで「どう分解するか」へ進むためである。  
9 バイト共通ヘッダと可変長ペイロードの組み合わせにより、固定長と可変長が同居する構造を読みやすい。

### この章の編集上の役割
- parser 的な見方を導入する
- 「単位」と「フィールド」の違いを見せる
- HTTP/2 の実装やログ読解への入口を作る

---

## 第5章 iterator / parser / validator で整理する

### この位置に置く理由
具体例を見たあとに、役割の違いを抽象化するためである。  
ここが早すぎると抽象論になり、遅すぎると後半で役割混同が起きやすい。

### この章の編集上の役割
- UTF-8 と HTTP/2 フレームに共通する整理軸を与える
- 実装設計や責務分担の前提を作る
- 中級書への橋を架ける

---

## 第6章 テストデータを自分で作る

### この位置に置く理由
ここで一度、受け身の理解を手元の実験へ変える必要があるからである。  
最小入力、壊れた入力、1 ケース 1 論点という視点をここで導入しておくと、後半の状態付き構造に入ったときの理解が安定する。

### この章の編集上の役割
- 能動的な理解へ切り替える
- encode / decode の往復を習慣化する
- HPACK 前の助走を作る

---

## 第7章 HPACK は状態を持つバイナリ構造である

### この位置に置く理由
ここで初めて「入力だけでは意味が決まらない」構造へ進むからである。  
この章より前に置くと、状態・分岐・構造が一度に出てきて負荷が高い。

### この章の編集上の役割
- 状態という新しい難しさを導入する
- HPACK の難しさを「圧縮」ではなく「状態」に置き直す
- 後続の stateful parser 論の素材を与える

---

## 第8章 state を持つ parser をどう考えるか

### この位置に置く理由
第7章で具体像を見たあとに、状態付き parser の設計原理を抽象化するためである。

### この章の編集上の役割
- 入力と履歴を分ける見方を導入する
- parser と state 管理の境界を整理する
- 上級編の Sans-I/O 的な分離へ進むための準備をする

---

## 第9章 バイナリハックから HTTP/2 実装へ

### この位置に置く理由
ここまで積み上げた見方を、実装読解へ接続するためである。  
実装論を早く出しすぎると、バイト列そのものを見る視点がぼやけるため、終盤に置く。

### この章の編集上の役割
- シリーズ全体との橋渡しを担う
- クラス責務や API 設計へ読者を接続する
- RST_STREAM / GOAWAY / Sans-I/O に入る前提を整える

## 第10章 他のプロトコルやフォーマットへ広げる

### この位置に置く理由
最後に、本書の学びを個別仕様に閉じないものとして回収するためである。

### この章の編集上の役割
- 本書の価値を汎用的な読み方として定義し直す
- HTTP/3 / QPACK や他形式への応用可能性を示す
- シリーズの先へ視野を広げる

## この順番でない場合に起きやすいこと

### HPACK を前半に持ってくる場合
状態・分岐・構造が一度に出てしまい、初学者には負荷が高い。  
また、難しさを「圧縮だから難しい」と誤解しやすい。

### iterator / parser / validator を前半に持ってくる場合
抽象論が先行し、読者が現物との対応を取りづらい。  
具体例のあとで整理した方が、役割の違いが手触りを持つ。

### テストデータ作成を終盤に回す場合
前半の理解が受け身のまま進みやすく、後半の状態付き構造で手元確認の習慣が不足する。

### 実装との接続を早く出す場合
ライブラリ設計や API の議論に読者の注意が引っ張られ、本書の核であるバイト列読解がぼやける。

## 読者体験として見た流れ

本書の読者体験は、次のような変化を想定している。

### 1. 最初は「何が難しいのか」を知る
バイト列が難しい理由は、16進数そのものではなく、読み方の軸が見えないことだと理解する。

### 2. 次に「どこで切るか」を学ぶ
固定長と可変長の違いを通じて、境界を見る習慣を持つ。

### 3. 身近な題材で可変長を体験する
UTF-8 を通じて、先頭バイト・継続バイト・妥当性に慣れる。

### 4. 構造化された対象へ進む
HTTP/2 フレームで、共通ヘッダとペイロードを分解して読む感覚を得る。

### 5. 役割分担の言葉を手に入れる
iterator / parser / validator の違いを明確にする。

### 6. 自分の手で確かめる
最小テストデータを作り、理解を能動化する。

### 7. 状態という難所へ入る
HPACK で「入力だけでは意味が決まらない」構造に触れる。

### 8. 状態付き構造の設計を考える
state を持つ parser の見方を整理する。

### 9. 実装読解への橋を渡る
HTTP/2 ライブラリや上級編との接続を理解する。

### 10. 他の題材へ持ち運ぶ
学んだ見方を別のプロトコルやフォーマットへ広げる。

## シリーズ内での位置づけ

本書は、シリーズ全体の中で**橋渡しの本**である。

### 入門書との接続
入門書で見たフレームや通信パターンを、「観察したもの」から「読めるもの」へ変換する。

### 中級書との接続
中級書でライブラリ設計やクラス責務を読むために、本書では iterator / parser / validator / state という整理軸を与える。

### 上級書との接続
上級書で RST_STREAM / GOAWAY / Sans-I/O や最小クライアントを扱う前に、本書で状態やテストデータの感覚を育てる。

したがって本書は、シリーズにおける寄り道ではない。  
**観察から読解へ、読解から設計へ進むための基礎工事**である。
