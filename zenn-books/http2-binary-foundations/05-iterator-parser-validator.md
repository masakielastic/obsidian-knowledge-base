## 第5章　iterator / parser / validator で整理する

### 導入対話

**初学者**

**解説者**

**実装者**

**初学者**  
「たしかに、UTF-8 の章では1文字ずつ進む感じがあって、HTTP/2 フレームの章ではフィールドへ分ける感じがありました」

**解説者**  
「その感覚は正しいです。本章では、その違いを言葉として固定します。そうすると、何を作ろうとしているのか、どこで責務が重くなっているのかが見えやすくなります」

**初学者**  
「つまり、この章は知識を増やすというより、今まで見たことの役割分担を整理する章なんですね」

**解説者**

---

### この章で扱うこと

ここまでの章では、UTF-8 と HTTP/2 フレームを題材として、バイト列を読むときに何が問題になるかを見てきた。  
UTF-8 では、先頭バイトを見て 1 文字の境界を見つけながら進んだ。  
HTTP/2 フレームでは、1 フレームを切り出し、その内部を共通ヘッダとペイロード、さらに各フィールドへ分解した。

この二つはどちらも「読む処理」に見える。  
しかし、同じ種類の仕事ではない。  
さらに、そこへ「正しいかどうかを判定する」という仕事が加わると、役割はもう一段増える。

本章の目的は、こうした違いを  
**iterator / parser / validator**  
という三つの語で整理することである。

この整理が必要なのは、用語を覚えるためではない。  
本当に重要なのは、**役割を混ぜないための視点**を持つことである。  
何を順にたどる仕事なのか。  
何を構造へ分解する仕事なのか。  
何を正しさとして判定する仕事なのか。  
この区別ができると、実装やテストや説明の見通しが大きく変わる。

---

### なぜ役割を分ける必要があるのか

バイト列を扱う処理は、ひとまとめに「デコード」と呼ばれがちである。  
しかし実際には、その中に複数の仕事が含まれている。

たとえば、UTF-8 の文字列を左から順にたどる処理を考える。  
ここでは主に、「今の位置から 1 文字分を取り出し、次の位置へ進む」ことが中心になる。  
一方で HTTP/2 フレームを読む処理では、「共通ヘッダを取り出し、Length や Type を読み、ペイロードを構造として分解する」ことが中心になる。  
さらに、その結果が仕様上許されるかどうかを判定する処理まで含めると、話はもう一段複雑になる。

これらを全部まとめて一つの仕事として考えると、問題が起きやすい。

まず、頭の中で何をしているのかが曖昧になる。  
次に、実装が重くなりやすい。  
境界を見つける処理と、構造を組み立てる処理と、エラー規則を適用する処理が一体化すると、修正もしにくくなる。  
さらに、テストを書くときにも、「何を確認しているテストなのか」がぼやける。

だから本章では、まず役割を切り分ける。  
この切り分けは、抽象論ではあるが、すでに前章までの具体例の上に立っている。  
つまり本章は、今まで見てきたことに名前を与える章である。

---

### iterator とは何か

iterator は、**順にたどるための役割**である。  
ここでの中心は、構造全体を高水準に理解することではない。  
今いる位置から、次の単位を取り出し、さらにその次へ進めるようにすることである。

この「次へ進む」という性質が重要である。  
iterator は、入力を左から右へ、あるいは何らかの規則に従って順に観察するための仕組みだと考えるとよい。

UTF-8 の章では、まさにこの感覚が中心にあった。  
今見ている位置から 1 文字分を見つける。  
その文字が 1 バイトなのか複数バイトなのかは先頭バイトで決まる。  
必要なだけ進んだら、次の文字の開始位置へ移る。  
この処理は、構造全体を複雑に組み立てるというより、**局所的な単位を順に取り出して前へ進む**ことに主眼がある。

iterator の仕事を素朴に言えば、  
「どこで次の単位が終わるかを知り、順番にたどれるようにすること」  
である。

ここでは、必ずしもその単位が仕様上正しいかどうかまで深く判定しなくてよい。  
また、その単位から高水準のオブジェクトを作らなくてもよい。  
まず大切なのは、**たどれること**そのものである。

---

### parser とは何か

parser は、**入力を構造へ分解する役割**である。  
iterator が順に進むことを中心にしていたのに対し、parser は「何がどのフィールドなのか」「どの部分がどの意味を持つのか」を組み立てることに重点がある。

HTTP/2 フレームの章では、この parser 的な見方が前面に出ていた。  
まず 1 フレームを切り出す。  
次に共通ヘッダを取り出す。  
さらにそのヘッダを Length、Type、Flags、Stream ID に分解する。  
そして Type に応じてペイロードの読み方を切り替える。  
これは単に順に進んでいるだけではない。  
**入力から意味のある構造を露出させている**。

parser の本質はここにある。  
バイト列は、そのままではただの並びに見える。  
parser は、それをフィールドや要素やノードのような、より説明しやすい構造に変える。  
その結果として、上位の処理は「生のバイト列」ではなく、「長さ」「種別」「識別子」のような形で対象を扱えるようになる。

言い換えると、parser は入力を**人間や上位レイヤーが扱いやすい単位へ持ち上げる**役割を担う。

ここで大事なのは、parser もまた境界や分岐を使うということである。  
ただし、その目的は「次へ進むこと」だけではない。  
**構造を明確にすること**が中心である。

---

### validator とは何か

validator は、**その入力や構造が正しいとみなせるかを判定する役割**である。

この役割は、iterator や parser と密接に関わるが、同じではない。  
たどることと、構造を作ることと、正しさを判定することは、似ていても別の問いに答えている。

validator が答える問いは、たとえば次のようなものである。

- この並びは UTF-8 として許されるか
    
- この Length と実際のデータ量は整合しているか
    
- この Type と Flags の組み合わせは妥当か
    
- この値の範囲は仕様上許されるか
    

ここで注目すべきなのは、validator が扱うのは**観察されたものに対する判断**だという点である。  
iterator は前へ進む。  
parser は構造へ分解する。  
validator は、それらによって得られた並びや構造が、仕様やルールに照らして妥当かを判定する。

この区別は実際には非常に重要である。  
「読めた」ことと「正しい」ことを同一視すると、実装も説明も混乱しやすい。  
前章までで何度か見たように、ある入力は何らかの形でたどれたり分解できたりしても、それが仕様上妥当とは限らない。  
validator は、その差を引き受ける役割である。

---

### iterator / parser / validator の違いを一文で言う

ここで三つの違いを、一度短く言い切っておく。

iterator は、**順にたどる**。  
parser は、**構造へ分解する**。  
validator は、**正しさを判定する**。

この三つは連続して現れることが多い。  
だから混ざりやすい。  
しかし、問いの種類は違う。

iterator が気にするのは、「次の単位はどこまでか」である。  
parser が気にするのは、「この単位はどのフィールドや要素に分かれるか」である。  
validator が気にするのは、「その並びや構造は許されるか」である。

この三つを頭の中で言い分けられるだけで、  
「今、自分は何を作ろうとしているのか」  
「今、何が足りないのか」  
がかなり明確になる。

---

### UTF-8 で見る iterator と validator

UTF-8 は、この三つの違いを見せるための小さな題材として便利である。

UTF-8 を 1 文字ずつ左から順にたどるとき、まず中心になるのは iterator 的な役割である。  
先頭バイトを見て、その文字が何バイト続くかを決める。  
必要な継続バイトのぶんだけ進み、次の文字の開始位置へ移る。  
ここでの主眼は、「順に文字をなめられること」にある。

一方で、その並びが UTF-8 として正しいかどうかを考え始めると、validator 的な役割が前面に出る。  
必要な継続バイトがあるか。  
継続バイトの形が妥当か。  
本来ありえない並びになっていないか。  
こうした問いは、「前へ進む」こととは別の仕事である。

このように UTF-8 では、iterator と validator の違いが比較的見えやすい。  
1文字ずつ進むだけなら、まず境界判定が中心になる。  
それに対して、「その並びを受け入れてよいか」を考えると、一段別の責務が立ち上がる。

本章では厳密な実装論に立ち入らないが、少なくとも頭の中では、  
**UTF-8 を順にたどる仕事**と  
**UTF-8 として妥当かを判定する仕事**  
を分けて考える癖をつけたほうがよい。

---

### HTTP/2 フレームで見る parser と validator

HTTP/2 フレームでは、parser 的な役割がさらに見えやすくなる。

1 フレームを取り出し、共通ヘッダを読み、Length、Type、Flags、Stream ID へ分解する。  
Type に応じてペイロードの読み方を切り替える。  
これはまさに parser の仕事である。  
ここでは入力から構造を露出させることが中心になっている。

そのうえで、さらに「このフレームは妥当か」と問うなら、validator の仕事が現れる。  
Length と実際のデータ量は合っているか。  
その Type に対してその Flags は許されるか。  
Stream ID の扱いに問題はないか。  
これらは、構造へ分解したあとに適用される判断である。

ここでも大事なのは、parser と validator を混同しないことである。  
フレームを構造へ分解できたことと、そのフレームが仕様上有効であることは別問題である。  
たとえばフィールドとしては読めても、その値の組み合わせが不正ということは起こりうる。

この差が見えていると、HTTP/2 を読む力はかなり安定する。  
バイト列を生のまま見るのではなく、まず構造を出す。  
そして、その構造に対して正当性の判断をかける。  
この順番が明確になるからである。

---

### parser は iterator を内側に含むことがある

ここで少しだけ注意しておきたいのは、実際の処理では parser が iterator 的な動きを内部で使うことがある、ということである。

たとえばバイト列を左から読み進めながらフィールドを切り出す parser では、内部的には位置を進める処理が必要になる。  
その意味では、parser の中に「順に進む」要素が含まれることは珍しくない。

しかし、それでも役割の中心は違う。  
iterator の中心は、順にたどれることにある。  
parser の中心は、構造を作ることにある。  
たとえ内部で同じようにカーソルを進めていたとしても、**何を目的にしているか**が違う。

この点を曖昧にすると、「全部 iterator ではないか」「全部 parser ではないか」という話になりやすい。  
だが本書では、動作の細部よりも、**責務として何を引き受けているか**を重視する。  
その観点で見ると、やはり三つは分けて考えたほうがよい。

---

### validator は parser の後にしか来ないのか

もう一つ注意点がある。  
validator はしばしば parser のあとに適用されるように見えるが、実際にはもっと局所的に働くこともある。

たとえば、あるフィールドを読んだ瞬間に「この値は範囲外である」と判定することもある。  
UTF-8 でも、継続バイトの形を見た時点で不正だと判断することがある。  
この意味では、validator は必ず「全部分解したあと」だけに現れるわけではない。

ただし、それでも問いの種類は変わらない。  
今やっているのが「順に進むこと」なのか、「構造を作ること」なのか、「許されるかを判定すること」なのかは、頭の中で区別できる。

本章の目的は、現実の実装を厳密に三層へ切り分けることではない。  
そうではなく、**責務を見分けるための言葉を持つこと**にある。  
処理がどれだけ密接に絡んでいても、何の役割を担っているのかを言い分けられることが重要である。

---

### 役割を混ぜると何が起きるか

iterator / parser / validator を分けて考える価値は、混ぜたときの問題を見るとよくわかる。

まず、関数やクラスの責務が重くなりやすい。  
境界判定、構造分解、妥当性判定、エラー処理、状態更新まで一つの場所に集まると、何を変えたときに何が壊れるのか見えにくくなる。

次に、テストの粒度が崩れやすい。  
あるテストが「次の単位を正しく進めること」を確認したいのか、  
「フィールドを正しく分解すること」を確認したいのか、  
「不正入力を拒否すること」を確認したいのか、  
目的が曖昧になる。

さらに、説明がしにくくなる。  
自分が何を作っているのかを言葉にできないと、レビューや設計議論でも混乱しやすい。

本書がこの章を置いているのは、この問題を早めに避けたいからである。  
後の章ではテストデータの設計や、状態付き構造である HPACK を扱う。  
そこへ進む前に、少なくとも  
「今の自分は iterator を考えているのか、parser を考えているのか、validator を考えているのか」  
を区別できるようにしておきたい。

---

### 実装設計の話に踏み込みすぎないための整理

ここで本章の立ち位置も確認しておきたい。  
iterator / parser / validator の話をすると、すぐに API 設計やクラス分割の話へ進みたくなる。  
実際、それらとは深く関係している。

ただし本章では、まだそこへ踏み込みすぎない。  
この本は、設計パターン集ではなく、バイト列の読み方を整える本だからである。

本章で必要なのは、実装の正解を決めることではない。  
まずは、**役割を区別するための視点を持つこと**である。  
設計の話は、その視点ができてからのほうが扱いやすい。  
中級書や後半の章で責務分担に触れるときにも、この章の整理が前提になる。

したがって、ここでは次の程度に留めておくのがよい。

- iterator は順にたどる役割
    
- parser は構造へ分解する役割
    
- validator は正しさを判定する役割
    
- 実際の処理では近接するが、責務としては分けて考えたほうがよい
    

この整理だけでも、後の見通しはかなり良くなる。

### 次章への橋――理解を手元に戻す

ここまでで、本書はかなり抽象的な整理に入った。  
第1章では難しさの軸を与え、第2章から第4章では固定長、可変長、UTF-8、HTTP/2 フレームを見てきた。  
そして本章で、役割の違いを iterator / parser / validator という言葉で整理した。

次章では、ここで得た整理を、**テストデータを自分で作る**という実践的な形へ戻す。  
この流れは重要である。  
抽象化したまま先へ進むのではなく、一度、自分の手で確かめられる最小入力へ戻る。

そのとき本章の整理が効いてくる。  
何をたどるテストなのか。  
何を分解するテストなのか。  
何を不正とみなすテストなのか。  
この区別があると、テストケースの意味がはっきりする。

つまり次章は、役割整理を実験可能な形に落とし直す章である。

### 整理ブロック

#### iterator / parser / validator の基本

| 役割 | 中心となる問い | 何をするか |
| --- | --- | --- |
| iterator | 次の単位はどこまでか | 順にたどる |
| parser | この入力はどんな構造か | 構造へ分解する |
| validator | この並びや構造は許されるか | 正しさを判定する |

#### UTF-8 と HTTP/2 フレームでの見え方

| 題材 | 前面に出やすい役割 | 典型的な関心 |
| --- | --- | --- |
| UTF-8 | iterator / validator | 1文字ずつ進む、継続バイトが妥当か |
| HTTP/2 フレーム | parser / validator | ヘッダ分解、Length や Type の整合性 |

#### 役割を混ぜると起きやすいこと

- 何をしている処理なのか説明しにくくなる
    
- 関数やクラスの責務が重くなる
    
- テストの目的が曖昧になる
    
- エラー処理の位置づけがぶれやすくなる
    

#### この章で押さえるべきこと

- iterator は順にたどる
    
- parser は構造へ分解する
    
- validator は正しさを判定する
    
- 実装上は近接していても、責務としては分けて考えたほうがよい
    
- この整理は後のテスト設計と状態付き構造の理解に効く
    

---

### この章で残すべきメンタルモデル

バイト列を扱う処理は一枚岩ではない。  
**順にたどること、構造へ分解すること、正しさを判定することは、別の役割である。**

---

## まとめ

- iterator は、入力を順にたどる役割であり、次の単位の境界を見つけながら前へ進む。
    
- parser は、入力をフィールドや要素へ分解し、構造として扱える形にする。
    
- validator は、その並びや構造が仕様上許されるかどうかを判定する。
    
- UTF-8 では iterator と validator の違いが見えやすく、HTTP/2 フレームでは parser と validator の違いが見えやすい。
    
- 実際の処理では三つが近接して現れても、責務としては分けて考えたほうが実装・説明・テストの見通しがよくなる。
    

## 次章への接続

次章では、ここまでの整理を踏まえて、  
**最小限のテストデータを自分で作る** という実践へ進む。  
読むことを理解したつもりで終わらせず、小さな入力を設計して確かめることで、境界・分岐・妥当性の感覚を手元に定着させていく。
