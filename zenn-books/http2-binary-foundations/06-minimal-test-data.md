
## 第6章　テストデータを自分で作る

### 導入対話

**初学者**  
「ここまで読んで、境界とか分岐とか、だいぶ整理できてきた気はします。でも、読むだけで本当に身につくのか少し不安です」

**解説者**  
「その不安は正しいです。理解したつもりになることと、自分の手で確かめられることは違います。だからこの章では、最小限のテストデータを自分で作る話をします」

**実装者**  
「実務でも、大きな実データだけ見ていると、どこが本質なのか見失いやすいです。小さい入力を自分で作れるようになると、境界や分岐や壊れ方が一気に見やすくなります」

**初学者**  
「テストデータというと、自動テスト用の大量データみたいなものを想像していました」

**解説者**  
「本章で扱うのは、その前段階です。まずは理解のための小さな入力です。  
1ケースで1論点だけを確かめる。  
その発想が重要です」

**初学者**  
「つまり、読む力を本当に自分のものにするには、入力を観察するだけでなく、自分で設計する必要があるんですね」

**解説者**  
「その通りです。この章では、UTF-8 と HTTP/2 フレームを題材に、最小入力、壊れた入力、encode / decode の往復という三つの観点から整理します」

---

### この章で扱うこと

ここまでの章で、本書は次のような階段を上ってきた。  
まずバイナリデータの難しさを、**境界・分岐・妥当性・状態**という観点で整理した。  
次に、固定長と可変長の違いを見た。  
さらに、UTF-8 では先頭情報で分岐する可変長データを、HTTP/2 フレームではフィールドへ分解する構造化バイナリを扱った。  
そして前章では、それらに関わる役割を iterator / parser / validator として整理した。

ただし、ここまでの理解は、まだ読む側の理解である。  
本章ではそこから一歩進み、**自分で入力を作る側**に回る。  
これが重要なのは、バイト列の理解は、観察だけでは曖昧なまま残りやすいからである。

巨大な実データや既存実装の出力だけを見ていると、そこには複数の論点が同時に入り込む。  
境界もある。分岐もある。妥当性もある。時には状態まで絡む。  
すると、どの規則を今確認しているのかが見えにくくなる。  
だから本章では、理解のためのテストデータは**小さく、単純で、論点が一つに絞られているべきだ**という立場を取る。

本章の主題は次の三つである。

- なぜ最小限の入力を自分で作るのか
    
- 正常系と壊れた入力をどう作り分けるのか
    
- encode / decode の往復がなぜ理解を支えるのか
    

この章の目的は、単に「テストを書くべきだ」と勧めることではない。  
**理解のためにどんな入力を作るべきかを言語化すること**にある。

---

### なぜ最小の入力を自作するのか

バイト列の理解において、最小の入力を自分で作ることには強い意味がある。

第一に、**何を確認したいのかが明確になる**。  
既存の巨大な入力を読むとき、そこには本質的でない要素が大量に含まれる。  
複数のフィールド、複数の分岐、補助的な値、場合によっては運よく通っているだけの部分まで混じる。  
その中から本当に確認したい論点を抜き出すのは難しい。

第二に、**境界が見やすくなる**。  
入力が大きいほど、どこで1単位が始まり、どこで終わるのかを追うのが難しくなる。  
しかし最小入力なら、どこを切るべきかがかなり明瞭になる。  
これは可変長データや構造化バイナリを学ぶうえで特に大きい。

第三に、**壊れ方を意図的に作れる**。  
既存の実データには、たいてい正常系しか含まれない。  
しかし理解にとって大切なのは、正しく動く例だけではない。  
どこで不正になるのか、何を validator が拒否すべきなのか、境界が崩れるとどうなるのかを知るには、壊れた入力を自分で作る必要がある。

第四に、**観察から設計へ移れる**。  
前章までで本書は、iterator / parser / validator という役割を整理した。  
自分でテストデータを作ると、この整理は抽象概念ではなくなる。  
どの入力が「順にたどれること」を確認しているのか。  
どの入力が「構造へ分解できること」を確認しているのか。  
どの入力が「不正を検出すること」を確認しているのか。  
その区別が手元のものになる。

最小入力の自作は、単なるテクニックではない。  
それは、**理解を受け身の観察から能動的な設計へ変える操作**である。

---

### 大きな実データが教えてくれないこと

実データを見ること自体は悪くない。  
むしろ、本番に近いデータには実際の複雑さがある。  
しかし学習の入口としては、実データだけに依存するのは危険である。

大きな実データには、たいてい複数の論点が同時に入っている。  
たとえば HTTP/2 の実際の通信ログを見れば、複数のフレームが並び、フラグの違いがあり、ストリームも複数あり、ペイロードもさまざまである。  
そこで何かを理解した気になっても、それが本当に Length の理解なのか、Type の理解なのか、単に実装の出力を眺めて納得しているだけなのかは曖昧になりやすい。

UTF-8 でも同じである。  
長いテキストを見れば、1バイト文字も複数バイト文字も入り、文字種も混ざる。  
その中で「UTF-8 を理解した」と言っても、実際には単に既知の文字列を見ているだけかもしれない。  
先頭バイトがどう分岐を作るのか、継続バイトがどこで必要になるのか、壊れた並びをどう見分けるのかは、長い入力だけでは案外つかみにくい。

つまり、実データは複雑さを見せてくれるが、**論点を分離してはくれない**。  
学習段階では、この分離を自分で行う必要がある。  
そのための方法が、最小入力の自作である。

---

### 1ケース1論点で設計する

本章で最も重要な原則を一つ挙げるなら、  
**1ケース1論点**  
である。

これは、1つのテストデータでは1つのことだけを確認する、という意味である。  
なるべく短く、なるべく単純にし、余計な要素を混ぜない。  
この原則は、テストの読みやすさのためだけでなく、理解のためにも重要である。

たとえば UTF-8 の最小例を作るなら、まず 1 バイト文字だけの例を作る。  
次に、複数バイト文字を 1 文字だけ持つ例を作る。  
さらに、継続バイトが足りない壊れた例を作る。  
ここで 1 つの入力の中に、1 バイト文字、2 バイト文字、3 バイト文字、さらに壊れた並びまで全部詰め込むと、何を見たいのかがぼやける。

HTTP/2 フレームでも同じである。  
Length の確認をしたいなら、極力単純なヘッダと最小限のペイロードでよい。  
Type による分岐を確認したいなら、他の変数はできるだけ固定する。  
壊れた長さの例を作るなら、今度はまさにその不整合だけを意図的に入れる。

1ケース1論点の原則は、前章の役割整理とも対応している。  
iterator を確認したいのか。  
parser を確認したいのか。  
validator を確認したいのか。  
この問いが明確であれば、余計な要素を入れたくなくなる。

理解のためのテストデータとは、網羅的で巨大なものではない。  
むしろ、**論点を孤立させた小さな入力**である。

---

### 正常系を最小化して作る

最初に作るべきなのは、正常系の最小入力である。  
ここでいう最小とは、仕様のあらゆる要素を削った結果として、なおその形式として成立する最小限の例である。

この発想はとても重要である。  
なぜなら、最小正常系を作ると、その形式にとって**絶対に必要なもの**が見えるからである。  
逆に言えば、そこに含まれないものは、少なくともその場面では本質ではない。

UTF-8 なら、まず 1 バイト文字 1 個の例が最小正常系の一つになる。  
この時点で、固定長のように見える最も単純なケースが確認できる。  
次に、複数バイト文字 1 個の例を作れば、先頭バイトと継続バイトの関係が露出する。  
ここでは、長い文字列や複雑な文章は不要である。  
むしろ邪魔になることさえある。

HTTP/2 フレームでも、最小正常系を考えると、共通ヘッダと最小限のペイロードだけを持つ例が有用になる。  
ここで必要なのは、ログらしい複雑な通信ではない。  
まず 1 フレームだけをきれいに切り出し、そのヘッダが示す Length と実際のペイロード長が整合していることが見えればよい。

最小正常系は、仕様の豊かさを見せるためのものではない。  
**構造の骨格を見せるためのもの**である。  
だからこそ、最初に作る価値がある。

---

### 壊れた入力を意図的に作る

正常系だけでは、読む力は完成しない。  
むしろ本書の観点から言えば、妥当性を本当に理解するには、**壊れた入力を自分で作ること**が欠かせない。

壊れた入力が重要なのは、それによって validator の仕事が初めて輪郭を持つからである。  
正常系だけを見ていると、iterator も parser も validator も、すべてが滑らかに動いてしまう。  
そのため、どこで何を判定しているのかが見えにくい。  
しかし不正入力を作ると、その境界が急に見え始める。

UTF-8 の場合なら、典型的には次のような壊し方がある。

- 継続バイトが足りない
    
- 継続バイトでないものが続いている
    
- 単独では開始になれない形のバイトが現れる
    

ここで大切なのは、壊す量を最小にすることである。  
複数箇所を同時に壊すと、どこで何が悪いのかがぼやける。  
1ケース1論点の原則は、不正入力でも同じである。

HTTP/2 フレームでも、不正の作り方はいろいろある。  
Length と実データ長をずらす。  
Type と Flags の整合を崩す。  
本来認められない値を入れる。  
ただしここでも、まずは一つの論点だけを壊すべきである。  
Length を確かめたいなら、他は正常にしておく。  
Flags を確かめたいなら、境界そのものは壊さない。

壊れた入力は、単なる失敗例ではない。  
それは、**何を妥当性として扱うかを明示するための入力**である。

---

### encode / decode の往復で理解する

本書では繰り返し、encode / decode の往復を重視している。  
この往復が大切なのは、バイト列をただ受け身で読む対象ではなく、**構造を持って組み立てられたもの**として見るためである。

decode だけをしていると、入力は最初から与えられたものに見える。  
そのため、「なぜこの位置にこの値があるのか」を十分に考えないまま読み進めてしまうことがある。  
しかし encode 側に立つと、どの情報をどこへ配置するか、長さをどこで決めるか、どのビット列が開始を示すかを意識せざるをえない。  
この視点が入ると、decode の見え方も変わる。

UTF-8 なら、ある文字が何バイトで表現されるのかを encode 側で考えることで、先頭バイトと継続バイトの役割がより明確になる。  
HTTP/2 フレームなら、Length をヘッダへ埋め込み、その後ろにペイロードを置くという構成を自分で考えてみると、共通ヘッダがなぜ先頭にあるのか、なぜまず 9 バイトを読むのかが腑に落ちやすい。

本章で言いたいのは、完全なエンコーダを実装しろということではない。  
そうではなく、**入力は誰かがそうなるように作ったものだ**という視点を持つことが大事だということである。  
encode / decode の往復を意識すると、バイト列は偶然の並びではなく、設計された構造として見えてくる。

---

### UTF-8 で作る最小テストデータ

ここで、UTF-8 を例に、どのような最小テストデータが理解に役立つかを整理してみたい。  
重要なのは具体的な文字そのものよりも、確認したい論点である。

最初の正常系としては、1 バイト文字 1 個の入力がよい。  
これは、もっとも単純なケースであり、UTF-8 の中でも固定長のように見える局面を確認できる。  
次に、複数バイト文字 1 個の入力を作る。  
ここで初めて、先頭バイトが長さを決め、継続バイトが後続するという構造が明確になる。  
さらに、1 バイト文字と複数バイト文字を短く並べた入力を作ると、「次の文字の開始位置へ進む」感覚が見えやすい。

不正入力としては、継続バイト不足の例がわかりやすい。  
また、継続バイトであるべき位置に不適切なバイトを置く例も有効である。  
これらは validator の観点を明確にする。  
どこで「たどること」と「正しいとみなすこと」が分かれるのかが見えるからである。

UTF-8 でのテストデータ設計は、そのまま可変長一般の訓練になる。  
どの先頭情報が長さを決めるのか。  
その後ろに何がどれだけ続くのか。  
何が不正な継続になるのか。  
これらを最小入力で確認できれば、可変長データの読み方はかなり安定する。

---

### HTTP/2 フレームで作る最小テストデータ

HTTP/2 フレームでは、今度は構造化バイナリとしての最小テストデータを考える。  
ここで最初に作るべきなのは、**共通ヘッダとごく小さなペイロードからなる1フレーム**である。

このとき大切なのは、フレーム種別の豊かさを見せることではない。  
まず見たいのは、次のような骨格である。

- 先頭の 9 バイトが共通ヘッダであること
    
- Length がペイロード長を決めること
    
- Type が後続の読み方の分岐点になること
    
- 共通ヘッダのあとに可変長ペイロードが続くこと
    

つまり、まずは**1 フレームの切り出しと分解**だけが見えればよい。  
複数フレームの連続や複数ストリームは、この段階では不要である。  
むしろ 1 フレームだけのほうが、構造の骨格が見えやすい。

不正入力としては、Length と実際のペイロード長が合わない例が特に有効である。  
これは境界の誤りがどこで問題化するかをはっきり見せる。  
また、Type と Flags の組み合わせに不整合を作る例も、validator の仕事を意識させる。

HTTP/2 フレームの最小テストデータは、第4章で扱った parser 的な見方を定着させる。  
入力を 1 フレームへ切り出し、共通ヘッダをフィールドへ分解し、そのうえで妥当性を考える。  
この流れが、小さな入力で安定して確認できることが重要である。

---

### テストデータは仕様の要約である

最小テストデータを作っていると気づくのは、テストデータが単なる検証用材料ではないということである。  
それはしばしば、**その形式の仕様をどのように理解しているかの要約**になる。

どのケースを正常系として最初に置くか。  
どの不正を「まず確認すべき壊れ方」とみなすか。  
何を1ケースに入れ、何を別ケースへ分けるか。  
こうした判断には、その形式の本質をどこに見ているかが表れる。

たとえば UTF-8 で、先頭バイトと継続バイトの関係を最初に試すなら、その人は UTF-8 を可変長データとして理解している。  
HTTP/2 フレームで、最初に共通ヘッダと Length の整合を確かめるなら、その人はまず境界と構造に注目している。  
逆に、いきなり巨大な複合ケースを持ってくるなら、まだ論点の分離が弱いかもしれない。

この意味で、テストデータ設計は仕様理解の副産物ではない。  
むしろ、**仕様理解を外に出して確かめる行為**である。  
本章が重視するのは、まさにこの点である。

---

### テストは実装のためだけでなく、理解のためにもある

一般にテストという言葉は、実装の品質保証と結びつけて語られやすい。  
もちろんそれは重要である。  
しかし本書では、それより一段手前の役割を強調したい。  
テストは、**理解を確かなものにするための装置**でもある。

実装を書く前でも、小さな入力を自分で設計してみることはできる。  
あるいは、既存実装にその入力を通して、何が起きるかを見ることもできる。  
ここで得られるのは、正解したかどうかだけではない。  
自分がどこを理解し、どこをまだ理解していないかの輪郭である。

たとえば、最小正常系は作れるのに壊れた入力がうまく作れないなら、妥当性の理解がまだ浅いのかもしれない。  
Length の不整合は作れるのに、Type による分岐の最小ケースが作れないなら、構造の理解がまだ曖昧なのかもしれない。  
このように、テストデータ設計そのものが理解の診断になる。

だから本章では、「後で実装するならテストを書こう」とは言わない。  
それよりも先に、**理解したいなら、最小入力を自分で作ろう**と言いたいのである。

---

### 次章への橋――状態が入る前に手元の感覚を固める

この章が本書のこの位置に置かれているのには理由がある。  
次章からは HPACK を通じて、**状態を持つバイナリ構造**に入っていく。  
そこでは、入力だけでは意味が決まらず、過去の履歴や更新が解釈に効いてくる。  
難しさの層が一段上がる。

その前に、本章で一度、理解を手元へ戻しておく必要がある。  
最小入力を作り、正常系と不正系を分け、encode / decode の往復で構造を確認する。  
この習慣がないまま状態付き構造へ進むと、複雑さに押し流されやすい。

言い換えると、本章は後半への助走である。  
ここで身につけるべきなのは、単なるテスト技法ではない。  
**複雑な対象に入る前に、論点を切り出して小さく確かめる姿勢**である。  
この姿勢が、次章以降で HPACK の状態性を読むときの土台になる。

### 整理ブロック

#### この章の中心原則

| 原則 | 意味 |
| --- | --- |
| 最小入力を作る | 構造の骨格だけを見えるようにする |
| 1ケース1論点 | 何を確認しているのかを明確にする |
| 正常系と不正系を分ける | validator の役割を見えるようにする |
| encode / decode を往復する | 入力を設計された構造として理解する |

#### UTF-8 で作る最小ケースの例

| 目的 | 例の性格 |
| --- | --- |
| 最小正常系 | 1 バイト文字 1 個 |
| 可変長の確認 | 複数バイト文字 1 個 |
| 順にたどる確認 | 短い混在列 |
| 妥当性確認 | 継続バイト不足、不正な継続位置 |

#### HTTP/2 フレームで作る最小ケースの例

| 目的 | 例の性格 |
| --- | --- |
| 最小正常系 | 共通ヘッダ＋最小限のペイロードを持つ 1 フレーム |
| 境界確認 | Length と実データ長の対応を見る |
| 分岐確認 | Type による読み方の違いを単純化して見る |
| 妥当性確認 | Length 不整合、Type と Flags の不整合 |

#### この章で押さえるべきこと

- 大きな実データは複数論点を同時に含む
    
- 理解のためには最小入力を自作するほうがよい
    
- 1ケース1論点が基本である
    
- 壊れた入力は validator の役割を見せる
    
- テストデータ設計は仕様理解の要約でもある
    

---

### この章で残すべきメンタルモデル

理解のためのテストデータは、  
**大きくて網羅的なものではなく、論点を一つだけ露出させる小さな入力**である。

## まとめ

- 最小のテストデータを自分で作ると、何を確認したいのかが明確になる。
    
- 大きな実データは複数の論点を同時に含むため、学習の入口としては不向きなことがある。
    
- 1ケース1論点の原則で、正常系と不正系を分けて作ることが重要である。
    
- UTF-8 では先頭バイトと継続バイト、HTTP/2 フレームでは共通ヘッダと Length / Type が最小ケースの中心になる。
    
- encode / decode の往復を意識すると、バイト列を設計された構造として理解しやすくなる。
