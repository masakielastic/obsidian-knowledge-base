## 第4章　HTTP/2 フレームは構造化バイナリの教材である

### 導入対話

**初学者**  
「UTF-8 は、1文字がどこまで続くかを見る教材だという話でした。HTTP/2 になると、今度は何を見ることになるんでしょうか」

**解説者**  
「第4章では、視点が少し変わります。UTF-8 では主に『1単位をどう切るか』を見ましたが、HTTP/2 フレームでは、**切ったあとにどう分解するか**が主題になります」

**実装者**  
「ログやダンプを見るときも、HTTP/2 はそこが重要です。ただバイト列を眺めるのではなく、共通ヘッダとペイロードに分け、さらにヘッダの各フィールドを読む必要があります」

**初学者**  
「つまり、UTF-8 は1文字単位の可変長、HTTP/2 フレームはもっと大きな構造を持つ単位ということですね」

**解説者**  
「その通りです。HTTP/2 フレームは、固定長の共通ヘッダと可変長のペイロードが組み合わさった構造です。だから、固定長と可変長がどう共存するかを見る教材としても優れています」

**初学者**  
「UTF-8 の章よりも、少し parser っぽくなりそうですね」

**解説者**  
「はい。この章では、バイト列をフィールドへ分解する見方を作ります。次の章で iterator / parser / validator を整理する前段として、とても大事な位置にあります」

---

### この章で扱うこと

前章では UTF-8 を題材に、先頭情報で分岐する可変長データを見た。  
そこでは主に、1文字がどこまで続くかを判定するという、局所的な境界の問題が中心だった。

この章で扱う HTTP/2 フレームは、そこから一段進んだ対象である。  
HTTP/2 フレームは、単に「何バイト続くか」を知れば終わりではない。  
**1つのフレームという単位を取り出したあと、その内部を複数のフィールドへ分解して読む必要がある。**

つまり本章の主題は、可変長そのものではなく、**構造化バイナリ**である。  
HTTP/2 フレームは、共通ヘッダとペイロードからなり、しかも共通ヘッダは固定長、ペイロードは可変長という形を持つ。  
したがって本章では、次の三つを中心に見ることになる。

- フレームという単位をどう切り出すか
    
- 共通ヘッダをどのようにフィールドへ分解するか
    
- 固定長ヘッダと可変長ペイロードがどう組み合わさるか
    

この章の目的は、HTTP/2 の全仕様を覚えることではない。  
読むべきなのは、**バイト列をフィールドへ分解する parser 的な見方**である。

---

### HTTP/2 フレームを「通信の話」としてだけ見ると見落とすもの

HTTP/2 というと、多くの読者はまず通信の流れを思い浮かべる。  
ストリーム、HEADERS、DATA、優先度、フロー制御、そうしたプロトコル上のふるまいである。  
もちろん、それらは HTTP/2 を理解するうえで重要である。

しかし本章では、その前にもう少し低い層を見たい。  
HTTP/2 の動きがどれほど複雑でも、実際に流れる単位はフレームである。  
そしてそのフレームは、結局のところバイト列である。

ここで注目したいのは、HTTP/2 フレームが**構造を持ったバイト列**だということである。  
ただ並んでいるのではない。  
先頭には一定幅の共通ヘッダがあり、その中に長さ、種別、フラグ、ストリーム識別子といった役割の違うフィールドが並んでいる。  
さらにその後ろには、種別ごとに意味が変わる可変長ペイロードが続く。

この見方を持つと、HTTP/2 フレームは単なる通信プロトコルの部品ではなくなる。  
それは、**固定長と可変長が組み合わさった構造化バイナリの教材**として見えてくる。

---

### フレームという単位をまず見る

HTTP/2 を読むとき、最初に考えるべきなのは「このフレームは何を意味するか」ではない。  
その前に必要なのは、「**どこまでが1フレームか**」である。

これは第1章以来繰り返していることだが、意味の前に境界がある。  
HTTP/2 でも同じである。  
バイト列の流れの中から、まず 1 つのフレームを取り出さなければ、その中を読むことはできない。

ここで HTTP/2 フレームが教材として優れているのは、境界の決め方が比較的明快だからである。  
共通ヘッダは固定長であり、その中にペイロード長が書かれている。  
したがって読み手は、まず固定長のヘッダを取り出し、そのヘッダが示す長さだけペイロードを読めば、1 フレーム全体の終わりを知ることができる。

これは、前章で見た二つの型が組み合わさった例である。  
共通ヘッダ自体は固定長である。  
しかしフレーム全体としては、ヘッダの中に書かれた長さによって可変長ペイロードの終わりが決まる。  
つまり HTTP/2 フレームは、**固定長ヘッダ＋長さ先行型の可変長本体**という構造を持つ。

このことが見えると、前章で整理した固定長と可変長の議論が、HTTP/2 にそのままつながっているとわかる。

---

### 9バイト共通ヘッダという設計

HTTP/2 フレームの読み方で最初に重要なのは、先頭にある**9バイトの共通ヘッダ**である。

この共通ヘッダは、HTTP/2 の各種フレームに共通する土台であり、フレームを読む入口でもある。  
読み手はまずこの 9 バイトを一塊として取り出す。  
この時点で、UTF-8 のような「先頭バイトを見て1文字の長さを知る」という読み方とは少し性格が異なる。  
ここでは、まず固定長のヘッダ全体を取り出し、その中をさらに分解することになるからである。

この差は重要である。  
UTF-8 では、1文字を切るために先頭バイトを見た。  
HTTP/2 フレームでは、まず 9 バイトの共通ヘッダを取り出し、そのあとで「このフレームの全長はどれだけか」「この後ろをどの規則で読むか」を判断する。  
つまり HTTP/2 では、**単位の切り出しと、ヘッダ内部のフィールド分解が段階的に行われる**。

この 9 バイト共通ヘッダがあることで、HTTP/2 フレームは構造としてとても読みやすくなる。  
共通部分が先頭にまとまっているからである。  
バイト列を読む側から見ると、毎回まず同じ型の土台に出会える。  
この安定性が、構造化バイナリとしての読みやすさを支えている。

---

### Length / Type / Flags / Stream ID を読む

共通ヘッダを読むとは、単に 9 バイトを眺めることではない。  
その 9 バイトを、さらに**役割の異なるフィールドへ分解する**ことである。

HTTP/2 の共通ヘッダでは、代表的には Length、Type、Flags、Stream ID というフィールドが並ぶ。  
ここで重要なのは、それぞれのフィールドが同じ種類の情報ではないという点である。

Length は、このフレームのペイロードが何バイトあるかを示す。  
つまり、境界を決める情報である。  
Type は、このフレームがどの種別であるかを示す。  
これは後続のペイロードをどの規則で読むかを決める分岐点になる。  
Flags は、そのフレーム種別の読み方に対して追加の条件や状態を与える。  
Stream ID は、そのフレームがどのストリームに属するかを示す識別情報である。

このように見ると、共通ヘッダは単なる前置きではない。  
それは、**この後ろをどう読むかを決めるための制御情報のまとまり**である。

ここで大事なのは、フィールド分解の感覚である。  
バイト列をそのまま意味づけるのではなく、まず役割ごとに切り分ける。  
どの部分が長さで、どの部分が種別で、どの部分が補助条件で、どの部分が識別子なのか。  
この切り分けをしたうえで、はじめて後続の解釈に進める。

これが、HTTP/2 フレームを通じて学ぶ parser 的な読み方の中心である。

---

### 固定長ヘッダと可変長ペイロード

HTTP/2 フレームを構造化バイナリの教材として見るとき、特に重要なのが、**固定長ヘッダと可変長ペイロードの組み合わせ**である。

共通ヘッダは 9 バイト固定である。  
したがって、その部分は毎回同じ位置関係で読める。  
一方でペイロードの長さはフレームごとに異なり、その終わりは Length フィールドによって決まる。

この構造には、読み方の順序がはっきり表れている。

まず固定長の共通ヘッダを読む。  
次に、Length を見てペイロードの長さを知る。  
さらに、Type を見てペイロードの読み方を決める。  
そのうえで、必要なだけペイロードを読み取る。

ここでは、境界と分岐が別のフィールドとして現れている。  
UTF-8 では、先頭バイトが長さの型も分岐も兼ねていた。  
しかし HTTP/2 フレームでは、Length と Type が明示的に分かれている。  
この違いによって、構造がより見通しよくなる。

つまり HTTP/2 フレームは、可変長の問題を抱えつつも、それを固定長ヘッダで整理している。  
読み手にとっては、可変長の難しさが完全に消えるわけではないが、  
**どこを見れば境界がわかり、どこを見れば分岐がわかるか**が明確になる。

このことが、HTTP/2 フレームを「構造化バイナリの教材」と呼ぶ理由である。

---

### UTF-8 と HTTP/2 フレームの共通点と違い

ここで、第3章の UTF-8 と比較しておくと、本章の位置づけがよりはっきりする。

両者の共通点は、どちらもバイト列をそのまま読むのではなく、**境界と分岐を見ながら解釈する対象**だということである。  
どちらも、ただ左から数字を眺めるだけでは読めない。  
何が1単位か、何が後続の解釈を決めるかを見つける必要がある。

ただし、構造の現れ方はかなり違う。

UTF-8 では、主に 1 文字という局所単位の境界が問題になった。  
先頭バイトを見て、その文字がどこまで続くかを判定した。  
そこでは「1文字をなめる」ような iterator 的な感覚が中心だった。

HTTP/2 フレームでは、まず 1 フレーム全体を切り出し、その内部をフィールドへ分解する。  
つまりここでは、単位の境界だけでなく、**単位の内部構造**が前面に出てくる。  
その意味で、HTTP/2 フレームは UTF-8 よりも parser 的である。

この違いを整理すると、次のようになる。

UTF-8 は、先頭情報で分岐する可変長単位の教材である。  
HTTP/2 フレームは、固定長ヘッダと可変長本体を持ち、しかもヘッダをフィールドへ分解して読む教材である。

この差が見えると、本書の流れが理解しやすくなる。  
第3章から第4章への移行は、題材が変わっただけではない。  
**局所的な単位の読解から、構造化された単位の読解へ**と、一段抽象度が上がっている。

---

### decode とは、フィールドへ分解すること

HTTP/2 フレームを読むときに重要なのは、decode という言葉をどう捉えるかである。

ここでの decode は、何か魔法のような変換ではない。  
本書の文脈では、まず **バイト列を構造へ分解すること** を指す。

たとえば、あるフレームの先頭 9 バイトを取り出し、それを Length、Type、Flags、Stream ID に分ける。  
さらに Length に従ってペイロードを取り出し、Type に応じてその中を読む。  
この一連の作業は、まさに decode である。

重要なのは、decode が「意味を与えること」である前に、「構造を露出させること」だという点である。  
フレームの生のバイト列を、役割の見える形に切り分ける。  
どの部分が制御情報で、どの部分が本体なのかを明らかにする。  
この作業なしには、HTTP/2 フレームはただの 16 進数列のままである。

この意味で、HTTP/2 フレームは encode / decode の往復を考える題材としてもよい。  
Length をどのようにヘッダに埋め込むか。  
Type や Flags をどう配置するか。  
そして decode 側では、それをどの順に取り出して読むか。  
この往復を意識することで、バイト列が単なる観察対象ではなく、**構造を持って組み立てられたもの**として見えてくる。

---

### parser 的な見方とは何か

この章で育てたいのは、HTTP/2 の個別仕様知識よりも、**parser 的な見方**である。

parser 的な見方では、対象を次のように扱う。

まず入力から1単位を切り出す。  
次に、その単位をフィールドへ分解する。  
さらに、種別や長さなどの制御情報をもとに、後続の読み方を決める。  
必要なら、そこからより高水準の構造へ持ち上げる。

この一連の流れは、UTF-8 の章で見た iterator 的な見方より一段複雑である。  
UTF-8 では、「今ここから1文字を取る」が主な仕事だった。  
HTTP/2 フレームでは、「1フレームを切り出し、その中のフィールドを読み分け、その種別に応じて本体を解釈する」という複数段階の仕事になる。

だからこそ、HTTP/2 フレームは parser 的な思考を学ぶ教材になる。  
しかも、いきなり HPACK のような状態付き構造へ進む前に、  
まだ状態を本格導入せずに、構造化バイナリをどう読むかだけに集中できる。  
この位置づけが重要である。

本章の段階では、まだ iterator / parser / validator の概念整理は完成しない。  
それは次章の役割である。  
しかし少なくともここでは、  
**HTTP/2 フレームは「バイト列をフィールドへ分解する」対象であり、その読み方は UTF-8 より parser に近い**  
という感覚を持てればよい。

---

### 妥当性はどこに現れるか

本章の中心は構造と分解だが、妥当性の観点もここで消えるわけではない。

HTTP/2 フレームでも、読み手はただ分解すれば終わりではない。  
Length が実際のペイロード長と整合しているか、Type と Flags の組み合わせが妥当か、Stream ID の使い方に問題がないか、といった問いが出てくる。  
ただし本章では、それらの細かい規則に深入りしない。

ここで押さえておきたいのは、妥当性の問題が**構造のあとに現れる**ことが多いという点である。  
まずフレームを切り出し、フィールドへ分解する。  
そのうえで、その値の並びが許されるかどうかを判定する。  
この順番が見えていることが大事である。

つまり HTTP/2 フレームでも、  
「どこまでが1単位か」を決める仕事、  
「どのフィールドに分かれるか」を露出させる仕事、  
「その内容が妥当か」を判定する仕事は、頭の中で分けておいたほうがよい。  
この整理が、次章の概念整理につながっていく。

---

### 次の章への橋――役割の違いを言語化する

ここまでで、UTF-8 と HTTP/2 フレームという二つの題材を見てきた。  
UTF-8 では、1文字ずつたどる感覚が前面に出た。  
HTTP/2 フレームでは、フィールドへ分解する感覚が前面に出た。  
さらに、どちらにも妥当性の論点があることも見えてきた。

次章では、こうした違いを**iterator / parser / validator**という言葉で整理する。  
つまり、今まで感覚的に見てきた役割の違いを、概念として言い分けられるようにする。

この順番が重要である。  
もし第5章の概念整理を先に置くと、読者は抽象論として受け取りやすい。  
しかし本章まで読むと、すでに具体例が手元にある。  
UTF-8 で見た「1文字ずつたどる仕事」と、HTTP/2 フレームで見た「フィールドへ分解する仕事」は、違う役割として実感できるはずである。

その実感を言語化するのが次章の役割である。

## まとめ

- HTTP/2 フレームは、通信の話としてだけでなく、構造化バイナリの教材として読める。
    
- 1フレームを読むには、まず固定長の共通ヘッダを取り出し、その後ろの可変長ペイロードを切り出す必要がある。
    
- 共通ヘッダは Length / Type / Flags / Stream ID などの役割の異なるフィールドへ分解して読む。
    
- HTTP/2 フレームは、固定長ヘッダと長さ先行型の可変長本体が組み合わさった構造を持つ。
    
- UTF-8 が局所的な単位の境界を学ぶ教材だったのに対し、HTTP/2 フレームはフィールド分解を学ぶ parser 的な教材である。

**iterator / parser / validator** の違いを整理する。
