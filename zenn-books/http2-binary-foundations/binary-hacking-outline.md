# バイナリハック入門 アウトライン

## 本書の狙い

本書は、バイナリデータを 16 進数の壁としてではなく、**境界・分岐・妥当性・状態**という観点から読むための本である。  
UTF-8、HTTP/2 フレーム、HPACK を題材として、バイト列の読み方を段階的に学ぶ。

本書を通じて、読者は次の力を身につけることを目指す。

- バイト列を見たときに、まず境界を探す発想を持つ
- 先頭情報による分岐を読めるようになる
- encode / decode の往復で構造を確認できる
- iterator / parser / validator の違いを整理できる
- state を持つ構造がなぜ難しいかを説明できる
- 自分で最小限のテストデータを作る姿勢を身につける

本書の背骨は、次の三段階である。

1. 可変長
2. 構造
3. 状態

この三段階を小さな題材から順に積み上げることで、個別仕様の暗記ではなく、他のフォーマットやプロトコルにも持ち運べる読み方を身につける。

## 章配列の意図

この本の章配列は、単に話題を並べたものではない。  
理解の負荷が少ない順に、次のような階段を作っている。

1. 難しさの正体を言語化する
2. 固定長と可変長を区別する
3. UTF-8 で可変長を学ぶ
4. HTTP/2 フレームで構造を学ぶ
5. iterator / parser / validator で概念整理する
6. テストデータ自作で理解を手元化する
7. HPACK で state を導入する
8. 状態付き parser の考え方へ進む
9. HTTP/2 実装との橋を架ける
10. 他分野への応用可能性を示す

つまり本書は、**可変長 → 構造 → 状態**という順で進みながら、最後にそれを実装読解へ接続する構成になっている。

## この順番にした理由

本書では、最初に「何が難しいのか」を言語化する。  
バイト列の難しさは、16進数そのものにあるのではなく、どこで切るのか、何が次の解釈を決めるのか、何をもって正しいとするのか、どこで過去の状態が効いてくるのか、という読み方の軸が見えていないことにある。

そのうえで、**可変長 → 構造 → 状態**の順に進む。  
状態付き構造はもっとも難しいため、その前に「どこまで読むか」「どうフィールドに分解するか」を先に身につける必要がある。  
そのため本書では、UTF-8 で可変長を学び、HTTP/2 フレームで構造を学び、HPACK で状態を学ぶ。

また、抽象論は先に置かず、具体例のあとで整理する。  
iterator / parser / validator という整理軸は重要だが、先に出すと抽象的に見えやすい。  
そこで UTF-8 と HTTP/2 フレームを一度見たあとで、第5章で整理する構成にしている。

さらに、第6章のテストデータ自作は、後半への助走として置いている。  
HPACK のような状態付き構造に入る前に、自分で最小入力や壊れた入力を設計する視点を持っておくことで、後半の理解が安定する。

---

## この順番でない場合に起きやすいこと

HPACK を前半に持ってくると、状態・分岐・構造が一度に現れ、初学者には負荷が高い。  
また、難しさを「圧縮だから難しい」と誤解しやすい。

iterator / parser / validator の整理を前半に持ってくると、抽象論が先行しやすい。  
本書では具体例を見たあとに整理することで、概念を手触りと結びつける。

テストデータ作成を終盤に回すと、前半が受け身の理解に留まりやすい。  
本書では第6章で一度、理解を自分の手に戻してから後半へ進む。

実装との接続を早く出しすぎると、ライブラリ設計や API の議論に意識が引っ張られ、本書の中心であるバイト列の読み方がぼやけやすい。  
そのため第9章で橋渡しとして扱う。
