## 第7章　HPACK は状態を持つバイナリ構造である

### 導入対話

**初学者**  
「ここまでは、境界とか分岐とか妥当性を見ながら読めばよかったですよね。でも HPACK になると、急に難しく感じます」

**解説者**  
「それは自然です。HPACK では、ここまでの章で扱ってきた難しさに加えて、**状態**が入ってくるからです」

**実装者**  
「実務でも、HPACK が難しいのは『圧縮だから』と説明されがちですが、それだけでは足りません。つまずきやすいのは、同じ入力でも前の履歴によって意味が変わるところです」

**初学者**  
「同じ入力でも意味が変わる、というのはかなり怖いですね。今ここにあるバイト列だけ見ても足りないんですか」

**解説者**  
「はい。そこがこの章の中心です。UTF-8 や HTTP/2 フレームでは、主に今見ている入力から境界や構造を決めました。HPACK では、それに加えて『これまでに何が起きたか』が必要になります」

**初学者**  
「つまり、ここで初めて本格的に『状態』を扱うわけですね」

**解説者**  
「その通りです。この章では、HPACK を圧縮方式としてだけではなく、**状態を持つバイナリ構造**として読み直します」

---

### この章で扱うこと

ここまでの章では、バイト列の難しさを段階的に分けてきた。  
第1章では、難しさを**境界・分岐・妥当性・状態**の四つに整理した。  
第2章では固定長と可変長の違いを見た。  
第3章では UTF-8 を、先頭情報で分岐する可変長データとして読んだ。  
第4章では HTTP/2 フレームを、固定長ヘッダと可変長ペイロードを持つ構造化バイナリとして扱った。  
第5章では iterator / parser / validator を区別し、第6章では最小テストデータを自分で作る姿勢を整理した。

この章では、そこへもう一段、新しい難しさが入る。  
それが**状態**である。

本章の主題は、HPACK を単なる「ヘッダ圧縮方式」としてではなく、**状態を持つバイナリ構造**として読むことである。  
ここで重要なのは、圧縮率の話や実装最適化の話ではない。  
見るべきなのは、次のような問いである。

- なぜ HPACK は入力だけでは意味が決まらないのか
    
- どこに分岐があり、どこで状態参照が起きるのか
    
- dynamic table は何を変えているのか
    
- なぜ HPACK の難しさは「圧縮」よりも「状態」にあるのか
    

この章の目的は、HPACK の全仕様を網羅することではない。  
むしろその逆である。  
個々の規則を羅列するのではなく、**何が新しい難しさとして入ってきたのか**を見える形にすることが主眼である。

---

### HPACK を「圧縮方式」とだけ見ると見落とすもの

HPACK は一般に、HTTP/2 のヘッダ圧縮方式として説明される。  
この説明自体は間違っていない。  
ヘッダフィールドをそのまま毎回送るのではなく、繰り返しを減らし、表現を小さくする仕組みだからである。

しかし本書の観点から見ると、その説明だけでは重要なことを見落としやすい。  
なぜなら、HPACK を難しくしている中心は、単に「短く表現する工夫」ではないからである。  
本当に難しいのは、**現在の入力だけでは解釈が完結しない**ことである。

たとえば、バイト列のある並びが「どのヘッダフィールドを表しているか」を知るためには、その場のビット列だけでなく、すでに構築されているテーブルの内容が必要になる場合がある。  
この時点で、問題は圧縮率ではなくなる。  
読者にとって重要なのは、「なぜ短くなるか」よりも、「なぜ今ここにあるバイト列だけでは読めないのか」である。

本書では、HPACK の難しさをこの観点から捉え直す。  
つまり、HPACK の核心は「ヘッダを小さくすること」ではなく、  
**状態を参照し、状態を更新する構造になっていること**にある。

---

### ここまでの題材との決定的な違い

UTF-8 でも HTTP/2 フレームでも、境界や分岐や妥当性の問題はあった。  
しかし、主な意味決定はその場の入力に依存していた。

UTF-8 では、先頭バイトを見れば、その文字が何バイト続くかを判断できた。  
HTTP/2 フレームでは、共通ヘッダを見れば Length や Type を読み、フレーム全体の構造を分解できた。  
もちろん妥当性の問題は別にあったが、少なくとも構造や基本的な意味づけは、目の前の入力から始められた。

HPACK では、ここが変わる。  
バイト列のある部分が、単独では十分な情報を持たないことがある。  
それは、**過去に何が送られ、どのようにテーブルが更新されてきたか**に依存するからである。

この違いは大きい。  
今までは、「入力をどう切るか」「入力をどう分解するか」が主な問いだった。  
HPACK では、それに加えて「**今の入力を読むために、過去の履歴として何が必要か**」という問いが入ってくる。

本章で読者がまずつかむべきなのは、この構造変化である。  
HPACK は単に規則が多いから難しいのではない。  
**意味決定の前提が入力の外へ広がるから難しい**のである。

---

### indexed header field という参照

HPACK の状態性を理解するうえで、最初に見たいのが indexed header field である。

indexed header field では、ヘッダフィールドそのものを毎回全部書くのではなく、ある番号を送ることで既知の項目を参照する。  
この発想自体は単純に見える。  
番号があり、その番号が指す項目がある。  
その意味では、配列参照のようなものだと言いたくなるかもしれない。

ただし HPACK で重要なのは、その参照先が固定ではない場合があることである。  
静的な表だけを見ていれば済む話ではなく、動的な表が加わると、同じ「番号を参照する」という形でも、その意味は履歴に依存する。

ここで本書として大切なのは、indexed header field を「圧縮の一種」としてだけ理解しないことである。  
もちろん結果として短くなる。  
しかし読解の観点から見ると、もっと重要なのは、  
**入力の一部が、別の場所に保存されている構造を参照して意味を得る**  
という点である。

この瞬間、入力だけでは完結しない読解が始まる。  
目の前のバイト列は、「それだけで何を表すか」が完全には決まらない。  
それを決めるには、今どのテーブルが存在しているかを知る必要がある。

---

### literal header field という書き方

一方で HPACK には、literal header field のように、ヘッダフィールドをその場で表す形もある。  
ここでは名前や値を、その場の入力から読める形で持つことができる。  
この意味では、indexed header field より「入力だけで読める」部分が多い。

しかしそれでも、literal header field は単純な直書きでは終わらない。  
どの形式の literal なのか、テーブルへ登録するのかしないのか、あるいは別の扱いをするのかによって、その後の状態更新に関わることがあるからである。

つまり literal は、その場で読める成分を増やす一方で、  
**現在の入力が将来の状態を変える入口**にもなっている。  
ここが重要である。

もし HPACK に indexed だけがあり、状態更新が一切なければ、話はかなり単純になる。  
参照先を固定的に読むだけで済むからである。  
しかし実際の HPACK では、literal を通じて dynamic table が更新され、その後の indexed 参照の意味まで変わっていく。  
そのため、literal header field は「生の値を読む場面」であると同時に、「**将来の解釈条件を作る場面**」でもある。

この時点で、HPACK が単なる表現形式ではなく、状態を伴う構造だということが見えてくる。

---

### integer 表現が見せるもの

HPACK では integer 表現も重要な部品である。  
ここは細部へ踏み込みすぎる必要はないが、本書の観点から見ると、integer 表現は二つの意味で重要である。

第一に、ここでもやはり**先頭情報による分岐**が現れることである。  
最初の数ビットや先頭の構造によって、その場で完結するのか、追加のバイトを読み進めるのかが決まる。  
この意味で integer 表現は、第3章で見た UTF-8 の「先頭が後続を決める」構造を、別の形で思い出させる。

第二に、integer 表現は HPACK の中で、長さやインデックスや各種の数値を運ぶために使われる。  
したがって、それ自体は局所的な可変長表現でありながら、最終的には**状態参照や状態更新のための部品**にもなる。

ここで重要なのは、HPACK の難しさが integer 表現そのものには尽きないという点である。  
整数の可変長表現だけなら、それは「先頭情報で分岐する可変長」の一種としてかなり局所的に扱える。  
しかし HPACK では、その整数が何を指すか、何を更新するかが、周囲の文脈や状態と結びつく。  
だから難しさの中心は、可変長そのものではなく、その先にある状態性へ移る。

---

### dynamic table が何を変えるのか

HPACK の状態性を最も強く表すのが dynamic table である。

dynamic table があると、これまでに現れたヘッダフィールドをあとで参照できるようになる。  
これ自体は圧縮の観点から見れば、繰り返しを減らすための仕組みである。  
しかし本書では、そこよりも読解上の変化を重視したい。

dynamic table が入ると、現在の入力の意味は、**過去に table がどう更新されてきたか**に依存する。  
そして現在の入力は、さらにその table を将来に向けて更新することがある。  
つまり HPACK の読解では、「今読む」ことと「次のために状態を変える」ことが結びつく。

この構造が、これまでの章との決定的な差である。

UTF-8 では、ある文字の解釈は、その文字のバイト列の範囲でほぼ閉じていた。  
HTTP/2 フレームでも、あるフレームの共通ヘッダとペイロードを見れば、そのフレームの構造はかなりの部分が決まった。  
しかし HPACK では、ある表現を理解するには、すでに作られている dynamic table を知らなければならない。  
しかも、その理解の結果として table 自体が変化する。

読者がここでつかむべきなのは、dynamic table を「高速化や圧縮のためのキャッシュ」とだけ見ないことである。  
それはむしろ、**現在の意味決定に入り込み、未来の意味決定も変える状態**である。

---

### 同じ入力でも状態により意味が変わる

本章の中心命題を最も短く言うなら、  
**同じ入力でも状態により意味が変わる**  
である。

これは、状態付き構造を扱うときの最重要ポイントである。  
あるインデックス値を見たとき、それが何を指すかは table の内容に依存する。  
同じような literal の操作でも、それが table を更新するかどうかで、その後の参照の意味が変わる。  
つまり、入力列の意味は入力列だけの関数ではない。

この性質は、読解の難しさを大きく変える。  
入力を1回読めば終わりではない。  
**いま参照している状態が何か**  
**この入力で状態がどう変わるか**  
の両方を追わなければならない。

このとき、読者は二つのものを同時に見る必要がある。  
一つは現在のバイト列である。  
もう一つは、そのバイト列の背後で維持されている履歴付きの構造である。  
この二重の視点が必要になることが、HPACK の難しさの核心である。

---

### HPACK の難しさは「圧縮」より「状態」にある

ここで、本章の中心的な整理をはっきり言葉にしておきたい。  
HPACK の難しさは、単にビット操作が多いからでも、規則が多いからでも、圧縮方式だからでもない。  
本質的には、**状態を持つ構造だから**難しい。

もちろん、圧縮方式として見れば、繰り返しを減らし、ヘッダ表現を短くするという設計意図がある。  
しかし読者がつまずくのは、たいていそこではない。  
つまずきやすいのは、今ここにある入力だけを見ても十分でないこと、過去の更新が必要なこと、現在の処理が未来の解釈条件を変えること、そうした性質である。

このことを明確に意識しておくと、HPACK の学び方も変わる。  
圧縮率やエンコードの技巧を先に追うのではなく、まず  
「どこで参照が起きているか」  
「どこで状態更新が起きているか」  
「同じ入力でも状態により意味が変わるとはどういうことか」  
を押さえるべきだとわかるからである。

本書では、この順番を重視する。  
それは、圧縮方式の細部よりも、**読むための認知の型**を優先しているからである。

---

### ここで見える境界・分岐・妥当性・状態

第1章で導入した四つの観点を、ここで改めて HPACK に当ててみると、本章の位置づけがはっきりする。

まず境界である。  
HPACK にも局所的な境界問題はある。  
integer 表現や literal の長さ表現など、どこまで読むかをその場で判断する部分があるからである。  
この点では、第2章、第3章で見た可変長の問題が消えるわけではない。

次に分岐である。  
先頭ビットや表現形式によって、これは indexed なのか literal なのか、追加のバイトを読むのかが決まる。  
ここにも先頭情報による分岐がある。

妥当性もある。  
存在しない参照をしていないか、長さや形式が妥当か、更新規則に反していないか、といった問いが必要になる。

そして最後に状態である。  
これが本章の新しい主役である。  
境界・分岐・妥当性はこれまでの章でも扱ってきた。  
しかし HPACK では、それらが**履歴に依存する読解**の上に載る。  
だからこそ難しさの質が変わる。

この整理ができると、HPACK は「全部が新しくて難しいもの」ではなくなる。  
境界・分岐・妥当性はすでに見た。  
新しく増えたのは、その上に乗る状態である。  
この認識が重要である。

---

### テストデータをどう考えるか

前章では、最小限のテストデータを自分で作る姿勢を扱った。  
その観点を HPACK に持ち込むと、難しさの質がさらに見える。

HPACK の最小正常系を考えるとき、ただ 1 個の表現を作ればよいとは限らない。  
なぜなら、ある表現の意味が、前提となる table の状態に依存するからである。  
つまり HPACK では、テストデータが「今の入力」だけで閉じない。  
**前提状態と入力の組**として考える必要がある。

たとえば、indexed header field を確認したいなら、その参照先が何であるかを決める前提状態が必要になる。  
literal header field で dynamic table 更新を確認したいなら、更新前と更新後の差を見られるようにしなければならない。  
この時点で、テスト設計も状態付きになる。

ここで無理に細かいケースを増やす必要はない。  
本章の段階では、まず次のことが見えれば十分である。

- HPACK では入力だけでなく前提状態が必要になる
    
- 現在の入力は future state を変えることがある
    
- したがって最小テストデータも「状態つき最小ケース」として考える必要がある
    

この認識が、第8章の state を持つ parser の整理につながっていく。

---

### この章の位置づけ

本章は、シリーズ全体の中でも橋の中央にある章である。  
ここまでの章では、境界、分岐、妥当性を持つバイト列を読めるようにしてきた。  
しかしそれらは、基本的にはその場の入力から出発する読解だった。

本章から、読解は一段難しくなる。  
HPACK では、過去の履歴を保持し、その履歴を参照しながら現在を読み、さらに現在の入力で履歴を更新する。  
つまり、読解が**時間方向の広がり**を持ち始める。

ただし、この章の役割は、HPACK の細部へ突っ込むことではない。  
それは上級編や実装論に近づきすぎる。  
本書で必要なのは、  
「HPACK では何が新しく難しいのか」  
を切り出して見せることである。

その答えは明確である。  
**HPACK は状態を持つバイナリ構造である。**  
この一文が本章の到達点である。

---

### 次章への橋――state を持つ parser をどう考えるか

本章では、HPACK の難しさを「状態」に置き直した。  
同じ入力でも状態によって意味が変わること、dynamic table が現在の解釈条件であると同時に将来の解釈条件でもあることを見た。

次章では、この具体像を素材にして、もう一段抽象化する。  
つまり、**state を持つ parser をどう考えるか**である。

ここで重要なのは、HPACK 固有の規則を増やすことではない。  
むしろ、入力・履歴・更新規則をどう分けて考えるか、parser と state 管理をどこで切り分けるか、という設計上の視点へ進む。

第7章が「状態という難しさの導入」であるなら、第8章は「その難しさをどう整理するか」の章である。  
だから本章では、まず HPACK の具体性の中で、状態が何を変えるかをはっきり見ておく必要がある。

### 整理ブロック

#### HPACK を本書でどう捉えるか

| 観点 | 本書での捉え方 |
| --- | --- |
| 圧縮方式 | 結果として短くする仕組みではあるが、本質的な難しさはそこにない |
| バイナリ構造 | indexed / literal / integer 表現などの分岐と境界を持つ |
| 状態付き構造 | dynamic table の参照と更新によって意味が状態に依存する |
| 学習上の主眼 | 圧縮率ではなく、状態が読解をどう変えるかを見る |

#### HPACK で見える主要な要素

| 要素 | 本書で重視する点 |
| --- | --- |
| indexed header field | 入力の一部がテーブル参照によって意味を得る |
| literal header field | その場で読めるだけでなく、将来の状態更新にも関わる |
| integer 表現 | 先頭情報による分岐を持つ局所的な可変長表現 |
| dynamic table | 現在の意味決定と将来の意味決定の両方に効く状態 |

#### HPACK の難しさの整理

| 難しさ | 内容 |
| --- | --- |
| 境界 | integer や literal の長さをどう切るか |
| 分岐 | indexed か literal か、どの形式かをどう見分けるか |
| 妥当性 | 参照や形式や更新がルールに合っているか |
| 状態 | 現在の入力の意味が前提状態に依存し、現在の入力が未来の状態も変える |

#### この章で押さえるべきこと

- HPACK は圧縮方式としてだけでは理解しにくい
    
- 真に新しい難しさは状態にある
    
- indexed 参照は入力外の構造を必要とする
    
- dynamic table は現在と未来の意味決定に効く
    
- 同じ入力でも状態により意味が変わる

HPACK の難しさは、  
**入力を読むことが、その場のバイト列だけでは完結しないこと**にある。

## まとめ

- HPACK は HTTP/2 のヘッダ圧縮方式だが、本書ではそれを状態を持つバイナリ構造として読む。
    
- indexed header field は、入力の一部がテーブル参照によって意味を得ることを示している。
    
- literal header field は、その場の値を表すだけでなく、dynamic table の更新にも関わる。
    
- dynamic table があることで、現在の入力の意味は前提状態に依存し、現在の入力は未来の解釈条件も変える。
    
- HPACK の本質的な難しさは、圧縮そのものよりも「同じ入力でも状態により意味が変わる」という点にある。
    

## 次章への接続

次章では、HPACK を具体例として使いながら、  
**state を持つ parser をどう考えるか** を整理する。  
入力・履歴・更新規則を分けて考える視点を導入し、状態付き構造を読むときの設計上の見方へ進んでいく。
