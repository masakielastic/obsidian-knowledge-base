# WordPressアーキテクチャ入門 執筆ガイドライン

## 1. 書籍の目的

この書籍の目的は、WordPress を  
**「テーマやプラグインの使い方の寄せ集め」ではなく、「Gutenberg 以降の設計思想と責務分界を持つシステム」**  
として理解できるようにすることです。

読者が最終的にできるようになるべきことは次の通りです。

- クラシック WordPress と Gutenberg / ブロックテーマの違いを説明できる
- PHP と JavaScript の両方が必要になる理由を説明できる
- `block.json` / `theme.json` / editor state / saved HTML の役割を区別できる
- WordPress を Web Components / React / SPA / Headless CMS と比較して位置づけられる
- WordPress のテスト戦略を、単なるテストピラミッドの暗記ではなく、保存契約・制約・表示資産を守る観点で説明できる
- WordPress を「古いから終わったもの」ではなく、「別の最適化を持つアーキテクチャ」として評価できる

## 2. 想定読者

- WordPress のテーマやプラグインに少し触れたことがある人
- PHP と JavaScript の基本構文は読める人
- MVC や DDD という言葉に抵抗がない人
- Gutenberg やブロックテーマを使ったことはあるが、構造として説明しにくい人
- 小規模 Web 制作会社や受託案件の現場にいて、今後必要になる理解を整理したい人
- WordPress を「古い CMS」と雑に片付けず、構造として理解したい人

## 3. 書籍の基本方針

### 重視すること

- 設計思想の理解
- 責務の分解
- 旧来の WordPress と新しい WordPress の連続性と断絶の整理
- PHP 側と JavaScript 側の責務分界の説明
- 保存 HTML という資産性の理解
- 他技術との比較による位置づけ
- DDD / CQRS / Event Sourcing を使った再解釈
- テスト戦略を実務に接続すること

### 深入りしないこと

- WordPress 完全初心者向けの操作説明
- テーマ開発やプラグイン開発のハンズオン手順の網羅
- DDD 一般理論の詳説
- React / Web Components / Headless CMS の入門書的説明
- 細かな API カタログ化
- 特定サービスやホスティングの製品比較中心の説明

## 4. 読者のスタート地点

この本は、次のような読者を前提とします。

- WordPress を少し触ったことはある
- しかし Gutenberg 以降を構造として説明できない
- 求人票にある「テーマが触れる」「プラグインが書ける」程度の実務要件では、アーキテクチャ理解に届かないと感じている
- WordPress を断片知識ではなく、比較可能な地図として理解したい

この本は、WordPress の使い方を覚える本ではなく、  
**WordPress を説明できるようになるための本**  
として設計する。

## 5. 対話形式の役割分担

登場人物は3役で固定します。

### 初学者

役割：

- 素朴な疑問や誤解を出す
- 読者が抱きやすい短絡的な理解を口にする
- 「WordPress は古いのでは」「React でよいのでは」などの素直な仮説を出す

注意：

- 賢くしすぎない
- DDD 的整理を先回りして言わない
- 解説役にならない
- 読者より一段先に進みすぎない

### 解説者

役割：

- 構造を整理する
- 比較軸を与える
- 責務分界を明確にする
- 誤解を修正しながら、章の中心命題を言語化する

注意：

- 何でも一度に説明しない
- 講義調の独演になりすぎない
- 比較だけで終わらず、最後は WordPress の用語と構造に戻す
- 章の中心命題から離れすぎない

### 実装者

役割：

- 実際の WordPress の構造や実装上の見え方を補足する
- PHP 側 / JavaScript 側 / editor state / block registration / 保存 HTML の具体像を示す
- テストや実務上の破綻点を補足する
- 抽象説明を現実の開発と運用に接続する

注意：

- 抽象説明を主導しない
- 細かな実装論に脱線しすぎない
- フレームワーク論争に広げすぎない
- 解説者の役を奪わない

## 6. 本書で扱う3つの仮説

この本の対話は、次の3つの仮説を扱うことを前提とします。

### 論点仮説

- この章では何を問題として切り出すか
- どの比較軸が有効か
- 何をいったん扱わないか

### アプローチ仮説

- どの技術との比較で理解しやすいか
- どこから説明すると混乱が減るか
- 何を観察すると WordPress の構造が見えるか

### 答え仮説

- 現時点ではこう理解するとよいのではないか、という中間結論
- ただし次章で見直されることがある
- 読者の理解を段階的に組み替えるための仮説

### 運用ルール

- 初学者は答え仮説を出しやすい
- 解説者は論点仮説を調整する
- 実装者はアプローチ仮説を支える
- 誤解は教育的価値があるため、必要なら一度残してよい
- ただし誤解を放置したまま章を終えない

## 7. 文章スタイル

### 基本トーン

- 技術的に正確
- 初学者にやさしい
- 構造理解に重点を置く
- 簡潔だが切り捨てすぎない
- 対話として自然
- 比較対象を出しても、最後は WordPress に戻す
- 音声読み上げでも不自然になりにくい

### 文体ルール

- です・ます調を基本とする
- 一文を長くしすぎない
- 一文一義を意識する
- 定義・整理・結論は明確に言い切る
- 比喩は補助的に使い、最後は正式な用語に戻す
- 「まず」「次に」「つまり」などの段階整理を積極的に使う
- 抽象と具体を往復する
- 用語の初出では、その章に必要な範囲で説明する

### 避けるべき文体

- WordPress を一方的に古いと断定する書き方
- 専門用語の説明なしの連打
- React / Headless / Web Components への安易な優劣論
- 読者が置いていかれる圧縮しすぎた説明
- 実装者が理論を延々と語る構成
- API 名や設定項目を列挙するだけの説明

## 8. 音声向け情報と視覚向け情報の分離

### 対話本文に置くもの

- 疑問
- 誤解
- 比較のきっかけ
- 概念説明
- 因果関係
- 責務分界
- 仮説の修正
- 「なぜ WordPress はそうなっているのか」の説明

### 対話の外に置くもの

- 箇条書き
- 用語整理
- 比較表の代わりになる短い一覧
- レイヤー整理
- 責務マップ
- テスト配分表
- 章末まとめ
- 「この章の核心」

### 運用ルール

- 対話本文では原則として箇条書きを使わない
- 2〜3項目程度の短い確認は、音声で自然なら許容してよい
- それ以上の一覧・比較・整理は対話の外へ出す
- 一覧や整理は原則として `### 整理` に置く
- 章末まとめは視覚的に一覧化してよい

## 9. 見出し構造の基準

Zenn の書籍では、基本的な見出しレベルは 2 までとする。

### 基本ルール

- 各ファイルのタイトルは先頭の `#`
- 章タイトル相当の大見出しは `##`
- 章内セクションは `###`
- 原則として `####` 以降は使わない
- 「整理」は `### 整理`
- 「この章の核心」は `### この章の核心`
- 「まとめ」は `## まとめ`

### 禁止事項

- 章ごとに見出しレベルを変える
- `####` 以降を常用する
- `まとめ` の見出しレベルを章ごとに変える
- 細かい見出しだけ増やして本文が薄くなる構成にする

## 10. 章フォーマットの基準

各章は原則として次の流れで構成する。

### 1. 導入

- 初学者が疑問や誤解を出す
- 解説者がこの章で扱う範囲を示す
- 必要なら早い答え仮説を先に出してよい

### 2. 本文

- 対話で概念を分解する
- 必要に応じて `###` セクションを置く
- 実装者が実際の WordPress の構造やテスト観点を補足する
- 章の中心命題から逸れたら戻す

### 3. 整理ブロック

- 見出し名は `### 整理`
- 本文で扱った内容を、用語・責務・流れ・比較軸として視覚的に整理する
- 新情報は追加しない
- 比較や責務マップを簡潔に圧縮する

### 4. この章の核心

- 見出し名は `### この章の核心`
- 1〜3文で書く
- 箇条書きは使わない
- その章の中心命題を圧縮して言い切る
- 新情報は追加しない

### 5. まとめ

- 見出し名は `## まとめ`
- 箇条書き3〜5項目で書く
- 用語の区別・責務・比較軸を再確認する
- 「この章の核心」を箇条書きに言い換えるだけにしない

### 6. 次章への接続

- 1〜2文で次章へつなぐ
- 次章で何の論点を扱うかを軽く示す

## 11. 章タイトルと章の中心命題

- 各章は原則として1つの中心命題に絞る
- 章タイトルだけ見て、何を整理する章か分かるようにする
- 主役となる概念と、補助的に触れる概念の比重を明確にする
- 本文の比重は章タイトルに対応していること
- 話題が広がった場合も、その章の中心命題に必ず戻る

## 12. 校正時の最重要品質基準

校正では、特に次の観点を優先して確認する。

### A. 概念の区別がぶれていないか

特に以下は必ず明確であること。

- クラシックテーマ / ブロックテーマ
- editor state / saved HTML
- `block.json` / `theme.json`
- capability / constraint
- UI 設計 / 制約設計
- Write モデル / Read モデル
- Snapshot / Event 的状態遷移
- unit / integration / E2E / manual

### B. 比較が WordPress 理解に回収されているか

- React の説明が React 入門になっていないか
- Web Components との比較が WordPress の特徴理解につながっているか
- Headless CMS との比較が優劣論で終わっていないか
- 新世代 CMS との比較が「需要の違い」の説明に回収されているか

### C. 読者の混乱ポイントに答えているか

- 章冒頭の疑問に本文が答えているか
- 話題が広がりすぎていないか
- PHP と JavaScript の両方が必要な理由が見えるか
- 求人票レベルの実務知識との差が言語化されているか

### D. 圧縮が効いているか

- 長い説明のあとに整理があるか
- まとめが機能しているか
- 章を読んだあとに3〜5点で復習できるか

### E. 章ごとの粒度が揃っているか

- 見出しの粒度が揃っているか
- まとめの長さがバラバラすぎないか
- 章タイトルと本文の比重が合っているか

## 13. 具体的なレビュー観点

### 用語

- 初出時に説明されているか
- 同じ意味の言葉が複数混在していないか
- 「古い」「モダン」「柔軟」などの語が曖昧なまま使われていないか
- 「裏でやってくれる」「いい感じに動く」などが責務を曖昧にしていないか

### 対話

- 誰が何を担うかが明確か
- 初学者が説明役になっていないか
- 解説者が独演になっていないか
- 実装者が補足役に収まっているか
- 誤解と修正の流れがあるか

### 構成

- 1セクション1テーマになっているか
- 話題の飛躍がないか
- 「この章の核心」と「まとめ」が重複しすぎていないか
- 章の中で論点仮説がぶれていないか

### 可読性

- 一文が長すぎないか
- 比較や整理が本文に埋もれていないか
- WordPress 固有用語と一般用語の関係が見えるか
- 例が抽象論だけで終わっていないか

### 読み上げ適性

- 対話部分が声に出して自然か
- 箇条書きは視覚ブロックに分離されているか
- 段階整理が機能しているか

## 14. まとめの基準

各章の最後の「まとめ」は、次の基準に従う。

- 箇条書き3〜5項目
- 新しい情報を追加しない
- 章タイトルと対応する内容に絞る
- 読者が短時間で復習できるようにする
- 用語の区別と責務の違いが再確認できる内容にする

### 悪い例

- 抽象的すぎる
- 本文にない情報を足す
- 1項目が長すぎる
- 章ごとにスタイルが変わる

### 良い例

- 用語の整理
- 比較軸の再確認
- 責務の再確認
- 次章の準備になる要点

## 15. リライト方針

校正や改善を行う際は、単なる誤字修正ではなく、必要に応じて以下を行う。

- 冗長な部分の圧縮
- 曖昧な主語や対象の明確化
- 長い台詞の分割
- 比較対象の位置づけの明確化
- 箇条書きと対話の役割分離
- 見出し粒度の統一
- まとめの統一フォーマット化
- 責務分解への接続の補強
- 「古い / 新しい」ではなく「何を最適化しているか」への言い換え

ただし、以下は避ける。

- 詳細すぎる実装知識の追加
- 章のスコープの勝手な拡張
- 対話形式を崩して説明文だけにしてしまうこと
- 書籍全体のトーンを急に変えること
- 他技術の詳細比較に流れすぎること

## 16. 出力時の期待

レビュー・校正・リライトを行うときは、必要に応じて次の観点で整理する。

- 構成上の問題
- 文章上の問題
- 用語運用上の問題
- 対話役割上の問題
- 比較軸運用上の問題
- 仮説運用上の問題
- 修正案
- 必要なら統一版のリライト

## 17. この書籍で特に守るべき一文

この本は、WordPress を  
**「テーマやプラグインの使い方」ではなく「編集可能な HTML 資産を、制約付きで安全に構築するシステム」**  
として理解させる本である。

校正・改善・リライトの判断は、常にこの方針に従って行うこと。
