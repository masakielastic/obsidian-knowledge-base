# Async for JS Developers 執筆ガイドライン

## 1. 書籍の目的

この書籍の目的は、JavaScript ユーザーが非同期処理を  
**Promise / await の使い方の暗記ではなく、責務分解と実行モデルの理解**として捉えられるようにすることです。

読者が最終的にできるようになるべきことは次の通りです。

- Promise / Future / Task / runtime / Event Loop の違いを説明できる
- Stream / Socket / Fiber / Event Loop を同じ種類のものとして混同しない
- 「値」「計算」「実行単位」「実行基盤」「責任スコープ」を区別できる
- Rust/Tokio や Python の構造化並行性を読むための土台を持てる

## 2. 想定読者

- ブラウザ JavaScript または Node.js を使ったことがある人
- Promise / async / await は使ったことがある人
- Event Loop や Stream の理解に自信がない人
- Rust/Tokio や Python asyncio をこれから学びたい人
- API の丸暗記ではなく、設計の見取り図をほしい人

## 3. 書籍の基本方針

### 重視すること

- 概念の区別
- 責務の分解
- 実行モデルの理解
- 読者の混乱ポイントの言語化
- JavaScript の既存メンタルモデルとの接続
- 他言語へ橋を架けるための比較可能な地図

### 深入りしないこと

- 各言語ランタイムの内部実装の網羅
- libuv や epoll や io_uring の詳細実装の深掘り
- Rust / Python の API カタログ化
- フレームワークやライブラリの個別比較の詳細
- ベンチマークや性能議論中心の説明

## 4. 対話形式の役割分担

登場人物は3役で固定します。

### 初学者

役割：

- 疑問・混乱・誤解を出す
- 読者のつまずきを代弁する
- 早い答え仮説を口にする

注意：

- 賢くしすぎない
- 正しい整理を先回りして言わない
- 専門用語を自力で整理しすぎない
- 説明役にならない

### 解説者

役割：

- 構造と考え方を説明する
- 論点を切り分ける
- 用語を定義し、責務の違いを言語化する

注意：

- 何でも一気に説明しない
- 講義調の独演になりすぎない
- 比喩だけで終わらず、最後は必ず正式な用語に戻す
- 章の中心命題から外れすぎない

### 実装者

役割：

- 実際の挙動や設計上の破綻点を補足する
- API / runtime / Event Loop / 実装上の見え方を具体化する
- 解説者の抽象説明を現実の動きに接続する
- アプローチ仮説を支える

注意：

- 抽象説明を主導しない
- 解説者の役を奪わない
- 枝葉末節の実装論に広げすぎない
- 章のスコープを超えて脱線しない

## 5. 本書で扱う3つの仮説

本書の対話は、次の3つの仮説を扱うことを前提とします。

### 論点仮説

- いま何を問題として切り出すか
- 探索範囲を決めるための仮説

### アプローチ仮説

- どこから見ると理解しやすいか
- どう確かめるかを決める仮説

### 答え仮説

- いまの段階ではこういう結論ではないか、という仮説
- 探索の優先順位を決めるための仮説

### 運用ルール

- 各章では、初学者が答え仮説を出しやすい
- 解説者は論点仮説を調整する
- 実装者はアプローチ仮説を支える
- 一度の誤解や試行錯誤は教育的価値があるため、必要に応じて残してよい
- ただし、誤解を放置したまま章を終えない

## 6. 文章スタイル

### 基本トーン

- 技術的に正確
- 初学者にやさしい
- 簡潔だが冷たすぎない
- 冗長すぎない
- 対話として自然
- 音声読み上げでも不自然になりにくい

### 文体ルール

- です・ます調を基本とする
- 一文を長くしすぎない
- 一文一義を意識する
- 定義・整理・結論は明確に言い切る
- 比喩は補助的に使い、最後は用語に戻す
- 「まず」「次に」「最後に」などの段階整理は積極的に使ってよい
- 抽象と具体を往復する
- 用語の初出では、その章に必要な範囲で説明する

### 避けるべき文体

- 曖昧な断定の連続
- 専門用語の説明なしの連打
- 技術書として軽すぎる口語
- 初学者が置いていかれる圧縮しすぎた説明
- 実装者が理論を延々と語る構成
- API 名や用語の列挙だけで意味を説明しない書き方

## 7. 音声向け情報と視覚向け情報の分離

この書籍では、  
**声に出して自然に読める情報**と、  
**目で見たほうが整理しやすい情報**を分けてよいものとします。

### 対話本文に置くもの

- 疑問
- 誤解
- 概念説明
- 因果関係
- 仮説の修正
- 設計上の見方
- 「なぜそう見えるのか」の説明

### 対話の外に置くもの

- 箇条書き
- 用語整理
- 比較表の代わりになる短い一覧
- 手順の一覧
- 責務マップ
- 章末まとめ
- 「この章の核心」

### 運用ルール

- 対話本文では原則として箇条書きを使わない
- 2〜3項目程度の短い確認は、音声で自然なら例外的に許容してよい
- それ以上の一覧・比較・整理は対話の外へ出す
- 一覧や整理は原則として `### 整理` に置く
- 長い整理は短文連続か、視覚ブロックに分離する
- 章末のまとめは視覚的に一覧化してよい

## 8. 見出し構造の基準

Zenn の書籍では、基本的な見出しレベルは **2まで** とします。  
理由は、`#` と `##` が目次の生成対象になるデフォルトであり、章構造を安定して読ませやすいからです。

### 基本ルール

- 各ファイルのタイトルは先頭の `#` を使ってよい
- 章タイトル相当の大見出しは `##`
- 章内セクションは `###`
- 原則として `####` 以降は使わない
- 「整理」は `### 整理`
- 「この章の核心」は `### この章の核心`
- 「まとめ」は `## まとめ`

### 禁止事項

- 章によって見出しレベルを変える
- `####` 以降を常用する
- `まとめ` の見出しレベルを章ごとに変える
- 見出しだけ細かく増やして本文が薄くなる構成にする

## 9. 章フォーマットの基準

各章は原則として次の流れで構成します。

### 1. 導入

- 初学者が疑問や混乱を出す
- 解説者がこの章で扱う範囲を示す
- 必要に応じて、誤った答え仮説を先に出してよい

### 2. 本文

- 対話で概念を分解する
- 必要に応じて `###` セクションを置く
- 実装者が API や runtime の挙動、設計上の破綻例を補足する
- 章の中心命題から逸れたら戻す

### 3. 整理ブロック

- 見出し名は原則として `### 整理`
- 本文で扱った内容を、用語・責務・流れ・観察ポイントとして視覚的に整理する
- 新情報は追加しない
- 比較や責務マップを簡潔に圧縮する

### 4. この章の核心

- 見出し名は `### この章の核心`
- 1〜3文程度の短い本文で書く
- 箇条書きは使わない
- その章の中心命題を圧縮して言い切る
- 新情報は追加しない

### 5. まとめ

- 見出し名は `## まとめ`
- 箇条書き3〜5項目で書く
- 復習用として、用語の区別・責務・流れを再確認する
- 「この章の核心」を箇条書きに言い換えるだけにならないようにする

### 6. 次章への接続

- 1〜2文で次章へつなぐ
- 次章で何の論点を扱うかを軽く示す

## 10. 章タイトルと章の中心命題

- 各章は原則として1つの中心命題に絞る
- 章タイトルを見ただけで、その章が何を整理する章か分かるようにする
- 1章の中で、主役となる概念と補助的に触れる概念の比重を明確にする
- 本文の比重は章タイトルに対応していること
- 話題が広がった場合も、その章の中心命題に必ず戻る

## 11. 校正時の最重要品質基準

校正では、特に次の観点を優先して確認してください。

### A. 概念の区別がぶれていないか

特に以下は必ず明確であること。

- Promise / Future
- 値 / 計算
- task / runtime / Event Loop
- Stream / Socket / Fiber
- 監視 / 実行 / 調停
- 並行実行 / 寿命管理
- Promise.all / TaskGroup 的な責任スコープ

### B. 実行モデルへの接続があるか

- 「誰が何を進めるのか」が説明されているか
- Event Loop や runtime の位置づけが示されているか
- Node.js / ブラウザ / Rust / Python の見え方の差が、責務の差ではなく見え方の差として整理されているか

### C. 読者の混乱ポイントに答えているか

- 章冒頭の疑問に本文がきちんと答えているか
- 途中で話題が広がりすぎていないか
- 初学者に必要な説明が飛ばされていないか

### D. 圧縮が効いているか

- 長い説明のあとに整理があるか
- まとめが機能しているか
- 章を読んだあとに3〜5点で復習できるか

### E. 章ごとの粒度が揃っているか

- 見出しの粒度が揃っているか
- まとめの長さがバラバラすぎないか
- 章タイトルと本文の比重が合っているか

## 12. 具体的なレビュー観点

### 用語

- 初出時に説明されているか
- 同じ意味の言葉が複数混在していないか
- 曖昧な言い換えが増えすぎていないか
- 「便利」「賢い」「裏で動く」などの表現が、責務を曖昧にしていないか

### 対話

- 誰が何を担うかが明確か
- 初学者が説明役になっていないか
- 解説者が長すぎる独演になっていないか
- 実装者が補足役に収まっているか
- 誤解と修正の流れがあるか

### 構成

- 1セクション1テーマになっているか
- 話題の飛躍がないか
- 「この章の核心」と「まとめ」が重複しすぎていないか
- 章の中で論点仮説がぶれていないか

### 可読性

- 一文が長すぎないか
- 読点が多すぎて息継ぎしにくくないか
- 長文の台詞が箇条書きの代わりになって崩れていないか
- 比較や整理が本文に埋もれていないか

### 読み上げ適性

- 対話部分が声に出して自然か
- 箇条書きは視覚ブロックに分離されているか
- 「まず」「次に」などの段階整理が機能しているか

## 13. まとめの基準

各章の最後の「まとめ」は、次の基準に従ってください。

- 箇条書き3〜5項目
- 新しい情報を追加しない
- 章タイトルと対応する内容に絞る
- 読者が章の内容を短時間で復習できるようにする
- 用語の区別と責務の違いが再確認できる内容にする

### 悪い例

- 抽象的すぎる
- 本文にない情報を足す
- 1項目が長すぎる
- 章ごとにスタイルが変わる

### 良い例

- 用語の整理
- 責務の再確認
- 流れの再確認
- 読者が次章の準備をしやすくなる要点

## 14. リライト方針

校正や改善を行う際は、単なる誤字修正ではなく、必要に応じて以下を行ってください。

- 冗長な部分の圧縮
- 曖昧な主語や対象の明確化
- 長い台詞の分割
- 箇条書きと対話の役割分離
- 見出し粒度の統一
- まとめの統一フォーマット化
- 実行モデルや責務分解への接続の補強
- 誤解を修正する流れの明確化

ただし、以下は避けてください。

- 詳細すぎる専門知識の追加
- 章のスコープを勝手に拡張すること
- 対話形式を崩して説明文だけにしてしまうこと
- 書籍全体のトーンを急に変えること
- 各言語の細部比較に流れすぎること

## 15. 出力時の期待

レビュー・校正・リライトを行うときは、必要に応じて次の観点で整理してください。

- 構成上の問題
- 文章上の問題
- 用語運用上の問題
- 対話役割上の問題
- 仮説運用上の問題
- 修正案
- 必要なら統一版のリライト

## 16. この書籍で特に守るべき一文

この本は、非同期処理を  
**「await の書き方」ではなく「責務と実行モデルの地図」として理解させる本**  
である。

校正・改善・リライトの判断は、常にこの方針に従って行ってください。
