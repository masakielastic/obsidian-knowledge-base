# 第8章 WordPress でテストするとは何を守ることか

**初学者**  
WordPress のテストというと、正直まだ少し曖昧です。  
普通のアプリケーションなら、ユニットテストや統合テストや E2E テストを組み合わせる、という話になりますよね。  
でも WordPress はテーマもブロックもエディタも絡むので、何を基準に考えればよいのか分かりにくいです。

**解説者**  
その戸惑いは自然です。  
この章の中心命題はこうです。  
WordPress でテストするとは、単にコードの正しさを確認することではなく、編集可能な HTML 資産が、制約付きで安全に保存・再編集・表示され続けることを守ることです。  
つまり、守る対象が少し違います。

**実装者**  
実務でも、ここを取り違えるとテストが空回りします。  
関数単体のテストだけを積み上げても、編集画面でブロックが崩れる、保存後に意図しない HTML になる、テーマの制約が効かず運用で破綻する、といった問題は防げません。  
WordPress は、コードだけでなく、編集体験と保存結果と表示結果の関係まで含めて見る必要があるんです。

**初学者**  
なるほど。  
普通の Web アプリのテストをそのまま当てはめるだけでは足りないのですね。

**解説者**  
その通りです。  
もちろん一般的なテストの考え方は役に立ちます。  
ですが WordPress では、何を守るべきかを先に定義しないと、テストの粒度も配分も決めにくい。  
この章では、その「守るべきもの」を整理していきます。

## まず、何を守るためにテストするのか

**初学者**  
最初にそこをはっきりさせたいです。  
WordPress でテストするとき、何を守ると考えればよいのでしょうか。

**解説者**  
大きく言えば、四つあります。  
保存された HTML、編集体験、テーマやブロックが与える制約、そして最終的な表示資産です。  
これらは別々のものに見えますが、WordPress では強くつながっています。

**実装者**  
たとえば、あるブロックを編集して保存したとき、保存結果の HTML が変わります。  
その HTML は公開画面で使われます。  
しかも後で再編集できなければ困る。  
さらに、そのブロックがテーマの制約やスタイル前提と噛み合っていないと、更新のたびに見た目が崩れる。  
つまり WordPress のテスト対象は、ひとつの関数の返り値だけではなく、編集から表示までの連鎖なんです。

**初学者**  
つまり、WordPress のテストは、単独の部品というより、つながりを守るためのものなのですね。

**解説者**  
その理解が重要です。  
WordPress は、編集可能な HTML 資産を長期運用するシステムでした。  
ならばテストも、その資産が安全に育つことを守らなければなりません。  
この視点がないと、テストが「書きやすいものだけ測る作業」になってしまいます。

## コードではなく契約を守るという発想

**初学者**  
「コードではなく契約を守る」という言い方が、少し気になります。  
どういう意味でしょうか。

**解説者**  
ここでいう契約とは、「このブロックはこういう構造で保存される」「このテーマではこういう制約のもとで編集できる」「このテンプレートではこの表示が成立する」といった、編集・保存・表示の約束です。  
テストは、その約束が破られていないかを確かめる手段です。

**実装者**  
実務で起きる不具合の多くは、内部関数の単純ミスだけではありません。  
あるブロックの属性変更で、既存コンテンツが再編集できなくなる。  
`theme.json` の調整で、エディタと公開画面の見え方がずれる。  
テンプレート変更で、意図していた patterns の流れが崩れる。  
こういう問題は、コード行単位ではなく、契約の破れとして現れます。

**初学者**  
なるほど。  
関数が正しく動いていても、全体の約束が崩れていれば問題なのですね。

**解説者**  
その通りです。  
WordPress のテストでは、「この関数が通ったから安心」では不十分です。  
大事なのは、編集者が安全に触れ、保存された HTML が意図通り残り、公開画面で破綻しないことです。  
だからテストの軸は、コード単体より契約全体に寄ります。

## 保存 HTML はなぜそんなに重要なのか

**初学者**  
この本では、保存 HTML が何度も出てきますよね。  
テストでも、やはりそこが重要なのでしょうか。

**解説者**  
非常に重要です。  
WordPress は、Gutenberg 以降も保存された HTML を重い資産として扱っています。  
エディタ内部に state があり、ブロック構造があり、それでも最終的に保存される HTML が公開と再編集の基盤になります。  
だから、この保存結果が崩れることは、かなり大きな問題です。

**実装者**  
たとえば、見た目上は同じでも、保存 HTML の構造が変わると、後方互換性や再編集可能性に影響することがあります。  
また、テーマ側の CSS やテンプレートは、保存された構造を前提にしていることが多いです。  
つまり HTML は、単なる出力結果ではなく、他の層が依存する契約面でもあります。

**初学者**  
そう考えると、保存 HTML のテストは、スクリーンショット確認よりも根本的ですね。

**解説者**  
その通りです。  
画面確認は大切ですが、保存 HTML の安定性は、それ以前の土台です。  
テストでは、見た目の変化だけでなく、保存される構造の変化にも敏感である必要があります。

## 編集体験もテスト対象になる

**初学者**  
でも、編集体験までテストするのは大変そうです。  
そこは手動確認だけではだめなのでしょうか。

**解説者**  
手動確認は重要です。  
ただし、編集体験そのものが WordPress の価値の一部である以上、完全に軽視はできません。  
ブロックが追加できるか。  
意図した入れ子が可能か。  
制約が効いているか。  
保存後に再編集しても崩れないか。  
こうした点は、編集体験のテスト対象です。

**実装者**  
特にブロックテーマ時代は、編集体験がそのまま運用品質につながります。  
更新担当者が迷う。  
意図しない操作で構造が崩れる。  
本来制限されているはずの変更が通ってしまう。  
こういう問題は、公開画面の最終確認だけでは見つけにくいです。  
だからエディタ上のふるまいも、テストの一部として考える必要があります。

**初学者**  
つまり WordPress では、編集しやすさや壊れにくさも品質なのですね。

**解説者**  
その通りです。  
普通のアプリケーションでも UI の品質は大事ですが、WordPress では特に、編集 UI がコンテンツ生産そのものに直結しています。  
だから編集体験は、周辺ではなく中核のテスト対象になります。

## 制約が守られているかもテストする

**初学者**  
この章のタイトルからすると、制約も守る対象に入るのですよね。

**解説者**  
はい。  
制約があるなら、その制約が本当に効いているかもテストしなければなりません。  
ブロックテーマ時代のテーマは、自由を与えるだけでなく、自由の範囲を設計していました。  
ならば、その範囲が壊れていないことを確かめる必要があります。

**実装者**  
たとえば、許していない色やスタイルが選べてしまう。  
想定外のブロック構成が入り込める。  
テンプレートパーツが不用意に崩せる。  
patterns を起点にしたはずの構造が運用で簡単に散ってしまう。  
こうした問題は、制約設計がテストされていないと起きやすいです。

**初学者**  
制約は設計しただけではだめで、維持されているか見ないといけないのですね。

**解説者**  
その通りです。  
制約設計は、書類上の方針ではなく、編集空間のふるまいとして実現されて初めて意味があります。  
だから WordPress のテストでは、自由度が広すぎないか、ガードレールが外れていないかも確認対象になります。

## 表示資産を守るとはどういうことか

**初学者**  
保存 HTML と編集体験と制約は分かってきました。  
では「表示資産を守る」とは、どういう意味でしょうか。

**解説者**  
表示資産とは、公開画面として読者が触れる HTML、レイアウト、導線、見た目の一貫性を含んだ資産です。  
WordPress は公開システムでもありますから、最終表示が崩れないことは当然重要です。  
ただし、ここでも単なる見た目チェックで終わらないのがポイントです。

**実装者**  
表示資産は、テーマ、保存 HTML、ブロックの構造、制約の効き方の結果として生まれます。  
だから見た目の崩れがあったとき、それは CSS の問題だけとは限りません。  
保存構造が変わったのかもしれない。  
ブロックの契約が変わったのかもしれない。  
テーマの制約が外れたのかもしれない。  
表示資産を守るとは、この結果面を安定して維持することです。

**初学者**  
最終画面は出口だけれど、出口だけ見ても原因は分からないのですね。

**解説者**  
その通りです。  
だから WordPress のテストは、表示だけ、保存だけ、編集だけを見るのではなく、それらのつながりを意識して配分する必要があります。

## pure unit test はどこで効くのか

**初学者**  
ここまで聞くと、ユニットテストの出番が少ないようにも思えてきます。  
pure unit test は役に立たないのでしょうか。

**解説者**  
そんなことはありません。  
pure unit test は、ロジックを切り出せるところでは有効です。  
変換処理、データ整形、条件分岐、補助関数、保存前後の一部ロジックなど、局所的に検証可能なものはあります。  
ただし、それだけで WordPress の品質は守れない、という話です。

**実装者**  
ユニットテストが役に立つのは、責務がきれいに切り出せる場所です。  
たとえば、ある設定値からクラス名を生成する処理や、属性の正規化、内部変換などはテストしやすい。  
でも「このブロックが編集画面で自然に使えて、保存後も崩れないか」はユニットテストだけでは見えません。  
だから WordPress では、ユニットテストを過大評価しないことが大事です。

**初学者**  
つまり、ユニットテストは必要だけれど、主役ではない場面が多いのですね。

**解説者**  
その整理でよいです。  
WordPress は UI と保存契約とテーマ制約が絡むので、純粋関数だけでは全体を代表しにくいのです。

## integration test は何をつなぐのか

**初学者**  
では integration test はどうでしょう。  
WordPress では何をつなぐ役割になるのですか。

**解説者**  
integration test は、ブロック、テーマ、保存処理、表示処理の接点を確かめる場面で重要です。  
個別要素は正しくても、つなぐと壊れることがある。  
WordPress ではまさにそこが多いです。

**実装者**  
たとえば、ブロック定義は正しい。  
テーマ側のスタイルも正しい。  
でも組み合わせると、保存 HTML が想定と少しずれて見た目が崩れる。  
あるいは、エディタではよく見えるのに公開画面では崩れる。  
こうした問題は統合点で起きます。  
integration test は、その継ぎ目を確認するために有効です。

**初学者**  
WordPress は継ぎ目が多いから、そこでの確認が大事なのですね。

**解説者**  
その通りです。  
そして WordPress の継ぎ目は、バックエンドとフロントエンドだけではありません。  
編集と保存、保存と表示、テーマ方針とブロック能力の継ぎ目もあります。  
だから integration test の意味も、少し広く捉える必要があります。

## なぜ E2E の比重が上がるのか

**初学者**  
では E2E テストは、かなり重要そうですね。

**解説者**  
はい。  
WordPress では、E2E の比重が相対的に上がりやすいです。  
なぜなら、実際の編集操作と保存と再表示の流れを通さないと見えない問題が多いからです。

**実装者**  
ブロックを追加する。  
属性を変える。  
入れ子にする。  
保存する。  
再読込する。  
もう一度編集する。  
公開画面で確認する。  
この一連の流れのどこかで壊れることがあります。  
特に Gutenberg やブロックテーマの問題は、途中だけ切り出すと見落としやすいです。

**初学者**  
確かに、それは E2E でないと見えにくそうです。

**解説者**  
その通りです。  
もちろん全部を E2E にするのは重いです。  
ですが WordPress では、E2E を軽視すると、本当に守りたいものが抜け落ちやすい。  
だから一般的なテストピラミッドをそのまま機械的に適用するより、E2E の重要度を高めに見る必要があります。

## なぜテストピラミッドをそのまま使いにくいのか

**初学者**  
今の話で気になりました。  
一般的には、ユニットテストを多く、E2E を少なくと言われますよね。  
WordPress ではそれがそのまま当てはまらないのですか。

**解説者**  
完全には当てはまりません。  
テストピラミッドの考え方自体は有用です。  
ただし WordPress では、重要な品質が UI・保存・再編集・表示の連鎖に宿るため、ユニットテストだけを厚くしても安心しにくいのです。

**実装者**  
特にブロックテーマ時代は、見た目と編集体験と制約が密接です。  
そのため、E2E や統合確認の価値が高くなります。  
ユニットテストが少なくてよいという話ではありません。  
むしろ、「どこに品質の本丸があるか」に合わせて配分を変えるべきなんです。

**初学者**  
つまり、テストピラミッドを否定するのではなく、守る対象に応じて重心を変えるのですね。

**解説者**  
その理解でよいです。  
WordPress のテストは、一般論を暗記するより、「何を守るシステムなのか」から逆算したほうがうまく設計できます。

## manual test はなぜ残るのか

**初学者**  
ここまで自動テストの話が多かったですが、manual test もかなり残りそうですね。

**解説者**  
はい。  
WordPress では manual test も重要です。  
特に編集体験の違和感、導線の自然さ、運用担当者が迷わないか、といった点は、完全には自動化しにくいです。  
また、テーマやサイトの性質によっては、見た目の印象差や編集負荷の違いも重要になります。

**実装者**  
実務では、「仕様上は正しいが、使うとつらい」という問題がよくあります。  
ボタンは押せる。  
保存もできる。  
でも更新担当者が迷う。  
期待した場所に設定がない。  
ちょっとした操作で意図が分からなくなる。  
こうした問題は manual test でないと拾いにくいです。

**初学者**  
なるほど。  
WordPress の品質には、人が触ったときの自然さも含まれるのですね。

**解説者**  
その通りです。  
だから WordPress のテスト設計では、自動テストと manual test を対立させるのではなく、守る対象ごとに役割分担させる必要があります。

## 実務での最小セットをどう考えるか

**初学者**  
実務では、理想的な全部盛りは難しいですよね。  
最小セットで考えるなら、どう整理すればよいでしょうか。

**解説者**  
最小セットでも、少なくとも次の発想は必要です。  
ひとつは、重要な保存 HTML の安定性を確認すること。  
ひとつは、主要な編集フローを E2E か手動で確認すること。  
ひとつは、テーマの制約が本当に効いているかを見ること。  
そして、切り出せるロジックにはユニットテストを当てることです。

**実装者**  
つまり、書きやすいテストから埋めるのではなく、壊れると困る契約から守る、という順番です。  
更新頻度が高い主要ブロック。  
運用上よく触るテンプレート。  
サイト全体の整合性に効く制約。  
ここを優先して押さえると、少ないコストでもかなり強くなります。

**初学者**  
全部を均等に守るのではなく、運用上の本丸から守るわけですね。

**解説者**  
その通りです。  
WordPress のテストは、テスト手法の一般論から入るより、守るべき編集可能な資産を見極め、その資産に対して適切な検証を割り当てるほうが実務的です。

## WordPress でテストするとは、運用可能性を守ることである

**初学者**  
最後に、この章全体をまとめるなら、WordPress のテストとは何なのでしょうか。

**解説者**  
一言で言えば、運用可能性を守ることです。  
ただし、その運用可能性は曖昧なものではありません。  
編集者が安全に触れ、保存結果が安定し、テーマの制約が効き、公開画面が崩れず、長期的に再編集可能であること。  
その全体を守るための営みが、WordPress のテストです。

**実装者**  
だから WordPress のテストは、コード品質の確認だけでは終わりません。  
編集可能な HTML 資産を、壊れにくく、説明可能で、保守しやすい形で維持するためのものです。  
その意味で、WordPress のテストは開発の後工程ではなく、設計思想の延長にあります。

**初学者**  
よく分かりました。  
テストは最後のチェックではなく、この本で言ってきた責務分界や制約設計を、本当に守れているか確かめるものなのですね。

**解説者**  
その通りです。  
だから WordPress でテストするとは何を守ることか、と問うこと自体に意味があります。  
守るべきものが見えていれば、テストの粒度も優先順位も自然に決まってくるからです。

### 整理

- WordPress のテストで守る対象は、コード単体ではなく、保存 HTML・編集体験・制約・表示資産の関係である。  
- テストの中心は、「このシステムが約束している編集・保存・表示の契約が破れていないか」を確認することにある。  
- 保存 HTML は、公開と再編集の基盤であり、重要なテスト対象である。  
- 編集体験も WordPress の品質の一部であり、操作可能性や壊れにくさも守るべき対象になる。  
- 制約設計を採用している以上、その制約が実際に効いているかもテストしなければならない。  
- pure unit test は局所ロジックに有効だが、WordPress の品質全体はそれだけでは守れない。  
- integration test と E2E test は、編集・保存・表示の継ぎ目を確認するうえで重要であり、特に E2E の比重が上がりやすい。  
- manual test も、編集の自然さや運用時の違和感を拾うために重要な役割を持つ。  

### この章の核心

WordPress でテストするとは、コードの正しさだけを確かめることではない。編集可能な HTML 資産が、制約付きで安全に保存され、再編集でき、表示され、長期運用に耐えることを守る営みである。

## まとめ

- WordPress のテストは、編集・保存・表示の契約を守るために行う。  
- 保存 HTML、編集体験、制約、表示資産は、切り離さずに考える必要がある。  
- ユニットテストは有効だが、それだけでは WordPress の品質を代表できない。  
- 統合テストと E2E テストは、継ぎ目と実運用フローを守るために重要である。  
- manual test も含めて、運用可能性を守る観点でテストを設計することが大切である。  

次章では、この構造理解とテスト観を、小規模制作会社の現場に引き寄せて考えます。  
求人票には書かれにくいが、Gutenberg 以降の WordPress を扱うために本当に必要な学習と責務分界を整理していきます。
