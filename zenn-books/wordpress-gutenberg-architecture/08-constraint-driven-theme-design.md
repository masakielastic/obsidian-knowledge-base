# 第7章 テーマ設計はなぜ制約設計なのか

**初学者**  
ここまで読んできて、ブロックテーマは「画面を描く」だけではなく、「編集可能な構造とその制約を設計するものだ」という話はだいぶ分かってきました。  
でも、テーマ設計をわざわざ「制約設計」と呼ぶのは少し大げさにも聞こえます。  
テーマはやはりデザインやレイアウトを作るものではないのですか。

**解説者**  
もちろん、デザインやレイアウトは重要です。  
ただ、ブロックテーマ時代になると、それだけではテーマの役割を説明しきれません。  
なぜなら、編集者がブロックを動かし、入れ替え、追加し、削除できる以上、テーマは完成画面だけでなく、「どこまで自由にしてよいか」を設計しなければならないからです。  
この章の中心命題はこうです。  
ブロックテーマ時代のテーマは、見た目を与えるだけのものではなく、編集可能な HTML 資産が運用で破綻しないように、自由の範囲を設計するものだ。だからテーマ設計は制約設計になるのです。

**実装者**  
実務では、この違いがかなり大きいです。  
初回納品時の画面がきれいでも、その後の更新で余白がばらつき、見出し階層が崩れ、導線が散らばり、担当者ごとにページの癖が変わることがあります。  
クラシックテーマ時代は、テーマ側が画面を強く握ることである程度それを防げました。  
でもブロックテーマ時代は、編集者が構造を動かせる。  
だから「どう描くか」だけでなく、「どう壊れにくくするか」を設計しないといけません。

**初学者**  
なるほど。  
テーマ設計が制約設計になるのは、編集の自由度が上がったからなのですね。

**解説者**  
その通りです。  
自由度が増えると、テーマは単なる見た目の提供者ではいられません。  
編集者が歩ける範囲を決めるガードレールの設計者になります。  
この章では、その意味を整理していきます。

## クラシックテーマでは、何を設計していたのか

**初学者**  
まず、比較のために確認したいです。  
クラシックテーマでは、テーマは何を設計していたのでしょうか。

**解説者**  
クラシックテーマでは、主に表示結果を設計していました。  
どのテンプレートがどの画面を担当し、どの HTML を出し、どの CSS で整えるか。  
もちろん `functions.php` やフックによる拡張もありますが、中心は「最終的にどう表示されるか」です。

**実装者**  
開発の感覚としても、クラシックテーマはかなり完成画面寄りです。  
固定ページテンプレート、投稿一覧、個別ページ、サイドバー、ヘッダー、フッターをどう分けるか。  
それぞれで何を出すか。  
管理画面の入力はあっても、最終的な画面はテーマ側がかなり強く制御できます。  
だからテーマ設計は、表示設計やレイアウト設計として理解しやすかったんです。

**初学者**  
つまり昔のテーマは、「こう見せる」を決める力が強かったのですね。

**解説者**  
そうです。  
もちろん完全に固定ではありませんが、編集者の自由度は現在ほど高くありませんでした。  
そのため、テーマの品質も主に「どれだけきれいに描けるか」「必要な画面をどれだけ漏れなく作れるか」で評価しやすかったのです。

## ブロックテーマでは、何が変わったのか

**初学者**  
では、ブロックテーマで何が変わったのでしょうか。

**解説者**  
一番大きいのは、テーマが完成画面だけでなく、編集過程の空間にも関わるようになったことです。  
ブロックテーマでは、テンプレートやテンプレートパーツがブロックで構成され、Site Editor で扱われます。  
編集者は本文だけでなく、サイト全体の構造にも触れられるようになります。  
するとテーマは、「完成形を出すもの」から、「編集可能な構造の範囲を定義するもの」へと性格を変えます。

**実装者**  
実務ではここが本当に大きいです。  
たとえばヘッダーやフッターをコードだけで固定していたときは、編集者が壊せる範囲は限定的でした。  
でも Site Editor でテンプレートやテンプレートパーツを扱えると、便利になる一方で、サイト全体の一貫性が崩れる余地も増えます。  
だからブロックテーマは、単に編集可能にするだけでなく、「どこまで触らせるか」を設計しないといけないんです。

**初学者**  
つまり、テーマが相手にする対象が、完成画面から編集空間へ広がったわけですね。

**解説者**  
その通りです。  
そして、編集空間を設計するということは、自由と制約の設計をするということでもあります。  
ここからテーマ設計は、自然に制約設計へ近づいていきます。

## なぜ自由を増やすだけではだめなのか

**初学者**  
でも、ブロックテーマの魅力は自由に編集できることですよね。  
なら、自由をできるだけ増やしたほうがよいのではないですか。

**解説者**  
そこが、もっとも誤解されやすい点です。  
編集者の自由度は価値があります。  
しかし、自由はそのままでは品質を保証しません。  
むしろ、自由度が高いほど、ばらつきや事故や再現性の低さが生まれやすくなります。

**実装者**  
現場ではよくあります。  
最初は「好きに更新できるほうが便利」と思っていても、数か月後にはページごとに余白の取り方が違う、見出しの階層が不揃い、画像サイズの扱いがばらばら、強調表現の癖が担当者ごとに違う、という状態になります。  
すると、更新は自由でも、サイト全体としては読みづらく、直しにくく、保守しづらくなります。

**初学者**  
自由に編集できることと、うまく運用できることは別なのですね。

**解説者**  
その通りです。  
テーマ設計で本当に必要なのは、「無制限の自由」ではありません。  
「安全に運用できる自由」です。  
そのためには、どこに自由を与え、どこで止めるかを設計する必要があります。  
この“止め方”が、制約設計です。

## 制約は何のためにあるのか

**初学者**  
制約という言葉に、まだ少し抵抗があります。  
制約は不便さの原因にも聞こえるからです。

**解説者**  
その感覚は自然です。  
ですが、ここで言う制約は、編集者を困らせるためのものではありません。  
運用を安定させ、品質を守り、判断コストを減らすためのものです。

**実装者**  
実務で考えると分かりやすいです。  
たとえば、色の選択肢が無限にあると、毎回「どの色を使うべきか」を編集者が判断しなければなりません。  
余白も自由、タイポグラフィも自由、レイアウトも自由だと、そのたびに小さな設計判断が発生します。  
でも実際の運用では、そうした判断を毎回ちゃんとできるとは限りません。

**初学者**  
確かに。  
自由が多いほど、毎回考えることも増えますね。

**解説者**  
そうです。  
制約は、自由を削るだけでなく、判断負荷を減らします。  
そしてサイト全体に一貫性を与えます。  
つまり制約は、運用のためのガードレールであり、品質を守るための共通ルールなのです。

## テーマは「壊れない編集環境」を設計する

**初学者**  
ここまで聞くと、テーマは見た目を作るというより、更新しても壊れない環境を作るものに思えてきました。

**解説者**  
その理解はかなり本質に近いです。  
ブロックテーマ時代のテーマは、完成画面のデザインだけでなく、「編集しても壊れにくい環境」を設計する必要があります。  
壊れないとは、単にエラーが出ないという意味ではありません。  
見た目の整合性、情報設計の一貫性、編集者の迷いにくさ、保守のしやすさを含んだ意味です。

**実装者**  
たとえば、更新担当者が毎回デザイナーのように余白や階層を設計しないといけないテーマは、見た目がきれいでも運用では弱いです。  
逆に、ある程度のルールが埋め込まれていて、自然に揃ったページを作りやすいテーマは強い。  
だから実務でよいテーマというのは、派手な自由度よりも、壊れにくさの設計がうまいことが多いです。

**初学者**  
テーマの品質を、初回のデザインだけでなく、更新後の状態でも見る必要があるのですね。

**解説者**  
その通りです。  
ブロックテーマ時代の品質評価は、表示品質だけでは足りません。  
編集品質と運用品質も見なければなりません。  
テーマ設計を制約設計として捉えるのは、そのためです。

## ハード制約・ソフト制約・誘導

**初学者**  
制約にもいろいろありそうです。  
全部を禁止で固めるわけではないですよね。

**解説者**  
はい。  
制約には強さの違いがあります。  
ここでは、ハード制約・ソフト制約・誘導という三つに分けると分かりやすいです。

**初学者**  
それぞれどう違うのでしょうか。

**解説者**  
ハード制約は、「できないようにする」制約です。  
たとえば、使える色を絞る、特定のレイアウト変更を許さない、サイト共通部分を不用意に崩せないようにする、といったものです。  
これは事故防止に強いです。

ソフト制約は、「できるが、ある程度のルールや既定値で揃える」制約です。  
完全に禁止はしないものの、自然に望ましい方向へ寄るようにする。  
既定の余白やスタイル選択肢、テンプレートの初期構成などがこれに近いです。

誘導は、「こうするとよい」という道筋を先に用意するやり方です。  
禁止ではなく、推奨される構造をすぐ使えるようにして、結果としてばらつきを減らします。

**実装者**  
現場では、この三つを混ぜて使います。  
全部をハード制約にすると窮屈になりすぎますし、全部を誘導だけで済ませると守られません。  
どこは絶対に守るべきで、どこは柔軟でもよくて、どこはサンプルで導けばよいのか。  
その見極めがテーマ設計の腕の見せ所です。

**初学者**  
制約というと全部禁止に聞こえていましたが、実際には濃淡があるのですね。

**解説者**  
そうです。  
テーマ設計が制約設計だというのは、何もかも縛るという意味ではありません。  
適切な強さで自由を整える、という意味です。

## patterns はなぜ重要なのか

**初学者**  
誘導の話で言うと、patterns の役割が大きそうですね。

**解説者**  
その通りです。  
patterns は、制約設計における誘導の代表例です。  
編集者がゼロから自由に組み立てるのではなく、「こういう単位で作ると整いやすい」という初期形を先に渡せます。  
これは、自由を残しながら品質を揃えるための強い手段です。

**実装者**  
実務でも、patterns はかなり効きます。  
見出し、本文、画像、CTA、問い合わせ導線などを、よく使う組み合わせで用意しておけば、更新担当者はそれを起点に編集できます。  
すると、毎回ゼロから構成を考える必要がなくなり、結果としてサイト全体の構造が揃いやすくなります。

**初学者**  
つまり patterns は便利機能というだけでなく、テーマの思想を編集者に渡す仕組みでもあるのですね。

**解説者**  
とてもよい言い方です。  
patterns は単なる時短機能ではありません。  
テーマが意図する構造、導線、情報のまとまり方を、編集体験の中で共有する仕組みです。  
だから制約設計の中でも、特に重要な役割を持ちます。

## `theme.json` は制約設計の中心にある

**初学者**  
ここで改めて `theme.json` の意味が見えてきました。  
あれはまさに制約設計のための仕組みなのですね。

**解説者**  
そうです。  
`theme.json` は、テーマが編集空間に対して与える方針と制約を表します。  
色、タイポグラフィ、余白、レイアウト、既定値などを通じて、「このサイトでは何をどこまで許すか」を定義します。  
だから `theme.json` は、単なる設定ファイルではなく、テーマの設計思想の一部です。

**実装者**  
クラシックテーマ時代は、最終的な見た目を CSS で整える発想が強かったです。  
でもブロックテーマ時代は、それだけでは遅い。  
編集段階で余計な選択肢を減らし、自然に揃う環境を作る必要があります。  
`theme.json` は、そのための中心的な道具です。

**初学者**  
事後調整ではなく、事前設計が大事になるわけですね。

**解説者**  
その通りです。  
テーマ設計が制約設計になるのは、完成後の修正より、編集前提のルール作りのほうが重要になるからです。

## UI 設計と制約設計はどう違うのか

**初学者**  
でも、ここまでの話だと UI 設計とかなり近いようにも思えます。  
UI 設計と制約設計は、どう違うのでしょうか。

**解説者**  
重なる部分はありますが、同じではありません。  
UI 設計は、見た目や操作感、情報の見せ方を設計します。  
一方、制約設計は、何ができて何ができないか、どこまで変更を許し、どこから先は揃えるかを設計します。  
UI は表現の設計、制約は行動可能範囲の設計です。

**実装者**  
実務では、UI だけきれいでも、制約が弱いと運用で崩れます。  
逆に、多少地味でも、制約がうまく効いているテーマは長持ちします。  
もちろん理想は両方そろうことです。  
でもブロックテーマ時代に見落とされがちなのは後者です。

**初学者**  
つまり、UI 設計は「どう見えるか」、制約設計は「どう振る舞えるか」に寄っているのですね。

**解説者**  
その整理でよいです。  
そしてブロックテーマでは、後者を設計しないと前者も守れません。  
だからテーマ設計の核心が、制約設計へ寄っていくのです。

## 小規模サイトでは、なぜ制約設計が特に重要なのか

**初学者**  
小規模サイトでは、そこまで厳密な設計は要らないようにも思えます。  
実は逆なのでしょうか。

**解説者**  
むしろ逆です。  
小規模サイトほど、制約設計が重要になることが多いです。  
なぜなら、専任のデザイナーや開発者が毎回更新に立ち会うわけではなく、運用担当者が限られた時間で更新することが多いからです。

**実装者**  
現場感覚でもそうです。  
企業サイト、店舗サイト、地域団体のサイトなどでは、更新担当者が Web 専門職ではないことが珍しくありません。  
そのとき、自由すぎる編集環境は助けにならず、負担になります。  
逆に、「この範囲で触れば大丈夫」「このパターンを使えば崩れにくい」というテーマは、とても強いです。

**初学者**  
なるほど。  
小規模サイトでは、自由より安心のほうが価値になる場面が多いのですね。

**解説者**  
そうです。  
だから小規模サイトにおけるよいテーマ設計とは、凝った自由度の高さではなく、壊れにくく、迷いにくく、長く使えることにあります。  
これはまさに制約設計の仕事です。

## テーマ設計は「編集者に何を委ねるか」を決める仕事である

**初学者**  
ここまでの話をまとめると、テーマ設計は「何を見せるか」より、「何を委ねるか」を決める仕事に思えてきました。

**解説者**  
そのまとめはとてもよいです。  
ブロックテーマ時代のテーマ設計は、完成画面を一方的に固定する仕事ではありません。  
編集者に委ねる範囲と、テーマが守る範囲を切り分ける仕事です。  
つまり、責務分界の設計です。

**実装者**  
そして、その責務分界は抽象論ではなく、実際の設定やテンプレートや patterns に落ちます。  
どこを固定するか。  
どこを選べるようにするか。  
どこは推奨だけにするか。  
この判断の積み重ねがテーマになります。

**初学者**  
テーマが単なる見た目の皮ではなく、運用ルールの実装でもある、ということですね。

**解説者**  
その通りです。  
だからテーマ設計は制約設計なのです。  
制約とは、創造性の敵ではありません。  
編集可能な HTML 資産を、安全に、長期にわたって扱うための、設計された自由の形なのです。

### 整理

- クラシックテーマは主に完成画面の表示結果を設計していた。  
- ブロックテーマでは、テーマが編集空間にも関わるため、完成画面だけでなく編集可能範囲を設計する必要がある。  
- 自由度を増やすだけでは、運用でばらつきや事故が起きやすくなる。  
- 制約は不便のためではなく、品質維持・判断負荷の削減・長期運用の安定化のためにある。  
- 制約には、ハード制約、ソフト制約、誘導という強さの違いがある。  
- patterns は、望ましい構造へ導く誘導として、制約設計の重要な手段である。  
- `theme.json` は、テーマが編集空間に与える方針と制約を表す中心的な仕組みである。  
- ブロックテーマ時代のテーマ設計は、「何を描くか」だけでなく、「何を編集者に委ね、何を守るか」を決める責務分界の設計である。  

### この章の核心

ブロックテーマ時代のテーマは、完成画面を作るだけでは足りない。編集者が安全に更新でき、サイト全体の整合性が壊れにくいように、自由の範囲とガードレールを設計する必要がある。だからテーマ設計は、見た目の設計であると同時に、制約設計でもある。

## まとめ

- ブロックテーマでは、テーマの設計対象が完成画面から編集空間へ広がった。  
- 編集自由度が高まるほど、テーマは自由の範囲を設計しなければならない。  
- 制約は不便のためではなく、品質と運用を守るためのガードレールである。  
- ハード制約、ソフト制約、誘導を組み合わせることが重要である。  
- テーマ設計の本質は、見た目だけでなく、編集者に何を委ね、何を守るかを決めることにある。  

次章では、この制約設計の発想をテストの話へつなげます。  
WordPress でテストするとは何を守ることなのかを、コード単体ではなく、保存 HTML・編集体験・制約・表示資産の関係から整理していきます。
