# はじめに

**初学者**  
WordPress は、ずっと身近にある技術でした。  
触ったことがある人も多いですし、テーマやプラグインの話もよく見かけます。  
でも、いざ「WordPress はどういうシステムなのか」と説明しようとすると、急に言葉に詰まる気がします。  
特に Gutenberg 以降は、前より便利になったはずなのに、かえって説明しにくくなったようにも感じます。

**解説者**  
その感覚は、とても自然です。  
この本は、まさにその「説明しにくさ」から出発します。  
WordPress は、単なる古い CMS でも、単なるテーマ開発の対象でもありません。  
Gutenberg 以降の WordPress は、編集、保存、表示、制約、運用が重なり合ったシステムです。  
しかも、その重なり方が、従来の WordPress の説明枠だけでは捉えにくくなっています。

**実装者**  
実務で触っていると、その難しさはよく分かります。  
テーマは触れる。  
プラグインも入れられる。  
必要ならコードも書ける。  
でも、ブロックテーマとは何を設計しているのか、`theme.json` は何を定義しているのか、なぜ PHP だけでなく JavaScript の理解も必要なのか、と聞かれると、一気に話が難しくなる。  
これは知識が足りないだけではなく、構造を整理する地図が不足しているからです。

## なぜいま WordPress をアーキテクチャとして学ぶのか

**初学者**  
WordPress の本というと、使い方やテーマ開発の手順を教える本が多い印象があります。  
それでも十分ではないのでしょうか。

**解説者**  
もちろん、それらの本には大きな価値があります。  
ですが、この本が扱いたいのは、手順そのものではありません。  
WordPress を、設計思想と責務分界を持つシステムとして理解することです。  
つまり、「どう使うか」より先に、「これは何を引き受けるシステムなのか」を問いたいのです。

**実装者**  
特に Gutenberg 以降は、その問いが重要になりました。  
クラシック WordPress では、テンプレート階層、`functions.php`、フックという見取り図でかなりの部分を説明できました。  
しかし今は、ブロック、エディタの状態、保存 HTML、テーマの制約設計、編集体験まで含めて考えないと全体像が見えません。  
だから WordPress をただの CMS 操作対象としてではなく、アーキテクチャとして学ぶ必要があります。

**初学者**  
なるほど。  
いまの WordPress は、昔の延長だけで説明しきれないからこそ、構造として学ぶ必要があるのですね。

**解説者**  
その通りです。  
しかも、これは一部の高度な開発者だけの話ではありません。  
小規模制作会社、受託開発、企業サイト運用、地域の事業者サイトのような現場でも、どこに何の責任があるのかを分かっているかどうかで、設計も学習も大きく変わります。  
だから本書は、WordPress を「説明できるようになる」ための本として書かれています。

## 「使える」と「説明できる」の違い

**初学者**  
「説明できるようになる」というのは、この本の大事な目標なのですね。

**解説者**  
はい。  
WordPress は、使える人が多い技術です。  
しかし、使えることと説明できることは同じではありません。  
管理画面を操作できる。  
既存テーマを修正できる。  
必要なプラグインを導入できる。  
そうした実務能力は大切です。  
けれど、それだけでは Gutenberg 以降の WordPress の構造を語るのは難しい。

**実装者**  
実際の現場でも、そこにギャップがあります。  
日々の仕事は回っているのに、なぜ今の WordPress がこうなっているのか、どこをテーマで持ち、どこをブロックで持ち、どこを運用ルールで吸収するのか、といった話になると整理しにくい。  
つまり、断片知識はあっても、全体地図がない状態です。

**初学者**  
この本は、その地図を作るための本なのですね。

**解説者**  
その通りです。  
本書の狙いは、個別の API や設定項目を網羅することではありません。  
WordPress を構造として説明できるようにし、断片知識を比較可能な地図へ変えることです。

## この本で扱うこと、扱わないこと

**初学者**  
では、この本は具体的に何を扱うのでしょうか。

**解説者**  
中心になるのは、次のような問いです。  
クラシック WordPress から Gutenberg へ何が変わったのか。  
ブロックテーマは何を設計対象にしたのか。  
Web Components、React、SPA、Headless CMS と比べたとき、WordPress はどこに位置づくのか。  
`block.json` と `theme.json` は何を定義しているのか。  
テーマ設計はなぜ制約設計なのか。  
WordPress でテストするとは何を守ることなのか。  
そして、こうした理解が小規模制作会社の現場や長期運用にどうつながるのか。  
本書は、これらを一つの流れとして整理します。

**実装者**  
逆に、扱わないこともあります。  
WordPress 完全初心者向けの操作説明。  
テーマ開発やプラグイン開発の手順を一から網羅するハンズオン。  
React や Web Components の入門書的な説明。  
あるいは特定のホスティングや製品比較を中心にした話です。  
それらは大事ですが、この本の主役ではありません。

**初学者**  
つまり、この本は入門手順書ではなく、構造理解の本なのですね。

**解説者**  
はい。  
本書の関心は、「どうやれば動くか」だけではなく、「なぜそのような設計になっているのか」にあります。

## この本の想定読者

**初学者**  
どんな読者に向いた本なのでしょうか。

**解説者**  
主な想定読者は、WordPress のテーマやプラグインに少し触れたことがある人です。  
PHP と JavaScript の基本構文は読める。  
MVC や DDD という言葉にも強い抵抗はない。  
けれど、Gutenberg 以降の WordPress を構造として説明するのは難しい。  
そんな読者を想定しています。

**実装者**  
小規模 Web 制作会社や受託案件の現場にいる人にも向いています。  
「テーマが触れる」「プラグインが書ける」という実務経験はある。  
でも、それだけでは今の WordPress を説明しきれないと感じている。  
あるいは、求人票の言葉より一段深い理解を身につけたい。  
そういう人にちょうどよいはずです。

**初学者**  
逆に、完全初心者だと少し難しいかもしれませんね。

**解説者**  
そうかもしれません。  
この本は操作入門ではないので、管理画面の基本や PHP・JavaScript の最初の文法からは始めません。  
ただし、専門用語を振り回す本にもしたくありません。  
できるだけ、構造理解に必要な範囲で言葉を置き、その場で整理しながら進めていきます。

## この本の読み方

**初学者**  
どう読めばよいでしょうか。  
最初から順に読むべきですか。

**解説者**  
基本的には、最初から順に読むのがよいです。  
本書は、問題提起から始まり、比較、抽象化、設計、実務、評価という流れで組んでいます。  
前の章で置いた区別が、後の章の前提になるからです。  
特に、クラシック WordPress と Gutenberg の違い、ブロックテーマ、`block.json` と `theme.json`、制約設計の話は連続しています。

**実装者**  
ただし、全部を一度で理解する必要はありません。  
本書は、知識を一気に詰め込むための本というより、見方を更新するための本です。  
最初はぼんやりしていてもかまいません。  
実際のテーマやブロックや管理画面に触れながら、あとで戻って読むと見え方がかなり変わるはずです。

**初学者**  
なるほど。  
一回で全部覚える本ではなく、観察の道具として読む本なのですね。

**解説者**  
その通りです。  
この本で扱う考え方は、実際の WordPress を見るためのレンズです。  
レンズを持ったうえでテーマやブロックや編集体験を見ると、これまで見えにくかった責務分界がかなりはっきりしてきます。

## 本書の立場

**初学者**  
最後に、この本の立場を確認したいです。  
WordPress を擁護する本なのか、批判的に見る本なのか、どちらなのでしょうか。

**解説者**  
どちらか一方ではありません。  
本書は、WordPress を礼賛するための本でも、時代遅れだと断じるための本でもありません。  
目指しているのは、WordPress を適切に評価できるようにすることです。  
そのために、まず構造を明らかにします。  
構造が見えれば、強みも制約も、どの需要に向いていてどこで無理が出るかも、かなり落ち着いて議論できるようになります。

**実装者**  
実務では、雑な評価ほど危険です。  
「古いからだめ」「モダンだから正しい」という言い方では、現場の課題は片づきません。  
大事なのは、何を最適化しているのか、どこに責任を置いているのか、どこが壊れやすいのかを見ることです。  
本書は、そのための土台を作るつもりで書かれています。

**初学者**  
分かりました。  
この本は、WordPress を好き嫌いで語る前に、まずちゃんと説明できるようになるための本なのですね。

**解説者**  
その通りです。  
では次章から、なぜ Gutenberg 以降の WordPress は説明しにくいのかを、順番に見ていきましょう。  
そこから、この本全体の問題設定がはっきりしてきます。

### この本の核心

この本は、WordPress を「テーマやプラグインの使い方の寄せ集め」としてではなく、「編集可能な HTML 資産を、制約付きで安全に構築し、長期運用するシステム」として理解するための本です。目的は、断片的な知識を増やすことではなく、WordPress を構造として説明できる地図を手に入れることにあります。

## まとめ

- 本書の目的は、WordPress の操作や機能を網羅することではなく、設計思想と責務分界を構造として理解することにある。  
- Gutenberg 以降の WordPress は、従来の説明枠だけでは捉えにくくなっており、構造として学ぶ必要がある。  
- 本書は、クラシック WordPress から Gutenberg、ブロックテーマ、比較、制約設計、テスト、実務、評価までを一つの流れとして扱う。  
- 想定読者は、WordPress に少し触れたことがあり、PHP と JavaScript の基本が読めるが、現在の WordPress を説明しにくいと感じている人である。  
- 本書は、WordPress を礼賛したり否定したりするためではなく、適切に評価し説明できるようになるための地図を提供する。
