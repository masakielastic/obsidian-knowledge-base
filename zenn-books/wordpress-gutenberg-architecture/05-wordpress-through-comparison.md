# 第4章 比較で見る WordPress の位置づけ

**初学者**  
ここまでで、Gutenberg 以降の WordPress は、クラシックな CMS の延長でありながら、ブロックや制約設計を前に出したシステムだ、という話がありましたよね。  
でも、そう言われてもまだ少し輪郭がぼんやりしています。  
結局 WordPress は、React や Web Components や Headless CMS と比べて、どういう立ち位置なのでしょうか。

**解説者**  
それを整理するのが、この章です。  
WordPress 単体だけを見ていると、何が特殊で、何が一般的なのかが見えにくくなります。  
だから比較が必要です。  
ただし、この章で大事なのは、比較で優劣を決めることではありません。比較を通じて、WordPress が何を設計対象にし、何を最適化しているのかを明らかにすることです。

**実装者**  
実務でも、「WordPress は古い」「React で作ればよい」「いまは Headless CMS の時代だ」といった話はよく出ます。  
でも、その手の議論は、比較軸が曖昧なまま進みがちです。  
更新主体は誰なのか。何を長く運用したいのか。HTML をどれだけ資産として重く見るのか。編集体験をどこに置くのか。  
そこを分けないと、比較がそのまま雑な優劣論になります。

**初学者**  
なるほど。  
この章は「どれが最新か」を決める章ではなく、「WordPress は何者なのか」を比較で見えるようにする章なのですね。

**解説者**  
その通りです。  
この章の中心命題はこうです。  
WordPress は、SPA や Headless CMS や Web Components の代用品ではなく、編集可能な HTML 資産を長期運用するための、独自の責務分界を持つシステムである。  
だから比較は、置き換え候補を探すためだけでなく、責務の違いを見抜くために使う必要があります。

## 比較はなぜ必要なのか

**初学者**  
そもそも、なぜ比較がそんなに大事なのでしょうか。  
WordPress 自体を丁寧に見ていけば十分ではないですか。

**解説者**  
WordPress だけを見ていると、内部の用語や慣習に慣れてしまって、その仕組みが何を前提にしているのかが見えにくくなるからです。  
比較を入れると、「ここは一般的な Web アプリに近い」「ここは CMS としてかなり独特だ」「ここは React 的だが、ここはまったく違う」といった輪郭が出てきます。

**実装者**  
たとえば Gutenberg を触っていると、React ベースの UI に見える部分があります。  
でも、保存の中心に HTML があり、再編集可能性を維持し、テーマや運用者との関係まで含めて考えると、ただの React アプリとはかなり違います。  
比較しないと、その違いを言葉にしにくいんです。

**初学者**  
比較しないと似ているところだけ目につき、比較すると違うところも見えるわけですね。

**解説者**  
その通りです。  
そして本書では、比較を WordPress 理解に回収することが重要です。  
React の説明をするために React を出すのではありません。  
Web Components の入門をするために比較するのでもありません。  
WordPress の責務分界を見やすくするために比較するのです。

## Web Components と比べると何が見えるか

**初学者**  
では、まず Web Components との比較からお願いします。  
ブロックはコンポーネントっぽいので、近そうに見えます。

**解説者**  
たしかに近く見える部分はあります。  
どちらも、画面を部品単位で組み立てる発想を持っています。  
再利用可能な単位を作り、それを組み合わせて全体を構成するという点では、ブロックとコンポーネントは直感的に似ています。

**実装者**  
ただし、役割はかなり違います。  
Web Components は、主にブラウザ上の UI 部品としての独立性を重視します。  
一方 WordPress のブロックは、UI 部品であると同時に、編集単位であり、保存単位であり、再編集可能なコンテンツ構造でもあります。  
つまりブロックは、表示だけでなく編集と保存までまたぐ単位です。

**初学者**  
Web Components は表示側の部品で、ブロックはもっと広い責務を持っているのですね。

**解説者**  
そうです。  
Web Components は、部品のカプセル化や再利用性の整理には向いています。  
しかし WordPress のブロックが扱っているのは、部品の表示だけではありません。  
編集者がどう触るか、保存時にどういう構造になるか、テーマの制約とどう噛み合うかまで含んでいます。  
だから「ブロックは Web Components みたいなもの」と言うと、半分しか当たっていません。

**実装者**  
もう少し強く言えば、Web Components は UI 実装の単位であり、WordPress のブロックは CMS の編集・保存契約の単位です。  
この違いは大きいです。  
前者はブラウザの部品化に寄り、後者はコンテンツ運用の構造化に寄ります。

**初学者**  
なるほど。  
同じ「部品化」でも、何を守ろうとしているかが違うのですね。

## なぜ Gutenberg は React を必要としたのか

**初学者**  
では React との比較はどうでしょう。  
Gutenberg は React 的だとよく言われますよね。

**解説者**  
はい。  
Gutenberg が React 的に見えるのは自然です。  
ブロックの追加、移動、入れ子、選択状態、各種 UI の切り替え、履歴管理など、状態を持つ複雑な編集画面を扱うからです。  
こうした問題は、宣言的 UI と状態管理の考え方と相性がよい。  
だから React 的なアプローチが必要になりました。

**実装者**  
実際、Gutenberg のエディタは、昔ながらのフォーム入力画面というより、状態を持つアプリケーションです。  
どのブロックが選択されているか。  
どの属性が変更されたか。  
今の編集中の構造は何か。  
Undo / Redo をどう扱うか。  
こうした問題は、PHP テンプレート中心の発想だけではうまく整理しにくいです。

**初学者**  
つまり React は、見た目を今風にするためではなく、エディタをアプリとして成立させるために必要だったのですね。

**解説者**  
その通りです。  
ここで重要なのは、React が WordPress の目的そのものではないことです。  
React は、編集体験を実現するための有力な手段です。  
しかし WordPress の中心目的は、React アプリを作ることではありません。  
編集可能なコンテンツを扱い、それを保存し、長期運用できるようにすることにあります。

**実装者**  
この違いを見失うと、「Gutenberg って結局 React でしょ」と言いたくなります。  
でも実際には違います。  
React はエディタを支える実装上の重要技術ですが、WordPress 全体の責務を代表しているわけではありません。

## なぜ WordPress は SPA になりきらないのか

**初学者**  
エディタがアプリ化しているなら、WordPress 全体も SPA に寄せればよいように思えます。  
それでもそうなっていないのはなぜでしょうか。

**解説者**  
WordPress が最適化している対象が、純粋な SPA とは違うからです。  
SPA は、クライアント側で状態を持ち、画面遷移をアプリケーションとして滑らかに扱うのに向いています。  
一方 WordPress は、編集体験の一部でアプリ化を取り入れながらも、保存された HTML やテーマによる表示、既存の公開基盤との互換を重視しています。

**実装者**  
言い換えると、WordPress は編集画面ではかなりアプリ的です。  
でも公開側まで完全に SPA に振り切ると、WordPress が長く最適化してきたものとずれてきます。  
たとえば、サーバーサイドでの安定した表示、HTML 資産としての扱いやすさ、テーマ文化、キャッシュ戦略、既存運用との接続です。

**初学者**  
つまり、エディタはアプリであっても、公開の思想は必ずしも SPA にしたいわけではないのですね。

**解説者**  
その通りです。  
WordPress は、編集のためにはアプリケーション的な構造を取り入れました。  
しかし公開の中心にあるのは、依然として HTML 資産です。  
だから WordPress は SPA の要素を取り込みつつも、SPA そのものにはなりきらない。  
ここに WordPress の中間的で独特な位置があります。

**実装者**  
この中間性は、弱さではなく設計上の選択です。  
すべてをクライアントアプリ化することよりも、編集と公開と運用のバランスを取ることを選んでいるんです。

## Headless CMS と WordPress の違い

**初学者**  
では Headless CMS との違いはどうでしょう。  
「今は Headless のほうがモダンだ」と言われることもありますよね。

**解説者**  
Headless CMS は、コンテンツ管理と表示を切り離す設計です。  
コンテンツは API 経由で配信され、表示側は別のフロントエンドが担当します。  
これは非常に明快な責務分離です。  
一方 WordPress は、少なくとも標準的な使い方では、編集・保存・表示がもっと密接に結びついています。

**実装者**  
Headless CMS では、表示側の自由度が高いです。  
React でも Vue でも、ネイティブアプリでも、別のシステムからでも取りにいける。  
その代わり、表示責務は完全に別途設計しなければなりません。  
WordPress は逆に、編集と表示が分断されていないぶん、HTML 資産としての一体的な運用がしやすいです。

**初学者**  
つまり Headless CMS は分離が強く、WordPress は一体性が強いのですね。

**解説者**  
そう整理すると分かりやすいです。  
ただし、ここでも単純な優劣にはなりません。  
分離が強いと柔軟性は上がりますが、責任も分かれ、設計コストも増えます。  
一体性が強いと柔軟性は絞られますが、編集から公開までを一つの運用としてまとめやすい。  
WordPress は後者に強いシステムです。

**実装者**  
特に小規模サイトや、編集者が最終公開物を意識しながら更新する現場では、この一体性がかなり効きます。  
コンテンツモデルを API のために抽象化しすぎるより、保存された HTML を含む公開物として見たほうが扱いやすいことが多いです。

## HTML を真実とする世界と JSON を真実とする世界

**初学者**  
この違いは、「何を真実のデータとみなすか」にも関係しそうですね。

**解説者**  
まさにそこです。  
Headless CMS や多くのアプリケーション設計では、JSON 的な構造データが中心になります。  
表示はそのデータをどう解釈するかの問題です。  
一方 WordPress は、Gutenberg 以降も、保存された HTML をかなり重いものとして扱います。

**実装者**  
ここは誤解されやすいところです。  
Gutenberg には editor state があり、ブロック構造もあります。  
それでも WordPress は、保存時に HTML を強く重視しています。  
だから WordPress は、JSON 中心世界に全面移行したわけではありません。  
むしろ、アプリケーション的な編集体験と HTML 資産の保存をつなぐ設計をしているんです。

**初学者**  
すると WordPress は、HTML を古いものとして捨てていないのですね。

**解説者**  
捨てていません。  
むしろ HTML を、表示結果であると同時に、長期運用される資産として扱っています。  
ここが WordPress の強みでもあり、説明しにくさの原因でもあります。  
純粋な JSON 世界でもなければ、純粋なテンプレート世界でもないからです。

**実装者**  
この点は、WordPress を過小評価しないためにも重要です。  
HTML を重視することは、単に古いだけではありません。  
編集者にとっての可視性、公開物としての安定性、既存の Web 基盤との相性、長期運用のしやすさといった価値があるんです。

## React や Headless CMS の代わりではなく、別の責務分界を持つ

**初学者**  
ここまで聞くと、WordPress は React や Headless CMS に負けているというより、そもそも役割が違うように思えてきました。

**解説者**  
その理解が重要です。  
もちろん、競合する局面はあります。  
同じサイト制作の候補として比べられることもあります。  
しかし、責務分界が違う以上、完全な一対一の置き換えとして見るのは雑です。

**実装者**  
React は UI アプリケーションを組み立てるための強力な道具です。  
Headless CMS はコンテンツ管理と配信の責務を強く分けるやり方です。  
WordPress は、編集者が直接扱うコンテンツ構造と、テーマによる表示と、保存された HTML 資産を、一つの運用系としてまとめる力が強いです。  
だから比較するなら、「どちらが新しいか」ではなく、「どこで責任を切るか」を見るべきです。

**初学者**  
責任を切る位置が違うから、向いている現場も違うのですね。

**解説者**  
その通りです。  
WordPress は、編集・表示・公開・運用の責務を適度に一体化したい場面で強い。  
一方で、表示を完全に別のアプリケーションとして設計したいなら、Headless のほうが自然なこともあります。  
つまり違いは、時代の前後ではなく、責務分界の設計思想にあります。

## WordPress をどう位置づければよいのか

**初学者**  
では最後に、WordPress の位置づけを一言で言うなら、どうなりますか。

**解説者**  
一言で言うならこうです。  
WordPress は、編集可能な HTML 資産を、制約付きで安全に構築し、長期運用するためのシステムです。  
Gutenberg 以降は、そのためにエディタをアプリ化し、ブロックという構造単位を導入しました。  
しかし最終的な目的は、フロントエンド技術の流行を追うことではなく、編集と公開の運用を成立させることにあります。

**実装者**  
だから WordPress は、Web Components でも、React アプリでも、Headless CMS でもありません。  
ただし、それらの一部の発想を取り込みながら、自分の責務分界を作っています。  
この“似ているが同じではない”という位置が、WordPress を難しくもし、面白くもしています。

**初学者**  
よく分かってきました。  
比較すると、WordPress が中途半端なのではなく、別の最適化を選んでいると見えてくるのですね。

**解説者**  
その通りです。  
そして、この見方がないと、「WordPress は古い」「React のほうが新しい」「Headless のほうが正しい」といった雑な議論に流れやすい。  
この章でやりたかったのは、その雑さを避けるための比較軸を用意することです。

### 整理

- Web Components は主に UI 部品の独立性と再利用性を重視するが、WordPress のブロックは編集・保存・再編集までまたぐ単位である。  
- Gutenberg が React を必要としたのは、見た目を今風にするためではなく、状態を持つ編集アプリを成立させるためである。  
- WordPress は編集画面ではアプリ化を進めたが、公開側では HTML 資産とテーマによる表示を重視しており、SPA にはなりきらない。  
- Headless CMS はコンテンツ管理と表示を強く分離するが、WordPress は編集から公開までの一体運用に強い。  
- Headless 的な世界では JSON 的構造データが中心になりやすいのに対し、WordPress は保存された HTML を重い資産として扱う。  
- したがって WordPress は、React や Headless CMS の代用品ではなく、編集可能な HTML 資産を長期運用するための独自の責務分界を持つシステムとして位置づけるべきである。  

### この章の核心

WordPress は、Web Components、React、SPA、Headless CMS のどれかに回収されるシステムではない。これらとの比較を通じて見えてくるのは、WordPress が編集可能な HTML 資産の長期運用を中心に据えた、独自の責務分界を持つシステムだということである。

## まとめ

- Web Components との比較では、ブロックが UI 部品以上の編集・保存単位であることが見える。  
- React との比較では、Gutenberg が状態を持つ編集アプリとして設計されていることが見える。  
- SPA との比較では、WordPress がアプリ化しつつも HTML 資産を手放していないことが見える。  
- Headless CMS との比較では、WordPress が編集と公開の一体運用に強いことが見える。  
- WordPress の位置づけは、古いか新しいかではなく、どの責務をどこで引き受けるかで捉えるべきである。  

次章では、この WordPress の構造をさらに抽象化し、Gutenberg を DDD / CQRS 的に読む視点へ進みます。  
editor state と saved HTML、Write モデルと Read モデルを切り分けることで、WordPress の内部構造を別のレンズから見ていきます。
