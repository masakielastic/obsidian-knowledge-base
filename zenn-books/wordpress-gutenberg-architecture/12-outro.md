# おわりに

**初学者**  
ここまで読んできて、最初に思っていた WordPress のイメージがかなり変わりました。  
最初は、テーマやプラグインの使い方がたくさんある CMS という感じでした。  
でも今は、編集、保存、表示、制約、運用がつながったシステムとして見えるようになってきました。

**解説者**  
それが、この本の到達点です。  
この本は、WordPress の機能をたくさん覚えるための本ではありませんでした。  
WordPress を、説明できる形で理解するための本でした。  
つまり、断片的な知識を、構造を持った地図へ変えることが目的だったのです。

**実装者**  
実務でも、その違いはとても大きいです。  
「テーマが触れる」「プラグインが書ける」「ブロックを追加できる」といった個別の技能はもちろん大事です。  
でも本当に効くのは、それらが WordPress 全体の中でどこに位置しているかを分かっていることです。  
どこが編集の責務で、どこが保存の責務で、どこがテーマの責務で、どこが運用ルールの責務なのか。  
それが見えていると、現場での判断がかなり変わります。

## WordPress を説明できるようになるとは何か

**初学者**  
この本を読んだあとで、WordPress を説明できるようになる、というのは具体的にどういう状態なのでしょうか。

**解説者**  
少なくとも、次のことを言葉にできる状態です。  
クラシック WordPress と Gutenberg 以降の違いを説明できること。  
なぜ PHP と JavaScript の両方が必要になるのかを説明できること。  
ブロック、`block.json`、`theme.json`、editor state、saved HTML の役割を区別できること。  
そして WordPress を、単なる古い CMS ではなく、編集可能な HTML 資産を制約付きで安全に構築し、長期運用するシステムとして位置づけられることです。

**実装者**  
もう少し現場寄りに言うと、「この問題はどこで解くべきか」が分かるようになることでもあります。  
テーマで吸収すべきか。  
ブロックで持つべきか。  
プラグインに切り出すべきか。  
それとも運用ルールで解くべきか。  
この切り分けができると、WordPress は急に扱いやすくなります。

**初学者**  
なるほど。  
全部を知っているというより、何をどう分けて見るかが分かっている状態なのですね。

**解説者**  
その通りです。  
WordPress を説明できるようになるとは、全機能の暗記ではありません。  
責務分界を見失わずに、構造として語れるようになることです。

## 断片知識を構造に変える意味

**初学者**  
この本ではずっと、断片知識を構造に変える、という話がありました。  
それにはどんな意味があるのでしょうか。

**解説者**  
一番大きいのは、学習と実務の両方で迷いにくくなることです。  
断片知識だけだと、新しい機能や案件に出会うたびに、毎回ゼロから判断しなければなりません。  
ですが構造が見えていると、新しいものも既存の地図のどこに置くかを考えられます。  
すると、学習も実装も再利用可能になります。

**実装者**  
実務では特に、未整理の現場に強くなります。  
既存テーマ、古いプラグイン、更新しにくい管理画面、限られた予算、非エンジニアの更新担当者。  
そういう状況で必要なのは、最新機能の暗記より、何が問題の中心で、どこに制約を置き、どこを柔らかくするかを見抜く力です。  
構造理解があると、それがかなりやりやすくなります。

**初学者**  
つまり構造理解は、きれいな理論のためだけではなく、現場で迷わないためにも必要なのですね。

**解説者**  
その通りです。  
この本が目指していたのは、まさにそこです。  
WordPress を、説明の難しい寄せ集めとしてではなく、比較可能で判断可能な地図として持てるようにすること。  
それができると、個々の技術知識にも意味が通ります。

## この本で見てきたもの

**初学者**  
最後に、この本全体で何を見てきたのかを振り返りたいです。

**解説者**  
最初に見たのは、なぜ Gutenberg 以降の WordPress が説明しにくいのか、という問題でした。  
そこから、クラシック WordPress から Gutenberg への変化を確認し、ブロックテーマが設計対象にしたものを見ました。  
次に、Web Components、React、SPA、Headless CMS との比較を通して、WordPress の位置づけを明らかにしました。  
さらに DDD / CQRS 的な見方で editor state と saved HTML を切り分け、`block.json` と `theme.json` の責務を整理しました。  
そのうえで、テーマ設計を制約設計として捉え、WordPress のテストを保存契約・編集体験・表示資産を守る営みとして読み直しました。  
最後に、小規模制作会社の現場や、新世代 CMS との比較を通して、WordPress がどんな需要に応えているのかを評価しました。

**実装者**  
こうして並べると、この本は機能紹介の本ではなく、ずっと「どこに何の責任があるか」を追っていたことが分かります。  
WordPress の難しさも、面白さも、そこにあります。  
PHP だけでも、JavaScript だけでも、テーマだけでも、ブロックだけでも説明しきれない。  
だからこそ、責務分界を軸に整理することに意味がありました。

**初学者**  
たしかに。  
読み終わってみると、個々の話題が全部つながっていたのですね。

## 次に学ぶべきこと

**初学者**  
この本のあと、次は何を学ぶのがよいでしょうか。

**解説者**  
大きく三つあります。  
ひとつは、実際のブロックテーマやテーマ設定を読み、`theme.json` や patterns がどのように制約設計として使われているかを観察することです。  
ひとつは、実在するブロックを見て、`block.json` と編集 UI と保存 HTML の関係を追うことです。  
もうひとつは、実案件や実サイトを観察し、更新運用で何が壊れやすいのかを見て、テーマ・ブロック・運用ルールの責務分界を考えることです。

**実装者**  
もし実装寄りに進むなら、次のような学び方がよいです。  
まず既存ブロックテーマを読む。  
次に、簡単なブロックや pattern を作ってみる。  
そのあとで、保存 HTML と再編集の関係、テーマ制約の効き方、E2E 的な確認を意識する。  
これを繰り返すと、WordPress の構造理解がかなり身体化します。

**初学者**  
なるほど。  
本を読み終えたあとも、ただ API を追うのではなく、責務分界を観察しながら学ぶのが大事なのですね。

**解説者**  
その通りです。  
この本の内容は、知識として閉じるより、観察の道具として使ってほしいのです。  
実際のテーマ、実際の更新体験、実際の壊れ方を見ると、この本で扱った概念がよりはっきり見えてきます。

## 最後に

**初学者**  
最初は、WordPress をここまで構造的に読む必要があるのかと思っていました。  
でも読み終わってみると、むしろこういう地図がないと、現在の WordPress は説明しにくいのだと分かりました。

**解説者**  
それが、この本の結論です。  
WordPress は、使えるだけでは足りません。  
とくに Gutenberg 以降は、構造として説明できることが、学習にも実務にも直結します。  
そのためには、断片知識を増やすだけでなく、責務分界と比較の視点を持つ必要があります。

**実装者**  
そしてその視点があると、WordPress は「古いから難しいもの」ではなくなります。  
むしろ、編集可能な HTML 資産をどう安全に運用するか、というとても現実的な課題に向き合ったシステムとして見えてきます。  
この見え方が手に入れば、テーマ開発も、ブロック開発も、運用改善も、かなり違って見えるはずです。

**初学者**  
よく分かりました。  
WordPress を説明できるようになるとは、個々の機能を全部知ることではなく、現場の中で何が起きているかを構造として語れるようになることなのですね。

**解説者**  
その通りです。  
そして、その構造を語れるようになること自体が、これからの WordPress 学習の土台になります。  
この本が、そのための地図になっていればうれしく思います。

### この本の核心

この本は、WordPress を「テーマやプラグインの使い方の寄せ集め」としてではなく、「編集可能な HTML 資産を、制約付きで安全に構築し、長期運用するシステム」として理解するための本でした。WordPress を説明できるようになるとは、その責務分界を見失わずに、断片知識を構造として語れるようになることです。

## まとめ

- WordPress を説明できるようになるとは、機能暗記ではなく、責務分界を構造として語れるようになることである。  
- Gutenberg 以降の WordPress は、編集・保存・表示・制約・運用をまたぐシステムとして理解する必要がある。  
- `block.json`、`theme.json`、editor state、saved HTML などの役割を区別できることが重要である。  
- 構造理解は、学習のためだけでなく、未整理の現場で適切な判断をするためにも役立つ。  
- この本の内容は、知識の一覧ではなく、実際の WordPress を観察し判断するための地図として使うべきものである。
