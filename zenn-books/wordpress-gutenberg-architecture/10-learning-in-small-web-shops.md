# 第9章 小規模制作会社の現場から見る WordPress 学習

**初学者**  
ここまで読んでくると、WordPress をちゃんと理解するには、思っていたよりずっと広い範囲が必要だと分かってきますね。  
でも、小規模制作会社の現場では、そこまで理論的に整理して学ぶ余裕がないようにも思えます。  
実際のところ、現場では何が求められているのでしょうか。

**解説者**  
そこが、この章の問いです。  
この章の中心命題はこうです。  
小規模制作会社の現場で必要なのは、WordPress の機能暗記ではなく、PHP と JavaScript をまたぐ責務分界と、更新運用まで含めた構造理解である。  
ただし、その必要性は求人票だけを見ていても見えにくいのです。

**実装者**  
実務感覚でも、これはかなり大きいです。  
小規模制作会社では、役割がきれいに分かれていないことが多いです。  
デザイナーが少しコードを触る。  
コーダーがテーマ調整もする。  
エンジニアが更新導線まで気にする。  
そういう現場では、WordPress を単なる CMS 操作やテーマ修正の知識だけで捉えると、途中で限界が来ます。

**初学者**  
つまり、現場は理論から遠いように見えて、実は構造理解がないと苦しくなるのですね。

**解説者**  
その通りです。  
しかも Gutenberg 以降は、その傾向が強まりました。  
だからこの章では、「現場で本当に必要な学習」とは何かを、求人票の言葉より一段深いところから整理していきます。

## 求人票に書かれないが重要な理解

**初学者**  
まず、求人票に書かれないが重要な理解、というのが気になります。  
求人票には「WordPress の実務経験」「テーマ開発経験」「PHP が読める」みたいなことが書かれている印象です。  
それでは足りないのでしょうか。

**解説者**  
足りないことが多いです。  
もちろん、求人票としてはそれで十分に機能します。  
限られた文字数で応募条件を示す必要があるからです。  
ただし、それはあくまで作業経験のラベルです。  
実際に現場で効いてくるのは、そのラベルの背後にある構造理解です。

**実装者**  
たとえば「テーマ開発経験あり」と書いてあっても、その中身はかなり違います。  
クラシックテーマの修正が中心なのか。  
ブロックテーマや `theme.json` に触れているのか。  
管理画面の更新導線まで考えていたのか。  
ブロックの制約と表示の関係を理解していたのか。  
同じ一言でも、実際の意味はかなり幅があります。

**初学者**  
確かに。  
求人票の言葉だけだと、どこまで求められているか分からないですね。

**解説者**  
そうです。  
だから小規模制作会社を目指して WordPress を学ぶときは、求人票の用語をそのまま学習項目に変換しないほうがよいのです。  
大事なのは、「現場で何が壊れると困るのか」「誰がどこまで責任を持つのか」を読むことです。

## 「プラグインが書ける」の先にあるもの

**初学者**  
では、「プラグインが書ける」の先にあるものとは何でしょうか。  
これはかなり評価されそうなスキルに見えます。

**解説者**  
たしかに評価されやすいです。  
ですが、それだけでは WordPress を扱えるとは言い切れません。  
プラグインが書けるとは、拡張点を使えるということです。  
しかし小規模制作会社の現場では、「どこを拡張すべきで、どこはテーマや運用で吸収すべきか」を判断できることのほうが重要な場合があります。

**実装者**  
実務では、全部をプラグイン化すればよいわけではありません。  
ちょっとした表示調整、更新しやすさの改善、パターン提供、テンプレートの整理、`theme.json` による制約付けで済むこともあります。  
逆に、本当にプラグインや独自ブロックが必要なケースもあります。  
だから大事なのは、「書ける」こと以上に、「どこで何を持つべきか」を判断できることなんです。

**初学者**  
なるほど。  
技術があることより、その技術をどこで使うかの判断が大事なのですね。

**解説者**  
その通りです。  
小規模制作会社では、過剰実装は保守負担になりやすい。  
逆に、必要な抽象化をしないと更新で破綻します。  
だから WordPress 学習は、実装スキルの獲得だけでなく、責務分界の見極めを含んでいなければなりません。

## PHP と JavaScript をまたぐ責務分界

**初学者**  
この本では何度も、PHP と JavaScript の両方が必要だと言ってきました。  
でも小規模制作会社だと、そこまで両方を深く求められないこともありそうです。

**解説者**  
たしかに、全員が両方を深く書ける必要はありません。  
ただし、両方の責務がどこにあるかを理解していないと、現在の WordPress は説明しにくいし、設計もしにくいです。  
小規模制作会社ほど役割が重なりやすいので、この責務分界の理解が効いてきます。

**実装者**  
たとえば PHP は、テンプレート、サーバーサイドの出力、既存テーマ文化、各種連携、公開側の安定性に強く関わります。  
一方 JavaScript は、ブロックエディタ、編集 UI、インタラクション、ブロックの内部表現に強く関わります。  
この区別が見えていると、「これはテーマの PHP 側で調整すべきか」「ブロックの編集側で吸収すべきか」が考えやすくなります。

**初学者**  
つまり、両方の文法を完璧に書けることより、どちらが何を担当しているかを分かっていることが重要なのですね。

**解説者**  
その通りです。  
特に Gutenberg 以降は、片方だけの知識で全体を説明しようとすると無理が出ます。  
小規模制作会社では「そこは別担当だから知らない」で済まないことも多いので、なおさらです。

## テーマだけのケースでは何を学ぶべきか

**初学者**  
実際の現場では、「うちはテーマだけで、独自ブロックまではやらない」という案件も多そうです。  
そういう場合は、何を学べばよいでしょうか。

**解説者**  
その場合でも、学ぶべきことはあります。  
まず、クラシックテーマとブロックテーマの違い。  
次に、テンプレートとテンプレートパーツの役割。  
そして `theme.json` による制約設計と、patterns による誘導設計です。  
独自ブロックを書かなくても、ブロックテーマ時代の運用はここに強く依存します。

**実装者**  
実務で言えば、独自ブロックなしでも、編集画面が壊れにくいか、更新担当者が迷わないか、サイト全体の一貫性が保てるかは大きな論点です。  
そのためには、使うブロックをどう組み合わせるか、どこまで自由にするか、どういう patterns を渡すかを考えなければいけません。  
これはテーマだけの仕事に見えて、かなり構造理解が要ります。

**初学者**  
なるほど。  
「独自ブロックを書かない＝古いやり方」というわけではなく、テーマ側にも学ぶべきことが多いのですね。

**解説者**  
そうです。  
テーマだけのケースでも、Gutenberg 以降の WordPress を理解していないと、見た目の修正屋で終わってしまいやすい。  
でも本当に求められるのは、更新運用まで含めた設計ができる人です。

## カスタムブロックありのケースでは何が増えるのか

**初学者**  
では、カスタムブロックありのケースでは、何が増えるのでしょうか。

**解説者**  
一言で言えば、「編集単位そのものを設計する責任」が増えます。  
テーマだけのときは、既存ブロックの組み合わせと制約が主な論点でした。  
カスタムブロックありになると、どんな単位をブロックとして成立させるべきか、そのブロックにどんな能力と制約を持たせるかを考える必要があります。

**実装者**  
ここでは `block.json` 的な契約理解が重要になります。  
そのブロックは何者か。  
どんな属性を持つのか。  
編集体験はどうあるべきか。  
保存 HTML はどうなるのか。  
テーマや `theme.json` の方針と噛み合うか。  
つまり、単に UI を追加するのではなく、CMS の構造単位を増やす仕事になります。

**初学者**  
それはたしかに、ただの部品追加ではないですね。

**解説者**  
その通りです。  
小規模制作会社でここまで扱う場合、React 的な UI 実装力だけでなく、WordPress 全体の責務分界が見えていることが重要になります。  
そうでないと、便利そうな部品を増やしただけで、保存契約や運用設計が壊れます。

## 学習の順番はどう考えるべきか

**初学者**  
ここまで聞くと、学ぶことが多くて順番に迷います。  
小規模制作会社を意識して学ぶなら、どういう順番がよいのでしょうか。

**解説者**  
まず、クラシック WordPress の基本構造を押さえることです。  
テンプレート階層、`functions.php`、フック、テーマとプラグインの違い。  
これは今でも土台になります。  
次に、Gutenberg によって何が変わったのかを理解する。  
ブロック、エディタのアプリ化、保存 HTML の位置づけ、Site Editor とブロックテーマの登場です。  
その上で、`theme.json` と patterns を通じて制約設計を学ぶ。  
最後に必要に応じて、`block.json` や独自ブロック開発へ進むのが自然です。

**実装者**  
実務感覚でも、その順番が無理が少ないです。  
いきなりカスタムブロックから入ると、WordPress 全体との接続が見えにくい。  
逆にクラシックな知識だけで止まると、今の編集運用を説明できない。  
だから「土台としてのクラシック構造」「転換点としての Gutenberg」「運用設計としてのブロックテーマ」「必要なら独自ブロック」という順番がよいです。

**初学者**  
なるほど。  
新しいところだけ追うのでも、古いところだけ守るのでもだめなのですね。

**解説者**  
その通りです。  
WordPress 学習は、歴史の順に全部追う必要はありません。  
ですが、どこが連続していて、どこが変わったのかは押さえる必要があります。  
それが現場での判断力につながります。

## 学習とは、API の暗記ではなく責務分界の習得である

**初学者**  
ここまで聞いていると、WordPress 学習って、API や関数を覚えることとは少し違いますね。

**解説者**  
はい。  
もちろん API や関数の知識も必要です。  
ですが小規模制作会社の現場から見た WordPress 学習の本質は、責務分界の習得にあります。  
どこをテーマで持つか。  
どこをブロックで持つか。  
どこをプラグインにするか。  
どこを運用ルールで吸収するか。  
この見極めができると、学習した知識が現場で生きます。

**実装者**  
逆に、関数名だけたくさん知っていても、どこに何を置くべきかが分からないと苦しいです。  
小規模制作会社では、過剰設計も場当たり実装もコストになります。  
だから必要なのは、機能の網羅より、責務の置き方を判断できることなんです。

**初学者**  
すると学習のゴールも、「全部知っている」ではなく、「どこを見ればよいか分かる」に近いですね。

**解説者**  
その表現はとてもよいです。  
すべてを暗記する必要はありません。  
ただ、問題が起きたときに、それがテーマの問題なのか、ブロックの問題なのか、編集運用の問題なのか、保存契約の問題なのかを切り分けられる必要があります。  
それが構造理解です。

## 小規模制作会社で効くのは「未整理の現場を整理する力」である

**初学者**  
小規模制作会社の現場という観点で、最後にもう一つ聞きたいです。  
結局、そういう現場で一番効くのは何なのでしょうか。

**解説者**  
一言で言えば、未整理の現場を整理する力です。  
小規模制作会社では、理想的なアーキテクチャ図が最初からあるわけではありません。  
既存テーマがあり、更新担当者の事情があり、クライアントの要望があり、保守予算も限られている。  
その中で、どこを整理し、どこを固定し、どこを柔らかく持つかを決める必要があります。

**実装者**  
実務では、「最新技術を知っている人」より、「この現場ではどこに制約を入れるべきか」「どこは無理に作り込まないほうがよいか」を判断できる人のほうが強いことがあります。  
WordPress は特にそうです。  
なぜなら、編集者、公開物、既存資産、テーマ、プラグイン、運用フローが全部絡むからです。

**初学者**  
なるほど。  
だからこの本でずっと責務分界や制約設計を重視してきたのですね。

**解説者**  
その通りです。  
小規模制作会社の現場で本当に役立つ WordPress 学習とは、断片的な技術の寄せ集めではありません。  
現場の曖昧さを整理し、壊れにくい着地点を設計するための学習です。  
その意味で、WordPress 学習は単なる CMS 学習ではなく、現場設計の学習でもあるのです。

### 整理

- 求人票の「WordPress 経験」「テーマ開発経験」は、作業経験のラベルであり、構造理解の深さまでは表さない。  
- 小規模制作会社で重要なのは、「何ができるか」だけでなく、「どこで何を持つべきか」を判断できること。  
- `プラグインが書ける`ことは有用だが、どこを拡張し、どこをテーマや運用で吸収すべきかの判断がより重要になる。  
- Gutenberg 以降の WordPress を扱うには、PHP と JavaScript の両方の責務分界を理解する必要がある。  
- テーマだけの案件でも、`theme.json`、patterns、制約設計、更新運用の理解が重要である。  
- カスタムブロックありの案件では、編集単位そのものを設計する責任が増える。  
- 学習の順番は、クラシック構造の理解、Gutenberg の転換点、ブロックテーマの制約設計、必要に応じた独自ブロック開発、という流れが自然である。  
- 小規模制作会社で本当に効くのは、未整理の現場を、責務分界と制約設計の観点から整理できる力である。  

### この章の核心

小規模制作会社の現場で必要なのは、WordPress の機能をたくさん知っていることではない。PHP と JavaScript をまたぐ責務分界を理解し、テーマ・ブロック・プラグイン・運用のどこに何を置くべきかを判断し、未整理の現場を壊れにくい形に整理できることである。

## まとめ

- 求人票の言葉だけでは、現場で必要な WordPress 理解は見えにくい。  
- 小規模制作会社では、実装力以上に責務分界の判断力が重要になる。  
- テーマだけの案件でも、Gutenberg 以降は制約設計と運用設計の理解が必要である。  
- カスタムブロックありの案件では、編集単位そのものを設計する力が求められる。  
- WordPress 学習の本質は、API の暗記ではなく、未整理の現場を整理するための構造理解にある。  

次章では、こうした構造理解を踏まえて、WordPress は古いのか、それとも別の最適化を持つシステムなのかを改めて評価します。  
「古い CMS」という雑な見方を超えて、WordPress をどう位置づけるべきかを考えていきます。
