# 第10章 WordPress は古いのか、それとも別の最適化なのか

**初学者**  
ここまで読んできて、WordPress を単純に古い CMS と言い切れないことはだいぶ分かってきました。  
でも世の中では、やはり「WordPress は古い」「いまどきは別のやり方がある」と言われがちですよね。  
結局、WordPress は古いのでしょうか。それとも、別の最適化を持つシステムなのでしょうか。

**解説者**  
その問いに答えるのが、この章です。  
この本全体を通して見えてきたのは、WordPress はたしかに長い歴史を持ち、古い層も多く抱えたシステムだということです。  
ただし、それだけで評価すると、本質を取り逃がします。  
この章の中心命題はこうです。  
WordPress は、単に古いシステムなのではなく、編集可能な HTML 資産を長期運用するための、独自の最適化を続けてきたシステムである。  
だから評価すべきなのは、新旧の印象ではなく、何を守り、何を最適化しているかです。

**実装者**  
実務でも、この違いはかなり大きいです。  
「古いからよくない」と見てしまうと、なぜ WordPress が今でも多くの現場で使われ、しかも Gutenberg のような大きな再設計を行っているのかが説明しにくくなります。  
逆に「全部モダン」と持ち上げるのも違います。  
大事なのは、WordPress がどんな需要に対して強く、どんな責務分界を採っているかを冷静に見ることです。

**初学者**  
なるほど。  
この章は、WordPress を擁護する章でも、否定する章でもなく、評価軸を整理する章なのですね。

**解説者**  
その通りです。  
ここでは「古い / 新しい」という雑な二分法をいったん脇に置き、WordPress が何を最適化してきたのかを見ていきます。

## 「古い / 新しい」という見方の限界

**初学者**  
まず、なぜ「古い / 新しい」という見方だけではだめなのでしょうか。  
分かりやすい基準には見えるのですが。

**解説者**  
分かりやすいのですが、粗すぎるのです。  
技術には必ず歴史がありますし、長く使われているものほど、複数の時代の要請を抱えます。  
そのとき、「古い」という言葉だけでは、何が問題で、何が価値として残っているのかを区別できません。

**実装者**  
実際、WordPress には古い部分があります。  
古い API、長い互換性の歴史、従来テーマ文化、プラグイン資産、PHP 中心の構造などです。  
でも同時に、Gutenberg によるエディタのアプリ化、ブロックによる構造化、`theme.json` による制約設計など、新しい要請に対応する変化もあります。  
この両方があるのに、「古い」の一語でまとめると、何も見えなくなります。

**初学者**  
つまり、WordPress は昔のまま止まっているわけではないのですね。

**解説者**  
その通りです。  
しかも重要なのは、変化の方向です。  
WordPress は「全部を最新の流行へ合わせる」ために変わっているのではありません。  
自分が重視してきた価値を保ちながら、編集体験や構造化の課題に対応しようとして変わっています。  
ここを見ないと、評価を誤りやすいのです。

## WordPress が最適化してきたものは何か

**初学者**  
では、WordPress は何を最適化してきたのでしょうか。

**解説者**  
一言で言えば、編集可能な Web 公開物を、広い人たちが長く運用できることです。  
もう少し分けて言うと、少なくとも次のようなものを重視してきました。  
公開しやすさ、更新しやすさ、テーマやプラグインによる拡張性、既存資産との接続、そして HTML 資産としての継続性です。

**実装者**  
たとえば、WordPress は最初からヘッドレス前提の CMS ではありませんでした。  
公開物そのものを作り、更新し、運用することを一体の流れとして持っています。  
この一体性は、とくに中小規模のサイトや、非エンジニアが更新に関わる現場でかなり強いです。  
編集者が書いて、見て、直して、そのまま公開物になる。  
これは単純に古いのではなく、ある種類の運用に最適化された設計です。

**初学者**  
なるほど。  
WordPress は、コンテンツを API 化して配ることより、公開物として持つことを重視してきたのですね。

**解説者**  
その通りです。  
だからこそ、Gutenberg 以降も保存された HTML が重い意味を持ちます。  
編集画面はアプリ化しましたが、公開資産としての HTML を捨てていません。  
これは古さの残骸ではなく、最適化の継続だと見ることができます。

## HTML 資産を長期運用するという思想

**初学者**  
この本では何度も HTML 資産という言葉が出てきました。  
なぜそこまで重要なのでしょうか。

**解説者**  
WordPress を理解するうえで、そこが中核だからです。  
WordPress は、表示結果としての HTML を単なる一時的レンダリングではなく、長期運用される公開資産として扱います。  
しかも、その資産は編集と切り離されていません。  
編集され、保存され、再編集され、表示されるという循環の中にあります。

**実装者**  
この発想は、純粋なアプリケーションデータ中心の設計とは少し違います。  
たとえば JSON を真実として API 経由で配る世界では、HTML はクライアントごとの表現結果になりやすいです。  
一方 WordPress では、HTML 自体がかなり中心にあります。  
テーマや CSS はその HTML を前提にし、公開もその HTML を基盤にします。  
だから WordPress は、HTML を「古い表示形式」として軽く扱っていません。

**初学者**  
それは、昔からの互換性だけの話ではないのですね。

**解説者**  
はい。  
互換性も大事ですが、それだけではありません。  
HTML は Web の基本的な公開資産でもあります。  
長く運用されるサイト、検索されるページ、共有される記事、更新担当者がそのまま成果物として認識できるページ。  
そうしたものを中心に置く思想が、WordPress にはあります。  
この点は、現在でも十分に意味があります。

## 新世代 CMS との違いは何か

**初学者**  
では、新世代 CMS と呼ばれるものとの違いは、どこにあるのでしょうか。

**解説者**  
違いは、優劣というより、重心です。  
新世代 CMS の多くは、コンテンツモデル、API 配信、フロントエンド分離、複数チャネルへの展開に強く最適化されています。  
これは非常に合理的です。  
一方 WordPress は、編集と公開の近さ、HTML 資産の継続性、テーマによる一体運用に強く最適化されています。

**実装者**  
つまり、新世代 CMS は「コンテンツをどう持ち出すか」に強く、WordPress は「公開物をどう運用し続けるか」に強いと言えます。  
もちろん重なる部分はあります。  
ですが、設計思想としてはかなり違います。  
だから WordPress を見て「API 中心でないから遅れている」と言うのは、評価軸をずらしていることがあります。

**初学者**  
なるほど。  
比べる相手が違えば、強みも弱みも違って見えるのですね。

**解説者**  
その通りです。  
新世代 CMS と WordPress は、同じ需要を全部奪い合う存在ではありません。  
重なる領域はありますが、強く応えている要求が違います。  
この違いを見ずに「新しいから勝ち、古いから負け」とすると、かなり雑な議論になります。

## 同じ需要か、重なるが別の需要か

**初学者**  
今の話で気になったのですが、WordPress と新世代 CMS は、同じ需要を満たしているのでしょうか。  
それとも、少し重なるが別物なのでしょうか。

**解説者**  
答えは後者です。  
かなり重なるが、完全には同じではありません。  
たとえば、企業サイト、メディアサイト、店舗サイト、採用サイトなどでは、どちらも候補になりえます。  
しかし、編集担当者の性質、公開物の寿命、テーマ変更の頻度、開発体制、運用責任の置き方によって、向く設計は変わります。

**実装者**  
小規模制作会社や地域の事業者向けサイトを考えると分かりやすいです。  
更新担当者がエンジニアではない。  
公開物をそのまま確認しながら更新したい。  
テーマやパターンである程度守りたい。  
こういう状況では WordPress の一体性が強い。  
一方で、複数の配信先に同一コンテンツを API で供給したい、フロントエンドは完全に別チームで構築したい、というなら別の選択肢が自然です。

**初学者**  
つまり、WordPress は古い需要にだけ応えているのではなく、今も存在する別種の需要に応えているのですね。

**解説者**  
その通りです。  
ここが非常に重要です。  
WordPress を「時代遅れの残り物」と見るのではなく、「今でも強く存在する需要に最適化されたシステム」と見るべきなのです。

## Gutenberg は何を意味しているのか

**初学者**  
ここまでの話を踏まえると、Gutenberg は何を意味しているのでしょうか。  
古い WordPress が無理やり新しく見せようとしているだけではないのですか。

**解説者**  
そうではありません。  
Gutenberg は、WordPress が自分の中核価値を保ちながら、編集体験と構造化の課題に正面から対応しようとした結果だと見るほうが適切です。  
つまり、HTML 資産と一体運用の思想は維持しつつ、編集をより構造化し、制約設計を導入し、エディタをアプリ化した。  
これは単なる見た目の刷新ではなく、最適化の更新です。

**実装者**  
実務感覚でも、Gutenberg は「WordPress が React っぽくなった」というだけではありません。  
むしろ、WordPress が自分の運用モデルを維持しながら、編集の難しさとバラつきに対応しようとした大きな構造変更です。  
だから Gutenberg を見ると、WordPress が止まっていないことがよく分かります。

**初学者**  
なるほど。  
Gutenberg は、古い WordPress を否定したのではなく、WordPress の最適化を更新したのですね。

**解説者**  
その通りです。  
古いものを全部捨てて新しいものへ移ったわけではありません。  
むしろ、守るべきものを守りながら、変えるべきところを変えた。  
そこに WordPress の連続性と断絶の両方があります。

## WordPress を過小評価しないための評価軸

**初学者**  
では、WordPress を過小評価しないためには、どんな評価軸を持てばよいでしょうか。

**解説者**  
少なくとも、次のような軸が必要です。  
何を公開資産として残したいのか。  
誰が更新するのか。  
編集体験と公開結果の距離をどれくらい近く保ちたいのか。  
どこで責務を切りたいのか。  
長期運用で何を安定させたいのか。  
このあたりです。

**実装者**  
さらに言えば、変更コストの置き方も重要です。  
表示層を自由に変えたいのか。  
それとも編集者が迷わず更新できることを優先したいのか。  
コンテンツを抽象データとして再利用したいのか。  
それとも HTML 公開物として一貫して扱いたいのか。  
こうした軸で見ると、WordPress が強い領域と弱い領域がかなり明確になります。

**初学者**  
つまり、「最新っぽいかどうか」ではなく、「どの運用に合っているか」で見るべきなのですね。

**解説者**  
その通りです。  
そしてその見方をすると、WordPress は決して単なる過去の遺物ではありません。  
むしろ、広い現場に今も存在する問題に対して、かなり現実的な答えを持っているシステムだと分かります。

## WordPress は古いのか、それとも別の最適化なのか

**初学者**  
では、最初の問いに戻ります。  
結局、WordPress は古いのでしょうか。それとも別の最適化なのでしょうか。

**解説者**  
答えは、両方です。  
WordPress は長い歴史を持つシステムであり、古い層を抱えています。  
その意味で、古さはあります。  
しかし評価の本質はそこではありません。  
WordPress は、古い層を抱えながら、編集可能な HTML 資産の長期運用という課題に対して、独自の最適化を続けてきたシステムです。  
だから「古いかどうか」だけで判断すると、半分しか見えていません。

**実装者**  
実務では、この両面を同時に見られることが大事です。  
古いから制約もある。  
でも古いからこそ残っている運用知、互換性、資産、公開モデルもある。  
そのうえで Gutenberg のような変化がある。  
この全体を見てはじめて、WordPress を適切に評価できます。

**初学者**  
分かってきました。  
WordPress は、時代遅れのシステムとして切り捨てるには、あまりにもはっきりした最適化の意図を持っているのですね。

**解説者**  
その通りです。  
この本全体でやってきたのは、まさにその意図を見えるようにすることでした。  
WordPress は古いか新しいか、ではなく、何を守り、何を引き受け、何を最適化しているのか。  
その問いで見ると、かなり像が変わります。

### 整理

- 「古い / 新しい」という見方だけでは、WordPress の価値と制約を十分に説明できない。  
- WordPress は長い歴史を持ち、古い層も抱えているが、それだけで評価すると本質を取り逃がす。  
- WordPress が最適化してきたのは、編集可能な Web 公開物を長く運用できること、一体的な更新体験、HTML 資産としての継続性である。  
- Gutenberg は、その中核価値を捨てずに、編集体験と構造化を更新しようとした大きな変化である。  
- 新世代 CMS は API 配信や責務分離に強く最適化される傾向があるのに対し、WordPress は編集と公開の一体運用に強い。  
- WordPress と新世代 CMS は需要が重なる部分もあるが、完全に同じ需要を満たしているわけではない。  
- WordPress を評価するには、新旧の印象ではなく、誰が更新し、何を資産として残し、どこで責務を切りたいのかという軸が必要である。  

### この章の核心

WordPress は、単に古いシステムなのではない。長い歴史を抱えながらも、編集可能な HTML 資産を長期運用するという課題に対して、独自の最適化を続けてきたシステムである。だから評価すべきなのは、新旧の印象ではなく、何を守り、何を最適化しているかである。

## まとめ

- WordPress には古さがあるが、それだけでは本質を説明できない。  
- WordPress の強みは、編集と公開を近く保ちながら、HTML 資産を長く運用できることにある。  
- Gutenberg は、WordPress の中核価値を保ちながら編集構造を更新した変化である。  
- 新世代 CMS との違いは、優劣よりも責務分界と最適化対象の違いにある。  
- WordPress を評価するには、「古い / 新しい」ではなく、「どの需要にどう応えているか」で見る必要がある。  

次は本書のまとめとして、WordPress を説明できるようになるとは何かを振り返ります。  
断片知識を構造に変えることが、なぜ現場でも学習でも重要なのかを、最後にもう一度整理していきます。
