# 第5章 Gutenberg を DDD / CQRS 的に読む

**初学者**  
ここまでで、Gutenberg 以降の WordPress は、クラシックなテンプレート中心の CMS というだけでは説明しにくい、という話は分かってきました。  
でも次に DDD や CQRS が出てくると、少し身構えてしまいます。  
WordPress をそんな難しい理論で読まないといけないのでしょうか。

**解説者**  
そこは最初にはっきりさせておきましょう。  
この章で DDD や CQRS を出すのは、WordPress を無理に流行の理論へ当てはめるためではありません。  
むしろ逆です。  
Gutenberg の構造を説明しやすくするための、再解釈のレンズとして使います。

**実装者**  
実務でも、Gutenberg をただ「ブロックエディタ」とだけ言うと、何が起きているのか整理しにくいです。  
編集画面の内部状態があり、保存された HTML があり、表示側の資産があり、それぞれの責務が少しずつ違う。  
その違いを見るには、Write と Read を分けて考える視点がかなり役立ちます。

**初学者**  
つまり、DDD や CQRS をそのまま実装していると言いたいのではなく、構造の見え方をよくするために使うのですね。

**解説者**  
その通りです。  
この章の中心命題はこうです。  
Gutenberg は、編集時の状態と保存・表示される結果を同一視すると分かりにくい。  
むしろ DDD / CQRS 的に、Write モデルと Read モデル、内部状態と保存資産を分けて考えると、構造がかなり見えやすくなる。  
ただし、Gutenberg は純粋な CQRS でも Event Sourcing でもありません。  
そこも丁寧に整理していきます。

## なぜ DDD / CQRS 的な読み方が必要なのか

**初学者**  
そもそも、なぜそこまでして読み替える必要があるのでしょうか。  
普通に「編集して保存して表示する」と考えるだけでは足りないのですか。

**解説者**  
直感的にはそれでも通じます。  
ですが、そのままだと「編集画面で見えているもの」と「保存されているもの」と「公開画面で使われるもの」の違いが曖昧になりやすいです。  
Gutenberg の説明が難しくなる理由のひとつは、これらを全部まとめて“コンテンツ”と呼んでしまうことにあります。

**実装者**  
実務でもよくあります。  
エディタ上では自由にブロックを並べ替えられる。  
でも保存後には HTML 資産として落ちる。  
しかも、その保存結果は再編集の前提にもなるし、テーマや CSS の前提にもなる。  
これを一つの層として扱うと、どこで何が起きているのか見えにくいんです。

**初学者**  
なるほど。  
“同じコンテンツ”と言っても、編集時と保存後と表示時で役割が違うのですね。

**解説者**  
その通りです。  
だから、Gutenberg を理解するには、状態を分けて考える必要があります。  
DDD / CQRS 的な読み方は、その分け方を与えてくれます。

## editor state と saved HTML は同じではない

**初学者**  
まず、いちばん大事そうなのはそこですね。  
editor state と saved HTML は、どう違うのでしょうか。

**解説者**  
editor state は、エディタ上で現在編集中の構造や選択状態や属性変更を含んだ、動的な内部状態です。  
一方 saved HTML は、保存の結果として残る HTML 資産です。  
両者は密接に関係していますが、同一ではありません。

**実装者**  
実感としては、editor state は作業中のモデルです。  
今どのブロックが選ばれているか。  
どの属性が変更されたか。  
親子関係がどうなっているか。  
Undo / Redo の履歴がどうなっているか。  
そうした編集中の事情が含まれます。  
でも saved HTML は、そうした一時的事情を全部そのまま持つわけではありません。  
保存可能な形に落ちた結果です。

**初学者**  
ということは、エディタで操作している世界と、保存される世界はつながっているけれど、重なりきってはいないのですね。

**解説者**  
その通りです。  
ここを区別しないと、「エディタで見えているものがそのまま保存される」と単純化してしまいます。  
しかし実際には、Gutenberg は編集のための内部状態を持ちながら、それを保存可能な HTML へ落とし込む構造を取っています。  
この差分が見えると、WordPress がなぜ単なる SPA にも、単なる静的テンプレートにもなりきらないのかが分かりやすくなります。

## Write モデルとしての Gutenberg

**初学者**  
では、CQRS 的に言う Write モデルとは、Gutenberg では何にあたるのでしょうか。

**解説者**  
大づかみに言えば、Gutenberg の editor state 側が Write モデルに近いです。  
編集者がブロックを追加し、属性を変え、並べ替え、入れ子にし、履歴をたどる。  
そうした変更の中心になる世界です。

**実装者**  
Write モデルという見方をすると、エディタがアプリケーションとして見えてきます。  
入力フォームの集合というより、状態遷移の場です。  
ある操作によって状態が変わり、その状態から次の操作可能性が決まる。  
だから Gutenberg を PHP テンプレートの延長だけで見ると違和感が出るんです。

**初学者**  
たしかに、これは「本文を入力して送信する」だけではないですね。  
編集操作そのものがかなり複雑です。

**解説者**  
その通りです。  
Write モデルとして見ると、Gutenberg の本質は「どう表示するか」だけではなく、「どう編集し、どう変化させるか」にあると分かります。  
この視点は、ブロックの UI や内部状態の意味を考えるうえで重要です。

## Read モデルとしての saved HTML

**初学者**  
では Read モデルは、saved HTML に近いのでしょうか。

**解説者**  
はい。  
厳密に一対一対応ではありませんが、学習上は saved HTML を Read モデルに近いものとして捉えると分かりやすいです。  
保存された HTML は、公開画面で利用され、テーマや CSS の対象になり、長期運用される資産でもあります。  
つまり「読む」「表示する」側に強く寄っています。

**実装者**  
ここで大事なのは、Read モデルが単なる二次的な結果ではないことです。  
WordPress では、保存された HTML 自体がかなり重い。  
公開物として機能するだけでなく、後の再編集や互換性の前提にもなります。  
だから saved HTML は、軽いキャッシュのようなものではなく、運用上の中心資産でもあります。

**初学者**  
なるほど。  
Write モデルが本体で Read モデルはおまけ、という感じではないのですね。

**解説者**  
その通りです。  
一般的な CQRS の説明だけをそのまま持ち込むと、Read モデルが軽く見えすぎることがあります。  
しかし WordPress では、Read 側に落ちる saved HTML が非常に重要です。  
だから Gutenberg を CQRS 的に読むときは、この点を WordPress 固有の特徴として押さえる必要があります。

## なぜ Write と Read を分けると見やすくなるのか

**初学者**  
ここまで聞くと、たしかに区別したほうがよさそうです。  
でも、分けると具体的に何が見やすくなるのでしょうか。

**解説者**  
少なくとも三つ見やすくなります。  
ひとつは、エディタがなぜアプリ的なのか。  
ひとつは、保存 HTML がなぜ重い資産なのか。  
もうひとつは、ブロックやテーマの変更がどこに影響するのかです。

**実装者**  
たとえば、あるブロックの編集 UI を変えるのは主に Write 側の話です。  
でも保存結果の HTML を変えると、Read 側の資産に影響します。  
さらに、既存コンテンツとの互換性やテーマ側のスタイルにも波及します。  
このとき Write と Read を分けて考えていれば、「何を変えているのか」が見えやすいです。

**初学者**  
つまり、変更の影響範囲を考えるときにも役立つのですね。

**解説者**  
その通りです。  
Gutenberg をただのリッチエディタと見ると、変更が UI の話なのか、保存契約の話なのかが混ざります。  
CQRS 的に読むと、その混ざりをかなりほぐせます。

## Undo / Redo はなぜ重要なのか

**初学者**  
アウトラインには Undo / Redo と内部イベント列という話もありました。  
これはどこにつながるのでしょうか。

**解説者**  
Undo / Redo は、Gutenberg が単なる入力欄ではなく、状態遷移を持つ編集アプリであることをよく示しています。  
編集とは、ある時点の完成形を一回作るだけではありません。  
操作の列があり、その列をさかのぼったり、やり直したりできる必要があります。  
これは Write 側の性格をかなり強く表しています。

**実装者**  
実際、Undo / Redo があるということは、内部的には「どんな変化が起きたか」をある程度追跡しているということです。  
もちろん、そのまま完全なイベントログとして永続化しているわけではありません。  
でも少なくとも編集アプリとしては、現在の状態だけでなく、そこへ至る変更列の意識があるわけです。

**初学者**  
なるほど。  
「今の形がある」だけでなく、「どう変わってきたか」が重要なのですね。

**解説者**  
その通りです。  
ここが、Gutenberg を Event Sourcing と比較したくなる理由のひとつでもあります。

## Event Sourcing と似ている点

**初学者**  
では Event Sourcing と似ている点は、どこにあるのでしょうか。

**解説者**  
似ているのは、「状態を一枚岩の静止物としてだけではなく、変更の列として見る」発想です。  
Gutenberg の editor state は、編集中の現在状態だけでなく、操作による変化の積み重ねとして理解できます。  
Undo / Redo の存在も、その見方を支えます。

**実装者**  
また、編集という行為自体が、段落を追加した、属性を変えた、並べ替えた、といった一連の操作から成り立っています。  
だから“イベント的に見る”ことには意味があります。  
特に editor state を考えるときは、「今の構造」だけでなく、「どういう操作の結果としてこの構造になったか」を意識したほうが自然です。

**初学者**  
たしかに、編集作業はイベントの連なりとして考えたほうが分かりやすいですね。

**解説者**  
そうです。  
だから Event Sourcing 的な連想は有効です。  
ただし、ここで大事なのは「似ている」と「同じ」は別だということです。

## Event Sourcing と違う点

**初学者**  
では、何が違うのでしょうか。

**解説者**  
一番大きいのは、Gutenberg が変更イベント列そのものを永続化の中心にしているわけではないことです。  
純粋な Event Sourcing では、イベント列が真実の記録になります。  
現在状態は、そのイベント列から再構成されます。  
しかし WordPress では、保存の中心にあるのは saved HTML です。  
つまり、真実の中心はイベント列ではなく、保存結果に強く寄っています。

**実装者**  
ここはかなり重要です。  
Gutenberg の内部にはイベント的な状態遷移があります。  
でも、そのまま「全イベントを永続化して、あとで再生する」設計ではありません。  
保存時には HTML 資産へ落ちます。  
しかもその HTML は公開物であり、長期運用の資産です。  
この点で、Gutenberg は純粋な Event Sourcing とはかなり違います。

**初学者**  
なるほど。  
イベント的に編集しているけれど、最後に残すものはイベント列ではないのですね。

**解説者**  
その通りです。  
だから Event Sourcing は比喩としては役立ちますが、完全に一致するモデルだと考えると誤解します。

## Snapshot 永続化として見ると何が分かるか

**初学者**  
アウトラインには、「純粋な Event Sourcing ではなく Snapshot 永続化であること」とありました。  
これはどういう意味でしょうか。

**解説者**  
簡単に言えば、WordPress が保存しているのは、イベント列ではなく、その時点での結果状態に近いものだということです。  
その結果状態が saved HTML です。  
つまり、操作の履歴を主に残すのではなく、ある時点で公開や再編集の基盤になるスナップショットを残している、と見ると分かりやすいです。

**実装者**  
この見方をすると、WordPress の実務上の性格がよく見えます。  
大事なのは「どう編集したかの完全履歴」より、「今どの状態が公開可能で、再編集可能か」です。  
だからスナップショットとしての保存結果が重くなる。  
Gutenberg は編集時には状態遷移を持ちながら、永続化ではスナップショットを重視しているんです。

**初学者**  
それは WordPress らしい感じがしますね。  
履歴の完全再現より、公開資産として今どう残るかを重く見ている。

**解説者**  
その通りです。  
この意味で、Gutenberg は「Event 的な編集」と「Snapshot 的な保存」を組み合わせたシステムとして読むと見やすいです。

## DDD 的に見ると、どこが見えるのか

**初学者**  
ここまでは CQRS や Event Sourcing の話が中心でした。  
DDD 的に読むとは、どういう意味でしょうか。

**解説者**  
ここでの DDD は、厳密な戦術パターンの話ではありません。  
もっと広く、「何を主要なドメイン概念として見るか」「どこに責務境界があるか」を整理するレンズとして使います。  
Gutenberg にとって重要なのは、ブロック、編集状態、保存資産、テーマ制約といった概念を混ぜずに扱うことです。

**実装者**  
たとえば、ブロックは単なる UI 部品ではありません。  
編集単位であり、保存単位でもあります。  
テーマは単なる見た目ではなく、制約を与える側でもあります。  
こうした概念の責務をはっきりさせると、WordPress の全体像がかなり整理されます。  
この「概念を混ぜない」という意味で、DDD 的な見方は役に立ちます。

**初学者**  
つまり DDD というより、ドメインの輪郭をちゃんと引くための考え方なのですね。

**解説者**  
その通りです。  
この章で DDD を持ち出す理由は、用語を増やすためではありません。  
WordPress の中にある概念の責務を、見えやすくするためです。

## Gutenberg は純粋な CQRS でも Event Sourcing でもない

**初学者**  
ここまで聞くと、Gutenberg は CQRS っぽくて、Event Sourcing っぽいけれど、どちらでもないという感じですね。

**解説者**  
その整理でよいです。  
Gutenberg は、Write と Read を分けて考えるとかなり見やすいですし、内部状態の変化をイベント的に捉えることにも意味があります。  
しかし、純粋な CQRS 実装でも純粋な Event Sourcing 実装でもありません。

**実装者**  
むしろ重要なのは、その“不完全さ”です。  
WordPress は理論をきれいに満たすことより、編集可能な HTML 資産を現実的に運用することを優先しています。  
だから編集時にはアプリ的で、保存時には HTML スナップショット的で、公開時にはテーマや既存 Web 基盤と強く結びつく。  
この折衷が WordPress の特徴です。

**初学者**  
理論的に中途半端なのではなく、現実的な設計の着地点なのですね。

**解説者**  
その通りです。  
この本全体でも見てきたように、WordPress は単なる古い CMS ではなく、別の最適化を持つシステムです。  
Gutenberg を DDD / CQRS 的に読むと、その“別の最適化”がかなり見えやすくなります。

## この見方は実務で何に効くのか

**初学者**  
最後に、この見方が実務でどう役立つのかを知りたいです。  
理論の整理としては面白いですが、実装や運用の判断に効くのでしょうか。

**解説者**  
かなり効きます。  
たとえば、ある変更が Write 側の UI 改善なのか、Read 側の保存契約変更なのかを分けて考えられます。  
また、互換性の影響がどこに出るかも見やすくなります。  
さらに、テーマやブロックや編集体験の責務分界も整理しやすくなります。

**実装者**  
実務では、「何を変えたのか」を正しく言えることが重要です。  
見た目だけ変えたつもりが保存構造を変えていた。  
編集 UI を直したつもりが既存コンテンツの再編集性に影響していた。  
そういう事故は珍しくありません。  
Write / Read / 保存契約 / 制約設計を分けて見られると、レビューもテストもかなりやりやすくなります。

**初学者**  
なるほど。  
この章の見方は、抽象的な整理だけでなく、影響範囲の見積もりにも効くのですね。

**解説者**  
その通りです。  
だからこの章で大切なのは、「Gutenberg は CQRS だ」と結論づけることではありません。  
むしろ、Gutenberg をそうしたレンズで読むと、WordPress の内部構造がかなり説明しやすくなる、ということです。

### 整理

- Gutenberg は、editor state と saved HTML を同一視すると分かりにくくなる。  
- editor state は編集中の動的な内部状態であり、Write モデルに近い。  
- saved HTML は保存結果として残る資産であり、Read モデルに近い。  
- Write と Read を分けて考えると、エディタのアプリ性、保存 HTML の重さ、変更の影響範囲が見やすくなる。  
- Undo / Redo や内部状態遷移は、Gutenberg をイベント的に見る視点を支える。  
- ただし、WordPress は変更イベント列そのものを永続化の中心にしているわけではなく、saved HTML というスナップショットを重視している。  
- そのため Gutenberg は、純粋な CQRS や純粋な Event Sourcing ではなく、Event 的な編集と Snapshot 的な保存を組み合わせたシステムとして読むのが適切である。  
- DDD 的な読み方は、ブロック、編集状態、保存資産、テーマ制約の責務境界を見やすくする。  

### この章の核心

Gutenberg は、編集時の内部状態と保存・表示される結果を分けて考えると見えやすくなる。つまり、Write モデルとしての editor state と、Read モデルとしての saved HTML を切り分けることが重要である。ただし WordPress は純粋な CQRS や Event Sourcing ではなく、Event 的な編集と Snapshot 的な保存を現実的に組み合わせたシステムである。

## まとめ

- Gutenberg を理解するには、editor state と saved HTML を分けて考える必要がある。  
- editor state は Write 側、saved HTML は Read 側として見ると整理しやすい。  
- Undo / Redo や状態遷移は、Gutenberg の編集アプリとしての性格を示している。  
- WordPress はイベント列を真実として保存するのではなく、saved HTML というスナップショットを重視する。  
- DDD / CQRS 的な読み方は、理論の適用そのものではなく、WordPress の責務分界を説明しやすくするためのレンズとして有効である。  

次章では、この見方をさらに具体的な設計要素へ下ろし、`block.json` と `theme.json` が何を定義しているのかを考えます。  
ブロックの契約とテーマの方針・制約を分けて見ることで、WordPress の編集空間がどう成立しているのかを整理していきます。
