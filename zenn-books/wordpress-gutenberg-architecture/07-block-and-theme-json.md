# 第6章 block.json と theme.json は何を定義しているのか

**初学者**  
`block.json` と `theme.json` は、WordPress のブロックテーマ時代でよく出てきますよね。  
でも正直、どちらも設定ファイルに見えてしまいます。  
違いをうまく説明できません。

**解説者**  
その混同はとても自然です。  
どちらも JSON ですし、どちらもテーマやブロックのふるまいに関わります。  
ただし、この章で一番大事なのは、`block.json` と `theme.json` は同じ種類の設定ではない、ということです。

**実装者**  
実務でも、ここが曖昧なままだと設計がぶれます。  
`block.json` はブロックという単位の契約に近いです。  
一方 `theme.json` は、テーマが編集空間全体に与える方針や制約に近いです。

**初学者**  
契約と方針、ですか。  
どちらも「できることを決める」ように見えるのですが、何が違うのでしょうか。

**解説者**  
この章の中心命題はこうです。  
`block.json` は「そのブロックは何者で、どんな能力を持ち、どう扱ってよいか」を定義する。  
`theme.json` は「そのサイトでは、どのような編集を許し、どのような制約のもとで揃えるか」を定義する。  
つまり前者は単位の契約、後者は空間の方針です。

## まず、なぜこの2つを分ける必要があるのか

**初学者**  
そもそも、なぜこれを別ファイルで持つ必要があるのでしょうか。  
ひとつにまとめたほうが分かりやすい気もします。

**解説者**  
ひとつにまとめると、責務が混ざるからです。  
ブロックという単位の定義と、テーマ全体のデザイン方針や制約は、関心の粒度が違います。  
この本では責務分界を重視してきましたが、ここでも同じです。  
個々の部品に関する契約と、全体空間に対するルールは分けて考えたほうが整理しやすいのです。

**実装者**  
たとえば、段落ブロックや画像ブロックのような部品には、「これはどんな名前のブロックか」「どんな属性を持つか」「どんな機能をサポートするか」といった情報が必要です。  
一方でテーマ側には、「このサイトではどの色を使わせるか」「余白やタイポグラフィをどう揃えるか」「エディタでどんな選択肢を見せるか」といった情報が必要です。  
同じ“設定”に見えても、対象が違います。

**初学者**  
なるほど。  
ブロックは部品の話で、テーマは空間の話なのですね。

**解説者**  
そうです。  
ただし、それだけだと少し抽象的です。  
この章では、`block.json` を schema / capability / 契約として、`theme.json` を policy / constraint として読む見方を固めていきます。

## `block.json` は何を定義しているのか

**初学者**  
では、まず `block.json` からお願いします。  
これは何を定義していると考えるのが一番よいのでしょうか。

**解説者**  
一番よいのは、`block.json` を「ブロックという単位の自己記述」として捉えることです。  
そのブロックが何者で、どのような名前で識別され、どんな属性やサポート機能を持ち、どんな編集・保存の文脈で扱われるかを示します。

**実装者**  
実務的に言うと、`block.json` があることで、ブロックが場当たり的な UI 部品ではなくなります。  
「このブロックは何か」「どんな入力を持つか」「どんな機能が許されているか」が、まとまった単位として見えるようになります。  
だからエディタ実装やテーマ側の対応も、暗黙の前提ではなく契約に寄せやすくなるんです。

**初学者**  
自己記述ということは、ブロック自身のプロフィールみたいなものですか。

**解説者**  
かなり近いです。  
ただし、単なる紹介文ではありません。  
プロフィールであると同時に、そのブロックをシステムがどう扱うかの境界面でもあります。  
だから `block.json` は、見た目の説明書というより、機械可読な契約に近いのです。

## schema としての `block.json`

**初学者**  
さっき schema という言葉が出ました。  
これはどういう意味ですか。

**解説者**  
ここでいう schema は、「そのブロックがどんな構造のデータを持つかを整理する枠組み」という意味です。  
ブロックは見た目の部品ですが、同時に属性を持つ構造単位でもあります。  
テキスト、配置、画像参照、スタイルの選択肢など、ブロックごとに持つ情報は違います。  
`block.json` は、そのブロックをどういう構造のものとして扱うかを整理する入口になります。

**実装者**  
ここで重要なのは、schema があることで、編集・保存・再編集の流れが安定しやすくなることです。  
何を持つブロックなのかが曖昧だと、エディタ側もテーマ側も、どこまで何を前提にしてよいか分かりません。  
逆に schema 的な整理があると、ブロックがシステムの一級単位として扱いやすくなります。

**初学者**  
つまり、`block.json` は「見た目の設定」ではなく、「そのブロックが持つ構造の輪郭」を定義しているのですね。

**解説者**  
その理解でよいです。  
そして、この輪郭があるからこそ、ブロックは単なる HTML の断片ではなく、再編集可能な構造単位として成立します。

## capability としての `block.json`

**初学者**  
では capability は何でしょうか。  
schema とどう違うのですか。

**解説者**  
schema が「そのブロックはどんな構造を持つか」だとすると、capability は「そのブロックに何を許すか」です。  
ブロックには、揃え方、色、余白、タイポグラフィ、配置、入れ子の扱いなど、さまざまな機能的能力があります。  
`block.json` は、そのブロックがどの能力を持つ前提でシステムに参加するかを表す面も持っています。

**実装者**  
ここを曖昧にすると、実装が破綻しやすいです。  
たとえば、あるブロックに本来想定していないスタイル変更を無制限に許すと、見た目が崩れたり、編集体験が複雑になったりします。  
逆に、必要な能力が定義されていないと、ブロックが使いにくくなります。  
つまり capability は、自由度の宣言であると同時に、責務の境界でもあるんです。

**初学者**  
能力を増やせば便利になる、という単純な話ではないわけですね。

**解説者**  
その通りです。  
能力は多ければよいわけではありません。  
どの能力をそのブロックに持たせるかは、編集上の責務と運用上の安全性を見ながら決める必要があります。  
この意味でも `block.json` は、単なる登録ファイルではなく、設計判断が現れる場所です。

## 契約としての `block.json`

**初学者**  
ここまで聞くと、`block.json` が契約だという意味も分かってきました。  
でも、なぜあえて契約という言葉を使うのでしょうか。

**解説者**  
契約という言葉を使う理由は、`block.json` が「このブロックはこういうものとして扱う」という約束の面を持つからです。  
ブロックを使う側、編集する側、表示する側、それぞれが無制限に想像で扱ってしまうと、システムは安定しません。  
だから「この単位はこういう能力と構造を持つ」と、ある程度明示する必要があります。

**実装者**  
実務感覚で言えば、契約があると変更の影響範囲を考えやすいです。  
あるブロックの属性やサポート機能を変えると、編集画面、保存内容、表示結果、既存コンテンツの互換性に影響が出るかもしれない。  
そういうとき、契約として捉えていれば、「これは単なる内部実装変更ではない」と判断しやすくなります。

**初学者**  
なるほど。  
ブロックを軽く変えたつもりでも、保存済みコンテンツとの関係まで含めると重い変更になりうるのですね。

**解説者**  
まさにそこです。  
Gutenberg 以降の WordPress では、ブロックは保存される構造単位です。  
だから `block.json` は、今この瞬間の UI 設定だけではなく、将来の再編集可能性や互換性とも関係してきます。

## `theme.json` は何を定義しているのか

**初学者**  
では `theme.json` に移りたいです。  
こちらは方針や制約だと言っていましたが、もう少し具体的にお願いします。

**解説者**  
`theme.json` は、テーマが編集空間全体に対して与えるルールの束として読むと分かりやすいです。  
個々のブロックのプロフィールではなく、このサイトやテーマではどのようなデザイン選択肢を許し、何を揃え、どこまでを編集者に委ねるかを定義します。

**実装者**  
つまり `theme.json` は、「このサイトでは自由にしてよい領域」と「揃えるべき領域」を編集体験の中に埋め込む仕組みです。  
後から CSS で見た目を整えるだけではなく、エディタの段階で選択肢をコントロールできます。  
ここがクラシックテーマ時代との大きな違いです。

**初学者**  
後から直すのではなく、先にルールを入れる感じですね。

**解説者**  
その通りです。  
ブロックテーマ時代のテーマは、出来上がった HTML にスタイルを当てるだけでは弱いのです。  
編集者が自由に組める以上、事後修正では間に合わないことが多い。  
だから `theme.json` は、編集前提のルールをテーマ側から定義する仕組みとして重要になります。

## policy としての `theme.json`

**初学者**  
policy という言葉は、設計思想や運用方針に近い感じでしょうか。

**解説者**  
はい。  
ここでの policy は、「このテーマでは、どういう方向へ編集を導きたいか」という方針です。  
色や余白やタイポグラフィの選択肢、レイアウトの基準、既定値の置き方などに、そのテーマの思想が表れます。

**実装者**  
たとえば、企業サイトでブランドカラーを厳密に守りたいのか、店舗サイトである程度柔軟に更新させたいのかによって、与える選択肢は変わります。  
編集者の自由度を高く取るのか、事故を減らすために絞るのかも違います。  
`theme.json` は、そうした方針をコードではなく構成として表現しやすくする場所です。

**初学者**  
ということは、`theme.json` を見ると、そのテーマが何を重視しているかも見えやすいのですね。

**解説者**  
その通りです。  
だから `theme.json` は、単なる技術ファイルというより、編集環境に埋め込まれた設計方針の表現だと考えると理解しやすいです。

## constraint としての `theme.json`

**初学者**  
そして constraint は制約ですよね。  
でも制約というと、単に不便にするだけにも聞こえます。

**解説者**  
制約は不便のためではありません。  
整合性を守るためです。  
ブロックエディタは自由度が高いので、何も制約しないと、ページごとにスタイルや余白や構造が散らばりやすくなります。  
そこで `theme.json` は、編集者が歩ける範囲をテーマ側で絞り、サイト全体の一貫性を守る役割を持ちます。

**実装者**  
実務では、制約がない自由は運用コストを上げます。  
最初の数ページはきれいでも、担当者が増え、更新回数が増えると、見た目のばらつきや予期しない崩れが出てきます。  
制約は、その場の自由を少し減らす代わりに、長期運用の安定性を上げるためのものです。

**初学者**  
なるほど。  
制約は自由の反対ではなく、運用のためのガードレールなのですね。

**解説者**  
とてもよい表現です。  
ブロックテーマ時代の設計では、制約は後ろ向きなものではありません。  
むしろ「安全に編集できる自由」を成立させる条件です。  
`theme.json` は、そのガードレールをテーマ側から明示する手段だと言えます。

## `block.json` と `theme.json` の関係

**初学者**  
ここまで聞いて、それぞれの役割は見えてきました。  
でも実際には、この2つはどう関係するのでしょうか。

**解説者**  
両者の関係は、「単位の能力」と「空間の方針」の関係として捉えると分かりやすいです。  
`block.json` は、そのブロックにどんな能力があるかを示します。  
`theme.json` は、その能力のうち何をどう使わせるか、サイト全体としてどう揃えるかを示します。

**実装者**  
たとえば、あるブロックが色や余白の変更に対応できるとしても、テーマ側がその選択肢を絞ることがあります。  
逆に、テーマ側が広いデザイン空間を用意しても、ブロック自身がその能力を持っていなければ使えません。  
つまり片方だけでは完結しません。  
部品の契約と、空間の制御が噛み合ってはじめて、編集体験が成立します。

**初学者**  
ということは、`block.json` が「何ができるか」、`theme.json` が「何をやってよいか」に近いのですね。

**解説者**  
非常によい整理です。  
もちろん現実はもう少し複雑ですが、学習の出発点としてはその理解が有効です。  
`block.json` は capability の側に寄り、`theme.json` は constraint の側に寄る。  
この軸で見ると、両者の違いがかなり見えやすくなります。

## できることと、やってよいこと

**初学者**  
今の話は大事そうですね。  
「できること」と「やってよいこと」は違う、ということでしょうか。

**解説者**  
その通りです。  
これは WordPress だけでなく、設計全般で重要な区別です。  
ある機能が技術的に実現可能であることと、それをそのサイトで許容することは同じではありません。

**実装者**  
ブロックが多機能でも、サイト全体としては絞ったほうがよい場合があります。  
逆に、ある程度の柔軟性を認めたほうが運用に合うこともあります。  
この調整を行う場所が `theme.json` です。  
だから `block.json` が capability を広げ、`theme.json` が policy と constraint で着地を決める、という見方ができます。

**初学者**  
それはまるで、API の仕様と利用ポリシーの違いみたいですね。

**解説者**  
かなり近いです。  
`block.json` はブロック単位の仕様に近く、`theme.json` はその仕様をどういう編集空間の中で運用するかのルールに近い。  
この違いを押さえると、ブロックテーマの設計意図が読みやすくなります。

## editor action との役割分担

**初学者**  
ここで少し疑問です。  
実際の編集では、ユーザーがエディタで操作しますよね。  
その操作との関係で `block.json` と `theme.json` はどう位置づければよいのでしょうか。

**解説者**  
よい問いです。  
編集者は editor action を通じて、ブロックを追加し、並べ替え、属性を変更し、構造を組み替えます。  
そのとき、何をどう操作できるかの前提を支えるのが `block.json` であり、どこまでその操作を許容し、どんな着地へ導くかを支えるのが `theme.json` です。

**実装者**  
つまり editor action は動的な操作です。  
それに対して `block.json` と `theme.json` は、操作の舞台装置に近いです。  
前者はブロックという役者の能力表、後者は舞台全体のルールと照明設計のようなものです。  
編集者はその舞台で動くわけです。

**初学者**  
なるほど。  
操作そのものを定義しているわけではなく、操作の前提条件を定義しているのですね。

**解説者**  
その通りです。  
だから `block.json` と `theme.json` を読むことは、単に設定項目を覚えることではありません。  
このサイトの編集空間が、どんな前提で成立しているかを読むことなのです。

## デザインシステムとの接続

**初学者**  
`theme.json` の話を聞いていると、デザインシステムにも近い気がしてきました。

**解説者**  
とても重要な観察です。  
`theme.json` は、テーマにおけるデザインシステムの入口として読めます。  
もちろん完全に同じではありませんが、色、タイポグラフィ、余白、レイアウトなどを統一的に扱い、編集者が使う選択肢として提示する点で、かなり近い役割を持ちます。

**実装者**  
クラシックテーマ時代は、デザインルールが CSS やテンプレートの中に分散しやすかったです。  
ブロックテーマ時代は、その一部を編集空間に持ち上げられるようになりました。  
すると、テーマは単なる見た目の実装ではなく、運用可能なデザインルールの提供者になります。

**初学者**  
デザインを完成画面に閉じ込めるのではなく、編集者が使うルールとして渡すわけですね。

**解説者**  
その通りです。  
だから `theme.json` を理解することは、テーマを CSS の集合として見るのではなく、編集環境に埋め込まれたデザインポリシーとして見ることでもあります。

## なぜこの区別が実務で重要なのか

**初学者**  
最後に、実務でなぜここまでこの区別が重要なのかを整理したいです。

**解説者**  
理由は、区別が曖昧だと責務が混ざるからです。  
ブロックが持つべき能力の問題なのか、テーマ全体として許すべき編集自由度の問題なのかが混ざると、設計判断が場当たり的になります。  
すると、あとから「なぜこのブロックだけ扱いが違うのか」「なぜこのサイトだけ自由度が高すぎるのか」といった混乱が起きやすくなります。

**実装者**  
実装の現場では、問題が起きたときの切り分けにも効きます。  
これはブロック定義の問題なのか。  
テーマ方針の問題なのか。  
エディタ操作の問題なのか。  
この見分けがつくと、設計もテストもレビューもしやすくなります。

**初学者**  
なるほど。  
単なるファイル知識ではなく、責務分解のために重要なのですね。

**解説者**  
まさにそこが、この章の着地点です。  
`block.json` と `theme.json` は、WordPress を「編集可能な HTML 資産を、制約付きで安全に構築するシステム」として理解するうえで、責務の違いをはっきり見せてくれる存在です。  
片方は部品の契約、もう片方は空間の方針。  
この区別が見えれば、ブロックテーマ時代の WordPress はかなり説明しやすくなります。

### 整理

- `block.json` は、ブロックという単位の自己記述であり、schema / capability / 契約として読める。  
- `theme.json` は、テーマが編集空間全体に与えるルールであり、policy / constraint として読める。  
- `block.json` は「そのブロックが何者で、何ができるか」に関わる。  
- `theme.json` は「そのサイトで何を許し、何を揃え、どこで止めるか」に関わる。  
- 前者は部品の契約、後者は空間の方針であり、粒度も責務も異なる。  
- 両者は対立するものではなく、ブロックの能力とテーマ全体の制約を噛み合わせることで、編集体験を成立させる。  

### この章の核心

`block.json` はブロックという単位の契約を定義し、`theme.json` は編集空間全体の方針と制約を定義する。だから両者の違いは、設定項目の違いではなく、責務の粒度の違いとして理解する必要がある。

## まとめ

- `block.json` は、ブロックの構造と能力を定義する契約である。  
- `theme.json` は、テーマ全体の編集方針と制約を定義する。  
- 「できること」と「やってよいこと」は違い、その区別が両者に表れている。  
- `block.json` と `theme.json` は、部品と空間という異なる粒度で編集体験を支える。  
- この区別を押さえると、ブロックテーマ時代の設計・実装・テストの切り分けがしやすくなる。  

次章では、この「能力」と「制約」の関係をさらに踏み込み、テーマ設計そのものを制約設計として捉え直します。  
ブロックテーマ時代に、なぜテーマが単なる UI 設計ではなく、編集環境の制約設計になるのかを見ていきます。
