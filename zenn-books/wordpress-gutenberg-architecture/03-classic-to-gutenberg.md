# 第2章 クラシック WordPress から Gutenberg へ

**初学者**  
WordPress って、もともとは PHP のテンプレートを組み合わせてページを出す CMS でしたよね。  
Gutenberg が入っても、見た目が少しリッチになっただけではないのですか。

**解説者**  
そこが、この章の出発点です。  
たしかに WordPress は長いあいだ、テンプレート階層と `functions.php`、そしてフックを中心に拡張される CMS として理解されてきました。ですが Gutenberg は、単なるエディタ刷新ではありません。クラシック WordPress の延長線上にありながら、設計上の重心をかなり移しました。:contentReference[oaicite:0]{index=0}

**実装者**  
実務でも、その違いはかなり大きいです。  
クラシックな時代は「このページをどの PHP テンプレートで描くか」が主役でした。Gutenberg 以降は「どのブロックをどう組み合わせ、どんな編集状態を経て、どんな HTML を保存するか」が重要になります。PHP だけ見ていても全体像がつかみにくくなりました。:contentReference[oaicite:1]{index=1}

**初学者**  
でも WordPress は今でも HTML を出力する CMS ですよね。  
そこまで大きく変わったと言うと、少し大げさにも聞こえます。

**解説者**  
その感覚は半分正しくて、半分違います。  
この章では、クラシック WordPress から Gutenberg への変化を、連続性と断絶の両方から整理します。章の中心命題はこうです。  
WordPress は「ページを組み立てる CMS」から、「編集可能な構造単位を扱う CMS」へ重心を移した、ということです。これは完全な別物化ではありませんが、従来の理解だけでは説明しきれない変化です。

## クラシック WordPress の基本的な見え方

**初学者**  
まず、クラシック WordPress の理解を確認したいです。  
昔ながらの WordPress は、どういう見方をすると整理しやすいのでしょうか。

**解説者**  
クラシック WordPress は、ひとことで言えば「ページを組み立てる CMS」として理解しやすいです。  
投稿や固定ページのデータがあり、テンプレート階層に従って適切な PHP テンプレートが選ばれ、必要なロジックやフックを通じて最終的な HTML が組み立てられます。

**実装者**  
たとえば、`single.php` や `page.php`、`archive.php` のようなテンプレートがありました。  
どのテンプレートが使われるかは、表示対象の種類によってある程度決まります。そこに `functions.php` でメニューやウィジェットやテーマ機能を登録し、各種フックで振る舞いを差し込んでいく。かなり PHP 中心の世界です。

**初学者**  
つまり、データが先にあって、それをテンプレートがどう表示するかを決める感じですね。

**解説者**  
その通りです。  
ここではページという単位が主役です。ページ全体が先にあり、その内部にどんな部品があるかは、テーマ側の実装やエディタ入力に強く依存していました。もちろんショートコードやカスタムフィールドのような仕組みはありましたが、今のブロックのように、編集単位そのものが一級の概念として前面に出ていたわけではありません。

**実装者**  
だからクラシック時代のテーマ開発では、「この画面をどのテンプレートで出すか」「どこでフックするか」「管理画面で入力された値をどう埋め込むか」が主要テーマになります。  
設計の中心は、レンダリングと拡張ポイントでした。

## `functions.php` とフックが担っていたもの

**初学者**  
`functions.php` とフックは、昔の WordPress を象徴する仕組みですよね。

**解説者**  
そうです。  
クラシック WordPress の柔軟性は、かなりの部分をフック文化が支えていました。テーマやプラグインは、コアの処理そのものを大きく書き換えずに、既存の流れの途中に処理を差し込めます。これは WordPress の大きな強みでした。

**実装者**  
実際、WordPress の拡張と聞いて多くの人が思い浮かべるのは、`add_action()` や `add_filter()` です。  
画面の前後に何かを出したり、クエリを調整したり、本文を加工したり、管理画面を拡張したりする。WordPress はこの仕組みで巨大なエコシステムを育ててきました。

**初学者**  
それなら、Gutenberg が入ってもフックで拡張する世界のままではないのですか。

**解説者**  
そこが連続性と断絶の分岐点です。  
フック文化そのものは消えていません。WordPress が拡張可能な CMS であることは変わりません。  
ただし、拡張の主戦場が変わりました。クラシック時代は、PHP が組み立てるページや処理の流れに割り込むことが中心でした。Gutenberg 以降は、エディタ上で扱う構造単位、つまりブロックの登録、属性、編集 UI、保存方法、制約へと関心が移ります。

**実装者**  
言い換えると、昔は「既にあるページ生成パイプラインに介入する」感覚が強かった。  
今は「そもそも何を編集単位として認めるか」「その単位をどう保存し、どう再編集可能にするか」を設計する感覚が強いです。

## 「ページを組み立てる CMS」という理解の限界

**初学者**  
でも WordPress は結局、ページを見せるための CMS ですよね。  
「ページを組み立てる CMS」という理解は、今でも十分ではないですか。

**解説者**  
最終的にページを出す、という意味では十分です。  
しかし、Gutenberg 以降の WordPress を説明するには不十分です。なぜなら、いまの WordPress では「ページがどう表示されるか」だけでなく、「編集者がどんな単位で内容を構築し、その構造がどう保存され、どこまで安全に再編集できるか」が中心問題になっているからです。:contentReference[oaicite:3]{index=3}

**実装者**  
クラシックな理解だと、本文はかなり大きな自由記述領域でした。  
その中に HTML を書いたり、ショートコードを埋めたり、独自フィールドをテンプレート側で読み出したりします。  
一方 Gutenberg は、見出し、段落、画像、カラム、グループのようなブロックごとに構造を明示します。この差は大きいです。

**初学者**  
なるほど。  
以前はページが先にあって中身が曖昧だったけれど、今は中身の単位が前に出てきたわけですね。

**解説者**  
その理解でよいです。  
クラシック WordPress では「ページをどう描くか」が第一でした。Gutenberg では「ページを何の単位で構成するか」が設計上の主題として前景化しました。

## ブロックという抽象単位

**初学者**  
この章でいちばん大事なのは、やはりブロックですか。

**解説者**  
そうです。  
Gutenberg の核心は、ブロックという抽象単位を WordPress の中心に据えたことです。  
ブロックは単なる見た目の部品ではありません。編集 UI、属性、保存形態、再編集可能性をまとめて扱う単位です。

**実装者**  
実務上も、ブロックはただのコンポーネントではありません。  
あるブロックには、どんな属性があり、エディタではどう編集され、保存時にはどんな HTML を出し、フロントではどう解釈されるか、という一連の契約が含まれます。  
つまりブロックは、UI の部品であると同時に、保存可能な構造単位でもあります。

**初学者**  
保存可能な構造単位、という言い方は面白いですね。

**解説者**  
ここが重要です。  
クラシック時代の WordPress でも、見た目上の部品はありました。ウィジェットやショートコードやテンプレートパーツがあります。  
しかし Gutenberg のブロックは、それらより一段深く、編集・保存・再表示をまたぐ単位として統合されています。この統合が、従来との大きな違いです。

## エディタのアプリ化

**初学者**  
Gutenberg は「エディタの刷新」と言われることが多いですが、本当にそれだけではないのですね。

**解説者**  
はい。  
Gutenberg の変化をただのエディタ刷新と見ると、構造を見失います。  
Gutenberg で起きたのは、エディタが単なる入力欄ではなく、状態を持つアプリケーションになったことです。ブロックを追加し、並べ替え、入れ子にし、属性を変え、履歴を持ちながら編集する。これは、従来のテキスト入力中心の編集体験とはかなり異なります。

**実装者**  
ここで JavaScript の比重が上がります。  
なぜ WordPress を理解するのに PHP と JavaScript の両方が必要なのか、と第1章で問題提起していましたが、その理由のひとつがこれです。PHP だけ見ていると、保存後の HTML やテーマ側の描画は理解できます。しかし、編集体験そのものや内部状態の変化は読み取れません。:contentReference[oaicite:6]{index=6}

**初学者**  
つまり、昔のエディタは入力欄だったけれど、今のエディタはアプリになった。

**解説者**  
そのまとめで大筋は合っています。  
ただし、ここでも単純に「モダンになった」と言うだけでは足りません。WordPress は完全な SPA になったわけではありませんし、保存の中心に HTML を残す設計も維持しています。  
エディタはアプリ化したが、保存と公開の思想は WordPress らしいまま残っている。このねじれが Gutenberg の理解を難しくも面白くもしています。

## 保存 HTML の位置づけはどう変わったのか

**初学者**  
保存の中心に HTML が残る、というのはどういう意味でしょうか。  
むしろ Gutenberg は内部で状態を持つなら、JSON 的なものが真実になったのではないですか。

**解説者**  
そこが WordPress らしいところです。  
Gutenberg には editor state があります。しかし最終的には、保存された HTML が重要な資産として残ります。  
つまり、編集時にはアプリケーション的な状態がありながら、保存時には HTML を重視する。この構造が、WordPress を純粋な SPA や純粋なアプリケーションデータ駆動のシステムと異なるものにしています。:contentReference[oaicite:7]{index=7}

**実装者**  
実務で見ると、この差はかなり効きます。  
たとえばある記事を開いたとき、データベース上にはブロック由来の HTML が保存されています。もちろんブロックコメントや属性表現も絡みますが、発想としては「構造を持った HTML 資産」が残る。  
だから WordPress は、編集アプリでありながら、長期運用される HTML 資産のシステムでもあるんです。

**初学者**  
なるほど。  
Gutenberg でアプリ化したからといって、保存対象が完全にアプリ内部のデータ形式へ移ったわけではないのですね。

**解説者**  
その通りです。  
ここで「保存 HTML の位置づけの変化」という言い方をすると、完全な交代ではなく、意味の変化として捉えるのがよいです。  
クラシック時代の HTML は、テンプレートや本文入力の結果として出てくるものでした。Gutenberg 以降の HTML は、再編集可能な構造単位の保存結果でもあります。  
同じ HTML でも、その背後にある編集モデルが変わっています。

## 連続性と断絶をどう見るか

**初学者**  
ここまで聞くと、WordPress が別物になったようにも聞こえます。  
でも「連続性もある」と言っていましたよね。

**解説者**  
はい。  
ここを片方だけで語ると誤解します。  
連続性としては、WordPress が今でも CMS であり、HTML を重視し、拡張可能で、テーマやプラグインの文化を維持している点があります。  
断絶としては、編集の中心単位がページ全体からブロックへ移り、エディタがアプリ化し、PHP だけでは説明できないシステムになった点があります。

**実装者**  
実務で一番混乱しやすいのは、この両方が同時に真であることです。  
昔の延長だと思うと JavaScript 側や editor state を見落とします。  
逆に完全な別物だと思うと、WordPress がなぜ HTML を重視し続けるのか、なぜ従来資産との互換や長期運用を気にするのかが見えなくなります。

**初学者**  
つまり、「同じ WordPress のまま、大事な設計単位が変わった」と理解するのが近いですか。

**解説者**  
とてもよい整理です。  
Gutenberg は WordPress を捨てたのではありません。  
WordPress の中核を、テンプレート中心の CMS 理解だけでは足りない地点へ動かしました。だからこそ、古い知識が無意味になったわけではないが、それだけでは不足するのです。

## 従来テーマとの連続性と断絶

**初学者**  
テーマ開発の観点では、どう整理するとよいでしょうか。  
従来テーマの知識はどこまで通用するのですか。

**解説者**  
ここでも、通用する部分と通用しない部分があります。  
テンプレート、スタイル、PHP による表示調整、フックによる拡張といった知識は依然として重要です。  
ただし、それだけでは不十分です。ブロックの存在によって、テーマは「画面を描くもの」から「編集可能な構造がどう成立するかを支えるもの」へと役割が変わり始めました。これは次章で本格的に扱います。

**実装者**  
従来テーマでは、テーマ側がかなり強くレイアウトを握っていました。  
もちろん柔軟性はありましたが、編集者が扱う構造単位は限定的でした。  
Gutenberg 以降は、編集者がブロックを使ってかなり多くの構造を作れます。するとテーマ側は、単に見た目を与えるだけでなく、どこまで自由にして、どこに制約をかけるかを考えなければなりません。

**初学者**  
それは、テーマの責任が重くなったということですか。

**解説者**  
重くなったというより、質が変わったと言うほうが正確です。  
クラシックテーマでは、表示ロジックとテンプレート設計が中心でした。  
Gutenberg 以降のテーマでは、編集体験や制約設計との接点が増えます。  
だから次章では、ブロックテーマが何を設計対象にしたのかを扱う必要があります。今回の章は、その前提として、なぜクラシックな理解だけでは足りなくなったのかを整理する章です。

### 整理

- クラシック WordPress は、テンプレート階層・`functions.php`・フックを中心に、ページを組み立てる CMS として理解しやすかった。  
- Gutenberg は、ページ全体よりもブロックという構造単位を前面に出した。  
- ブロックは見た目の部品ではなく、編集 UI・属性・保存・再編集可能性を束ねる単位である。  
- エディタは単なる入力欄ではなく、状態を持つアプリケーションとしての性格を強めた。  
- それでも WordPress は保存 HTML を重視し続けており、完全な SPA や純粋な JSON 中心システムにはなっていない。  
- したがって Gutenberg 以降の WordPress は、クラシック WordPress の連続上にありながら、設計上の主役を変えたシステムとして理解する必要がある。  

### この章の核心

Gutenberg は WordPress を別の製品にしたのではなく、WordPress の中心を「ページの描画」から「編集可能な構造単位の管理」へ移した。だからクラシック WordPress の知識は今でも有効だが、それだけでは現在の WordPress を説明できない。

## まとめ

- クラシック WordPress の中心は、テンプレート階層・`functions.php`・フックによるページ組み立てだった。  
- Gutenberg 以降は、ページ全体ではなくブロックが設計上の主要単位になった。  
- エディタは入力欄からアプリケーションへ近づき、JavaScript 側の理解が不可欠になった。  
- 保存 HTML は今も重要であり、WordPress はアプリ化しながら HTML 資産の思想を維持している。  
- Gutenberg への移行は、連続性と断絶の両方として捉える必要がある。  

次章では、この変化がテーマ設計をどう変えたのかを扱います。  
とくに、ブロックテーマが何を設計対象にし、なぜ `theme.json` やテンプレートパーツのような仕組みが重要になるのかを見ていきます。
