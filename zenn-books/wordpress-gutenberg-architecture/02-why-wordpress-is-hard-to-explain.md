# 第1章 なぜ Gutenberg 以降の WordPress は説明しにくいのか

**初学者**  
WordPress って、昔からよく使われてきた CMS ですよね。  
なのに、Gutenberg 以降の WordPress は説明しにくい、と言われるのはなぜでしょうか。  
新しい機能が増えただけなら、むしろ説明しやすくなってもよさそうです。

**解説者**  
そこが、この本の出発点です。  
Gutenberg 以降の WordPress が説明しにくいのは、単に機能が増えたからではありません。  
従来の WordPress を理解するための言葉や見取り図だけでは、いまの WordPress の構造を十分に表せなくなったからです。

**実装者**  
実務で触っている人ほど、その感覚を持ちやすいです。  
投稿できる。テーマも触れる。プラグインも入れられる。  
そこまでは分かるのに、「Gutenberg 以降の WordPress はどういうシステムなのか」と聞かれると、急に言葉が詰まる。  
これは知識不足というより、整理の枠組みが追いついていないことが多いです。

**初学者**  
たしかに、WordPress を使える人は多いのに、説明できる人は少ない気がします。  
この章では、その理由を整理するわけですね。

**解説者**  
はい。  
この章の中心命題はこうです。  
Gutenberg 以降の WordPress は、クラシックな WordPress の延長上にありながら、設計の重心が変わったため、従来の説明枠だけでは足りなくなった。  
だから説明しにくいのです。

## 「使える」と「説明できる」は違う

**初学者**  
まず、そこから気になります。  
WordPress が使えるなら、説明もできそうな気がするのですが、違うのでしょうか。

**解説者**  
違います。  
「使える」は操作や実務経験の蓄積です。  
一方で「説明できる」は、その仕組みをどんな単位で捉え、どこに責務があり、何が変わったのかを言葉にできることです。  
この二つは重なりますが、同じではありません。

**実装者**  
たとえば、ある人が長年 WordPress の案件をやっていて、既存テーマを修正したり、プラグイン設定を調整したり、必要なカスタマイズをこなしていたとします。  
それでも、「なぜ Gutenberg 以降は PHP だけでは全体像を把握しにくいのか」「ブロックテーマは何を設計対象にしているのか」と聞かれると、答えにくいことがあります。  
日々の作業では困っていなくても、構造の言語化は別の能力だからです。

**初学者**  
なるほど。  
仕事で扱えていることと、システムとして説明できることは別なのですね。

**解説者**  
そうです。  
しかも WordPress は歴史が長く、学び方も断片化しやすい。  
「テーマの作り方」「フックの使い方」「カスタム投稿タイプの登録方法」「エディタの使い方」といった形で個別に覚えやすい反面、それらを一つの地図として統合しにくいのです。  
Gutenberg 以降は、その断片化がさらに進みました。

## クラシック WordPress の理解は、なぜ強かったのか

**初学者**  
でも、昔の WordPress は説明しやすかったのですか。

**解説者**  
比較的しやすかったです。  
もちろん完全に簡単だったわけではありませんが、少なくとも中心に置くべき見取り図が比較的はっきりしていました。  
投稿や固定ページのデータがあり、テンプレート階層があり、`functions.php` やフックがあり、テーマやプラグインで拡張する。  
この枠組みで、かなりの部分を説明できました。

**実装者**  
実務感覚でも、クラシック WordPress は PHP 中心の CMS として整理しやすいです。  
「どのテンプレートが選ばれるか」「どこにフックするか」「本文やカスタムフィールドをどう表示するか」という発想で、主要な構造をかなり追えました。  
テーマ開発も、最終的な画面をどのテンプレート群で組み立てるか、という見方が中心です。

**初学者**  
つまり、昔の WordPress は「ページをどう出すか」を軸に理解しやすかったのですね。

**解説者**  
その通りです。  
ページ全体が主役であり、そのページを PHP テンプレート群でどう描くかが中心でした。  
もちろん細部には複雑さがありますが、少なくとも説明の重心は見えやすかった。  
そのため、多少断片的な知識でも、一つの物語にまとめやすかったのです。

## Gutenberg が持ち込んだ変化は「機能追加」ではない

**初学者**  
では Gutenberg は、その分かりやすい構図を崩したのでしょうか。

**解説者**  
はい。  
ただし、単に新しいエディタを追加したからではありません。  
Gutenberg が持ち込んだのは、「何を主要な単位として WordPress を理解するか」の変化です。  
従来はページやテンプレートが説明の主役でした。  
Gutenberg 以降は、ブロック、編集状態、保存された構造、テーマによる制約といったものが前に出てきます。

**実装者**  
ここで重要なのは、Gutenberg がエディタの見た目を変えただけではないことです。  
ブロックという単位で内容を扱い、エディタ自体が状態を持つアプリケーションに近づきました。  
その結果、WordPress は「HTML を出す PHP の CMS」だけでは説明しにくくなります。

**初学者**  
なるほど。  
外から見るとエディタが変わっただけに見えるけれど、実際にはシステムの見方が変わっているのですね。

**解説者**  
その通りです。  
説明しにくさの原因は、表面上の機能差ではなく、構造の主役交代にあります。  
ところが、多くの人は古い説明枠のまま新しい WordPress を見ようとする。  
だから違和感が生まれます。

## PHP と JavaScript の両方が必要になる

**初学者**  
Gutenberg 以降の WordPress を語るとき、よく「PHP だけでは足りない」と言われます。  
これはなぜですか。

**解説者**  
WordPress の重要な責務が、PHP 側だけに閉じなくなったからです。  
クラシック時代も JavaScript はありましたが、システムの中心説明は PHP 側に寄せやすかった。  
しかし Gutenberg 以降は、ブロックの編集体験、状態遷移、インタラクション、UI の構成が JavaScript 側に大きく乗っています。

**実装者**  
実際、保存後の HTML やフロント表示を見るだけなら PHP 側の理解でかなり追えます。  
けれど、なぜその構造になったのか、編集画面で何が起きているのか、どの能力がどこで制御されているのかを理解しようとすると、JavaScript 側を見ないと分かりません。  
逆に JavaScript だけ見ても、保存 HTML やテーマの最終表示、WordPress の長期運用の思想はつかみにくいです。

**初学者**  
つまり、片方だけでは半分しか見えないのですね。

**解説者**  
その通りです。  
そして、これが説明を難しくする大きな要因です。  
技術の学習は、どうしても自分の得意領域に寄ります。  
PHP 寄りの人はエディタ内部を見落としやすい。  
JavaScript 寄りの人は、WordPress がなぜ HTML を重視し続けるのかを軽く見やすい。  
両方をまたぐ必要があるため、単純な説明に落とし込みにくいのです。

## エディタが「入力欄」ではなくなった

**初学者**  
JavaScript が重要になるのは分かってきました。  
でも、それは具体的には何が変わったからなのでしょうか。

**解説者**  
大きいのは、エディタが単なる入力欄ではなくなったことです。  
クラシックな感覚では、エディタは本文を書く場所でした。  
多少リッチになっても、あくまで入力欄の延長として見やすかった。  
しかし Gutenberg 以降のエディタは、ブロックを追加し、入れ替え、入れ子にし、スタイルを変え、履歴を持ちながら編集するアプリケーションです。

**実装者**  
この変化はかなり本質的です。  
入力欄なら、最終的に送信されるデータだけを見ればかなり説明できます。  
でも状態を持つアプリケーションになると、画面上で可能な操作、内部状態、保存結果、再編集可能性が絡んできます。  
だから WordPress を説明するための言葉も、「本文をどう出すか」だけでは足りなくなるんです。

**初学者**  
エディタがアプリになると、説明の粒度も変わるわけですね。

**解説者**  
そうです。  
しかも WordPress は、そのアプリ化をしながら、保存対象としての HTML も重視しています。  
完全にアプリケーションデータ中心へ振り切ったわけではない。  
この中間的な立ち位置が、さらに説明を難しくしています。

## ブロックは「部品」以上のものになった

**初学者**  
ブロックという言葉も、説明しにくさの原因ですよね。  
UI 部品のようであり、保存単位のようでもあり、少しつかみどころがありません。

**解説者**  
その感覚は正しいです。  
ブロックは単なる見た目の部品ではありません。  
編集 UI、属性、保存形態、再編集可能性を束ねる単位です。  
つまり、表示と編集と保存をまたぐ抽象単位になっています。

**実装者**  
ここが従来の WordPress とかなり違うところです。  
昔もショートコードやウィジェットやテンプレートパーツはありました。  
でもブロックは、それらよりも一段深く、システムの中心単位として扱われています。  
そのため、「部品」と言うだけでは足りないし、「データ」と言うだけでも足りない。  
説明するときに、既存の言葉へ完全には収まりません。

**初学者**  
つまり、既存の用語のどれか一つに当てはめようとすると無理が出るのですね。

**解説者**  
その通りです。  
Gutenberg 以降の WordPress が説明しにくいのは、新しい概念が増えたからというより、既存の分類に収まりきらない概念が中心に来たからでもあります。

## 従来テーマの知識だけでは足りない

**初学者**  
テーマを触ったことがある人なら、Gutenberg 以降も理解しやすいのではないですか。

**解説者**  
ある程度の土台にはなります。  
ただし、それだけでは足りません。  
従来テーマの知識は、テンプレート階層、`functions.php`、フック、CSS による表示調整といった理解を支えます。  
しかし Gutenberg 以降は、ブロック、Site Editor、テンプレートの編集可能性、`theme.json` の制約設計など、別の論点が前に出てきます。

**実装者**  
従来テーマの延長だけで考えると、「どのテンプレートファイルを置くか」「どのフックを使うか」に意識が寄りすぎます。  
でもブロックテーマ時代は、「編集者がどこまで構造を変えられるか」「どの選択肢を見せるか」「壊れにくい更新環境をどう作るか」も重要です。  
ここでは、テーマが設計する対象そのものが変わっています。

**初学者**  
だから、テーマ経験がある人でも、新しい WordPress を前にすると違和感が出るのですね。

**解説者**  
そうです。  
知識が無意味になるわけではありません。  
むしろ従来知識は重要です。  
ただ、それだけで全体を説明できるほど、現在の WordPress は単純ではなくなっています。

## 「プラグインが書ける」と「WordPress を理解している」は同じではない

**初学者**  
では、プラグインが書ける人ならどうでしょう。  
かなり深く理解していそうに思えます。

**解説者**  
それも部分的には正しいです。  
しかし、やはり同じではありません。  
プラグインが書けるということは、WordPress の拡張点や API に慣れていることを意味します。  
ですが、それは必ずしも WordPress 全体の構造理解と一致しません。

**実装者**  
極端に言えば、特定のフックや管理画面拡張には詳しくても、ブロックエディタの状態管理やテーマ側の制約設計には詳しくないことがあります。  
逆に、ブロック開発には慣れていても、WordPress が長年積み上げてきた PHP 側の互換性やテンプレート文化には疎いこともあります。  
つまり「書ける」は局所的でありうるのです。

**初学者**  
なるほど。  
技術的に何かを実装できることと、全体像を持っていることは別なのですね。

**解説者**  
その通りです。  
しかも WordPress は裾野が広く、案件の種類もさまざまです。  
そのため、実務経験があっても、見えている範囲が偏りやすい。  
Gutenberg 以降の説明しにくさは、この経験の偏りとも関係しています。

## なぜ求人票レベルの言葉では足りないのか

**初学者**  
実務でよく見るのは、「テーマが触れる」「プラグインが書ける」「WordPress のカスタマイズ経験あり」といった表現です。  
このあたりの言葉では、なぜ足りないのでしょうか。

**解説者**  
それらは作業経験のラベルであって、構造理解のラベルではないからです。  
もちろん採用や案件の現場では便利です。  
しかし、それだけでは「現在の WordPress をどういうシステムとして理解しているか」は分かりません。

**実装者**  
実際、「テーマが触れる」と言っても、クラシックテーマ中心なのか、ブロックテーマや `theme.json` まで見ているのかで意味が変わります。  
「プラグインが書ける」も、管理画面の追加なのか、カスタムブロックの設計まで含むのかで話が違います。  
求人票の言葉は実務上は必要ですが、説明の粒度としては粗いんです。

**初学者**  
だから、仕事では通じるのに、学習や説明の場では曖昧になるのですね。

**解説者**  
その通りです。  
この本が目指すのは、その曖昧さをほどくことです。  
作業ラベルではなく、構造ラベルで WordPress を説明できるようにする。  
それが、この章から始める理由でもあります。

## WordPress は「古い CMS」と言うだけでは片付かない

**初学者**  
ここまで聞くと、説明しにくさの背景はかなり分かってきました。  
でも世の中では、「WordPress は古いからややこしい」と言われがちですよね。  
それで片付けるのはだめなのでしょうか。

**解説者**  
だめというより、不十分です。  
たしかに WordPress には長い歴史があり、古い仕組みも抱えています。  
しかし説明しにくさの本質は、単なる古さではありません。  
古い構造を抱えながら、新しい編集モデルや制約設計を取り込んでいることにあります。

**実装者**  
もし本当に古いだけなら、単純に置き換えれば済む話です。  
でも現実には、WordPress は長期運用される HTML 資産、広いエコシステム、既存テーマ文化、編集者向けの UI、そして Gutenberg 以降のブロック構造を同時に抱えています。  
だからこそ、一言で整理しにくいんです。

**初学者**  
古いから分かりにくいのではなく、複数の時代の要請を同時に引き受けているから分かりにくいのですね。

**解説者**  
非常によいまとめです。  
WordPress は単純な意味で“古いもの”なのではありません。  
むしろ、古い最適化と新しい最適化を接続しながら進化しているシステムです。  
それゆえに説明には、比較と整理と責務分解が必要になります。

## この本は何を埋めるのか

**初学者**  
では、この本は具体的に何を埋めようとしているのでしょうか。

**解説者**  
断片知識と構造理解の間にあるギャップです。  
WordPress を少し触ったことがあり、PHP と JavaScript の基本も読める。  
しかし Gutenberg 以降を構造として説明しにくい。  
そうした読者に向けて、この本は WordPress を「説明できる地図」として再整理していきます。

**実装者**  
だから本書は、操作説明や API カタログの本ではありません。  
WordPress を、編集可能な HTML 資産を制約付きで安全に構築するシステムとして理解し直すための本です。  
その入口として、まず「なぜ説明しにくいのか」を言語化しておく必要があります。

**初学者**  
分かりました。  
最初に難しさの正体を押さえることで、後の章で出てくるブロック、テーマ、制約、比較の話がつながりやすくなるのですね。

**解説者**  
その通りです。  
説明しにくさは、学習者の能力不足ではありません。  
見取り図の更新が必要なのです。  
この章は、その更新の必要性を確認するための章です。

### 整理

- Gutenberg 以降の WordPress が説明しにくいのは、機能が増えたからではなく、設計の重心が変わったからである。  
- 「使える」と「説明できる」は異なり、断片的な実務経験だけでは構造理解に届かないことがある。  
- クラシック WordPress は、テンプレート階層・`functions.php`・フックを中心に、PHP 主体の CMS として説明しやすかった。  
- Gutenberg 以降は、ブロック、編集状態、保存構造、制約設計が前に出てきて、従来の説明枠だけでは足りなくなった。  
- PHP と JavaScript の両方を見ないと全体像をつかみにくく、これが説明難易度を上げている。  
- 「テーマが触れる」「プラグインが書ける」といった作業ラベルは便利だが、現在の WordPress の構造理解を表すには粗すぎる。  
- WordPress は単に古いのではなく、古い最適化と新しい最適化を同時に抱えているため、比較と責務分解が必要になる。  

### この章の核心

Gutenberg 以降の WordPress が説明しにくいのは、従来の WordPress の延長上にありながら、構造の主役が変わったからである。だから必要なのは断片知識の追加ではなく、WordPress 全体を捉える見取り図の更新である。

## まとめ

- WordPress を使えることと、構造として説明できることは同じではない。  
- クラシック WordPress の説明枠は強力だったが、Gutenberg 以降はそれだけでは足りない。  
- Gutenberg は単なる機能追加ではなく、ブロックと編集状態を中心に据える変化をもたらした。  
- 現在の WordPress を理解するには、PHP 側と JavaScript 側の両方を見る必要がある。  
- この本は、断片知識を構造的な地図へ組み替えることを目的とする。  

次章では、クラシック WordPress から Gutenberg へ何が変わったのかを、連続性と断絶の両方から整理します。  
テンプレート階層、`functions.php`、フックによる拡張という従来の理解が、どこまで引き継がれ、どこで足りなくなったのかを見ていきます。
