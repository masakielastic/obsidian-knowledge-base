## 第8章　RST\_STREAM と GOAWAY から終了処理を学ぶ

**初学者**「HTTP/2 では 1 本の接続の上に複数の通信が載る、という話はここまででだいぶ分かってきました。でも、止めるときの話になるとまた混乱します。ある通信だけを中断したい場合と、接続全体を終わらせたい場合で、何がどう違うのかがまだ曖昧です」

**解説者**「そこは HTTP/2 の重要な境界です。HTTP/1.1 的な感覚だと、通信を止めることと接続を終えることが比較的近く見えることがあります。しかし HTTP/2 では、1 本の接続の中に複数のストリームが同居しています。だから終了処理も階層的に考えなければなりません」

**実装者**「実務でも、その違いが分からないとつらいです。個別リクエストの失敗を接続障害のように扱ってしまったり、逆に接続終了の兆候を単なる個別失敗として見てしまったりします。retry や graceful shutdown の判断もぶれます」

**解説者**「この章では、RST\_STREAM と GOAWAY を手がかりに、HTTP/2 の終了処理を整理します。目的はフレームの細部を暗記することではありません。どの粒度の責務が終わるのかを見分けられるようになることです」

### HTTP/2 では「止まる」に複数の粒度がある

HTTP/2 の終了処理を考えるとき、最初に捨てたいのは、「止まる」という言葉を一種類で済ませる癖です。HTTP/2 では、あるストリームだけが終わることもあれば、接続全体が終わる方向へ入ることもあります。この区別は、本書全体で繰り返してきた Connection と Stream の境界そのものです。執筆ガイドラインでも、終了処理と失敗処理の区別、接続単位とストリーム単位の責務の区別は最重要品質基準の一部として挙げられています。

HTTP/2 の便利さは、1 本の接続を共有しながら複数の通信を並行して進められることでした。ですが、この便利さは終了処理を難しくします。なぜなら、「個別通信を止めたい」場面と「もうこの接続では新しい通信を受け付けたくない」場面が分かれるからです。

ここで大切なのは、終了処理を「例外が起きたから終わる」という話だけで見ないことです。HTTP/2 における終了処理は、**どの責務単位をどの範囲で閉じるか**という設計問題です。第5章でも、Error と shutdown は「どの粒度の責務が終わるのか」で切るべきだと整理しました。本章はその話を、HTTP/2 の具体的な終了手段に即して掘り下げる章です。

### RST\_STREAM は何を止めるのか

RST\_STREAM は、名前の通りストリーム単位の中断を表します。ここで重要なのは、「接続が壊れた」という意味ではなく、**この個別通信はここで打ち切る**という粒度だということです。

私は RST\_STREAM を、個別通信に対する明示的な打ち切りとして理解するのがよいと思います。あるストリームに関して、これ以上正常な進行を続けない。ヘッダーやデータの処理をこれ以上続けない。だが、接続全体はまだ使えるかもしれない。ほかのストリームは継続してよいかもしれない。ここがポイントです。

この見方を持つと、RST\_STREAM は「通信全体の失敗」ではなく、「共有接続の中の個別失敗」だと分かります。HTTP/2 では複数の通信が一本の接続を共有している以上、個別通信だけを止める手段が必要になります。RST\_STREAM はそのためのものです。

実務上も、この粒度差は大きいです。ある RPC 呼び出しが失敗した、あるファイル転送だけを中断したい、あるレスポンスはこれ以上読まなくてよい。こうした場面で、接続全体を壊してしまうのは過剰です。RST\_STREAM の存在は、HTTP/2 が共有接続を前提にしていることの裏返しだと言えます。アウトラインでも、第8章は「ストリーム単位の失敗」と「接続単位の終了」を対比させる章として設計されています。

### GOAWAY は何を止めるのか

一方、GOAWAY は接続単位の終了方向を表します。ここでも誤解したくないのは、GOAWAY が単に「今すぐ全部切る」という話ではないことです。むしろ、**この接続はもう終わる方向に入る**ことを伝えるための手段として理解したほうが本質に近いです。

HTTP/2 では、接続全体を終了させたい場面でも、すでに走っているストリームが存在することがあります。そのため、接続終了は「個別通信を全部一斉に破壊する」だけでは済みません。どこまで新規を受け付けるか、どこまで既存通信を流し切るか、といった方針が必要です。

この意味で GOAWAY は、接続全体に対する shutdown の意思表示です。新しいストリームを作らせない、あるいはこれ以降の処理方針を相手に伝える。その結果として、接続は終了へ向かいます。

ここが RST\_STREAM との決定的な違いです。RST\_STREAM は個別通信を止める。GOAWAY は共有基盤そのものの寿命を閉じる方向へ進める。どちらも「止める」ですが、止めている対象が違います。

この違いが見えると、Session と Stream の責務差もより明確になります。第3章では Session が接続全体の責務、Stream が個別通信の責務を表すと整理しました。GOAWAY は Session 側の問題であり、RST\_STREAM は Stream 側の問題です。本章は、その区別を終了処理から逆向きに確認しているとも言えます。

### 個別中断と接続終了を分ける必要はなぜあるのか

ここで一度、「なぜわざわざ二つの粒度を分けるのか」を正面から整理しておきます。

理由は単純で、HTTP/2 の価値が共有接続と多重化にあるからです。もし個別通信の失敗ごとに接続全体を壊していたら、1 本の接続に複数の通信を載せる利点が大きく損なわれます。逆に、接続を終わらせたいのに個別通信単位の停止しか表現できなければ、共有基盤としての寿命管理ができません。

つまり、HTTP/2 の設計では「複数の通信を一緒に走らせる」ために、開始だけでなく終了も階層化する必要があります。これは flow control が接続単位とストリーム単位の両方を持っていたのと同じ構造です。多重化された共有接続では、制御面も終了面も階層的になります。

私はここで、HTTP/2 の終了処理を「stop に複数の粒度がある世界」だと覚えるのがよいと思います。個別通信だけを止める stop と、共有基盤ごと閉じる stop は違う。その違いを明示的に扱えること自体が、HTTP/2 の設計の一部なのです。

### graceful shutdown とは何か

GOAWAY の話をすると、graceful shutdown という言葉が出てきます。これも中級では重要な概念です。ただし、ここでも詳細な実装手順を暗記することより、「何を目指しているか」を押さえるほうが大事です。

graceful shutdown とは、乱暴に全部を切るのではなく、**接続を終える方向へ進みつつ、可能な範囲で既存の通信を整然と収束させる**考え方です。HTTP/2 では 1 本の接続に複数のストリームが存在し得るため、接続終了にも段階が必要になります。

このとき重要なのは、新しいストリームをもう増やさないことと、すでに進行中のものをどう扱うかを分けて考えることです。GOAWAY は、その分離を支える接続単位の通知として読むと分かりやすいです。

graceful shutdown を理解すると、HTTP/2 の接続は単なるパイプではなく、寿命管理を持った共有基盤だという感覚がよりはっきりします。開始と同じくらい、終わり方にも設計が必要なのです。これはアウトラインでも、第8章の主要トピックに graceful shutdown が含まれている理由です。

### error と shutdown は同じではない

ここで、あえて一段整理しておきます。error と shutdown は近いですが、同じではありません。執筆ガイドラインでも、「終了処理 / 失敗処理」の区別は明確であるべきだとされています。

error は、何かが期待通りに進まなかったことを表します。個別ストリームの失敗であることもあれば、接続全体の異常であることもあります。

shutdown は、今後の受け付けや進行をどのように終わらせるかという方針に近い概念です。必ずしも壊れたから起きるとは限りません。運用上の都合、再起動、負荷調整、役割切り替えなどでも shutdown は起こり得ます。

この違いが見えないと、終了処理の設計が乱れます。たとえば GOAWAY を「エラーが起きたから全部失敗」とだけ見ると、graceful shutdown の意味が消えます。逆に RST\_STREAM を軽い shutdown のように見てしまうと、個別通信の失敗通知と接続寿命の管理が混ざります。

私はここで、error は状態の悪化や異常、shutdown は寿命管理の意思決定、と分けて見るのがよいと思います。もちろん現実には両者が絡み合うことはありますが、設計上は分けておいたほうが安定します。

### retry や recovery にどう影響するのか

HTTP/2 の終了処理で実務上重要なのは、retry や recovery の判断に直結することです。アウトラインでも、第8章の主要トピックに retry や recovery の考え方が含まれています。

あるストリームが RST\_STREAM で打ち切られたとき、それは接続全体の破綻を意味しないかもしれません。すると、「その個別通信をどう扱うか」という判断が中心になります。再試行するか、上位へ失敗を返すか、そのストリームだけの問題として閉じるか。焦点は個別通信にあります。

一方、GOAWAY が出てきたときは、共有接続の寿命が終わる方向に入っています。すると個別通信だけでなく、「今後どの接続で続けるか」「新規要求をどう扱うか」「どのタイミングで再接続するか」といった、接続全体の方針が問題になります。

つまり retry や recovery の単位も、RST\_STREAM と GOAWAY で変わります。これは単なる技術的な違いではなく、責務の単位の違いです。個別通信の再処理なのか、接続基盤の切り替えなのか。この差が分かると、HTTP/2 の障害対応や運用設計もかなり見えやすくなります。

### API 設計ではどう見えるのか

本書は API 設計との接続を重視してきました。本章でもそこを確認しておきます。RST\_STREAM と GOAWAY の違いは、ライブラリの API ではたいてい「個別ストリームの終了通知」と「セッション全体の終了・shutdown 通知」の差として見えます。

第2章では、ライブラリは複雑さを消すのではなく API の形へ再配置すると整理しました。本章の話も同じです。利用者が生のフレーム名を毎回意識しなくても、API にはたいてい「この通信が終わった」「このセッションはもう新規を受け付けない」といった粒度差が現れます。

もし API 上でこの差が曖昧なら、利用者は個別失敗と接続終了を混同しやすくなります。だから終了処理の API は、単なるイベント一覧ではなく、どの責務単位の寿命が変わったのかを読み取れる形であるべきです。これは第3章の Session / Stream / Event の話や、第5章の Error / shutdown 責務の分け方ともつながっています。

### HTTP/2 の終了処理から見えてくるもの

ここまでの話をまとめると、HTTP/2 の終了処理は「終わり方の設計」です。

個別通信だけを止める仕組みが必要なのは、接続共有を前提としているからです。  
接続全体を終える仕組みが別に必要なのは、共有基盤の寿命を管理しなければならないからです。  
graceful shutdown が重要なのは、複数の通信が走る接続を乱暴に切るわけにはいかないからです。

つまり、RST\_STREAM と GOAWAY の違いは、単なるフレームの違いではありません。HTTP/2 がどの粒度で責務を分け、どの粒度で寿命を管理しているかを示す、とても教育的な違いです。

私はこの章を通して、読者に「終了処理を見ると責務境界がよく見える」という感覚を持ってもらいたいと思っています。始め方より終わり方のほうが、境界が露わになることがあるからです。

### 整理ブロック：RST\_STREAM と GOAWAY の違い

| 観点 | RST\_STREAM | GOAWAY |
| --- | --- | --- |
| 対象 | 個別ストリーム | 接続全体 |
| 主な意味 | この通信を打ち切る | この接続は終了方向へ入る |
| 影響範囲 | 主にそのストリーム | 複数ストリームと新規受付方針 |
| 責務の帰属 | Stream 側 | Connection / Session 側 |

### 整理ブロック：終了処理をどう読むか

| 見方 | 内容 |
| --- | --- |
| 個別中断 | 共有接続は残しつつ、特定通信だけを止める |
| 接続終了 | 共有基盤の寿命を閉じる方向へ進める |
| graceful shutdown | 新規と既存を分けて、整然と収束させる |
| error と shutdown の違い | 異常の通知と寿命管理の方針は分けて考える |

### 整理ブロック：retry / recovery への影響

| 状況 | 中心になる判断 |
| --- | --- |
| 個別ストリームの中断 | その通信だけをどう扱うか |
| 接続全体の終了方向 | 新規要求、再接続、既存通信の収束をどう扱うか |

### この章で残すべきメンタルモデル

HTTP/2 では、「止める」に複数の粒度があります。  
RST\_STREAM は個別通信を止めます。  
GOAWAY は接続全体を終える方向を示します。

この違いは、単なるフレームの違いではありません。  
HTTP/2 が、共有基盤としての接続と、個別通信としてのストリームを分けて管理していることの表れです。

だから終了処理を理解するとは、  
どの責務単位の寿命が変わったのかを見分けられるようになることです。

## まとめ

- HTTP/2 では終了処理も階層的であり、個別中断と接続終了を分けて考える必要がある
    
- RST\_STREAM は個別ストリームの打ち切りを表し、GOAWAY は接続全体の終了方向を表す
    
- graceful shutdown は、新規受付と既存通信の収束を分けて考える接続単位の設計である
    
- error と shutdown は近いが同じではなく、異常通知と寿命管理は分けて考えたほうがよい
    
- retry や recovery の判断は、個別通信の問題か接続全体の問題かで大きく変わる
    

次章では、ここまで整理してきた接続、ストリーム、イベント、終了処理の分離をさらに一段進めて、Sans-IO という考え方への入口を扱います。そこで、I/O とプロトコル状態を分けたくなる理由が見えてきます。
