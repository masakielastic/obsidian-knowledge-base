# HTTP/2 中級書 概要

## この本の位置づけ

この本は、HTTP/2 入門書の次に読むことを想定した中級者向けの書籍である。  
入門書では、HTTP/2 を主に「観察」と「基本構造」の観点から扱った。具体的には、Connection / Stream / Frame の区別、主要フレームの役割、curl や nghttpd のログの大まかな読み方、典型的な通信パターンの理解を目標にした。

本書では、その次の段階として、HTTP/2 を **ライブラリ利用・非同期処理・クラス責務・レイヤー分割** の観点から捉え直す。目的は、HTTP/2 を単に「知っている」状態から、**ライブラリや API の設計を読み、実務上の責務分担を考えられる状態**へ進むことである。

本書は上級書ではない。そのため、RFC 全体の精読、HPACK の符号化詳細、フロー制御アルゴリズムの実装、フレームのビット単位解析、生データを前提とした詳細実装などは中心テーマにしない。そうした内容は、次の上級書で扱う。

---

## この本の目的

本書の目的は次の通りである。

- HTTP/2 ライブラリが何を隠し、何を利用者に露出しているかを理解する
- Connection / Stream / Frame の区別を、実装責務や API 設計と結びつけて考える
- HTTP/2 と非同期処理の相性を、速度論ではなく状態管理の観点から説明できるようにする
- クラス責務やレイヤー分割を通じて、HTTP/2 を「使う側」「設計する側」の両方から見られるようにする
- gRPC / Connect / HTTP/3 への橋渡しとして、HTTP/2 のどの抽象を再利用できるかを整理する

---

## 想定読者

本書が想定している読者は、次のような人である。

- HTTP/2 入門レベルの知識はあり、その先へ進みたい人
- ライブラリやフレームワーク越しに HTTP/2 を使っているが、内部構造がまだ曖昧な人
- Node.js などの HTTP/2 API を見て、なぜその形になっているのかを理解したい人
- 非同期処理やイベント駆動と HTTP/2 の関係を整理したい人
- 将来的に RFC 読解や Sans-IO、HTTP/3 に進みたい人

一方で、本書は次のような読者には少し物足りない可能性がある。

- いきなり RFC の詳細や HPACK の実装を読みたい人
- フレーム列をバイト単位で扱うことを今すぐ目標にしている人
- HTTP/2 実装そのものを書き始めたい人

その場合は、本書を飛ばして上級書の一部を参照したほうが早い場面もある。

## 本書で扱うこと

本書では、主に次の内容を扱う。

### 1. 入門書の次に見るべき景色
入門書で得た知識を整理しなおし、中級書では何を見るべきかを明確にする。ここでは「観察から設計へ進む」という本書の軸を共有する。

### 2. HTTP/2 ライブラリの抽象化
ライブラリは複雑さを消すものではなく、別の形にまとめ直すものである。HTTP/2 のライブラリがどの複雑さを隠し、どの責務を API として外に出しているかを整理する。

### 3. Node.js の http2 モジュールを題材にした API 読解
Node.js の API を具体例として、Session / Stream / Event などの責務分担を考える。これは Node.js の使い方を学ぶ章ではなく、API から HTTP/2 の構造を読む章である。

### 4. 非同期処理と HTTP/2
HTTP/2 を request/response の延長としてだけではなく、複数ストリームの並行進行と状態変化を扱うプロトコルとして捉える。イベント駆動や非同期処理がなぜ自然に見えてくるのかを整理する。

### 5. クラス責務
Connection 管理、Stream 管理、Frame のパースとシリアライズ、エラー処理、アプリとの境界など、どこで責務を切ると見通しがよくなるかを検討する。

### 6. レイヤー分割
低レイヤー / 中レイヤー / 高レイヤーの分業という観点から、HTTP/2 ライブラリ、ミドル層、gRPC / Connect、アプリケーションの関係を整理する。

### 7. フロー制御を実務的に理解する
入門書では概要だけにとどめた flow control を、「なぜ利用者も無関係ではいられないのか」という観点から整理する。

### 8. エラー処理と終了処理
RST_STREAM と GOAWAY の違いを軸に、接続全体と個別ストリームの失敗や終了をどう考えるべきかを整理する。

### 9. Sans-IO への入口
上級書への橋として、なぜ I/O と状態遷移を分けた設計が欲しくなるのかを説明する。

### 10. HTTP/3 への橋
HTTP/2 で身につけた抽象が HTTP/3 にどうつながるのか、逆に transport の違いによって何が変わるのかを概観する。

### 11. 設計レビューによる総復習
架空の HTTP/2 ライブラリ設計レビュー会を題材にして、本書全体の論点を横断的に復習する。Connection / Stream の境界、Frame の parse と dispatch、Event 設計、非同期処理、flow control、RST_STREAM と GOAWAY、Sans-IO、application boundary といった論点を、実際の設計判断の言葉で言い直す章である。

- HPACK の符号化詳細

- 実装アルゴリズムの詳細
- 各言語ランタイム比較の深掘り
- 生のバイト列を扱う高度な解析
- HTTP/2 実装のフルチュートリアル

これらは重要だが、本書の主題は「実装設計の見取り図を得ること」であって、「仕様の隅々を網羅すること」ではない。

ただし、いきなり専門家向けの本文だけに入ると、読者がその章の目的や論点をつかみにくい。そこで各章の冒頭には、短い **導入対話** を置く。

- その章で起きやすい誤解を先に表面化する
- その章の問いを明確にする

つまり、本書では

- **導入対話** = メンタルモデルの準備
- **一人称本文** = 知識の圧縮と整理
- **章末整理** = 要点と次章への橋渡し

という三層構成を採用する。

## この本で育てたいメンタルモデル

本書で読者に残したいのは、細かな知識の断片よりも、次のようなメンタルモデルである。

### 1. HTTP/2 は「複数の並行する状態」を扱うプロトコルである
HTTP/2 は request/response を1回ずつ順番に処理する世界よりも、複数のストリームと状態変化を並行して扱う世界のほうが自然に見える。

### 2. ライブラリは複雑さを消すのではなく再配置する
複雑さは消えない。Session, Stream, Event, Backpressure などの形で再配置される。API はその痕跡を残す。

### 3. プロトコルの区別は、そのまま責務設計の手がかりになる
Connection と Stream の違い、接続単位と個別ストリーム単位の違いは、クラス責務やエラー処理の設計に深く関わる。

### 4. 低・中・高レイヤーの分業が重要である
HTTP/2 は単独の知識ではなく、プロトコル処理・中間抽象・アプリケーション利用の分業の上に成り立つ。

### 5. HTTP/2 は HTTP/3 への土台でもある
HTTP/3 は transport が変わっても、多くの抽象を HTTP/2 から引き継いでいる。HTTP/2 を整理することは HTTP/3 への準備にもなる。
