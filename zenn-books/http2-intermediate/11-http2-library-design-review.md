## 第11章　HTTP/2 ライブラリ設計レビュー会

**初学者**「この本をここまで読んで、Connection と Stream を分けることや、非同期処理が自然に見える理由は分かってきました。でも、いざ自分で設計を考えると、全部がつながって見えてしまいます。Frame の処理、イベント、flow control、終了処理、アプリとの境界。結局、どこで何を切ればいいのか、自信が持てません」

**解説者**「それは自然です。HTTP/2 は、概念を知っただけでは設計できません。逆に、設計の場に持ち込んでみると、どの概念が本当に重要だったかが見えてきます」

**実装者**「実務でもそうです。『HTTP/2 を知っています』と『HTTP/2 ライブラリの責務分割を説明できます』のあいだには、かなり大きな差があります。API の形、ログの見え方、終了処理、backpressure の扱いまで考え始めると、知識が判断に変わる必要があります」

**解説者**「では今日は、架空の HTTP/2 ライブラリをレビューする形で、この本全体を復習しましょう。目的は正解のクラス図を一つ作ることではありません。どの境界を守るべきか、どこで複雑さを隠し、どこで露出させるべきかを、設計レビューの言葉で言い直すことです」

### 最初の設計案は、なぜ危ういのか

**初学者**「では、まず単純な案を出します。`Connection` クラスを一つ置いて、その中でフレームの parse、ストリーム管理、送受信、イベント通知、エラー処理を全部まとめて扱います。最初の実装なら、これでよいのではないでしょうか」

**実装者**「最初にそう書きたくなる気持ちはよく分かります。試作品なら、実際そういう形から始まることもあります」

**解説者**「ただ、その設計案は、この本で分けてきた責務を再び一箇所へ押し戻してしまいます。接続全体の責務と個別通信の責務が混ざりますし、低レイヤーの bytes 処理と中レイヤーの意味処理と高レイヤー向け通知も混ざります」

**初学者**「つまり、最初から分けて考えないと、あとでほどけなくなるんですね」

**解説者**「そうです。HTTP/2 ライブラリ設計で最初に問うべきなのは、『何をまとめると楽か』ではなく、『どの責務は混ぜてはいけないか』です。ここを誤ると、あとで flow control、shutdown、retry、observability のすべてが曖昧になります」

私はこの章で、設計レビューをひとつの復習方法として使いたいと思います。  
なぜなら、レビューでは知識の有無よりも、境界を説明できるかどうかが問われるからです。  
本書の中級編としての価値も、まさにそこにありました。HTTP/2 を「便利なライブラリの裏側」ではなく、「抽象化・責務分割・状態管理の教材」として読むことです。

### Connection と Stream を分けるのは、クラスの趣味ではない

**解説者**「最初に守るべき境界は、やはり Connection と Stream の区別です。これは本書全体の中心線でした」

**初学者**「でも、Connection の中に Stream の一覧を持つのは自然ですよね」

**解説者**「それは自然です。ただし、保持することと責務を一体化することは違います。Connection の中に Stream の参照があるのはよくても、Connection が各ストリームの進行責務まで抱え込むべきではありません」

**実装者**「あるストリームだけが RST\_STREAM で中断されたとき、必要なのは接続全体を壊すことではなく、その個別通信の状態遷移を処理することです。逆に GOAWAY を受けたときは、接続全体の寿命と、新規ストリームをどう扱うかが問題になります。この二つを同じ粒度で扱うと、終了処理の意味が壊れます」

ここで思い出したいのは、HTTP/2 が 1 本の接続に複数の通信を載せるプロトコルだという点です。  
だからこそ、共有基盤の責務と個別通信の責務を分ける必要があります。  
Session / Stream の API 分割も、クラス責務の議論も、終了処理の階層性も、すべてこの差を守るためにありました。

設計レビューで最初に見るべきなのは、この境界が構造として守られているかどうかです。

### Frame の parse は、意味処理と混ぜない

**初学者**「では、フレームの parse は Connection に持たせてもよいでしょうか。接続からバイト列を受け取るのだから、自然に見えます」

**解説者**「それもよく出る案です。ただ、parse はなるべく意味処理から切り離したほうがよいです」

**実装者**「理由は単純です。parse の責務は『どう読めるか』であって、『どう意味づけるか』ではないからです。長さ、種別、フラグ、ストリーム ID を読むところまでは、比較的低レイヤーに留められます」

**解説者**「その先で『これは接続単位の処理か』『どのストリームへ配送すべきか』『END\_STREAM が立っているのでどの状態へ進むか』を考えるのは、もう中レイヤーの責務です」

この分離は、第5章で見た「Frame は責務の中心ではなく、責務を受け渡す表現単位に近い」という見方とつながっています。  
フレームの bytes 処理と、接続やストリームの意味処理を分けると、仕様としての正しさと、ライブラリとしての抽象化を別々に扱いやすくなります。

### dispatch は、見えにくいが中核である

**初学者**「parse の次は、フレームをどこへ渡すか決める必要がありますよね。これは Connection のメソッドで十分ではないでしょうか」

**実装者**「十分なこともあります。ただ、その責務を見失うと Connection が巨大化しやすいです」

**解説者**「dispatch は、低レイヤーの bytes 処理でもなければ、アプリケーションロジックでもありません。どのフレームが接続単位の処理へ行き、どのフレームが特定ストリームへ配送されるかを決める中層です」

**初学者**「地味ですが、たしかにそこが多重化の中心ですね」

**解説者**「その通りです。HTTP/2 では多重化が本質なので、dispatch の責務は意外に重いのです」

私はこの責務を特に重要だと思っています。  
なぜなら、HTTP/2 の難しさはフレームそのものではなく、複数の責務単位へ正しく意味を配ることにあるからです。  
dispatch を意識できると、「parse の話」と「状態遷移の話」と「利用者への通知」がどこでつながるのかが見えてきます。

### Event は「何を見せるか」の設計そのものである

**初学者**「Node.js の `http2` のようにイベントが多い API を見ると、何をどこまで露出すべきか迷います」

**解説者**「その迷いは、ライブラリ設計の中心です。イベント設計は、複雑さの再配置の仕方そのものだからです」

**実装者**「露出しすぎると利用者が低レイヤーの事情を背負います。隠しすぎると、flow control や shutdown のような、本当に必要な差分まで見えなくなります」

**解説者**「基準は、『利用者の判断に影響する状態変化かどうか』です。個別ストリームの開始、ヘッダー受信、データ受信、終了、中断。接続全体の shutdown 開始や新規受付停止。これらは意味があります。一方、内部的な window 計算や parse の途中経過をそのままイベントにする必要はありません」

第3章で見た通り、Event は単なる通知機構ではありません。  
どの状態変化を利用者へ見せ、どれを内部責務に留めるかという抽象化の選択です。  
設計レビューでは、イベント一覧を見るだけで、そのライブラリが何を利用者責務として残しているかがかなり読めます。

### 非同期処理は流行ではなく、並行状態の表現である

**初学者**「イベントの話になると、結局 async や event loop に寄る設計になる気がします。これは言語文化の問題でしょうか」

**解説者**「本書ではそう整理しませんでした。HTTP/2 と非同期処理の相性は、流行ではなく、並行する複数の状態を扱う構造から来ています」

**実装者**「あるストリームではヘッダーを受け取り、別のストリームではデータ送信が止まり、接続全体では GOAWAY を受けるかもしれない。この世界は、一回の戻り値で終わる直線的なモデルだけでは表しにくいです」

つまりレビューで問うべきなのは、「この API は async か」ではありません。  
「この API は並行する状態変化を、どの責務単位で利用者に扱わせるのか」です。  
callback でも Promise でも stream API でもかまいません。  
重要なのは、状態管理の構造に忠実であることです。

### flow control は接続かストリームかの二択ではない

**初学者**「flow control は特に設計が難しいです。接続単位とストリーム単位の両方があるので、誰の責務なのか決めづらいです」

**解説者**「その感覚は正しいです。flow control は、この本でもっとも混ざりやすい責務のひとつでした」

**実装者**「しかも利用者には backpressure や詰まりとして見えるので、完全に内部へ押し込めるわけでもありません」

**解説者**「設計の原則としては、接続単位の window 管理とストリーム単位の window 管理を分けるべきです。ただし、いまこのストリームでどれだけ送れるかという判断は、両者の調停が必要です。だから flow control は、接続かストリームかの二択ではなく、中層の coordination が必要な責務です」

この点は、第5章のクラス責務と第7章の「実務で困る制御面」という見方が交差する場所です。  
flow control は内部アルゴリズムであると同時に、利用者には送信待ちや詰まりとして現れるため、設計レビューでも見逃せません。  
レビューで「flow control は Connection が持ちます」と言われたらストリーム側の事情を、  
「各 Stream が自分で判断します」と言われたら共有接続側の調停を、  
それぞれ問い返す必要があります。

### RST\_STREAM と GOAWAY を同じ「失敗」にしてはいけない

**初学者**「終了処理については、利用者向けにはエラーイベントひとつにまとめてしまうのはだめでしょうか」

**解説者**「かなり危うい設計です」

**実装者**「RST\_STREAM と GOAWAY は、どちらも『止まる』ように見えて、責務の粒度が違います。前者は個別ストリームの中断で、後者は接続全体の shutdown 方向です」

**解説者**「もし API 上でその差を潰してしまうと、利用者は retry や recovery の判断を誤ります。個別通信だけをやり直せばよいのか、接続全体を張り直すべきなのかが分からなくなるからです」

第8章で見たように、HTTP/2 の終了処理は階層的です。  
したがって設計レビューでは、終了系の API が「どの責務単位の寿命が変わったのか」を読める形になっているかを確認すべきです。  
これは単なるイベント名の問題ではなく、HTTP/2 が接続共有を前提としていることの表れです。

### Sans-IO を意識するなら、I/O とプロトコル核を分ける

**初学者**「ここまで来ると、I/O と状態遷移を分けたくなってきます。Sans-IO を意識するなら、どこから切り出せばよいのでしょうか」

**解説者**「第9章で整理した通り、Sans-IO は大きな枠組みを最初から導入することではありません。I/O とプロトコル状態を同じ場所に閉じ込めない見方です」

**実装者**「最低限でも、  
bytes の読み書き、  
frame の parse / serialize、  
入力を受けて接続・ストリーム状態を更新し、イベントや送信候補を出すプロトコル核、  
この三つは分けて考えたほうがよいです」

**解説者**「特に最後の部分が重要です。HTTP/2 の本質は、ソケットから読んだという事実そのものではなく、その結果として何の状態が変わり、どんなイベントを返し、何を送るべきかにあります」

Sans-IO の完成形をこの本で作る必要はありません。  
ただ、レビューの場で「このライブラリは Sans-IO か」と問うより、  
「I/O と状態遷移はどこで分かれていますか」と問えるようになることのほうが重要です。  
それができれば、HTTP/2 を読み書き手続きではなく、状態遷移と責務境界の問題として見られるようになります。

### application boundary は、見せすぎず隠しすぎず切る

**初学者**「最後に、アプリケーション境界はどこで切ればよいでしょうか。HTTP/2 の事情はできるだけ隠すべきですか」

**解説者**「そこは隠しすぎも見せすぎも問題です」

**実装者**「アプリケーションはフレーム単位の事情まで知る必要はありません。ですが、通信の開始、進行、終了、中断、接続 shutdown、送信待ちの存在など、自分の判断に影響する情報は知る必要があります」

**解説者**「第6章で見た低・中・高レイヤーの整理を思い出してください。低レイヤーは正しさを担い、中レイヤーは利用可能な抽象を作り、高レイヤーは業務上の意味を扱います。レビューの観点では、『この API は高レイヤーに必要以上の HTTP/2 を背負わせていないか』『逆に必要な差分まで消していないか』を見るべきです」

HTTP/2 の事情をそのままアプリへ露出すると、上位がプロトコル実装を再開し始めます。  
一方で、shutdown や backpressure や終了処理の粒度差を完全に隠すと、高レイヤーの判断材料がなくなります。  
このバランスを取ることが、中レイヤー設計の価値です。

### よい設計レビューとは、何を確認する場か

**初学者**「ここまで聞いてくると、HTTP/2 ライブラリの設計レビューで本当に見るべき点が、かなり整理できてきました」

**実装者**「私なら、まず四つを見ます。  
第一に、Connection と Stream の責務が混ざっていないか。  
第二に、parse、dispatch、状態遷移、イベント生成が同一塊になっていないか。  
第三に、flow control と shutdown が粒度差を保って表現されているか。  
第四に、アプリ境界で必要な差分を消していないか、です」

**解説者**「私はそこにもう一つ加えます。  
そのライブラリは、何を隠し、何を利用者に残すのかを自分で説明できるか。  
これです」

**初学者**「それは第2章からずっと続いてきた問いですね」

**解説者**「はい。この本を通して学んできたことを一文で言えば、HTTP/2 ライブラリを『便利な箱』としてではなく、『複雑さの再配置装置』として読めるようになることでした。レビュー会は、その見方を実戦の言葉へ変える場です」

この章の目的も、まさにそこにあります。  
HTTP/2 の断片知識を並べ直すことではなく、  
それらを設計判断の言葉に変換することです。  
それができれば、本書全体の復習としては十分です。

### 整理ブロック：レビュー会で最初に確認すべきこと

| 観点 | 確認する問い |
| --- | --- |
| Connection / Stream の分離 | 共有基盤の責務と個別通信の責務が混ざっていないか |
| parse と意味処理の分離 | bytes 処理と状態遷移が一体化していないか |
| dispatch の存在 | 接続単位とストリーム単位への配送判断が曖昧になっていないか |
| Event 設計 | 利用者の判断に必要な状態変化が適切な粒度で露出しているか |
| flow control | 接続単位・ストリーム単位・調停責務が意識されているか |
| shutdown | RST\_STREAM と GOAWAY の粒度差が API 上でも保たれているか |
| application boundary | HTTP/2 の事情を見せすぎず、隠しすぎてもいないか |

### 整理ブロック：この章で回収している本書全体の論点

| 本書の論点 | この章での確認の仕方 |
| --- | --- |
| ライブラリは複雑さを再配置する | 何を隠し、何を見せるかをレビュー観点にする |
| Session / Stream / Event | 責務の粒度として API を読み直す |
| 非同期処理と状態管理 | 並行する状態変化をどう表現するかとして見る |
| クラス責務の分割 | parse、dispatch、state、event の境界で確認する |
| flow control | backpressure と調停責務として確認する |
| RST\_STREAM / GOAWAY | 終了処理の階層性として確認する |
| Sans-IO | I/O とプロトコル核の境界として確認する |

### この章で残すべきメンタルモデル

HTTP/2 ライブラリの設計レビューとは、  
クラス図の好みを論じることではありません。

何を隠し、何を露出するのか。  
どの責務を接続単位で持ち、どの責務をストリーム単位で持つのか。  
状態遷移、flow control、終了処理、イベント生成を、どの粒度で利用者に見せるのか。  
その判断を確認する場です。

この本全体で学んだのは、HTTP/2 の断片知識ではなく、  
そうした判断を支える地図でした。

## まとめ

- HTTP/2 ライブラリ設計で最優先の境界は Connection と Stream の分離である
    
- Frame の parse と意味処理、dispatch、イベント生成は分けて考えたほうが安定する
    
- 非同期処理は流行ではなく、並行する状態変化を表現する構造上の要請である
    
- flow control と shutdown は、接続単位とストリーム単位の粒度差を保ったまま設計すべきである
