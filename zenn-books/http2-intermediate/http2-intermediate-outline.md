# 第2章 HTTP/2 ライブラリは何を隠し、何を見せるのか

## 章の役割
ライブラリを「便利な箱」ではなく、責務の再配置装置として理解する。  
HTTP/2 を直接扱わずとも、ライブラリ越しにどんな抽象を見ているのかを言語化する章。

## この章で答えたい問い
- HTTP/2 ライブラリを使えば HTTP/2 を理解しなくてよいのか
- どんな複雑さがライブラリ内部に押し込まれているのか
- 利用者の側にどんな責務が残るのか

## 主要トピック
- 抽象化と責務移動
- 高水準 API と低水準 API
- セッション・ストリーム・イベント・バックプレッシャー
- ライブラリが隠すものと露出するもの
- 「便利さ」の中身

## 残したいメンタルモデル
- ライブラリは複雑さを消さない
- 複雑さは API の形として残る
- API を読むことは責務分割を読むことでもある

## 深入りしないこと
- 特定ライブラリの細かな API リファレンス
- 各言語ライブラリの大規模比較

## 次章への橋
次章では具体例として Node.js の http2 モジュールを見ながら、セッション・ストリーム・イベントの責務を読む。

# 第4章 非同期処理と HTTP/2 はなぜ相性がよいのか

## 章の役割
HTTP/2 を request/response の順番処理だけで考えるのがなぜ苦しいかを説明し、非同期処理やイベント駆動との相性を整理する。

## この章で答えたい問い
- なぜ HTTP/2 は同期的な発想だけでは理解しにくいのか
- 非同期処理が自然に見えるのはなぜか
- 速いから非同期なのか、それとも状態管理の都合なのか

## 主要トピック
- 複数ストリームの並行進行
- 状態変化とイベント
- request/response 脳と stream/event 脳
- イベントループとの親和性
- 言語ランタイムごとの一般的な事情を軽く整理

## 残したいメンタルモデル
- 非同期は単なる高速化テクニックではない
- HTTP/2 は「並行する状態」を扱う都合上、非同期やイベント駆動が自然に見える

## 深入りしないこと
- 各言語の async 機構の詳細比較
- event loop 実装論

## 次章への橋
次章では、こうした状態管理をクラス責務としてどう分割するかを整理する。

---

# 第5章 クラス責務をどう切るか

## 章の役割
HTTP/2 を題材に、クラス責務やオブジェクト境界を整理する。  
中級書の中心章の一つ。

## この章で答えたい問い
- Connection と Stream の責務はどこで分けるべきか
- Frame の parse / serialize はどこに置くべきか
- エラー処理や flow control はどこが持つべきか
- アプリとの境界はどう切るべきか

## 主要トピック
- Connection 管理責務
- Stream 管理責務
- Frame 処理責務
- Error / shutdown 責務
- Flow control 責務
- Logging / observability の責務
- Application boundary

## 残したいメンタルモデル
- プロトコル概念とクラス責務は一対一ではない
- ただし、概念区分は責務設計の重要な手がかりになる
- 接続単位とストリーム単位の境界は最重要

## 深入りしないこと
- 特定言語の OOP スタイル論争
- 完全な実装パターンの押しつけ

## 次章への橋

# 第6章 低・中・高レイヤーで HTTP/2 を捉え直す

## 章の役割
HTTP/2 を単独の知識ではなく、複数レイヤーにまたがる分業の問題として捉える。  
ライブラリ・ミドル層・アプリの役割を整理する章。

## この章で答えたい問い
- 低レイヤーは誰が担当すべきか
- 中レイヤーとは何をする層か
- gRPC / Connect / アプリはどの層に位置づくのか
- curl のようなツールはどの位置にあるのか

## 主要トピック
- 低レイヤー: frame / state / HPACK / parse
- 中レイヤー: stream 管理 / event / backpressure / abstraction
- 高レイヤー: RPC / framework / app
- 共通基盤化とコミュニティ分業
- 学習コストの分担

## 残したいメンタルモデル
- HTTP/2 は公共基盤的な技術である
- 何を共通化し、何を各フレームワークや言語に任せるかが大きな設計課題になる

## 深入りしないこと
- コミュニティ政策論の詳細
- 各ライブラリの勢力図

## 次章への橋
こうした分業の中でも利用者が無関係ではいられない話題として、次章で flow control を扱う。

# 第7章 フロー制御を「実務で困る仕組み」として理解する

## 章の役割
入門書では軽く触れた flow control を、中級書では「ライブラリ利用者にも関係する現実的な制御面」として整理する。

## この章で答えたい問い
- flow control はなぜ必要か
- なぜ詳細実装を知らなくても、無視はできないのか
- どういうときに「詰まり」として見えるのか

## 主要トピック
- 接続単位とストリーム単位の制御
- WINDOW_UPDATE の役割
- backpressure の感覚
- アプリやミドル層に見える症状
- 利用者が意識すべき最小限の理解

## 残したいメンタルモデル
- flow control は単なる細かな仕様ではない
- データの流れを調整する制御面であり、アプリの体感にも影響する

## 深入りしないこと

- RFC の数式や詳細規則

## 次章への橋

# 第8章 RST_STREAM と GOAWAY から終了処理を学ぶ

## 章の役割
HTTP/2 における「止め方」「失敗の伝え方」を整理する。  
接続全体と個別ストリームの境界を、終了処理から逆に理解する章。

## この章で答えたい問い
- なぜ個別中断と接続終了を分ける必要があるのか
- RST_STREAM と GOAWAY はどう違うのか
- graceful shutdown は何を意味するのか
- 再試行やアプリ設計にどう影響するのか

## 主要トピック
- ストリーム単位の失敗
- 接続単位の終了
- graceful shutdown
- error と shutdown の関係
- retry や recovery の考え方

## 残したいメンタルモデル
- 1本の接続に複数の通信が載るため、終了処理も階層的に考える必要がある
- stop には複数の粒度がある

## 深入りしないこと
- 全エラーコードの網羅
- 実装依存の retry 戦略詳細

## 次章への橋
接続・状態・イベントの分離をさらに進める視点として、次章で Sans-IO の入口を扱う。

## 章の役割

## 深入りしないこと

# 第11章 HTTP/2 ライブラリ設計レビュー会

## 章の役割
本書全体の総復習として、架空の HTTP/2 ライブラリ設計レビューを通じて各章の論点を再統合する。  
知識の要約ではなく、設計判断の言葉で HTTP/2 を言い直せるようにする章。  
同時に、中級から上級への橋として、責務分割を自分で守る側へ視点を移す章でもある。

## この章で答えたい問い
- HTTP/2 ライブラリ設計で最初に守るべき境界は何か
- Frame 処理、dispatch、状態遷移、イベント生成はどう切り分けるべきか
- flow control と shutdown の粒度差を API 設計でどう保つべきか
- Sans-IO を意識するなら、I/O とプロトコル核をどう見分けるべきか
- 設計レビューでは何を確認すればよいのか

## 主要トピック
- Connection / Stream の分離
- parse と意味処理の分離
- dispatch の責務
- Event 設計
- 非同期処理と並行状態の表現
- flow control の調停責務
- RST_STREAM と GOAWAY の粒度差
- Sans-IO を意識した I/O と状態遷移の分離
- application boundary
- 設計レビュー観点の整理

## 残したいメンタルモデル
- HTTP/2 ライブラリ設計レビューとは、クラス図の好みを論じることではない
- 何を隠し、何を露出するかを確認する場である
- 接続単位、ストリーム単位、状態遷移、制御面、終了処理の粒度差を守れるかが重要である
- 本書で学んだのは断片知識ではなく、そうした判断を支える地図である

## 深入りしないこと
- 特定実装の完全なクラス図提示
- 実装言語ごとの詳細なデザインパターン比較
- Sans-IO 実装の詳細設計
- テスト戦略の詳細技法

## 次への橋
- 上級書では、この章で設計観点として整理した責務差を、実装・テスト・仕様読解へ接続する
- HTTP/3 や Sans-IO の具体設計を学ぶときも、何を隠し何を露出するかという視点を持ち込める
