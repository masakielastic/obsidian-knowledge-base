## 第9章　Sans-IO という考え方への入口

**初学者**「ここまで読んで、HTTP/2 は接続全体の責務とストリーム単位の責務が分かれていて、イベントや終了処理も階層的だということはかなり見えてきました。でもその先で Sans-IO という言葉が出てくると、急に難しく感じます。I/O を分けると言われても、通信プロトコルなのだから I/O と切り離せないのではないか、と思ってしまいます」

**解説者**「その反応は自然です。Sans-IO は、I/O を不要にする考え方ではありません。そうではなく、I/O とプロトコル処理を同じ場所に閉じ込めないようにする設計思想です。HTTP/2 のように状態遷移とイベント生成が中心になるプロトコルでは、この分離が欲しくなる理由があります」

**実装者**「実務でも、そこを分けたい場面は多いです。たとえばテストのしやすさ、イベントループやランタイムへの依存の減らし方、サーバーとクライアントで共通に使える部分の整理などです。ソケット読み書きと状態遷移が一体化していると、変更も検証もつらくなります」

**解説者**「この章では、Sans-IO の完成形を教えるのではなく、なぜそういう設計が欲しくなるのかを整理します。HTTP/2 を通じて見えてきた責務分割を、もう一段抽象化する章だと思ってください」

### Sans-IO は何を分けようとしているのか

まず最初に、Sans-IO を誤解しないようにしておきます。Sans-IO は「I/O がいらない」という意味ではありません。通信プロトコルである以上、最終的には必ずどこかでソケットやストリームやバッファとのやり取りが必要です。

ここで Sans-IO が言おうとしているのは、**プロトコル処理の本質は I/O そのものではなく、受け取った入力に対する状態遷移とイベント生成にある**、ということです。これはアウトラインの第9章でも残したいメンタルモデルとして明示されています。

HTTP/2 をここまで学んできた流れで言い換えるなら、プロトコル処理の中心は「ソケットから読んだ」という事実そのものではなく、「その結果として接続状態がどう変わり、どのストリームに何が起き、上位へどんなイベントを渡すべきか」です。逆に送信側でも、「今どのフレームをどういう順で書くか」は重要ですが、その前提には「現在の状態では何を送るべきか」「何をまだ送ってはいけないか」という判断があります。

Sans-IO は、この判断の核を I/O の周辺事情から切り出して見ようとします。つまり、  
**I/O は境界、状態遷移は中核**  
という見方です。

この見方があると、HTTP/2 の理解がさらに一段深まります。第3章では Session / Stream / Event を API 設計として見ました。第5章ではクラス責務として見ました。第6章では低・中・高レイヤーの分業として見ました。Sans-IO は、その中でもとくに「中核となるプロトコル状態」と「外側の I/O 実行環境」を分けたい、という発想です。

### なぜ I/O と状態遷移を分けたくなるのか

ここで本章の中心問いに答えます。なぜ I/O と状態遷移を分けたくなるのでしょうか。

理由の第一は、HTTP/2 の本質が単なる読み書きではないからです。HTTP/2 は、接続全体の状態、複数ストリームの状態、flow control、終了処理、イベント生成など、**並行する複数の状態を扱うプロトコル**でした。第4章で見た通り、非同期処理と相性がよいのも、これが速度の問題ではなく状態管理の問題だからでした。

もし I/O と状態遷移がひとつの巨大なコード塊になっていると、「ソケットから読んだらすぐに内部状態も更新し、そのまま相手へ書き返し、同時にアプリ向けイベントも投げる」といった具合に、あらゆる責務が一箇所へ集まりやすくなります。すると、Connection と Stream の境界も、終了処理の粒度差も、flow control の調停も、全部が I/O の都合に引きずられます。

一方で、I/O と状態遷移を分けると、入力はまず「プロトコルエンジンへ渡す材料」になります。そこで接続やストリームの状態が更新され、必要なイベントや送信候補が得られます。その後に、実際のソケット書き込みやイベントループとの統合を別の層で行えます。

私はこの違いを、**出来事を処理するのか、出来事に反応して状態を更新するのか**という差として見ると分かりやすいと思います。Sans-IO は後者を中心に据えます。

### HTTP/2 は Sans-IO を考えたくなる題材である

すべての通信処理が同じ程度に Sans-IO を欲しがるわけではありません。しかし HTTP/2 は、とくにその動機が見えやすい題材です。

なぜなら、HTTP/2 には次の特徴があるからです。

ひとつは、接続全体とストリーム単位の二層構造があることです。第3章と第5章で見たように、Session / Stream の区別や Connection / Stream の責務差は、設計の中心でした。これがあるため、I/O の読み書き手続きよりも、「どの粒度の状態が変わったか」が重要になります。

もうひとつは、イベント生成が本質に近いことです。ヘッダーを受け取った、データを読める、ストリームが終了した、接続が shutdown 方向に入った。HTTP/2 の利用者が知りたいのは、こうした意味のある変化です。第3章で Event は単なる通知機構ではなく、状態変化をどの粒度で露出するかの設計判断だと整理しました。Sans-IO は、この「イベントを生むプロトコル核」を切り出す発想と相性がよいのです。

さらに、flow control や shutdown のような制御面もあります。第7章と第8章で見たように、HTTP/2 では「いま送れるか」「どの粒度を閉じるか」が重要でした。これらは I/O の都合だけで決まるのではなく、プロトコル状態によって決まります。だから状態遷移の核を独立して見たくなります。

この意味で、HTTP/2 は Sans-IO への入口として非常によい教材です。I/O を分けること自体が目的なのではなく、**状態遷移とイベント生成が主役だと気づくこと**が目的です。

### Sans-IO で何が見えやすくなるのか

では、Sans-IO 的に考えると何が見えやすくなるのでしょうか。

まず見えやすくなるのは、プロトコル処理の中核です。ソケット操作やイベントループの integration は実装環境によって変わりますが、HTTP/2 の接続状態、ストリーム状態、flow control、終了処理のような核は、本質的にはより安定した責務です。Sans-IO では、その安定した部分を前面に出せます。

次に見えやすくなるのは、イベントの意味です。I/O と密結合している設計だと、「読めたから呼ぶ」「書けたから進める」といった実行順序に意識が引っ張られます。Sans-IO 的に見ると、「この入力を受けて、どの状態が変わり、どのイベントを上位へ返すか」が中心になります。すると API 設計も、イベントループの都合より責務境界の都合で考えやすくなります。

さらに、クライアントとサーバー、あるいは異なるランタイム間の共通部分も見えやすくなります。I/O の実行方法は違っても、状態遷移やイベント生成の核が共通なら、そこを共有基盤として扱いやすくなります。第6章で見た共通基盤化の話ともつながります。低レイヤーの bytes 処理と中レイヤーの抽象化のあいだに、さらに「I/O に依存しないプロトコル核」を意識できるわけです。

### テストしやすさはなぜ上がるのか

Sans-IO の利点として、よくテストしやすさが挙げられます。アウトラインでも主要トピックに含まれています。

これは単なるおまけではありません。HTTP/2 のように状態遷移が複雑なプロトコルでは、テストしやすさは設計の質にかなり直結します。

I/O と状態遷移が一体化していると、テストしたいのは「このフレーム入力に対してストリームが閉じるべきか」という状態遷移なのに、実際にはソケット、イベントループ、タイミング、書き込み可否といった周辺環境まで一緒に持ち込むことになりがちです。すると、何を検証したいのかがぼやけます。

一方、Sans-IO 的に状態遷移の核が分かれていれば、「この入力を与えたら、どのイベントが返り、どの内部状態になり、どの送信候補が出るか」を直接テストできます。つまり、**プロトコルとして正しいか**を、I/O 環境から切り離して確かめやすくなります。

私はこれを、テストしやすさというより「問いの粒度が合う」と表現したいです。HTTP/2 で本当に検証したいことは、しばしば状態遷移と責務境界です。Sans-IO は、その問いを素直に立てやすくします。

### ランタイム依存性をどこまで減らせるのか

Sans-IO が求められるもう一つの理由は、ランタイム依存性を減らしやすくすることです。アウトラインでは「ランタイム依存性の縮小」も主要トピックとして挙げられています。

HTTP/2 を扱う環境はさまざまです。イベントループ型のランタイムもあれば、スレッドベースの実装もあります。クライアントもサーバーもあります。I/O API の形も異なります。こうした差異に対して、プロトコル状態遷移の核まで毎回作り直していたら、再利用もしにくいし、理解の地図も安定しません。

Sans-IO は、I/O 部分を完全に共通化することを目指すのではなく、**共通化できる中心核と、環境依存の境界面を見分けやすくする**発想だと言えます。

もちろん、現実には完全分離は簡単ではありません。タイミング、バッファ管理、送信キュー、observability、API の ergonomics など、境界ににじみ出る要素は多くあります。だから本章では Sans-IO を魔法の解決策としては扱いません。むしろ、中級者として大事なのは、「どこまでが環境依存で、どこからがプロトコル核なのかを意識すること」です。

### Sans-IO は HTTP/2 のどのレイヤーに位置づくのか

第6章の低・中・高レイヤーの地図に照らすと、Sans-IO は単純にどれか一層へぴたりとはまりません。ですが、主に **低レイヤーと中レイヤーの境界付近を整理する発想** として捉えると分かりやすいです。

低レイヤーの frame parse や状態遷移の正しさに深く関わりつつ、中レイヤーのイベント生成や抽象化にもつながる。けれども実際のソケット統合やアプリケーションの業務意味までは担わない。こうした位置づけです。

つまり Sans-IO は、新しい「高レイヤー API」そのものではありません。むしろ、I/O に引きずられずに HTTP/2 の状態遷移とイベント生成を整理したいときに現れる、設計上の焦点だと見るほうがよいでしょう。

この見方をすると、第2章で見た「ライブラリは複雑さを再配置する」という話も、より深く理解できます。Sans-IO は、その再配置の中で「I/O の複雑さ」と「状態遷移の複雑さ」をさらに分けてみようとする考え方なのです。

### Sans-IO をどう学べばよいのか

ここで、この章のスコープも明確にしておきます。執筆ガイドラインでも、本書は上級編向けの詳細実装を勝手に大量追加しないことが求められています。

したがって、本章では Sans-IO のフル設計や高度な状態機械の実装には入りません。ここで読者に持って帰ってほしいのは、次のような問いです。

なぜこのライブラリはイベントループと密結合しているのか。  
なぜ別の実装ではプロトコルコアを切り出しているのか。  
どの部分が I/O 依存で、どの部分が HTTP/2 の状態遷移そのものなのか。  
どこを共通化すると長持ちし、どこは環境に任せたほうが自然なのか。

これらの問いが立てられるだけでも、Sans-IO という言葉はかなり実りあるものになります。重要なのは用語を覚えることではなく、責務と境界をより細かく見る目を得ることです。

### Sans-IO から逆に見える HTTP/2 の姿

この章の最後に、Sans-IO から逆向きに HTTP/2 を見直しておきます。

HTTP/2 は、接続共有と多重化を持つプロトコルでした。  
そのため、Connection と Stream の責務が分かれました。  
複数の状態変化を扱うため、Event が重要になりました。  
flow control や shutdown によって、制御面と終了面も階層化されました。

ここまで来ると、HTTP/2 の中心が「ソケットから読んでそのまま返すこと」ではなく、「入力に応じて状態を更新し、意味のあるイベントを生成し、必要な出力候補を決めること」にあると見えてきます。

Sans-IO は、その見え方をさらにはっきりさせるための考え方です。I/O は必要です。しかし主役ではありません。主役は、HTTP/2 の状態遷移と責務境界です。

私は、この章を通して読者に「Sans-IO は上級者だけの特殊趣味ではなく、ここまで学んできた責務分割をさらに洗練して見るための自然な延長線上にある」と感じてもらえれば十分だと思っています。

| 観点 | I/O 側 | プロトコル側 |

### この章で残すべきメンタルモデル

Sans-IO は、I/O を不要にする考え方ではありません。  
I/O とプロトコル状態遷移を同じ場所に閉じ込めないようにする設計思想です。

HTTP/2 の本質は、  
接続やストリームの状態がどう変わり、どんなイベントを生み、何を次に送るべきか、という判断にあります。

だから Sans-IO は、  
HTTP/2 を「読み書きの手続き」ではなく「状態遷移とイベント生成の核」として見るための入口になります。

## まとめ

- Sans-IO は I/O をなくす考え方ではなく、I/O とプロトコル状態遷移を分けて考える発想である
    
- HTTP/2 は接続共有、多重化、イベント生成、flow control、shutdown を持つため、Sans-IO を考えたくなる題材である
    
- 分離の中心にあるのは、ソケット操作ではなく状態遷移とイベント生成を主役として見ることだ
    
- Sans-IO にすると、テストしやすさやランタイム依存性の整理がしやすくなる
    
- 本章の目的は Sans-IO の完成形を学ぶことではなく、なぜその設計が欲しくなるのかを理解することである
    

次章では、本書の締めくくりとして、HTTP/2 でここまで整理してきた抽象が HTTP/3 にどうつながるかを確認します。そこで、何が引き継がれ、何が transport の変化によって新しく難しくなるのかを見ていきます。
