
## 第2章　HTTP/2 ライブラリは何を隠し、何を見せるのか

**初学者**「HTTP/2 対応のライブラリを使えば、フレームのことや状態管理を細かく知らなくても済みますよね。だったら、HTTP/2 の複雑さはライブラリが全部引き受けてくれている、と考えていいのでしょうか」

**解説者**「半分は正しくて、半分は危うい理解です。たしかにライブラリは多くの複雑さを内部に押し込みます。ですが、複雑さそのものを消しているわけではありません。利用者が触れる API の形、イベントの種類、エラーの出方、バックプレッシャーの扱い方として、その複雑さは別の場所に現れます」

**実装者**「実務でもそこを誤解すると、使い方がちぐはぐになります。たとえばストリーム単位の問題を接続全体の問題として扱ってしまったり、逆に接続全体の shutdown を個別リクエストの失敗と同じように見てしまったりします」

**解説者**「この章では、ライブラリが何を隠し、何を見せるのかを整理します。狙いは API を暗記することではありません。抽象化がどこに複雑さを移したのかを読むことです」

### ライブラリは複雑さを消さない

HTTP/2 を直接実装しない立場から見ると、ライブラリはとても強力です。フレームを自分で組み立てなくていい。ストリーム ID の割り当てを毎回意識しなくていい。SETTINGS の往復や、ある程度の状態管理を自動で処理してくれる。これだけを見ると、複雑さが消えたように見えます。

しかし、私はここで「消えた」という言い方は使いません。より正確には、**複雑さが別の形に変換された**と考えるべきです。

HTTP/2 が本来持っている難しさは、主に三つあります。ひとつは、1本の接続の上で複数の通信が同時進行すること。ひとつは、接続単位とストリーム単位で責務の粒度が異なること。もうひとつは、フレーム、状態遷移、制御メッセージ、終了処理が単純な request/response よりずっと多層的であることです。

ライブラリは、こうした難しさをそのまま利用者に見せるのではなく、扱いやすい抽象へ変換します。Session や Connection オブジェクト、Stream オブジェクト、イベント通知、非同期 API、エラーオブジェクト、バックプレッシャー制御などがその例です。つまり、複雑さは消えたのではなく、**API の表面へ再配置された**のです。これは本章で最も重要な見方です。

この視点を持つと、ライブラリの API を見る目が変わります。メソッド名やイベント名を並べて覚えるのではなく、「なぜこういう分かれ方をしているのか」「なぜこの責務はこのオブジェクトに置かれているのか」を読むようになります。

### ライブラリが隠すもの

では、HTTP/2 ライブラリは何を隠すのでしょうか。まず隠すのは、**低レイヤーの手続きの細部**です。

たとえば、HEADERS フレームや DATA フレームをどの順序で受け取り、どうやってパースし、どのストリームに結びつけるか。SETTINGS のやり取りをどのタイミングで反映するか。PING や WINDOW\_UPDATE の扱いをどう内部状態に反映するか。これらをすべて利用者が毎回意識していたら、アプリケーションを書く以前にプロトコル実装を書いているのとほとんど変わりません。

ライブラリは、この低レイヤーの複雑さを内部に押し込みます。利用者は通常、「ヘッダーを送る」「データを受け取る」「ストリームが閉じたことを知る」といった、もう少し高い粒度で HTTP/2 を扱えます。

また、ライブラリは**状態遷移の細かな整合性**もかなり隠します。HTTP/2 では、ストリームは開いて終わるだけではなく、片方向が閉じている状態や、途中でリセットされた状態など、段階的な状態を持ちます。接続全体にも、通常運転の状態と shutdown に向かう状態があります。こうした状態管理を直接 exposed しすぎると、利用者は毎回 RFC 的な整合性まで背負うことになります。そこでライブラリは、よく使う操作を安全な API としてまとめます。

さらに、**フレーム列と利用者の意図の間の翻訳**も隠されます。アプリケーション開発者が考えたいのは、「レスポンスヘッダーが来た」「ボディを読み進める」「このリクエストを打ち切る」といった意図です。内部ではそれが複数のフレームや状態変化に対応していても、多くのライブラリはそこを一つ上の表現に持ち上げます。

ここまでは、いわば「隠してよい複雑さ」です。利用者全員が毎回触る必要はありません。むしろ隠したほうが安全で、利用しやすくなります。

### それでも利用者の前に残るもの

隠されるものが多い一方で、どうしても利用者の前に残るものがあります。それは、**抽象化しても消せない責務**です。

もっとも代表的なのは、接続単位とストリーム単位の区別です。HTTP/2 では複数のストリームが1本の接続を共有します。したがって、個別の通信に関する責務と、接続全体の責務は分けて考えなければなりません。ライブラリが Session と Stream のような別オブジェクトを用意するのは、この区別を API に反映しているからです。これは単なるクラス設計の好みではありません。プロトコルの構造が、そのまま利用者向けの抽象に影響しているのです。

また、**イベントや非同期性**も消しきれません。HTTP/2 は、複数のストリームが並行して進み、それぞれのヘッダー、データ、終了、エラーが異なる順序で到着し得るプロトコルです。同期的な request/response の連鎖だけで自然に表現するのは難しい。だから多くのライブラリは、イベントやコールバックや Promise やストリーム API の形で、この並行性を利用者の前に残します。

さらに、**バックプレッシャーや流れの制御感覚**も残ります。フロー制御の数式や詳細ルールまでは隠せても、「一度に無制限に送れるわけではない」「受信側・送信側の都合で流れが止まることがある」といった感覚までは消せません。ここは利用者が最低限知っておく必要があります。後の章で詳しく扱いますが、この時点では「流量の管理は内部実装だけの話ではなく、API 利用者の体感にも出てくる」と押さえておけば十分です。

そして最後に、**失敗と終了の粒度**も利用者に残ります。個別のストリームだけが失敗したのか。接続全体が閉じる方向に入ったのか。この違いは retry の判断や、上位アプリケーションの扱い方に直結します。ライブラリはエラーの見せ方を整理してくれますが、区別そのものまでは消せません。

### 高水準 API と低水準 API の違い

HTTP/2 ライブラリには、抽象化の度合いに幅があります。ここでは大ざっぱに、高水準 API と低水準 API という二つの見方で整理しておきます。

高水準 API は、利用者が「HTTP/2 を使っている」ことをあまり意識しなくても使えるように作られています。リクエストを投げる、レスポンスを受け取る、ヘッダーを扱う、といった操作が中心で、フレームや細かな状態遷移はほとんど見えません。多くのアプリケーション開発では、この層で十分です。

一方、低水準 API は、Session、Stream、イベント、フロー制御、エラー通知などが比較的露出しています。こちらは「何が起きているか」をより細かく制御したいときに使われます。たとえばサーバー実装、ゲートウェイ、RPC 基盤、デバッグ性の高いミドル層などでは、この粒度が重要になります。

ここで重要なのは、高水準 API が「HTTP/2 を理解しなくてよい API」ではないことです。高水準であるほど、内部の複雑さをうまく隠しています。しかし何か問題が起きたとき、あるいは性能や shutdown や流量制御を調整したくなったときには、その背後にある Session、Stream、接続共有、終了処理の概念を知っていないと説明がつきません。

逆に、低水準 API は露出が多いぶん難しく見えますが、複雑さの出どころを理解するには非常に教育的です。本書では特定ライブラリのリファレンス解説は避けますが、次章では Node.js の http2 モジュールを題材に、この責務の露出のされ方を読みます。

### API を読むとは、責務分割を読むこと

HTTP/2 ライブラリの API を見るとき、多くの人は「どう使うか」を先に見ます。もちろんそれは重要です。しかし中級者に必要なのは、その一歩先です。私は、**API を読むとは責務分割を読むことだ**と考えています。

なぜ Session と Stream が別なのか。なぜ接続確立後に複数のイベントが飛ぶのか。なぜ request オブジェクトと response オブジェクトの背後に、ストリームやセッションの文脈が残っているのか。なぜ shutdown に関する通知が、個別リクエストの失敗とは別に存在するのか。こうした疑問は、使い方の細部ではなく、設計の筋道を読む問いです。

HTTP/2 の構造上、1本の接続には複数の通信が同居します。したがって、接続全体に関わる責務は一箇所に集約したい。他方で、個別通信ごとのヘッダー、データ、終了、エラーはストリーム単位で扱いたい。すると API も自然に、その二層を反映した形になります。

また、フレームそのものをそのまま利用者に見せないとしても、イベントや状態遷移の通知は必要です。なぜなら HTTP/2 は、利用者にとっても「相手がいつ何を送ってきたか」に反応する必要があるプロトコルだからです。つまり、**フレームを隠しても、状態変化までは隠せない**のです。

この観点から見ると、ライブラリごとの差も読みやすくなります。違いは単なる流儀ではなく、「どこまで隠し、どこから利用者に責務を渡すか」の判断の違いとして見えてきます。

### 「便利さ」の中身を言語化する

HTTP/2 ライブラリを使うと便利だ、という言い方はよくします。ですが中級では、その便利さの中身を具体的に言語化したいところです。

便利さの一部は、確かに単純な省力化です。面倒なフレーム処理や状態遷移の整合性チェックを自動で処理してくれるから便利です。

しかし、もっと本質的な便利さは、**責務の置き場所を明確にしてくれること**にあります。セッション管理はここ。個別ストリームの処理はここ。終了通知はこの形。送受信の流れに関する制御はこの API。このように、ライブラリは単にコード量を減らすだけではなく、考える単位を整理してくれます。

だからこそ、便利さの裏側には設計があります。便利だから中身を見なくてよい、ではありません。便利であるほど、どの責務がどこへ移されたのかを理解したほうが、利用も troubleshooting もはるかに安定します。

HTTP/2 は、とくにこの点が重要です。HTTP/1.1 的な感覚で「1リクエストに1レスポンス」という線形モデルを前提にすると、接続共有やストリーム並行や階層的な終了処理が見えにくくなります。ライブラリの便利さは、その複雑さを利用可能な形へ圧縮した結果です。その圧縮の仕方を読むことが、中級者にとっての学習になります。

### ライブラリ利用者に残る最低限の責任

ここまでを踏まえると、HTTP/2 ライブラリの利用者に残る最低限の責任も見えてきます。

第一に、接続単位とストリーム単位を混同しないことです。個別通信の失敗と、接続全体の問題は別です。この区別が曖昧だと、ログの読み方もエラーハンドリングも不安定になります。

第二に、イベントや非同期性を単なる実装都合として切り捨てないことです。HTTP/2 ではそれが自然な表現です。並行する複数の状態変化を扱う以上、イベントや非同期 API は本質に近い場所にあります。

第三に、流れの制御や shutdown の感覚を持つことです。詳細なアルゴリズムまでは不要でも、「送れる」「止まる」「閉じる」に粒度の違いがあることは理解しておく必要があります。

第四に、API の形を責務分割として読むことです。どのオブジェクトが何を代表しているか、どのイベントがどの状態変化を意味しているかを見れば、ライブラリの背後にある HTTP/2 の構造が見えてきます。これができるようになると、単なる利用者から一段進んで、設計を読み解ける利用者になります。

### 整理ブロック：ライブラリが主に隠すもの

| 隠すもの | 典型的な内容 | なぜ隠すのか |
| --- | --- | --- |
| フレーム処理の細部 | parse、serialize、対応づけ | 毎回利用者が扱うには粒度が低すぎるから |
| 細かな状態遷移の整合性 | ストリーム状態、接続状態 | 安全性と利用しやすさを高めるため |
| 制御フレームの手続き | SETTINGS、PING、WINDOW\_UPDATE など | 利用者の主目的はアプリの通信だから |
| 生のやり取りの翻訳 | フレーム列から意図的な操作への変換 | アプリケーションの関心に近づけるため |

### 整理ブロック：それでも利用者に残るもの

| 利用者に残るもの | どのように見えるか | なぜ残るのか |
| --- | --- | --- |
| 接続単位とストリーム単位の区別 | Session / Stream などの API 分離 | プロトコル自体が階層構造を持つから |
| 非同期性とイベント | callback、event、Promise、stream API | 複数通信の並行進行を表す必要があるから |
| バックプレッシャー感覚 | 書き込み停止、読み取り待ち、送受信制御 | 流量制御は利用者の体感にも影響するから |
| 終了と失敗の粒度 | 個別中断、接続終了、shutdown 通知 | retry や復旧判断に直結するから |

### 整理ブロック：高水準 API と低水準 API

| 観点 | 高水準 API | 低水準 API |
| --- | --- | --- |
| 見える粒度 | リクエスト、レスポンス、ヘッダー | Session、Stream、イベント、制御 |
| 向いている場面 | 一般的なアプリ開発 | サーバー、ミドル層、基盤、詳細制御 |
| 利点 | 使いやすい、学習コストが低い | 構造が見えやすい、制御しやすい |
| 注意点 | 問題発生時に裏側の理解が必要 | 責務の理解がないと難しい |

### この章で残すべきメンタルモデル

ライブラリは HTTP/2 の複雑さを消しません。  
複雑さを、利用者が扱いやすい API、イベント、オブジェクトの形へ変換します。

だから、HTTP/2 ライブラリを読むとは、便利な使い方を覚えることではありません。  
どの複雑さが内部へ隠され、どの責務が利用者側へ残されたのかを読むことです。

Session と Stream の分離、イベント駆動、バックプレッシャー、終了処理の粒度は、その再配置の結果として見えています。

## まとめ

- HTTP/2 ライブラリは複雑さを消すのではなく、API の形へ再配置する
    
- ライブラリはフレーム処理や細かな状態整合性を主に内部へ隠す
    
- それでも接続単位とストリーム単位の区別、非同期性、終了処理の粒度は利用者に残る
    
- 高水準 API と低水準 API の違いは、どこまで責務を露出するかの違いでもある
    
- API を読むことは、責務分割を読むことでもある
    

次章では、具体例として Node.js の http2 モジュールを見ながら、Session、Stream、Event がどのように責務を分担しているかを読み解きます。
