
## 第6章　低・中・高レイヤーで HTTP/2 を捉え直す

初学者「ここまでで、Connection と Stream の責務や、イベント駆動が自然に見える理由はかなり整理できてきました。でも今度は、別の迷いが出てきます。どこまでが HTTP/2 ライブラリの仕事で、どこからがフレームワークやアプリケーションの仕事なのかが、まだ少し曖昧なんです」

解説者「その迷いは重要です。HTTP/2 を理解するというのは、Connection や Stream の内部だけを理解することではありません。どの責務を共通基盤としてまとめ、どの責務をミドル層へ持ち上げ、どこから先をアプリケーションへ渡すかまで見えるようになる必要があります」

実装者「実務では、その境界が曖昧だとつらくなります。低レイヤーの事情をアプリが背負いすぎたり、逆に高レイヤーが本来知るべき shutdown や backpressure の情報まで隠されてしまったりします」

解説者「この章では、HTTP/2 を低・中・高レイヤーの分業として整理します。目的は『正しい層の数』を決めることではありません。何を共通化し、何を各層に委ねるべきか、その見取り図を得ることです」

### HTTP/2 を単独の知識として見るだけでは足りない

入門段階では、HTTP/2 はひとつのプロトコルとして学ばれます。それで十分な場面も多いです。ですが中級になると、それだけでは足りません。なぜなら、実際の利用や設計の現場では、HTTP/2 は単独で存在しているのではなく、**複数の層にまたがる分業の中心**として現れるからです。これはアウトラインでも第6章の役割として明示されています。

たとえば、フレームの parse や状態管理を行う低レイヤーがあります。その上に、ストリーム管理やイベント化やバックプレッシャーのような、中間的な抽象化を担う層があります。さらにその上には、RPC、フレームワーク、アプリケーションのような高レイヤーがあります。実際に私たちが HTTP/2 を「使う」ときは、ほとんどの場合、このどれかの層から関わっています。

ここで重要なのは、HTTP/2 の理解を「フレームがこう流れる」だけで終わらせないことです。どの責務をどの層が引き受けるのか、どの層が何を知り、何を知らなくてよいのかが見えて初めて、HTTP/2 を設計の言葉で扱えるようになります。執筆ガイドラインでも、本書の目的のひとつとして「低・中・高レイヤーで HTTP/2 を捉え直せること」が明示されています。

私はここで、HTTP/2 を「公共基盤に近い技術」として見るのがよいと思っています。すべてのアプリケーションが自前でフレーム解析や状態遷移を実装すべきではありません。しかし同時に、すべての事情を最上位の API に押し込んで忘れてよいわけでもありません。だからこそ、レイヤーの見取り図が必要になります。

### 低レイヤーは何を担当するのか

まず低レイヤーです。ここが担当するのは、HTTP/2 のもっともプロトコル寄りの部分です。アウトラインでは、frame、state、HPACK、parse などがここに置かれています。

具体的には、フレームの parse / serialize、接続やストリームの状態遷移の基礎、制御フレームの処理、フロー制御に関わる低レベルな数値更新、ヘッダー圧縮の扱いなどがここに入ります。第5章で言えば、frame codec や parser のような責務が低レイヤー寄りでした。

低レイヤーの特徴は、**正しさと一貫性が最優先される**ことです。アプリケーションごとの都合で振る舞いを変えすぎると困ります。逆に言えば、低レイヤーは利用者の業務上の意味をあまり知りません。たとえば「この通信は注文作成 API だから重要だ」といったことは知りません。知るべきなのは、フレームの種類、ストリーム ID、状態遷移、送受信可能性といった、プロトコルの基礎事実です。

この層を厚くしすぎると、利用者にとって難しくなります。ですが薄すぎると、上位が毎回 bytes 処理や状態整合性まで背負うことになります。したがって、低レイヤーは「上位が毎回再実装すべきでない、共通のプロトコル基盤」を担う層だと考えると整理しやすいです。

ここで注意したいのは、低レイヤーが「見えない場所」だから重要でないわけではないことです。むしろ HTTP/2 の安定性はこの層に大きく依存します。ただし、本書の中心はこの低レイヤーそのものの全面解説ではありません。ここから一段上にある中レイヤーこそ、中級者にとって最も教育的です。

### 中レイヤーは何をする層なのか

中レイヤーは、この章の中心です。アウトラインでも、stream 管理、event、backpressure、abstraction などが中レイヤーに置かれています。

私は中レイヤーを、**低レイヤーの事実を、上位が扱える責務単位へ変換する層**だと考えています。

低レイヤーが知っているのは、たとえば「HEADERS フレームが届いた」「このストリーム ID の window が変わった」「接続が GOAWAY 状態へ入った」といった事実です。しかし上位がそのまま欲しいとは限りません。アプリケーションが知りたいのは、「この通信のヘッダーが読める」「いまは送信が詰まっている」「新しい通信はもう受け付けるべきではない」といった、より意味のある単位です。

この翻訳を担うのが中レイヤーです。Session や Stream のオブジェクト、イベント駆動の API、backpressure の表現、個別通信と接続全体の区別、終了処理の粒度差などは、まさにこの層の仕事です。第2章から第5章までで見てきた多くの話題は、実は中レイヤーの設計問題でした。ライブラリが何を隠し、何を見せるのかという問いも、この層で具体化します。

中レイヤーの難しさは、低レイヤーの正しさを壊さずに、上位にとって扱いやすい抽象を作らなければならないことです。抽象化が浅すぎると、上位がプロトコルの泥をかぶります。抽象化が深すぎると、shutdown や flow control や retry 判断に必要な情報まで消えてしまいます。

私は、この中レイヤーこそ HTTP/2 を「便利なライブラリの裏側に隠れた仕組み」ではなく、「抽象化・責務分割・状態管理の教材」として理解するための中心だと思っています。これは執筆ガイドラインの核心とも一致します。

### 高レイヤーは何を知り、何を知らなくてよいのか

高レイヤーには、RPC フレームワーク、Web フレームワーク、アプリケーション、あるいは業務ロジックが位置づきます。アウトラインでは gRPC / Connect / app のようなものが高レイヤーの例として挙げられています。

この層が主に知りたいのは、通信の業務上の意味です。要求が来た、レスポンスを返す、ストリームが終わった、失敗した、再試行するか判断する。こうした判断は、HTTP/2 の bytes 処理や細かな状態遷移そのものではなく、それらが翻訳された結果の上で行われます。

高レイヤーは、低レイヤーの事情をすべて知る必要はありません。むしろ知りすぎると困ります。毎回フレーム列や HPACK の事情まで意識して業務処理を書くのは不自然です。

ただし、高レイヤーが何も知らなくてよいわけでもありません。たとえば、接続が閉じる方向に入ったこと、個別通信が中断されたこと、送信が詰まっていること、接続共有の都合で複数通信が相互に影響し得ること。こうした情報は、抽象化された形で高レイヤーに届く必要があります。さもないと retry、graceful shutdown、負荷制御、タイムアウト設計が不自然になります。

つまり高レイヤーは、「HTTP/2 のすべて」を知る必要はありませんが、**自分の判断に影響する形へ翻訳された HTTP/2 の事情**は知る必要があります。ここで中レイヤーの質が問われます。何を露出し、何を隠すかのバランスが悪いと、高レイヤーは過剰に賢くなるか、逆に無知すぎて不安定になります。

### 低・中・高レイヤーは上下関係ではなく分業である

ここで誤解を避けたいのは、低・中・高レイヤーという言い方が、単なる優劣や上下関係を意味するわけではないということです。これは**責務の分業**です。

低レイヤーが基礎であることは確かですが、だからといって常にそこが設計の主役とは限りません。高レイヤーが最終的な利用目的に近いことも確かですが、だからといってそこで全部決めてよいわけでもありません。どの層も、それぞれ異なる問いに答えています。

低レイヤーは、「HTTP/2 として正しいか」に答えます。  
中レイヤーは、「HTTP/2 を利用可能な抽象にどう変換するか」に答えます。  
高レイヤーは、「その抽象を使って何を実現するか」に答えます。

この分業を意識すると、議論の噛み合わなさも整理しやすくなります。ある人が「この API は使いにくい」と言っているとき、それは中レイヤーの問題かもしれません。別の人が「仕様上それは許されない」と言っているなら、低レイヤーの問題かもしれません。さらに「アプリ側ではこの情報が必要だ」と言っているなら、高レイヤーの要求かもしれません。

HTTP/2 の議論が難しく見えるのは、しばしばこれらの層の問いが混ざるからです。本章では、それを分けて見るための地図を作っています。

### gRPC や Connect はどの層にいるのか

高レイヤーの例として分かりやすいのが gRPC や Connect です。これらは HTTP/2 の上に成り立つ通信の利用形態ですが、HTTP/2 そのものではありません。アウトラインでも、gRPC / Connect / HTTP/3 へ進むための土台を作ることが本書の目的のひとつとされています。

gRPC や Connect が主に扱うのは、RPC 呼び出し、メッセージ境界、ステータス、サービス定義、アプリケーションレベルの契約です。これは高レイヤーの関心です。ただし、それらは HTTP/2 のストリーム、ヘッダー、終了処理、フロー制御の上に載っています。だから完全に HTTP/2 を忘れてよいわけではありません。

ここで大切なのは、gRPC や Connect は HTTP/2 の代替ではなく、**HTTP/2 の中レイヤーや低レイヤーの上に立つ高レイヤーの利用形態**だと理解することです。すると、「gRPC があるから HTTP/2 を知らなくてよい」という誤解も避けやすくなります。高レイヤーは中レイヤーや低レイヤーの設計に依存しているからです。

また、逆に HTTP/2 ライブラリ側が gRPC の都合を背負いすぎるのも問題です。共通基盤としての HTTP/2 と、特定の RPC スタイルは分けて考えたほうが長持ちします。これも分業の視点です。

### curl のようなツールはどこに位置づくのか

アウトラインでは、curl のようなツールがどの位置にあるのかも問いとして挙げられています。

これを考えるのも、レイヤー感覚を養うのに役立ちます。curl は最上位の業務ロジックではありませんが、低レイヤーのフレーム parser そのものでもありません。利用者から見るとかなり高水準です。URL、ヘッダー、リクエスト、レスポンスという単位で HTTP 通信を扱えるからです。

しかし内部では、接続管理、プロトコル選択、ストリーム処理、流量制御との付き合い方など、中レイヤー寄りの責務もかなり背負っています。つまり、curl のようなツールは「上位向けの顔を持ちながら、中レイヤーの実装を内包する存在」と見ると分かりやすいです。

このように実際のソフトウェアは、きれいに一層だけに住むとは限りません。だからこそ、「どの責務をどの層から借りているか」を見ることが重要です。本章の目的は、製品やライブラリをひとつの箱として見るのではなく、その中にどのレイヤーの責務が含まれているかを分解して読めるようにすることです。

### 共通基盤化はなぜ重要なのか

HTTP/2 をレイヤーで捉えると、共通基盤化の意味も見えてきます。アウトラインでは、この章の主要トピックとして「共通基盤化とコミュニティ分業」「学習コストの分担」が挙げられています。

HTTP/2 の低レイヤーを、すべてのアプリケーションやフレームワークが毎回自前実装するのは非効率ですし、危険です。フレーム parse、状態遷移、フロー制御、終了処理といった基礎は、共通基盤としてある程度まとまっていたほうがよいです。

一方で、中レイヤーや高レイヤーは、用途や言語ランタイムやフレームワーク文化によって変わり得ます。どこまでイベントを露出するか、どのように backpressure を見せるか、アプリにどの単位で通信を渡すかは、一つの答えに固定しにくいです。

だから分業が必要になります。低レイヤーはなるべく共有し、中レイヤーは利用モデルに応じて抽象化し、高レイヤーは業務や RPC やフレームワークに最適化する。この考え方があると、「なぜここは共通化され、なぜここは各ライブラリに委ねられるのか」を説明しやすくなります。

私はこれを、学習コストの分担でもあると考えています。すべての人が低レイヤーの詳細を知る必要はありません。しかし中レイヤーの考え方を知っていれば、ライブラリ設計や API の違いを理解しやすくなります。この本が中級編として中レイヤーを重視する理由はそこにあります。

### レイヤーを意識すると何が見えやすくなるのか

ここまでの話を、読者の得るメリットとして整理しておきます。

まず、ライブラリを「便利な箱」としてではなく、どの層の責務を持っているのかで読めるようになります。第2章で見た「ライブラリは複雑さを消すのではなく再配置する」という見方が、ここでレイヤーの形に展開されます。

次に、Node.js の `http2` モジュールのような API も、単にイベントが多いとかクラスが分かれているという見方ではなく、「低レイヤーの事実を中レイヤーの抽象へ変換している」と読めるようになります。これは第3章から第5章の議論を上から見直した姿です。

さらに、gRPC や Connect や将来の HTTP/3 の学習にもつながります。どの部分が引き継がれ、どの部分が新しくなるのかを、レイヤーごとに分けて考えられるからです。本書全体の出口が HTTP/3 や Sans-IO につながることも、アウトラインで明示されています。

最後に、実務で起きる症状の切り分けがしやすくなります。これは低レイヤーの不具合か、中レイヤーの抽象化不足か、高レイヤーの誤用か。こうした問いを立てられるようになるだけでも、HTTP/2 の理解はかなり実戦的になります。

### 整理ブロック：低・中・高レイヤーの役割

| レイヤー | 主に扱うもの | 中心的な問い |
| --- | --- | --- |
| 低レイヤー | frame、state、parse、HPACK、制御処理 | HTTP/2 として正しいか |
| 中レイヤー | stream 管理、event、backpressure、抽象化 | HTTP/2 をどう利用可能な形に変換するか |
| 高レイヤー | RPC、framework、app、業務処理 | その抽象を使って何を実現するか |

### 整理ブロック：各レイヤーが知るべきこと

| レイヤー | 知るべきこと | 知りすぎなくてよいこと |
| --- | --- | --- |
| 低レイヤー | フレーム、状態遷移、整合性、制御 | 業務上の意味やサービス契約 |
| 中レイヤー | 接続とストリームの責務、イベント化、終了処理の粒度 | アプリ固有の業務ロジック |
| 高レイヤー | 通信の意味、終了通知、送受信制御の結果 | bytes 処理や細かな parse 詳細 |

### 整理ブロック：代表的な技術の見え方

| 例 | 主に位置づく層 | 補足 |
| --- | --- | --- |
| frame parser / codec | 低レイヤー | bytes と仕様整合性を担う |
| Session / Stream API | 中レイヤー | 利用者向け抽象化の中心 |
| gRPC / Connect | 高レイヤー | HTTP/2 の上に立つ利用形態 |
| curl のようなツール | 中〜高レイヤー寄り | 高水準の顔を持ちつつ中層を内包する |

### この章で残すべきメンタルモデル

HTTP/2 は、ひとつの知識の塊ではありません。  
低レイヤーのプロトコル基盤、中レイヤーの抽象化、高レイヤーの利用形態にまたがる分業の対象です。

低レイヤーは正しさを支えます。  
中レイヤーは使える形へ変換します。  
高レイヤーはその抽象の上で意味を扱います。

中級者として特に重要なのは中レイヤーです。  
ここを見ると、ライブラリが何を共通基盤化し、何を利用者へ露出し、どこから先をアプリに委ねているのかが見えてきます。

## まとめ

- HTTP/2 は単独の知識ではなく、低・中・高レイヤーにまたがる分業として捉えると理解しやすい
    
- 低レイヤーは frame、state、parse などの共通プロトコル基盤を担う
    
- 中レイヤーは Session、Stream、Event、backpressure などを通じて利用可能な抽象を作る
    
- 高レイヤーは gRPC / Connect / framework / app のように、通信の業務上の意味を扱う
    
- どこを共通化し、どこを各層に委ねるかを見ることで、HTTP/2 の設計と利用の地図が安定する
    

次章では、このレイヤー分業の中でも利用者が無関係ではいられない制御面として、flow control を扱います。そこでは、低レイヤーの細かな仕様ではなく、「実務で何が詰まりとして見えるのか」という観点から整理していきます。
