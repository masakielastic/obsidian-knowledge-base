
## 第3章　Node.js の http2 モジュールから API 設計を読む

**初学者**「HTTP/2 の API を見ると、`Session` と `Stream` が分かれていたり、イベントがたくさん出てきたりしますよね。あれは Node.js の流儀だからそうなっているのでしょうか。それとも HTTP/2 の構造そのものが、ああいう API を要求しているのでしょうか」

**解説者**「よい問いです。もちろん Node.js らしい表現の仕方はあります。ですが、それだけではありません。Session と Stream の分離や、イベント駆動の形は、HTTP/2 が持っている責務の分かれ方をかなり素直に反映しています」

**実装者**「実務でも、その見方を持つと API が急に読みやすくなります。『このイベントは何を通知しているのか』ではなく、『この通知は接続の責務なのか、個別通信の責務なのか』と見るようになるからです」

**解説者**「この章では Node.js を題材にしながら、API 設計がどこまでプロトコル構造を映しているのかを整理します。Node.js は題材です。本題は HTTP/2 の責務分割です」

### Node.js の API を題材にする理由

本章で Node.js の `http2` モジュールを取り上げるのは、Node.js が唯一の正解だからではありません。むしろ、Session、Stream、イベントという形が比較的見えやすく、HTTP/2 の責務分割を読む教材として都合がよいからです。アウトラインでも、本章の目的は Node.js の使い方そのものではなく、API から HTTP/2 の抽象構造を読み取ることにあると整理されています。

私は、特定の言語やライブラリの API を読むとき、まず「その API は何を楽にしようとしているか」よりも、「その API は何をどこで区切っているか」を見ます。HTTP/2 のように接続共有、多重化、イベント的な状態変化が本質にあるプロトコルでは、この見方が特に有効です。

Node.js の API に Session と Stream という区別があるのは、たまたまクラスを二つ作りたかったからではありません。HTTP/2 自体が、**接続全体に関わる責務**と、**個別の通信単位に関わる責務**を明確に分ける必要があるからです。API は、その差を利用者に読ませる窓になっています。

### Session は何の責務を持つのか

まず見るべきなのは Session です。Node.js では client side でも server side でも、HTTP/2 の通信はまずセッションという単位で始まります。このセッションは、HTTP/2 の接続全体を表す抽象だと考えると分かりやすいです。

ここでいう「接続全体」とは、単に TCP ソケットを包んだもの、という意味ではありません。HTTP/2 では 1 本の接続の上に複数のストリームが載ります。したがって Session が背負うのは、「複数の個別通信を成立させるための共通基盤」としての責務です。

たとえば、SETTINGS の反映、PING、接続全体の寿命、GOAWAY のような終了方向の制御、全体としてのエラー状態などは、個別ストリームではなくセッション側の関心事です。ストリームを新しく開くことも、接続全体の文脈の中で行われます。つまり Session は、「個別の request/response の入れ物」ではなく、**複数のストリームが共存する場を管理する責務**を持っています。

この見方を持つと、Session が存在する理由がはっきりします。HTTP/1.1 的な感覚だと、接続は見えにくいことがあります。1 リクエストごとの処理に意識が寄りやすいからです。しかし HTTP/2 では、接続は単なる土台ではありません。設定、フロー制御、終了処理、ストリーム生成の前提を共有する、能動的な管理単位です。だから API でも Session が独立して現れます。

### Stream は何の責務を持つのか

次に Stream です。Stream は HTTP/2 における個別通信単位を表す抽象です。ひとつのリクエストとレスポンスの往復を、より正確にはその往復を支える論理的な通信路を表していると考えるとよいでしょう。

ここで重要なのは、Stream は「接続から独立したミニ接続」ではないという点です。独自の寿命や状態は持ちますが、あくまで Session の中に属しています。ストリーム ID の空間も、生成順序も、接続全体の文脈に依存します。つまり Stream は自律的な存在ではなく、**接続の中で並行進行する個別責務の単位**です。

この区別は設計上とても大きいです。ヘッダーやデータの送受信、個別ストリームの終了、RST\_STREAM による中断のような話は Stream 側の責務として自然です。他方で、接続全体を閉じるかどうか、今後新しいストリームを受け付けるかどうか、全体設定をどう扱うかは Session 側の責務です。

言い換えると、Stream はアプリケーションに最も近い HTTP/2 の単位ですが、それでも完全にアプリケーションそのものではありません。アプリケーションが「ひとつの通信」として見たいものを、HTTP/2 の構造に沿って載せるための単位です。だからこそ Node.js の API でも、ヘッダー受信やデータ受信、終了通知などが Stream に寄って見えるのです。

### Session と Stream が分かれている意味

Session と Stream を別にする理由は、オブジェクト指向の見た目を整えるためではありません。HTTP/2 では、接続全体に関わる関心事と、個別通信に関わる関心事が衝突しやすいからです。ここを一つの巨大なオブジェクトに押し込むと、責務の境界がすぐ曖昧になります。

たとえば、ある通信が失敗したとします。この失敗が個別ストリームに閉じた話なのか、接続全体の不健全さを示す話なのかは、設計上まったく違います。前者ならそのストリームだけを中断すればよいかもしれません。後者なら新しいストリーム作成を止めたり、再接続や graceful shutdown を考えたりする必要があります。

また、複数ストリームが並行している以上、各ストリームの状態変化は相互に独立して起こり得ます。にもかかわらず、それらは同じ接続上の資源を共有しています。つまり HTTP/2 の設計では、「分けて考えたいが、切り離し切れない」という関係が常にあります。Session と Stream の分離は、その緊張関係を API 上で扱いやすくした結果です。

この意味で、Node.js の `http2` API は HTTP/2 の構造をかなり正直に映しています。Session は共有基盤、Stream は個別通信単位。この二層を意識するだけで、HTTP/2 の見え方はかなり安定します。これは執筆ガイドラインでも重視されている「Connection / Stream / Frame の区別を実装責務や API 設計と結びつける」という観点そのものです。

### Event はなぜ必要なのか

Node.js の API を見ると、イベントが多いと感じるかもしれません。これは Node.js の文化的な事情もありますが、それだけで説明すると本質を外します。イベントが多いのは、HTTP/2 が本質的に**状態変化を段階的に通知したくなるプロトコル**だからです。

HTTP/1.1 的な単純化されたモデルでは、「リクエストを送る」「レスポンスが返る」という直線的な理解がしやすいです。しかし HTTP/2 では、複数のストリームが同時に進みます。あるストリームではヘッダーが先に来て、別のストリームではデータが進行し、さらに別のストリームでは終了やエラーが発生するかもしれません。接続全体の設定変化や shutdown の兆候も、その途中に割り込みます。

こうした状況を、単一の戻り値や一回きりの完了通知だけで表現するのは難しいです。だから API はイベントやコールバックや非同期反復のような形を取ります。重要なのは、イベントは「実装上たまたまそうした」のではなく、**複数の責務単位で起きる状態変化を利用者に段階的に見せるためのインターフェース**だということです。

私はここで、Event を「通知機構」とだけ理解しないほうがよいと考えています。Event は、HTTP/2 のどの変化を利用者が意識すべきかを示す、抽象化の選択でもあります。何をイベントにし、何を内部処理に閉じ込めるかには設計判断があります。つまり Event も、API 設計が責務分割をどう表しているかの一部なのです。

### Event の責務は何を表すのか

では、Event は何を表しているのでしょうか。大きく分けると、二種類あります。

ひとつは、**接続全体の状態変化**です。セッションが確立した、閉じる方向に入った、設定が変わった、エラーが起きた、といったものです。これらは Session の責務に属します。個別ストリームの問題ではなく、共有基盤の側で起きたことだからです。

もうひとつは、**個別ストリームの状態変化**です。ヘッダーを受け取った、データを読める、個別通信が終わった、途中でリセットされた、といったものです。こちらは Stream に属します。アプリケーションに近い形で観察したいのはこちらのほうです。

この二種類が混ざって見えると、API は急に読みにくくなります。逆に、「このイベントは接続の話か、個別通信の話か」と常に問いながら読むと、かなり整理されます。イベント名の細かな違いより、まず責務の帰属先を見るのです。

これは実務上も有効です。ログや監視でも、何が接続全体の不調を示し、何が個別通信の失敗を示しているのかが分かると、障害の切り分けがやりやすくなります。API 設計を責務分割として読む習慣は、そのまま observability の読み方にもつながります。

### API 設計はどこまでプロトコル構造を反映しているのか

ここで一度、問いを正面から整理します。Node.js の API 設計は、どこまで HTTP/2 のプロトコル構造を反映しているのでしょうか。

結論から言えば、**責務の骨格はかなり反映しているが、表現はそのままではない**、です。

そのままではない、というのは当然です。API は利用者のための抽象化ですから、フレームや状態遷移を一対一対応で見せる必要はありません。むしろ多くは見せないほうがよいです。すべてを露出すると、利用者が毎回プロトコル実装者の視点を背負うことになります。

一方で、骨格はかなり反映しています。接続全体の文脈があり、その上で複数の個別通信が並行進行し、その各所で状態変化が起きる。この構造は隠しきれません。だから Session、Stream、Event という分割は、プロトコルの責務構造を高水準に写像したものとして読むことができます。

ここで注意したいのは、API とプロトコル概念は一対一対応ではないということです。たとえば Frame は Node.js の高水準 API の中心には出てきません。しかし、Frame を見せなくても Stream や Event の設計には Frame 的な状態変化の影が残ります。つまり、**API は概念を削っているのではなく、利用者に必要な単位へ束ね直している**のです。

### Node.js の API から見える HTTP/2 の設計感覚

ここまでをまとめると、Node.js の `http2` API から見える HTTP/2 の設計感覚は次のようになります。

第一に、HTTP/2 は接続共有を前提としたプロトコルであり、接続全体の責務を表す Session が必要になります。これは単なる transport のハンドルではなく、設定、寿命、全体制御の単位です。

第二に、その Session の中で複数の Stream が並行進行します。Stream は個別通信単位であり、ヘッダー、データ、終了、中断など、アプリケーションに比較的近い責務を持ちます。

第三に、これらの責務は直線的な関数呼び出しだけでは表現しにくいため、Event が重要になります。Event は単なる通知ではなく、状態変化をどの粒度で利用者に露出するかという抽象化の結果です。

第四に、API 設計はプロトコル構造をそのまま再現するのではなく、利用者に必要な責務単位へ変換しています。だから API を読むことは、「HTTP/2 の何がどこへ再配置されたか」を読むことでもあります。

この感覚を持って次章へ進むと、なぜ HTTP/2 と非同期処理が相性よく見えるのかを、単なる Node.js の流儀ではなく、状態管理の都合として説明できるようになります。

### 整理ブロック：Session / Stream / Event の対応

| 抽象 | 主に表すもの | 代表的な責務 |
| --- | --- | --- |
| Session | 接続全体の文脈 | 設定、寿命、全体制御、ストリーム生成 |
| Stream | 個別通信単位 | ヘッダー、データ、個別終了、中断 |
| Event | 状態変化の通知面 | 接続や通信の変化を段階的に利用者へ伝える |

### 整理ブロック：接続単位とストリーム単位の違い

| 観点 | 接続単位 | ストリーム単位 |
| --- | --- | --- |
| 関心事 | 共有基盤の管理 | 個別通信の進行 |
| 代表例 | SETTINGS、GOAWAY、全体エラー | HEADERS、DATA、RST\_STREAM |
| API 上の表れ | Session | Stream |
| 失敗時の影響 | 複数通信へ波及し得る | その通信に主に閉じる |

### 整理ブロック：Event をどう読むか

| 見方 | 内容 |
| --- | --- |
| 通知機構として読む | 何が起きたかを利用者へ知らせる仕組み |
| 責務分割として読む | どの状態変化を露出し、どれを内部へ隠すかの設計判断 |
| HTTP/2 の構造として読む | 並行する複数の状態変化を扱う必要の反映 |

### この章で残すべきメンタルモデル

Node.js の `http2` API にある Session・Stream・Event の分離は、単なる流儀ではありません。  
HTTP/2 が持っている接続全体の責務、個別通信の責務、状態変化の通知という三つの面を、高水準の API に写したものです。

Session は共有基盤です。  
Stream は個別通信単位です。  
Event は、その両者で起きる状態変化を利用者に見せる窓です。

API を読むとは、メソッドの一覧を覚えることではありません。  
責務がどこで切られ、どの変化がどの粒度で露出しているかを読むことです。

## まとめ

- Node.js の `http2` API は、HTTP/2 の責務構造を読む教材として有用である
    
- Session は接続全体の責務を表し、Stream は個別通信単位の責務を表す
    
- Event は単なる通知ではなく、状態変化をどの粒度で露出するかという抽象化の結果である
    
- API 設計はプロトコル構造をそのまま再現するのではなく、利用者向けの責務単位へ変換している
    
- Session / Stream / Event の分離を読むことで、HTTP/2 の構造を高水準 API から理解できる
    

次章では、この Event 中心の見え方を踏まえて、HTTP/2 と非同期処理がなぜ相性よく見えるのかを整理します。そこでは速度の話よりも、並行する状態をどう扱うかが中心になります。
