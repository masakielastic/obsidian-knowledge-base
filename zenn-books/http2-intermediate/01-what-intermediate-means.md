## 第1章　入門書の次に、何が見えるようになるべきか

**初学者**「HTTP/2 の入門書を読むと、ストリームがあることや、複数の通信を1本の接続に載せられることは分かります。でも、そこで理解が止まってしまうんです。ライブラリを使う段になると、急に Session や Stream やイベントが出てきて、何をどの粒度で考えればいいのか曖昧になります」

**解説者**「それは自然なつまずきです。入門書は、まずプロトコルの地図を与える本だからです。HTTP/2 が何を導入したのか、何が HTTP/1.1 と違うのかを認識できるようにすることが主な役割です。けれども、中級書ではそこから一歩進めて、その地図がライブラリや API や設計判断の中でどう現れるかを読む必要があります」

**実装者**「実務でも同じです。ログを見てフレーム名が分かるだけでは足りません。どの問題が接続単位の話なのか、どの問題がストリーム単位の話なのかが見えないと、API の使い方も切り分けも曖昧になります」

**解説者**「この章の問いは単純です。入門書の次に、何が見えるようになるべきか。本書はその答えを最初に共有するところから始めます」

### 入門書までで分かっていること

入門書を読み終えた段階で、読者はすでに重要な土台を持っています。HTTP/2 がバイナリフレームを使うこと、1本の接続の上で複数のストリームを並行して扱うこと、主要なフレームにそれぞれ役割があること、そして HTTP/1.1 の head-of-line blocking を軽減する方向で設計されていること。このあたりが見えていれば、入門としては十分です。

私はこの段階を、**認知地図を得た状態**だと考えています。都市の地図を最初に受け取った状態に近いです。駅の名前は分かる。主要な道路も分かる。だが、どこで渋滞が起きやすいのか、物流がどう流れているのか、どの業務がどの施設に集約されているのかまではまだ見えていません。

HTTP/2 の入門でも同じことが起きます。Connection、Stream、Frame という単語は知っていても、それぞれが設計上どんな責務の単位なのかまでは、まだ十分に整理されていないことが多いのです。

ここで大事なのは、入門書の理解を過小評価しないことです。中級に進むとは、入門が間違っていたという意味ではありません。視点が変わるだけです。観察の地図から、利用と設計の地図へ進む。それが中級の役割です。

### 中級で見えるようになるもの

中級で見えるようになるべきものは、プロトコルの事実そのものよりも、**抽象化の形**です。

HTTP/2 ライブラリを使うとき、私たちはフレームを毎回手で組み立てているわけではありません。多くの場合、Session や Stream のようなオブジェクト、あるいはイベントやコールバックや非同期 API を通して HTTP/2 を扱います。ここで重要なのは、ライブラリが複雑さを消しているのではなく、**複雑さを別の形に並べ替えている**という点です。これは本書全体を通じて繰り返し使う見方です。

たとえば、接続全体に関わる問題と、個別の通信に関わる問題は別です。SETTINGS や GOAWAY のように接続単位で効く話と、HEADERS や DATA や RST\_STREAM のようにストリーム単位で効く話は、責務の持ち方が異なります。入門書ではそれを「種類の違い」として学びますが、中級では「設計上の境界」として読み直します。

また、HTTP/2 は非同期処理と相性がよいと言われますが、それも単なる高速化の話ではありません。1本の接続の上で複数のストリームが並行して進み、それぞれが異なるタイミングで状態を変える以上、イベント駆動や非同期 API が自然に見えるのです。つまり中級では、現象を説明するだけでなく、その現象が API の形や責務分割にどう影響するかまで見えるようになる必要があります。

### 利用・設計・仕様の三段階で考える

この本では、HTTP/2 の理解を大きく三段階で捉えます。

第一段階は、**利用の地図**です。ライブラリやランタイムが提供する API を使い、何を利用者が意識し、何をライブラリに任せるのかを把握する段階です。Session と Stream が分かれている理由、イベントがどの責務を表しているか、バックプレッシャーやフロー制御がどのように利用者に見えてくるかを考えます。

第二段階は、**設計の地図**です。Connection、Stream、Frame という区別を、クラス責務やレイヤー分割に接続する段階です。ここでは「プロトコルの概念がそのままクラスになる」とは限りません。むしろ、どの概念をどの責務単位で束ねるか、どこで abstraction を切るかが問題になります。本書が主に扱うのはこの段階です。

第三段階は、**仕様と実装の地図**です。RFC を読み、生のフレーム列を追い、HPACK や状態遷移やエラー処理の細部まで扱う段階です。これは重要ですが、本書の主戦場ではありません。本書はそこへ進む前に、利用と設計の間にある大きな溝を埋める本です。

この三段階を先に意識しておくと、「いま自分はどの地図を見ているのか」を見失いにくくなります。HTTP/2 の議論は、同じ単語を使っていても、利用の話をしているのか、設計の話をしているのか、仕様の話をしているのかで意味が変わりやすいからです。

### 中級と上級は何が違うのか

中級と上級の違いは、知識量の差だけではありません。何を直接扱うかの違いです。

中級では、抽象化されたものを読み解きます。ライブラリの API、イベントモデル、クラス責務、レイヤー分割を通して、HTTP/2 の構造を理解します。ここで問われるのは、便利な API の背後にある設計判断を説明できることです。

上級では、抽象化そのものを疑ったり、分解したりします。フレームの順序、状態遷移、ヘッダー圧縮、フロー制御、エラーコード、実装上の境界条件などを直接扱います。つまり、上級は「ライブラリが何を隠しているか」をより低い層まで降りて確認する段階です。

私はこの違いを、次のように整理すると分かりやすいと思います。

> 入門は認知地図です。  
> 中級は利用と設計の地図です。  
> 上級は仕様と実装の地図です。

この区別は、本書のスコープを守るためにも重要です。本書は RFC の網羅や HPACK の内部動作や生データ読解を主目的にはしません。そこに中途半端に踏み込みすぎると、かえって中級書としての焦点がぼけます。

### HTTP/2 を「分かった」と言うために必要なこと

HTTP/2 を「分かった」と言うとき、何をもってそう言うのか。この本では、その基準を少し引き上げます。

フレーム名を知っているだけでは足りません。多重化できると説明できるだけでも足りません。次のことが言語化できるようになって、初めて中級の入口に立ったと言えます。

ひとつは、HTTP/2 ライブラリが何を隠し、何を利用者に見せているかを説明できることです。複雑さは消えていない、API の形に再配置されている、と言えることが重要です。

もうひとつは、Connection、Stream、Frame の区別を、単なる用語ではなく責務の単位として説明できることです。どこまでが接続単位の関心事で、どこからがストリーム単位の関心事なのかが見えると、API 設計もエラー処理も shutdown も読みやすくなります。

さらに、HTTP/2 と非同期処理の相性を速度論ではなく状態管理として説明できることも大切です。HTTP/2 は、同時に進行する複数の通信と、その状態変化を扱う都合上、非同期やイベント駆動と相性がよい。ここが見えると、Node.js のような API も表面的な作法ではなく構造として理解できます。

最後に、HTTP/2 を低・中・高レイヤーで捉え直せることです。どこまでが基盤ライブラリの仕事で、どこからがミドル層の抽象化で、どこからがアプリケーションや RPC フレームワークの関心事なのか。このレイヤー感覚は、gRPC、Connect、HTTP/3、Sans-IO に進むための土台になります。

### 本書で読者が得る地図

ここまでの話を踏まえると、本書が提供するのは「HTTP/2 の全部」ではありません。提供するのは、**HTTP/2 を抽象化・責務分割・状態管理の教材として読むための地図**です。これは執筆ガイドラインでも中心方針として明示されている通りです。

次章以降では、まず HTTP/2 ライブラリが何を隠し、何を見せるのかを考えます。そこから Node.js の http2 モジュールを題材に API 設計を読み、非同期処理との相性を整理し、クラス責務やレイヤー分割へ進みます。後半ではフロー制御、終了処理、Sans-IO、そして HTTP/3 への橋を扱います。つまり本書全体は、観察中心の理解から、利用・設計・抽象化中心の理解へ視点を移していく構成になっています。

この章の役割は、その移動を読者と合意することにあります。いまから読むのは、HTTP/2 の「答え合わせ」ではありません。HTTP/2 をどういう地図で読み直すか、その基準を共有するための本です。
