## 第7章　フロー制御を「実務で困る仕組み」として理解する

**初学者**「flow control は入門書でも出てきましたが、正直まだ遠い話に感じています。仕様として必要なのは分かるんです。でも、ライブラリを使うだけなら、そこまで意識しなくてもいいのではないかとも思ってしまいます」

**解説者**「その感覚は自然です。flow control は、最初は低レイヤーの細かな規則に見えます。ただ、中級では見方を変えます。flow control は単なる仕様の飾りではありません。データの流れをどこまで進められるかを決める制御面であり、利用者にも『詰まる』『進まない』『待たされる』という形で見えてきます」

**実装者**「実務ではまさにそこです。送っているつもりなのに進まない。特定のストリームだけが遅いように見える。接続全体が重いように感じる。こうした症状の背景に flow control がいることがあります。詳細アルゴリズムを全部知らなくても、どの粒度で何が制御されているかは分かっていたほうがよいです」

**解説者**「この章では、flow control を『ライブラリ内部の難しい仕組み』としてではなく、『実務で困る症状とつながる制御面』として整理します。重要なのは、数式を覚えることではなく、何がどの単位で止まり得るのかを見えるようにすることです」

### flow control を「仕様の細部」とだけ見ると見誤る

HTTP/2 の flow control は、入門段階ではどうしても細かな仕組みに見えます。WINDOW\_UPDATE があり、ウィンドウサイズがあり、送れる量に制約がある。そこまでは分かっても、「それが自分の設計やライブラリ利用にどう関係するのか」が見えにくいのです。

ですが本書の方針では、flow control は単なる細かな仕様ではありません。執筆ガイドラインでも、flow control のアルゴリズム詳細や RFC の数式を主目的にせず、**実務で見える症状と設計上の理由を接続する**ことが重視されています。さらにアウトラインでも、第7章は flow control を「実務で困る仕組み」として理解する章として位置づけられています。

私はここで、flow control を次のように捉えるのがよいと思います。  
**flow control とは、データの流れを完全な自由放任にせず、受け手と送り手の都合を調整する制御面である。**

この見方に変えると、flow control は単なる低レイヤーの特殊知識ではなくなります。なぜ今このデータが進まないのか。なぜある通信だけ遅いのか。なぜ API が「まだ送れない」「待て」と言ってくるのか。こうした疑問の背景として見えてきます。

### なぜ flow control が必要なのか

まず、そもそもなぜ flow control が必要なのかを整理します。理由は単純で、HTTP/2 は 1 本の接続の上で複数のストリームが並行して進むからです。その世界では、送れる側が好きなだけ送り続けると、受ける側や接続全体の資源管理が破綻しやすくなります。

ここで言う資源とは、メモリ、処理能力、バッファ、アプリケーションが消費できる速度などです。相手がまだ読み切れていないのに大量のデータを押し込み続けると、単純に受信側が苦しくなります。しかも HTTP/2 では、複数のストリームが接続を共有しています。ある通信が大量に流れていることが、別の通信の見え方にも影響し得ます。

つまり flow control は、「速いほうが全部勝つ」世界を避けるための仕組みです。受け手の都合や共有接続の都合を無視せず、データの流れを段階的に調整する。そのために送れる量に制限を設け、消費できた分だけ先へ進めるようにします。

ここで重要なのは、flow control が性能低下のための足かせではなく、**多重化された接続を破綻させないための秩序**だということです。HTTP/2 の便利さである多重化は、裏側でこうした制御面を必要とします。便利さは、制御の不在ではなく、制御の上に成り立っています。

### 接続単位とストリーム単位の flow control がある

HTTP/2 の flow control をややこしくしているのは、制御の粒度が一つではないことです。アウトラインでも、本章の主要トピックとして「接続単位とストリーム単位の制御」が最初に挙げられています。

これは本書全体で繰り返してきた構造と同じです。HTTP/2 では、接続全体に関わる責務と、個別ストリームに関わる責務が分かれています。flow control も例外ではありません。

接続単位の flow control は、共有基盤としての接続全体でどれだけデータを流せるかに関わります。これは、同じ接続を共有する複数ストリーム全体に影響する制御です。

一方、ストリーム単位の flow control は、ある個別通信がどれだけ先へ進めるかに関わります。あるストリームが止まっていても、別のストリームは進めるかもしれません。逆に接続全体の都合で、個別ストリームも巻き込まれて止まることがあります。

ここで大事なのは、flow control を「送れるか送れないか」という単純な二値で見ないことです。実際には、「そのストリームとしては送れるのか」「接続全体としては送れるのか」という二層の問いがあります。第5章で見たように、flow control は接続かストリームかの二択ではなく、階層的に考えるべき責務です。

この構造が分かると、「なぜ一部だけ遅いのか」「なぜ全体が重いのか」を切り分けやすくなります。

### WINDOW\_UPDATE は何をしているのか

ここで WINDOW\_UPDATE の役割を、中級者向けの粒度で押さえておきます。詳細な数値規則には入りませんが、概念だけは明確にしておきたいところです。

WINDOW\_UPDATE は、受け手側が「ここまでならさらに受け取れる」と送り手側に伝えるための仕組みです。つまり、単に「窓がある」という静的な話ではなく、**消費に応じて流れを前へ進めるための合図**です。

私はこれを、許容量の回復通知として捉えると分かりやすいと思います。送り手は無制限に送り続けるのではなく、相手からの許容量の範囲で進めます。そして受け手が処理を進められたら、そのぶんを WINDOW\_UPDATE のような形で知らせ、さらに送れるようにします。

この見方の利点は、flow control を「禁止の仕組み」ではなく、「進め方を段階的に調整する仕組み」として見られることです。止めるためだけではありません。適切に前進させるための制御です。

実務で重要なのは、WINDOW\_UPDATE の細かな値を暗記することより、「相手が消費しない限り、自分は思ったように前へ進めないことがある」と理解することです。これは API の書き込み挙動、データ到着の間隔、バックプレッシャーの見え方に直結します。

### flow control は backpressure の感覚として見える

HTTP/2 の flow control を利用者の体感に近づける言葉として、backpressure があります。アウトラインでも、本章の主要トピックに backpressure の感覚が含まれています。

backpressure とは、ざっくり言えば「流れを押し込みすぎないために、下流の都合が上流へ跳ね返ってくること」です。HTTP/2 では、受信側の消費速度や接続全体の制約が、送信側の進み方に影響します。

このとき利用者が見るのは、必ずしも「flow control が働いた」という明示的な表示ではありません。むしろ、「書けるはずなのにすぐには書けない」「データが来るはずなのに間が空く」「ある処理だけ妙に遅い」といった、体感上の詰まりです。

ここが大切です。flow control は低レイヤーの数値更新ですが、利用者には backpressure の感覚として現れます。つまり、本章で覚えるべきなのは数式ではなく、**詰まりは制御面の現れでもある**ということです。

この感覚がないと、HTTP/2 の問題をすべてアプリケーション側のバグやネットワーク遅延だけで説明しようとしてしまいます。もちろん実際にはさまざまな要因がありますが、flow control という制御面がひとつの候補として見えるかどうかで、切り分けの質はかなり変わります。

### どのように「詰まり」として見えるのか

では、flow control は実務でどのような症状として見えるのでしょうか。ここでは典型的な見え方を整理しておきます。

まず、**送信が思ったように進まない**という見え方があります。アプリケーションの側では「書いたつもり」でも、実際には即座に先へ流せるとは限りません。ライブラリやランタイムによって見え方は違いますが、書き込みの待ち、flush の遅れ、あるいは送信可能通知の待ちとして現れることがあります。

次に、**特定ストリームだけ遅い**という見え方があります。接続全体は生きていて、ほかの通信もある程度動いているのに、あるストリームだけ進みが悪い。これはストリーム単位の制御や、そのストリームに対する消費の遅れが関係しているかもしれません。

さらに、**接続全体が重く感じられる**こともあります。これは接続単位の制御面の影響を疑う場面です。HTTP/2 は多重化されているため、共有基盤の詰まりが複数の通信に波及して見えることがあります。

最後に、**終了や失敗と見分けにくい遅さ**もあります。flow control による停止は、必ずしも「壊れた」ことを意味しません。単に今は進めないだけかもしれません。しかし利用者から見ると、遅いことと失敗していることは区別しにくいです。だからこそ、終了処理やタイムアウトや retry を考えるときにも、flow control の存在を頭に置いておく必要があります。

ここでの要点は、flow control は「動いているが進まない」という中途半端な症状を生みやすいことです。完全な成功でも完全な失敗でもなく、制御の都合で流れが止まったり再開したりする。その曖昧さが、実務での理解を難しくします。

### ライブラリ利用者が意識すべき最小限の理解

ここで、本章の中心となる問いに答えます。詳細実装を知らなくても、利用者は何を理解しておくべきなのでしょうか。

第一に、HTTP/2 では無制限に送受信できるわけではないことです。送る側の意志だけでは前へ進みません。相手の消費や接続全体の制御が関わります。

第二に、flow control には接続単位とストリーム単位の二層があることです。したがって「この通信だけ遅いのか」「接続全体が重いのか」を分けて考える必要があります。これは本書全体の Connection / Stream の区別とも一致します。

第三に、flow control は backpressure の感覚として API 利用者にも現れることです。ライブラリが内部で処理してくれるからといって、完全に忘れてよいわけではありません。送信待ち、読み取り待ち、進行の偏りといった形で、利用者のコードや運用判断にも影響します。

第四に、遅いことと壊れていることをすぐに同一視しないことです。flow control による待ちなのか、接続やストリームの失敗なのか、アプリケーション側の処理遅れなのかを切り分ける視点が必要です。

この程度の理解があるだけでも、HTTP/2 の「なんとなく進まない」をかなりましな形で説明できるようになります。本章の目的はそれです。詳細アルゴリズムの習得ではなく、**症状と制御面を結びつける視点**を得ることです。

### flow control をどのレイヤーで考えるべきか

第6章のレイヤーの話に戻ると、flow control は低・中・高のどの層とも無関係ではありません。

低レイヤーでは、window の更新や制御フレームの処理のような仕組みそのものが扱われます。ここでは正しさと整合性が重要です。

中レイヤーでは、その制御面を利用者にどう見せるかが問題になります。送信可能通知、イベント、ストリーム API、書き込み待ち、バッファ制御など、利用可能な抽象へ翻訳するのは中レイヤーの仕事です。第2章から第5章で繰り返してきた「何を隠し、何を露出するか」という問いが、そのままここに現れます。

高レイヤーでは、flow control を意識した運用や設計判断が必要になります。たとえば大量データ送信の扱い、shutdown の設計、タイムアウト、再試行、アプリ側の消費速度などです。高レイヤーは細かな window 値を知らなくてもよいですが、「下流の都合で流れが止まり得る」ことは知っていたほうがよいです。

このように flow control は、低レイヤーの詳細でありながら、中レイヤーの抽象化を通じて、高レイヤーの体感や判断にも影響します。だから中級書で扱う価値があります。これはまさに、本書が HTTP/2 を状態管理と責務分割の教材として理解させる方針と一致しています。

### flow control を学ぶ意味

最後に、この章で flow control を扱う意味を整理しておきます。

HTTP/2 を中級として理解するとは、便利な API の背後にある設計上の理由を言語化できるようになることでした。flow control はその好例です。なぜライブラリが送信待ちや backpressure のような形を持つのか。なぜ多重化された通信で「一部だけ遅い」「全体が重い」という現象が起きるのか。なぜ shutdown や retry やタイムアウトの話が、単純な request/response の完了だけでは済まないのか。これらは flow control を無視すると説明しにくくなります。

私は、flow control を知ることは「低レイヤーに詳しくなること」以上に、「進まないという症状を構造的に読めるようになること」だと思っています。HTTP/2 の実務では、そのほうが価値が高い場面が多いです。

### 整理ブロック：flow control をどう捉えるか

| 見方 | 内容 |
| --- | --- |
| 仕様の細部として見る | window や制御フレームの規則に注目する |
| 制御面として見る | データの流れを送り手と受け手の都合で調整する仕組みと見る |
| 実務の症状として見る | 詰まり、待ち、進行の偏りとして観察する |

### 整理ブロック：接続単位とストリーム単位の違い

| 粒度 | 何が制御されるか | どう見えやすいか |
| --- | --- | --- |
| 接続単位 | 共有基盤として全体で流せる量 | 複数通信にまたがる重さ、全体的な進みの悪さ |
| ストリーム単位 | 個別通信として流せる量 | 特定通信だけの遅さ、個別の進行停止 |

### 整理ブロック：利用者に見える典型症状

| 症状 | 背景として考えられること |
| --- | --- |
| 書いたつもりなのに進まない | 送信側がすぐには前進できない |
| 特定ストリームだけ遅い | ストリーム単位の制御や消費の偏り |
| 接続全体が重い | 共有接続側の制御面の影響 |
| 失敗か待ちか分かりにくい | flow control による停止と異常終了の区別が必要 |

### この章で残すべきメンタルモデル

flow control は、HTTP/2 の細かな仕様の一部ではあります。  
しかし中級者にとって重要なのは、それを「実務で困る制御面」として見ることです。

HTTP/2 では、接続単位とストリーム単位の両方でデータの流れが調整されます。  
その結果、利用者には backpressure や詰まりとして現れます。

つまり flow control とは、  
「なぜ今この通信が思ったように進まないのか」を説明するための、重要な背景知識です。

## まとめ

- flow control は単なる仕様の細部ではなく、データの流れを調整する制御面である
    
- HTTP/2 では接続単位とストリーム単位の両方で flow control が働く
    
- WINDOW\_UPDATE は、消費に応じてさらに進めるための合図として理解できる
    
- 利用者には flow control が backpressure や詰まりとして見えることがある
    
- 中級者として重要なのは、遅さや停止を制御面と結びつけて考えられることである
    

次章では、この制御面の理解を踏まえて、RST\_STREAM と GOAWAY を題材に終了処理を整理します。そこで、個別通信を止めることと接続全体を閉じることが、どれほど違う責務なのかが見えてきます。
