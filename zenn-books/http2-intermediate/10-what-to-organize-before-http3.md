## 第10章　HTTP/3 を学ぶ前に、HTTP/2 で整理しておくこと

**初学者**「HTTP/2 の中級編をここまで読んで、接続とストリーム、イベント、flow control、shutdown、Sans-IO までだいぶ整理できた気がします。でも、その先で HTTP/3 を学ぼうとすると、また全部やり直しになるのではないかと少し不安です」

**解説者**「その不安は自然です。たしかに HTTP/3 では transport が変わりますし、新しく難しくなる部分もあります。ですが、だからといって HTTP/2 の理解が無駄になるわけではありません。むしろ、HTTP/2 で整理しておくべき抽象があるからこそ、HTTP/3 を『完全な新世界』としてではなく、連続した設計の延長として学べます」

**実装者**「実務でも、その見方は大事です。HTTP/3 を学ぶときに transport の違いばかり見てしまうと、逆に共通部分を見失います。ストリーム、状態管理、ヘッダー圧縮、flow control、終了処理の粒度といった観点は、HTTP/2 を通して整理しておいたほうが理解が安定します」

**解説者**「この章では、HTTP/3 の細部には入りません。その前に、HTTP/2 で何を整理しておけばよいのかを確認します。言い換えると、この本で作ってきた地図のうち、何が次の学習へ持ち越せるのかを明確にする章です」

### HTTP/3 は HTTP/2 と無関係な新世界ではない

まず最初に押さえたいのは、HTTP/3 は HTTP/2 と無関係な別世界ではないということです。アウトラインでも、第10章で残したいメンタルモデルとして「HTTP/3 は HTTP/2 と無関係な新世界ではない」「多くの抽象は引き継がれ、transport が入れ替わることで景色が変わる」と明示されています。

HTTP/2 を通じて本書が整理してきたのは、単なるフレーム名や API の断片ではありませんでした。接続全体と個別通信の区別、状態遷移、イベント生成、flow control、終了処理の粒度、レイヤー分業、そして Sans-IO 的な見方です。これらは transport の詳細そのものではなく、**通信プロトコルをどう抽象化し、どう責務分割し、どう状態管理するか**という見方でした。

この意味で、HTTP/2 の学習は HTTP/2 に閉じた知識ではありません。本書が一貫して目指してきたのは、HTTP/2 を「抽象化・責務分割・状態管理の教材」として理解することでした。だからこそ、その理解は HTTP/3 にも持ち込めます。

もちろん、同じではありません。transport が変われば、見え方も難しさも変わります。ただ、その違いを本当に理解するためにも、まず何が連続しているのかを押さえておく必要があります。

### まず持ち越せるのは「ストリームを中心に見る視点」である

HTTP/2 を通して本書が繰り返し強調してきたのは、request/response の直線モデルだけでは足りず、stream と event を中心に見たほうが自然だということでした。第4章では、HTTP/2 と非同期処理の相性を速度論ではなく状態管理として整理しました。

この視点は、HTTP/3 を学ぶ前にも強く役立ちます。なぜなら、HTTP/3 でも個別通信単位としての stream という発想は中心に残るからです。もちろん transport の違いによって挙動や前提は変わりますが、「複数の通信が並行して進み、それぞれが独自の状態を持つ」という見方自体は捨てなくてよいのです。

私はここで、HTTP/2 の学習成果として最も大きいのは、「通信を一本の直線ではなく、複数の進行中の状態として見る感覚」だと思っています。これがあると、HTTP/3 に進んだときも、単に新しい用語を覚えるのではなく、どの抽象が継承され、どの部分だけが transport 由来で変わるのかを見分けやすくなります。

### Connection / Stream / Event の整理は今後も効く

第3章では、Node.js の `http2` モジュールを題材に、Session / Stream / Event という API 分割が HTTP/2 の責務構造をかなり素直に映していると整理しました。第5章では、それをクラス責務の設計へつなげました。

このとき本当に大事だったのは、特定 API の名前ではありません。接続全体の責務と、個別通信の責務と、状態変化の通知面を分けて考えることでした。

HTTP/3 に進んでも、この整理は無駄になりません。むしろ「何が接続全体に属し、何が個別ストリームに属し、何がイベントとして上位へ露出すべきか」という問いは、より重要になります。transport が変わると、同じ単語でも背後の意味合いが少し変わることはあります。しかし、責務を粒度ごとに切って考えるという学習姿勢はそのまま持っていけます。

この本が単なる HTTP/2 の使い方本ではなく、中級編として意味を持つのはここです。概念を API に結びつけ、API を責務分割として読み、さらにレイヤー分業と Sans-IO へつなげてきたからこそ、HTTP/3 を学ぶ足場になります。

### header compression や flow control も「考え方」は引き継げる

本書では HPACK の符号化詳細には深入りせず、flow control もアルゴリズム詳細ではなく「実務で困る制御面」として扱いました。これは執筆ガイドラインでも意図的なスコープとして明示されています。

この方針は、HTTP/3 への橋としても有効です。なぜなら、HTTP/3 を学ぶときにも重要なのは、いきなり圧縮方式や制御規則の細部へ潜ることではなく、**なぜこうした制御面が必要になるのか**を先に理解しておくことだからです。

たとえば flow control について、本書では「接続単位とストリーム単位の両方で流れが調整され、利用者には backpressure や詰まりとして見える」と整理しました。  
この見方は、HTTP/3 を学ぶときにも役立ちます。細部が変わっても、「多重化された通信では制御面が必要であり、それは利用者の体感にも出る」という理解は持ち越せます。

ヘッダー圧縮についても同じです。本書では深入りしませんでしたが、「ヘッダー表現は単なる文字列の受け渡しではなく、プロトコルの効率化と状態管理に関わる」と理解しておくこと自体が、次の学習の地ならしになります。

つまり、HTTP/2 で細部を全部覚えなくてもよい代わりに、**制御や圧縮を設計上の必然として見る感覚**は持ち帰るべきです。それが HTTP/3 でも効きます。

### 終了処理の粒度という発想は特に重要である

第8章では、RST\_STREAM と GOAWAY を通して、HTTP/2 における終了処理は階層的であり、個別通信を止めることと接続全体を終えることは別だと整理しました。

私は、この「終了処理の粒度」という発想は、とくに HTTP/3 へ持っていく価値が高いと思っています。なぜなら、transport が変わっても、複数の通信単位をどう止め、共有基盤をどう閉じるかという問題は消えないからです。

HTTP/2 を学ぶ前は、止めることを一種類だと思っていたかもしれません。しかし本書を通じて、stop には複数の粒度があり、それぞれで retry や recovery や graceful shutdown の判断が変わることを見てきました。これは今後、どんなプロトコルを学ぶときにも役立つ視点です。

つまり、本書で身につけてほしかったのは「RST\_STREAM と GOAWAY の名前」よりも、「どの責務単位の寿命が変わったのかを見分ける目」です。この目は HTTP/3 を学ぶときにも重要です。

### transport が変わると何が新しく難しくなるのか

ここまで連続性を強調してきましたが、もちろん HTTP/3 では新しく難しくなることもあります。アウトラインでも、第10章は「何が引き継がれ、何が transport の違いで変わるか」を整理する章として設計されています。

ただし、本章ではその詳細に踏み込みません。ここで読者に残したいのは、「新しさはどこから来るのか」を雑にでも言語化できることです。

その中心にあるのは、やはり transport の違いです。HTTP/2 では、共有基盤の性質が API 設計や状態管理の見え方にかなり影響していました。HTTP/3 では、その基盤が変わることで、同じ stream や flow control や shutdown という言葉でも、背景となる制約や設計判断の景色が変わります。

ここで大事なのは、transport の違いを「全部やり直し」として受け取らないことです。むしろ、「抽象は似ているが、その抽象を支える基盤が変わったので、難しさの出どころがずれる」と理解するほうが中級者らしい見方です。

この理解があると、HTTP/3 を学ぶときも、新しい概念と古い概念を混ぜずに整理しやすくなります。何が継承され、何が transport の交代によって再設計されているのかを見分けられるからです。

### HTTP/2 中級編の学びをどう持ち込むか

ここで、本書全体の学びを HTTP/3 へ持ち込むときの見方を整理しておきます。

第一に、HTTP/2 を単なる過去の規格知識として捨てないことです。本書が重視してきたのは、HTTP/2 そのものの網羅ではなく、設計判断の地図でした。だから、その地図は次にも持っていけます。

第二に、HTTP/3 を学ぶときも、まず API やライブラリの表面だけでなく、何を隠し、何を見せ、どの責務をどこへ置いているかを見ることです。これは第2章以来の一貫した視点でした。

第三に、低・中・高レイヤーの分業を意識することです。どこまでが共通基盤で、どこからが利用モデルで、どこからがアプリケーションの意味なのか。このレイヤー感覚があると、HTTP/3 の学習でも迷いにくくなります。第6章の地図はそのためにありました。

第四に、Sans-IO 的な見方を持つことです。I/O の違いに目を奪われすぎず、状態遷移とイベント生成の核はどこにあるのかを見る。この視点があると、新しい transport を学ぶときも中心を見失いにくくなります。

### 何を整理しておけば、HTTP/3 を学びやすいのか

では、HTTP/3 を学ぶ前に、HTTP/2 で最低限どこまで整理しておけばよいのでしょうか。本書の到達点に引きつけて言えば、次の四つです。

ひとつ目は、**ストリーム中心の見方**です。request/response の直線モデルだけでなく、複数の通信が並行する状態としてプロトコルを見る感覚です。

ふたつ目は、**接続単位と個別通信単位の責務差**です。共有基盤の管理と、個別通信の進行・終了を分けて考えることです。

みっつ目は、**制御面と終了面を設計として見ること**です。flow control や shutdown を細かな仕様ではなく、共有接続を成立させるための制御として理解することです。

よっつ目は、**I/O と状態遷移を分けて見る感覚**です。これは Sans-IO の入口として整理したものでした。新しい transport を学ぶときに、とても効きます。

この四つが整理できていれば、HTTP/3 を学ぶときに「何もかも新しい」とは感じにくくなります。むしろ、「ここは持ち越せる」「ここは transport の違いで新しくなる」と分けて考えられるようになります。

### この本の終わり方

本書のアウトラインでは、第10章はシリーズ全体の出口を作る章とされています。

私は、その出口は単に「次は HTTP/3 を読みましょう」という案内では足りないと思っています。この本の終わり方として大事なのは、HTTP/2 を通して何を学んだのかを、もう一度正確に言い直すことです。

本書は、HTTP/2 の仕様詳細を網羅する本ではありませんでした。ライブラリ利用、非同期処理、責務分割、レイヤー設計という観点から、HTTP/2 を読み直す本でした。つまり、HTTP/2 の知識を増やす本というより、**プロトコルを見る目を鍛える本**だったのです。

だから本書の終わりは、HTTP/2 の完了ではありません。次のプロトコルや次の層を学ぶための足場ができた、という終わり方になります。HTTP/3 はその最も自然な次の題材です。

### この章で残すべきメンタルモデル

HTTP/3 を学ぶ前に大切なのは、HTTP/2 を「過去の規格」として終わらせないことです。  
本書で整理してきたのは、ストリーム、状態管理、責務分割、制御面、終了面、そして I/O とプロトコル核の分離でした。

これらは HTTP/2 だけの知識ではありません。  
通信プロトコルを設計として読むための地図です。

だから HTTP/3 は、HTTP/2 と無関係な新世界ではありません。  
多くの抽象は引き継がれ、transport が変わることで新しい難しさが前景化します。  
HTTP/2 を通してその地図を持てたこと自体が、次へ進む準備になります。

## まとめ

- HTTP/3 は HTTP/2 と無関係な別世界ではなく、多くの抽象を引き継ぎつつ transport が変わることで景色が変わる
    
- HTTP/2 で整理しておくべきなのは、ストリーム中心の見方、責務分割、制御面、終了面、Sans-IO 的な見方である
    
- flow control や shutdown を設計上の必然として見る感覚は、HTTP/3 学習にも持ち越せる
    
- 新しく難しくなるのは主に transport の違いから来るが、共通抽象があるからこそ差分として理解できる
    
- 本書の価値は HTTP/2 の網羅ではなく、プロトコルを抽象化・責務分割・状態管理の教材として読む目を作ることにある
    

これで本書の本文は終わりです。次に進むなら、上級編として HPACK、RFC 読解、生データ、Sans-IO の具体設計へ進む道と、HTTP/3 入門・中級へ進む道の二つがあります。どちらに進むとしても、この本で作った地図は土台として残ります。
