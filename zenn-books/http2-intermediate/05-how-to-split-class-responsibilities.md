## 第5章　クラス責務をどう切るか

**初学者**「ここまで読んで、Connection と Stream を分けて考える必要があることは分かってきました。でも、実装やライブラリ設計を考えると、今度は別の迷いが出てきます。Frame の処理はどこに置くべきなのか、エラーや shutdown は誰が持つべきなのか、flow control は接続側なのかストリーム側なのか。全部が関係していて、境界が曖昧に見えるんです」

**解説者**「その迷いは自然です。HTTP/2 は責務をきれいに完全分離できる題材ではありません。接続単位とストリーム単位は分けたいのに、実際には互いに影響し合います。だから大事なのは、責務が混ざらないようにしつつ、必要な連携はどこで受けるかを決めることです」

**実装者**「実務でも、ここを曖昧にするとすぐにつらくなります。接続全体の shutdown の話が個別ストリームのコードに散ったり、フレーム処理とアプリ向けイベント生成が一体化したり、flow control の都合がどこからでも触れるようになったりします」

**解説者**「この章では、HTTP/2 を題材にクラス責務の切り方を整理します。正解のクラス図を一つ示すのではなく、どの責務をどの粒度で持たせると設計が崩れにくいか、その判断の筋道を見ていきます」

### プロトコル概念とクラス責務は一対一ではない

最初に、ひとつ大事な前提を置いておきます。HTTP/2 の概念区分と、クラス責務は一対一対応ではありません。これは本章で最も重要な注意点です。アウトラインでも、「プロトコル概念とクラス責務は一対一ではない」「ただし、概念区分は責務設計の重要な手がかりになる」と整理されています。

たとえば、HTTP/2 には Connection、Stream、Frame という区別があります。これをそのまま `Connection` クラス、`Stream` クラス、`Frame` クラスにすれば整理できそうに見えます。しかし実際には、それだけでは足りません。

なぜなら、Frame は「責務の単位」というより、しばしば**表現の単位**だからです。HEADERS や DATA や WINDOW\_UPDATE は、たしかにフレームという形で運ばれますが、利用者が考えたい責務はそれよりひとつ上にあります。あるストリームのヘッダー処理、接続全体の flow control、終了処理、イベント通知といった責務は、単一のフレームオブジェクトだけでは表しきれません。

逆に Connection や Stream も、そのままひとつの巨大クラスに押し込めばよいわけではありません。接続単位の責務といっても、ストリーム管理、設定反映、shutdown 制御、全体エラー、ログ出力などがあり、それぞれ性質が違います。つまり、HTTP/2 の概念は責務を切るための地図にはなりますが、そのまま完成したクラス図にはなりません。

私はここで、次のように考えるのがよいと思います。  
**プロトコル概念は境界の候補を示す。クラス責務は、その境界を実装上どの粒度で安定させるかを決める。**

この順番を逆にすると危険です。クラス構造の都合で HTTP/2 の境界を忘れると、あとで接続単位とストリーム単位が混ざります。反対に、概念をそのままクラスに写そうとすると、今度は責務が粗すぎたり細かすぎたりします。

### 最優先の境界は Connection と Stream である

HTTP/2 の責務設計で、最初に守るべき境界は Connection と Stream の区別です。これは執筆ガイドラインでも、最重要品質基準のひとつとして明示されています。接続単位の責務とストリーム単位の責務を区別できることが、この本全体の中心だからです。

Connection 側で持つべきものは、接続全体にかかる関心事です。たとえば、SETTINGS の反映、GOAWAY の扱い、接続寿命の管理、新しいストリームの生成条件、接続単位の flow control、全体エラーの扱いなどです。これらは特定のひとつの通信に閉じません。共有基盤としての接続が持つべき責務です。

一方で Stream 側で持つべきものは、個別通信の進行に関する責務です。たとえば、ヘッダー受信、データ受信、個別ストリームの状態遷移、RST\_STREAM による中断、アプリケーションへの通知などです。これらは、そのストリームがどう進み、どう終わるかという話です。

この境界が曖昧になると、設計はすぐ崩れます。あるストリームの失敗をきっかけに接続全体を不用意に壊してしまったり、逆に接続全体の shutdown 状態を各ストリームが勝手に判断し始めたりします。HTTP/2 は 1 本の接続に複数の通信が載るからこそ、**個別責務と共有責務の切り分け**が最優先になります。

私はこの境界を、実装上の利便性より優先して守るべきだと考えています。多少参照が増えても、多少メソッドが分散しても、Connection と Stream の責務が混ざらないことのほうが長期的には重要です。

### Frame の parse / serialize はどこに置くべきか

次に迷いやすいのが、Frame の parse / serialize をどこに置くかです。ここで避けたいのは、Connection や Stream が直接すべてのフレームバイト列を読み書きする設計です。それをやると、通信の意味とバイト列処理が密結合になります。

私はこの責務は、基本的には**低レイヤーの専用責務**として切り出すべきだと考えています。具体的には、frame codec、parser、serializer のような層です。理由は単純で、フレームのエンコード・デコードは「何を意味するか」よりも先に、「どう読めるか」「どう書けるか」の責務だからです。

たとえば HEADERS フレームを受け取ったとき、バイト列からフレーム種類・長さ・フラグ・ストリーム ID を読むところまでは低レイヤーです。その先で「これはどのストリームに配送すべきか」「END\_STREAM が立っているので状態をどう変えるか」「アプリケーションへ何を通知するか」といった判断は、Connection や Stream、あるいはその中間の dispatcher 的責務が担うべきです。

この分離を守ると、parse / serialize の責務は比較的安定します。フレームをどう読むかはプロトコル仕様に強く依存し、アプリ設計にはあまり依存しません。一方、受け取ったフレームをどう意味づけし、どこへ配送するかはライブラリの抽象化や API 設計に依存します。この二つを分けておくと、仕様の変化と抽象化の変化を別々に扱いやすくなります。

ここで言いたいのは、「Frame クラスを作れ」ということではありません。そうではなく、**フレーム処理は意味処理と混ぜないほうがよい**ということです。Frame は責務の中心ではなく、責務を受け渡すための素材に近いのです。

### dispatch の責務はどこにあるのか

フレーム処理を切り出すと、次に必要になるのが dispatch です。つまり、受け取ったフレームやイベントを、接続全体の処理へ回すのか、特定ストリームの処理へ回すのかを決める責務です。

これは Connection の中に置くこともできますし、別の dispatcher 的なコンポーネントとして置くこともできます。どちらでもよいのですが、重要なのは、**配送判断は parse 責務でも application 責務でもない**と明確に意識することです。

なぜなら、配送判断は HTTP/2 の多重化そのものに関わるからです。どのストリーム ID に属するフレームか、接続単位で処理すべき制御フレームか、すでに閉じたストリームに対するものか、今の接続状態で受理できるか。こうした判断は、低レイヤーの bytes 処理でもなければ、アプリ側の business logic でもありません。まさにプロトコルの中層にあたる責務です。

この中層を意識しないと、Connection が何でも知っている巨大クラスになりやすいです。逆に Stream 側へ押し込みすぎると、接続全体の状態をストリームが勝手に読みに行く不安定な構造になりがちです。dispatch は、Connection と Stream の間をつなぐが、どちらにも飲み込まれない責務として扱うと整理しやすくなります。

### Error と shutdown は誰が持つべきか

HTTP/2 の設計で難しいのは、失敗と終了に複数の粒度があることです。これはアウトラインでも第8章の中心テーマとして後で扱いますが、責務設計の段階でもすでに重要です。

まず個別ストリームの失敗があります。RST\_STREAM による中断や、特定通信だけの異常終了です。これは基本的に Stream 側の責務です。そのストリームの状態遷移として処理し、上位へ通知するのが自然です。

一方で接続全体の終了があります。GOAWAY、全体エラー、graceful shutdown のような話です。これは Connection 側の責務です。新しいストリーム生成を止めるか、既存ストリームをどう扱うか、どの時点で接続寿命を終えるかは共有基盤の判断だからです。

ここで危険なのは、Error と shutdown を「例外を投げる場所」の問題としてだけ捉えることです。そうではありません。HTTP/2 における Error と shutdown は、**どの粒度の責務が終わるのか**という設計問題です。個別通信だけを止めるのか、接続全体を止めるのか。ここが曖昧だと、ライブラリ利用者に見える API も曖昧になります。

私はこの点で、Connection が持つべきものは「接続単位の終了方針」であり、Stream が持つべきものは「個別通信単位の終了状態」だと整理すると分かりやすいと思います。Error handling はその粒度に沿って設計されるべきです。

### Flow control はどこが持つべきか

flow control は、この章で最も「混ざりやすい」責務です。なぜなら、HTTP/2 には接続単位の flow control とストリーム単位の flow control の両方があるからです。したがって、これを完全にどちらか一方へ押し込むのは無理があります。

ここで大切なのは、flow control を「単なる送信可否フラグ」として扱わないことです。flow control は、データの流れを調整する制御面であり、接続共有と個別通信の両方にまたがっています。執筆ガイドラインでも、この本は flow control を単なる細かな仕様ではなく、利用者の体感にも影響する制御面として理解させる方針を取っています。

設計としては、接続単位の window 管理と、ストリーム単位の window 管理を分けるべきです。ただし、それらをどう組み合わせて「今このストリームでどこまで送れるか」を判断するかは中層の責務になります。つまり flow control も、低レイヤーの数値更新だけでは足りず、Connection と Stream の境界をまたぐ coordination が必要です。

ここを Stream の中に全部押し込むと、接続全体の容量を各ストリームが勝手に見に行く構造になりやすいです。逆に Connection の中に全部押し込むと、個別ストリームの事情が見えなくなります。したがって、flow control は「接続側の責務」と「ストリーム側の責務」と「両者を調停する責務」に分けて考えるのが安定します。

この章ではアルゴリズムの詳細には入りません。ただ、中級者として重要なのは、flow control が**接続かストリームかの二択ではない**と分かることです。多重化されたプロトコルだからこそ、制御面も階層的になるのです。

### Logging / observability は誰の責務か

HTTP/2 の設計では、ログや観測性も脇役ではありません。これを単なる付加機能と考えると、あとで troubleshooting が難しくなります。アウトラインでも logging / observability は責務候補として挙げられています。

ここでのポイントは、ログ出力を Connection や Stream の本体責務に埋め込みすぎないことです。もちろん、それぞれが観測に必要な情報を持っているのは事実です。接続 ID、ストリーム ID、状態遷移、フレーム種別、終了理由などは、それぞれの責務の中でしか分かりません。

しかし、「どこで何を記録するか」という判断は、プロトコル処理そのものとは少し別です。私は observability は、プロトコル責務に寄り添うが、その本体にはなりきらない横断的責務として扱うのがよいと思います。イベント発火や状態遷移通知の地点で観測情報を採れるようにし、出力形式や収集方法は分離する。このほうが本体設計を汚しにくいです。

ここでも第3章で見た Event の考え方が役に立ちます。Event は状態変化を利用者に見せるだけでなく、観測のフックにもなります。すると observability は、Connection や Stream の中に `print` を散らす話ではなく、**責務境界で起きたことをどう外へ観測可能にするか**という設計問題になります。

### Application boundary はどこで切るべきか

最後に、アプリケーションとの境界です。これは HTTP/2 ライブラリ設計でとても重要です。プロトコル実装がどこまで面倒を見るのか、どこから先をアプリケーションや RPC フレームワークに渡すのかを決める必要があります。

私はここで、アプリケーション境界は「意味の確定点」で切るのがよいと考えています。つまり、ヘッダーがどう届いたか、データがどのフレームに分かれていたか、WINDOW\_UPDATE がいつ来たかといった低レイヤーの事情を、そのままアプリに渡しすぎないことです。

アプリケーションが主に知りたいのは、「このストリームで要求が始まった」「ヘッダーを受け取った」「ボディを読める」「この通信は終わった」「この理由で中断された」といった、個別通信単位の出来事です。だからアプリ境界は、多くの場合 Stream か、その少し上の request/response abstraction の地点になります。

一方で、GOAWAY や接続 shutdown のような接続全体の出来事は、アプリ全体の受け付け方針に影響することがあります。このため、アプリ境界を切るときも、接続単位の通知が完全に不要になるわけではありません。ここでも、個別通信と共有基盤の二層構造がそのまま現れます。

つまり application boundary は、「HTTP/2 の事情をどこまで上へ見せるか」という問題です。見せすぎると上位がプロトコル実装を再開してしまいます。隠しすぎると shutdown や backpressure や retry 判断に必要な情報が消えます。このバランスを取ることが、ライブラリ設計の中核になります。

### 「何を隠し、何を露出するか」で責務を切る

ここまでをひとつの基準でまとめると、HTTP/2 のクラス責務設計は、**何を隠し、何を露出するか**で考えると整理しやすいです。

フレームの bytes 処理はなるべく隠す。  
接続単位とストリーム単位の違いは明確に露出する。  
flow control の詳細アルゴリズムは隠しても、送受信が止まり得るという制御面は露出する。  
個別失敗と接続終了の粒度差は露出する。  
アプリケーションには意味のあるイベントを渡しつつ、フレーム列そのものは隠す。

こう考えると、クラス責務は単なる分担表ではなく、抽象化の設計そのものだと分かります。HTTP/2 はこの点でとても教育的です。便利な API を作るには、まず責務境界をはっきりさせる必要があるからです。

### 整理ブロック：主要な責務と置き場所の考え方

| 責務 | 主な置き場所 | 理由 |
| --- | --- | --- |
| 接続全体の寿命・設定・新規作成制御 | Connection | 個別通信に閉じない共有基盤だから |
| 個別通信の進行・終了・中断 | Stream | ストリーム単位で意味が完結するから |
| Frame の parse / serialize | 低レイヤーの codec / parser | bytes 処理と意味処理を分けたいから |
| フレームの配送判断 | 中層の dispatch / routing | parse と application の中間責務だから |
| 接続終了方針 | Connection | GOAWAY や shutdown は接続全体の問題だから |
| 個別通信の終了状態 | Stream | RST\_STREAM などは個別通信の問題だから |
| 観測フック | 横断的責務 | 本体責務に埋め込みすぎると汚れやすいから |

### 整理ブロック：混ざりやすい責務

| 責務 | なぜ混ざりやすいか | どう考えるとよいか |
| --- | --- | --- |
| Flow control | 接続単位とストリーム単位の両方がある | 二層管理と調停責務に分ける |
| Error / shutdown | 個別失敗と全体終了が似て見える | どの粒度が終わるのかで切る |
| Event 生成 | 低レイヤーの事実と上位向け意味づけが混ざる | 状態変化の通知面として中層で整理する |
| logging / observability | どこでも出せそうに見える | 責務境界で観測可能にする |

### 整理ブロック：application boundary の見方

| 見せるもの | 隠すもの |
| --- | --- |
| ストリーム単位の開始、進行、終了、中断 | フレーム単位の細かな手続き |
| 接続 shutdown のような運用判断に必要な通知 | bytes 処理や細かな parse 詳細 |
| backpressure や送受信停止の存在 | flow control の低レベル計算過程 |

### この章で残すべきメンタルモデル

HTTP/2 の概念区分は、そのままクラス図にはなりません。  
しかし、Connection と Stream の区別は、責務設計の最重要境界です。

Frame の bytes 処理は低レイヤーへ寄せる。  
接続全体の責務と個別通信の責務は分ける。  
flow control や shutdown のような階層的な問題は、二択ではなく中層の調停も含めて考える。

つまり、HTTP/2 のクラス責務設計とは、  
概念をそのまま写すことではなく、抽象化が崩れない境界を選ぶことです。

## まとめ

- HTTP/2 の概念区分とクラス責務は一対一ではないが、概念区分は重要な手がかりになる
    
- 最優先で守るべき境界は Connection と Stream の区別である
    
- Frame の parse / serialize は bytes 処理として低レイヤーへ寄せ、意味処理と分けるべきである
    
- flow control、error、shutdown は階層的な責務であり、単純に一箇所へ押し込まないほうがよい
    
- application boundary は、HTTP/2 の事情を見せすぎず隠しすぎない地点で切る必要がある
    

次章では、この責務分割の視点をさらに引き上げて、低・中・高レイヤーという大きな分業で HTTP/2 を見直します。そこでは、ライブラリ、ミドル層、アプリケーションがどこで役割を分けるべきかが見えてきます。
