
## 第4章　非同期処理と HTTP/2 はなぜ相性がよいのか

初学者「HTTP/2 の話になると、非同期処理やイベント駆動の話がよく出てきますよね。でも、正直に言うと、そこがまだ腑に落ちていません。HTTP/2 は速いから非同期と相性がよい、という程度の理解で止まっています」

解説者「その理解は入口としては悪くありませんが、中心を外しています。HTTP/2 と非同期処理の相性がよいのは、単に速いからではありません。1本の接続の上で複数の通信が並行して進み、しかもそれぞれが別々のタイミングで状態を変えるからです」

実装者「実務でも、そこを取り違えると混乱します。『非同期にすると速い』という話だけで考えると、なぜ Session と Stream が分かれ、なぜイベントが多く、なぜ shutdown や flow control が扱いにくく見えるのかが説明できません」

解説者「この章では、HTTP/2 が同期的な request/response の発想だけでは理解しにくい理由を整理します。非同期処理を、性能の小技ではなく、並行する状態を扱うための表現として読み直していきます」

### 非同期処理を速度論だけで見ると何を見失うのか

HTTP/2 と非同期処理の関係を説明するとき、最初に出てきやすいのは性能の話です。確かに、同期的に一つずつ待ち合わせるより、複数の処理を並行して進められるほうが、待ち時間を有効に使いやすいことは多いです。ですが、私はこの説明だけでは不十分だと考えています。

なぜなら、HTTP/2 で非同期処理が自然に見える理由は、単なる高速化ではなく、**同時に進行する複数の状態を扱わなければならない**ことにあるからです。これは執筆ガイドラインでも重視されている観点であり、「HTTP/2 と非同期処理の相性を、速度論ではなく状態管理の観点から言語化できる」ことが本書の到達点の一つとして明示されています。

同期処理の発想は、一本の手続きが順番に進む状況に向いています。ある操作をして、その結果を待ち、次の操作へ進む。HTTP/1.1 の単純化された理解では、この発想でもかなりの範囲を説明できます。1リクエストを送り、1レスポンスを受け取り、それが終わってから次へ進む、という線形モデルが頭に入りやすいからです。

しかし HTTP/2 では、一本の接続の上に複数のストリームが並行して存在します。あるストリームではヘッダーを受け取っている最中かもしれない。別のストリームではデータ送信が詰まっているかもしれない。さらに別のストリームでは終了通知が来るかもしれない。その途中で接続全体の設定や shutdown の方向も変わり得ます。こうなると、世界の見え方は「一つ終わってから次へ」ではなく、「複数の状態が同時に進んでいる」に変わります。

HTTP/2 と非同期処理の相性は、まさにこの世界観の一致にあります。

### request/response 脳では何が苦しくなるのか

ここで少し強い言い方をします。HTTP/2 を理解するときに最初に苦しくなるのは、**request/response だけで世界を捉えようとする癖**です。

もちろん、HTTP/2 でもアプリケーションの表面では request と response が重要です。けれども、プロトコルの内部ではそれだけでは足りません。1本の接続に複数の request/response が同居し、それぞれが別々のペースで進行するからです。しかも、それらは独立しきっていません。接続全体のフロー制御、設定、寿命、終了処理を共有しています。

この構造を、完全に同期的な直線モデルで捉えようとすると、いくつかの無理が出ます。どの request に対して今のイベントが起きたのか。接続全体の問題なのか、個別の通信の問題なのか。ある通信を止めたいのか、接続全体を閉じたいのか。こうした問いが、単純な call-and-return の流れからははみ出してきます。

第3章で見た Session と Stream の分離は、まさにこの苦しさへの答えでした。Session は接続全体の責務を表し、Stream は個別通信の責務を表します。そして Event は、それぞれの責務単位で起きる状態変化を段階的に知らせる仕組みでした。つまり、非同期処理が重要に見えるのは、Node.js がたまたまそういう文化だからではなく、**HTTP/2 の構造が request/response だけでは収まりきらない**からです。

私はここで、「request/response 脳」と「stream/event 脳」を分けて考えると整理しやすいと思っています。前者は一往復の完了を中心に世界を見る発想です。後者は、個別通信と接続全体の状態変化を並行して追う発想です。HTTP/2 は後者のほうが自然に見えるプロトコルです。

### HTTP/2 が自然にイベント駆動へ寄る理由

では、なぜ HTTP/2 ではイベント駆動が自然に見えるのでしょうか。理由は単純で、重要な変化が一回きりの完了通知では足りないからです。

あるストリームは、まずヘッダーを受け取ります。その後でデータが何回かに分かれて届くかもしれません。途中でフロー制御の影響を受けるかもしれません。最後に正常終了するか、あるいは途中でリセットされるかもしれません。これだけでも、一つの「通信」が複数の段階を持つことが分かります。

さらに、接続全体にも状態があります。新しいストリームを開けるのか。設定は変わったか。接続は閉じる方向に入ったか。こうした情報は、個別通信の完了とは独立して進行します。

このように、HTTP/2 では重要な出来事が段階的かつ並行的に発生します。そのため API は、戻り値一発で全体を表現するよりも、イベント、コールバック、Promise、非同期ストリームなどを組み合わせた形になりやすいのです。これが「HTTP/2 と非同期処理は相性がよい」と感じられる本当の理由です。相性がよいというより、**そう表現したほうが構造に忠実**なのです。

ここで強調したいのは、非同期処理は飾りではないということです。HTTP/2 の構造を隠すための見た目でもありません。むしろ、接続共有と多重化の結果として生じる状態変化を、利用者が無理なく扱うための表現です。

### 並行進行とは、同時実行のことではない

ここでひとつ誤解を避けておきます。HTTP/2 の説明で「複数のストリームが同時に進む」と言うと、すぐに CPU の並列実行やスレッドの話を連想しがちです。ですが、本章で言いたい「並行進行」は、まず **状態管理の上で同時に扱う必要がある** という意味です。

これは実行モデルの詳細と同じではありません。シングルスレッドのイベントループでも、複数のストリームの状態を交互に進めることはできます。逆にマルチスレッドであっても、接続全体とストリーム単位の責務をきちんと分けていなければ整理は崩れます。

重要なのは、「複数のものが一度に計算されるか」よりも、「複数のものが未完了のまま存在し、それぞれが別々に変化し得るか」です。HTTP/2 ではそれが起きます。だから非同期処理やイベント駆動が自然に見えます。ここを性能やスレッドの話だけにすると、本質がずれます。

この観点は、中級で特に重要です。各言語の async 機構や event loop の実装差分に深入りする前に、HTTP/2 が要求しているのは「並行する状態を扱う心構え」であると押さえるほうが、学習の足場として安定します。アウトラインでも、本章は言語ランタイムごとの詳細比較ではなく、状態変化とイベントループとの親和性を軽く整理する章として位置づけられています。

### 非同期処理は「待ち方」を整理する技術でもある

HTTP/2 と非同期処理の関係を考えるとき、もう一つ大切なのは、非同期処理が「同時にたくさん動かす技術」であるだけでなく、**待ち方を整理する技術**でもあることです。

HTTP/2 では、いろいろな待ちが発生します。相手からヘッダーが来るのを待つ。データが届くのを待つ。送信可能になるのを待つ。接続全体の終了方向を待つ。個別ストリームの終端を待つ。これらの待ちは、すべて同じ種類ではありません。

同期的な設計では、ひとつの待ちが全体を止めやすくなります。しかし HTTP/2 では、あるストリームで待ちが発生しても、別のストリームは進めることがあるし、接続全体として観察すべき状態変化もあります。すると、「待つ」という行為を一本化するより、待ちを責務ごとに分解して扱えるほうが自然です。

この意味で、非同期処理は HTTP/2 の多重化を生かすためだけではなく、**待ちの粒度を接続単位・ストリーム単位・イベント単位に分解するための表現**でもあります。私はこれをかなり重要な見方だと思っています。速度だけでなく、「どこで何を待っているのか」が見えるようになるからです。

### イベントループとの親和性はどこから来るのか

HTTP/2 とイベントループの相性がよいと言われるのも、同じ理由で説明できます。イベントループは、何かが完了するたびに順次処理を進める仕組みです。このモデルは、複数のストリームや接続状態の変化が断続的に発生する HTTP/2 と噛み合いやすいです。

あるストリームでヘッダー受信のイベントが来たら処理する。別のストリームでデータが届いたら処理する。接続全体で shutdown の兆候が出たら新規作成を止める。こうした振る舞いは、イベントが発生するたびに適切な責務単位へ処理を配るモデルと相性がよいです。

もちろん、イベントループ以外の実装モデルでも HTTP/2 は扱えます。本章の主張は「HTTP/2 はイベントループでしか実装できない」ではありません。そうではなく、「HTTP/2 が持つ段階的・並行的な状態変化は、イベントループ的な発想と自然に噛み合う」ということです。ここを押さえると、async/await であれ callback であれ、表現は違っても背後にある要請は同じだと見えてきます。

### HTTP/2 の非同期性は利用者にも残る

第2章でも触れた通り、ライブラリは複雑さを内部へ押し込みます。しかし非同期性だけは、かなりの部分が利用者の前に残ります。なぜなら、接続共有と多重化の結果として必要になる「状態変化への反応」は、利用者の責務に近いからです。

たとえば、アプリケーションは個別ストリームの終了を知る必要があります。途中で中断されたのか、正常に終わったのかも知りたいでしょう。接続全体が閉じる方向に入ったなら、新しい処理をどこまで受け付けるかを考える必要があります。流量制御の影響で書き込みが止まるなら、その待ち方を考えなければなりません。

つまり、HTTP/2 の非同期性は単なる内部実装の都合ではなく、**利用者が何にどう反応する必要があるか**という問題でもあります。だから API はイベントや非同期インターフェースを露出しますし、利用者もそれを避けて通れません。

この意味で、非同期処理は「HTTP/2 にたまたまよく使われる技法」ではありません。HTTP/2 の構造を高水準で扱おうとしたときに、ほぼ必然的に前に出てくる表現です。

### 「非同期にすると速い」から「非同期でないと整理しづらい」へ

この章で視点をひとつずらしておきたいと思います。HTTP/2 と非同期処理の関係を、これからは「非同期にすると速い」ではなく、**非同期でないと整理しづらい**という向きで捉えてください。

もちろん、性能は大事です。ですが、中級者として重要なのは、なぜ API がその形になり、なぜ責務がそのように切られ、なぜイベントが必要になるのかを説明できることです。その中心には、多重化された複数のストリームと、接続全体の状態を同時に扱う必要があります。

この理解があると、次章のクラス責務の話も読みやすくなります。Connection と Stream をなぜ分けるのか。Frame の処理はどこに置くのか。flow control や error や shutdown の責務はどこが持つべきか。これらはすべて、「並行する状態をどの単位で管理するか」という問いにつながっているからです。

### 整理ブロック：HTTP/2 で同期的発想が苦しくなる理由

| 苦しくなる点 | 何が起きているか | なぜ同期的発想だけでは足りないか |
| --- | --- | --- |
| 複数ストリームの共存 | 個別通信が別々のタイミングで進む | 一つ終わってから次へ、では表しにくい |
| 接続全体の状態変化 | SETTINGS、shutdown、全体エラーなどがある | 個別 request/response だけでは追えない |
| 段階的な進行 | headers、data、end、reset などが分かれる | 完了通知一回では情報が足りない |
| 待ちの多様性 | 読み待ち、書き待ち、終了待ちがある | 待ちを一本化すると責務が混ざる |

### 整理ブロック：request/response 脳と stream/event 脳

| 見方 | 中心に置くもの | 向いている理解 |
| --- | --- | --- |
| request/response 脳 | 一往復の完了 | 単純な線形処理の理解 |
| stream/event 脳 | 並行する状態変化 | HTTP/2 の多重化と共有接続の理解 |

### 整理ブロック：非同期処理をどう捉えるか

| 見方 | 内容 |
| --- | --- |
| 速度論としての非同期 | 待ち時間を有効利用するための技法 |
| 状態管理としての非同期 | 並行する複数の状態変化を扱うための表現 |
| 待ち方の整理としての非同期 | 接続単位・ストリーム単位・イベント単位で待ちを分解する方法 |

### この章で残すべきメンタルモデル

HTTP/2 と非同期処理の相性がよいのは、速いからではありません。  
1本の接続の上で複数のストリームが並行して進み、しかも接続全体にも独自の状態があるからです。

HTTP/2 は、request/response だけで世界を捉えるより、  
stream と event を中心に、並行する状態変化を扱うほうが自然に見えます。

非同期処理は性能の小技ではありません。  
HTTP/2 の多重化と共有接続を、利用者が無理なく扱うための表現です。

## まとめ

- HTTP/2 と非同期処理の相性は、速度よりも状態管理の都合で説明できる
    
- 複数ストリームの並行進行と接続全体の状態変化が、同期的な直線モデルを苦しくする
    
- request/response 脳より、stream/event 脳のほうが HTTP/2 を自然に捉えやすい
    
- 非同期処理は、同時実行そのものよりも、並行する状態と待ち方を整理する表現として重要である
    
- イベントループとの親和性は、段階的な状態変化を責務ごとに処理しやすい点にある
    

次章では、この「並行する状態をどう管理するか」という視点を、クラス責務の設計へつなげます。Connection、Stream、Frame、flow control、error、shutdown を、どの単位で持つべきかがそこで見えてきます。
