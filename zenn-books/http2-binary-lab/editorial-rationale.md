# editorial-rationale

## この実習編を作る理由

本実習編は、『バイナリハック入門 ― UTF-8・HTTP/2・HPACK で学ぶ構造と状態』の続編として構想された教材である。  
本編の役割は、バイト列を「16 進数の壁」としてではなく、**境界・分岐・妥当性・状態**という観点から読むための見方を作ることにあった。

一方で、観点を理解しただけでは、実際の短いバイト列やテストデータを前にしたときに、すぐ読めるようになるとは限らない。  
HTTP/2 や HPACK の実データは、仕様を知っていても、どこから見ればよいのかが曖昧になりやすい。さらに、実装記事やサンプルコードでは、TLS、ALPN、ソケット処理、送受信ループ、エラー処理など、テストデータの読解そのものとは別の責務が大量に混ざる。そのため、中級者にとっても「読めるコード」である一方、「何を送っているか」「何が不正なのか」「どこで状態が必要になるのか」は見失いやすい。

そこで本実習編では、本編で獲得した観点を、**実際の最小データに着地させて鍛える**ことを目的とする。

---

## 本編との役割分担

本編と実習編は、扱う主題が似ていても役割が異なる。

### 本編の役割
本編は、次の観点を身につけるための本である。

- バイナリデータには境界がある
- 可変長データでは分岐の見方が重要になる
- 構造を切り出せることと、妥当であることは別である
- 状態を持つ構造では、単発の parser だけでは足りない
- iterator / parser / validator / state 管理を分けて考える必要がある

つまり本編は、**何を見るべきかを学ぶ本**である。

### 実習編の役割
これに対して実習編は、実際のデータを前にして、

- どこで境界を見るか
- 何を分岐点として扱うか
- どの時点で validator を意識するか
- どこから state が必要になるか

を、短いバイト列やテストデータを通じて反復練習するための資料である。

つまり実習編は、**作った観点を実データに適用する本**である。

この違いを明確にするため、本実習編では本編の要約や仕様の網羅は主目的としない。  
また、実装コードのフルチュートリアルも主役にしない。主役はあくまで、**最小の正常系・最小の不正系・1 ケース 1 論点のテストデータ**である。

---

## 付録ではなく独立実習編とした理由

本企画は当初、本編の付録や補遺として扱う可能性もあった。  
しかし最終的には、独立した実習編として設計する方針を採った。理由は三つある。

### 1. 読者に要求する動作が違う
本編では、概念整理と視点の獲得が中心である。  
一方、実習編では、短いバイト列を目の前に置き、順序立てて読む訓練が中心になる。

つまり、本編が「理解する」動作を中心にしているのに対し、実習編は「読む」「切る」「止まる」「判断する」という動作を中心にする。  
この違いは大きく、付録として圧縮すると、演習としての段階設計が弱くなりやすい。

### 2. 正常系・不正系・状態依存を段階的に扱う必要がある
実習資料では、単なる例示ではなく、次のような段階を意識する必要がある。

- 正常な最小データを読む
- 壊れた最小データを読む
- 単発では読めず、状態が必要になるデータを読む
- 公開テストケースを読む
- ツールの出力や適合性検査の結果を読む

この段階をきちんと設計するには、付録ではなく一冊の流れとして組み立てたほうが教育効果が高い。

### 3. 到達目標が具体的に存在する
本実習編には、比較的はっきりした到達目標がある。

- Qiita の最小 HTTP/2 クライアントに埋め込まれた定数を単独で読める
- `http2jp/http2-frame-test-case` のケースを読める
- `http2jp/hpack-test-case` のケースを読める
- `summerwind/h2spec` の結果を見て、どの層の問題か説明できる

こうした到達目標を持つ以上、補遺として軽く添えるよりも、独立した実習教材として設計したほうが自然である。

## なぜ UTF-8 を実習の導入に入れるのか

本実習編は HTTP/2 バイナリを主題とするが、冒頭に UTF-8 の生データを使ったウォームアップを入れる方針を採っている。  
これは話題を広げたいからではなく、**分岐付き可変長データの読み方を、小さな題材で再確認するため**である。

UTF-8 は表面的には「1〜4 バイト文字」として説明されがちだが、実装の観点ではそれだけでは足りない。  
実際には、1 バイトから 4 バイトまでの長さ分類の背後に、先頭バイトごとの分岐条件があり、さらに第 2 バイト以降の許容範囲も先頭バイトに応じて変わる。つまり UTF-8 は、教育的には「1〜4 バイトの文字コード」というよりも、**先頭バイトによって読み方と妥当性条件が分岐する小さな可変長フォーマット**として捉えたほうが有益である。

この性質は、HTTP/2 フレームや HPACK を読むときの感覚とよく似ている。

- 最初に境界を切る
- 先頭側の情報で分岐する
- 長さが合っていても妥当とは限らない
- 後続の解釈には追加条件が必要になる

したがって、UTF-8 は本実習編の主題そのものではないが、HTTP/2 に入る前の**小さな助走**として適している。

## なぜ最小テストデータを重視するのか

HTTP/2 や HPACK を学ぶとき、実運用のキャプチャや大きな実装コードから入る方法もある。  
しかし本実習編では、それを最初の入口にはしない。入口として重視するのは、**最小の正常系・最小の不正系・1 ケース 1 論点**である。

この方針を採る理由は、読解対象に含まれる論点をできるだけ分離したいからである。

たとえば、大きな通信ログやキャプチャでは、次の論点が同時に混ざりやすい。

- フレーム境界
- フレーム種別
- flags の影響
- stream の文脈
- HPACK block
- state 依存
- ネットワーク入出力
- TLS や ALPN
- 実装固有のログ表現

こうした要素が一度に出てくると、どの難しさに引っかかっているのかが見えにくくなる。  
一方、最小テストデータであれば、「今はフレーム境界だけを見る」「今は長さ不整合だけを見る」「今は state が必要になる瞬間だけを見る」といった形で、論点を切り分けて扱える。

本実習編の中心は、仕様知識の網羅ではなく、**読む動作の訓練**である。  
そのため、テストデータも「現実に近い巨大なもの」より、「思考の焦点が定まる小さなもの」を優先する。

## 公開テストケースを到達目標に置く理由

本実習編では、公開されている次の資産を重要な到達目標としている。

- `http2jp/http2-frame-test-case`
- `http2jp/hpack-test-case`
- `summerwind/h2spec`

この三者は、いずれも HTTP/2 / HPACK を扱うが、役割が異なる。

### `http2-frame-test-case`
HTTP/2 フレーム単体を対象にしたテストケース集である。  
ここでは、各フレームがどのような構造を持ち、何が正常で何が不正なのかを、比較的静的な形で読める。

### `hpack-test-case`
HPACK の raw-data と headers の対応や、state を含む story を追うためのテストケース集である。  
ここでは、フレーム読解の先にある「状態を持つ圧縮構造」の読解が問われる。

### `h2spec`
実装全体に対して仕様準拠を検査する conformance tool である。  
ここでは、単なる静的データ読解ではなく、「どのように振る舞うべきか」「どの仕様違反がどの層の責務に属するか」を考えることになる。

この三つを到達目標に置くことで、実習編は単なる自作問題集ではなく、**公開資産を読めるようになるための橋渡し教材**になる。  
また、実装やライブラリを読む際にも、「このケースは frame parser の責務なのか」「validator の責務なのか」「state 管理の責務なのか」といった見方をしやすくなる。

## ツールをどう位置づけるか

本実習編では、`nghttp`、`h2i`、`nghttpd`、`h2spec` などのツールにも触れる。  
ただし、それらは本文前半の主役ではない。

実習編の中心は、あくまで**手で読める最小データ**である。  
ツールは、その読解経験を

- ログに接続する
- 対話的な観測に接続する
- ローカル環境での検証に接続する
- conformance testing に接続する

ための**橋渡し**として使う。

この順番にすることで、読者はツールの出力を「読むための補助」として使えるようになる。  
逆に、先にツールから入ると、表示された情報を受け取るだけになりやすく、生のバイト列やテストケースとどう対応しているのかが曖昧になりやすい。

## この実習編の教育方針

本実習編の教育方針は、次のように整理できる。

### 1. 仕様を暗記させるのではなく、読む手順を身につけさせる
重要なのは、項目を覚えることそのものではなく、短いデータを前にしてどの順で見ればよいかを定着させることである。

### 2. 意味を急がず、まず境界を見る
最初から意味や用途に飛びつくのではなく、まずどこまでがひとまとまりかを切る。  
この姿勢は、UTF-8、HTTP/2 フレーム、HPACK のいずれにも共通する。

### 3. parser と validator を混同しない
構造を切り出せることと、仕様上正しいことは別である。  
この差を、正常系だけでなく不正系でも体験させる。

### 4. 状態が必要になる瞬間を意識して区切る
単発のデータで読める範囲と、前後関係がなければ読めない範囲を区別する。  
これにより、state 管理の必要性を抽象論ではなく実感として理解できる。

### 5. 教材の出口を公開資産に接続する
自作の例題だけで閉じず、公開 test case や conformance tool に接続することで、学習の出口を具体化する。

---

## まとめ

本実習編は、本編で獲得した「境界・分岐・妥当性・状態」という観点を、最小のバイト列・公開 test case・conformance tool の結果へと段階的に適用していくための教材である。

本編が「なぜそう見るのか」を扱ったのに対し、実習編は「どう実際に読むのか」を扱う。  
そのために、本実習編では次の方針を採る。

- 付録ではなく独立実習編として設計する
- UTF-8 を助走として用いる
- HTTP/2 フレームを最小データから読む
- 壊れたケースを通じて validator を理解する
- HEADERS から HPACK へ段階的に入る
- state が必要になる瞬間を別章で扱う
- `http2-frame-test-case` と `hpack-test-case` を到達目標に置く
- `h2spec` を通じて実装全体の振る舞いへ橋渡しする

この構成によって、本実習編は、HTTP/2 バイナリを「仕様の知識」としてではなく、**読解可能な構造として扱うための訓練の場**になることを目指す。
