## 第7章　状態が必要になると何が変わるか

この章では、HPACK に state が入ると何が変わるかを扱います。  
ここまでの章では、できるだけ単発で読める最小ケースを使ってきました。HTTP/2 フレームも、最小の HPACK も、まずは「その場にあるデータだけを見れば読める」形から始めました。  
しかし、HPACK はそこでは終わりません。ある段階から、**今目の前にある raw-data だけでは意味が決まらない**という状況が出てきます。

この章の中心は、その変化をはっきり言葉にすることです。

- どこまでは単発で読めたのか
    
- どこから前提 state が必要になるのか
    
- state が入ると parser と validator の仕事はどう変わるのか
    
- なぜ dynamic table manager のような責務が必要になるのか
    

本章では、状態依存の複雑な実装論に一気に入るのではなく、まず **state が必要になる最小の瞬間** を小さな例で確認します。  
つまり本章は、HPACK を「ただの圧縮表現の列」として読む段階から、「前後関係を保持しながら読む構造」として理解する段階への橋渡しです。

### この章で何を訓練するか

この章では、次のことができるようになることを目指します。

dynamic table を導入する最小ケースを読めること。  
単発では読めないケースを見分けられること。  
同じ raw-data でも、前提 state によって解釈が変わることを説明できること。  
state manager が必要になる理由を言語化できること。  
そして、**ここから難しさの種類が変わる**と自覚して読めることです。

ここではまだ、複雑な story を大量に追いません。  
また、実装の最適化や高速化の話にも入りません。  
今はまず、「state が必要になるとはどういうことか」を、最小の例でつかむことを優先します。

---

## 単発で読めるケースと、state が必要なケース

前章までの HPACK 最小例では、1 個または数個の表現を順に読み、その場で最終的な headers を復元できました。  
そのとき使っていた前提は、次のようなものでした。

- static table は固定である
    
- literal はその場に必要な情報を持っている
    
- raw-data の意味は、その block 自体を読めば決まる
    

この前提がある限り、HPACK は「少し複雑な可変長データ」として読めます。  
しかし dynamic table が入ると事情が変わります。

dynamic table が入ると、ある表現の意味を知るために、**それ以前に何が追加されたか**を知らなければならなくなります。  
すると、読解は単発では完結しません。  
今見ている表現の前に、どんな header field が table に積まれているかを覚えておく必要が出てきます。

ここで難しさが変わります。

それまでは「どこで切るか」「どの種類の表現か」が中心でした。  
state が入ると、そこに「現在の table はどんな状態か」が加わります。  
つまり、構造の読解に加えて、**過去の入力の要約を保持する責務**が必要になるのです。

---

## state とは何を持つことか

この章でいう state は、抽象的な「前後関係」ではありません。  
もっと具体的には、少なくとも次のようなものです。

- それまでに dynamic table へ追加された項目
    
- その現在の並び順
    
- 今この瞬間に参照可能なインデックスの意味
    

つまり state を持つとは、「前の block までに構築された参照先の一覧を保持する」ということです。

ここで大事なのは、state は単なる補助情報ではないということです。  
dynamic table を使う表現に入った瞬間、それは **意味決定そのものの一部** になります。

たとえば、あるインデックス参照が現れたとして、その番号だけを見ても十分ではありません。  
その番号が static table を指しているのか、dynamic table の現在の何番目を指しているのかは、前提 state に依存します。  
したがって、bytes を切れたとしても、それだけでは header を復元できない場合があります。

これが、本章で確認したい「state が必要になる」ということです。

---

## 読み方がどう変わるか

state が入るといっても、最初の読み順が全部崩れるわけではありません。  
ここも重要です。

まず raw-data を見る。  
次に 1 個の表現を切る。  
その先頭バイトから種類を判定する。  
ここまでは前章までと同じです。

変わるのはその次です。  
種類判定のあとに、「この表現を解釈するには現在の table state が必要か」を確認する必要が出ます。

つまり読み順は、次のように一段増えます。

まず bytes を切る。  
次に表現の種類を判定する。  
そのあとで、state なしで意味が決まるか、state が要るかを判断する。  
state が要るなら、現在の table を参照して意味を確定する。

この一段が増えるだけで、読解の性質はかなり変わります。  
前章までは「その場の bytes だけで説明できる」ことが多かったのに対し、本章からは「その場の bytes と、ここまでの state を合わせて説明する」必要が出てきます。

---

## なぜ state manager が必要になるのか

HPACK の説明だけを読んでいると、dynamic table は「あとで使うための追加表」くらいに見えるかもしれません。  
しかし実装の責務として見ると、これは単なる補助ではありません。

state が必要になると、少なくとも次の仕事が発生します。

- 新しく追加された header field を table に入れる
    
- その順序を維持する
    
- 後続の参照で、その時点の table を使って解釈する
    
- 場合によっては古い項目の押し出しや消失も考える
    

ここで見えてくるのが、parser と state manager の役割差です。

parser は、表現の種類や長さや文字列を切り出します。  
しかし parser だけでは、「この参照が現在何を意味するか」は決めきれません。  
そのため、state manager が必要になります。

本章では実装コードを書きません。  
ただし少なくとも、**dynamic table が出た瞬間に、単発 parser だけでは責務が足りなくなる**ということは、はっきり意識しておきます。

---

## 実習 1　dynamic table に 1 件追加する最小ケースを見る

### このケースで見ること

最初の実習では、dynamic table を使う前に、まず table に 1 件追加する最小ケースを見ます。  
狙いは、「状態依存の読み方は、参照より前に追加から始まる」と確認することです。

### データ

1 個の literal 系表現があり、それが単に header をその場で表すだけでなく、後続で再利用できる形で table に追加されるケースを考えます。

### まず何を見るか

先頭バイトを見て、これは単なる static 参照ではなく、literal 系の表現であると判断します。  
次に、ヘッダ名と値を読みます。  
ここまでは前章の literal の読み方と大きくは変わりません。

### 次に何を見るか

ここで終わらず、この表現が「復元された header を返す」だけでなく、「dynamic table に新しい項目を追加する」という効果を持つと考えます。

### 最後に何を見るか

この block を読み終えた時点で、現在の dynamic table に 1 件の項目が存在すると整理します。

### このケースで十分な到達点

このケースを見て、

- literal 表現として header 自体を読める
    
- さらに、その結果が dynamic table に追加される
    
- したがって、この block の読了後に state が変化する
    

と説明できれば十分です。

### このケースの意義

このケースが重要なのは、state は突然空から現れるのではなく、前の入力の結果として作られると分かるからです。  
つまり、後の参照は単独では理解できません。  
それ以前の「追加」があって初めて意味を持ちます。

---

## 実習 2　次の block でその項目を参照する

### このケースで見ること

次は、前の実習で追加した項目を、次の block で参照する最小ケースを見ます。  
ここで初めて、「今ある bytes だけでは意味が決まらない」状況を体験します。

### データ

前の block で 1 件の header field が dynamic table に追加されているとします。  
その次の block には、短い参照表現だけが入っているケースを考えます。

### まず何を見るか

現在の block の先頭バイトを見て、これは参照系の表現だと判断します。  
ここまでは前章の static 参照と似ています。

### 次に何を見るか

参照先のインデックスを読みます。  
しかし、ここで止まってはいけません。  
そのインデックスが何を意味するかは、現在の table state を見なければ決まりません。

### 最後に何を見るか

前の block の結果として 1 件追加されていた dynamic table を参照し、そのインデックスがどの header field を指しているかを確定します。

### このケースで十分な到達点

このケースを見て、

- 現在の raw-data だけでは header が決まらない
    
- 1 つ前の block で追加された項目が前提になる
    
- したがって、この block は state 依存である
    

と説明できれば十分です。

### このケースの意義

このケースが、本章の最重要ポイントです。  
前章までの最小例では、block 単独で読めました。  
しかしここでは、それができません。  
つまり、**同じ「参照」に見えても、static table 参照と dynamic table 参照では読解の前提が違う**のです。

---

## 実習 3　同じ raw-data でも state が違うと意味が変わる

### このケースで見ること

ここでは、まったく同じ raw-data でも、前提 state が違えば意味が変わることを確認します。  
これが state 依存の本質です。

### データ

短い参照表現だけからなる同一の raw-data を 2 回考えます。  
ただし、1 回目は dynamic table に何かが入っている状態、2 回目は別の項目が入っている状態、という前提を置きます。

### まず何を見るか

bytes 自体は同じです。  
先頭バイトも同じです。  
参照先インデックスも同じです。  
parser の観点では、両者の切り出し結果は同じに見えます。

### 次に何を見るか

しかし、そのインデックスが指す先は、前提となる dynamic table の内容によって変わります。  
つまり、同じ bytes でも、state が違えば復元される header が変わります。

### 最後に何を見るか

この差を通じて、「bytes だけで意味が閉じていない」と確認します。

### このケースで十分な到達点

このケースを見て、

- parser の出力だけでは意味が足りない
    
- 同じ raw-data でも state によって解釈が変わる
    
- したがって、state は付随情報ではなく意味決定の一部である
    

と説明できれば十分です。

### このケースの意義

このケースにより、state を「実装都合のキャッシュ」程度に軽く見る誤解を防げます。  
dynamic table は単なる高速化の補助ではありません。  
HPACK の一部の表現は、最初から state を前提として意味を持っています。

---

## 実習 4　単発で読めるケースと state 依存ケースを並べる

### このケースで見ること

ここでは、static table の単発ケースと dynamic table の状態依存ケースを並べて比べます。  
狙いは、難しさの種類がどこで変わるのかを見えるようにすることです。

### データ

片方は static table を参照するだけの短いケース。  
もう片方は dynamic table を前提とする短いケースです。

### まず何を見るか

両者とも先頭バイトを見て、参照表現らしいことを確認します。  
どちらも bytes の切り方だけなら大差ないように見えるかもしれません。

### 次に何を見るか

しかし static table のケースでは、その場ですぐ意味が決まります。  
一方 dynamic table のケースでは、前提 state を参照しなければ意味が確定しません。

### 最後に何を見るか

この差を、「構文の違い」ではなく「必要な責務の違い」として整理します。

### このケースで十分な到達点

このケースを見て、

- static table のケースは単発 parser でかなり読める
    
- dynamic table のケースは state manager なしでは意味が確定しない
    
- したがって、ここで難しさの質が変わる
    

と説明できれば十分です。

### このケースの意義

この比較によって、本章で難しくしているものが何かがはっきりします。  
問題は bytes の長さではありません。  
問題は、**意味決定に過去の入力が必要になること**です。

## 実習 5　短い story として読む

### このケースで見ること

最後に、追加と参照を 2 手か 3 手の短い story として読みます。  
狙いは、状態依存ケースを「巨大な時系列」ではなく、小さな物語として追う感覚を作ることです。

### データ

1 手目で literal によって項目が追加される。  
2 手目でそれが参照される。  
必要なら 3 手目でもう一度別の参照や追加が起こる。  
その程度の短い流れを考えます。

### まず何を見るか

各手ごとに raw-data を読み、その場でどんな header が得られるかを確認します。  
同時に、その手の終了時点で dynamic table がどう変化したかも記録します。

### 次に何を見るか

次の手では、その更新済み table を前提として表現を解釈します。  
つまり、各手の意味は「その手の bytes」だけでなく、「前手までの state」に依存します。

### 最後に何を見るか

この短い story 全体を通じて、table の変化と後続参照の関係を追います。

### このケースで十分な到達点

このケースを見て、

- state 依存ケースは単発の正誤問題ではなく、小さな時系列として読む必要がある
    
- 各手で table がどう変化したかを追うことが重要である
    
- その tracking がないと、後続の参照を正しく解釈できない
    

と説明できれば十分です。

### このケースの意義

この実習は、次章以降で公開 test case を読むための準備です。  
特に HPACK の test case では、単発で読める例だけでなく、story を追う必要がある例が現れます。  
その入口として、本章ではごく短い story にとどめます。

---

## parser、validator、state manager の責務差

ここまで来ると、HPACK の読解には少なくとも三種類の責務があると見えてきます。

まず parser があります。  
これは bytes を切り、表現の種類を判定し、必要な長さ付き要素を取り出します。

次に validator があります。  
これは、その切り出した表現が妥当か、長さが足りているか、許される参照かを確認します。

そして state manager があります。  
これは dynamic table を更新し、後続参照が現れたときに、その時点の意味を与えます。

この三つは似ているようで違います。  
parser だけでは、dynamic table 参照の意味は決まりません。  
validator だけでも、参照先を作れません。  
state manager だけでも、そもそも bytes を切れません。

本章で残したいのは、この責務差です。  
それが見えると、HPACK の実装や test case を読むときに、「どの層の仕事でつまずいているのか」を整理しやすくなります。

---

## state が入っても、まず境界を見る

state 依存が出てくると、読者はつい「とにかく表の中身を覚えなければ」と焦りやすくなります。  
しかしここでも、最初の基本動作は変わりません。

まず bytes を切る。  
次に表現の種類を判定する。  
そのあとで、state が必要かどうかを確認する。  
必要なら現在の table を参照する。

つまり、state が入っても「まず境界」という原則は崩れません。  
崩れるのは、意味決定がその場の bytes だけで閉じなくなることです。

この区別を保つことが重要です。  
state を意識するあまり、構造読解そのものが曖昧になってはいけません。  
逆に、構造だけを見て state を無視しても正しく読めません。  
両方を順番に扱う必要があります。

---

## この章でまだ扱わないこと

ここでは、まだ次のことは本格的に扱いません。

- dynamic table の大きな story
    
- 複数の追加と押し出しが複雑に絡むケース
    
- HPACK の詳細な不正ケース
    
- 実装全体としてのメモリ管理や効率化
    
- HTTP/2 フレーム列全体の中での header block の断片化
    

これらは次の公開 test case 読解や、さらに先の実装読解で重要になります。  
今はまず、**state が必要になる最小の瞬間を見て、単発読解から時系列読解へ切り替わる地点を理解すること**を優先します。

---

## この章で残すべきメンタルモデル

本章で残したいモデルは次のものです。

前章までの HPACK は、その場の raw-data だけでかなり読めた。  
しかし dynamic table が入ると、raw-data だけでは意味が決まらない。  
ある表現の意味は、前の入力によって作られた state に依存する。  
したがって HPACK には、parser や validator に加えて state manager という責務が必要になる。

要するに、**state が必要になるとは、今見ている bytes の意味が、過去の入力の要約に依存するようになること**です。

この感覚が持てれば、本章の役割は果たせています。

## まとめ

- dynamic table が入ると、HPACK は単発読解だけでは足りなくなる
    
- state は前の入力の結果として作られ、後続の意味決定に使われる
    
- 同じ raw-data でも、前提 state が違えば解釈が変わることがある
    
- state 依存ケースでは、parser だけでなく state manager の責務が必要になる
    
- それでも最初の読み順は変わらず、まず bytes を切り、次に種類を判定する
    
- 本章の中心は、単発読解から時系列読解へ切り替わる地点を理解することである
