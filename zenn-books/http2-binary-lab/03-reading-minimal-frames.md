## 第3章　最小フレームを読む

この章では、前章で切り出した共通ヘッダ 9 バイトを土台にして、最小の正常系フレームを実際に読んでいきます。  
ここでの目的は、HTTP/2 の全フレーム種別を覚えることではありません。狙いはもっと限定的です。**共通ヘッダを読んだあと、どこから種別ごとの解釈が始まるのかを体で理解すること**です。

前章では、まず境界を切り、Length、Type、Flags、Stream Identifier を順に読むところまでを扱いました。この章では、その次の一歩として、SETTINGS、SETTINGS ACK、DATA、GOAWAY という最小の例を見ます。  
ただし、ここでもまだ仕様の百科事典にはしません。大事なのは、最小データを前にして、

ここまでは共通処理で読める。  
ここから先は Type ごとの処理になる。  
このフレームでは payload はどこまで読めば十分か。

と切り分けられることです。

この章は、「フレーム種別の意味を覚える章」というより、**共通ヘッダから payload へ入る最初の訓練**です。

### この章で何を訓練するか

この章では、次のことができるようになることを目指します。

最小の SETTINGS を読めること。  
SETTINGS ACK を読めること。  
最小の DATA を読めること。  
最小の GOAWAY を読めること。  
そして、**どこまでが共通処理で、どこから種別ごとの処理か**を説明できることです。

ここでは、複雑な flags の組み合わせ、壊れたフレーム、HEADERS の先にある HPACK、stream 文脈の詳細な妥当性判定までは入りません。今はまず、正常系の最小データを使って、共通ヘッダの先へ一歩だけ進みます。

---

## 最小フレームを読むときの基本姿勢

最小フレームを読むときも、順序は変わりません。

まず境界を見る。  
次に共通ヘッダを読む。  
そのあとで payload の意味に入る。  
最後に、そのフレームでどこまで読めれば十分かを確認する。

ここでありがちな失敗は、Type を見た瞬間に仕様知識を思い出そうとして、境界や payload 長の確認が雑になることです。  
ですが、実際の読解では逆です。先に Length でフレーム境界を切れていなければ、その後に出てくる payload の解釈も安定しません。

したがって本章でも、各ケースでまず共通ヘッダから始めます。  
そのうえで、「この Type なら payload をどう読むか」という最小限の分岐だけを扱います。

---

## 共通処理と種別ごとの処理

この章で特に意識したいのは、HTTP/2 フレーム読解には少なくとも二段階あるということです。

第一段階は、すべてのフレームに共通する処理です。  
ここでは、

- 9 バイトの共通ヘッダを切る
    
- Length を読む
    
- Type を読む
    
- Flags を読む
    
- Stream Identifier を読む
    

という動作を行います。

第二段階は、Type ごとに変わる処理です。  
ここでは、

- SETTINGS なら payload を 6 バイト単位で見る
    
- DATA なら payload をデータ本体として見る
    
- GOAWAY なら先頭側に固定長の情報があると考える
    

という具合に、payload の読み方が変わります。

この二段階を混ぜないことが大事です。  
最初の parser が担うのは共通処理です。そのあとで種別ごとの parser に引き渡す、と考えると整理しやすくなります。

---

## 実習 1　空の SETTINGS を読む

### このケースで見ること

最初のケースでは、もっとも小さな SETTINGS を読みます。  
ここで確認したいのは、Type が SETTINGS のとき、payload が 0 バイトなら、共通ヘッダのあとに追加で読むものがない、ということです。

### データ

空の SETTINGS フレームを考えます。  
Length は 0、Type は SETTINGS、Flags は 0、Stream Identifier は 0 です。

### まず何を見るか

最初の 3 バイトを見て、Length が 0 であることを確認します。  
これによって、フレーム全体は共通ヘッダ 9 バイトだけで終わると分かります。

次に Type を見て SETTINGS だと確認します。  
Flags は 0 です。  
Stream Identifier は 0 です。

### 次に何を見るか

ここで Type が SETTINGS なので、本来なら payload の中に設定項目が並ぶ可能性があります。  
しかし Length が 0 なので、今回は payload 自体がありません。

つまりこのケースでは、Type ごとの処理に入るとしても、「読む payload はない」と判断して終われます。

### このケースで十分な到達点

空の SETTINGS を見て、

- 共通ヘッダ 9 バイトでフレームが完結する
    
- Type は SETTINGS である
    
- payload を読む段階に進んでも、今回は追加の内容がない
    

と説明できれば十分です。

### このケースの意義

このケースのよいところは、共通ヘッダと種別ごとの処理の境目が見えやすいことです。  
共通ヘッダまでは確実に読める。  
SETTINGS であることも分かる。  
しかし payload は 0 なので、それ以上の複雑な処理は不要です。

つまり、「種別ごとの処理がある」といっても、最小ケースではそこをほとんど空のまま通過できることが分かります。

---

## 実習 2　SETTINGS ACK を読む

### このケースで見ること

次は SETTINGS ACK を見ます。  
ここでの狙いは、Type が同じ SETTINGS でも、Flags によって意味づけが変わることを確認することです。

### データ

SETTINGS ACK フレームを考えます。  
Length は 0、Type は SETTINGS、Flags は ACK、Stream Identifier は 0 です。

### まず何を見るか

最初の 3 バイトを見て、Length が 0 であることを確認します。  
したがって、このフレームもやはり共通ヘッダ 9 バイトだけで完結します。

次に Type を見て SETTINGS だと確認します。  
ここまでは空の SETTINGS と同じです。

### 次に何を見るか

次に Flags を見ます。  
ここで、先ほどの空の SETTINGS と違って ACK が立っていることを確認します。

この段階で重要なのは、「Type が同じでも Flags によって読み方の分岐が入る」と理解することです。  
今回のケースでは payload はやはり 0 なので、payload 側の複雑さはありません。  
しかし、共通ヘッダのうち Flags が変わるだけで、フレームの役割が変わることが見えます。

### このケースで十分な到達点

SETTINGS ACK を見て、

- Type は SETTINGS である
    
- しかし Flags に ACK が立っているので、空の SETTINGS と同一ではない
    
- Length は 0 なので、フレーム境界の切り方は同じである
    

と説明できれば十分です。

### このケースの意義

このケースは、同じ Type の中でも Flags が後続の意味づけを変えるという最小例です。  
つまり、共通ヘッダの中で分岐は一度しか起きないわけではありません。Type が大きな分岐で、その中に Flags による細かな分岐が入ることがあります。

本章ではまだ ACK の妥当性条件には踏み込みません。  
それは次章の「壊れたフレームを読む」で扱います。  
ここではまず、**Type のあとにも分岐が残っている**と分かれば十分です。

## 実習 3　1 バイト payload の DATA を読む

### このケースで見ること

次は最小の DATA を見ます。  
ここで確認したいのは、payload が 0 ではないフレームに入ったとき、Length がそのまま境界決定に効くことです。

### データ

Length が 1、Type が DATA、Flags は 0、Stream Identifier は 1 のような単純な DATA フレームを考えます。  
payload には 1 バイトだけデータが入っています。

### まず何を見るか

最初の 3 バイトを見て、Length が 1 であることを確認します。  
この時点で、フレーム全体の長さは

9 バイトの共通ヘッダ  
＋  
1 バイトの payload

だと分かります。

次に Type を見て DATA だと確認します。  
Flags を見ます。  
Stream Identifier を見ます。

### 次に何を見るか

ここで初めて payload に入ります。  
今回は DATA フレームなので、payload はそのままデータ本体として読めます。  
しかも Length は 1 なので、読むべき payload は 1 バイトだけです。

このケースでは、「payload の意味」がかなり単純です。  
そのため、共通ヘッダを読んだあとに payload へ入る流れを、もっとも素直な形で確認できます。

### このケースで十分な到達点

このフレームを見て、

- Length 1 によってフレーム全体の境界が決まる
    
- Type DATA なので、payload はデータ本体として扱う
    
- 今回は 1 バイトしかないので、そこまで読めば十分
    

と説明できれば十分です。

### このケースの意義

空の SETTINGS と比べると、このケースでは初めて「共通ヘッダのあとに実際に読む payload がある」状態になります。  
その意味で、共通処理と種別ごとの処理のつながりが一段だけ具体的になります。

ただし、それでもまだ複雑ではありません。  
DATA はここでは「payload をそのまま読む」という最小の例です。  
だからこそ、境界と payload の接続を見る訓練に向いています。

---

## 実習 4　最小の GOAWAY を読む

### このケースで見ること

次は GOAWAY を見ます。  
ここでの狙いは、Type によって payload の中にも最低限の構造が現れることを確認することです。

### データ

GOAWAY フレームの最小ケースを考えます。  
GOAWAY には、最低限、先頭側に固定長の情報が入ります。したがって、空の SETTINGS や 1 バイト DATA よりも、payload の内部に少しだけ構造があります。

### まず何を見るか

最初の 3 バイトを見て Length を確認します。  
次に Type を見て GOAWAY だと確認します。  
Flags を見ます。  
Stream Identifier を見ます。

ここまではいつも通りの共通処理です。

### 次に何を見るか

Type が GOAWAY なので、payload をただの生データとして扱うのではなく、先頭側に決まった構造があるものとして読みます。  
この時点で重要なのは、「Type が変わると payload の内部構造も変わる」と分かることです。

本章では GOAWAY の中身を細部まで追いません。  
ですが少なくとも、DATA のように「ただの payload 1 バイト」ではなく、先頭に意味のあるまとまりがあることは確認します。

### このケースで十分な到達点

GOAWAY を見て、

- 共通ヘッダの読み方は他のフレームと同じ
    
- ただし payload の内部には GOAWAY 固有の構造がある
    
- したがって、共通ヘッダのあとに Type ごとの parser が必要になる
    

と説明できれば十分です。

### このケースの意義

このケースは、HTTP/2 フレームの payload が「全部同じような生データ」ではないことを見せる役割を持ちます。  
SETTINGS では payload が 6 バイト単位で並び、DATA では生データになり、GOAWAY では先頭に固定的な構造が置かれます。

つまり、Type は単なるラベルではありません。  
Type は、「このあと payload をどんな構造として読むか」を分岐させる情報です。

## 実習 5　コード中の定数をフレームとして読む

### このケースで見ること

ここでは、実装コードや記事の中に埋め込まれた定数を、コード全体から切り離して読む練習をします。  
狙いは、ソケット処理や TLS やループ処理に意識を奪われず、**中に埋まっているフレームそのものを単独で読めるようになること**です。

### データ

たとえば最小 HTTP/2 クライアントの説明記事には、SETTINGS、SETTINGS ACK、GOAWAY などの定数が直接書かれていることがあります。  
ここでは、そのバイト列を「コードの一部」としてではなく、「HTTP/2 フレームそのもの」として読みます。

### まず何を見るか

最初にやることは変わりません。  
コード断片を見ても、いきなり処理の意味を追わないことです。  
まずその定数列から 9 バイトの共通ヘッダを切り出します。

Length はいくつか。  
Type は何か。  
Flags はどうか。  
Stream Identifier は何か。

この順で確認します。

### 次に何を見るか

そのあとで、Type に応じて payload を読みます。  
SETTINGS なら payload の有無や長さを確認し、GOAWAY なら payload 先頭の構造を意識します。

### このケースで十分な到達点

コード中の定数を見て、

- まずフレーム境界を切れる
    
- 共通ヘッダを単独で読める
    
- そのあとで payload の最小限の意味に入れる
    

という流れが取れれば十分です。

### このケースの意義

この練習が重要なのは、実装記事やサンプルコードにはフレーム以外の情報が大量に混ざるからです。  
I/O、ソケット、TLS、エラー処理、ループ、文字列リテラル、ログ出力。  
そうした周辺要素に引きずられると、実は何を送っているのかが見えにくくなります。

だからこそ、コード中の定数を一度「純粋なフレーム」に戻して読む必要があります。  
この感覚は、今後ライブラリ実装や test case を読むときにも役立ちます。

---

## 最小フレームを並べて見えてくること

ここまでの 4 つの最小例を並べると、いくつかのことが見えてきます。

第一に、共通ヘッダの読み方はいつも同じだということです。  
SETTINGS でも DATA でも GOAWAY でも、最初にやることは変わりません。  
9 バイトを切り、Length、Type、Flags、Stream Identifier を順に読むだけです。

第二に、Type が変わると payload の読み方が変わるということです。  
SETTINGS では payload がなくてもよいし、あれば設定項目の列として読む必要があります。  
DATA では payload をデータとして読めます。  
GOAWAY では payload の先頭に固有の構造があります。

第三に、Flags は Type の中でさらに細かな分岐を作るということです。  
SETTINGS ACK はその最小例でした。

この三点が見えていれば、本章の狙いは達成できています。

---

## この章でまだ扱わないこと

ここでは、まだ次のことは本格的に扱いません。

- 不正な Length
    
- 不正な SETTINGS ACK
    
- stream 0 と非 0 の細かな妥当性
    
- HEADERS フレームの payload の詳細
    
- HPACK の中身
    
- state を必要とする読み方
    

これらは、すべて次の章以降で段階的に扱います。  
本章の中心は、正常系の最小フレームを通して、共通ヘッダのあとに何が起きるかを見えるようにすることです。

---

## この章で残すべきメンタルモデル

HTTP/2 フレームは、まずすべて同じ入口から読みます。  
共通ヘッダ 9 バイトを切り、Length、Type、Flags、Stream Identifier を確認する。  
ここまでは共通処理です。

そのあとで、Type に応じて payload の読み方が分岐します。  
これが種別ごとの処理です。

つまり、本章で残したいモデルは次の形です。

**まず共通ヘッダを読む。  
そのあとで Type ごとの parser に入る。  
Flags はその中の追加分岐になることがある。**

このモデルが安定すれば、後続の章で壊れたフレームを読むときにも、HEADERS の入口に立つときにも、どこで何を見ればよいかがぶれにくくなります。

## まとめ

- 最小フレームを読むときも、まず共通ヘッダから始める
    
- SETTINGS は payload 0 の最小例として、共通処理と種別処理の境目を見やすい
    
- SETTINGS ACK は、同じ Type でも Flags が分岐を作ることを示す
    
- DATA は、Length によって payload 境界が決まり、そのままデータ本体に入る最小例である
    
- GOAWAY は、Type によって payload の内部構造が変わることを示す
    
- コード中の定数も、周辺処理から切り離してフレームとして読めるようにすることが重要である
    

## 次章への接続

次章では、同じフレーム読解の型を使いながら、今度は壊れたフレームを扱います。  
そこで、構造として切り出せることと、仕様上妥当であることが別だという点を、実際の不正データで確認していきます。
