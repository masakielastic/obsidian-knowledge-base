## 第5章　HEADERS フレームの入口に立つ

この章では、HTTP/2 フレームとしての HEADERS を読みます。  
ただし、ここで目指すのは HEADERS のすべてを理解することではありません。むしろ逆です。大事なのは、**HTTP/2 フレームとしてどこまで読めるかを確認し、その先で一度止まること**です。

前章では、壊れたフレームを通して、構造として読めることと、仕様上妥当であることが別だと確認しました。  
この章では、さらに別の切り分けを学びます。  
それは、**フレーム層として読める範囲**と、**その先にある別レイヤー**を分けることです。

HEADERS は、HTTP/2 を学び始めた読者が一気に混乱しやすいフレームです。  
理由は単純で、HEADERS という名前のフレームの中に、さらに HPACK という別の構造が入っているからです。  
そのため、最初から全部を一つの塊として理解しようとすると、

- どこまでがフレーム共通ヘッダか
    
- どこからが HEADERS フレーム固有の話か
    
- どこからが HPACK block なのか
    

が混ざってしまいます。

そこで本章では、あえて途中で止まります。  
HEADERS フレームを見たときに、

ここまでは HTTP/2 フレームとして読める。  
ここから先は HPACK の領域であり、今はまだ深入りしない。

と判断できるようになることが、この章の中心です。

### この章で何を訓練するか

この章では、次のことができるようになることを目指します。

HEADERS フレームを、まず HTTP/2 フレームとして読めること。  
共通ヘッダ 9 バイトを切ったうえで、HEADERS としての入口を確認できること。  
flags の最低限の読み方を押さえられること。  
payload の先が HPACK block の入口であると説明できること。  
そして、**ここから先は別レイヤーだと一度止まれること**です。

ここではまだ、HPACK の内部読解には本格的に入りません。  
dynamic table も扱いません。  
Huffman の詳細も扱いません。  
今はまず、HEADERS フレームの入口に立ち、境界を切り分けることを優先します。

---

## なぜ HEADERS で一度止まるのか

SETTINGS や DATA や GOAWAY は、Type ごとの payload の読み方が変わるといっても、HTTP/2 フレームの延長として理解しやすい面があります。  
しかし HEADERS は少し違います。

HEADERS フレームの payload には、単なる「このフレーム専用の固定構造」だけではなく、ヘッダ表現そのものを圧縮した別のデータ構造が入っています。  
つまり、HEADERS は HTTP/2 フレームであると同時に、HPACK という別レイヤーへの入口でもあります。

このときに起こりやすい失敗は、HEADERS フレームを見た瞬間に「ヘッダ一覧を取り出すにはどうするか」へ飛んでしまうことです。  
しかし、その前に必要なのは、まずフレームとしての境界を切ることです。

- このフレームの Length はいくつか
    
- Type は HEADERS か
    
- Flags はどうなっているか
    
- Stream Identifier は何か
    
- payload はどこからどこまでか
    
- その payload のどこまでが HEADERS フレーム側の責務で、どこからが HPACK 側か
    

これが見えていない状態で HPACK の話に入ると、読解対象の層が混ざります。

したがって本章では、HEADERS を「全部理解する章」ではなく、**HEADERS の入口で止まる章**として扱います。

---

## HEADERS フレームを見るときの基本姿勢

HEADERS を読むときも、最初の順序は他のフレームと同じです。

まず境界を見る。  
次に共通ヘッダを読む。  
そのあとで Type 固有の最小限の情報を見る。  
そして、payload の先頭から先は HPACK block だと判断したら、一度止まる。

この「一度止まる」が本章の要点です。

ここで無理に全部を理解しようとしないことが、実は重要な技能です。  
HTTP/2 フレーム読解の役割と、HPACK 読解の役割を混同しないためには、「今どの層を読んでいるか」を意識しながら進む必要があります。

本章では、HEADERS を見たら次のように考えます。

まず、これは HTTP/2 フレームである。  
したがって共通ヘッダ 9 バイトを切る。  
Type が HEADERS なので、payload を HEADERS フレームの payload として見る。  
ただし、その中身はすべて HTTP/2 フレーム層の仕事ではない。  
ある地点から先は HPACK block として扱う必要がある。

この認識が持てれば、本章の目的はほぼ達成です。

---

## HEADERS フレームの何が難しいのか

HEADERS フレームが難しく感じられるのは、少なくとも三つの要素が同時に出てくるからです。

第一に、HTTP/2 フレームとしての共通ヘッダがあります。  
ここは他のフレームと同じです。

第二に、HEADERS フレーム固有の flags や payload の先頭側の扱いがあります。  
ここは Type 固有の領域です。

第三に、その先には HPACK で表現された header block fragment があります。  
ここは別レイヤーです。

つまり HEADERS は、「フレーム種別の一つ」であるだけでなく、「別の構造へ降りる入口」でもあります。  
この二重性があるため、SETTINGS や DATA よりも層の切れ目を意識しないと読みにくくなります。

本章では、この難しさを解決するために、次のような方針を取ります。

- まず共通ヘッダだけでフレーム境界を切る
    
- 次に HEADERS として最低限必要な分岐だけを見る
    
- そのあとで「ここから先は HPACK」と明示的に切る
    
- HPACK の中身は次章以降に回す
    

これにより、HEADERS を「いきなり全部読まなければならない塊」ではなく、「段階的に降りていく構造」として扱います。

---

## flags をどう見るか

この章では flags の完全な網羅を目的にしません。  
ただし、HEADERS を読むうえで flags を見ないわけにはいきません。  
なぜなら、HEADERS では flags が「payload をどこまでそのまま読めるか」に影響するからです。

ここで大事なのは、flags の名前を暗記することよりも、**flags によって後続の構造の見え方が変わる**と理解することです。

たとえば HEADERS では、flags によって

- このフレームだけでヘッダブロックが終わるのか
    
- 前に追加の情報がつくのか
    
- さらに後続フレームとの関係を意識すべきか
    

といったことが変わります。

本章では細かな組み合わせを追いません。  
ですが少なくとも、「Type が HEADERS だと分かっただけでは十分ではなく、flags を見てから payload の入口を考える必要がある」と押さえておきます。

つまり、HEADERS フレームの読み順はこうです。

まず共通ヘッダを読む。  
Type が HEADERS だと分かる。  
そのあとで flags を見て、payload の先頭の扱いを決める。  
そのうえで、header block fragment の入口を探す。

この順序を守ることが重要です。

---

## 実習 1　HEADERS と分かった時点で何が言えるか

### このケースで見ること

最初の実習では、まだ payload の中身に深入りせず、共通ヘッダを読んだだけで何が言えるかを確認します。

### データ

Length が 0 ではない。  
Type は HEADERS。  
Flags と Stream Identifier が続く。  
そういう最小の HEADERS フレームを考えます。

### まず何を見るか

最初に共通ヘッダ 9 バイトを切ります。  
Length を読みます。  
Type を見て HEADERS だと確認します。  
Flags を見ます。  
Stream Identifier を見ます。

ここまでは他のフレームと同じです。

### 次に何を見るか

Type が HEADERS だと分かった時点で、payload をただの生データとは見ません。  
この payload は、最終的にはヘッダ表現へつながるデータだと分かります。

ただし、この時点ではまだ「どのヘッダが入っているか」は分かりません。  
分かるのは、**このフレームの payload は HPACK block を含む領域になる**ということです。

### このケースで十分な到達点

このフレームを見て、

- まず HEADERS フレームとして共通ヘッダまでは読める
    
- payload は単なる DATA ではなく、header block を含む
    
- ただし header block の詳細はまだ別レイヤーの話である
    

と説明できれば十分です。

### このケースの意義

この実習の狙いは、「HEADERS と分かった瞬間に全部読めた気にならない」ことです。  
Type HEADERS は、終点ではなく入口です。

---

## 実習 2　先頭 9 バイトだけを読む

### このケースで見ること

今度はあえて、HEADERS フレームの先頭 9 バイトだけを独立して見ます。  
狙いは、HEADERS でも最初の仕事は共通ヘッダ読解だと確認することです。

### データ

HEADERS フレーム全体が与えられているとして、そのうち先頭 9 バイトだけを取り出して考えます。

### まず何を見るか

Length を見ます。  
Type を見ます。  
Flags を見ます。  
Stream Identifier を見ます。

このとき、まだ payload の中身には入らないことが大切です。

### 次に何を見るか

Length に従って、payload がどこから始まりどこまで続くかだけを確認します。  
それ以上はまだ読みません。

### このケースで十分な到達点

HEADERS フレームであっても、

- 最初にやることは他のフレームと同じ
    
- 先頭 9 バイトだけで、かなり多くのことが分かる
    
- payload の意味へ飛ぶ前に、まずフレーム境界を固める
    

と説明できれば十分です。

### このケースの意義

この実習は地味ですが重要です。  
HEADERS を難しく感じるときほど、最初の 9 バイトへ戻る必要があります。  
ここへ戻れれば、層の切れ目を見失いにくくなります。

---

## 実習 3　flags を見て payload の入口を考える

### このケースで見ること

ここでは、HEADERS フレームでは flags を見ないと payload の入口の理解が曖昧になることを確認します。

### データ

Type は HEADERS。  
Flags は 0 ではない。  
Length もある。  
そういう最小例を考えます。

### まず何を見るか

共通ヘッダ 9 バイトを読みます。  
Type が HEADERS であることを確認します。  
そのあとで flags に注目します。

### 次に何を見るか

ここで考えるべきなのは、flags が payload の読み方に何らかの影響を与えるということです。  
つまり、header block fragment の入口を雑に「payload の先頭から全部」と決めてよいとは限りません。

本章では、具体的なすべての分岐は追いません。  
ですが、少なくとも「flags を見たあとで payload の先頭構造を考える」という順序を押さえます。

### このケースで十分な到達点

このフレームを見て、

- HEADERS では flags が後続の読み方に影響する
    
- したがって payload の入口は Type だけでは決めきれない
    
- 共通ヘッダのあとに flags を確認してから中へ進む必要がある
    

と説明できれば十分です。

### このケースの意義

この実習によって、HEADERS では flags が「意味の飾り」ではなく、payload の構造理解に関わることが見えてきます。

---

## 実習 4　残りを header block fragment として切り出す

### このケースで見ること

ここでは、HEADERS フレームの payload のうち、HTTP/2 フレーム層として見たあとに、残りを一まとまりの header block fragment として切り出す感覚を作ります。

### データ

共通ヘッダを読み終え、flags から前置き部分の有無もひとまず確認できたとします。  
そのあとに残る領域を考えます。

### まず何を見るか

共通ヘッダから payload の開始位置を決めます。  
必要なら HEADERS フレーム固有の先頭側の情報を考慮します。  
そのうえで、残りのまとまりを見ます。

### 次に何を見るか

この残りが、header block fragment です。  
ここで初めて、「ここから先は HPACK の入力になる」と言えます。

重要なのは、この段階ではまだ「中身を展開しない」ことです。  
今やるべき仕事は、**HPACK の入力領域として切り出すこと**です。

### このケースで十分な到達点

このフレームを見て、

- HTTP/2 フレームとして読める範囲を先に確定する
    
- そのあとに残る領域を header block fragment として切り出す
    
- その領域は次章で扱う HPACK の対象である
    

と説明できれば十分です。

### このケースの意義

この実習が重要なのは、「中身を読むこと」より前に「どこを中身として渡すか」が必要だからです。  
HTTP/2 フレーム parser と HPACK decoder の間には、受け渡し点があります。  
その受け渡し点を見つけることが、本章の核心です。

---

## 実習 5　HEADERS を全部読まない練習をする

### このケースで見ること

最後の実習では、あえて「全部読まない」ことを練習します。  
これは消極的な態度ではなく、責務分離を守るための積極的な判断です。

### データ

任意の HEADERS フレームを考えます。  
Length があり、Type は HEADERS、flags も立っている。  
payload には何らかの header block が含まれています。

### まず何を見るか

共通ヘッダを読みます。  
Type を確認します。  
flags を見ます。  
Stream Identifier を見ます。  
payload 境界を決めます。

### 次に何を見るか

header block fragment の入口までを確認します。  
そして、そこで止まります。

### このケースで十分な到達点

このフレームを見て、

- ここまでは HTTP/2 フレームとして読める
    
- ここから先は HPACK の入力である
    
- したがって、この章ではここで止まる
    

と判断できれば十分です。

### このケースの意義

技術書では「分かるところまで全部説明する」ことが親切に見えることがあります。  
しかし実際には、層をまたぐ地点で一度止まるほうが、読者の理解を安定させます。  
本章の「止まる」は、そのためのものです。

---

## HTTP/2 フレームとしてどこまで読めるか

ここまでを整理すると、HEADERS フレームについて本章で読めるのは次の範囲です。

まず共通ヘッダ 9 バイトを切る。  
Length によってフレーム全体の境界を決める。  
Type が HEADERS であると確認する。  
flags を見て、payload の先頭側の扱いを考える。  
そして、header block fragment の入口を特定する。

この時点で、HTTP/2 フレームとしての仕事はかなり進んでいます。  
少なくとも、「どこからどこまでが HEADERS フレームで、そのうちどこが HPACK に渡る領域か」は説明できます。

逆に、本章ではまだ次のことはしません。

- header block fragment の中身を展開する
    
- static table や literal header を読む
    
- dynamic table を追う
    
- header list 全体を復元する
    

それらは次章以降の仕事です。

---

## HEADERS を読むときの責務分離

この章の内容を実装寄りに言い換えると、少なくとも次のような責務分離が見えてきます。

最初の parser は、フレーム共通ヘッダを読む。  
次の Type 別 parser は、HEADERS フレームとしての入口を読む。  
そして、その結果として得られた header block fragment を HPACK 側へ渡す。

この分け方が見えていると、HEADERS を「巨大な一枚岩」として見なくて済みます。  
また、後で HPACK のテストケースを読むときにも、「これはもうフレーム層ではなく HPACK 層の話だ」と判断しやすくなります。

つまり本章の中心は、HEADERS というフレームの説明そのものより、**レイヤーの切れ目を見つけること**にあります。

---

## この章でまだ扱わないこと

ここでは、まだ次のことは本格的に扱いません。

- HPACK の static table
    
- literal header field の表現
    
- Huffman の詳細
    
- dynamic table
    
- CONTINUATION をまたぐ複雑なケース
    
- ヘッダ一覧の最終復元
    

これらはすべて重要ですが、この章で先取りすると、HEADERS フレームの入口という主題がぼやけます。  
今はまだ、HTTP/2 フレームとしての HEADERS を読み、HPACK への入口で止まることを優先します。

---

## この章で残すべきメンタルモデル

HEADERS フレームを見たとき、最初にやることはヘッダ一覧の復元ではありません。  
まず共通ヘッダを読む。  
次に flags を見て payload の入口を考える。  
そのあとで、header block fragment の範囲を切り出す。  
そして、**ここから先は HPACK だと判断して一度止まる**。

本章で残したいのは、この読み方です。

つまり、HEADERS は一枚岩ではありません。  
HTTP/2 フレームとして読める範囲と、HPACK として読むべき範囲がある。  
この二つを切り分けられれば、本章の目的は達成です。

## まとめ

- HEADERS フレームは、HTTP/2 フレームであると同時に HPACK への入口でもある
    
- 最初にやることは他のフレームと同じで、共通ヘッダ 9 バイトを読むことである
    
- HEADERS では flags が payload の入口理解に影響する
    
- payload のうち、ある地点から先は header block fragment として HPACK に渡される
    
- 本章では HPACK の中身に深入りせず、入口で止まることが重要である
    
- HEADERS を読むとは、フレーム層と圧縮層の境界を見つけることでもある
    

## 次章への接続

次章では、いよいよ HPACK の最小例を読みます。  
そこで、今章で切り出した header block fragment を入力として、static table や literal header の最小ケースを読み、**HEADERS の先にある別の可変長構造**へ入っていきます。
