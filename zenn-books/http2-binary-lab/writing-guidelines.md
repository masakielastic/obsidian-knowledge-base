# HTTP/2 Binary Lab 執筆ガイド

## このファイルの役割

このファイルは、  
**『HTTP/2 Binary Lab（仮）』**  
の執筆・校正・リライトを支援するための品質基準と文章スタイルガイドである。

本実習編は、『バイナリハック入門 ― UTF-8・HTTP/2・HPACK で学ぶ構造と状態』の続編にあたり、本編で扱った  
**境界・分岐・妥当性・状態**  
という観点を、実際の最小テストデータに着地させて鍛えることを目的とする。

したがって執筆時には、仕様項目の網羅や実装コード全体の説明量よりも、  
**短いバイト列をどの順で読めばよいか**  
**どこで parser と validator を分けて考えるか**  
**どこから state が必要になるか**  
が見えることを優先する。

---

## 1. 実習編の目的

本実習編の目的は、UTF-8、HTTP/2 フレーム、HPACK を題材として、  
本編で獲得した読み方を、実際のデータに適用できるようにすることにある。

読者が最終的にできるようになるべきことは次の通りである。

- 短いバイト列を前にして、まず境界を見るという手順を取れる
- 先頭バイトや共通ヘッダから、何が分岐点かを説明できる
- 構造を切り出せることと、仕様上妥当であることを分けて考えられる
- 単発の parser で足りる範囲と、state が必要になる範囲を区別できる
- 最小の正常系と最小の不正系を比較しながら理解できる
- `http2-frame-test-case` のケースを読める
- `hpack-test-case` のケースを読める
- `h2spec` の失敗結果を見て、どの層の問題か説明できる

本実習編は、HTTP/2 実装のフルチュートリアルを提供する本ではない。  
**読む観点を、短いデータで反復練習し、公開 test case とツール出力へ接続する本**である。

---

## 2. 想定読者

- 『バイナリハック入門』を読み終えた読者
- あるいは同程度に、境界・分岐・妥当性・状態の考え方を理解している読者
- HTTP/2 の用語は知っているが、実際のバイト列や test case を読む自信がない読者
- 実装コード全体より先に、テストデータやログの見方を身につけたい読者
- HTTP/2 上級編や実装読解へ進む前に、観察の型を固めたい読者

補足として、本実習編の読者は、RFC の全面読解やフル実装をすぐに目指す人に限らない。  
むしろ、仕様や実装へ入る前に「どこから読むと構造が見えるのか」を身につけたい読者を主対象とする。

---

## 3. 実習編の基本方針

### 重視すること

- 最小の正常系
- 最小の不正系
- 1 ケース 1 論点
- 境界の見つけ方
- 分岐点の見つけ方
- parser と validator の責務分離
- state が必要になる瞬間の切り分け
- UTF-8 を助走にした可変長データの読解
- HEADERS と HPACK の境界の見極め
- 公開 test case の読解
- ツール出力を読むための橋渡し

### 深入りしすぎないこと

- RFC の網羅
- フレーム種別の完全カタログ化
- HPACK 実装の完全解説
- Huffman の詳細解読を長く続けること
- 特定言語でのフル実装チュートリアル
- ネットワーク I/O や TLS 周辺の詳細説明
- HTTP/2 上級編に属する本格的な状態遷移論
- RST_STREAM / GOAWAY / Sans-IO の本格設計論
- QUIC / QPACK の詳細

本実習編は、仕様を漏れなく説明する本ではない。  
**読む順序を定着させ、何が次の学習課題かを切り分けられるようにする本**である。

## 4. 本編との役割分担

### 本編の役割

本編は、次のような観点を作る本である。

- バイト列には境界がある
- 可変長データでは分岐の見方が重要になる
- 切り出せることと妥当であることは別である
- 状態を持つ構造では単発の parser だけでは足りない
- iterator / parser / validator / state 管理を分けて考える必要がある

### 実習編の役割

実習編は、その観点を実データに適用する本である。

- どこから境界を見るか
- どこで意味を急がず止まるか
- どの時点で validator を意識するか
- どこから先は state なしでは読めないか
- どの test case が何を試しているか

したがって、実習編では本編の要約を繰り返しすぎない。  
また、仕様の百科事典にも、実装コード解説の長文にも寄りすぎない。  
主役はあくまで、**短いバイト列と、それを読む手順**である。

## 5. 章の役割と書き分け

本実習編では、章ごとに役割をはっきり分ける。

### 導入章の役割

- その章で何を訓練するかを明示する
- 何をまだ扱わないかを明示する
- 読者がどこで混乱しやすいかを先に言葉にする
- その章で使う観点を宣言する

### 本文の役割

- 短いデータを順に読む
- 見る順序を固定する
- どこで止まるべきかを示す
- 正常系と不正系を比較する
- 具体例から読み方へ一般化する

### 整理ブロックの役割

- 比較表
- バイト列の分解図
- 正常系 / 不正系の対比
- parser / validator / state の責務表
- 公開 test case を読むときの着眼点

### 章末の役割

- その章で残すべきメンタルモデルを圧縮する
- 次章で何が増えるのかを示す
- 新しい情報を足さずに復習可能な形へ整理する

## 6. 文体上の基本姿勢

本編の方針を継承しつつ、実習編では次を特に重視する。

- 「私はこう思う」よりも、「本実習編ではここで止まる」「ここではこの粒度で読む」を優先する
- 随筆的な私語りではなく、読解手順と判断基準が見える文体を採る
- 説明はです・ます調を基本とする
- 一文一義を意識する
- 定義・区別・結論は言い切る
- 「なぜここで止まるのか」「なぜこの順で見るのか」を省略しない
- 見た目の印象ではなく、構造上の意味へ戻す
- 例は必要最小限に絞る
- 比喩は導入で使ってもよいが、本文では必ず用語へ戻す

### 実習編で特に重視する文体

- まず境界を見る
- 次に分岐点を見る
- 次に payload や後続バイトに入る
- 最後に妥当性や state の話へ進む

この順序が文そのものから見えることが望ましい。

## 7. 実習編で特に守るべき補助原則

- UTF-8 は文字コードの説明としてではなく、分岐付き可変長データの教材として扱う
- HTTP/2 フレームは仕様項目の列挙ではなく、構造化バイナリの教材として扱う
- HEADERS は「HTTP/2 フレーム」と「HPACK の入口」の境界として扱う
- HPACK は圧縮理論の説明で終わらせず、状態付き構造として扱う
- 公開 test case は「答え合わせ」ではなく、「何を試しているかを読む対象」として扱う
- ツールは本文前半の主役ではなく、後半の橋渡しとして使う
- 個別知識を増やすより、読む順序を定着させることを優先する

## 8. 実習問題の設計原則

### 1. 最小性

- 最小の正常系を優先する
- 最小の不正系を対で置く
- 余計なフィールドや余計な条件をできるだけ外す

### 2. 1 ケース 1 論点

- 一つの演習で複数の難しさを同時に持ち込みすぎない
- そのケースで何を見てほしいかを明確にする
- 章をまたぐ論点は、その章では先取りしない

### 3. 比較可能性

- 正常系と不正系を近い形で並べる
- 何バイト違うのか
- どのフラグが違うのか
- どの前提 state が違うのか
- そうした差分が見える形にする

### 4. 責務の切り分け

- どこまでが iterator 的な走査か
- どこからが parser か
- どこで validator が働くか
- どこから state 管理が必要か

この線引きを、各演習で明示する。

### 5. 段階性

- まず単発で読めるもの
- 次に不正系
- その次に、状態が必要になるもの
- 最後に、公開 test case やツール出力

この順で難度を上げる。

---

## 9. 各章で必ず明示すること

各章では、冒頭または前半で次の項目を明示する。

- この章で何を訓練するか
- この章の主な観点  
  - 境界  
  - 分岐  
  - 妥当性  
  - 状態
- この章では何をまだ扱わないか
- この章で置く最小データの意味
- どこまで読めれば十分か

これにより、読者が「全部理解しなければ次へ進めない」と感じにくくなる。

## 10. 音声向け情報と視覚向け情報の分離

本編の方針を継承し、本実習編でも  
**声で追いやすい情報** と  
**目で見たほうが理解しやすい情報**  
を分ける。

### 本文に置くもの

- 読む順序
- どこで止まるか
- どこが分岐点か
- 正常系と不正系の意味の違い
- parser / validator / state の責務差
- その章で残すべきメンタルモデル

### 整理ブロックに置くもの

- バイト列の分解
- フィールド表
- 比較表
- フラグの差分
- 正常系 / 不正系の並列表
- 公開 test case の項目対応
- ツール出力と生データの対応

### 運用ルール

- 長い列挙は本文でだらだら続けない
- 一覧は整理ブロックへ出す
- バイト列例は必要最小限に絞る
- 同じ章で表と長文説明を重複させすぎない
- 章末は復習しやすい形式に固定する

## 11. 見出し構造の基準

見出しレベルは全章で統一する。

### 基本ルール

- 章タイトル：`##`
- 章内セクション：`###`
- 「この章で残すべきメンタルモデル」：`###`
- 「まとめ」：`##`

### 禁止事項

- 章ごとに見出しレベルを変える
- 一部の章だけ極端に細分化する
- 章末整理の見出し名を毎回変える

## 12. 章フォーマットの基準

各章は原則として次の流れで構成する。

### 1. 章の狙い

- この章で何を訓練するか
- 何をまだ扱わないか
- どの観点を主に使うか

### 2. 導入

- その章で起きやすい誤解を短く出す
- 必要なら短い導入対話を置いてよい
- ただし、対話だけで章を進めない

### 3. 本文

- 最小データを順に読む
- 読む順序を固定する
- 途中で止まる理由を示す
- 正常系と不正系を比較する
- 個別例から一般化する

### 4. 整理ブロック

- 比較表
- バイト列の分解図
- 責務分担表
- 公開 test case との対応

### 5. この章で残すべきメンタルモデル

- 新しい情報を増やさない
- その章の中核を短く圧縮する
- 次章で使う抽象を残す

### 6. まとめ

- 3〜5 項目程度
- 新しい情報を追加しない
- 章タイトルと対応する内容に絞る

### 7. 次章への接続

- 1〜2 文で十分
- 次に増える難しさを示す

## 13. 実習章での説明順序の基準

演習データを扱う本文では、原則として次の順序を守る。

### 基本順序

1. まず全体の長さを見る  
2. どこまでが境界かを見る  
3. 先頭側の情報で分岐点を見る  
4. 必要なら payload や後続バイトへ進む  
5. そこで妥当性を見る  
6. 必要なら state の有無を確認する  
7. 最後に、そのケースが何を試しているかをまとめる

### 避けるべき順序

- いきなり意味や用途から入る
- まだ境界が切れていないのに payload 解釈へ飛ぶ
- 妥当性の議論を、構造読解の前に持ち込む
- state が必要なケースを、単発ケースとして説明する

---

## 14. 公開 test case を扱う章の書き方

`http2-frame-test-case` や `hpack-test-case` を扱う章では、単に「こういうリポジトリがある」と紹介するだけで終わらせない。

### 必ず示すこと

- そのケースは何を試しているのか
- どの層のテストなのか
- 正常系か不正系か
- 単発で読めるか、state が必要か
- どの項目を先に見るべきか
- どこまで自力で説明できれば十分か

### 避けること

- JSON 項目の列挙だけで終わること
- テスト名の紹介だけで終わること
- リポジトリ構造の説明に紙幅を使いすぎること
- 本文が単なる利用手順書になること

---

## 15. ツールを扱う章の書き方

`nghttp`、`h2i`、`nghttpd`、`h2spec` を扱うときは、ツールそのものの完全マニュアルにしない。

### 位置づけ

- `nghttp` はフレームログとの対応を見るための橋渡し
- `h2i` は低レベルな送受信の観察に入るための橋渡し
- `nghttpd` はローカル観測対象を用意するための補助
- `h2spec` は conformance 結果を読むための入口

### 必ず示すこと

- そのツールが本文前半のどの演習とつながるか
- 生データや test case と比べて、何が省略され、何が見やすくなるか
- ツールの出力をどの層の問題として読むか

### 避けること

- インストール手順の長文化
- オプション一覧の羅列
- 実習編の本体をツール操作説明に置き換えること

## 16. 校正時の最重要品質基準

校正では、次の観点を最優先で確認する。

### A. 読む順序が見えるか

- どこから見ればよいかが明示されているか
- 意味を急がず、まず境界を見る流れになっているか
- 読者が途中で迷子にならないか

### B. 観点が明確か

- どこが境界か
- 何が分岐点か
- 何を妥当性として扱うか
- どこで state が入るか
- どこまでが parser の責務か
- どこから validator / state 管理の責務か

### C. 1 ケース 1 論点になっているか

- 一つの実習に難しさを詰め込みすぎていないか
- 正常系と不正系の差が見えるか
- 比較対象の置き方が適切か

### D. 抽象と具体の接続があるか

- UTF-8 / HTTP/2 / HPACK の具体例が抽象に戻れているか
- バイト列例が単なる例示で終わっていないか
- 公開 test case やログ読解につながっているか

### E. 章の問いに答えているか

- 章冒頭で掲げた訓練目標に本文が答えているか
- 途中で話題が広がりすぎていないか
- 上級編の話題が侵食していないか

### F. 圧縮が効いているか

- 長文のあとに整理があるか
- 章末で復習できる形になっているか
- 抽象語だけが増えていないか

## 17. 具体的なレビュー観点

### 用語

- 初出時に必要な説明があるか
- 同じ概念に複数の言い換えを乱用していないか
- iterator / parser / validator / state が混同されていないか
- フレーム層と HPACK 層の用語が不用意に混ざっていないか

### 本文

- 読む手順が見えるか
- どこで止まるべきかが示されているか
- 「なぜここで止まるか」が書かれているか
- 個別例を一般化できているか

### 実習設計

- 最小性が守られているか
- 不正系が適切に対で置かれているか
- その章の前提を超えた難しさが紛れ込んでいないか
- 状態依存ケースを早すぎる段階で出していないか

### 構成

- 1 セクション 1 テーマになっているか
- 話題の飛躍がないか
- 整理ブロックが本文の代わりになっていないか
- 章末のメンタルモデルとまとめが重複しすぎていないか

### 可読性

- 一文が長すぎないか
- 16 進数列が長すぎて本文の主線を壊していないか
- 例が多すぎて焦点がぼけていないか
- 抽象語が連続しすぎていないか

## 18. まとめの基準

各章の最後の「まとめ」は、次の基準に従う。

- 箇条書き 3〜5 項目
- 新しい情報を追加しない
- 章タイトルと対応する内容に絞る
- 次章へ必要な前提が残るようにする
- 読む観点の区別が再確認できるようにする

良いまとめは、次を短く再確認できるものとする。

- どこが境界だったか
- 何が分岐点だったか
- どこで validator が必要だったか
- どこから state が必要になったか
- 次章で何が増えるか

## 19. リライト方針

校正や改善では、必要に応じて次を行う。

- 冗長な説明の圧縮
- 長い実習説明の分割
- 具体例と抽象の対応の明確化
- 正常系 / 不正系の対比強化
- 公開 test case への接続補強
- ツール説明の圧縮
- 上級話題の切り出し
- 章末まとめの統一

ただし、次は避ける。

- 仕様項目の網羅へ戻すこと
- 実装コード解説の本に変えてしまうこと
- 一つの章に複数の難しさを詰め込みすぎること
- 導入の短さを失って長い講義に変えること
- 書籍全体のトーンを章ごとに変えること

## 20. 実習編で特に守るべき一文

この実習編は、HTTP/2 と HPACK のデータを、  
**「仕様の知識として眺める対象」ではなく、「境界・分岐・妥当性・状態の観点から段階的に読める構造」として扱えるようにする本**  
である。

執筆・校正・改善の判断は、常にこの方針に従って行う。

## 21. シリーズ内での位置づけを崩さないための注意

本実習編は、本編の続編であり、HTTP/2 上級編や実装設計論への橋渡しでもある。  
したがって、次のバランスを崩さない。

- 本編よりは具体的に手を動かす
- ただし上級編の実装論へ踏み込みすぎない
- 中級書の API 設計論とも役割を混同しない
- 本実習編は「観点を実データで鍛える本」であることを保つ
- 仕様暗記やツール操作習熟ではなく、読解の型の定着を中心にする
