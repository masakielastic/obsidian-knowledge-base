# HTTP/2 Binary Lab テストデータ設計原則

## このファイルの役割

このファイルは、  
**『HTTP/2 Binary Lab（仮）』**  
で扱う実習用テストデータの設計原則を整理するための文書である。

本実習編では、仕様を項目ごとに説明することよりも、  
**短いバイト列をどの順で読み、どこで止まり、どこで判断を分けるか**  
を身につけることを優先する。

したがって、テストデータも「現実にありそうな複雑なもの」を最初から置くのではなく、  
**最小の正常系・最小の不正系・1 ケース 1 論点**  
を基本として設計する。

---

## 1. テストデータ設計の基本方針

本実習編のテストデータは、単なる例示ではなく、  
**読む観点を訓練するための教材**  
として設計する。

そのため、テストデータ設計では次の四つを最優先とする。

- 境界が見えること
- 分岐点が見えること
- 妥当性判定の位置が見えること
- 状態が必要になる瞬間が見えること

個々のデータは、RFC の仕様項目を網羅するためではなく、  
**読者が「どこから読めばよいか」を学ぶために置く**。

---

## 2. 最小性を優先する

### 基本原則

各テストデータは、できるだけ小さくする。  
ここでいう「小さい」とは、単にバイト数が短いという意味だけではない。  
次のような意味で小さくする。

- 余計なフィールドを含めない
- 論点を増やさない
- まだ扱っていない層を混ぜない
- 一つのケースで複数の難しさを同時に持ち込まない

### ねらい

最小性を優先する理由は、読者が  
**どの部分でつまずいているのかを見失わないようにするため**  
である。

複雑なキャプチャや実装由来の大きなデータは、現実には有用である。  
しかし教育の入口としては、次の難しさが同時に混ざりやすい。

- 境界切り出し
- フィールド解釈
- 妥当性判定
- stream 文脈
- HPACK 解読
- state 管理
- ログ表現の省略
- 実装固有の都合

本実習編では、これらをまとめて出さず、  
**小さく切り分けて順に出す**。

---

## 3. 1 ケース 1 論点を守る

### 原則

一つのテストデータには、原則として一つの主論点だけを持たせる。

たとえば次のように切り分ける。

- フレーム境界だけを見るケース
- Length の読み方だけを見るケース
- SETTINGS ACK の妥当性だけを見るケース
- HEADERS の入口だけを見るケース
- static table の HPACK だけを見るケース
- dynamic table による state 必要性だけを見るケース

### 避けるべき例

- HEADERS フレームの読解と、HPACK の状態依存と、不正な長さを一つのケースで同時に扱う
- 一つのケースに複数の flags の意味を詰め込む
- フレーム不正と stream 文脈不正を同時に扱う
- HPACK の literal と dynamic table と Huffman を一度に導入する

### 理由

実習編で大事なのは、「正答」ではなく  
**どの観点で読むべきかを定着させること**  
である。

そのため、一つのケースで複数の難しさが競合しないようにする。

---

## 4. 正常系と不正系を対で置く

### 原則

可能な限り、正常系だけでなく、それに対応する最小の不正系を対で置く。

### 例

- 正常な UTF-8 と、不正な UTF-8
- 空の SETTINGS と、不正な SETTINGS ACK
- 正しい Length と、Length 不整合
- static table の単発ケースと、前提 state がないと読めないケース

### 理由

正常系だけでは、読者は  
**「どこまでが parser の仕事で、どこから validator の仕事か」**  
を見分けにくい。

不正系を対で置くことで、次が見えやすくなる。

- 構造は切り出せるが妥当ではない
- 長さは足りているが許容範囲ではない
- 単発では読めるように見えるが、実際には state が要る

### 運用上の注意

不正系は、正常系との差分が見えるように設計する。  
差分はできるだけ少なくする。

よい差分の例は次のようなもの。

- 1 バイトだけ違う
- フラグだけ違う
- 第 2 バイトの範囲だけ違う
- 前提 state の有無だけ違う

---

## 5. 差分が見えるように並べる

### 原則

テストデータは単独で置くだけでなく、  
**比較によって意味が見えるように並べる**。

### 並べ方の基本

- 正常系 → 不正系
- 単純なケース → 少し複雑なケース
- 単発で読めるケース → state が必要なケース
- 生データ → 公開 test case
- 手で読む → ツール出力に接続する

### 比較の軸

比較の軸は、章ごとに一つに絞る。

- 何バイト違うか
- どのフィールドが違うか
- どのビットが違うか
- どの前提が違うか
- どの責務に属する違いか

### 理由

差分が見えないままデータを並べると、読者は個別ケースを暗記しやすくなる。  
本実習編の目的は暗記ではないので、  
**どの違いが、どの判断につながるか**  
が見える並べ方を採る。

---

## 6. 層を混ぜすぎない

### 原則

テストデータは、できるだけ層ごとに分けて導入する。

### この実習編で意識する層

- UTF-8 のバイト列
- HTTP/2 フレーム共通ヘッダ
- HTTP/2 フレーム payload
- HEADERS フレームの入口
- HPACK block
- HPACK の state
- 公開 test case
- ツール出力
- conformance result

### ねらい

たとえば HEADERS フレームを扱うときでも、最初から HPACK の内部状態まで同時に説明しない。  
まずは、

- フレーム共通ヘッダ
- HEADERS の payload が始まる地点
- ここから先は HPACK だという境界

を見せる。

そのうえで、次章以降で HPACK 自体に入る。

### 理由

層を混ぜすぎると、読者は  
「今どのレイヤーを読んでいるのか」  
を見失う。

本実習編では、**層をまたぐ瞬間自体を教材化する**。  
だからこそ、日常的な説明ではなく、層の切れ目を意識したデータ設計が必要になる。

---

## 7. state は遅れて導入する

### 原則

state が必要になるケースは、十分に単発読解を経験したあとで導入する。

### 先に置くべきもの

- 単発で完結する UTF-8 の正常系 / 不正系
- 単独で読める HTTP/2 フレーム
- 最小の DATA / SETTINGS / GOAWAY
- static table 中心の HPACK

### 後で置くべきもの

- dynamic table の追加
- 以前の header set に依存するケース
- story を追わないと意味が定まらない `hpack-test-case`
- `h2spec` の結果に現れる文脈依存の失敗

### 理由

state が必要なケースは、それだけで難しさの種類が変わる。  
単発読解がまだ不安定な段階で state を入れると、

- 今は境界でつまずいているのか
- 妥当性でつまずいているのか
- state 依存でつまずいているのか

が見えにくくなる。

### 補助原則

state を扱う章では、必ず  
**何が前提として保持されているか**  
を文章または整理ブロックで明示する。

---

## 8. UTF-8 は助走として使う

### 原則

UTF-8 のテストデータは、HTTP/2 に入る前の助走として使う。  
ただし、文字コードそのものの解説に紙幅を使いすぎない。

### 目的

UTF-8 を使う目的は、次を小さな題材で確認することである。

- 可変長データでは先頭バイトが分岐点になる
- 長さが合っていても妥当とは限らない
- parser と validator は別である
- 正常系と不正系の差は少数バイトに現れる

### テストデータ選定の基準

UTF-8 では、次の三段階が見えるように選ぶ。

- 長さの違い
- 分岐条件の違い
- 妥当性条件の違い

### 置くべきケースの型

- 1 バイト ASCII
- 2 バイトの正常例
- 3 バイトの正常例
- 4 バイトの正常例
- `E0` と `ED` の比較
- `F0` と `F4` の比較
- continuation byte の不正例
- 途中で切れた例

### 理由

UTF-8 は本実習編の主題ではないが、  
**可変長フォーマットをどう読むか**  
という感覚を育てるには非常に適している。

---

## 9. HTTP/2 フレームでは共通ヘッダを独立して扱う

### 原則

HTTP/2 フレームのテストデータは、まず共通ヘッダ 9 バイトの読解を独立した段階として扱う。

### 目的

次を分けて学ばせるためである。

- フレーム境界を切ること
- Length / Type / Flags / Stream ID を読むこと
- その後に payload の種別ごとの意味へ進むこと

### 置くべきケースの型

- 接続プレフェース
- 空の SETTINGS
- SETTINGS ACK
- 1 バイト payload の DATA
- Stream ID 0 と 1 の比較
- Length が 0 の例
- Length が 1 の例

### 理由

ここを独立させないと、読者は最初からフレーム種別の意味や payload 解釈に意識を取られやすい。  
本実習編では、  
**まず共通ヘッダが切れること**  
を基礎技能として扱う。

---

## 10. HEADERS は入口だけで一度止まる

### 原則

HEADERS フレームは、いきなり全部を説明しない。  
まずは  
**HTTP/2 フレームとしてどこまで読めるか**  
で一度止まる。

### 置くべきこと

- HEADERS フレームの共通ヘッダ
- flags の最低限の読み方
- payload の始点
- ここから先が HPACK block であること

### 後回しにすること

- HPACK の詳細解読
- dynamic table
- Huffman の詳細
- 複雑な header block の断片化

### 理由

HEADERS は、HTTP/2 フレーム読解と HPACK 読解の境界を意識するための重要な教材である。  
だからこそ、  
**一度入口だけで止まるケース**  
が必要になる。

---

## 11. HPACK は static から始める

### 原則

HPACK のテストデータは、まず static table だけで読めるケースから始める。

### 段階

1. static table の単発ケース  
2. literal header の最小ケース  
3. state を使わない複数ケース  
4. dynamic table を導入するケース  
5. story を追う必要があるケース

### 理由

HPACK の難しさは、圧縮方式そのものよりも、  
**可変長・複数表現・状態依存が同時に出てくること**  
にある。

したがって最初は、state のないケースから始めて、  
**HEADERS の先にある別レイヤーを読む感覚**  
を先に作る。

---

## 12. 公開 test case は到達目標として配置する

### 原則

`http2-frame-test-case` と `hpack-test-case` は、最初の素材ではなく、  
ある程度の最小演習を終えたあとの到達目標として置く。

### 理由

公開 test case は非常に有用だが、読者にとってはすでに「少し加工された教材」である。  
何を見ればよいかがまだ定まっていない段階で読むと、

- JSON 項目が多く見える
- 何が主論点か分からない
- 正常系と不正系の違いが抽象的に見える

という状態になりやすい。

### 運用方針

公開 test case に入る前に、実習編側で次を先に経験させる。

- 最小の生データ
- 正常系 / 不正系の対比
- 単発読解
- HEADERS と HPACK の境界
- state が必要になる瞬間

そのうえで、公開 test case を  
**「既に知っている読み方を、少し大きな素材に適用する対象」**  
として扱う。

---

## 13. ツール出力は終盤で接続する

### 原則

`nghttp`、`h2i`、`nghttpd`、`h2spec` などのツールを使うケースは、  
実習編の後半または終盤で接続する。

### 理由

ツール出力は便利だが、すでに整理された情報でもある。  
そのため、最初から使うと読者は

- 生データで何が省略されているか
- ツールがどの責務まで肩代わりしているか
- 自分がどこを読んでいるのか

を見失いやすい。

### 位置づけ

- `nghttp` はフレームログとの対応を確認する素材
- `h2i` は低レベルなやり取りの観察へ入る素材
- `nghttpd` はローカルの観測対象
- `h2spec` は conformance result を読む素材

つまり、ツール出力は  
**手で読めるようになったものを、観測可能な環境へ接続するための最後の段階**  
として使う。

---

## 14. ケースの粒度を章ごとにそろえる

### 原則

章ごとに扱うケースの粒度をそろえる。  
一部の章だけ極端に大きなデータを扱わない。

### ねらい

各章で読者に求める動作を揃えるためである。

- この章は境界を見る章
- この章は最小フレームを読む章
- この章は不正系を見る章
- この章は HEADERS の入口で止まる章
- この章は state が必要になる章

こうした章の役割に対して、データの大きさや複雑さが極端にずれると、  
章の目的がぶれやすい。

### 補助原則

もし大きめのケースを置くなら、その章では必ず  
**どこまで読めれば十分か**  
を明示する。

---

## 15. ケース名と説明文の原則

### ケース名の原則

ケース名は、できるだけ  
**何を見てほしいか**  
が分かるものにする。

よい例：

- 空の SETTINGS を読む
- SETTINGS ACK の妥当性を見る
- HEADERS の入口で止まる
- static table の単発ケース
- dynamic table が必要になる最小例

避けたい例：

- 例1
- パターンA
- テストデータ3
- いろいろな HEADERS

### 説明文の原則

ケース説明では、次を短く書く。

- このケースの主論点
- どの観点で読むか
- どこまで読めれば十分か
- 何はまだ扱わないか

---

## 16. 実習問題ごとの最小テンプレート

各実習問題は、原則として次のテンプレートで設計する。

### 1. このケースで見ること

- 主論点を一文で書く

### 2. データ

- バイト列または test case の抜粋を示す

### 3. まず何を見るか

- 境界
- 先頭バイト / 共通ヘッダ
- 分岐点

### 4. 次に何を見るか

- payload
- 後続バイト
- flags
- 文脈

### 5. 妥当性または state

- どこで validator が入るか
- どこから state が必要か

### 6. このケースで十分な到達点

- 何を説明できればよいかを明示する

このテンプレートを守ることで、章ごとの説明が安定する。

---

## 17. 悪いテストデータ設計の例

本実習編では、次のような設計を避ける。

### 悪い例 1
最初から大きなキャプチャを置き、どこを見ればよいかを説明しない。

### 悪い例 2
一つのケースに複数の不正を入れる。

### 悪い例 3
正常系だけを並べて、validator の役割が見えない。

### 悪い例 4
HEADERS を出した瞬間に HPACK の全要素をまとめて説明する。

### 悪い例 5
dynamic table を早すぎる段階で出して、単発読解が定着していない。

### 悪い例 6
ツール出力をそのまま貼り、元の生データとの対応を示さない。

### 悪い例 7
公開 test case の JSON 項目を列挙するだけで、何を試しているかを説明しない。

---

## 18. よいテストデータ設計の例

### よい例 1
空の SETTINGS と SETTINGS ACK を並べて、flags だけの差で意味が変わることを見せる。

### よい例 2
UTF-8 の正常例と不正例を 1 バイト差で並べ、長さと妥当性の違いを見せる。

### よい例 3
HEADERS の先頭 9 バイトだけをまず読ませ、残りを HPACK block として切り分ける。

### よい例 4
static table の単発ケースを読ませたあとに、dynamic table が必要なケースを置く。

### よい例 5
公開 test case に入る前に、実習編側で同型の最小問題を先に置く。

### よい例 6
`h2spec` の結果を示す前に、読者が connection preface や GOAWAY を既に学んでいる状態を作る。

---

## 19. 実習編で特に守るべき一文

本実習編のテストデータは、  
**「仕様の断片を並べるための材料」ではなく、「境界・分岐・妥当性・状態を読めるようにするための教材」**  
として設計する。

テストデータの選定・圧縮・並べ替えは、常にこの方針に従って行う。

---

## 20. まとめ

本実習編のテストデータ設計では、次の原則を守る。

- 最小性を優先する
- 1 ケース 1 論点を守る
- 正常系と不正系を対で置く
- 差分が見えるように並べる
- 層を混ぜすぎない
- state は遅れて導入する
- 公開 test case は到達目標として置く
- ツール出力は終盤で接続する
- テストデータを、読む観点を訓練する教材として扱う

この原則に従うことで、実習編全体が  
**「バイト列をどう読むか」を段階的に身につけるための一貫した教材**  
としてまとまりやすくなる。
