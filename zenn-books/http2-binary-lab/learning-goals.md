# HTTP/2 Binary Lab 学習目標

## このファイルの役割

このファイルは、  
**『HTTP/2 Binary Lab（仮）』**  
における学習目標を整理するための文書である。

本実習編は、『バイナリハック入門 ― UTF-8・HTTP/2・HPACK で学ぶ構造と状態』の続編として、  
本編で扱った  
**境界・分岐・妥当性・状態**  
という観点を、実際の最小テストデータに適用して鍛えることを目的とする。

したがって、本実習編の学習目標は、仕様項目の暗記量ではなく、  
**短いバイト列・公開 test case・ツール出力を、どの順で見て、どこで判断を分けられるか**  
という読解力として定義する。

---

## 1. 学習目標の全体像

本実習編の最終目標は、次の一文で表せる。

**HTTP/2 と HPACK の最小バイト列を、境界・分岐・妥当性・状態の観点から段階的に読み、最終的に公開 test case と conformance tool の結果を手がかりに、何が起きているかを説明できるようになること。**

この目標は、さらに次の三段階に分かれる。

- 最小バイト列を手で読める
- 公開 test case を読める
- ツールや conformance result を読める

---

## 2. 実習編の基本到達目標

本実習編を読み終えた読者は、少なくとも次のことができるようになることを目指す。

### A. 最小データを前にして読む順序を取れる

- まず境界を見る
- 次に先頭バイトや共通ヘッダを見る
- 何が分岐点かを確認する
- その後で payload や後続バイトへ進む
- 最後に妥当性や state の有無を考える

### B. 構造と妥当性を分けて考えられる

- バイト列を切り出せることと、仕様上正しいことを別に考えられる
- parser の仕事と validator の仕事を混同しない
- 「読めるが正しくない」ケースを説明できる

### C. state が必要になる瞬間を区別できる

- 単発で読めるケース
- 単発では読めず、前提 state が必要なケース
- 同じ raw-data でも前提 state によって解釈が変わるケース

を区別できる。

### D. 実データと公開資産に接続できる

- Qiita の最小 HTTP/2 クライアントに埋め込まれた定数を読める
- `http2-frame-test-case` のケースを読める
- `hpack-test-case` のケースを読める
- `h2spec` の結果を見て、どの層の問題かを説明できる

---

## 3. 本実習編が目指す読解力

本実習編では、知識量そのものよりも、次の読解力を育てる。

### 1. 境界読解力

- どこまでが一つのまとまりかを見抜ける
- 接続プレフェースとフレームの違いを区別できる
- 共通ヘッダと payload の境界を切れる
- HEADERS フレームの入口で、「ここから先は HPACK」と止まれる

### 2. 分岐読解力

- UTF-8 の先頭バイトで長さと条件が変わることを説明できる
- HTTP/2 の Length / Type / Flags / Stream ID が何を分岐させるかを説明できる
- static table / literal / dynamic table の違いを見分けられる

### 3. 妥当性読解力

- 長さが合っているだけでは不十分だと説明できる
- 不正な SETTINGS ACK を見分けられる
- stream 0 と非 0 の文脈不整合を説明できる
- UTF-8 の continuation byte 条件違反と、HTTP/2 のフレーム不整合を同じ認知フレームで理解できる

### 4. 状態読解力

- static table の単発ケースと dynamic table の状態依存ケースを区別できる
- 以前の入力に依存して次の解釈が変わることを説明できる
- state manager が必要になる理由を言語化できる

### 5. 責務分解力

- iterator 的走査
- parser
- validator
- HPACK decoder
- state 管理
- connection / stream 文脈

を区別しながら読める。

---

## 4. 章ごとの学習目標

以下では、章ごとに何をできるようになるべきかを整理する。

---

## 第0章　この実習編で何を訓練するのか

### 到達目標

- 本編と実習編の役割の違いを説明できる
- 実習編が仕様の網羅ではなく、読む手順の訓練であることを説明できる
- 各章で何を見るのか、何をまだ扱わないのかを意識して読める
- 境界・分岐・妥当性・状態の四観点を、実習の観点として再確認できる

### 到達判定の目安

- 「この本は何を覚える本か」ではなく「何を読めるようになる本か」で説明できる

---

## 第1章　UTF-8 をウォームアップにする

### 到達目標

- UTF-8 を単なる 1〜4 バイト分類ではなく、分岐付き可変長データとして説明できる
- 先頭バイトによって後続バイト条件が変わることを説明できる
- 途中で切れたケースと、長さは足りているが不正なケースを区別できる
- UTF-8 で parser と validator を分けて考えられる

### 到達判定の目安

- 正常例と不正例を見て、どこで境界を切り、どこで妥当性違反と判断するかを説明できる

---

## 第2章　HTTP/2 フレームの境界を読む

### 到達目標

- 接続プレフェースがフレームではないと説明できる
- HTTP/2 フレーム共通ヘッダ 9 バイトを切り出せる
- Length / Type / Flags / Stream ID の位置を説明できる
- まだ payload の意味に飛ばず、まず共通ヘッダを読む姿勢を取れる

### 到達判定の目安

- 短いフレーム例を見て、共通ヘッダだけを正しく読み上げられる

---

## 第3章　最小フレームを読む

### 到達目標

- 最小の SETTINGS、SETTINGS ACK、DATA、GOAWAY を読める
- 共通ヘッダの後に、どこから種別ごとの解釈が必要になるかを説明できる
- Qiita の最小クライアントに埋め込まれた定数を、周辺コードから切り離して読める

### 到達判定の目安

- 最小の正常系フレームを前にして、「何のフレームで、どこまで読めるか」を説明できる

---

## 第4章　壊れたフレームを読む

### 到達目標

- Length 不整合を見分けられる
- 不正な SETTINGS ACK を見分けられる
- stream 0 と非 0 の文脈差を説明できる
- 構造読解と妥当性判定を分けて説明できる

### 到達判定の目安

- 「このケースは切り出せるが妥当ではない」と言える
- どこが parser の範囲で、どこから validator の範囲かを言える

---

## 第5章　HEADERS フレームの入口に立つ

### 到達目標

- HEADERS フレームを、HTTP/2 フレームとしてどこまで読めるか説明できる
- HEADERS の payload が HPACK block の入口であることを説明できる
- まだ HPACK 全体を理解しなくても、フレーム層として読む範囲を切り分けられる

### 到達判定の目安

- HEADERS フレームを見て、「ここから先は別レイヤー」と言える

---

## 第6章　HPACK の最小例を読む

### 到達目標

- static table を使う最小ケースを読める
- literal header の最小ケースを読める
- raw-data と headers の対応を説明できる
- HPACK を「HEADERS の中にある別の可変長構造」として理解できる

### 到達判定の目安

- 単発の HPACK 最小例について、どこで区切り、どの表現を使っているかを説明できる

---

## 第7章　状態が必要になると何が変わるか

### 到達目標

- dynamic table を導入する最小ケースを読める
- 単発では読めないケースを見分けられる
- 前提 state の有無が解釈にどう影響するかを説明できる
- state manager が必要になる理由を言語化できる

### 到達判定の目安

- 同じ raw-data でも、前提 state によって解釈が変わることを説明できる

---

## 第8章　`http2-frame-test-case` を読む

### 到達目標

- `http2-frame-test-case` のケースを読める
- そのケースが正常系か不正系かを説明できる
- そのケースが何を試しているかを言葉にできる
- フレーム構造の読解から、テスト意図の読解へ進める

### 到達判定の目安

- 公開 test case を見て、「どのフレームの、どの論点のテストか」を説明できる

---

## 第9章　`hpack-test-case` を読む

### 到達目標

- `hpack-test-case` の `headers` と `raw-data` の対応を説明できる
- static / literal / dynamic の違いを見分けられる
- state なしで読めるケースと、story を追う必要があるケースを区別できる
- HPACK の公開 test case の主論点を説明できる

### 到達判定の目安

- ある test case について、「これは単発で読めるか、前提 state が必要か」を言える

---

## 第10章　ツール・ログ・conformance を読む

### 到達目標

- `nghttp` の verbose 出力を、フレーム読解と対応づけられる
- `h2i` が何を観察するための道具かを説明できる
- `nghttpd` をローカル観測対象として使う意味を説明できる
- `h2spec` の結果を見て、どの層の問題が指摘されているかを説明できる
- 公開 test case と conformance tool の役割の違いを説明できる

### 到達判定の目安

- `h2spec` の失敗結果を見て、「これは frame parser の問題か、validator の問題か、state や接続管理の問題か」を概ね説明できる

---

## 第11章　この先どこへ進むか

### 到達目標

- 最小クライアント実装を読むとき、どこがバイナリ読解の対象かを切り分けられる
- frame parser / validator / HPACK decoder / state manager などの責務分担を意識して実装を見られる
- 今後の発展課題を、自分の現在地に応じて選べる

### 到達判定の目安

- 「次に何を学ぶべきか」を、闇雲ではなく責務単位で説明できる

---

## 5. 三段階の到達目標

本実習編の目標は、初級・中級・橋渡しの三段階で整理できる。

### 第1段階　最小データを手で読める

ここでは次ができればよい。

- UTF-8 の最小正常系 / 不正系を読める
- HTTP/2 フレーム共通ヘッダを読める
- 最小の SETTINGS / DATA / GOAWAY / HEADERS を読める
- 正常系と不正系を比較できる

### 第2段階　公開 test case を読める

ここでは次ができればよい。

- `http2-frame-test-case` を読める
- `hpack-test-case` を読める
- 各ケースの主論点を説明できる
- 単発ケースと状態依存ケースを区別できる

### 第3段階　ツールと conformance result を読める

ここでは次ができればよい。

- `nghttp` の出力をフレームと結びつけて読める
- `h2spec` の失敗結果を責務ごとに説明できる
- 実装全体を読解する入口として、どの層の問題かを切り分けられる

---

## 6. できるようにならなくてよいこと

本実習編は、次を直ちに到達目標にはしない。

- RFC の全面読解
- HTTP/2 全フレーム種別の網羅的暗記
- HPACK の完全実装
- Huffman 符号の詳細理解
- HTTP/2 クライアントやサーバのフル実装
- Sans-IO 設計の本格実践
- QUIC / QPACK の詳細理解
- 運用環境での大規模キャプチャ解析

これらは重要な発展課題ではあるが、本実習編の中心は  
**読む順序を身につけること**  
である。ここを超えて目標を広げすぎない。

---

## 7. 学習目標の判定原則

本実習編で「理解できた」とみなす基準は、次のように置く。

### よい理解の判定

- データを見て、まずどこを見るかを言える
- どこで止まるべきかを言える
- どこが分岐点かを言える
- 妥当性の問題か、状態依存の問題かを言える
- 何をまだ扱っていないかを言える

### 不十分な理解の兆候

- 仕様用語を列挙できるが、読む順序が説明できない
- 正常系は説明できるが、不正系との違いが曖昧
- HEADERS と HPACK の境界が曖昧
- parser と validator が混ざっている
- state が必要なケースを単発ケースとして扱ってしまう
- test case 名は読めるが、何を試しているかを言えない

---

## 8. 本実習編で特に育てたいメンタルモデル

本実習編では、次のメンタルモデルが残ることを目指す。

### モデル1
可変長データは、  
**どこまで読むかを決める規則** と  
**その読み方が正当かを確かめる規則**  
の二層でできている。

### モデル2
バイナリ読解では、  
**意味を急ぐ前に、まず境界を見る**  
ことが重要である。

### モデル3
切り出せることと、正しいことは別である。

### モデル4
同じデータでも、前提 state が変われば意味が変わることがある。

### モデル5
公開 test case や conformance result は、  
**答えを教えてくれるもの** ではなく、  
**何を試しているかを読む対象**  
である。

---

## 9. 実習編で特に守るべき一文

本実習編の学習目標は、  
**HTTP/2 と HPACK のデータを、仕様の断片として知ることではなく、境界・分岐・妥当性・状態の観点から段階的に読めるようになること**  
にある。

学習目標の追加・修正・整理は、常にこの方針に従って行う。

---

## 10. まとめ

本実習編の学習目標は、次の三段階に整理できる。

- 最小バイト列を手で読める
- 公開 test case を読める
- ツール出力と conformance result を読める

そのために、読者が最終的に獲得すべき力は次の通りである。

- 境界を切る力
- 分岐点を見つける力
- 妥当性を判断する力
- state の必要性を見抜く力
- 責務を分解して考える力
- test case やツール結果の主論点を読む力

この学習目標に従うことで、本実習編は  
**本編で作った観点を、実データ読解の技能へ変える教材**  
としてまとまりやすくなる。
