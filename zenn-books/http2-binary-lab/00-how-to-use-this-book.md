## はじめに――この実習編で何を訓練するのか

この本は、HTTP/2 や HPACK の仕様項目を暗記するための本ではありません。  
この本が目指すのは、短いバイト列、公開 test case、ツール出力を前にしたときに、**どこから見ればよいかを判断できるようになること**です。

本編『バイナリハック入門 ― UTF-8・HTTP/2・HPACK で学ぶ構造と状態』の役割は、バイト列を  
**境界・分岐・妥当性・状態**  
という観点から読むための見方を作ることにありました。  
それに対して本実習編の役割は、その見方を実際の最小データに適用し、手を動かしながら読解の型として定着させることにあります。

したがって、この本では次のような方針を取ります。

- 最小の正常系から始める
- 最小の不正系を対で置く
- 1 ケース 1 論点を守る
- まず境界を見る
- 次に分岐点を見る
- そのあとで妥当性や状態の話へ進む
- 公開 test case やツール出力は、最後に接続する

この順番を崩さないことが、この本の使い方の中心です。

### 本編と実習編の違い

本編と実習編は、似た題材を扱いますが、読者に求める動作が違います。

本編は、何を見るべきかを学ぶ本です。  
たとえば、次のような観点を作ることが本編の中心でした。

- バイナリデータには境界がある
- 可変長データでは分岐の見方が重要である
- 構造を切り出せることと、妥当であることは別である
- 状態を持つ構造では、単発の parser だけでは足りない

一方、実習編は、実際のデータを前にして、

- どこで境界を見るか
- どこを分岐点として扱うか
- どこで validator を意識するか
- どこから state が必要になるか

を順に練習する本です。  
つまり本編が「見方を作る本」だとすれば、実習編は「その見方を使えるようにする本」です。

### この本で一貫して使う四つの観点

この本では、各章を通して次の四つの観点を繰り返し使います。

#### 境界

まず大事なのは、どこまでが一つのまとまりかを切ることです。  
UTF-8 なら 1 文字分の範囲、HTTP/2 なら 1 フレーム分の範囲、HEADERS なら HTTP/2 フレームとして読める範囲と HPACK block の境界を意識します。

意味を急ぐ前に、まず境界を切る。  
これがこの本で最も大事な基本動作です。

#### 分岐

境界を切ったあとに見るのは、どこで読み方が変わるかです。  
UTF-8 では先頭バイトが長さや後続条件を変えます。  
HTTP/2 では Length、Type、Flags、Stream ID がその後の読み方を左右します。  
HPACK では static table、literal、dynamic table といった表現の違いが分岐点になります。

#### 妥当性

構造を切り出せることと、仕様上正しいことは別です。  
長さが合っているだけでは不十分なケースがありますし、読めるように見えても不正なケースがあります。  
この本では、正常系だけでなく不正系も対で扱うことで、parser と validator の責務を分けて考えられるようにします。

#### 状態

単発で読めるものと、前後関係がなければ読めないものは違います。  
HPACK の dynamic table が典型ですが、同じ raw-data でも前提 state によって解釈が変わる場合があります。  
この本では、state が必要になるケースを早く出しすぎず、単発で読めるケースを十分に経験したあとで導入します。

### 責務の分け方を先に揃える

この本では、実装上の責務もできるだけ分けて考えます。  
特に次の区別は重要です。

#### iterator 的走査

バイト列を前から順に見ていく、もっとも素朴な走査です。  
まだ意味づけをしすぎず、「どこまで進めるか」を見る段階です。

#### parser

構造を切り出し、どこに何が置かれているかを読む役割です。  
HTTP/2 フレーム共通ヘッダ 9 バイトを読むことや、HEADERS の先頭までをフレームとして切り出すことは parser の仕事です。

#### validator

切り出した構造が仕様上妥当かを判定する役割です。  
たとえば Length 不整合、不正な SETTINGS ACK、文脈に合わない stream の使い方などは validator の話です。

#### state 管理

単発では決まらない意味を、前後関係を踏まえて保持する役割です。  
HPACK の dynamic table は、その代表例です。

この本では、これらを混ぜて説明しません。  
「今はどの責務の話をしているのか」を意識しながら読むことで、実装記事やツール出力を見たときにも迷いにくくなります。

### この本の到達目標

この本を読み終えたとき、目指す到達点は次の三段階です。

#### 1. 最小バイト列を手で読める

- UTF-8 の正常系と不正系を見分けられる
- HTTP/2 フレーム共通ヘッダを切り出せる
- 最小の SETTINGS、DATA、GOAWAY、HEADERS を読める
- 「読めるが正しくない」ケースを説明できる

#### 2. 公開 test case を読める

- `http2-frame-test-case` のケースを読める
- `hpack-test-case` の `headers` と `raw-data` の関係を説明できる
- 単発で読めるケースと、state が必要なケースを区別できる

#### 3. ツールや適合性検査の結果を読める

- `nghttp` の verbose 出力をフレーム読解と結びつけられる
- `h2i` や `nghttpd` を観測の道具として位置づけられる
- `h2spec` の失敗結果を見て、どの層の問題かを大まかに説明できる

この本は、HTTP/2 のフル実装をすぐに書けるようになることを直接の目標にはしません。  
その前段階として、**読む順序を身につけること**を目標にします。

### 各実習をどう読むか

この本の各実習では、原則として次の順序を守ってください。

#### 1. まず境界を見る

最初に確認するのは、「どこまでが一つのまとまりか」です。  
意味や用途を急がず、まず範囲を切ります。

#### 2. 次に先頭側の情報を見る

UTF-8 なら先頭バイト、HTTP/2 なら共通ヘッダ、HPACK なら最初の表現種別です。  
ここで、その後の読み方を決める分岐点を探します。

#### 3. そのあとで payload や後続バイトへ進む

境界と分岐点が見えてから、中身を読みます。  
まだそこまで切れていない段階で payload の意味に飛ばないことが大切です。

#### 4. 最後に妥当性と状態を考える

その構造は仕様上正しいか。  
単発で読めるのか。  
前提 state が必要か。  
この確認は最後に行います。

この順序を守るだけで、難しそうに見えるデータもかなり整理して読めるようになります。

### この本で最初からは目指さないこと

この本は、次のことを直ちに目標にはしません。

- RFC の全面読解
- 全フレーム種別の網羅的暗記
- HPACK の完全実装
- Huffman の詳細な解読
- HTTP/2 クライアントやサーバのフル実装
- Sans-IO 設計の本格実践
- QUIC / QPACK の詳細理解

これらは重要な発展課題ですが、この本の中心ではありません。  
この本でまず固めたいのは、**どこから読めば構造が見えるか**という基本動作です。

### この本の章立てをどう使うか

章は、難しさが段階的に増えるように並んでいます。

最初に UTF-8 をウォームアップとして使い、可変長データの基本的な読み方を小さな題材で確認します。  
そのあとで HTTP/2 フレーム共通ヘッダへ進み、最小フレーム、壊れたフレーム、HEADERS の入口、HPACK の最小例、state が必要になるケースへと進みます。  
後半では公開 test case とツール出力に接続し、最後に実装読解や今後の学習課題へ橋を架けます。

この順番には意味があります。  
たとえば HEADERS は、いきなり全部を理解しようとせず、まず HTTP/2 フレームとしてどこまで読めるかで一度止まります。  
HPACK も static table だけで読める最小例から入り、dynamic table はあとで導入します。  
公開 test case やツールは、最初の素材ではなく、読解経験を接続する到達目標として置かれています。

### 読み進めるときのコツ

この本を読むときは、各章で次の三つを意識してください。

第一に、その章で何を訓練しているかを見ることです。  
一つの章ですべてを理解しようとせず、その章の主論点に集中してください。

第二に、まだ扱わないことを意識することです。  
この本は、あえて一度止まる場所を作っています。  
そこでは「まだここから先は読まない」という判断も重要です。

第三に、正常系と不正系の差を丁寧に見ることです。  
何が 1 バイト違うのか、どのフラグが違うのか、どの前提 state が違うのか。  
その差分こそが、読解力を育てます。

### この章で残すべきメンタルモデル

この本は、HTTP/2 や HPACK の知識を増やす本である前に、  
**短いデータを前にして、まず境界を切り、次に分岐点を見つけ、最後に妥当性と状態を考える本**です。

この順序が身につけば、最小バイト列も、公開 test case も、ツール出力も、同じ観点で少しずつ読めるようになります。  
この章でまず持ち帰ってほしいのは、その読み方の型です。

## まとめ

- 本編は見方を作る本であり、実習編はその見方を実データに適用する本である
- この本では、境界・分岐・妥当性・状態の四観点を一貫して使う
- parser、validator、state 管理の責務を混同しないことが重要である
- 各実習では、まず境界、次に分岐点、そのあとに payload、最後に妥当性と状態を見る
- この本の目標は、最小データ、公開 test case、ツール出力を順に読めるようになることである
