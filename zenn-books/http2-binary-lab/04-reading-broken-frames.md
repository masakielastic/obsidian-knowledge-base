## 第4章　壊れたフレームを読む

この章では、正常系の最小フレームから一歩進んで、不正なフレームを読みます。  
ここでの中心は、「HTTP/2 の不正ケースをたくさん暗記すること」ではありません。大事なのは、**どこまでは構造として読めるのか、どこから先が妥当性違反なのかを分けて考えられるようになること**です。

前章までは、共通ヘッダ 9 バイトを切り、Length、Type、Flags、Stream Identifier を読んだうえで、最小の正常系 payload へ入る練習をしました。  
この章では、その同じ読み方を不正データにも適用します。

ここで身につけたいのは、次の感覚です。

- バイト列を切り出せることと、仕様上正しいことは別である
    
- parser が扱う仕事と validator が扱う仕事は違う
    
- どこで止まるべきかを説明できることが重要である
    

正常系だけを読んでいると、HTTP/2 フレーム読解は「共通ヘッダを読めば終わり」のように見えやすくなります。  
しかし実際には、Length が実データ長と合わないこともあれば、ACK を立てた SETTINGS に不正な payload がついていることもあります。  
また、stream 0 と非 0 のように、構造は読めても文脈上正しくないケースもあります。

この章では、不正ケースを通して、**構造読解と妥当性判定は別段階である**ことを体に入れます。

### この章で何を訓練するか

この章では、次のことができるようになることを目指します。

Length と実データ長の不整合を見分けられること。  
不正な SETTINGS ACK を見分けられること。  
stream 0 と非 0 の文脈差を説明できること。  
そして、「このケースは切り出せるが妥当ではない」と言えることです。

ここではまだ、実装全体のエラー処理や接続終了の流れまでは追いません。  
今はまず、1 フレーム単位で、

- どこまで parser が進めるか
    
- どこで validator が止めるべきか
    

を言葉にできるようにします。

---

## 壊れたフレームを読むときの基本姿勢

不正フレームを読むときも、最初の順序は正常系と変わりません。  
ここが重要です。

まず境界を見る。  
次に共通ヘッダを読む。  
そのあとで Type ごとの payload 解釈へ進む。  
最後に、その構造が妥当かどうかを判断する。

不正データを前にすると、つい最初から「どこが悪いのか」を探したくなります。  
ですが、その前に必要なのは、**どこまで読めるかを冷静に切り分けること**です。

たとえば Length が不正でも、共通ヘッダそのものは読めるかもしれません。  
SETTINGS ACK に payload がついていても、Type が SETTINGS で Flags に ACK が立っていること自体は読めます。  
stream 文脈が不正でも、Stream Identifier の値自体は読めます。

つまり、不正ケースで大切なのは、「全部読めない」と雑にまとめないことです。  
読める段階と、読めなくなる段階、読めても許されない段階を分けて考えます。

---

## parser と validator を分ける

本章の中心はここです。  
parser と validator は似ているようで、役割が違います。

parser の仕事は、構造を切り出すことです。  
HTTP/2 なら、

- 9 バイトの共通ヘッダを切る
    
- Length を読む
    
- Type を読む
    
- Flags を読む
    
- Stream Identifier を読む
    
- 必要なら payload の範囲を切る
    

というところまでが parser の中心です。

validator の仕事は、その切り出した構造が仕様上許されるかを判断することです。  
たとえば、

- Length が実データ長と一致するか
    
- SETTINGS ACK に payload があってよいか
    
- その Type が stream 0 で許されるか
    
- reserved bit をどう扱うか
    

といった判断は validator の領域です。

この二つを混ぜると、不正フレームを見たときに「どこが壊れているのか」が曖昧になります。  
本章では、各ケースについて「どこまでは parser で進めるか」「どこから validator が止めるべきか」を明示します。

---

## 実習 1　Length が長すぎるフレームを読む

### このケースで見ること

最初のケースでは、共通ヘッダの Length が、実際に与えられたデータより大きいケースを見ます。  
ここで確認したいのは、Length は「payload の意味」ではなく、まず境界決定そのものに関わる情報だということです。

### データ

たとえば、共通ヘッダでは Length が 4 になっているのに、実際には payload が 2 バイトしか続いていないようなフレームを考えます。

### まず何を見るか

最初にやることはいつも通りです。  
9 バイトの共通ヘッダを切り、Length、Type、Flags、Stream Identifier を読みます。

この段階では、Length が 4 だと読めます。  
Type も Flags も Stream Identifier も読めるかもしれません。

### 次に何を見るか

そのあとで、Length に従って payload の範囲を切ろうとします。  
ここで必要なのは 4 バイトですが、実データを数えると 2 バイトしかありません。

### 妥当性

このケースでは、共通ヘッダは読めます。  
Type も読めます。  
しかし、Length が要求するだけの payload が存在しないので、フレーム全体の境界を正しく成立させられません。

つまり、これは「Type ごとの意味に入る前に、境界確定の段階で止まる」ケースです。

### このケースで十分な到達点

このフレームを見て、

- 共通ヘッダ自体は読める
    
- しかし Length に従った payload 範囲を満たせない
    
- したがって、境界の段階で不正と分かる
    

と説明できれば十分です。

### このケースの意義

このケースは、Length が単なる補助情報ではなく、フレーム読解の土台だということを示します。  
Length が崩れると、そのフレームの payload 解釈だけでなく、次のフレームの開始位置まで壊れます。

---

## 実習 2　Length が短すぎるフレームを読む

### このケースで見ること

今度は逆に、Length が実際のデータより短いケースを見ます。  
ここで確認したいのは、Length 不整合には「足りない」だけでなく「余る」ケースもあるということです。

### データ

共通ヘッダでは Length が 1 になっているのに、実際にはそれより多くの payload らしきバイトが続いているケースを考えます。

### まず何を見るか

やはり最初は共通ヘッダを切ります。  
Length は 1 と読めます。  
Type、Flags、Stream Identifier も読めます。

### 次に何を見るか

Length に従えば、このフレームの payload は 1 バイトだけです。  
したがって parser は、共通ヘッダの直後 1 バイトでこのフレームを終えるはずです。

しかし実際には、そのあとにもバイト列が続いています。  
ここで考えるべきなのは、「それは次のフレームなのか、それともこのフレームに紛れた余剰なのか」です。

### 妥当性

Length で切った位置までなら、ひとまず 1 フレームとして読めるかもしれません。  
ですが、その直後に余剰がある場合、入力全体としては不整合が生じます。

このケースは、「長さ不足」とは違って、いったん境界を切ったあとに余りが出るタイプの不整合です。

### このケースで十分な到達点

このフレームを見て、

- parser は Length に従って 1 バイトで payload を切る
    
- しかし入力全体を見ると余剰があり、整合的なフレーム列として扱いにくい
    
- したがって、これも妥当性の問題になる
    

と説明できれば十分です。

### このケースの意義

このケースが重要なのは、「Length 不整合」という言葉の中に二種類の壊れ方があると分かるからです。  
必要長に足りないケースと、指定長より余るケースでは、止まる場所が少し違います。

## 実習 3　ACK 付き SETTINGS に payload があるケースを読む

### このケースで見ること

ここでは、不正な SETTINGS ACK を見ます。  
狙いは、「構造としてはかなり読めるが、仕様上は許されない」というタイプの違反を体験することです。

### データ

Type は SETTINGS、Flags には ACK が立っている。  
しかし Length は 0 ではなく、payload がついている。  
そういうフレームを考えます。

### まず何を見るか

共通ヘッダを切ります。  
Length を読みます。  
Type を見て SETTINGS だと確認します。  
Flags を見て ACK が立っていることを確認します。  
Stream Identifier も読みます。

ここまでは、かなり普通に読めます。

### 次に何を見るか

Type が SETTINGS で Length が 0 以外なら、本来は payload に設定項目が並ぶ形を想像したくなります。  
しかし今回は Flags に ACK が立っています。

ここで初めて、「この組み合わせは許されるのか」という validator の問いが出てきます。

### 妥当性

このケースは、Length もある、payload もある、Type も Flags も読める。  
つまり parser の観点ではかなり先まで進めます。  
しかし、SETTINGS ACK に payload があるのは不正です。

したがって、このケースは「構造は読めるが、仕様上の意味づけの段階で止まる」例です。

### このケースで十分な到達点

このフレームを見て、

- 共通ヘッダも payload 範囲も読める
    
- しかし SETTINGS と ACK の組み合わせでは payload が許されない
    
- したがって validator の段階で reject される
    

と説明できれば十分です。

### このケースの意義

このケースは、本章で最も「parser と validator の違い」が見えやすい例の一つです。  
長さ不足のように境界段階で壊れるのではなく、構造は成立しているのに意味づけで不正になるからです。

## 実習 4　stream 0 が不適切なフレームを読む

### このケースで見ること

次は、stream 文脈に関する不正を見ます。  
ここでは、「フレームの型は読めるが、その stream で使ってよいとは限らない」ということを確認します。

### データ

たとえば、個別 stream に属することを期待するフレームが、Stream Identifier 0 で送られているケースを考えます。

### まず何を見るか

共通ヘッダを切ります。  
Length を読みます。  
Type を読みます。  
Flags を読みます。  
最後に Stream Identifier を見て、0 だと確認します。

### 次に何を見るか

ここで重要なのは、「0 という値そのものは読めている」ということです。  
つまり parser は失敗していません。  
問題は、その Type のフレームが stream 0 で現れてよいかどうかです。

### 妥当性

このケースでは、フレーム構造はきちんとあります。  
payload も読めるかもしれません。  
しかし文脈としては不正です。

したがって、これは「型の問題」ではなく、「その型がどの文脈で使われるか」の問題です。

### このケースで十分な到達点

このフレームを見て、

- Stream Identifier の値は読める
    
- しかしその Type に対して stream 0 は不適切である
    
- したがってこれは文脈上の妥当性違反である
    

と説明できれば十分です。

### このケースの意義

このケースによって、妥当性には少なくとも二種類あることが見えてきます。  
一つは Length のような構造寄りの妥当性です。  
もう一つは stream 文脈のような意味寄りの妥当性です。

本章では、その両方を「validator の仕事」としてまとめて見ています。

## 実習 5　reserved bit を含むフレームを見る

### このケースで見ること

ここでは、共通ヘッダの細部にある reserved bit を含むケースを考えます。  
狙いは、「ビット単位で見れば何かが立っていることは読めるが、それをどう扱うかは別問題」という感覚を持つことです。

### データ

Stream Identifier を表す 4 バイトのうち、先頭側の reserved bit が立っているように見えるケースを考えます。

### まず何を見るか

いつも通り、共通ヘッダを読みます。  
Length、Type、Flags を確認し、最後に Stream Identifier へ進みます。

### 次に何を見るか

ここで、単純に 32 ビット整数として値を見るのではなく、予約されたビットが混ざっていることに注意します。  
つまり、バイト列は読めるが、その全ビットをそのまま識別子として扱ってよいわけではありません。

### 妥当性

このケースでは、「どのビットが識別子本体で、どのビットが reserved なのか」を切り分ける必要があります。  
そして、その reserved bit をどう扱うかは実装の validator 側の判断や仕様解釈と結びつきます。

### このケースで十分な到達点

このケースを見て、

- バイト列としては読める
    
- しかし Stream Identifier は単純な 32 ビット値として雑に扱えない
    
- 予約領域を含む以上、共通ヘッダ読解にも細かな条件がある
    

と理解できれば十分です。

### このケースの意義

このケースは、共通ヘッダの読み方自体にも、ただの「4 バイト整数を取る」以上の注意があることを示します。  
本章では深入りしませんが、共通ヘッダもまた完全に自明ではないと分かれば十分です。

---

## 壊れ方の種類を整理する

ここまでの不正ケースを並べると、壊れ方には少なくともいくつかの層があることが見えてきます。

一つ目は、Length 不整合のように、境界決定そのものが崩れるケースです。  
これは比較的早い段階で止まります。

二つ目は、SETTINGS ACK に payload があるように、構造は読めるが Type と Flags の意味づけで不正になるケースです。  
これは parser のあとで validator が止めます。

三つ目は、stream 0 のように、構造も payload も見えるが、文脈で不正になるケースです。  
これはより意味寄りの妥当性違反です。

この整理ができると、今後 test case や conformance tool の結果を見たときにも、「どの層で失敗しているのか」を考えやすくなります。

---

## 正常系と不正系をどう比較するか

本章のケースは、前章の最小正常系と対にして読むと効果的です。

空の SETTINGS に対して、不正な SETTINGS ACK を置く。  
単純な DATA に対して、Length 不整合の DATA を置く。  
正しい stream 文脈の例に対して、stream 0 が不適切な例を置く。

こうして比較すると、何が一つだけ違うのかが見えやすくなります。  
この本で繰り返し使う「正常系と不正系を近い形で並べる」という方針は、まさにこのためにあります。

大事なのは、不正ケースを「怖いもの」や「例外の山」として見ることではありません。  
むしろ、正常系との小さな差分として見ることで、何を validator が見ているのかを具体的に理解します。

---

## この章でまだ扱わないこと

ここでは、まだ次のことは本格的に扱いません。

- 接続全体としてどのエラーコードを返すべきか
    
- GOAWAY や RST\_STREAM の実際の送出
    
- HEADERS と HPACK に関する複雑な不正ケース
    
- 状態をまたぐ不整合
    
- conformance tool が返す失敗結果の全体像
    

これらは重要ですが、本章の中心ではありません。  
今はまだ、1 フレーム単位で「どこで壊れているか」を説明できることを優先します。

---

## この章で残すべきメンタルモデル

不正フレームを読むときも、最初の読み順は正常系と同じです。  
まず境界を見る。  
次に共通ヘッダを読む。  
そのあとで payload の構造に入る。  
最後に妥当性を判断する。

そして、本章で特に残したいのは次の区別です。

**切り出せることと、正しいことは別である。**  
**parser が進めることと、validator が許すことは別である。**

この二つが見えていれば、壊れたフレームを前にしても、どこで何が崩れているのかを落ち着いて説明できるようになります。

## まとめ

- 不正フレームも、最初は正常系と同じ順序で読む
    
- Length 不整合は、境界決定の段階で起きる妥当性違反である
    
- 不正な SETTINGS ACK は、構造は読めても意味づけで不正になる例である
    
- stream 0 の不適切な利用は、文脈上の妥当性違反である
    
- reserved bit のように、共通ヘッダ読解にも細かな条件がある
    
- parser と validator の責務を分けて考えることが重要である
