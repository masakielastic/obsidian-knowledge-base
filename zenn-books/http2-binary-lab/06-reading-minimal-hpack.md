## 第6章　HPACK の最小例を読む

この章では、前章で切り出した header block fragment の中へ入ります。  
ここから扱うのは、HTTP/2 フレームそのものではなく、HEADERS フレームの先にある別の構造です。つまり本章の主題は、**HPACK を最小例で読むこと**です。

ただし、ここでも目標は「HPACK の全仕様を理解すること」ではありません。  
この章で重視するのは、圧縮理論の詳細でも、Huffman の完全解説でもありません。まず必要なのは、**HPACK を可変長の別レイヤーとして見て、どこで区切り、どの種類の表現を読んでいるかを説明できるようになること**です。

前章では、HEADERS フレームを見たときに、ここまでは HTTP/2 フレームとして読める、ここから先は HPACK だと一度止まる練習をしました。  
本章では、その「ここから先」を初めて読みます。

ここで身につけたいのは、次の感覚です。

- HPACK は HEADERS の中にある別の可変長構造である
    
- 最初の 1 バイト付近が、その後の表現の種類を大きく分岐させる
    
- raw-data と最終的な headers の対応を、小さな例で追える
    
- ただし最初から state や story を持ち込まない
    
- まずは単発で読める最小例から始める
    

つまり本章は、**HEADERS の先にある別の parser の入口を作る章**です。

### この章で何を訓練するか

この章では、次のことができるようになることを目指します。

static table を使う最小ケースを読めること。  
literal header の最小ケースを読めること。  
raw-data と headers の対応を説明できること。  
HPACK を、HTTP/2 フレームとは別の可変長構造として見られること。  
そして、**単発で読める最小ケースと、まだ先送りすべき複雑さを区別できること**です。

ここではまだ、dynamic table は扱いません。  
state をまたぐ story も扱いません。  
Huffman の詳細な読解にも入りません。  
今はまず、single-shot で読める HPACK の最小例を使って、「何が始まっているのか」を見えるようにします。

---

## HPACK をどう見るか

HTTP/2 フレームを読んでいるときは、共通ヘッダ 9 バイトが入口でした。  
HPACK に入ると、その入口は変わります。

ここではまず、**header block fragment は、さらに小さな表現の列からできている**と考えます。  
その表現にはいくつかの種類があり、先頭バイトの上位側のビットパターンによって、大まかな分類が決まります。

本章で大事なのは、その分類を完璧に暗記することではありません。  
重要なのは、**最初の 1 バイトを見て、これはどの種類の表現かを判断し、その種類ごとの読み方に分岐する**ということです。

この意味で、HPACK は前章までの HTTP/2 フレームとよく似ています。

HTTP/2 フレームでは、まず共通ヘッダがあり、Type が payload の読み方を分岐させました。  
HPACK では、まず表現の先頭バイトがあり、その先頭ビット列が後続の読み方を分岐させます。

したがって、本章の読み順も似ています。

まず、1 個の表現の開始地点を見る。  
次に、先頭バイトでどの種類かを判断する。  
そのあとで、その種類に応じて後続バイトを読む。  
最後に、その表現がどのヘッダへ対応するかを確認する。

この順序を守ることが重要です。

---

## なぜ static から始めるのか

HPACK を難しくしているのは、単に圧縮されているからではありません。  
本当に難しいのは、**複数の表現形式があり、しかも後になると状態依存まで入ってくること**です。

もし最初から dynamic table や story を持ち込むと、読者は次のどこで迷っているのかが分からなくなります。

- 先頭バイトの種類判定で迷っているのか
    
- 長さ付き文字列の切り方で迷っているのか
    
- ヘッダ名と値の組み合わせで迷っているのか
    
- state 依存で迷っているのか
    

そのため、本章ではまず static table を使う最小ケースから始めます。  
static table のよいところは、単発で読めることです。  
つまり、「前に何が送られたか」を知らなくても、1 個の表現だけで完結して読めます。

これにより、本章では HPACK を「状態付き圧縮機構」としてではなく、まず **HEADERS の中に入っている別の可変長構造**として理解できます。

---

## raw-data と headers の関係を見る

本章では、header block fragment を「なんとなく圧縮された塊」として見ないようにします。  
代わりに、**raw-data と最終的な headers の対応を見る**ことを重視します。

たとえば、ある短い raw-data があったとして、それが最終的に

- `:method: GET`
    
- `:path: /`
    

のようなヘッダ表現につながるなら、その対応を小さく追います。

ここで重要なのは、「どう圧縮率が高いか」ではありません。  
見るべきなのは、**この表現はどのヘッダ名・値の組を指しているのか**です。

HTTP/2 フレームのときも、共通ヘッダを読んだあとに payload の意味へ入る必要がありました。  
HPACK でも同じで、先頭バイトで表現の種類を読んだあとに、その表現がどんなヘッダを表すかを追う必要があります。

---

## 本章で使う読み順

本章では、各ケースを次の順序で読みます。

まず、1 個の HPACK 表現の開始地点を見る。  
次に、先頭バイトを見て、表現の種類を判断する。  
そのあとで、必要ならインデックスや長さ付き文字列を読む。  
最後に、その表現がどの header に対応するかを確認する。

ここでも大事なのは、いきなり最終的な header 一覧から逆算しないことです。  
必ず raw-data 側から順に見ます。

つまり、

まず bytes を見る。  
次に構造を切る。  
そのあとで意味に入る。

という順序を守ります。  
これは本書全体で繰り返してきた型そのものです。

---

## 実習 1　static table を使う最小ケースを読む

### このケースで見ること

最初の実習では、static table の既存エントリを 1 個参照するだけの最小ケースを見ます。  
狙いは、HPACK にも「まず最小の正常系がある」と確認することです。

### データ

header block fragment が 1 個の表現だけで構成され、その表現が static table のある項目を指しているケースを考えます。  
たとえば `:method: GET` のように、よく使われる組を既存表からそのまま参照する最小例です。

### まず何を見るか

最初の 1 バイトを見ます。  
ここで大事なのは、「これは文字列がそのまま書いてあるのではなく、何かを指している表現らしい」と判断することです。

つまり、先頭ビットから「インデックス参照型だ」と分かることが第一歩です。

### 次に何を見るか

そのあとで、どのインデックスを指しているのかを読みます。  
今回は最小例なので、追加の複雑な拡張長や長い値は考えません。

そして、そのインデックスが static table のどの項目に対応するかを確認します。

### 最後に何を見るか

その結果として、最終的な header が何になるかを確認します。  
ここで初めて raw-data と `:method: GET` のような意味がつながります。

### このケースで十分な到達点

このケースを見て、

- 先頭バイトからインデックス参照だと分かる
    
- そのインデックスを static table で引ける
    
- 結果として 1 個の header を復元できる
    

と説明できれば十分です。

### このケースの意義

このケースのよいところは、HPACK が必ずしも「文字列を全部読む」処理ではないと分かることです。  
場合によっては、短い表現が既存表の項目を直接指して終わります。

つまり HPACK の最初の入口は、「文字列が見えるかどうか」ではありません。  
まずは、**これは参照なのか、埋め込みなのか**を見分けることです。

---

## 実習 2　static table を 2 個並べた最小ケースを読む

### このケースで見ること

今度は、header block fragment の中に表現が 1 個ではなく、2 個続くケースを見ます。  
狙いは、HPACK が 1 個の表現で終わるとは限らず、複数の表現の列として読む必要があることを確認することです。

### データ

2 個のインデックス参照が並び、たとえば `:method: GET` と `:path: /` のような 2 つの header に対応するケースを考えます。

### まず何を見るか

最初の表現の開始地点を決めます。  
先頭バイトから種類を判定し、インデックスを読み、1 個目の header を得ます。

### 次に何を見るか

そこで終わらず、次のバイト位置へ進みます。  
次の表現についても同じことを行います。

### 最後に何を見るか

2 個の表現を順に読んだ結果として、最終的に 2 個の header が得られることを確認します。

### このケースで十分な到達点

このケースを見て、

- HPACK block は 1 個の表現ではなく、複数の表現の列として読める
    
- 各表現について、先頭バイトから種類を判定する
    
- その結果として header 群が復元される
    

と説明できれば十分です。

### このケースの意義

このケースによって、HPACK の「可変長構造らしさ」が少し見えてきます。  
つまり、先頭から順に 1 個ずつ表現を切っていき、その列として読む必要があります。  
この感覚は、あとで literal や dynamic table のケースに進んでも重要です。

---

## 実習 3　literal header の最小ケースを読む

### このケースで見ること

ここでは、既存表を参照するだけでなく、literal header を使う最小ケースを見ます。  
狙いは、「HPACK には参照だけでなく、文字列を伴う表現もある」と確認することです。

### データ

たとえば、ヘッダ名や値の少なくとも一部を長さ付き文字列として含む、最小の literal header ケースを考えます。  
ここでは最小例にとどめ、極端に長い文字列や複雑なエンコードは扱いません。

### まず何を見るか

先頭バイトを見て、今度は static table の完全参照ではなく、literal 系の表現だと判断します。  
ここが最初の分岐です。

### 次に何を見るか

そのあとで、必要ならヘッダ名が既存表を参照しているのか、それとも名前自体が文字列として入っているのかを確認します。  
さらに、値について長さ付き文字列を読みます。

ここで重要なのは、「文字列が現れたからすぐ意味へ飛ぶ」のではなく、まず長さを読んでから文字列本体へ進むことです。

### 最後に何を見るか

その literal 表現が、最終的にどのヘッダ名と値の組になるかを確認します。

### このケースで十分な到達点

このケースを見て、

- 先頭バイトから literal 系だと分かる
    
- 長さ付き文字列を順に切り出せる
    
- その結果として 1 個の header を復元できる
    

と説明できれば十分です。

### このケースの意義

このケースは、HPACK が「参照だけ」ではなく、「埋め込み文字列も扱う」ことを示します。  
同時に、文字列が見えても読む順序は変わらないことを確認できます。  
つまり、**まず種類判定、次に長さ、最後に文字列**です。

---

## 実習 4　名前は既存、値だけ literal のケースを読む

### このケースで見ること

今度は、ヘッダ名は既存のものを使い、値だけを literal として与えるような最小ケースを見ます。  
狙いは、HPACK の literal が「全部生文字列」とは限らないことを確認することです。

### データ

ヘッダ名は static table の既知項目を参照し、値だけが長さ付き文字列で与えられるケースを考えます。

### まず何を見るか

先頭バイトを見て literal 系だと判断します。  
そのあとで、名前部分はインデックス参照だと読みます。

### 次に何を見るか

続いて値の長さを読み、値の文字列本体を切り出します。

### 最後に何を見るか

名前と値を組み合わせて、1 個の完全な header を得ます。

### このケースで十分な到達点

このケースを見て、

- literal 系でも、全部が生文字列とは限らない
    
- 名前は参照、値は埋め込み、という混合形がある
    
- その場合も、順に構造を切っていけば読める
    

と説明できれば十分です。

### このケースの意義

このケースは、HPACK の表現が「参照」と「literal」の二択で単純に分かれるわけではないことを示します。  
ただし、本章の段階では複雑に考えすぎる必要はありません。  
大事なのは、**どの部分が既存表から来て、どの部分が今回の block に埋め込まれているか**を区別することです。

---

## 実習 5　raw-data と復元 header の対応を一行ずつ確認する

### このケースで見ること

最後の実習では、短い header block fragment について、raw-data の並びと復元される headers の対応を、あえて一行ずつ確認します。  
狙いは、HPACK を「ブラックボックスの圧縮結果」として見ないことです。

### データ

static table の参照と literal が 1 つずつ含まれるような短いケースを考えます。

### まず何を見るか

最初の表現を切り出し、それがどの header を表すかを確認します。  
次の表現を切り出し、それがどの header を表すかを確認します。

### 次に何を見るか

この対応を、表現ごとに並べて見ます。  
つまり、

この bytes はこの表現。  
この表現はこの header。  
次の bytes は次の表現。  
次の表現は次の header。

という対応を順に追います。

### このケースで十分な到達点

このケースを見て、

- raw-data の各部分が、どの表現に対応するか
    
- 各表現が、どの header に対応するか
    
- その列として、最終的な header 群ができるか
    

を説明できれば十分です。

### このケースの意義

この実習は、本章全体のまとめです。  
HPACK は「圧縮されていて難しい」ものではありますが、少なくとも最小ケースでは、raw-data と header の対応を追えます。  
ここが見えると、HPACK は急に「読めない塊」ではなくなります。

---

## HPACK でも parser と validator を分けて考える

本章では主に正常系を扱っていますが、ここでも parser と validator を分けて考える姿勢は有効です。

parser の仕事は、

- 1 個の表現の開始地点を見つける
    
- 先頭バイトから種類を判定する
    
- 必要な長さ付き要素を切り出す
    

ことです。

validator の仕事は、その切り出した表現が HPACK として妥当か、長さが足りるか、許される参照か、といったことを確認することです。

本章ではまだ不正ケースを詳しく扱いません。  
ですが、前章までの流れを踏まえるなら、HPACK にも同じ区別があると先に意識しておく価値があります。

---

## Huffman を今は深追いしない理由

HPACK を学び始めると、Huffman の話が気になりやすいかもしれません。  
たしかに重要です。  
しかし本章でそれを主役にしない理由は単純で、今はまず **HPACK がどんな単位の列として読めるか** を理解することが先だからです。

もし最初から Huffman の詳細に入ると、読者は

- 先頭バイトの種類判定
    
- 長さ付き文字列の切り出し
    
- 名前と値の区別
    
- static table と literal の違い
    

といった基本動作より前に、符号化の細部へ意識を取られます。

そのため、本章では Huffman を「文字列の中身をどう表すかに関わる別の複雑さ」と位置づけ、中心には置きません。  
まずは、HPACK の大枠の構造が見えることを優先します。

---

## この章でまだ扱わないこと

ここでは、まだ次のことは本格的に扱いません。

- dynamic table
    
- 以前の入力に依存する story
    
- 同じ raw-data が state によって違う意味になるケース
    
- Huffman の詳細な復号
    
- HPACK の不正ケース
    
- 複雑な header block の断片化
    

これらは次章以降で扱います。  
本章の中心は、**単発で読める最小の HPACK 表現を通して、先頭バイトによる種類分岐と raw-data / headers の対応を見えるようにすること**です。

---

## この章で残すべきメンタルモデル

HPACK を読むときも、いきなり復元後の header 一覧へ飛んではいけません。  
まず raw-data を見る。  
次に 1 個の表現を切る。  
その先頭バイトから、どの種類の表現かを判断する。  
必要ならインデックスや長さ付き文字列を読む。  
最後に、その表現がどの header を表すかを確認する。

本章で残したいのは、この読み方です。

つまり、HPACK は「読めない圧縮データ」ではなく、  
**先頭バイトで種類が分かれ、複数の表現の列として順に読める可変長構造**です。

このモデルができれば、次章で state が必要になるケースに入ったときにも、どこから難しさが増えたのかを見失いにくくなります。

## まとめ

- HPACK は HEADERS フレームの先にある別レイヤーの可変長構造である
    
- 最初の 1 バイト付近が、その表現の種類を大きく分岐させる
    
- 本章ではまず static table を使う最小ケースから始める
    
- literal header の最小例では、種類判定、長さ、文字列の順で読む
    
- raw-data と復元される headers の対応を小さく追うことが重要である
    
- 本章では dynamic table や Huffman の詳細にはまだ踏み込まない
    

## 次章への接続

次章では、dynamic table を導入し、**state が必要になると何が変わるか**を見ます。  
そこで初めて、同じ raw-data でも前提 state によって意味が変わることを確認し、HPACK が単発読解だけでは足りなくなる地点へ進みます。
